# Iteration 004: Response Hint Mechanisms and Routing Hint Design

## Focus: Analyze existing MCP response hint and redirect mechanisms, then design a reusable response hint injection layer that nudges assistants toward CocoIndex for semantic code discovery and Code Graph for structural analysis without causing over-aggressive redirects.

## Findings:
1. The universal reusable hint carrier already exists: every structured MCP response envelope has a top-level `hints: string[]` field alongside `summary`, `data`, and `meta`. Success, empty, and error responses all flow through the same envelope factory, so routing guidance does not need a new schema or transport. Error responses already flatten `recovery.hint`, `recovery.actions`, and `recovery.toolTip` into `hints[]`, which proves the channel is intended for actionable redirect/recovery text rather than pure diagnostics. 2. Auto-surface is the first existing hint injection mechanism. `appendAutoSurfaceHints()` parses the returned JSON envelope, normalizes `envelope.hints`, appends a human-readable string (`Auto-surface hook: injected ...`), and writes structured metadata to `meta.autoSurface`. This is important because it shows the server already has a sanctioned post-dispatch enrichment pattern that mutates both human-facing hints and machine-readable metadata before final serialization. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/hooks/response-hints.ts:86-140] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:813-817]

3. Session priming plus PrimePackage is the second existing hint injection mechanism, but it is explicitly post-selection. On the first tool call, `primeSessionIfNeeded()` builds a `PrimePackage` containing `specFolder`, `currentTask`, `codeGraphStatus`, `cocoIndexAvailable`, and `recommendedCalls`; later `injectSessionPrimeHints()` appends strings like `Active spec folder: ...`, `Code graph: ..., CocoIndex: ...`, and `Recommended next calls: ...` into `envelope.hints`, while also persisting the full structured object into `meta.sessionPriming`. This makes PrimePackage useful as a corrective hint layer for the second tool decision, but not as a first-call enforcement mechanism. 4. The current context extraction path is already suitable for routing-intent detection. `extractContextHint()` pulls a query-like string from `input`, `query`, `prompt`, `specFolder`, `filePath`, or joined `concepts[]`, and both `autoSurfaceAtToolDispatch()` and the memory-aware path reuse that extraction before dispatch. A routing hint layer can therefore piggyback on the same normalized query text instead of inventing a new request parser. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:121-137] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:453-475] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:741-758]

5. There is already a generic post-dispatch enrichment slot beyond auto-surface/session-prime: passive enrichment. After `dispatchTool()` returns, the server may call `runPassiveEnrichment(...)`; if it returns `enrichment.hints`, those are appended into `envelope.hints` and reserialized. This is the cleanest architectural precedent for a routing hint injector because it is tool-agnostic, late-bound, non-fatal, and designed to add hints without changing core handler logic. A routing layer should mimic this pattern rather than be implemented ad hoc inside individual handlers. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:788-805]

6. Token-budget and error-recovery hinting prove that `hints[]` is already a mixed advisory channel, not a narrowly scoped auto-surface channel. After enrichment, the server may append `Token budget enforced: ...` or `Response exceeds token budget ...`, and fallback error paths also emit recovery hints. This means routing hints can coexist with other guidance as long as they are brief, deduplicated, and lower-volume than hard recovery messages. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:843-879] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/lib/response/envelope.ts:240-275]

7. `buildServerInstructions()` remains the only pre-selection, all-CLI injection point, but it is not itself a response hint mechanism. At startup, the server computes a short instruction string and passes it through `serverWithInstructions.setInstructions(...)`. Today that string covers memory counts, channel availability, key memory tools, stale-memory warning, and session-recovery digest; it does not encode semantic-vs-structural routing. Therefore the response hint layer should be treated as the corrective/iterative half of enforcement, while `buildServerInstructions()` carries the first-call routing policy for hook-compatible and non-hook CLIs alike. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:605-646] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:1593-1598]

8. The best response-hint design is a dedicated routing enrichment stage in `context-server.ts`, positioned after `dispatchTool()` and before token-budget enforcement, parallel to passive enrichment and before final metadata sync. Concretely: add a `runRoutingHintEnrichment({ toolName, args, result, sessionPrimeContext, autoSurfacedContext })` helper that (a) reuses `extractContextHint()` for the query string, (b) classifies intent as `semantic_discovery`, `structural_graph`, `exact_lookup`, or `unknown`, (c) checks environment state such as `cocoIndexAvailable` and `codeGraphStatus`, and (d) appends at most one concise hint plus structured `meta.routingHint = { intent, recommendedTool, reason, confidence, suppressedReason? }`. Example emitted strings:
   - `Routing hint: semantic code discovery detected. Prefer cocoindex_code-search for concept/intent queries.`
   - `Routing hint: structural dependency query detected. Prefer code_graph_query/code_graph_context for callers, imports, and impact.`
   - `Routing hint: code graph is stale/empty. Prefer CocoIndex until code_graph_scan refreshes graph health.`
   This fits the already established enrichment model and keeps the mechanism transport-neutral because it only uses existing envelope fields. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:788-879] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:316-364] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:366-415]

9. The semantic-search trigger should be pattern-based but conservative. The classifier should only emit a CocoIndex redirect when the extracted query strongly looks like intent-based code discovery: phrases such as `how does`, `where is the logic`, `find code that`, `implementation of`, `similar code`, `pattern for`, `search codebase for`, or `unfamiliar module`, combined with the absence of exact-path or exact-token signals. Structural terms such as `callers`, `imports`, `impact`, `dependency`, `outline`, `references`, or `what calls` should route toward Code Graph instead. Queries that look like exact matches or known-path lookups (`src/...`, `.ts`, `.py`, regex syntax, glob syntax, explicit symbol text) should suppress the routing hint entirely. This preserves the intended division from prior iterations: semantic discovery -> CocoIndex; structural relationships -> Code Graph; exact lookup -> Grep/Glob. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:121-137] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:605-646]

10. The main failure modes of over-aggressive routing are false semantic redirects, stale-graph redirects, hint fatigue, tool thrash, and late-stage interference with recovery/error guidance. The escape hatches should therefore be explicit:
   - Confidence gate: only emit when classifier confidence crosses a high threshold; otherwise set `meta.routingHint.suppressedReason = 'low_confidence'`.
   - Exact-match short-circuit: suppress when query appears path-like, regex-like, symbol-like, or contains explicit file extensions/glob operators.
   - Correct-tool short-circuit: do not redirect if the current tool already matches the inferred intent.
   - Availability gate: never recommend Code Graph when `codeGraphStatus` is `stale` or `empty`; never recommend CocoIndex if `cocoIndexAvailable` is false.
   - Error-channel priority: suppress routing hints on error responses unless the routing issue is itself the clearest recovery action, so recovery hints remain dominant.
   - Session dedupe: emit once per normalized query or once per session phase to avoid repeated nudges.
   - Soft-suggestion mode: keep the hint advisory, not blocking; do not reject the current tool call or rewrite results.
   - Token cap: reserve a tiny budget (for example one short hint) so routing hints do not displace substantive response content.
   These escape hatches make the layer safe as a correction mechanism while avoiding the classic failure mode where every vaguely exploratory prompt gets over-redirected to CocoIndex. 11. Final design conclusion: response hints can and should enforce the "next tool choice" using the already-deployed `hints[]`/`meta.*` enrichment pipeline, but they cannot solve the root cause alone because they arrive after a tool has already been chosen. The root-cause fix still requires a concise routing stanza in `buildServerInstructions()` for first-call behavior across hook-compatible and non-hook CLIs. The response hint layer is therefore best designed as a secondary corrective loop that reinforces the same routing matrix after the first tool call and adapts recommendations to current graph/CocoIndex availability. ## Key Questions Answered:
- Q7: Answered. Existing reusable hint/redirect mechanisms are the universal response `hints[]` array, auto-surface hint append + `meta.autoSurface`, session-prime/PrimePackage hint append + `meta.sessionPriming`, passive enrichment hint append, error recovery hints, and token-budget warning hints. `buildServerInstructions()` is the separate pre-selection mechanism.
- Q10: Answered. The major failure modes are false semantic redirects, stale-graph redirects, repeated nagging, tool thrash, and interference with recovery/error hints. Safe escape hatches are high-confidence gating, exact-match suppression, correct-tool suppression, availability checks, one-shot/session dedupe, and advisory-only wording.
