# Iteration 001: Tool Selection Decision Path Mapping
## Focus: Map the current tool selection decision path — trace how an AI session receives tool information from MCP server instructions, PrimePackage, tool descriptions, and instruction files.

## Findings:
1. `buildServerInstructions()` is the only pre-tool, server-wide instruction surface inside this MCP server. At startup, `context-server.ts` calls `buildServerInstructions()` and passes the resulting string to `serverWithInstructions.setInstructions(...)` before connecting transport, so any MCP client that surfaces server instructions can receive this text before choosing a tool. Today that text only reports memory counts, active/stale counts, search channels, key memory tools, and an optional session-recovery digest; it does not contain any routing policy for Grep/Glob vs CocoIndex vs Code Graph. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:605-646] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:1593-1598]
2. Tool descriptions are the second major decision surface, but they are capability-oriented rather than routing-oriented. `ListToolsRequestSchema` returns `TOOL_DEFINITIONS`, so clients see the descriptions from `tool-schemas.ts` when enumerating tools. The current `code_graph_query` description explains structural operations (`outline`, `calls_from`, `calls_to`, `imports_from`, `imports_to`), and `code_graph_context` explains compact graph neighborhoods plus CocoIndex-seeded expansion, but neither description says when to prefer Code Graph over CocoIndex or Grep/Glob. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:685-687] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/tool-schemas.ts:637-693]
3. PrimePackage is a post-selection correction mechanism, not a pre-selection router. On the first tool call, `primeSessionIfNeeded()` builds a `PrimePackage` from the incoming tool args and code-graph snapshot, recording `specFolder`, `currentTask`, `codeGraphStatus`, `cocoIndexAvailable`, and `recommendedCalls`. That package is only created after a tool has already been chosen, and later injected into the tool response envelope as hints such as `Code graph: fresh, CocoIndex: available` and `Recommended next calls: ...`. This means PrimePackage can improve the second tool decision, but it cannot prevent the first misjudgment. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:317-364] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:366-415] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:530-565] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:833-839]
4. The exact current decision path is layered and mostly heuristic: runtime instruction files provide the strongest explicit routing policy, server instructions provide only session/memory context, tool descriptions provide per-tool capability summaries, and PrimePackage/auto-surface hints arrive only after the first tool response. For Copilot/Gemini-style runtime docs, the code-search protocol explicitly says to use CocoIndex first for concept/intent queries, Grep for exact text, Glob for file paths, and to fall back to CocoIndex when the graph is stale. Codex has a similar `Query-Intent Routing` section that maps semantic discovery to CocoIndex and structural navigation to `code_graph_query` / `code_graph_context`. [SOURCE: GEMINI.md:71-105] [SOURCE: CLAUDE.md:34-67] [SOURCE: CODEX.md:14-33]
5. The best MCP-server injection point for all CLIs is therefore `buildServerInstructions()`, because it is the only centralized surface in this server that can talk about both internal Code Graph tools and the external CocoIndex provider from one place before tool selection happens. By contrast, `tool-schemas.ts` can only influence tools owned by this MCP server, and PrimePackage is necessarily late. This matters because CocoIndex is not defined in `tool-schemas.ts`; it is registered separately in `opencode.json` as the `cocoindex_code` provider. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:605-646] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/tool-schemas.ts:637-693] [SOURCE: opencode.json:40-49]
6. A secondary universal injection point is to strengthen the descriptions of `code_graph_query`, `code_graph_context`, `code_graph_status`, and `session_resume` so they encode routing intent for structural tasks and stale-graph fallback. Those descriptions are returned to every MCP client via `ListToolsRequestSchema`, so they are likely the strongest per-tool lever available from this server. However, because CocoIndex lives on another server, per-tool description changes here cannot fully enforce semantic-vs-structural routing by themselves. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:685-687] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/tool-schemas.ts:654-744] [SOURCE: opencode.json:40-49]
7. `buildServerInstructions()` can be extended within a tight token budget. A concise addition under ~200 tokens would be enough to encode a routing policy such as: `Routing: exact token/path lookups -> grep/glob; semantic/intent queries or unfamiliar code -> CocoIndex; callers/imports/neighborhood/impact -> code_graph_query/context; if graph is stale/empty, use CocoIndex until code_graph_scan/session_resume restores graph health.` This would fill the current gap without requiring new tools. The current function already appends dynamic lines and session-recovery guidance, so adding 1-2 routing lines is mechanically straightforward. ## Key Questions Answered:
- Q1: Answered — the decision path is runtime docs -> MCP server instructions -> MCP tool descriptions -> first tool choice -> PrimePackage/result hints for subsequent choices.
- Q2: Answered — the strongest universal MCP injection point is `buildServerInstructions()`, with `tool-schemas.ts` as a secondary per-tool lever and PrimePackage only as a post-selection correction layer.
- Q3: Answered — `buildServerInstructions()` currently returns memory inventory + search-channel + key-tool + optional session-recovery text, and can be safely extended with a concise routing stanza.
