# Iteration 003: Query Classification Patterns for Tool Routing
## Focus: Analyze query classification patterns that distinguish semantic discovery (CocoIndex), structural navigation (Code Graph), and exact-text/path lookup (Grep/Glob), then design a lightweight routing classifier.

## Findings:
1. The MCP server already contains a lightweight structural-vs-semantic classifier, but it is only a 3-state `structural | semantic | hybrid` heuristic and is currently used inside `memory_context`, not as a universal first-tool router. `classifyQueryIntent()` scores keyword hits at 1x, regex-pattern hits at 2x, and picks a dominant class only when one side exceeds 65% of the total score; otherwise it falls back to `hybrid`. `memory_context` then uses that result to optionally attach code-graph context for `structural` or `hybrid` queries, while `semantic` or low-confidence queries simply fall through to the existing semantic path. 2. Structural queries are characterized by relationship and graph-navigation language rather than general discovery language. The existing classifier and tests consistently key on verbs/nouns such as `calls`, `imports`, `exports`, `extends`, `implements`, `callers`, `callees`, `dependencies`, `definition`, `outline`, `impact`, `hierarchy`, plus patterns like `who calls`, `what imports`, `show callers`, `class hierarchy`, and `impact of`. Tool descriptions line up with the same shape: `code_graph_query` advertises `outline`, `calls_from/calls_to`, and `imports_from/imports_to`, while `code_graph_context` advertises `neighborhood`, `outline`, and `impact`. These are strong routing cues for Code Graph. 3. Semantic queries are characterized by concept, behavior, or intent language: the user wants examples, patterns, implementations, purpose, or explanation rather than graph edges. The existing classifier already keys on terms like `similar`, `related`, `example`, `pattern`, `usage`, `approach`, `strategy`, `find`, `search`, `discover`, `explain`, `understand`, `purpose`, `concept`, and domain concepts such as `authentication`, `middleware`, or `validation`, along with patterns like `find code ... about`, `implementations of`, `examples of`, and `what is the pattern/purpose`. 4. The missing category is exact-text/path lookup. Runtime instructions already encode the rule (`Grep` for exact text/token/symbol, `Glob` for known file/path patterns), but the MCP-side classifier does not model that class at all: the classifier type only exposes `structural | semantic | hybrid`, and its tests cover only those cases. That gap explains why assistants can still misjudge literal lookups and send them to semantic or structural tools. The most reliable exact-match signals are lexical surface-form markers rather than conceptual verbs: `exact`, `exactly`, `literal`, `verbatim`, `string`, `text`, `token`, `regex`, `grep`, `contains`, quoted snippets, slash-heavy path fragments, file extensions, glob characters (`*`, `**`, `{}`), and prompts like `file named`, `path`, or `where is function named exactly X`. These are high-precision cues and should be evaluated before semantic/structural scoring. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/lib/code-graph/query-intent-classifier.ts:11-19] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/tests/query-intent-classifier.vitest.ts:11-94] [SOURCE: CLAUDE.md:44-50] [SOURCE: AGENTS.md:78-104]

5. A lightweight classifier can therefore be designed as a conservative 4-label router: `exact_text`, `structural`, `semantic`, and `hybrid`. Recommended decision order:
   - Stage A: detect exact-match/path requests with high-precision regexes and token features. Example strong patterns: quoted substrings, explicit `exact|exactly|literal|verbatim|string|regex|grep`, path/glob markers, or `file named/path`.
   - Stage B: if exact-match is not strong, reuse the existing structural/semantic scorer almost unchanged.
   - Stage C: if structural and semantic scores are both present and close, return `hybrid`; if no signal exists, default to `hybrid` rather than forcing a wrong redirect.
   - Stage D: keep escape hatches: any query with relational verbs (`who calls`, `what imports`, `impact of`) should override incidental quotes, while explicit literal/path search should override broad semantic verbs such as `find`.
   This design stays cheap, explainable, and implementation-local: it extends the current heuristic instead of introducing embeddings or model-based classification. 6. For Q9, server-instruction changes remain the highest-leverage enforcement surface, tool-description changes are second, and response hints are corrective-only. `buildServerInstructions()` is pushed through `setInstructions(...)` before transport is connected, so it is the only investigated server-wide, pre-tool-choice injection point. `ListToolsRequestSchema` returns `TOOL_DEFINITIONS`, so tool descriptions are also visible before selection, but only for this MCP server's tools; they cannot fully steer CocoIndex because it is a separate provider. By contrast, PrimePackage hints are injected only after a tool call succeeds, so they can improve the second choice but cannot prevent the first misroute. 7. The practical enforcement shape is now clear: put the 3-way/4-way routing rule into `buildServerInstructions()` as the universal policy, mirror concise structural-vs-semantic trigger wording in `tool-schemas.ts` and the CocoIndex-side description where available, and use response hints only as recovery guidance after a misroute. The classifier does not need to hard-dispatch tools; it only needs to produce routing hints like `exact_text -> grep/glob`, `semantic -> CocoIndex`, `structural -> code_graph_query/context`, `hybrid -> CocoIndex first, then graph enrichment if graph healthy`. That keeps false redirects low while giving all runtimes the same pre-selection decision rubric.
