# Iteration 033: CocoIndex Code Integration Options

## Focus

Assess how the repo's existing CocoIndex Code integration works, what it is good at, where it stops short for structural code understanding, and whether the planned code graph work should reuse parts of CocoIndex or remain a separate retrieval channel.

## Findings

1. CocoIndex Code in this repo is a semantic code search subsystem with two access modes: a full CLI (`ccc`) for search and index lifecycle operations, and an MCP server (`ccc mcp`) that exposes exactly one tool, `search`. The MCP surface is intentionally narrow: index management (`init`, `index`, `status`, `reset`, `daemon`) stays CLI-only, while MCP is limited to natural-language search with language/path filters and result-count control. [SOURCE: `.opencode/skill/mcp-coco-index/SKILL.md:187-242`; `.opencode/skill/mcp-coco-index/references/tool_reference.md:21-24`; `.opencode/skill/mcp-coco-index/references/tool_reference.md:183-199`; `.opencode/skill/mcp-coco-index/README.md:42-46`; `.opencode/skill/mcp-coco-index/README.md:143-175`]

2. CocoIndex semantic search is chunk-embedding retrieval, not symbol-table or AST traversal. The documented pipeline is: scan project files while respecting `.gitignore`, split files with language-aware boundaries into 1000-character chunks with 250-character minimum and 150-character overlap, embed those chunks, store vectors in SQLite via `sqlite-vec`, embed the query with the same model, then rank by cosine similarity; language/path filters are applied after ranking rather than replacing semantic ranking. The daemon auto-starts, supports incremental indexing, and keeps serving subsequent searches. [SOURCE: `.opencode/skill/mcp-coco-index/SKILL.md:278-317`; `.opencode/skill/mcp-coco-index/references/settings_reference.md:71-118`; `.opencode/skill/mcp-coco-index/references/settings_reference.md:142-160`; `.opencode/skill/mcp-coco-index/README.md:46-47`; `.opencode/skill/mcp-coco-index/README.md:133-139`]

3. CocoIndex is configured as a repo-local MCP server named `cocoindex_code` across runtimes, using the checked-in binary `.opencode/skill/mcp-coco-index/mcp_server/.venv/bin/ccc` with the `mcp` argument and `COCOINDEX_CODE_ROOT_PATH` set to `"."`. The config templates standardize this contract for OpenCode, Codex, Claude, Gemini, and Agents, and the helper scripts explicitly scan the expected config files (`.mcp.json`, `opencode.json`, `.agents/settings.json`, `.gemini/settings.json`, `.claude/mcp.json`, `.codex/config.toml`) to validate downstream adoption. The live repo wiring in `opencode.json`, `.codex/config.toml`, `.claude/mcp.json`, `.agents/settings.json`, and `.gemini/settings.json` matches that pattern. [SOURCE: `.opencode/skill/mcp-coco-index/assets/config_templates.md:20-33`; `.opencode/skill/mcp-coco-index/assets/config_templates.md:61-81`; `.opencode/skill/mcp-coco-index/assets/config_templates.md:150-167`; `.opencode/skill/mcp-coco-index/scripts/common.sh:21-28`; `.opencode/skill/mcp-coco-index/scripts/common.sh:150-164`; `opencode.json:37-45`; `.codex/config.toml:24-32`; `.claude/mcp.json:27-35`; `.agents/settings.json:43-52`; `.gemini/settings.json:43-52`]

4. The overlap between CocoIndex and a tree-sitter code graph is real but partial. Both are code-discovery mechanisms intended to improve "where does this live?" workflows, and CocoIndex already preserves some code structure at chunk boundaries by preferring function/class splits where possible. But the current graph machinery in Spec Kit Memory is a different thing: `query-router.ts` treats `graph` and `degree` as separate retrieval channels, and `graph-search-fn.ts` shows that today's `graph` channel is causal-memory traversal over `causal_edges` plus FTS5 lookup, not code-structure relations like imports, calls, or definitions. That means CocoIndex covers semantic similarity over code chunks, while a tree-sitter code graph would cover explicit structural relations over code entities. [SOURCE: `.opencode/skill/mcp-coco-index/references/settings_reference.md:144-160`; `.opencode/skill/system-spec-kit/mcp_server/lib/search/query-router.ts:23-25`; `.opencode/skill/system-spec-kit/mcp_server/lib/search/query-router.ts:40-72`; `.opencode/skill/system-spec-kit/mcp_server/lib/search/graph-search-fn.ts:4-6`; `.opencode/skill/system-spec-kit/mcp_server/lib/search/graph-search-fn.ts:88-126`; `.opencode/specs/02--system-spec-kit/024-compact-code-graph/research/research.md:53-59`]

5. Inference from the sources: a future code graph channel could share CocoIndex's operational infrastructure, but it should not share CocoIndex's primary index format. CocoIndex is built around chunk vectors whose dimensions are tied to the embedding model; switching models requires destroying and rebuilding the index because mixed dimensions corrupt results. By contrast, a code graph channel needs first-class node/edge storage and graph scoring. The cleanest reuse layer is therefore around project-root detection, bootstrap/readiness flows, and possibly optional embeddings for graph-node summaries, not around reusing CocoIndex's chunk-vector tables as the graph store itself. [SOURCE: `.opencode/skill/mcp-coco-index/SKILL.md:245-253`; `.opencode/skill/mcp-coco-index/SKILL.md:256-276`; `.opencode/skill/mcp-coco-index/README.md:55-59`; `.opencode/skill/mcp-coco-index/README.md:135-139`; `.opencode/skill/system-spec-kit/mcp_server/lib/search/graph-search-fn.ts:149-206`; `.opencode/specs/02--system-spec-kit/024-compact-code-graph/research/research.md:86-95`]

6. CocoIndex's main strengths are concept-first lookup, multi-language coverage, incremental daemon-backed indexing, and a deliberately safe MCP surface for agents. Its main limitations for structural code queries are also explicit in the docs: search is approximate and must be verified, exact token/symbol search still belongs to Grep, low-scoring results are often noise, repeated `refresh_index=true` calls can cause concurrency errors, and the MCP surface has no notion of structural operations such as "find all callers", "expand import graph", or "return AST parents/children". Even its chunker is structure-aware only for splitting, not for exposing structural graph primitives. [SOURCE: `.opencode/skill/mcp-coco-index/README.md:63-80`; `.opencode/skill/mcp-coco-index/README.md:137-139`; `.opencode/skill/mcp-coco-index/SKILL.md:329-347`; `.opencode/skill/mcp-coco-index/SKILL.md:372-377`; `.opencode/skill/mcp-coco-index/SKILL.md:383-401`; `.opencode/skill/mcp-coco-index/SKILL.md:538-545`]

7. The evidence in the current spec packet still supports making code graph a separate channel instead of "just extending CocoIndex". The packet's own decision record says code graph is a different problem from the current hook/compact work and defers it to its own spec folder. The synthesized research matrix already ranks "build clean-room code graph" as the recommended path and "enhance CocoIndex Code" as only an alternative. Combined with the difference in data models above, that points toward keeping CocoIndex as the semantic code-search channel and adding code graph as a sibling retrieval channel in the hybrid router rather than folding graph semantics into CocoIndex's single-tool MCP surface. [SOURCE: `.opencode/specs/02--system-spec-kit/024-compact-code-graph/decision-record.md:48-56`; `.opencode/specs/02--system-spec-kit/024-compact-code-graph/spec.md:127-130`; `.opencode/specs/02--system-spec-kit/024-compact-code-graph/research/research.md:65-71`; `.opencode/specs/02--system-spec-kit/024-compact-code-graph/research/research.md:75-82`]

8. The best integration pattern is a hybrid one: keep CocoIndex responsible for semantic candidate generation, then let a new code-graph channel add structural evidence and reranking inside the existing multi-channel search architecture. The repo already has a channel router and hybrid-search pipeline that can accommodate another channel, and the research plan for this spec already sketches the right shape: tree-sitter extraction into `code_nodes` and `code_edges`, followed by registration of a `code-graph-channel.ts` as a sixth channel. That pattern preserves CocoIndex's strengths, avoids overloading its MCP contract, and lets code graph mature independently without losing the option to share readiness/bootstrap conventions or optional embedding summaries later. [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/lib/search/query-router.ts:40-72`; `.opencode/specs/02--system-spec-kit/024-compact-code-graph/research/research.md:53-59`; `.opencode/specs/02--system-spec-kit/024-compact-code-graph/research/research.md:84-95`; `.opencode/specs/02--system-spec-kit/024-compact-code-graph/research/research.md:219-222`]

## Novelty Justification

Prior packet research already identified "enhance CocoIndex" as an alternative, but it did not pin down the repo's actual CocoIndex runtime contract, the exact MCP/CLI split, the embedding/chunk/vector mechanics, or the practical boundary between what CocoIndex can reuse versus what a tree-sitter code graph must own. This iteration adds those implementation-facing details and turns the earlier high-level option into a concrete integration decision.

## Recommendations for Implementation

1. Treat CocoIndex as the semantic code-search channel, not as the home for structural traversal logic.
2. Add code graph as a separate sixth retrieval channel inside the existing hybrid router, following the packet's planned `code_nodes`/`code_edges` plus `code-graph-channel.ts` direction.
3. Reuse CocoIndex-style operational patterns where they help: repo-local installation, explicit root-path override, readiness checks, and incremental refresh conventions.
4. Keep the storage models separate: CocoIndex chunk vectors in its own index, code graph entities/relations in dedicated graph tables. Only add embeddings to graph nodes later if reranking benefits from them.
5. For agent UX, use a two-step retrieval pattern: semantic prefilter with CocoIndex, then structural expansion/rerank with the code graph channel for caller/callee/import/definition questions.
6. Do not widen CocoIndex's MCP surface for graph operations. A separate code graph tool or internal channel keeps destructive/index-management concerns and structural-query semantics cleaner.
