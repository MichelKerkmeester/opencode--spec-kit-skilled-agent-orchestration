# Review 004: Handler Defaults & Parameter Audit
## Dimension: Correctness & Parameter Safety

## Findings (P0/P1/P2 with file:line citations)

No P0 findings.

No P1 findings.

- P2: The nested `memory_context -> memory_search` path performs two separate `checkDatabaseUpdated()` calls, and the second one can be reported as a generic strategy failure instead of a database-state failure. `memory_context` does an explicit database-state check and returns an `E_INTERNAL` database-specific envelope if that first check throws, but the routed deep/focused/resume strategies immediately call `handleMemorySearch()`, which performs a second uncaught `checkDatabaseUpdated()`. If that inner call throws, `memory_context` catches it at the strategy wrapper and downgrades it to `E_STRATEGY`, which obscures the real failure mode and recovery hint. This is a diagnostics/problem-isolation issue rather than a result-filtering regression, but it is the only material correctness concern I found in this handler audit. Citations: `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-context.ts:984-996`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-context.ts:1163-1185`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts:396-400`, `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-context.js:674-692`, `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-context.js:811-835`, `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-search.js:207-214`.

## Recommendations

- Keep the current `minState` fix as-is. The reviewed source and dist handlers no longer default `minState` inside `memory_search`, and the `memory_context` deep/focused/resume strategy builders now omit `minState` instead of hardcoding it. I did not find another hidden `minState='WARM'` or `minState='COLD'` path in these four files. Citations: `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts:420-425`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-context.ts:628-710`, `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-search.js:211-214`, `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-context.js:401-477`.

- Treat `enableDedup=true` as the only remaining default in these handlers that can actually remove otherwise valid results, but only on follow-up calls within the same trusted session. It does not remove first-time results in a fresh session, because `memory_context` mints a new ephemeral `effectiveSessionId` when the caller omits `sessionId`, and the dedup filter only drops rows that were already marked sent in that session. In other words: this is intentional session continuity behavior, not another null-state regression. Citations: `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-context.ts:1005-1008`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-context.ts:1041-1067`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-context.ts:1098-1108`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts:360-380`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts:854-927`, `.opencode/skill/system-spec-kit/mcp_server/lib/session/session-manager.ts:385-398`, `.opencode/skill/system-spec-kit/mcp_server/lib/session/session-manager.ts:898-944`.

- `includeArchived=false` is correct and does not exclude legitimate non-archived rows by itself. The archive filters preserve rows where `is_archived` is `NULL` or `0`, so the flag only removes explicitly archived records. Any remaining misses under this default would come from other gates such as scope, tier, expiry, quality threshold, or the fixed result limit, not from archival logic. Citations: `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts:427-430`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/pipeline/stage1-candidate-gen.ts:119-130`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/sqlite-fts.ts:53-66`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/vector-index-queries.ts:179-188`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/vector-index-queries.ts:221-228`.

- The other audited defaults are ranking and shaping defaults, not silent hard filters. `autoDetectIntent=true`, `useDecay=true`, `rerank=true`, `applyLengthPenalty=true`, `enableSessionBoost`, and `enableCausalBoost` can change ordering and therefore affect which items survive the configured `limit`, but they are not removing rows through a separate filter chain in the way the old `minState` defaults did. The four `memory_context` strategy builders are internally consistent: `quick` routes to trigger matching; `deep` and `focused` pass through the search defaults without reintroducing state filters; and `resume` intentionally narrows behavior with `includeConstitutional:false`, `enableDedup:false`, and `useDecay:false` for continuity-oriented recall. Citations: `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts:410-437`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts:648-685`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-context.ts:608-718`, `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-search.js:211-214`, `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-search.js:403-439`, `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-context.js:389-480`.

- Follow-up hardening: add one focused regression test that exercises `memory_context(mode='deep'|'focused'|'resume')` with a mocked inner `checkDatabaseUpdated()` failure and asserts the final envelope preserves database-state classification instead of collapsing to generic `E_STRATEGY`. That would lock down the only concrete issue surfaced by this audit.
