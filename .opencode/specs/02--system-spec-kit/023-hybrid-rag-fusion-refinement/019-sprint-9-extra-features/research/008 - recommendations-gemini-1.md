# Document 2 - Actionable Recommendations for system-speckit

This section outlines how to integrate the insights from Cognee, ArtemXTech, and QMD into the `system-speckit` ecosystem and its Memory MCP server.

## 1. Implement "Context Trees" for Spec Folders (High Impact, Low Effort)

**Concept:** 
Adapted from QMD, a Context Tree attaches high-level semantic meaning to directory paths. When an agent reads a file in `specs/003-parent/121-child/`, it currently lacks the broader context of what `003-parent` aims to achieve unless it explicitly reads the parent's `spec.md`.

**Implementation Strategy:**
Enhance the Memory MCP server's file reading tools to automatically prepend Context Tree metadata.
*   [Assumes: The MCP server currently serves raw file strings].

**Code Pattern Example:**
```javascript
// Inside the Memory MCP Server
async function readSpecFileWithContext(filePath) {
  const content = await fs.readFile(filePath, 'utf8');
  const specDir = path.dirname(filePath);
  
  // Look up the Context Tree cache (generated by generate-context.js)
  const contextData = await getContextTreeForPath(specDir);
  
  return `
--- SYSTEM CONTEXT ---
Path Context: ${contextData.description || "Unknown Phase"}
Parent Objective: ${contextData.parentGoal || "None"}
----------------------
${content}
`;
}
```

## 2. Smart Markdown Chunking for Memory Retrieval (High Impact, Medium Effort)

**Concept:**
When an agent requests `/memory:search`, returning full Level 3+ `spec.md` or `decision-record.md` files (often >500 LOC) blows out the context window. Adapted from QMD, we should implement **Smart Chunking** at the AST level.

**Implementation Strategy:**
Update `generate-context.js` to parse Markdown using `remark` or `marked`. Index each heading (`## Task 1`, `### Decisions`) as a separate node. 

**Code Pattern Example:**
Add a new MCP tool specifically for targeted reading:
```javascript
{
  name: "read_spec_section",
  description: "Read only a specific heading section from a spec file",
  parameters: { filePath: "string", heading: "string" }
}
```
*Benefits:* The agent can query `read_spec_section("specs/001/spec.md", "Testing Strategy")`, retrieving 30 lines instead of 300.

## 3. SQLite-Backed Hybrid Search (Medium Impact, High Effort)

**Concept:**
Currently, `memory_search()` likely relies on basic text matching or standard vector DBs. We can adopt QMD's approach by moving the memory index to a unified SQLite database using `FTS5` (Full Text Search) and `sqlite-vec`.

**Implementation Strategy:**
- Migrate the output of `generate-context.js` from raw JSON files to a local `memory.sqlite` file.
- Implement a two-stage search:
  1. BM25 keyword search (critical for finding exact spec IDs like `121-child`).
  2. Vector search (for conceptual queries like "Where did we discuss database migrations?").

**Risks:** 
Introducing native SQLite dependencies (`sqlite-vec`) can complicate cross-platform installations (macOS/Linux/Windows binaries).

## 4. Metadata-First Querying via Checklists (High Impact, Low Effort)

**Concept:**
Inspired by ArtemXTech's "Bases Query", we can treat the required `checklist.md` in Level 2+ specs as a structured database. Agents shouldn't need to read the whole spec to know if it's done.

**Implementation Strategy:**
Create an MCP tool that parses `checklist.md` files and returns a structured JSON summary of progress.

**Code Pattern Example:**
```javascript
// MCP Tool: speckit_get_status
function getSpecStatus(specPath) {
    const checklistStr = fs.readFileSync(path.join(specPath, 'checklist.md'), 'utf8');
    const total = (checklistStr.match(/\[.\]/g) || []).length;
    const completed = (checklistStr.match(/\[x\]/g) || []).length;
    
    return JSON.stringify({
        spec: specPath,
        status: completed === total ? "Complete" : "In Progress",
        completion_percentage: (completed / total) * 100
    });
}
```
*Benefits:* The `@orchestrate` agent can instantly poll the status of all child specs without consuming thousands of context tokens reading Markdown.

## 5. Lightweight Graph Edges for System-Spec-Kit (Medium Impact, Medium Effort)

**Concept:**
Cognee's Triple-Store is too heavy, but its Graph concept is highly relevant to `system-speckit`, which already has implicit relationships (Parents, Children, Phases).

**Implementation Strategy:**
Instead of a full Neo4j database, encode graph relationships directly into the memory JSON index or SQLite DB using explicit standard edges:
- `DEPENDS_ON`
- `IMPLEMENTS`
- `BLOCKS`

When an agent requests a spec via MCP, automatically append a "Dependency Graph" footer:
```markdown
--- DEPENDENCY GRAPH ---
[<- Blocks]: specs/004-api-auth
[-> Depends On]: specs/002-database-schema
```
*Benefits:* This satisfies the "SYSTEMS Lens" (from your GEMINI.md rules) by explicitly showing the agent the side effects and dependencies of the file they are about to modify, preventing scope-lock violations.

## Summary of Adoption Pathway
1. **Phase 1 (Immediate):** Implement **Metadata-First Querying** and **Context Trees** via string manipulation in the existing MCP read tools. This requires no new dependencies.
2. **Phase 2 (Near-term):** Upgrade `generate-context.js` to perform **Smart Markdown Chunking**, allowing the MCP server to serve specific headers rather than whole files.
3. **Phase 3 (Long-term):** Migrate the `memory/` index to an **SQLite Hybrid Search** (`FTS5` + `sqlite-vec`) for robust, localized, agent-driven retrieval without external vector cloud dependencies.
