### Score: 72

- **Command injection via folder names:** No current finding. `projectRoot` is never interpolated into the shell command; it is only supplied as the `cwd` option, so folder names containing spaces or shell metacharacters do not become part of the executed command. The dynamic pieces inside command strings are bounded integers (`HEAD~${diffWindow}`) or fixed literals.
- **Crash recovery:** The current recovery path is too coarse. `catch { return emptyResult(); }` swallows exit status, stderr, and exception type, so callers cannot distinguish “git not installed”, “not a valid repository”, “timeout”, and “empty repo” from a legitimate empty result.
- **Empty repo / no commits / unborn branch:** This is not handled explicitly. In a freshly initialized repo with no commits, `git status --porcelain` can return untracked files successfully, but `git rev-list --count HEAD` exits non-zero; the extractor then returns a fully empty payload instead of `uncommittedCount > 0` with `commitCount = 0`.
- **Recommended fix direction:** Keep the current non-fatal behavior, but special-case expected git failures. For example: treat unborn `HEAD` / exit 128 as `revCount = 0`, preserve `statusEntries`, and return a degraded-but-informative result; for `ENOENT`, timeout, or unexpected stderr failures, emit a structured warning before falling back.
