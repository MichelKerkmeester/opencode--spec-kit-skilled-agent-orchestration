## Findings
1. **High** `task_preflight` / `task_postflight` accept `NaN` scores. `validateScores()` only checks `typeof value === 'number'` plus `< 0` / `> 100`, and both comparisons are false for `NaN`, so bad inputs can be persisted and then poison `delta_*`, `learning_index`, and downstream averages. References: [session-learning.ts:189](\/Users\/michelkerkmeester\/MEGA\/Development\/Opencode%20Env\/Public\/.opencode\/skill\/system-spec-kit\/mcp_server\/handlers\/session-learning.ts#L189), [session-learning.ts:413](\/Users\/michelkerkmeester\/MEGA\/Development\/Opencode%20Env\/Public\/.opencode\/skill\/system-spec-kit\/mcp_server\/handlers\/session-learning.ts#L413)

2. **High** batch session dedup breaks for memories without an `id`. `shouldSendMemoriesBatch()` only returns decisions in a `Map<number, boolean>`, so id-less rows are never represented there, and `filterSearchResults()` only consults the map when `r.id != null`; already-sent id-less memories can therefore still be returned to the caller. References: [session-manager.ts:343](\/Users\/michelkerkmeester\/MEGA\/Development\/Opencode%20Env\/Public\/.opencode\/skill\/system-spec-kit\/mcp_server\/lib\/session\/session-manager.ts#L343), [session-manager.ts:706](\/Users\/michelkerkmeester\/MEGA\/Development\/Opencode%20Env\/Public\/.opencode\/skill\/system-spec-kit\/mcp_server\/lib\/session\/session-manager.ts#L706)

3. **Medium** generating `CONTINUE_SESSION.md` mutates session state. `writeContinueSessionMd()` calls `recoverState()`, and `recoverState()` flips `status='interrupted'` back to `active`; simply reading state to render a checkpoint file can clear the interrupted marker before any real resume happens. References: [session-manager.ts:897](\/Users\/michelkerkmeester\/MEGA\/Development\/Opencode%20Env\/Public\/.opencode\/skill\/system-spec-kit\/mcp_server\/lib\/session\/session-manager.ts#L897), [session-manager.ts:1071](\/Users\/michelkerkmeester\/MEGA\/Development\/Opencode%20Env\/Public\/.opencode\/skill\/system-spec-kit\/mcp_server\/lib\/session\/session-manager.ts#L1071)

4. **Medium** `session-learning` schema init is stale across DB swaps. `schemaInitialized` is a process-global boolean, not tied to a database handle, so once one DB has been initialized, later DBs skip schema creation entirely and can fail with `no such table: session_learning`. References: [session-learning.ts:163](\/Users\/michelkerkmeester\/MEGA\/Development\/Opencode%20Env\/Public\/.opencode\/skill\/system-spec-kit\/mcp_server\/handlers\/session-learning.ts#L163), [session-learning.ts:166](\/Users\/michelkerkmeester\/MEGA\/Development\/Opencode%20Env\/Public\/.opencode\/skill\/system-spec-kit\/mcp_server\/handlers\/session-learning.ts#L166)
