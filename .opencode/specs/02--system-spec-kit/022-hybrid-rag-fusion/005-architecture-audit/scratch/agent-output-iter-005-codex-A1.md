## Findings
- High: Provider failover is effectively disabled on the normal lazy-init path. [`shared/embeddings.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings.ts#L326) always creates the provider with `warmup: false`, but the cloud-to-`hf-local` fallback only runs in the warmup/create branches in [`factory.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/factory.ts#L191) and [`factory.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/factory.ts#L233). In practice, once OpenAI/Voyage initializes successfully, later API outages only retry the same provider and never fail over locally.

- High: The persistent embedding cache can return semantically different vectors under the same cache key. [`embedding-pipeline.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts#L111) hashes normalized raw content, but the sync save path actually generates and stores a weighted embedding from `buildWeightedDocumentText(...)` at [`embedding-pipeline.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/embedding-pipeline.ts#L150). The retry path uses the same hash but generates/stores an unweighted embedding at [`retry-manager.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/providers/retry-manager.ts#L257) and [`retry-manager.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/providers/retry-manager.ts#L276). A later cache hit can therefore silently swap weighted and unweighted vectors for unchanged content.

- Medium: Path-security is bypassed by the background retry loader. The validator exists in [`shared/utils/path-security.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/utils/path-security.ts#L17) and is re-exported from [`path-security.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/utils/path-security.ts#L8), but retry processing reads `memory.file_path` directly from the DB and passes it to `readFile()` with no validation in [`retry-manager.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/providers/retry-manager.ts#L423) and [`retry-manager.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/providers/retry-manager.ts#L546). A poisoned `memory_index.file_path` row can make the retry worker read arbitrary local files.

- Medium: The structured logger has no redaction and is not fail-safe for arbitrary metadata. [`logger.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/utils/logger.ts#L46) blindly `JSON.stringify`s whatever object it is handed and emits it verbatim to stderr. Any caller that logs `apiKey`, `Authorization`, raw content, or file paths will leak them, and a circular object will throw from `JSON.stringify`, turning logging into a crash path instead of best-effort diagnostics.

- Medium: Logging can stall hot paths during provider or retry failures. The logger writes synchronously to stderr in [`logger.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/utils/logger.ts#L54), and the retry-heavy paths emit per-attempt/per-iteration logs in [`openai.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/openai.ts#L185), [`voyage.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/voyage.ts#L203), and [`retry-manager.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/providers/retry-manager.ts#L467). Under outage conditions this can turn the failure path itself into event-loop backpressure.

- Medium: API key validation is weaker than it looks outside auto-detection. Placeholder/short-key screening exists only in `resolveProvider()` at [`factory.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/factory.ts#L61) and [`factory.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/factory.ts#L84), but explicit provider construction only checks for a non-empty string in [`openai.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/openai.ts#L95) and [`voyage.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/voyage.ts#L107). Obvious placeholders or whitespace-only keys can still get accepted until the first network call fails.
