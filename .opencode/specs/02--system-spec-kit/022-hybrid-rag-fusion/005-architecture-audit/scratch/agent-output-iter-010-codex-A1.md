## Security Findings
- **MEDIUM** Unvalidated database-stored paths are read by the background embedding retry worker. Evidence: the retry loop falls back to `loadContentFromFile(memory.file_path)` in [retry-manager.ts#L423](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/providers/retry-manager.ts#L423), `loadContentFromFile` reads that path directly with no containment check in [retry-manager.ts#L546](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/providers/retry-manager.ts#L546), and the loaded content is then sent into `generateDocumentEmbedding(...)` in [retry-manager.ts#L276](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/providers/retry-manager.ts#L276). Elsewhere, the codebase explicitly re-validates DB-stored paths before file reads, e.g. [vector-index-queries.ts#L1237](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/vector-index-queries.ts#L1237). If an attacker can poison `memory_index.file_path` via a tampered DB or restored state, the retry job can read arbitrary local files and potentially exfiltrate their contents through a remote embedding provider. Remediation: re-run `validateFilePath`/`validate_file_path_local` before every retry-path read, reject invalid rows, and avoid sending file-derived content to providers unless the path is re-authorized.

- **LOW** User-facing MCP errors leak internal filesystem and backend details. Evidence: raw database exceptions are returned to clients in [memory-crud-list.ts#L137](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-crud-list.ts#L137) and [memory-crud-stats.ts#L158](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-crud-stats.ts#L158), and `memory_save` exposes the pending recovery path plus raw rename failure text in [memory-save.ts#L1162](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts#L1162) and [memory-save.ts#L1165](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts#L1165). That can disclose absolute paths, schema/SQL details, and OS-level errors to callers. Remediation: return stable public error codes/messages, keep raw exception text and absolute paths in server logs only, and sanitize any recovery hints before surfacing them.

- **LOW** A maintenance script still builds SQL via string concatenation instead of bound parameters. Evidence: [run-chk210-quality-backfill.ts#L83](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/scripts/evals/run-chk210-quality-backfill.ts#L83), [run-chk210-quality-backfill.ts#L89](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/scripts/evals/run-chk210-quality-backfill.ts#L89), and [run-chk210-quality-backfill.ts#L118](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/scripts/evals/run-chk210-quality-backfill.ts#L118) interpolate values directly into SQL passed to `sqlite3`. The current quote-doubling reduces straightforward injection, so I do not see a clean exploit in the current code, but this is still a brittle pattern and an easy place for a future regression. Remediation: replace shell-style SQL strings with parameterized `better-sqlite3` statements or a binding-safe `sqlite3` interface.
