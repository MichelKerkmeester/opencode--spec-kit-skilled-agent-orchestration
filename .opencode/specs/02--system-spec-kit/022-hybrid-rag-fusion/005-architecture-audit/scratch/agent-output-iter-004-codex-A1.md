## Findings
- High: Governed saves validate `sessionId` but never persist it to `memory_index`, so session-scoped rows become invisible to later dedup, conflict checks, and retention sweeps. The post-insert governance field map omits `session_id` in [scope-governance.ts#L295](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/governance/scope-governance.ts#L295), the update allowlist also omits it in [db-helpers.ts#L13](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/db-helpers.ts#L13), but same-path dedup explicitly filters on `session_id` in [create-record.ts#L64](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/create-record.ts#L64) and the save path passes a normalized `sessionId` scope in [memory-save.ts#L850](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts#L850). In practice, a governed/session-scoped save can be written, then immediately fail to match its own scope on the next save or retention run.

- High: Memory-type inference gives tier mapping precedence over document/path rules, so common spec docs are systematically misclassified. `parseMemoryContent()` always supplies the extracted/default importance tier in [memory-parser.ts#L190](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts#L190), and `inferMemoryType()` checks tier first in [type-inference.ts#L228](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/config/type-inference.ts#L228) before path rules in [type-inference.ts#L246](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/config/type-inference.ts#L246). That means default tiers like `important`/`normal` collapse `plan.md`, `tasks.md`, and `checklist.md` into `declarative`, even though their canonical document mappings expect `procedural`/`prospective` in [memory-types.ts#L348](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/config/memory-types.ts#L348). This will skew decay/retention behavior for spec artifacts.

- Medium: The redaction gate has a context-free allowlist for any 40-character hex string, so hex-format secrets can bypass redaction if they happen to look like a Git SHA. The exemption is hard-coded in [redaction-gate.ts#L21](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/extraction/redaction-gate.ts#L21) and applied unconditionally in [redaction-gate.ts#L35](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/extraction/redaction-gate.ts#L35). That is safe for actual commit hashes, but unsafe for SHA1-style tokens or opaque IDs emitted by tools outside a Git context.

- Medium: Community persistence can retain stale memberships after the graph changes. `storeCommunityAssignments()` only deletes rows for memories missing from `memory_index` in [community-detection.ts#L451](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/graph/community-detection.ts#L451); it does not remove rows for memories that still exist but are absent from the new assignment set. `getCommunityMembers()` and `applyCommunityBoost()` then consume those stale rows in [community-detection.ts#L490](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/graph/community-detection.ts#L490) and [community-detection.ts#L525](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/graph/community-detection.ts#L525). If a node loses all causal edges, it can still be boosted as if it belonged to an old community.

- Medium: Signal-vocabulary boosts do not affect which trigger matches survive the limit. `matchTriggerPhrases()` sorts and slices first in [trigger-matcher.ts#L460](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/parsing/trigger-matcher.ts#L460), then `matchTriggerPhrasesWithStats()` applies boosts afterward in [trigger-matcher.ts#L484](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/parsing/trigger-matcher.ts#L484), and the handler uses that returned order directly in [memory-triggers.ts#L252](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-triggers.ts#L252). So “correction” and “preference” signals currently change reported weights, but not actual retrieval ranking when the result set is truncated.

- Low: Code-fence entity extraction can emit partial/incorrect technology names for common fence labels with non-word characters. The regex in [entity-extractor.ts#L87](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/extraction/entity-extractor.ts#L87) captures `\w+`, so ```` ```c++ ```` becomes `c`, and labels like `objective-c` or `f#` are truncated or missed. That will pollute `technology` entities with false positives and undercount real languages.
