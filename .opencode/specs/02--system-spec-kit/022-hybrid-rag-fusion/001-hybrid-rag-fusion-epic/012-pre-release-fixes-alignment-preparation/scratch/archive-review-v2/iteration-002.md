## Findings

### P2-004: T11 governance checks are minimal -- no sanitization of indexed content
- Dimension: security
- Evidence: [SOURCE: scripts/core/memory-indexer.ts:148-157]
- Impact: The T11 fix adds title/content emptiness checks but does NOT sanitize the content being indexed. Content flows directly from generate-context.js output to vectorIndex.indexMemory() without any input sanitization, HTML escaping, or injection prevention. In the MCP pipeline path, governance hooks include quality gates, sufficiency checks, and template compliance. The script path bypasses all of these. However, the input is generated by the system itself (not user-supplied external data), so the attack surface is limited.
- Skeptic: The content indexed via the script path is always generated by generate-context.js from AI-composed JSON or collected session data. It never processes arbitrary user-uploaded files or external HTTP input. The script runs locally in the user's environment. The governance bypass is a design trade-off documented in T11 checklist: "Basic governance, not full hook parity -- documented as acceptable in review."
- Referee: P2. The governance bypass is documented and the threat model is limited (local script, AI-generated content). Not a security vulnerability but an acknowledged gap.
- Final severity: P2

### P2-005: Input normalizer warns but does not reject unknown fields
- Dimension: security
- Evidence: [SOURCE: scripts/utils/input-normalizer.ts:786-789]
- Impact: Unknown fields in input data produce a console warning but are silently dropped. There is no option to reject (throw) on unknown fields. If an attacker or misconfigured caller passes unexpected fields (e.g., `__proto__`, `constructor`), they are warned but the data object is still processed. However, the code only reads explicitly named properties from the data object (never dynamically iterates unknown keys into the normalized output), so prototype pollution is not exploitable here.
- Skeptic: The input data comes from JSON.parse() of CLI arguments or file content. JSON.parse() does not create `__proto__` or `constructor` properties on the resulting object. The code accesses only known property names via explicit key reads. There is no dynamic key copying. The warn-only approach is the correct UX for typo detection per the spec (T06/T08).
- Referee: P2. The security posture is adequate for the threat model (local CLI tool, JSON.parse input). The warn-only approach is intentional per spec. No exploitation vector exists.
- Final severity: P2
