# Research: Memory Generation Quality (JSON Mode)

> 3-agent deep investigation of generate-context.js JSON mode quality issues.
> Date: 2026-03-24 | Agents: 3x GPT-5.4 (codex exec, high reasoning, workspace-write)

---

## Executive Summary

Memory files generated by `generate-context.js` JSON mode have 5 systemic quality defects caused by path fragment contamination in trigger/topic pipelines and semantic starvation in the summarizer. The active contamination path is in `workflow.ts` (not `deriveMemoryTriggerPhrases()` as initially assumed), which re-injects folder tokens AFTER filtering. JSON mode content thinness stems from the summarizer consuming only `userPrompts` (one synthetic entry from `sessionSummary`), while `exchanges`, `toolCalls`, `nextSteps`, and `technicalContext` never reach it. **Recommended fix: combined architecture** — source-stripping + centralized sanitizer + pre-write prevention + JSON enrichment. Ultra-think review (88% confidence) confirmed root causes correct, recommended simplifying from 5 steps to 3, splitting into 2 PRs, and explicitly fixing `ensureMinTriggerPhrases` in Step 1.

## 1. Root Cause Map (from Agent 1 — newInfoRatio: 0.67)

### Contamination Map

| Entry Point | Filter | Escape Path | Impact |
|---|---|---|---|
| `workflow.ts:1056-1058` adds `folderNameForTriggers` (leaf folder name, normalized) to trigger extraction source | `filterTriggerPhrases():122-158` catches `/`, `\`, leading `NNN `, short tokens, substring shingles | Normalized folder phrases like "memory generation quality" have no separators or leading numbers — pass through | Folder-name phrases in `trigger_phrases` |
| `workflow.ts:1101-1106` re-adds full folder phrase via `unshift()` AFTER filtering | None — post-filter insertion | Even if extraction/filtering removed it, it's re-added directly | Guarantees at least one folder-derived trigger |
| `workflow.ts:1107-1125` appends individual folder tokens not in `FOLDER_STOPWORDS` | Exact-match stopword list only | `generation`, `epic` not in stopwords for current naming pattern | Single-word folder residue in triggers |
| `frontmatter-editor.ts:96-115` `ensureMinTriggerPhrases()` backfills from leaf folder tokens when <2 triggers | No stopword list, no path-fragment filter | Blocked tokens like `memory`, `quality` can re-enter | Fallback reintroduces contamination |
| `topic-extractor.ts:31-36` pushes `specFolderName` into `weightedSegments` | Aggressive stopwords + bigram scoring | `generation` not in aggressive stopwords; folder bigrams score well | Folder-derived key_topics |
| `frontmatter-editor.ts:118-136` `ensureMinSemanticTopics()` falls back to first folder token | No path-aware filter | First surviving folder token becomes sole topic | Folder token as only topic in low-signal sessions |
| `memory-frontmatter.ts:50-57` `buildSpecTokens()` tokenizes FULL spec path | None in `deriveMemoryTriggerPhrases()` | Full-path tokens: system, spec, kit, hybrid, rag, fusion, epic, etc. | **LATENT** — not called by active JSON-mode workflow path |

### Key Correction

`deriveMemoryTriggerPhrases()` in `memory-frontmatter.ts` is **NOT the active JSON-mode writer path**. Only `deriveMemoryDescription()` is called from workflow. The active contamination flows through `workflow.ts` directly. The helper is a latent/alternate contamination source.

### Filter Gap Summary

| Filter | Catches | Misses |
|---|---|---|
| `filterTriggerPhrases()` | `/`, `\`, leading `NNN `, short tokens, substrings | Normalized folder phrases without separators or numbers |
| `FOLDER_STOPWORDS` | Common folder words (system, spec, kit, etc.) | `generation`, `epic`, dynamic naming patterns |
| `PATH_FRAGMENT_PATTERNS` (post-save) | Short words, directory names, leading numbers, separators | 3-word normalized phrases, 2-word fragments without separators |

## 2. JSON Mode Content Gap Analysis (from Agent 2 — newInfoRatio: 0.72)

### Root Problem

The semantic summarizer is fed almost entirely by `allMessages`, and Step 7.5 builds `allMessages` only from `collectedData.userPrompts`. In JSON mode, the slow-path normalizer collapses scalar inputs into **one synthetic `userPrompt`** from `sessionSummary`, leaving the summarizer starved.

### Field-by-Field Gap Analysis

| JSON Input Field | Normalized Form | Pipeline Consumer | Output Quality |
|---|---|---|---|
| `sessionSummary` | 1 synthetic userPrompt + 1 observation (truncated to 200 chars) | Step 7.5 → summarizer (one message) | **Main cause of thin summaries** — compressed to one message |
| `filesModified` | `FILES[]` with synthetic description | File sections only, not messages | Low for content depth |
| `keyDecisions` | Decision observations + `_manualDecisions` | Decision extractors | Medium — survives but doesn't enrich intent/plan/result |
| `nextSteps` | 1 followup observation | Status heuristics only | Medium-low — not used for outcomes/plan text |
| `technicalContext` | Implementation observation + `TECHNICAL_CONTEXT` | Template section | Medium-low — never becomes implementation messages |
| `triggerPhrases` | `_manualTriggerPhrases` side-channel | Memory indexer | Low for body content |
| `exchanges` | Retained raw, **NOT promoted to messages** | Activity counts only | **Highest missed opportunity** — contains user/assistant dialogue |
| `toolCalls` | Retained raw, **NOT promoted to messages** | Activity counts only | **High missed opportunity** — contains implementation evidence |
| `userPrompts` (if provided) | Preserved as-is on fast path | Direct source for `allMessages` | **High quality** — the existing escape hatch |
| `observations` | Preserved/synthesized, deduped | Secondary extractors | Partial — helps file/decision extraction, not semantic summary |

### Best Enrichment Targets (ordered by impact)

1. **`exchanges`** — Already contain user intent + assistant result text. Convert to multi-message semantic input.
2. **`toolCalls`** — Concrete implementation evidence. Promote to implementation/result messages.
3. **`nextSteps` + `technicalContext`** — Structured fields that should seed plan/implementation message classes.
4. **`filesModified`** — Convert to implementation/result change messages instead of file-list-only metadata.

### Title Truncation

Title generation is richer than body because it considers raw `sessionSummary`, `QUICK_SUMMARY`, `TITLE` candidates before 110-char truncation. Truncation is a secondary issue — the primary problem is thin semantic body content.

## 3. Fix Recommendations (from Agent 3 — newInfoRatio: 0.82)

### Recommended Architecture: Combined

**Decision**: Adopt combined architecture — source-stripping + centralized sanitizer + pre-write prevention + JSON enrichment.

This is the only option that covers: workflow trigger phrases, memory-frontmatter fallback triggers, `key_topics`, JSON-mode thin-content, and future regressions.

### Implementation Sequence

| Step | What | Where | Priority |
|---|---|---|---|
| **1** | Stop source pollution | Remove `folderNameForTriggers` insertion + post-filter reinsertion in workflow. Narrow `buildSpecTokens()`. Remove/gate `specFolderName` weighting in `extractKeyTopics()`. | **P0** |
| **2** | Create shared semantic sanitizer | One helper for trigger phrase + key topic sanitation. Extend to structural phrase detection beyond slash prefixes. | **P0** |
| **3** | Promote review to pre-write prevention | Reuse `PATH_FRAGMENT_PATTERNS` before file write. Keep post-save review for audit. | **P1** |
| **4** | Enrich JSON normalization | Improve slow-path synthesis: promote `exchanges` to messages, `toolCalls` to implementation evidence, `nextSteps`/`technicalContext` to plan/implementation classes. | **P1** |
| **5** | Tighten regression coverage | Extend 6 test suites (see below). | **P1** |

### Ruled-Out Approaches

| Approach | Why Rejected |
|---|---|
| Centralized filter only | Doesn't cover `key_topics` (bypasses filter entirely). Still leaks on post-filter reinsertion. |
| Distributed source-stripping only | Too brittle — no final guard if future contributors add new contamination source. |
| Post-save auto-repair as primary fix | Too reactive — hides non-deterministic generation bugs. |

## 7. Ultra-Think Review (Opus, 88% confidence)

Multi-strategy review (Analytical, Critical, Pragmatic) validated 11 of 12 code-line claims. Root cause analysis confirmed correct.

### Corrections to Research

1. **`ensureMinTriggerPhrases` omitted from fix plan Step 1** — the research identifies it as contamination source (Finding 1, row 4) but the 5-step fix plan does not explicitly patch it. Must be fixed in the same step by applying `FOLDER_STOPWORDS` inside the function. Same for `ensureMinSemanticTopics`.

2. **`_manualTriggerPhrases` interaction not analyzed** — manual triggers merged at `workflow.ts:1068-1083` BEFORE auto-extracted ones. Could mask or compound folder contamination. Minor gap — does not change core diagnosis.

### Simplification Recommendation

The 5-step combined architecture is **over-engineered**. Simplify to 3 steps, 2 PRs:

| Step | PR | What | Files | ~LOC |
|---|---|---|---|---|
| A | PR1 | Stop contamination: delete reinsertion `:1101-1106`, expand `FOLDER_STOPWORDS` (add `generation`, `epic`, `audit`, `alignment`, `enforcement`, `remediation`), apply stopwords in `ensureMinTriggerPhrases` + `ensureMinSemanticTopics`, reduce `specFolderName` weight in `extractKeyTopics` | workflow.ts, frontmatter-editor.ts, topic-extractor.ts | ~40 |
| B | PR2 | Enrich JSON slow-path: promote `exchanges` to `userPrompts` (top-10 cap, dedup vs sessionSummary), promote `toolCalls` to implementation messages (summarized, not verbatim) | input-normalizer.ts | ~25 |
| C | Both | Targeted regression tests | 4 test files | ~80 |

**Deferred** (not needed now): Shared semantic sanitizer module (premature architecture), pre-write prevention promotion (nice-to-have defense-in-depth).

### Risk Assessment

| Risk | Severity | Mitigation |
|---|---|---|
| **Retrieval regression** — removing folder reinsertion means memories not discoverable by folder-name trigger matching | Medium | Let folder phrases survive naturally through extraction. Remove only FORCED reinsertion. `spec_folder` frontmatter already enables `memory_search` by folder. |
| **Circular contamination in ensureMin** — Step 1 removes source pollution but ensureMin STILL backfills without stopwords | Medium | Apply `FOLDER_STOPWORDS` inside `ensureMinTriggerPhrases()` and `ensureMinSemanticTopics()` in same PR. |
| **Duplicate content from enrichment** — promoted exchanges overlap with sessionSummary | Medium | Cap at 10 exchanges, dedup by checking if exchange text is substring of existing sessionSummary prompt. |
| **Over-filtering legitimate terms** — "rag fusion" stripped when session IS about RAG fusion | Low-Med | Keep phrases independently sourced from session content even if they match folder name. |

### Additional Missing Tests

| Test | Why Important |
|---|---|
| `ensureMinTriggerPhrases` with FOLDER_STOPWORDS | After fix, backfill must not reintroduce stopped words |
| Manual trigger + folder contamination interaction | Verify `_manualTriggerPhrases` don't mask folder contamination |
| Exchange-to-userPrompt deduplication | Promoted exchanges must not duplicate sessionSummary content |
| Parameterized folder name patterns | Test "001-simple", "022-hybrid-rag-fusion", "013-memory-generation-quality", deeply nested paths |
| FOLDER_STOPWORDS completeness | Assert dynamic folder tokens (generation, epic, audit, alignment, enforcement, remediation) included |
| Summarizer output with enriched input | Verify 5+ promoted exchanges produce classified messages across intent/plan/implementation/result |

### Exchange Promotion Contract (define before implementing PR2)

- **Max promoted**: 10 exchanges
- **Dedup**: Check if exchange text is substring of existing sessionSummary prompt
- **toolCalls format**: Summarized text (not verbatim JSON dumps)
- **Fast-path guard**: If `userPrompts` already has 3+ entries, skip promotion (rich input already present)

### Verdict

Research is thorough enough to proceed to implementation. No critical unknowns remain. Recommend the simplified 3-step / 2-PR approach over the original 5-step plan.
