### Cross-Cutting Observations

**State management:** All 7 stateful modules use module-level `let db: Database.Database | null` singletons. There's no shared lifecycle manager—each module's `init()` must be called in the correct order (e.g., `archival-manager` depends on `tier-classifier` at runtime). A missed `init()` silently returns empty results rather than throwing.

**Concurrency:** `better-sqlite3` is synchronous, so single-tick operations are safe. However, `co-activation.ts:populateRelatedMemories` is `async` and yields between a DB read and a DB write—another MCP call could interleave and create a last-write-wins race on `related_memories`.

**Session lifecycle:** `working-memory.ts` has `cleanupOldSessions()` but no automatic invocation—it must be called by the server. `archival-manager.ts` has an `unref()`'d background interval but no guarantee `cleanup()` is called on server shutdown. Both are potential resource leaks in long-lived processes.
