### Task B — Persistent Embedding Cache Evaluation (`mcp_server/lib/providers/retry-manager.ts`)

1. **Current behavior under rate-limit starvation**
- In `retryEmbedding()` (`retry-manager.ts:219-227`), it always calls `generateDocumentEmbedding(normalizedContent)` without persistent cache lookup.
- The only cache hit path is the process-local in-memory map in `shared/embeddings.ts` (`embeddingCache`, size 1000). That cache is lost on restart and does not help if calls keep failing (no successful value to cache).
- Result: repeated retries re-hit provider for same content (`MAX_RETRIES=3`, backoff), then mark failed.

2. **Impact severity**
- Moderate-to-high under sustained 429/5xx conditions.
- Repeated provider calls for identical content amplify throttling pressure and burn retry budget quickly.
- After restarts, retries effectively begin from scratch cache-wise.
- Backlogs can convert to permanent `failed` states even for content already embedded elsewhere.

3. **SQLite-backed cache design sketch**
- Reuse existing cache primitives already used in `memory-save`:
  - `lookupEmbedding`, `storeEmbedding` from `mcp_server/lib/cache/embedding-cache.ts`
  - table already created via schema (`vector-index-schema.ts` REQ-S2-001).
- In `retryEmbedding()` flow:
  - Resolve `modelId` via embedding provider helper.
  - Prefer `memory.content_hash` from `memory_index` as key (fallback to computed hash if missing).
  - `lookupEmbedding(db, contentHash, modelId)` before provider call.
  - On hit: decode buffer -> write to `vec_memories` and set status `success`.
  - On miss: call provider, then `storeEmbedding(...)` on success.
- Add counters/logging for cache hit/miss in retry job output.
- Add tests: hit path skips provider call; miss path stores; restart persistence behavior.

4. **Recommendation**
- **Go (implement now).**
- Reason: low implementation risk (infrastructure already present), clear operational upside during provider starvation, and direct alignment with existing REQ-S2-001 pattern in `memory-save`.
