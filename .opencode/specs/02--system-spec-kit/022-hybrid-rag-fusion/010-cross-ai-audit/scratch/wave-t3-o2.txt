### 2. DATA INTEGRITY — CHECK-THEN-ACT (TOCTOU)

| Sev | File:Line | Description | Suggested Fix |
|-----|-----------|-------------|---------------|
| **P1** | `causal-edges.ts:156-162` | `insertEdge()` — `countEdgesForNode()` check → then INSERT. Classic TOCTOU. Two concurrent auto-edge insertions could both pass the MAX_EDGES_PER_NODE guard and both insert, exceeding the limit. With better-sqlite3 sync API this requires two MCP tool calls with an async gap between count and insert in the same event-loop tick — low probability but possible. | Move the count check inside a transaction with the INSERT, or use a CHECK constraint/trigger at the DB level. |
| **P1** | `index-refresh.ts:152-175` | `markFailed()` — SELECT retry_count → compute new status → UPDATE. If two concurrent calls both read retry_count=2, both compute "retry", both update to retry_count=3 with status "retry" instead of one getting "failed". | Wrap in `db.transaction()`, or use a single UPDATE with `CASE WHEN retry_count + 1 >= 3 THEN 'failed' ELSE 'retry' END`. |
| **P2** | `checkpoints.ts:375-384` | `createCheckpoint()` — COUNT → compare → DELETE for max checkpoint enforcement. INSERT and overflow cleanup are separate. A concurrent create could exceed MAX_CHECKPOINTS temporarily. | Move the count + delete inside a transaction with the INSERT. |
