### FINDING-01: Normalized decision confidence is rendered as a percent without conversion
- **File:** `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:40-42,329-335; .opencode/skill/system-spec-kit/templates/context_template.md:533; .opencode/skill/system-spec-kit/scripts/lib/ascii-boxes.ts:92-105`
- **Severity:** HIGH
- **Category:** BUG
- **Current Behavior:** `normalizeConfidence()` converts decision values like `70` to `0.7`, but both the Markdown template and decision-tree header render `{{CONFIDENCE}}%` / `${confidence}%` directly. A normalized decision therefore displays as `0.7%` instead of `70%`.
- **Expected Behavior:** If decision confidence is normalized to `[0,1]`, renderers must convert it back to percentage form at display time; otherwise the stored value should remain in `0-100` and thresholds should be adjusted consistently.
- **Root Cause:** The P1 fix normalized the producer side in `decision-extractor.ts`, but downstream consumers still assume `CONFIDENCE` is a percent value.
- **Suggested Fix:** Define one canonical unit for decision confidence. If the canonical unit is `[0,1]`, add a shared render helper (for template/context rendering and ASCII tree rendering) that outputs `Math.round(CONFIDENCE * 100)` or equivalent.

### FINDING-02: Confidence is not system-wide guaranteed to be in `[0,1]`
- **File:** `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:123-125; .opencode/skill/system-spec-kit/scripts/lib/simulation-factory.ts:262-303; .opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:208-245; .opencode/skill/system-spec-kit/templates/context_template.md:162`
- **Severity:** MEDIUM
- **Category:** DESIGN
- **Current Behavior:** The null-data fallback in `decision-extractor.ts` returns simulated decisions with `CONFIDENCE: 85`, and `collect-session-data.ts` passes preflight confidence through unchanged (`55`, `81`, etc.) for percent-style rendering. So the codebase currently mixes normalized ratios and raw percentages under the same generic “confidence” concept.
- **Expected Behavior:** Either all confidence fields should share one documented unit, or percent-based and ratio-based confidences should be explicitly separated by type/field name.
- **Root Cause:** There is no single cross-system contract for confidence units, and no validation layer enforcing one. Different modules use the same label for different scales.
- **Suggested Fix:** Introduce explicit unit-specific fields/types such as `confidenceRatio` vs `confidencePercent` (or a shared normalization/render contract), then update simulation data, templates, and extractors to honor that contract uniformly.

### FINDING-03: Decimal normalized confidences in text are parsed incorrectly, and NaN safety is incomplete
- **File:** `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:40-42,329-335`
- **Severity:** MEDIUM
- **Category:** BUG
- **Current Behavior:** The extractor only matches integer confidence text via `/confidence:?\s*(\d+)(?:%|\b)/i` and uses `parseInt()`. Text like `confidence: 0.8` is parsed as `0`, then normalized to `0`; meanwhile negative numeric input would clamp to `0`, but `normalizeConfidence(NaN)` itself would return `NaN` because the helper has no internal `Number.isFinite()` guard.
- **Expected Behavior:** Decimal ratios like `0.8` should round-trip to `0.8`, percentage strings like `80%` should become `0.8`, negatives should clamp to `0`, and invalid/NaN inputs should fall back safely instead of producing `NaN`.
- **Root Cause:** Integer-only regex + `parseInt()` loses decimal information, and the helper assumes a valid numeric caller.
- **Suggested Fix:** Parse with a decimal-capable pattern and `Number()`/`parseFloat()`, then harden `normalizeConfidence()` to reject non-finite inputs before clamping.
