### FINDING-01: `filesModified` legacy objects are normalized into invalid `FILE_PATH` objects
- **File:** `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:49,253-258; .opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:148,692`
- **Severity:** HIGH
- **Category:** BUG
- **Current Behavior:** `normalizeInputData()` treats every `filesModified` item as a string. If the caller provides the legacy shape that `collectSessionData` explicitly accepts (`{ path, changes_summary }`), the normalizer emits `FILES: [{ FILE_PATH: <object> }]`. When `collectSessionData()` processes that payload, the malformed file path propagates downstream and blows up (`filePath.replace is not a function`).
- **Expected Behavior:** Both `string[]` and `{ path, changes_summary }[]` forms should normalize into `FILES` entries with a string `FILE_PATH` and a meaningful description.
- **Root Cause:** Producer and consumer disagree on the `filesModified` item type: the normalizer hard-codes `string[]`, while the collector accepts object entries.
- **Suggested Fix:** Change `RawInputData.filesModified` to `Array<string | { path: string; changes_summary?: string }>` and normalize each branch explicitly:
  - string -> `{ FILE_PATH: item, DESCRIPTION: 'File modified...' }`
  - object -> `{ FILE_PATH: item.path, DESCRIPTION: item.changes_summary ?? 'File modified...' }`

### FINDING-02: Manual normalization silently drops collector-supported optional fields
- **File:** `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:45-60,243-299; .opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:135-154,777-835`
- **Severity:** HIGH
- **Category:** BUG
- **Current Behavior:** In manual mode, `normalizeInputData()` rebuilds a narrow object containing only `SPEC_FOLDER`, `FILES`, `observations`, `userPrompts`, `recentContext`, and `_manual*` fields. Optional collector inputs like `preflight` and `postflight` are discarded before `collectSessionData()` can read them, so learning-delta fields are always empty/null.
- **Expected Behavior:** Optional fields that `collectSessionData()` understands should survive normalization unchanged unless intentionally transformed.
- **Root Cause:** The producer constructs a fresh `NormalizedData` object instead of cloning and augmenting the original payload.
- **Suggested Fix:** Start from a cloned copy of the raw input and then overlay canonical fields, or explicitly preserve `preflight`, `postflight`, and other collector-recognized keys during normalization.

### FINDING-03: Hybrid payloads short-circuit before aliases and manual fields are canonicalized
- **File:** `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:238-241,249-297; .opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:139,681-706,727-748`
- **Severity:** HIGH
- **Category:** BUG
- **Current Behavior:** If the input contains any one of `userPrompts`, `observations`, or `recentContext`, `normalizeInputData()` returns the raw object unchanged. That means mixed-shape payloads keep `specFolder` as camelCase, and leave `sessionSummary`, `keyDecisions`, `technicalContext`, and `triggerPhrases` unconverted, so `collectSessionData()` never sees `SPEC_FOLDER` or synthesized observations for those manual fields.
- **Expected Behavior:** Partially normalized payloads should still be canonicalized field-by-field so the collector gets one consistent input shape.
- **Root Cause:** The early-return check assumes “has one canonical array” means “fully normalized payload”.
- **Suggested Fix:** Replace the blanket passthrough with incremental normalization:
  - always canonicalize `specFolder -> SPEC_FOLDER`
  - always merge manual `sessionSummary` / `keyDecisions` / `technicalContext` into `observations`
  - preserve existing `userPrompts` / `observations` / `recentContext` while filling missing canonical fields

### FINDING-04: `importanceTier` is validated by the producer but ignored by the consumer
- **File:** `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:54,332-335; .opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:705-706,775,817`

- **Category:** DESIGN
- **Current Behavior:** `normalizeInputData()`/`validateInputData()` accepts `importanceTier` as a valid input field, but there is no end-to-end path that preserves or consumes it. `collectSessionData()` always recomputes `IMPORTANCE_TIER` from detected session characteristics instead.
- **Expected Behavior:** Either an explicit input `importanceTier` should be propagated and honored, or the producer should reject/remove that field from the accepted contract.
