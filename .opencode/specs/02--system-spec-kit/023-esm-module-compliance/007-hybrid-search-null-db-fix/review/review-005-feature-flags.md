### P1

- `isFeatureEnabled()` silently disables documented default-ON search flags whenever `SPECKIT_ROLLOUT_PERCENT` is set below 100 and the caller does not supply identity. The implementation explicitly returns `false` for blank identity after it has already accepted the flag as enabled by env value, so "default ON" stops being true in every no-identity call path. That affects most of `search-flags.ts`, because only `SPECKIT_PRESSURE_POLICY` and `SPECKIT_AUTO_RESUME` even expose an identity parameter; the rest are global wrappers that call `isFeatureEnabled()` with just the flag name. This is a configuration-safety bug, not just a policy choice, because the source comments advertise these flags as ON by default unless explicitly disabled. `.opencode/skill/system-spec-kit/mcp_server/lib/cognitive/rollout-policy.ts:53-74`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:23-29`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:31-45`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:47-181`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:210-257`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:304-553`
- `SPECKIT_SEARCH_FALLBACK` sits inside that blast radius, so a partial rollout can silently downgrade hybrid retrieval from the 3-tier quality-aware fallback to the older adaptive two-pass behavior. In `collectRawCandidates()`, the tiered path merges primary and retry candidate sets, while the adaptive path uses only retry-or-primary. In `searchWithFallback()`, disabling the flag stops delegation to `searchWithFallbackTiered()` and falls back to the older adaptive pipeline. If `SPECKIT_ROLLOUT_PERCENT` is accidentally set to 1-99 in production, search still returns results, but recall and rescue behavior can regress without any explicit `SPECKIT_SEARCH_FALLBACK=false` setting. `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:79-85`, `.opencode/skill/system-spec-kit/mcp_server/lib/cognitive/rollout-policy.ts:64-71`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/hybrid-search.ts:1679-1698`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/hybrid-search.ts:1731-1758`

### P2

- `SPECKIT_FILE_WATCHER=true` and `RERANKER_LOCAL=true` are not reliable operator overrides under partial rollout. Both helpers first require explicit opt-in, but then immediately call `isFeatureEnabled()` without identity. That means an operator can set the env to `true` and still get `false` back whenever `SPECKIT_ROLLOUT_PERCENT` is below 100. This mixed opt-in plus rollout semantics is easy to misread and should be documented or normalized. `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:272-289`, `.opencode/skill/system-spec-kit/mcp_server/lib/cognitive/rollout-policy.ts:64-71`
- `SPECKIT_DEGREE_BOOST` defaults ON through `isFeatureEnabled()`, but unlike the surrounding flags it has no adjacent doc comment declaring that default or how to disable it. That is an auditability gap rather than a runtime failure, but it makes this flag the easiest one to misclassify during future reviews. `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:255-257`, `.opencode/skill/system-spec-kit/mcp_server/lib/cognitive/rollout-policy.ts:53-66`

## Recommendations

1. Split rollout helpers into two explicit APIs: one for global/server-wide flags that should stay default-ON unless the env disables them, and one for identity-scoped gradual rollout. Right now `isFeatureEnabled()` is trying to serve both cases and fails closed for the global case. `.opencode/skill/system-spec-kit/mcp_server/lib/cognitive/rollout-policy.ts:59-74`
2. If the existing API must stay in place, at minimum pass a deterministic fallback identity for global flags, for example `flagName`, so `SPECKIT_ROLLOUT_PERCENT=50` yields stable partial rollout instead of blanket disablement. That should be applied to `SPECKIT_SEARCH_FALLBACK`, `SPECKIT_GRAPH_SIGNALS`, `SPECKIT_CONTEXT_HEADERS`, and the other no-identity wrappers in `search-flags.ts`. `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:27-28`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:83-85`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:210-257`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:268-553`
3. Add targeted tests for `SPECKIT_ROLLOUT_PERCENT=0`, `50`, and `100` with `undefined`, blank, and concrete identity values. The minimum regression set should cover `SPECKIT_SEARCH_FALLBACK`, `SPECKIT_FILE_WATCHER`, `RERANKER_LOCAL`, and `SPECKIT_GRAPH_SIGNALS`. `.opencode/skill/system-spec-kit/mcp_server/lib/cognitive/rollout-policy.ts:10-27`, `.opencode/skill/system-spec-kit/mcp_server/lib/cognitive/rollout-policy.ts:59-74`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:272-289`
4. Decide whether `SPECKIT_SEARCH_FALLBACK` is release-critical. If it is, either exempt it from global rollout percent entirely or emit a conspicuous warning when it resolves `false` because identity is missing. Right now the downgrade to adaptive fallback is silent. `.opencode/skill/system-spec-kit/mcp_server/lib/search/hybrid-search.ts:1679-1698`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/hybrid-search.ts:1731-1758`
5. Fill in the missing default-state documentation for `SPECKIT_DEGREE_BOOST` and standardize which flags are rollout-gated, which are strict opt-ins, and which are pure config knobs. The file is mostly consistent today; this one undocumented exception makes audits slower and riskier than they need to be. `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:255-257`
