# Iteration 029 — Generate Operator Guidance From Runtime Policy

Date: 2026-04-10

## Research question
Is `system-spec-kit`'s operator guidance surface too hand-maintained and too large, especially across `AGENTS.md`, constitutional notes, commands, and YAML assets?

## Hypothesis
Babysitter will show that a thinner human-facing instruction surface is possible when runtime policy is machine-readable and the system can generate harness-specific guidance on demand.

## Method
I compared Spec Kit's current rule surface with Babysitter's generated instruction model, repository guidance, and harness-specific playbooks.

## Evidence
- Spec Kit's top-level AGENTS file carries mandatory gates, workflow tables, documentation policy, confidence rules, agent routing, MCP rules, and skill protocol in one very large operator contract. [SOURCE: AGENTS.md:159-252]
- `system-spec-kit` reinforces that model by making spec-folder creation and documentation workflow a central, mandatory skill behavior. [SOURCE: .opencode/skill/system-spec-kit/SKILL.md:10-13] [SOURCE: .opencode/skill/system-spec-kit/SKILL.md:32-59]
- Babysitter's top-level repo guidance is comparatively compact at the front: one main rule (`always use the babysitter skill`), then an autogenerated block describing deterministic runtime behavior and key commands. [SOURCE: .opencode/specs/system-spec-kit/999-agentic-system-upgrade/001-research-agentic-systems/002-babysitter-main/external/CLAUDE.md:3-10] [SOURCE: .opencode/specs/system-spec-kit/999-agentic-system-upgrade/001-research-agentic-systems/002-babysitter-main/external/CLAUDE.md:23-68]
- Babysitter can generate harness-specific instructions on demand through explicit CLI surfaces such as `instructions:babysit-skill`, `instructions:process-create`, `instructions:orchestrate`, and `instructions:breakpoint-handling`. [SOURCE: .opencode/specs/system-spec-kit/999-agentic-system-upgrade/001-research-agentic-systems/002-babysitter-main/external/CLAUDE.md:237-242]
- The plugin skills themselves stay thin because they delegate the detailed playbook generation to the runtime instruction generator. [SOURCE: .opencode/specs/system-spec-kit/999-agentic-system-upgrade/001-research-agentic-systems/002-babysitter-main/external/plugins/babysitter-codex/skills/babysit/SKILL.md:33-47] [SOURCE: .opencode/specs/system-spec-kit/999-agentic-system-upgrade/001-research-agentic-systems/002-babysitter-main/external/plugins/babysitter-opencode/skills/babysit/SKILL.md:21-35]

## Analysis
Spec Kit's current rule surface does not just express policy. It manually duplicates it across AGENTS, constitutional notes, commands, YAML assets, and skill docs. That is powerful, but it is also brittle because every UX simplification now requires synchronized edits in multiple explanatory layers. [SOURCE: AGENTS.md:159-252] [SOURCE: .opencode/skill/system-spec-kit/SKILL.md:102-119]

Babysitter points to a more sustainable direction: keep the runtime contract machine-readable, then generate the harness-facing instructions from that contract. The human-facing top layer becomes shorter because it is mostly pointing at one authoritative system instead of repeating it. [SOURCE: .opencode/specs/system-spec-kit/999-agentic-system-upgrade/001-research-agentic-systems/002-babysitter-main/external/CLAUDE.md:3-10] [SOURCE: .opencode/specs/system-spec-kit/999-agentic-system-upgrade/001-research-agentic-systems/002-babysitter-main/external/CLAUDE.md:237-242]

That is the clearest Phase 3 automation lesson: system-spec-kit needs fewer hand-written behavioral surfaces and more generated ones.

## UX / System Design Analysis

- **Current system-spec-kit surface:** Operators learn the system through a large hand-written AGENTS contract plus more policy spread across skills, commands, constitutional docs, and YAML assets.
- **External repo's equivalent surface:** Babysitter keeps the front door smaller and leans on generated, harness-specific orchestration instructions from runtime-owned commands.
- **Friction comparison:** Spec Kit has higher maintenance burden and higher cognitive load because the operator surface is manually layered. Babysitter reduces drift risk by generating detailed playbooks from runtime commands.
- **What system-spec-kit could DELETE to improve UX:** Delete large amounts of duplicated behavioral prose across AGENTS, command setup text, and constitutional mirror docs once a generated guidance path exists.
- **What system-spec-kit should ADD for better UX:** Add a machine-readable policy/runtime manifest plus `instructions:*` style generators for harness-specific operator playbooks.
- **Net recommendation:** REDESIGN

## Conclusion
confidence: high

finding: `system-spec-kit` should redesign its operator guidance around a machine-readable policy source and generated instruction surfaces. The current hand-maintained rule stack is too heavy and too drift-prone.
