# Iteration 001 — ResearcherSkill Architecture & Innovation

## Focus
Q1: Architectural innovations from ResearcherSkill that could improve sk-deep-research

## Findings

### 1. ResearcherSkill treats exploration as a branchable DAG, while sk-deep-research is still effectively linear
ResearcherSkill makes non-linearity a first-class part of the loop. It says the experiment history is non-linear, allows forks from any prior `keep`, records them in `.lab/branches.md`, and requires the agent to consider results from all branches when thinking. The guide reinforces that the agent can split one promising point into multiple divergent paths and later combine wins across branches. By contrast, our live deep-research contract enforces "one focus per iteration" and keeps wave orchestration and segment transitions as reference-only, so the current executable model is still a single active path with occasional recovery pivots rather than a real exploration graph. The practical upgrade is to add explicit `researchTracks` or branch records to state so divergent lines of inquiry can coexist, be compared, and later merged without overloading `Next Focus` into doing all the coordination work. [SOURCE: /tmp/deep-research-029/ResearcherSkill/researcher.md:130-146; /tmp/deep-research-029/ResearcherSkill/GUIDE.md:214-226; .opencode/skill/sk-deep-research/SKILL.md:220-246]

### 2. The `.lab/` model is a stronger state architecture than our current generic `scratch/` bundle
ResearcherSkill does not just persist state; it separates experiment knowledge from code state and makes that separation explicit. `.lab/` is described as the "single source of truth" for experiment history, is protected from git cleanup/reset behavior, and has role-specific artifacts: `config.md`, `results.tsv`, `log.md`, `branches.md`, `parking-lot.md`, and later `summary.md`. That gives both machine-readable and human-readable views of the search. Our current skill externalizes state well, but the canonical set is still a generic five-file bundle: config JSON, JSONL log, strategy markdown, per-iteration markdown, and `research/research.md`. We do have an optional `research-ideas.md`, but we do not yet have a durable branch registry, experiment journal, or concise results table. A high-value adaptation would be to keep `scratch/` but introduce a `.lab`-like sub-structure inside it, especially `branches.md`, a compact tabular `results.tsv`, and a narrative `log.md`, so the loop is easier to audit, resume, and steer. [SOURCE: /tmp/deep-research-029/ResearcherSkill/researcher.md:18-22; /tmp/deep-research-029/ResearcherSkill/researcher.md:68-77; /tmp/deep-research-029/ResearcherSkill/GUIDE.md:43-65; .opencode/skill/sk-deep-research/references/state_format.md:15-25; .opencode/skill/sk-deep-research/references/loop_protocol.md:175-200]

### 3. Its Phase 0 resume check is safer and more user-aligned than our current auto-resume
ResearcherSkill starts with a dedicated Phase 0 that checks whether `.lab/` already exists, reads the key files, summarizes objective/metrics/active branches/counts/current best/last status, and then asks the user whether to resume or start fresh. A fresh start archives the old lab before proceeding. Our loop protocol does support resume, but it is orchestrator-centric: if config, JSONL, and strategy agree, it increments the iteration counter and continues; if they do not, it halts for repair. The missing piece is the human checkpoint between "resumable" and "should resume." For sk-deep-research, adopting a pre-resume summary plus an explicit "resume or archive-and-restart" decision would make long-running research less brittle, especially when prior iterations converged on the wrong framing or when a user wants to preserve history but intentionally restart the inquiry. [SOURCE: /tmp/deep-research-029/ResearcherSkill/researcher.md:26-37; .opencode/skill/sk-deep-research/references/loop_protocol.md:45-69]

### 4. ResearcherSkill's convergence model is heuristic and pattern-aware in ways our weighted stop logic is not
Our current convergence system is rigorous but mostly numerical. It evaluates max-iteration and answered-question hard stops, then a weighted vote across rolling average, MAD noise floor, and question entropy. ResearcherSkill instead keeps a table of qualitative signals such as 5+ discards in a row, repeated thought experiments, same code area touched 3+ times, branch stagnation, and alternating keep/discard behavior. The guide explicitly says these are not a hard-coded state machine but signals the agent interprets. That is architecturally important: ResearcherSkill uses convergence signals not only to stop but to redirect strategy. The strongest upgrade is not replacing our composite algorithm, but layering heuristic "steering signals" on top of it. We should keep the quantitative stop vote, but add pattern detectors that can trigger branch creation, focus diversification, or variable isolation before the system reaches formal convergence. [SOURCE: /tmp/deep-research-029/ResearcherSkill/researcher.md:190-202; /tmp/deep-research-029/ResearcherSkill/GUIDE.md:192-210; .opencode/skill/sk-deep-research/references/convergence.md:25-120]

### 5. The explicit hypothesis strategy library is richer than our current recovery taxonomy
ResearcherSkill gives the agent a named library of ten move types: ablation, amplification, combination, inversion, isolation, analogy, simplification, scaling, decomposition, and sweep. The text frames them as tools rather than rigid rails, which keeps autonomy while still making search behavior legible. Our current stuck recovery is much narrower: try opposites, combine prior findings, or audit low-value iterations, selected from a small routing function. That is useful for recovery, but it is not a full strategy language for the whole loop. A practical improvement would be to promote strategy selection into a first-class field in state, so each iteration can declare a strategy such as `isolation` or `decomposition`, and the orchestrator can later analyze which moves are productive for which research topics. [SOURCE: /tmp/deep-research-029/ResearcherSkill/researcher.md:173-188; .opencode/skill/sk-deep-research/references/convergence.md:278-321]

### 6. Thought experiments are first-class citizens in ResearcherSkill; sk-deep-research lacks an equivalent iteration mode
ResearcherSkill explicitly distinguishes real experiments from thought experiments. Thought experiments make no code changes, produce no benchmark metric, and are logged with status `thought`, but they are treated as legitimate knowledge-producing work rather than filler. The guide argues this prevents wasting runs when the agent can invalidate a path analytically first. Our state schema does not currently have a comparable first-class status; iteration records are modeled around `complete`, `timeout`, `error`, and `stuck`, with `newInfoRatio` as the main progress measure. For sk-deep-research, this suggests adding an explicit `mode` or `status` for `thought`, `survey`, or `synthesis-only` iterations. That would let the loop preserve reasoning-heavy passes without forcing them into the same semantics as evidence-gathering passes, and it would make convergence smarter by detecting repeated analytical looping separately from genuinely new evidence collection. [SOURCE: /tmp/deep-research-029/ResearcherSkill/researcher.md:112-118; /tmp/deep-research-029/ResearcherSkill/GUIDE.md:105-107; /tmp/deep-research-029/ResearcherSkill/GUIDE.md:230-241; .opencode/skill/sk-deep-research/references/state_format.md:111-125]

### 7. The qualitative rubric system solves a problem our research loop currently leaves underspecified
ResearcherSkill supports a qualitative primary metric by defining 3-5 criteria, assigning weights that sum to 1.0, using a fixed scale, and converting the result into a composite score. The newer 1.1.0 change adds an especially useful refinement: before building the rubric, the agent asks whether the user already knows the criteria or needs help discovering them, in which case the skill generates an external research prompt and derives the rubric from that output rather than guessing. sk-deep-research currently has rich convergence math, but no formal mechanism for judging the quality of a research direction, recommendation set, or synthesis when the problem is not reducible to source novelty alone. An optional rubric layer for "recommendation quality" or "evidence usefulness" would be especially valuable for architecture and design research, where `newInfoRatio` can stay high even if the findings are not decision-useful. [SOURCE: /tmp/deep-research-029/ResearcherSkill/researcher.md:45-60; /tmp/deep-research-029/ResearcherSkill/researcher.md:163-171; /tmp/deep-research-029/ResearcherSkill/CHANGELOG.md:10-15; .opencode/skill/sk-deep-research/references/convergence.md:128-147]

### 8. Periodic re-validation is an elegant guard against drift and false local optima
ResearcherSkill requires re-validation every 10 real experiments and again at wrap-up: re-run the current head, compare against the recorded best, log drift, and potentially fork back from the best experiment. That creates a recurring check against hidden regressions and stale assumptions. Our loop has progressive synthesis and structured convergence records, but no scheduled "re-score the current best understanding against prior best" step. For sk-deep-research, the equivalent would be a periodic synthesis audit every N iterations: re-read the highest-value prior iterations, compare them to the current `Next Focus` and current answer set, and log whether the research is truly improving or just moving. This would make progressive synthesis less accumulative and more self-correcting. [SOURCE: /tmp/deep-research-029/ResearcherSkill/researcher.md:144-156; /tmp/deep-research-029/ResearcherSkill/GUIDE.md:245-260; .opencode/skill/sk-deep-research/SKILL.md:203-210; .opencode/skill/sk-deep-research/references/state_format.md:23-25]

### 9. ResearcherSkill uses bounded consultation rather than absolute non-interaction
ResearcherSkill is strongly autonomous, but not absolutist. It allows user consultation in two narrow cases: when the only viable path crosses scope boundaries, and when all strategies/branches/parking-lot ideas are exhausted. It also inserts user involvement up front for qualitative criteria selection and during resume-or-fresh decisions. Our skill is stricter: the live contract says the agent should not ask the user and should make best-judgment decisions, only escalating after recovery tiers fail. The ResearcherSkill model is better calibrated for ambiguous research because it keeps most of the autonomy benefit while preserving user control at decision boundaries with non-obvious consequences. A good upgrade would be to keep default autonomy but add explicit consultation checkpoints for resume choice, rubric-definition ambiguity, and exhaustion after branch-level recovery fails.
