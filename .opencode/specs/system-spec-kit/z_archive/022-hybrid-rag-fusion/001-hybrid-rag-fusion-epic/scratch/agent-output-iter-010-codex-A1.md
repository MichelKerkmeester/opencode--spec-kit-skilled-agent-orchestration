**4. Over-Engineering Assessment**
- Yes, for the current use case this subsystem is over-engineered.
- The strongest evidence is architectural mismatch: `memory_search` explicitly notes that tier-classifier is legacy V1 material `[memory-search.ts:11](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts#L11)`, while the live 4-stage pipeline does not import the audited modules directly.
- Large portions of the audited code are either offline-only (`archival-manager`), trigger-only (`attention-decay`, `tier-classifier`), orchestration-only (`pressure-monitor`), or fully unused (`temporal-contiguity`, `cognitive.ts`).
- What looks justified today is the session-attention path around `working_memory`, because the system still uses that data for session boost and resume context.

**5. Simpler Alternatives**
- Keep `working_memory` plus `session-boost`, and treat that as the only online cognitive signal.
- Inline the trigger-path scoring into `memory-triggers.ts`: one session-decay score plus optional simple state buckets, instead of separate `attention-decay.ts` and most of `tier-classifier.ts`.
- Move archival behind an explicit maintenance task or admin command instead of an always-on background subsystem.
- Delete or park `temporal-contiguity` until a caller exists.
- Remove `configs/cognitive.ts` until the co-activation pattern is actually consumed by runtime code.
