# Hybrid RAG Fusion Audit

## Executive Summary
The subsystem is mostly hand-tuned, not data-driven. I found many explicit constants, profile tables, clamps, and bonuses, but no learned calibration layer or parameter-fitting pipeline in the reviewed code. RSF is dormant/shadow-only, while live fusion relies primarily on RRF plus downstream heuristic boosts/penalties. I also found that the three score-resolution paths do **not** agree on precedence or units, which is a real consistency risk ([rrf-fusion.ts:22](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L22), [rsf-fusion.ts:7](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L7), [types.ts:49](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/pipeline/types.ts#L49), [ranking-contract.ts:32](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/pipeline/ranking-contract.ts#L32), [stage4-filter.ts:203](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/pipeline/stage4-filter.ts#L203)).

## 1. RRF `k` and RSF weights: hardcoded, configurable, or data-driven?
- RRF `k` defaults to `60`, citing the original SIGIR 2009 paper as a “robust default”; it is configurable by caller option and by env var `SPECKIT_RRF_K`, but I found no code that learns or tunes it from retrieval data ([rrf-fusion.ts:22](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L22), [rrf-fusion.ts:36](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L36), [rrf-fusion.ts:113](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L113)).
- RRF per-list weights are configurable at call time via `RankedList.weight`; if omitted, most channels default to `1.0` and `graph` gets a hardcoded `1.5` boost ([rrf-fusion.ts:67](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L67), [rrf-fusion.ts:38](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L38), [rrf-fusion.ts:243](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L243)).
- Adaptive fusion profiles are also hardcoded: intent profiles specify semantic/keyword/recency/graph weights, and doc type shifts them by a fixed `0.1`; again, no evidence of offline fit or online learning in code ([adaptive-fusion.ts:60](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/adaptive-fusion.ts#L60), [adaptive-fusion.ts:83](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/adaptive-fusion.ts#L83), [adaptive-fusion.ts:135](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/adaptive-fusion.ts#L135)).
- RSF has no tunable source-weight map at all. It does equal averaging after per-source min-max normalization, with hardcoded penalties for missing channels and a hardcoded `+0.10` cross-variant bonus ([rsf-fusion.ts:100](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L100), [rsf-fusion.ts:210](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L210), [rsf-fusion.ts:323](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L323)).
- RSF is explicitly marked dormant/shadow-only for offline comparison, not live ranking, so even its fixed rules are not currently shaping production order ([rsf-fusion.ts:7](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L7)).

**Verdict:** configurable in places, but not data-driven in the reviewed code.

## 3. Is there score normalization between vector vs BM25 vs FTS?
- In live RRF, not really. RRF discards raw channel magnitudes and fuses on rank only, so channel comparability comes from position, not calibrated score normalization ([rrf-fusion.ts:22](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L22), [rrf-fusion.ts:247](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L247)).
- RRF can optionally min-max normalize the **final fused RRF scores** to `[0,1]`, but that is post-fusion normalization, not per-channel calibration ([rrf-fusion.ts:281](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L281), [rrf-fusion.ts:478](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L478)).
- RSF does explicit per-source min-max normalization before averaging, so RSF is the only reviewed mechanism that truly calibrates channel score scales against each other ([rsf-fusion.ts:68](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L68), [rsf-fusion.ts:101](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L101), [rsf-fusion.ts:208](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L208)).
- Composite scoring has its own normalization for composite outputs, but that is downstream and unrelated to vector/BM25/FTS channel alignment ([composite-scoring.ts:838](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/scoring/composite-scoring.ts#L838)).
- There is also a latent unit mismatch: some paths treat `similarity` as `0-100` and divide by 100, while `extractScoringValue` returns raw `similarity` as-is ([types.ts:65](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/pipeline/types.ts#L65), [stage4-filter.ts:215](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/pipeline/stage4-filter.ts#L215)).

**Verdict:** RRF sidesteps normalization by using ranks; RSF does real per-channel normalization; the wider pipeline does not maintain a single consistent numeric scale.

## 6. Score distribution shape: cliffs or dead zones?
This part is formula-based inference; I did not run live histograms.

- RRF raw scores are very compressed because `k=60` makes even rank-1 worth only about `0.0164` before bonuses; then a fixed `+0.10` convergence bonus can dominate rank differences, creating a big multi-source cliff ([rrf-fusion.ts:36](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L36), [rrf-fusion.ts:247](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L247), [rrf-fusion.ts:272](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L272)).
- RRF and RSF normalization both have a plateau mode: if all scores are equal, every result becomes `1.0`, which is a dead zone for differentiation ([rrf-fusion.ts:505](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L505), [rsf-fusion.ts:77](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L77)).
- RSF has explicit coverage cliffs: pairwise single-source rows are halved, and multi-list rows are scaled by `countPresent / totalSources`; that strongly favors cross-channel agreement over strong single-channel evidence ([rsf-fusion.ts:173](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L173), [rsf-fusion.ts:287](/Users/michelkerkmeester/MEGA/Development/Opencode Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/rsf-fusion.ts#L287)).

## Bottom Line
The system is sophisticated, but its weighting/calibration story is mostly heuristic. The only clearly external reference is RRF’s literature default `k=60`; most other numbers are hand-set, sometimes explicitly labeled “initial” or “provisional.” The biggest concrete risk I found is not lack of knobs, but lack of a single consistent score contract across pipeline stages.
