# Iteration 002: Hook vs Non-Hook CLI Guidance Gap Analysis

## Focus: Analyze how non-hook CLIs currently receive tool guidance versus hook-compatible Claude Code, and map the runtime detection, fallback, and PrimePackage delivery path.

## Findings:
1. Runtime detection explicitly separates hook-capable and hookless/default-fallback runtimes. `detectRuntime()` returns `claude-code` with `hookPolicy: 'enabled'`, `codex-cli` with `hookPolicy: 'unavailable'`, `copilot-cli` with `hookPolicy: 'disabled_by_scope'`, and `gemini-cli` with a dynamic policy derived from `.gemini/settings.json`. `getRecoveryApproach()` then collapses this to `hooks` only when hooks are enabled; otherwise it returns `tool_fallback`. Cross-runtime tests assert that only Claude uses `hooks` in the default matrix, while Codex, Copilot, and Gemini all use `tool_fallback`. 2. Non-hook CLIs already receive some guidance via MCP, not only via instruction files, but that MCP guidance is generic recovery/bootstrap guidance rather than active search-tool routing. `CODEX.md` explicitly says Codex does not have hook-based context injection and must manually recover with `memory_context(...resume...)` and `code_graph_status()`, but it also says the MCP server auto-primes on the first tool call. The same pattern appears in the context-prime agent docs: the agent calls `session_resume()` and `session_health()` to emit a compact Prime Package, but the package is framed as session/bootstrap context rather than a tool-routing policy. 3. The current PrimePackage flow is: first MCP tool call enters the context-server call handler, the server runs `primeSessionIfNeeded(name, args)`, `primeSessionIfNeeded()` builds a `PrimePackage`, then after the actual tool returns the server injects the priming hints into the response envelope via `injectSessionPrimeHints()`. The raw priming payload is also written to `meta.sessionPriming`. This means the package reaches the model only after the model has already selected and invoked some tool; it is not a pre-tool-choice directive channel. 4. The PrimePackage content is operationally useful but too weak for routing enforcement. It contains `specFolder`, `currentTask`, `codeGraphStatus`, `cocoIndexAvailable`, and `recommendedCalls`. The `recommendedCalls` logic only suggests `code_graph_scan`, `memory_context({ input: "resume previous work", mode: "resume", profile: "resume" })`, or `memory_match_triggers({ prompt: "<your task>" })`. It never emits the missing decision rule: semantic/concept search -> CocoIndex, structural/call/import analysis -> Code Graph, exact token/path lookup -> Grep/Glob. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:321-363]

5. `buildServerInstructions()` is the only startup-scoped MCP instruction channel investigated that could reach all runtimes uniformly, but it currently advertises memory inventory and session recovery only. It emits indexed-memory counts, channel availability, memory tool names, and a short session-recovery digest with recommendations like `call session_resume()`, `run code_graph_scan`, or `call memory_context(resume)`. It does not mention CocoIndex/Code Graph routing rules at all, and the file comment states these instructions are computed once at startup and not refreshed during the session. For non-hook CLIs, this is the clearest MCP-wide place where routing guidance could exist but currently does not. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/context-server.ts:602-645]

6. The server has response-hint plumbing, but it currently reinforces surfaced memory/session state, not tool-routing correction. `appendAutoSurfaceHints()` only injects counts/timing for constitutional and triggered memories. `injectSessionPrimeHints()` only adds hints for constitutional-memory count, active spec folder, code graph availability, CocoIndex availability, and recommended next calls. No investigated hook or response-hint code path currently classifies a query as semantic vs structural vs exact-match and redirects accordingly. 7. Hook-compatible Claude Code differs mainly in transport timing, not in the richness of routing content. The runtime matrix and fallback tests show Claude gets hooks while Codex/Copilot/Gemini default to tool fallback. However, the actual priming payload built by the MCP server is the same generic package for whichever runtime makes the first tool call. In other words: hook-capable runtimes can receive context earlier via hooks, but the currently delivered content still lacks the concrete routing directives needed to stop Grep/Glob overuse. 9. The key information currently lost for non-hook CLIs is the actionable routing rule itself at the moment of tool choice. Non-hook runtimes can recover session context through instruction files and MCP bootstrap metadata, but the investigated MCP surfaces do not currently say: use CocoIndex for semantic/code-understanding requests, use Code Graph for structural/caller/import/impact questions, and reserve Grep/Glob for exact text or known-path lookups. As a result, non-hook CLIs receive availability/status signals without the higher-priority behavioral rule that should govern tool selection.
