# Iteration 025: Recovery Payload Format Analysis

## Focus

Analyze the recovery payload and hook-envelope formats used by Spec Kit Memory search and hook layers, then translate those findings into a concrete stdout contract for `compact-inject.js`.

## Findings

1. The structured recovery payload is a small four-field object with enum-constrained fields. `recovery-payload.ts` defines `RecoveryStatus = "no_results" | "low_confidence" | "partial"`, `RecoveryReason = "spec_filter_too_narrow" | "low_signal_query" | "knowledge_gap"`, `RecoveryAction = "retry_broader" | "switch_mode" | "save_memory" | "ask_user"`, and `RecoveryPayload { status, reason, suggestedQueries, recommendedAction }`. The builder input is `RecoveryContext { query, hasSpecFolderFilter, evidenceGap?, resultCount, avgConfidence?, lowConfidenceThreshold? }`. Source: `.opencode/skill/system-spec-kit/mcp_server/lib/search/recovery-payload.ts:22-53`

2. The recovery payload is constructed in the search formatter, not in the hook layer. For empty results, `formatSearchResults()` directly calls `buildRecoveryPayload({ query, hasSpecFolderFilter, resultCount: 0 })` and attaches the result as `data.recovery` inside the standard MCP empty envelope. For non-empty weak/partial results, the formatter computes average confidence from `computeResultConfidence()`, builds `recoveryCtx` from `query`, `specFolder`, `formatted.length`, `avgConfidence`, and `Boolean(extraData?.evidenceGap)`, then conditionally attaches `data.recovery` when `shouldTriggerRecovery()` returns true. Sources: `.opencode/skill/system-spec-kit/mcp_server/formatters/search-results.ts:394-427`, `.opencode/skill/system-spec-kit/mcp_server/formatters/search-results.ts:582-680`

3. The data sources feeding the recovery payload are a mix of direct request inputs and post-pipeline annotations. `memory-search.ts` passes the normalized query and resolved `specFolder` into `formatSearchResults()`, so the recovery builder sees the same query/filter context used by the actual retrieval. Evidence-gap intent originates in Stage 4 filtering/TRM, where `detectEvidenceGap()` computes `evidenceGapDetected` plus `evidenceGapWarning`. However, the current main handler only forwards `evidenceGapWarning` in `extraData`, while the formatter checks `extraData.evidenceGap` for partial classification. That means the design clearly expects an evidence-gap boolean to feed recovery, but the primary `memory-search` handler path shown here does not currently pass that boolean through. Sources: `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts:708-719`, `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-search.ts:784-795`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/pipeline/stage4-filter.ts:261-279`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/pipeline/types.ts:297-309`, `.opencode/skill/system-spec-kit/mcp_server/formatters/search-results.ts:603-613`

4. The recovery payload is not the same thing as the hook compact-inject output format. The recovery payload lives under `envelope.data.recovery` in MCP search responses. The hook layer, by contrast, operates on the outer MCP envelope shape `summary`, `data`, `hints`, and `meta`. `response-hints.ts` parses the serialized envelope in `result.content[0].text`, appends human-readable hint strings, writes `meta.autoSurface`, and reserializes the whole envelope with a synchronized `meta.tokenCount`. This means `compact-inject.js` should treat recovery payloads as one possible input fragment, not as the final stdout contract. Sources: `.opencode/skill/system-spec-kit/mcp_server/lib/response/envelope.ts:16-35`, `.opencode/skill/system-spec-kit/mcp_server/hooks/response-hints.ts:59-104`

5. `recovery-payload.ts` and `response-hints.ts` are related conceptually but are separate layers with no direct import relationship. `recovery-payload.ts` is search-formatting logic for weak/empty retrieval UX. `response-hints.ts` is hook/transport decoration logic for already-serialized MCP envelopes. The context server wires the latter in after tool dispatch and before token-budget enforcement, so hook metadata reflects the fully decorated response. In practice: search recovery yields `data.recovery`; hook decoration yields `hints[]`, `meta.autoSurface`, `meta.autoSurfacedContext`, `meta.tokenBudget`, and a re-synced `meta.tokenCount`. Sources: `.opencode/skill/system-spec-kit/mcp_server/context-server.ts:373-399`, `.opencode/skill/system-spec-kit/mcp_server/context-server.ts:401-433`, `.opencode/skill/system-spec-kit/mcp_server/formatters/search-results.ts:656-680`, `.opencode/skill/system-spec-kit/mcp_server/hooks/response-hints.ts:71-104`

6. Response hints are plain strings plus structured envelope metadata. `appendAutoSurfaceHints()` normalizes `envelope.hints` to an array of strings, appends a summary string of the form `Auto-surface hook: injected X constitutional and Y triggered memories (Zms)`, and writes `meta.autoSurface = { constitutionalCount, triggeredCount, surfaced_at, latencyMs }`. The context server then also stores the full raw surfaced payload at `meta.autoSurfacedContext` and injects `meta.tokenBudget`, with final `meta.tokenCount` re-estimated from the fully serialized envelope. The tests confirm that the final envelope includes all of those fields and that the final token count matches the serialized text. Sources: `.opencode/skill/system-spec-kit/mcp_server/hooks/response-hints.ts:71-104`, `.opencode/skill/system-spec-kit/mcp_server/context-server.ts:393-399`, `.opencode/skill/system-spec-kit/mcp_server/tests/context-server.vitest.ts:1044-1130`

7. The recovery payload object is typically small, but it has no absolute schema-level maximum. The shared token estimator is intentionally `ceil(chars / 4)`, not a model-specific tokenizer. Using that estimator on representative JSON payloads gives roughly 147 chars / 37 tokens for a minimal payload, 217 chars / 55 tokens for a typical three-suggestion payload, and 269 chars / 68 tokens for a longer-but-still-realistic payload. Hard shape limits do exist: exactly four top-level fields and at most three `suggestedQueries`. But there is no hard max string length because one suggestion can be the original query itself when a spec-folder filter is active, and query text is not length-bounded by the payload type. Operational ceilings therefore come from the enclosing envelope and hook budgets, not from `RecoveryPayload` itself. Sources: `.opencode/skill/system-spec-kit/shared/utils/token-estimate.ts:1-11`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/recovery-payload.ts:107-146`, `.opencode/skill/system-spec-kit/mcp_server/tests/d5-recovery-payload.vitest.ts:264-273`, `.opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:52-55`, `.opencode/skill/system-spec-kit/mcp_server/hooks/memory-surface.ts:294-316`, `.opencode/skill/system-spec-kit/mcp_server/context-server.ts:397-433`

8. For `compact-inject.js`, stdout should be a concise recovery/context brief, not a raw dump of the MCP envelope or the raw auto-surface JSON. The packet spec is explicit that SessionStart stdout is model-visible context, while envelope metadata alone is weaker than true prompt injection. It also establishes that resume/compact flows should use `memory_context({ mode: "resume", profile: "resume" })` to get the compact `{ state, nextSteps, blockers }` brief rather than verbose search results. Based on the current payload formats, the hook should read cached auto-surface/search output, extract only the semantically useful pieces (`summary`, key surfaced memories, `data.recovery` when present, and maybe a shortened hint), and emit a human-readable block or `hookSpecificOutput.additionalContext`. If the hook prints raw MCP JSON, it will waste token budget on transport metadata (`meta.tokenCount`, `cacheHit`, `tokenBudget`) and surface implementation details instead of recovery context. Sources: `.opencode/specs/system-spec-kit/024-compact-code-graph/spec.md:11-14`, `.opencode/specs/system-spec-kit/024-compact-code-graph/spec.md:23-29`, `.opencode/specs/system-spec-kit/024-compact-code-graph/001-precompact-hook/spec.md:50-66`, `.opencode/specs/system-spec-kit/024-compact-code-graph/002-session-start-hook/spec.md:14-15`, `.opencode/specs/system-spec-kit/024-compact-code-graph/002-session-start-hook/spec.md:25-26`, `.opencode/specs/system-spec-kit/024-compact-code-graph/002-session-start-hook/spec.md:45-49`, `.opencode/specs/system-spec-kit/024-compact-code-graph/decision-record.md:69-74`

9. Schema evolution is currently managed by additive rollout rather than explicit versioning. The recovery payload is behind `SPECKIT_EMPTY_RESULT_RECOVERY_V1`, which defaults to ON but can still be disabled. The formatter comments repeatedly describe recovery attachment as additive. There is no `version` field inside `RecoveryPayload` or `meta.autoSurface`. Separately, `response-hints.ts` explicitly preserves its helper return type to avoid a breaking API change across 10+ call sites. The implication for hook-cache design is that any persisted compact cache should add a cache-level schema/version marker if cross-version compatibility matters, because neither the recovery payload nor the auto-surface metadata is self-versioning today. Sources: `.opencode/skill/system-spec-kit/mcp_server/lib/search/recovery-payload.ts:10-18`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/recovery-payload.ts:221-224`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/search-flags.ts:508-514`, `.opencode/skill/system-spec-kit/mcp_server/formatters/search-results.ts:418-419`, `.opencode/skill/system-spec-kit/mcp_server/formatters/search-results.ts:664-665`, `.opencode/skill/system-spec-kit/mcp_server/hooks/response-hints.ts:51-56`

## Evidence

- Recovery payload definition and heuristics live entirely in `.opencode/skill/system-spec-kit/mcp_server/lib/search/recovery-payload.ts`, including enum types, suggestion generation, action mapping, and the default-on feature flag export.
- Search response attachment happens in `.opencode/skill/system-spec-kit/mcp_server/formatters/search-results.ts`, where `data.recovery` is injected for empty, weak, or partial result sets.
- Upstream evidence-gap generation happens in Stage 4 filtering, but the inspected `memory-search.ts` path forwards only `evidenceGapWarning`, not the boolean the formatter actually reads.
- Hook-layer response decoration happens in `.opencode/skill/system-spec-kit/mcp_server/hooks/response-hints.ts` and is called by `.opencode/skill/system-spec-kit/mcp_server/context-server.ts` before token-budget enforcement.
- The spec packet defines SessionStart stdout as model-visible context and requires `profile: "resume"` for compact brief formatting, which is the right contract for `compact-inject.js`.

## Novelty Justification

Most of this iteration adds implementation-level detail that was not captured in the spec packet summary: the exact `RecoveryPayload` and `RecoveryContext` types, the formatter-level attachment points, the distinction between `data.recovery` and hook `hints/meta`, the concrete token-size estimates derived from the shared estimator, and the likely propagation mismatch where the formatter expects `extraData.evidenceGap` but the inspected main handler only forwards `evidenceGapWarning`. Those are materially new details for implementation, not just a restatement of earlier “use resume profile” findings.

## Recommendations for Implementation

1. Format `compact-inject.js` stdout as a concise brief, not raw MCP JSON. Prefer a short markdown/context block with sections like `State`, `Next Steps`, `Blockers`, `Critical Memories`, and optionally `Recovery Guidance` when `data.recovery` exists.
2. Use `memory_context({ mode: "resume", profile: "resume" })` for fallback paths, because the packet explicitly says `mode: "resume"` alone returns search-style output that is too verbose for hook injection.
3. If the hook consumes cached MCP envelopes, parse and normalize them. Keep `summary`, a small subset of surfaced memory titles, and `data.recovery`; drop transport-only metadata such as `tokenCount`, `cacheHit`, and most `pipelineMetadata`.
4. Plumb a top-level boolean `extraData.evidenceGap` into `formatSearchResults()` if the intended behavior is that coverage gaps should force `status: "partial"` even when result count is otherwise high enough.
5. Add a cache-level schema marker such as `cacheVersion` or `payloadVersion` to any persisted compact-context file. The current recovery payload and hook metadata are additive but not self-versioning.
