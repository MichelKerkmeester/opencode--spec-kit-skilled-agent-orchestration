# Iteration 5 - Exclude-Rule Confidence Tier Definitions

## Summary

Q6 is answered with a three-tier exclusion model for code-graph scan scope. The current scanner has two exclusion layers: hard segment filters in `EXCLUDED_FOR_CODE_GRAPH` and `getDefaultConfig().excludeGlobs`, then inherited `.gitignore` matching during traversal. Locally, those layers already exclude dependency caches, compiled output, runtime databases, external/non-framework project trees, and archived/future spec folders. [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/lib/utils/index-scope.ts:31-48`; `.opencode/skill/system-spec-kit/mcp_server/code_graph/lib/indexer-types.ts:112-120`; `.opencode/skill/system-spec-kit/mcp_server/code_graph/lib/structural-indexer.ts:1178-1268`; `.gitignore:23-87`; `.gitignore:89-126`]

The main research outcome is that "always exclude" should be reserved for infrastructure/caches whose contents are never authoritative source for this code graph by default. Build-output and dependency-vendor names are only medium confidence because verified public repositories use the same path segments for committed source-like or test-critical content: Kubernetes commits `vendor/` source trees, Bazel has canonical Java source under a `build/` package segment, React has build tooling under `scripts/rollup/`, ESLint has committed `tests/fixtures`, and Go has committed `src/cmd/go/testdata` files. [SOURCE: `https://github.com/kubernetes/kubernetes/tree/e1cf06b9b41f3534fd1ce475a18ef20d46e8f876/vendor`; `https://github.com/bazelbuild/bazel/tree/ad2e223235bcacd1d026b1ee8ff851347bc41f7f/src/main/java/com/google/devtools/build`; `https://github.com/facebook/react/tree/561ed529b3a6a16e5b2b76fa5ee86c09f959686c/scripts/rollup`; `https://github.com/eslint/eslint/tree/54080dad4f77bb39a1a843933d4ff3a2b7c175e2/tests/fixtures`; `https://github.com/golang/go/tree/02b3e0d4dd9a94b005189a10d398db61326bf1c0/src/cmd/go/testdata`]

Local repo-specific candidates belong in the low tier unless an operator explicitly confirms the scan target is system-spec-kit-only. Examples from this repo include `/a_nobel_en_zn`, `/ai_systems`, `.opencode/specs/00--*`, `z_archive`, `z_future`, and `scratch` trees. They are bloat candidates for this packet, but they can contain useful project, archive, fixture, or research context for other tasks. ## Tier Definitions

| Tier | Definition | Default action | Confirmation rule |
|---|---|---|---|
| High confidence / always-exclude | Infrastructure, dependency caches, interpreter caches, runtime indexes, or metadata paths with near-zero source-authority value for code graph answers. | Exclude without prompting; allow explicit include overrides only for narrow fixture investigations. | No confirmation needed unless the user explicitly asks to index tool/cache internals. |
| Medium confidence / usually-exclude | Conventional build outputs, vendored trees, generated trees, or output folders that are often noise but have common real-world source/test exceptions. | Exclude by default for general scans, but surface the pattern and allow operator confirmation or include overrides. | Prompt/summarize before adding a new global default or when the path appears under `src/`, `tests/`, `fixtures/`, or language package namespaces. |
| Low confidence / per-repo decision | Repo-local names or lifecycle folders whose meaning is project policy, not ecosystem convention. | Do not globally exclude; treat as candidate rules in repo-level config. | Require explicit operator approval and record the reason in the packet/config. |
