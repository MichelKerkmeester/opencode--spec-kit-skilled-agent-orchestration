# Iteration 4: Implementation-Ready Delta List

## Focus
Turn the already-established workflow-owned emitter contract into an implementation-ready delta list: pin the exact opt-out/config plumbing, define a file/stream-shaped extractor API that stays faithful to the phase-012 template, and identify the packet spec/tasks/checklist changes needed to align phase 013 with the live synthesis boundary.

## Findings
1. The clean opt-out path is a new top-level `resourceMap.emit` boolean in both deep-loop config templates, defaulting to `true`, with `--no-resource-map` mapped in the command setup flow directly onto that field. Both command entrypoints already describe CLI-flag-to-config mapping for non-executor settings, both config templates currently lack any `resourceMap` block, and `parseExecutorConfig()` only validates the `executor` subtree, so pushing the flag into `executor-config.ts` would be the wrong coupling point. The workflow delta is: add `resourceMap.emit` to `assets/deep_research_config.json` and `assets/deep_review_config.json`, teach `/spec_kit:deep-research` and `/spec_kit:deep-review` to parse `--no-resource-map`, and guard a new synthesis-time `step_emit_resource_map` with `config.resourceMap.emit != false`. 2. The shared extractor API should be shaped around per-iteration delta files, not the packet's imagined nested `findings[]` payloads. The live contract is one JSONL file per iteration containing one canonical `{"type":"iteration",...}` row plus event rows, and the phase-012 template only exposes `Path | Action | Status | Note` plus summary counts. That means the stable boundary is a two-stage API such as `collectResourceMapEvidence({ mode, deltaFiles, repoRoot }) -> { rows, summary, degraded }` followed by `renderResourceMap({ rows, summary, scope, generatedAt }) -> markdown`, where review counts and research citation totals are encoded in `Note` text and summary totals instead of custom columns. ## Reflection
- What worked and why: Reading the live command flag-mapping docs, config templates, executor validator, and synthesis-facing docs together turned a fuzzy "add an opt-out" requirement into a concrete plumbing path, because each surface owns one slice of the real runtime contract.
- What did not work and why: The phase packet alone remained misleading on three fronts: it still assumes reducer-local invocation, still proposes custom columns, and still treats new standalone feature/playbook files as the main doc surface. That packet drift is now large enough that it cannot be used as an implementation checklist without revision.
- What I would do differently: Jump to the command/config validator surfaces earlier whenever a packet requirement mentions a CLI flag or config knob. That would have exposed the executor-vs-artifact boundary sooner.
