# Iteration 009 — Focus: Schema/Runtime Pressure-Test of `/spec_kit:start` Drafts

## 1. Focus
Pressure-test iteration-008's `/spec_kit:start` markdown/YAML drafts against live helper contracts and packet requirements, with emphasis on artifact transaction boundaries, `graph-metadata.json.manual.*` shape, placeholder-upgrade exit criteria, `generate-context.js` reuse risk, and `recommend-level.sh` invocation semantics.

## 2. Runtime Sources Checked
- Packet state and scope: `research/deep-research-config.json`, `research/deep-research-strategy.md`, `research/deep-research-state.jsonl`, `spec.md`, and iterations `001`-`008`.
- Helper/runtime surfaces: `.opencode/skill/system-spec-kit/scripts/dist/memory/generate-context.js`, `.opencode/skill/system-spec-kit/scripts/tests/task-enrichment.vitest.ts`, `.opencode/skill/system-spec-kit/scripts/dist/spec-folder/generate-description.js`, `.opencode/skill/system-spec-kit/scripts/spec/recommend-level.sh`.
- Graph metadata references: sibling sample `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/011-skill-advisor-graph/graph-metadata.json` and prior packet reasoning in `iteration-004.md`, `iteration-005.md`, and `iteration-008.md`.

## 3. Finding 1 — Existing helpers do not currently provide true three-artifact atomicity
The packet spec requires `/start` to treat `spec.md`, `description.json`, and `graph-metadata.json` as a single transactional unit, but the live helper contract still models recoverable partial state, not all-or-nothing commit semantics. The strongest contradiction is inside the packet itself: `NFR-R02` demands cleanup or exact recovery steps on any partial failure, while the edge-case section explicitly says that if `generate-description.js` crashes after `spec.md` is written, `spec.md` remains and metadata generation is retried later. `generate-description.js` also behaves like a single-file emitter: it writes only `description.json` after its own validation path succeeds, and exits non-zero on missing/unreadable `spec.md` rather than coordinating rollback of any other artifact. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/spec.md:188-189] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/spec.md:206-207] [SOURCE: .opencode/skill/system-spec-kit/scripts/dist/spec-folder/generate-description.js:109-114]

Recommendation: iteration-008's draft should stop implying that existing helpers already satisfy `REQ-005`/`NFR-R02`. `/start` either needs a new staged-write wrapper (`*.tmp` + final commit only after all three outputs validate) or the spec language must be narrowed to "recoverable partial state with explicit repair commands" instead of "transactional unit."

## 4. Finding 2 — `manual.*` must emit object arrays, and the safe minimum shape is stricter than the draft currently states
The draft's `manual_relationships` field currently guarantees only array presence, but the packet requirements and live sample both point to object-array entries, not bare packet strings. `REQ-007` requires `reason` and `source`, `NFR-S03` makes `source` mandatory for provenance, and the sibling `graph-metadata.json` sample shows the concrete shape already in use: `{ "packet_id", "reason", "source" }` under `manual.related_to`. Prior packet reasoning in iteration 004 also concluded that the relationship capture should stay inside `manual.depends_on`, `manual.related_to`, and `manual.supersedes` as "small objects," not a new free-form metadata block. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/spec.md:132-133] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/spec.md:185-185] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/011-skill-advisor-graph/graph-metadata.json:13-22] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/research/iterations/iteration-004.md:17-17]

Recommendation: `/start` should emit the minimum safe shape as `[{ packet_id, reason, source, spec_folder?, title? }]`, with `packet_id` normalized as the authoritative edge target and `spec_folder`/`title` treated as optional denormalized hints. The YAML contract should say this explicitly instead of only saying "arrays only."

## 5. Finding 3 — Placeholder-upgrade needs explicit seed markers and anchor-scoped exit criteria
The packet direction is consistent that unresolved deep-research placeholders are re-entry state, not "healthy populated folder" state, but iteration-008's draft still leaves detection and success criteria too heuristic. The packet spec says deep-research seeds a Level 1 `spec.md` with TODO placeholders in Scope and Requirements, and later says `/start` must detect deep-research-authored placeholders and upgrade them via interview. Iteration 005 already tightened the exit rule: re-entry stays active until the placeholder sentinels are cleared or intentionally replaced with `"N/A - insufficient source context"`. Iteration 008 then classified `placeholder-upgrade` as "deep-research seed markers or unresolved TODO placeholders" and required those placeholders to be replaced before success. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/spec.md:122-122] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/spec.md:160-160] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/spec.md:214-214] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/research/iterations/iteration-005.md:64-67] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/research/iterations/iteration-008.md:195-221]

Recommendation: `/start` should not use a generic `"contains TODO"` test. Deep-research seed-create needs explicit machine markers for each placeholder-bearing anchor, and placeholder-upgrade should only exit when every tracked seed marker is resolved, `selected_level` is set, and no anchor-specific placeholder sentinel remains.

## 6. Finding 4 — `generate-context.js` is a memory-save pipeline, not a lightweight metadata repair primitive
The live `generate-context.js` CLI is JSON-primary and explicitly describes itself as creating a saved context artifact, indexing it into Spec Kit Memory, and refreshing root-level `graph-metadata.json` as part of that same workflow. It accepts structured JSON via `--stdin`, `--json`, or a JSON file, and requires a target spec folder in structured mode; direct spec-folder-only invocation is no longer supported. That means `/start` can technically call it, but only as a full save pipeline with memory-quality gates attached. The strongest evidence against using it for start-only repair is the skipped thin-save test: when explicit JSON contains almost no evidence, the expected runtime behavior is `INSUFFICIENT_CONTEXT_ABORT` and zero written files. That is exactly the wrong failure model for a create/repair command whose job is canonical metadata creation rather than session narrative capture. [SOURCE: .opencode/skill/system-spec-kit/scripts/dist/memory/generate-context.js:62-89] [SOURCE: .opencode/skill/system-spec-kit/scripts/dist/memory/generate-context.js:343-387] [SOURCE: .opencode/skill/system-spec-kit/scripts/tests/task-enrichment.vitest.ts:1225-1273]

Recommendation: `/start` should not rely on `generate-context.js` for empty-folder creation or metadata-only repair. Reuse `generate-description.js` plus the existing graph-metadata regeneration path directly, and reserve `generate-context.js` for runs that actually have session evidence worth saving. If the product wants a start-only repair convenience, it needs a lighter metadata-only helper rather than a thin JSON call into the memory save pipeline.

## 7. Finding 5 — `recommend-level.sh` is reusable from YAML, but only after `/start` derives numeric inputs
`recommend-level.sh` is a clean shell helper with a stable CLI surface: it accepts `--loc`, `--files`, risk flags, phase flags, and `--json`, then maps the weighted score to Level 0-3 and optional phase recommendations. A sample JSON invocation exits `0` and returns machine-readable fields including `recommended_level`, `level_name`, `confidence`, `recommended_phases`, `suggested_phase_count`, `inputs`, `breakdown`, and `thresholds`. The important constraint is that the helper does not accept free-text feature descriptions or raw research asks; it consumes only numeric estimates and boolean risk flags. That means `/start` can invoke it from YAML, but only after it derives a scoring proxy from interview answers or a small fixed rubric. [SOURCE: .opencode/skill/system-spec-kit/scripts/spec/recommend-level.sh:14-28] [SOURCE: .opencode/skill/system-spec-kit/scripts/spec/recommend-level.sh:37-61] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/spec.md:133-133] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/012-spec-kit-commands/spec.md:206-206] [SOURCE: local command `bash .opencode/skill/system-spec-kit/scripts/spec/recommend-level.sh --loc 100 --files 3 --json` (exit 0)]

Recommendation: `/start` should call `recommend-level.sh --json` from YAML only after it computes provisional `loc/files/risk` estimates. The resulting JSON should be stored as the recommendation record, while the user-confirmed override remains the authoritative selected level.

## 8. Pressure-Test Matrix And Next Focus
### PRESSURE-TEST MATRIX
| Concern | Finding | Recommendation |
|---|---|---|
| Artifact transaction boundaries | Current helpers support recoverable partial state, not atomic three-artifact commit. | Add a staged-write wrapper or narrow `NFR-R02` to explicit repair semantics. |
| `graph-metadata.json.manual.*` shape | Draft is too loose; safe minimum is object arrays with provenance, not string arrays. | Emit `{ packet_id, reason, source, spec_folder?, title? }` objects under `manual.*`. |
| Placeholder-upgrade exit criteria | "Contains TODO" is too broad; re-entry needs seed-marker-aware resolution checks. | Add explicit deep-research seed markers and clear/replace-all exit gating per tracked anchor. |
| `generate-context.js` reuse for start-only runs | Full memory save pipeline is too heavy and can fail on thin JSON context. | Do not use it for creation/repair; use direct metadata helpers or add a lighter repair helper. |
| `recommend-level.sh` invocation contract | Script is reusable, but it scores structured numeric inputs, not free text. | Invoke `--json` only after `/start` derives loc/files/risk proxies; store recommendation separately from selected level. |

### Ruled Out
- Treating the current `generate-description.js` + `generate-context.js` combination as proof of runtime transactionality.
- Allowing `/start` to emit bare packet strings in `manual.*` and hoping later normalization fixes them.
- Treating generic `TODO` detection as sufficient proof of deep-research placeholder state.

### Next Focus
Iteration 010 should turn these pressure-test findings into packet-local spec deltas: tighten `REQ-005`/`NFR-R02`, formalize the `manual.*` object schema in the `/start` drafts, define the seed-marker contract for placeholder-upgrade, and split `/start` save behavior into metadata repair vs memory save so the runtime story is implementable without hidden helper changes.
