## Summary
This iteration answered Q5 and Q10 by comparing `PHRASE_INTENT_BOOSTERS` in `.opencode/skill/scripts/skill_advisor.py` against the per-skill `graph-metadata.json` files and the compiled `.opencode/skill/scripts/skill-graph.json`. The main result is that the graph currently replaces only a very small slice of the hardcoded phrase relationships, while `intent_signals` already duplicate a meaningful part of the phrase table but are dropped during compilation. The highest-ROI path is to move canonical intent routing into compiled metadata, keep only a small manual override layer, and make graph propagation strong enough to cover the few multi-skill relationships that are already modeled as edges.

## Findings
1. The runtime graph path is topology-only today. `skill_advisor.py` loads `skill-graph.json`, applies adjacency-based transitive boosts, and then applies family affinity, but it never reads `intent_signals` or `domains` because those fields are not present in the compiled artifact (`.opencode/skill/scripts/skill_advisor.py:68-140`, `.opencode/skill/scripts/skill_advisor.py:1488-1498`). The compiler currently emits only `families`, `adjacency`, `conflicts`, and `hub_skills`, and it excludes `prerequisite_for` plus all non-edge metadata (`.opencode/skill/scripts/skill_graph_compiler.py:233-290`). This means the graph cannot currently replace most hardcoded routing phrases even when the source metadata already describes them.

2. Only 2 of the 9 multi-skill phrase relationships in `PHRASE_INTENT_BOOSTERS` are now graph-backed. The redundant or partially redundant relationships are `spec folder review -> sk-code-review`, because `sk-deep-review` already `depends_on` `sk-code-review` at `0.8` (`.opencode/skill/scripts/skill_advisor.py:707`, `.opencode/skill/sk-deep-review/graph-metadata.json:7-18`), and `figma css -> sk-code-web`, because `mcp-figma` already `enhances` `sk-code-web` at `0.4` (`.opencode/skill/scripts/skill_advisor.py:716`, `.opencode/skill/mcp-figma/graph-metadata.json:7-18`). The remaining 7 multi-skill phrase relationships do not have graph equivalents yet: `pr review`, `merge readiness`, `ready to merge`, `api network`, `cross-ai review`, `codex review`, and `claude code review` (`.opencode/skill/scripts/skill_advisor.py:616-638`, `.opencode/skill/scripts/skill_advisor.py:728-750`).

3. Even those 2 graph-backed relationships are not strong enough to remove the hardcoded phrase entries yet. The current transitive math is `enhances * 0.3`, `siblings * 0.15`, and `depends_on * 0.2`, with a floor of `0.1` before a boost is emitted (`.opencode/skill/scripts/skill_advisor.py:98-119`). For `spec folder review`, the hardcoded primary boost is `2.0` to `sk-deep-review`, so the graph-derived `sk-code-review` boost would be `2.0 * 0.8 * 0.2 = 0.32`, which is much smaller than the explicit `0.8`. For `figma css`, the graph-derived `sk-code-web` boost would be `0.8 * 0.4 * 0.3 = 0.096`, which falls below the `0.1` cutoff and is dropped entirely. So the relationships exist in the graph, but one is only partially substitutable and the other is effectively inert.

4. `intent_signals` can already auto-generate a meaningful exact subset of the phrase table. I found 21 exact phrase-skill matches where the hardcoded phrase string is already present verbatim in a skill’s `intent_signals`. The cleanest exact duplicates are: `code review`, `pr review`, `security review`, and `quality gate` for `sk-code-review` (`.opencode/skill/sk-code-review/graph-metadata.json:19-20`); `deep research` and `research loop` for `sk-deep-research` (`.opencode/skill/sk-deep-research/graph-metadata.json:15-16`); `deep review`, `review loop`, `iterative review`, and `spec folder review` for `sk-deep-review` (`.opencode/skill/sk-deep-review/graph-metadata.json:17-18`); `semantic search`, `code search`, and `coco index` for `mcp-coco-index` (`.opencode/skill/mcp-coco-index/graph-metadata.json:13-14`); `improve agent` and `evaluate agent` for `sk-improve-agent`; and the exact CLI phrases `gemini cli`, `delegate to gemini`, `codex cli`, `delegate to codex`, `copilot cli`, and `delegate to copilot` from their respective metadata.

5. A second tier of phrase entries is also derivable from `intent_signals`, but only with simple alias templates rather than exact string copy. I found roughly 29 phrase-skill pairs that are near matches to existing signals, such as `use claude code`, `claude code cli`, and `delegate to claude code` from the `claude code` / `delegate to claude` signal family; `find code that` and `semantic code search` from the `find code` / `code search` family; and `:review`, `/improve:agent`, `/improve:prompt`, `sk-code-review`, and similar explicit naming wrappers that are just templated variants of canonical intent or skill identity. This suggests the graph system does not need a 1:1 replacement phrase table; it needs compiled canonical signals plus a deterministic alias-expansion layer.

6. `intent_signals` should be included in the compiled output. Right now a zero-edge skill such as `system-spec-kit` still carries valuable routing metadata in its source file (`spec folder`, `save context`, `save memory`, `spec kit`, `memory search`), but none of that survives compilation because the skill only contributes to `families`, not adjacency (`.opencode/skill/system-spec-kit/graph-metadata.json:13-14`, `.opencode/skill/scripts/skill_graph_compiler.py:249-267`). Including `intent_signals` in the compiled artifact would let the runtime use graph metadata as the canonical source for exact intent routing even when a skill has no edges.

7. `domains` are worth compiling too, but not as exact phrase boosters. A quick size check showed the current compiled `skill-graph.json` is about `1808` bytes, while adding raw per-skill `domains` plus `intent_signals` would raise it to about `5112` bytes, a delta of `3304` bytes. That is still tiny in absolute terms for a local JSON load, but it would exceed the compiler’s current `2KB` warning threshold. The bigger issue is semantic noise: domains like `review`, `cli`, `prompt`, `agent`, `css`, and `session` are too broad to safely behave like exact routing phrases. My assessment is: compile `intent_signals` for canonical routing; compile `domains` only for low-weight fallback scoring, coarse recall, or analytics.

8. Highest ROI improvement #1: compile a per-skill `skill_metadata` section that includes at least `intent_signals`, and preferably `domains`, alongside the current topology output. This is the gating improvement because every other graph-driven routing improvement depends on the runtime being able to see that metadata from a single compiled source instead of duplicating it manually in `skill_advisor.py`.

9. Highest ROI improvement #2: auto-generate the exact phrase boosters from compiled `intent_signals` and keep only exceptions in `PHRASE_INTENT_BOOSTERS`. This would immediately eliminate the 21 exact duplicates without reducing expressiveness, and it would move canonical routing ownership to `graph-metadata.json`, which is already the declared source-of-truth for skill relationships and signals.

10. Highest ROI improvement #3: add a deterministic alias-expansion layer for canonical signals and skill identities. The small, high-value template set is straightforward: slash-prefixed variants, skill-name variants with hyphen/space normalization, imperative wrappers like `use X`, delegation wrappers like `delegate to X`, and a few command-style wrappers such as `:review`. This is a better fit than hand-maintaining dozens of near-duplicate phrase entries.

11. Highest ROI improvement #4: add a drift validator that compares phrase boosters to graph metadata. At minimum it should report three cases: exact signal duplicates that can be deleted, multi-skill phrase relationships whose secondary targets are already graph-backed, and multi-skill phrase relationships that still have no graph-edge representation. That would keep the graph and phrase layer from silently diverging again.

12. Highest ROI improvement #5: strengthen routing-time graph propagation for edge-backed secondary skills. The current coefficients and `0.1` floor are conservative enough that an existing graph relation like `mcp-figma -> sk-code-web` does not materialize from the `figma css` phrase at all. A routing-focused edge mode, per-edge-type minimum floor, or higher multiplier for phrase-originated boosts would let the graph replace more of the remaining manual secondary relationships instead of merely annotating them.

## New Questions
1. Should the compiler emit `intent_signals` and `domains` inside `skill-graph.json`, or should it write a second compact artifact such as `skill-routing-metadata.json` to avoid changing the current topology contract?
2. Do we want graph-backed secondary routing to be approximate by design, or should certain edges support a stronger “recommended companion skill” mode that can fully replace hardcoded secondary boosts?
3. Which of the 7 remaining multi-skill phrase relationships should become explicit graph edges, and which should remain hand-curated because they represent product policy rather than reusable structural relationships?
4. Should `domains` stay intentionally broad, or should the system add a narrower `routing_keywords` / `intent_aliases` field so low-noise routing terms do not have to be inferred from generic taxonomy labels?
