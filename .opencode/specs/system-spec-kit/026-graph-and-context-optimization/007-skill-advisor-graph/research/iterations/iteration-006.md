## Summary
This iteration audited the remaining MCP and CLI skill `graph-metadata.json` files that were not covered in iteration 1, cross-checked each one against its `SKILL.md` frontmatter description, and re-tested the two open semantic questions from the prompt. The main result is that six of the seven files are structurally and semantically in reasonable shape as written: the three CLI orchestrators are fully consistent with the established `0.5` sibling pattern, and the `mcp-clickup` / `mcp-figma` / `mcp-chrome-devtools` dependencies on `mcp-code-mode` all match their documented Code Mode routing. The two substantive gaps are narrower: `mcp-chrome-devtools -> sk-code-web` looks directionally correct but probably a bit underweighted, and `mcp-coco-index` still looks like a semantically important orphan even though its zero-edge metadata is schema-valid.

## Findings
1. All seven target files are valid JSON and pass the graph compiler's metadata validation, so this pass did not find any schema or structural breakage. Running `python3 .opencode/skill/skill-advisor/scripts/skill_graph_compiler.py --validate-only` reported `VALIDATION PASSED: all metadata files are valid`, which means the questions here are about modeling quality rather than broken file format (`.opencode/skill/skill-advisor/scripts/skill_graph_compiler.py`, validation run on 2026-04-13).

2. `mcp-clickup` is correctly modeled for the relationships that currently exist in the graph. Its metadata declares only `depends_on -> mcp-code-mode (0.9)` (`.opencode/skill/mcp-clickup/graph-metadata.json:6-16`), and that matches both the frontmatter description that the MCP path is the Code Mode path for enterprise ClickUp features (`.opencode/skill/mcp-clickup/SKILL.md:2-5`) and the usage section emphasizing docs, goals, webhooks, and bulk operations (`.opencode/skill/mcp-clickup/SKILL.md:19-26`). I do not see evidence in the current skill set for an additional graph edge beyond that dependency, because there is no separate local ClickUp CLI skill node to model the "CLI vs MCP" split as a sibling relation.

3. `mcp-figma` is also correctly modeled at the current level of granularity. The `depends_on -> mcp-code-mode (0.9)` edge is consistent with its frontmatter description that Figma access is "via Code Mode" (`.opencode/skill/mcp-figma/graph-metadata.json:6-18`, `.opencode/skill/mcp-figma/SKILL.md:2-4`), and the `enhances -> sk-code-web (0.4)` edge is supported by the design-to-code positioning in the body text (`.opencode/skill/mcp-figma/SKILL.md:12-14`, `.opencode/skill/mcp-figma/SKILL.md:30-44`). This still looks like a valid moderate overlay rather than a missing or contradictory relationship.

4. The CLI sibling weights are fully consistent and satisfy the "all should be `0.5`" check. `cli-codex`, `cli-copilot`, `cli-gemini`, and the already-existing `cli-claude-code` all record symmetric `siblings` edges to the other three peers, every one at `0.5`, with the same context string `CLI orchestrator peer` (`.opencode/skill/cli-claude-code/graph-metadata.json:6-18`, `.opencode/skill/cli-codex/graph-metadata.json:6-18`, `.opencode/skill/cli-copilot/graph-metadata.json:6-18`, `.opencode/skill/cli-gemini/graph-metadata.json:6-18`). That lines up with the packet's declared sibling weight band of `0.4-0.6`, so there is no mismatch here (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/007-skill-advisor-graph/spec.md:115-120`).

5. `mcp-chrome-devtools -> sk-code-web` is the one audited edge that looks slightly underweighted rather than missing. The current metadata sets `enhances -> sk-code-web (0.5)` and already captures the right direction (`.opencode/skill/mcp-chrome-devtools/graph-metadata.json:6-18`), but the supporting documents show stronger overlap than the current mid-band weight implies. `mcp-chrome-devtools` explicitly covers browser debugging, DOM inspection, performance work, and CSS-adjacent browser tasks (`.opencode/skill/mcp-chrome-devtools/SKILL.md:12-26`), while `sk-code-web` explicitly says its debugging phase uses DevTools and its domains include both `browser` and `css` (`.opencode/skill/sk-code-web/SKILL.md:21-36`, `.opencode/skill/sk-code-web/SKILL.md:47-59`, `.opencode/skill/sk-code-web/graph-metadata.json:18-19`). My read is that the edge is correct but could reasonably be strengthened from `0.5` to `0.6` without overclaiming a prerequisite relationship.

6. `mcp-coco-index` is the remaining likely graph gap. Its zero-edge metadata is technically valid (`.opencode/skill/mcp-coco-index/graph-metadata.json:6-15`), but the skill description and surrounding workflow docs show that it is not actually isolated in use. Its frontmatter and activation section frame it as the default semantic search surface for unfamiliar-code exploration and concept-based code discovery (`.opencode/skill/mcp-coco-index/SKILL.md:2-4`, `.opencode/skill/mcp-coco-index/SKILL.md:20-41`), and `system-spec-kit` explicitly routes "semantic/concept queries" to CocoIndex (`.opencode/skill/system-spec-kit/SKILL.md:775-777`). That means "zero edges" is probably under-modeling a real workflow coupling. I would not call the exact target fully settled yet, but I do think this audit strengthens the case that `mcp-coco-index` should have at least one directional `enhances` relationship to a workflow skill that consumes semantic search. The cleanest current candidate among graph nodes is `system-spec-kit`, because that skill explicitly delegates semantic query routing to CocoIndex.

7. The three CLI skills' frontmatter descriptions do not reveal any hidden graph inconsistencies beyond the sibling mesh. `cli-codex` emphasizes code generation, code review, web research, architecture analysis, and parallel processing (`.opencode/skill/cli-codex/SKILL.md:2-5`, `.opencode/skill/cli-codex/SKILL.md:23-55`), `cli-copilot` emphasizes collaborative planning and cloud delegation (`.opencode/skill/cli-copilot/SKILL.md:2-5`, `.opencode/skill/cli-copilot/SKILL.md:23-52`), and `cli-gemini` emphasizes web research plus architecture analysis (`.opencode/skill/cli-gemini/SKILL.md:2-5`, `.opencode/skill/cli-gemini/SKILL.md:23-52`). Those capability differences show up appropriately in `domains` and `intent_signals`, but they do not justify changing the sibling weights. The current graph is treating them as same-family peers with different direct routing signals, which is consistent with both the metadata and the packet design.

## New Questions
1. Should `mcp-chrome-devtools -> sk-code-web` stay at the current medium overlay weight `0.5`, or should we strengthen it slightly to `0.6` now that both the metadata domains and the `sk-code-web` workflow text explicitly center browser/DevTools debugging?
2. If `mcp-coco-index` stops being a zero-edge skill, should its first edge be `enhances -> system-spec-kit`, or do we want to reserve `system-spec-kit` for broader workflow links and instead connect CocoIndex to a code-workflow skill such as `sk-code-full-stack` or `sk-code-web`?
3. More generally, are "tooling utility" skills like `mcp-coco-index` intentionally allowed to remain graph orphans as long as their `intent_signals` are strong, or does the packet want the graph to model their repeated workflow companionship too?
