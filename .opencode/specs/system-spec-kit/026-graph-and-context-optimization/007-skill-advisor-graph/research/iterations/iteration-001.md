## Summary
Reviewed the five requested `graph-metadata.json` files against their `SKILL.md` frontmatter and the advisor's ambiguous-keyword boosters. The strongest gap is `system-spec-kit`: it has an empty edge inventory even though both its frontmatter and advisor routing make it a recurring companion to review/spec-validation work. A second important gap is operational rather than declarative: `mcp-code-mode` declares `prerequisite_for` edges, but the compiled graph excludes that edge type, so those relations do not participate in transitive graph boosts.

## Findings
1. `system-spec-kit` is under-modeled in the skill graph. Its metadata declares no edges at all (`.opencode/skill/system-spec-kit/graph-metadata.json:6-12`), but its frontmatter explicitly spans documentation, validation, and context preservation (`.opencode/skill/system-spec-kit/SKILL.md:2-3`), and the advisor repeatedly co-boosts it for `audit`, `context`, `plan`, `save`, `session`, and `task` signals (`.opencode/skill/skill-advisor/scripts/skill_advisor.py:578-598`). This is a confirmed relationship gap, even if exact edge directions still need calibration.
2. `mcp-code-mode` has a source-metadata vs runtime-graph mismatch. Its metadata records `prerequisite_for` edges to `mcp-figma`, `mcp-clickup`, and `mcp-chrome-devtools` (`.opencode/skill/mcp-code-mode/graph-metadata.json:11-15`), and its frontmatter says it is mandatory for all MCP tool calls including ClickUp, Figma, and Chrome DevTools (`.opencode/skill/mcp-code-mode/SKILL.md:3`, `.opencode/skill/mcp-code-mode/SKILL.md:19-20`). But the compiler explicitly drops `prerequisite_for` from compiled adjacency (`.opencode/skill/skill-advisor/scripts/skill_graph_compiler.py:253-264`), and the advisor only propagates `depends_on`, `enhances`, and `siblings` (`.opencode/skill/skill-advisor/scripts/skill_advisor.py:96-120`). Result: these edges exist in source metadata but are inert for graph-based boosting.
3. `mcp-code-mode` also exposes a node-coverage gap. The frontmatter names Notion and Webflow alongside ClickUp/Figma/Chrome DevTools (`.opencode/skill/mcp-code-mode/SKILL.md:3`, `.opencode/skill/mcp-code-mode/SKILL.md:19-20`), but the current graph metadata only models the three skills that already have local graph nodes (`.opencode/skill/mcp-code-mode/graph-metadata.json:11-15`). This is not a simple missing edge inside the current graph; it indicates the graph cannot represent all capabilities named by the umbrella skill.
4. `sk-code-review` captures its baseline-to-overlay relationship strongly in one direction, but the effective graph is asymmetric. The frontmatter describes adaptive compatibility with `sk-code-opencode`, `sk-code-web`, and `sk-code-full-stack` (`.opencode/skill/sk-code-review/SKILL.md:3`, `.opencode/skill/sk-code-review/SKILL.md:12`), and its metadata encodes those as `enhances` edges with weight `0.7` (`.opencode/skill/sk-code-review/graph-metadata.json:8-12`). The reverse graph files only point back with `0.3` `enhances` weights, which means overlay-first routing often will not lift `sk-code-review` past the advisor's `0.1` transitive threshold (`.opencode/skill/skill-advisor/scripts/skill_advisor.py:98-104`). This looks more like a weight calibration problem than a completely missing edge.
5. `sk-deep-review` is mostly accurate on its explicit review-loop ties, but it still lacks a graph connection to the spec-validation system it routinely works alongside. The metadata correctly records `depends_on -> sk-code-review` and `siblings -> sk-deep-research` (`.opencode/skill/sk-deep-review/graph-metadata.json:7-13`). However, its intent signals include `spec folder review` (`.opencode/skill/sk-deep-review/graph-metadata.json:17-18`), while `system-spec-kit` owns spec-folder/validation/context routing (`.opencode/skill/system-spec-kit/SKILL.md:3`, `.opencode/skill/skill-advisor/scripts/skill_advisor.py:919-927`). That coupling is visible in routing behavior but absent from the graph.
6. `cli-claude-code` has accurate peer edges to the other CLI orchestrators, but the graph does not model its review companionship. The frontmatter explicitly includes `code review` in its description (`.opencode/skill/cli-claude-code/SKILL.md:3`), its metadata already covers CLI sibling relations (`.opencode/skill/cli-claude-code/graph-metadata.json:9-12`), and advisor phrase routing co-boosts `cli-claude-code` with `sk-code-review` for review-oriented prompts such as `claude code review` (`.opencode/skill/skill-advisor/scripts/skill_advisor.py:745-750`) and more general `validate` / `opinion` phrases (`.opencode/skill/skill-advisor/scripts/skill_advisor.py:602-604`). This is a plausible missing cross-family edge, but with lower confidence than the `system-spec-kit` gap.

## Missing Edges Identified
- High confidence: add a graph relationship between `system-spec-kit` and `sk-deep-review`. Best current fit looks like `system-spec-kit -> enhances -> sk-deep-review` because spec-folder context and validation increase deep-review usefulness, but the direction should be confirmed before editing metadata.
- Medium confidence: add a graph relationship between `system-spec-kit` and `sk-code-review`. Best current fit looks like `system-spec-kit -> enhances -> sk-code-review` for audit/spec-validation tasks.
- Medium confidence: add a graph relationship between `cli-claude-code` and `sk-code-review`. Best current fit looks like `cli-claude-code -> enhances -> sk-code-review` for cross-AI review/validation flows.
- Not a raw missing edge, but a structural gap: decide whether `mcp-code-mode` should compile `prerequisite_for` edges into runtime adjacency, or whether those relations should instead be duplicated as reverse `depends_on` edges on dependent MCP skills only.
- Not a raw missing edge, but a node-model gap: decide whether Notion and Webflow need dedicated graph nodes/skills so `mcp-code-mode` can represent the full set of capabilities named in frontmatter.
