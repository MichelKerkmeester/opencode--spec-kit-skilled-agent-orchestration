## Findings
- **Hooks exist, but they are not a prompt-injection API.** Copilot CLI exposes `sessionStart`, `userPromptSubmitted`, `preToolUse`, `postToolUse`, `sessionEnd`, and `errorOccurred`, but the hook reference says `sessionStart` output is ignored, `userPromptSubmitted` output is ignored, and `postToolUse`/`errorOccurred` output is ignored; only `preToolUse` denial is currently processed. That means the documented hook surface can observe/log and gate tools, but cannot natively inject a `SessionStart:startup` or `UserPromptSubmit` brief into model context. [SOURCE: https://docs.github.com/en/copilot/reference/hooks-configuration] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/use-hooks]
- **`--agent` / `/agent` selects or delegates to custom agents as subagents, not as an always-on startup overlay for the main session.** The docs describe custom agents as specialized profiles instantiated when assigned, and the creation guide says work is carried out using a subagent with its own context window; `copilot --agent <name> --prompt ...` is an explicit per-run selection, not a hidden lifecycle hook that prepends dynamic context to every main-agent turn. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/about-custom-agents] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/create-custom-agents-for-cli] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features]
- **Copilot CLI does have a native startup-context mechanism, but it is static-file based custom instructions, not dynamic hook/MCP injection.** Docs say custom instructions are loaded at session start from `.github/copilot-instructions.md`, `.github/instructions/**/*.instructions.md`, root `AGENTS.md`, and `$HOME/.copilot/copilot-instructions.md`; changes become active on the next submitted prompt, and `COPILOT_CUSTOM_INSTRUCTIONS_DIRS` adds more instruction search roots. This is the closest native equivalent to startup context, but it is static text on disk unless an external wrapper rewrites files before launch. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions] [SOURCE: https://docs.github.com/en/copilot/reference/custom-instructions-support] [SOURCE: file:bash `copilot help environment` on 2026-04-22]
- **The documented plugin model is packaging, not a new context channel.** Plugins can bundle agents, skills, hooks, MCP configs, and LSP configs, but the plugin docs present them as an install/distribution vehicle for those existing surfaces. Nothing in the plugin docs or CLI help adds a separate per-session/per-prompt payload injection path beyond the bundled hooks/instructions/agents. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/about-cli-plugins] [SOURCE: file:bash `copilot plugin --help` on 2026-04-22]
- **The live local config/env surface shows no hidden arbitrary prompt-payload knob.** On this machine, `~/.copilot/config.json` contains login/banner/trusted folder state, `~/.copilot/mcp-config.json` only defines MCP servers and their server-side env, and `copilot help environment` lists model/provider/home/permissions/instruction-dir variables but nothing for passing arbitrary startup or advisor payload text directly into the model. [SOURCE: file:bash `ls -la ~/.copilot && cat ~/.copilot/config.json && cat ~/.copilot/mcp-config.json && env | grep -iE '^(COPILOT|GITHUB|GH)_'` on 2026-04-22] [SOURCE: file:bash `copilot help environment` on 2026-04-22]

## Questions Answered
- **Q1:** Yes, Copilot CLI has a documented hook API, but it is not equivalent to Claude's prompt-mutating hook surface because `sessionStart` and `userPromptSubmitted` outputs are explicitly ignored. Evidence: hooks how-to + hook reference. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/use-hooks] [SOURCE: https://docs.github.com/en/copilot/reference/hooks-configuration]
- **Q2:** The `/agent` / `--agent` model is a custom-agent/subagent selection mechanism, not a system-prompt startup injector for the main session. Evidence: custom-agent concept + creation docs + customization matrix. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/about-custom-agents] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/create-custom-agents-for-cli] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features]
- **Q5:** Yes, there are documented instruction files Copilot reads automatically (`AGENTS.md`, `.github/copilot-instructions.md`, path-specific instructions, `$HOME/.copilot/copilot-instructions.md`), but no evidence in this pass of a dedicated startup-payload config key beyond those files. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions] [SOURCE: https://docs.github.com/en/copilot/reference/custom-instructions-support] [SOURCE: file:bash `ls -la ~/.copilot && cat ~/.copilot/config.json` on 2026-04-22]
- **Q7:** The documented env surface includes `COPILOT_CUSTOM_INSTRUCTIONS_DIRS`, model/provider/home/permission variables, and auth tokens, but no env var for arbitrary prompt payload injection.
