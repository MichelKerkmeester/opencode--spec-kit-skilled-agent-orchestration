# Iteration 8: Q8 environment propagation source audit
## Focus
This iteration focused on Q8, because Q1 and Q2 were already synthesis-ready after iterations 6 and 7. The goal was to separate Codex hook command environment behavior from the separate shell-tool subprocess environment policy.
## Actions Taken
1. Re-read the official Codex hooks page for the documented command-hook runtime contract, especially stdin, cwd, timeout, matcher, and stdout context semantics: https://developers.openai.com/codex/hooks
2. Re-opened upstream `command_runner.rs` to inspect the actual hook subprocess construction path: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/command_runner.rs
3. Checked upstream `spawn.rs` to contrast hook subprocesses with model-requested shell tool subprocesses, which do explicitly receive an environment map after `env_clear()`: https://github.com/openai/codex/blob/main/codex-rs/core/src/spawn.rs
4. Checked the official Codex configuration reference for `shell_environment_policy.*`, confirming that Codex documents a subprocess environment policy surface but does not state it specifically governs hooks: https://developers.openai.com/codex/config-reference
5. Checked Rust's `std::process::Command` documentation for the default environment inheritance behavior that applies when a `Command` is spawned without `env_clear`, `env`, `envs`, or `env_remove`: https://doc.rust-lang.org/std/process/struct.Command.html
6. Rechecked local read-only evidence: `codex --version` reports `codex-cli 0.122.0`; `~/.codex/config.toml` has no `[shell_environment_policy]`; `~/.codex/hooks.json` still registers only Superset `notify.sh` for `SessionStart`, `UserPromptSubmit`, and `Stop`; no modification was made to `~/.codex/hooks.json`.
## Findings
- Q8: The hook runner path constructs a Tokio `Command`, sets `current_dir(cwd)`, pipes stdin/stdout/stderr, and sets `kill_on_drop(true)`, but in the inspected hook runner source it does not call `env_clear`, `env`, `envs`, or `env_remove`. Primary source: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/command_runner.rs Relevance: source-level evidence that hook commands are not given a sanitized or Codex-constructed environment in this runner path.
- Q8: Rust's `Command::new` default is to inherit the current process environment, and inheritance is disabled only by explicit environment methods such as `env_clear` or `env_remove`. Primary source: https://doc.rust-lang.org/std/process/struct.Command.html Relevance: combined with `command_runner.rs`, this supports the source-level inference that hook commands inherit the Codex process environment.
- Q8: Codex's model-requested shell tool subprocess path is different: `spawn.rs` takes an explicit `env` map, applies network proxy env when needed, calls `cmd.env_clear()`, then calls `cmd.envs(env)`. Primary source: https://github.com/openai/codex/blob/main/codex-rs/core/src/spawn.rs Relevance: prevents incorrectly projecting shell-tool environment filtering onto hooks; the inspected hook runner does not use this spawn helper path.
- Q8: Codex documents `shell_environment_policy.inherit`, `exclude`, `include_only`, `ignore_default_excludes`, and `set` as controls for subprocess environment inheritance and overrides. Primary source: https://developers.openai.com/codex/config-reference Relevance: this policy is real, but the evidence found in this pass ties it to shell-tool subprocess construction, not to hook command construction.
- Q1 collateral: The official hook docs still state that every command hook receives one JSON object on stdin, with common fields including `session_id`, `transcript_path`, `cwd`, `hook_event_name`, and `model`. Primary source: https://developers.openai.com/codex/hooks Relevance: reconfirms the already synthesis-ready stdin contract for `SessionStart` and `UserPromptSubmit`.
- Q2 collateral: The official hook docs still state that `SessionStart` plain stdout is added as extra developer context, and JSON stdout can provide `hookSpecificOutput.additionalContext`, also added as extra developer context. Primary source: https://developers.openai.com/codex/hooks Relevance: reconfirms stdout-to-model-context injection semantics for `SessionStart`.
- Q2 collateral: The official hook docs list the `UserPromptSubmit` event as supporting prompt inspection and blocking, and prior source evidence remains consistent with additive context rather than prompt replacement; no new contradictory source appeared in this pass. Primary source: https://developers.openai.com/codex/hooks Relevance: keeps Q2/Q7 synthesis stable while Q8 is being closed.
