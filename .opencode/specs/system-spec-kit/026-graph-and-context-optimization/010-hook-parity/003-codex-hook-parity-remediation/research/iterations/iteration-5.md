## Findings
- Q3: `UserPromptSubmit` supports the Claude-style blocker pattern. JSON stdout `{"decision":"block","reason":"..."}` marks the hook run as blocked, sets `should_stop=true`, and uses the reason as feedback; exit code `2` also blocks when stderr contains a non-empty reason. Primary sources: https://developers.openai.com/codex/hooks and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/user_prompt_submit.rs
- Q7: `UserPromptSubmit` can decline/block the prompt, but the researched schema/source do not expose a prompt transform surface. The generated output schema allows `continue`, `decision`, `reason`, `stopReason`, `suppressOutput`, `systemMessage`, and `hookSpecificOutput.additionalContext`; it does not include `updatedInput` or a replacement prompt field. Primary sources: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/schema/generated/user-prompt-submit.command.output.schema.json and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/output_parser.rs
- Q3: `Stop` uses `decision:"block"` and exit code `2` differently from `UserPromptSubmit`: they do not reject a user prompt; they create continuation feedback/prompt fragments so Codex keeps going. `continue:false` takes precedence over continuation decisions from other matching `Stop` hooks. Primary sources: https://developers.openai.com/codex/hooks and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/stop.rs
- Q3: `PreToolUse` can block Bash execution with either `hookSpecificOutput.permissionDecision:"deny"` plus `permissionDecisionReason`, legacy `decision:"block"` plus `reason`, or exit code `2` plus stderr. Plain stdout is ignored; JSON-like invalid stdout fails the hook but fails open for tool execution unless a supported block reason was parsed. Primary sources: https://developers.openai.com/codex/hooks and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/pre_tool_use.rs
- Q3: `PostToolUse` does not block a command that already ran. `decision:"block"` records feedback and replaces/feeds back on the tool result path; exit code `2` with stderr surfaces feedback to the model but source tests keep the hook run status completed, not blocked. Primary sources: https://developers.openai.com/codex/hooks and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/post_tool_use.rs
- Q3: Across researched events, arbitrary non-zero exit codes other than event-specific code `2` are treated as hook failures with an error entry like `hook exited with code N`; they are not the supported blocker path. Primary sources: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/user_prompt_submit.rs, https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/stop.rs, https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/pre_tool_use.rs, and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/post_tool_use.rs
- Q5 collateral: Current docs state multiple matching command hooks for the same event are launched concurrently; source discovery assigns `display_order`, and `Stop` aggregation concatenates multiple continuation reasons in declaration/display order after all results return. Primary sources: https://developers.openai.com/codex/hooks, https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/discovery.rs, and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/stop.rs
- Q6 collateral: The generated output schemas' `HookEventNameWire` enum and discovery source both list six configurable events: `PreToolUse`, `PermissionRequest`, `PostToolUse`, `SessionStart`, `UserPromptSubmit`, and `Stop`. Primary sources: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/schema/generated/user-prompt-submit.command.output.schema.json and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/discovery.rs
