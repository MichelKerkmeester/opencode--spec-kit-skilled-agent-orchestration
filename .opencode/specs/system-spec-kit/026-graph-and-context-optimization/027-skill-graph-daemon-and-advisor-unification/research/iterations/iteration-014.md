## Analysis
Adopt repo-level corpus statistics now. The local extractor is already framed as a TF-IDF/N-gram hybrid, but the code only has within-document term-frequency scoring plus length bonuses. That is enough to rank repeated phrases inside one skill, but it cannot know that terms like `workflow`, `code`, `mcp`, or `cli` are common across the skill corpus. Standard TF-IDF controls solve exactly that problem by making IDF depend on total document count and document frequency, with `max_df`/`min_df` providing corpus-specific common-term and rare-term filters.

The implementation should stay TypeScript-first and index-time, not prompt-time. The current corpus is only 21 production skill metadata files, the skill graph already scans all metadata on startup, and the SQLite graph already stores content hashes plus a key/value metadata table. A full corpus-stat recompute whenever the skill graph observes a changed skill metadata file is simpler and safer than maintaining a fragile incremental DF table. The prompt-blocking advisor should only read the latest `corpusStatsGeneration` and precomputed per-skill derived weights.

The baseline should re-compute on any graph-generation change that can affect candidate terms: added/removed skills, changed `graph-metadata.json`, or a changed derived-input manifest from B5. Because IDF shifts globally, even one changed skill can alter scores for other skills; with the present corpus size, recomputing the whole baseline is cheap and avoids stale cross-skill weights. The practical cadence should be startup scan plus debounced index-time regeneration after relevant file changes, with no per-prompt recomputation.

Store the corpus baseline as graph-level metadata, not in each `SKILL.md`: `document_count`, `vocabulary_hash`, `source_fingerprint`, `idf_version`, generated timestamp, normalization settings, top common terms, and optional per-term DF/IDF rows if the scorer needs explainability. Per-skill derived metadata can then reference the corpus generation that produced its derived terms. This keeps B3's hybrid storage, B4's lower-trust derived lane, and B5's freshness model coherent while leaving later C4/C7 work to tune exact weights against the 200-prompt corpus.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Maintain repo-level DF/IDF stats as an index-time skill-graph baseline, recomputed on startup and on debounced graph/derived-input changes, not during prompt routing. The corpus is currently small and the existing scan/hash model makes full recomputation simpler and safer than incremental DF maintenance.
