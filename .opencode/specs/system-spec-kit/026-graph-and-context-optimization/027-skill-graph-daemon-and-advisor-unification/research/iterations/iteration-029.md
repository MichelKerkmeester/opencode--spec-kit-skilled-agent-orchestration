## Analysis
The install/bootstrap answer should be adopted now, but as integration cleanup inside the existing `spec_kit_memory` MCP server, not as a new MCP declaration. `opencode.json` already starts a single system-spec-kit MCP entry through `context-server.js`, and the MCP server already dispatches `skill_graph_*` tools through the same tool registry. The install guide and installer, however, still present the system as memory-first: canonical path tables omit `skill-graph.sqlite`, expected tool lists omit the live `skill_graph_*` tools, and the setup script only configures the `spec_kit_memory` entry.

The runtime bootstrap is closer to the desired Phase 027 shape than the docs suggest. Context-server startup initializes the skill-graph database, logs fresh/load/reindex states, runs a background `startupSkillGraphScan()`, and exposes a manual `skill_graph_scan` path with a workspace escape guard. That means D6 does not need to invent an install-time daemon bootstrap from scratch; it should make the installer and guides verify the already-shipped startup behavior and document the manual fallback when the MCP server is not running.

The advisor install/uninstall flow is currently split across a separate Skill Advisor setup guide, hook reference, and OpenCode plugin bridge. That split creates operator drift: the central install hub lists a Skill Advisor setup guide via a broken symlink, the hook reference documents Claude/Gemini/Copilot/Codex but not OpenCode `opencode.json` registration, and rollback is described as an env var rather than a complete uninstall matrix. Consolidation should fix the symlink/path, add advisor hook/plugin verification to the system-spec-kit install flow, and make uninstall/rollback explicit: disable prompt-time advice with `SPECKIT_SKILL_ADVISOR_HOOK_DISABLED=1`, unregister runtime hook/plugin surfaces when removing the advisor, but keep the core `spec_kit_memory` MCP entry unless the user is uninstalling the whole MCP server.

D6 therefore reinforces D4-D5: the advisor should live as a native subsystem of the system-spec-kit MCP server, with install/bootstrap owned there. The important boundary is that `skill_advisor.py` remains a compatibility fallback until D7 decides the backward-compat matrix; D6 should only update bootstrap docs/scripts to create or verify generation state, skill graph SQLite freshness, hook/plugin registration, and prompt-safe status checks.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Adopt install/bootstrap consolidation now because runtime skill-graph bootstrap already exists inside `context-server.js`, but installer/docs lag behind the architecture. The safe scope is guide/symlink/verification updates plus advisor hook/plugin rollback documentation, not a new MCP server registration.
