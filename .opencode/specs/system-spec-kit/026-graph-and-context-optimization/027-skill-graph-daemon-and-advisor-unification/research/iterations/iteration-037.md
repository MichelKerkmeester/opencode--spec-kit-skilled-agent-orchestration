## Analysis
The roadmap should stage the work by trust boundary, not by track label. The first packet must establish graph/freshness correctness because A2, A3, A4, and A8 make the daemon's watched scope, hash-aware SQLite index, post-commit publication, and fail-open recovery the durable base for every later advisor feature (`iteration-002.md:43`-`iteration-002.md:54`; `iteration-003.md:37`-`iteration-003.md:46`; `iteration-004.md:43`-`iteration-004.md:55`; `iteration-008.md:46`-`iteration-008.md:53`). The second packet should add derived metadata and precision safeguards before the native scorer consumes generated triggers, because B3 stores generated signals in `graph-metadata.json.derived`, B4/B7 require lower-trust evidence separation and anti-stuffing gates, and X3 says derived keywords must not enter the direct `(signal)` boost lane (`iteration-011.md:29`-`iteration-011.md:41`; `iteration-012.md:32`-`iteration-012.md:47`; `iteration-015.md:44`-`iteration-015.md:54`; `iteration-034.md:37`-`iteration-034.md:50`).

The third and fourth packets can then move runtime behavior into native MCP. C2, C3, C4, and C6 show the adopt-now runtime shape: an advisor-specific skill projection, bounded `skill_edges` traversal, centralized analytical fusion, deterministic hot-path lexical/SQL work, and semantic or learned ranking only as shadow/prototype (`iteration-017.md:37`-`iteration-017.md:52`; `iteration-018.md:32`-`iteration-018.md:47`; `iteration-019.md:45`-`iteration-019.md:60`; `iteration-021.md:39`-`iteration-021.md:54`). D2 and D3 then give that runtime its concrete module/tool surface under `lib/skill-advisor`, `handlers/skill-advisor`, and `tools/advisor-tools.ts`, while D4 keeps the Python process only as a compatibility bridge until parity (`iteration-025.md:55`-`iteration-025.md:70`; `iteration-026.md:44`-`iteration-026.md:93`; `iteration-027.md:48`-`iteration-027.md:63`).
