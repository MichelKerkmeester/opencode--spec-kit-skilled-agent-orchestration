## Analysis
C and D are coherent because Track C repeatedly defines reuse as adapter-level borrowing from memory MCP rather than physical reuse of memory tables or direct memory-causal APIs. C1 adopts memory retrieval primitives as a pattern, C2 requires an advisor-specific skill projection instead of pushing skills into `memory_index`, C3 keeps causal-style traversal over `skill_edges`, C4 copies the centralized analytical fusion model, and C6/C8 keep prompt-time embedding or learned scoring out of the adopt-now path until stricter gates pass. Those decisions already prevent C from demanding that advisor matching live inside `lib/search` or memory-specific handlers.

Track D gives the same design a concrete module home. D2's target layout keeps graph authority in `lib/skill-graph/`, puts advisor runtime/scoring/projection/signals/conflicts/health under `lib/skill-advisor/`, adds `handlers/skill-advisor/`, and exposes public access through a new `tools/advisor-tools.ts` family. D3's `advisor_recommend`, `advisor_status`, and `advisor_validate` surface is additive beside the existing `skill_graph_*` tools; it does not replace graph scan/query/status/validate, and it does not require C's scorer to become a memory-search MCP tool.

The live code supports that split. `lib/skill-graph` already stores typed skill nodes and edges, indexes source/target/type lookups, performs transactional index updates, and exposes bounded traversal. `lib/skill-advisor` already owns hook-time policy, freshness, prompt cache, rendering, diagnostics, metrics, fail-open behavior, and the typed advisor result envelope. The only current gap is implementation, not architecture: schemas and dispatchers currently register `skill_graph_*` but no `advisor_*` family, and `buildSkillAdvisorBrief()` still calls the Python subprocess. D3/D4 explicitly cover those as additive MCP and native-runtime migration work.

Therefore X4 should not be marked as a conflict. The correct placement is: graph/index/traversal primitives remain in `lib/skill-graph` and existing `skill_graph_*` MCP tools; advisor projection, scoring fusion, ambiguity handling, status, validation, and compatibility wrappers live in `lib/skill-advisor`, `handlers/skill-advisor`, and `tools/advisor-tools`; memory search/fusion remains a reference implementation and optional shadow/prototype source, not the physical home for skill advisor routing.
