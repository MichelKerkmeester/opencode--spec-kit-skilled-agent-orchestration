## Analysis
The current ambiguity renderer is intentionally narrow. It first filters to passing recommendations, then declares ambiguity only when the top two passing recommendations have confidence values within `0.05`; it does not inspect raw score margin, uncertainty delta, reasons, graph edge type, or causal provenance at render time (`render.ts:72`-`render.ts:82`). The model-visible ambiguous string is also gated by token budget: the top-two form appears only when `metrics.tokenCap` or render options exceed the default 80-token cap, otherwise the same recommendation set becomes a single "use" brief (`render.ts:111`-`render.ts:136`; `advisor-renderer.vitest.ts:41`-`advisor-renderer.vitest.ts:57`).

Most ambiguity intelligence already lives upstream of `render.ts`. The Python advisor computes uncertainty from evidence count, missing intent, and ambiguous multi-skill matches (`skill_advisor.py:2061`-`skill_advisor.py:2108`), then calibrates confidence with margin and ambiguity penalties (`skill_advisor.py:2139`-`skill_advisor.py:2165`). It also has targeted disambiguation for known wording collisions such as deep-research/deep-review versus code-review (`skill_advisor.py:2200`-`skill_advisor.py:2302`). That division is healthy because `render.ts` is explicitly a prompt-boundary guard that ignores reasons, descriptions, stdout/stderr, and prompt text (`render.ts:93`-`render.ts:99`).

Causal-style skill edges would improve ambiguity handling, but they should improve the scorer and uncertainty model before rendering, not turn the renderer into a graph reasoner. Positive edges already support bounded analytical boosts with a ghost-candidate guard: `enhances`, `siblings`, and `depends_on` only boost targets that already had prompt evidence (`skill_advisor.py:552`-`skill_advisor.py:598`). Conflict edges are even more directly relevant to ambiguity because the runtime can add `0.15` uncertainty when two mutually conflicting skills both pass threshold (`skill_advisor.py:666`-`skill_advisor.py:708`), and the skill graph already has indexed `conflicts_with` storage and bidirectional conflict queries (`skill-graph-db.ts:20`-`skill-graph-db.ts:25`; `skill-graph-queries.ts:144`-`skill-graph-queries.ts:157`).

So the best Phase 027 answer is adopt-now with a strict boundary. C3 already says to use skill-owned `skill_edges` instead of memory `causal_edges` for skills, and C4 already says graph/causal contributions should be deterministic, capped, and explainable (`iteration-018.md:32`-`iteration-018.md:39`; `iteration-019.md:45`-`iteration-019.md:57`). The renderer can keep its top-two `<= 0.05` display rule, while causal-style edges adjust confidence, uncertainty, and pass/fail status upstream; once conflicts are populated, conflicting top-two passes should more often fail the uncertainty gate or render as explicit ambiguity rather than a single decisive recommendation.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Adopt causal-style skill-edge ambiguity handling now, but only upstream in scoring and uncertainty calibration; keep `render.ts` as a deterministic, sanitized top-two display boundary.
