## Analysis
The update boundary should be the SQLite graph mutation, not the watcher event and not a combined filesystem-plus-database transaction. The current skill-graph indexer already parses outside the critical section, then deletes missing nodes, upserts changed nodes, and replaces changed outgoing edges inside one `better-sqlite3` transaction. That is the right unit of atomicity for Phase 027 because status and query readers can trust the committed `skill_nodes`/`skill_edges` set, while failed scans leave the prior graph intact rather than publishing a half-updated topology.

Phase 020's freshness model is still useful, but as a commit signal and cache invalidation layer rather than as the transaction itself. Freshness already derives a source signature from source files plus graph artifacts and combines it with a generation counter for caching. Phase 027 should reuse that pattern by bumping or otherwise refreshing advisor generation only after `indexSkillMetadata()` succeeds, then letting non-live freshness and signature mismatch invalidate stale advisor output. The existing `incrementAdvisorGeneration()` API is present, but search found it used only by tests and freshness utilities today, so wiring it into successful skill-graph updates is part of the adopt-now work.

The repo's atomic write helpers are a warning against pretending filesystem and SQLite updates can be made fully atomic together. The transaction manager explicitly documents the DB-committed/rename-failed window and uses pending-file recovery to reconcile that class of issue later. For the skill graph, the daemon does not need to write the watched `graph-metadata.json` source files, so it can avoid the hardest cross-resource case: keep source writes owned by editors or generators, debounce/await stable files, commit DB graph updates in one SQLite transaction, then publish freshness/generation metadata after commit.

Concurrency should remain intentionally simple for the first Phase 027 slice. SQLite WAL supports concurrent readers with a single writer, and the current daemon runner already prevents overlapping in-process scans and queues one follow-up rescan. A later A8 failure-mode pass can decide whether to add a database-backed scan lease like memory indexing, but A4 does not need a new two-phase transaction model or a separate prototype to proceed.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Adopt the existing single-SQLite-transaction graph update as the authoritative consistency boundary, then publish freshness/generation/cache invalidation only after successful commit. Reject a cross-resource two-phase transaction for Phase 027 because current evidence shows no need to atomically write source files and SQLite rows together.
