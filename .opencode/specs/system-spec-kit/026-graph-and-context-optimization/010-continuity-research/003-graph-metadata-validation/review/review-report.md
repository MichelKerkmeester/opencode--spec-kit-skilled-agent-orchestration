# Root Review Report: 019-research-graph-metadata-validation

## 1. Overview

Ten iterations covered parser correctness, corpus verification, doc accuracy, backfill behavior, and packet traceability for the root `019-research-graph-metadata-validation` packet and sub-phases `001` through `005`. ## 4. Recommended Remediation

- Extend `keepKeyFile()` coverage to reject `cd ... && ...` and similar command-shaped strings, then verify that `deriveEntities()` no longer inherits those values from `key_files`.
- Update `graph-metadata-backfill.vitest.ts` and the `004-normalize-legacy-files` packet docs so they match the inclusive-default backfill contract that phase 005 now documents.
- Repair the root 019 packet and phases 001-004 to a clean strict-validation pass, or explicitly scope/defer the remaining checklist items instead of leaving them open under “complete” delivery claims.
- Tighten canonical packet-doc preference so local canonical docs win over unrelated external packet references that merely share the same filename suffix.

## 5. Residual Risk

The review did not find a security or data-loss defect, but the current graph-metadata surface is still not trustworthy enough for release-quality traceability. The parser is producing noisy and occasionally misattributed metadata, the backfill verification lane is stale, and the 019 packet tree cannot yet serve as clean audit evidence for the shipped work.
