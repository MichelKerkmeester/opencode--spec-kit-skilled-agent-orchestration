# Iteration 10: smallest safe v1 slices

## Focus
This iteration stress-tests the iteration-9 sequence against actual Public implementation slices. The goal is to define the smallest safe v1 for each `adopt now` lane, identify the exact cross-session analytics gap left by `024-compact-code-graph/003-stop-hook-tracking`, and choose the first concrete follow-on packet by weighting prerequisites, blast radius, and reversibility. ## Findings
### 1. FTS capability cascade is still the cleanest first packet

Iteration 9's "FTS first" ordering survives the v1 stress test. `023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard` already hardened multi-word FTS correctness, descendant scope filtering, runtime observability, and hybrid search behavior, so the remaining adoption work is a narrow backend-capability contract rather than a new retrieval system. That gives this lane the best prerequisite score, the broadest unblock value, and the highest rollback safety. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md] [SOURCE: .opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard/implementation-summary.md]

#### Smallest Safe v1: FTS capability cascade

- **Scope:** Add a runtime backend-selection contract to the existing search lane only: `FTS5 + BM25` when available, `FTS4` when `FTS5` is unavailable, and `LIKE` as the final fallback. Do not import branch-ranked recall, transcript hydration, or schema changes. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-008.md] [SOURCE: .opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard/implementation-summary.md]
- **Inputs:** Existing search query text, current spec-folder scope filters, SQLite compile capability detection, and the current search pipeline metadata surface. [SOURCE: .opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard/implementation-summary.md]
- **Outputs:** The same logical keyword-search result contract at every backend tier, plus explicit backend metadata such as `keywordBackend: "fts5" | "fts4" | "like"` so degraded operation is observable instead of silent. [SOURCE: .opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard/implementation-summary.md]
- **Dependencies:** Only `023/013` as the existing correctness and observability substrate. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md]
- **Smallest deliverable:** One packet that adds capability detection, backend selection, forced-degrade test coverage, and user-visible diagnostics without changing the public query surface. [SOURCE: .opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard/implementation-summary.md]
- **Risk:** `LIKE` fallback will be noisier and less rank-stable than FTS-backed paths, so degradation must be explicit. [SOURCE: .opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard/implementation-summary.md]
- **Exit test:** A representative multi-word search returns usable results in all three forced modes, descendant scoping still works, and diagnostics show which backend executed. [SOURCE: .opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard/implementation-summary.md]

### 2. SessionStart fast path is safe only as a cached-summary augmentation

Iteration 8 and iteration 9 were right to keep this lane inside Public's existing startup and resume surfaces. `024/002` already owns source-aware routing and `024/003` already extracts a short session summary at Stop time, so the safe v1 is a Stop-time write plus SessionStart read of a curated summary payload. It must not replace `session_bootstrap()` / `session_resume()` or import Claude's recency heuristic. #### Smallest Safe v1: SessionStart precomputed-summary fast path

- **Scope:** Persist one bounded `precomputedSummary` object in hook state at Stop time and let `resume` plus `compact` flows read it before falling back to the normal prime flow. Leave `startup` and `clear` behavior unchanged. [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/002-session-start-hook/implementation-summary.md] [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md]
- **Inputs:** Existing `lastSpecFolder`, the Stop-hook summary extracted from the last assistant message, token-pressure metadata, and a freshness timestamp. ### 3. Auditor/discoverer split is safe if v1 stays read-first

The discoverer/auditor split is implementable now because `022/008` already unified the signal substrate and deliberately kept shared trigger parity intact. The safe cut is not "auto-consolidate memory"; it is "discover candidates separately from verifying proposed writes," with both outputs structured and reviewable before execution. [SOURCE: .opencode/specs/system-spec-kit/022-hybrid-rag-fusion/009-perfect-session-capturing/008-signal-extraction/implementation-summary.md] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-008.md] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md]

#### Smallest Safe v1: extract-learnings auditor/discoverer split

- **Scope:** Add two explicit roles to the existing consolidation flow: a discoverer that emits ranked candidate learnings and an auditor that only validates proposed changes for `STALE`, `CONTRADICT`, `MERGE`, or `DATE_FIX`. Keep execution manual or separately approved. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-008.md] [SOURCE: .opencode/specs/system-spec-kit/022-hybrid-rag-fusion/009-perfect-session-capturing/008-signal-extraction/implementation-summary.md]
- **Inputs:** Canonical script-side extracted signals, existing trigger-parity baselines, current memory candidates, and any proposed update text. [SOURCE: .opencode/specs/system-spec-kit/022-hybrid-rag-fusion/009-perfect-session-capturing/008-signal-extraction/implementation-summary.md]
- **Outputs:** `discovererCandidates` ranked 3-7 items and `auditorVerdicts` constrained to verification labels. No direct memory writes in v1. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-008.md]

- **Smallest deliverable:** One structured discover pass plus one structured audit pass wired into an existing review flow, without changing storage semantics. ### 4. The ingestion lane is where `024/003` only partially unlocks the next step

This is the exact place where iteration 9's sequence needs a safety refinement. `024/003` already gives Public incremental transcript parsing, per-session token snapshots, summary extraction, and token-pressure feedback, but it also documents three blockers for true cross-session analytics: snapshots live in hook-state JSON rather than queryable tables, counts are lower-bound estimates, and cost estimation omits cache-read/cache-write pricing. That means `024/003` unlocks collection, not analytics-grade normalization. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md] [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md]

#### Smallest Safe v1: get-token-insights ingestion

- **Scope:** Introduce a read-only batch ingestor that converts existing session artifacts into normalized analytics tables for cross-session queries. Keep the Stop hook as the producer of raw session facts; do not rewrite `024/003`. [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md]
- **Inputs:** Hook-state JSON produced by `024/003`, transcript paths already captured there, session metadata, and pricing rules extended to include cache-read and cache-write tiers. [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/research.md]
- **Outputs:** A minimum normalized schema with one session-level table and one hook-level or turn-summary table, sufficient to query cost, cache ratio, model mix, and hook overhead across sessions. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/research.md] [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md]
- **Dependencies:** `024/003` as producer. No dependency on dashboard rendering. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md]
- **Smallest deliverable:** A narrowly scoped follow-on packet under the `024-compact-code-graph` family that adds an offline ingestor plus normalized analytics tables and leaves the Stop hook intact. The memory registry still treats `024-compact-code-graph` as an active packet family with prior closure work, which supports keeping this as a packet-local extension instead of a new subsystem family. [SOURCE: /Users/michelkerkmeester/.codex/memories/MEMORY.md] [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md]

- **Exit test:** Two or more completed sessions can be ingested into normalized tables, queried without scanning hook-state files directly, and show consistent session cost plus cache metrics across runs. ### 5. The exact gap is "analytics-grade cross-session storage," not "missing more Stop-hook collection"

The precise gap left by `024/003` is that it writes session-local JSON that is sufficient for current-session pressure feedback but insufficient for analytics-style cross-session aggregation. The smallest spec that closes that gap is therefore not a rewrite of the Stop hook. It is a narrow follow-on spec that adds a normalized read model over existing session artifacts and extends pricing normalization to cache tiers. [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/research.md]

### 6. Dashboard data contracts are not safe as a standalone v1 yet

The dashboard-contract lane is only safe if it is explicitly scoped as a post-ingestion contract slice. Iteration 9 correctly placed it after ingestion, and the v1 stress test shows why: without normalized analytics tables, Public would be freezing KPI, trend, and recommendation fields before deciding which metrics are durable in its own pipeline. That makes the standalone contract lane premature today. Reclassify it to `prototype later` unless it is bundled as a thin, derived-contract add-on after ingestion lands. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-008.md] [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md]

#### Smallest Safe v1: get-token-insights dashboard data contracts

- **Scope:** Expose only derived metrics that are directly backed by the normalized ingestion tables: `kpis`, `cost_by_project`, `project_spend`, `model_split`, `skill_usage`, `agent_delegation`, `hook_performance`, `findings`, and `recommendations`. Exclude HTML, charts, and speculative diagnostics. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/research.md]
- **Inputs:** Only the normalized analytics tables from the ingestion lane. No direct reads from hook-state JSON. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md]
- **Outputs:** A versioned JSON contract document or API payload definition for downstream renderers. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/research.md]
- **Dependencies:** Ingestion must land first. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md]
- **Smallest deliverable:** One contract-focused follow-on after ingestion, not before. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md]
- **Risk:** If shipped before ingestion, the contract will encode unstable calculations and force later compatibility debt. [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md]
- **Exit test:** Every exported field maps to a normalized source column or deterministic derived metric, and no field requires direct hook-state parsing at read time. ### 7. First follow-on packet recommendation: the FTS capability cascade

Weighted across the three requested criteria, the first concrete follow-on packet in Public should be the FTS capability cascade lane.

- **Prerequisites already satisfied:** Highest score. `023/013` already shipped the relevant correctness and observability hardening, so the remaining work is a narrow compatibility layer. [SOURCE: .opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard/implementation-summary.md]
- **Blast radius:** Highest practical unblock value. Iteration 9 already concluded that recall should move before SessionStart injection, because startup augmentation is only meaningfully improved if it can point into a stable retrieval surface. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/iterations/iteration-009.md]
- **Reversibility:** Best rollback profile. Backend detection and fallback metadata can be feature-flagged or removed without touching hook-state schemas, startup flows, or analytics storage. [SOURCE: .opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/013-fts5-fix-and-search-dashboard/implementation-summary.md]

The ingestion lane ranks second because it unlocks more downstream reporting work than any other remaining lane, but its normalized-table work is less reversible than the FTS contract and it still needs the gap-closing spec above to become analytics-safe.
