# Iteration 7: memory hierarchy comparison

## Focus
This iteration compares Claudest's documented memory hierarchy against `Code_Environment/Public`'s current Spec Kit Memory model. The goal is to separate true complements from duplicate storage layers and make a concrete adoption recommendation for Public.

## Findings
- Claudest treats memory as a layered file hierarchy that is partly always loaded: `extract-learnings` defines `~/.claude/CLAUDE.md`, `<repo>/CLAUDE.md`, `memory/MEMORY.md`, and `memory/*.md` topic files as distinct placement targets, with L0-L2 loaded every session and L3 on demand (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/skills/extract-learnings/SKILL.md`; `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/README.md`). Public already has repo-wide instruction loading in [`CLAUDE.md`](/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/CLAUDE.md) plus auto-surfaced constitutional and trigger-driven memory retrieval in [`.opencode/skill/system-spec-kit/SKILL.md`](/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skill/system-spec-kit/SKILL.md), so adding a new always-loaded repo `MEMORY.md` layer would mostly duplicate existing instruction and recovery surfaces. Recommendation: `(reject)`.
- Claudest's `MEMORY.md` is an agent-managed "working knowledge" note that `extract-learnings` reads, edits, and trims, with detailed topic files used only when the summary becomes too long (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/skills/extract-learnings/SKILL.md`). Public does not have a top-level [`MEMORY.md`](/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/MEMORY.md) at all; instead it saves spec-folder-scoped context through [`generate-context.js`](/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skill/system-spec-kit/scripts/dist/memory/generate-context.js) and indexes that data for `memory_context()` / `memory_search()` retrieval via [`.opencode/skill/system-spec-kit/SKILL.md`](/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skill/system-spec-kit/SKILL.md). These are complementary in function only if Public wants a human-curated, repo-level summary document, but redundant as memory infrastructure because Spec Kit already stores and retrieves the same class of project knowledge with better scoping. Recommendation: `(prototype later)` only as a derived summary view, not as a second source of truth.
- Claudest's strongest transferable idea is not the extra file layer but the placement rubric: the README and `extract-learnings` skill explicitly classify whether a learning belongs in global instructions, repo instructions, concise working memory, or deeper topic files, and they require explicit approval before writing (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/README.md`; `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/skills/extract-learnings/SKILL.md`). Public already has analogous destinations spread across [`CLAUDE.md`](/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/CLAUDE.md), spec folders, spec-folder memory saves, and tiered MCP memory indexing in [`.opencode/skill/system-spec-kit/SKILL.md`](/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skill/system-spec-kit/SKILL.md); what it lacks is a compact "where should this learning live?" rule set. Recommendation: `(adopt now)` as a placement/consolidation heuristic mapped onto existing Public surfaces rather than by importing Claudest's file hierarchy.
- Claudest's recall system is conversation-first: the plugin README and top-level external `CLAUDE.md` describe SQLite-backed session sync, startup context injection, and skill-driven search over prior conversations, with archival memory as a later distillation step (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/README.md`; `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/CLAUDE.md`). Public's Spec Kit Memory is already a richer retrieval stack with five-channel search, intent-aware routing, importance tiers, trigger matching, and `/spec_kit:resume` recovery in [`.opencode/skill/system-spec-kit/SKILL.md`](/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/.opencode/skill/system-spec-kit/SKILL.md) and [`CLAUDE.md`](/Users/michelkerkmeester/MEGA/Development/Code_Environment/Public/CLAUDE.md). That makes Claudest's hierarchy complementary as a curation pattern, but redundant as a system replacement. Recommendation: `(reject)` replacing Public's model; keep Public's retrieval engine and only borrow curation rules if needed.

## Ruled Out
- Treating Claudest's hierarchy as a drop-in replacement for Public's memory stack. Public already has stronger retrieval, richer scoping, and explicit spec-folder context preservation.
- Treating a repo-root `MEMORY.md` as necessary for Public. The current model works without one because retrieval is MCP-backed and spec-folder-scoped rather than file-rooted.
