- **Goal statement:** Ship a narrow backend-capability contract inside Public's existing memory-search path so lexical search can degrade predictably across SQLite builds while preserving one logical search API and explicit diagnostics. This remains the first recommended follow-on packet because iteration 10 already narrowed the safe v1 to backend selection plus observability, not a new recall subsystem. - **Claudest capability detection contract, as implemented today:** `detect_fts_support()` probes `PRAGMA compile_options`, returns `"fts5"` when `ENABLE_FTS5` is present, falls back to `"fts4"` when either `ENABLE_FTS4` or `ENABLE_FTS3` is present, and returns `None` otherwise. This is capability-probe driven, not retry-through-exception driven. - **Claudest fallback ordering and query rewriting, as implemented today:** `search_sessions()` splits the raw query on whitespace, sanitizes each token with `sanitize_fts_term()`, drops empties, wraps surviving terms in quotes, and joins them with `OR` for `MATCH`. The `"fts5"` branch runs `ORDER BY bm25(branches_fts)`, the `"fts4"` branch reuses `MATCH` but orders by `b.ended_at DESC`, and the no-FTS path rewrites to `b.aggregated_content LIKE ?` clauses joined with `AND`, each parameterized as `%term%`, also ordered by `b.ended_at DESC`. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/skills/recall-conversations/scripts/search_conversations.py:22-106]
- **Error-class observation from the source:** Claudest does not currently hop tiers on `sqlite3.OperationalError` inside `search_sessions()`. The only probe failure it explicitly swallows is the generic `Exception` around `PRAGMA compile_options`; after that, the selected query path executes directly. That means Public's packet should treat exception-driven degrade as an extra hardening layer, not as something already proven by the Claudest source. - **Packet-ready Public contract:** Keep Public's current lexical surface but add a three-tier backend selector: `fts5_bm25` when `memory_fts` exists and the FTS5 path succeeds, `fts4_match` when an FTS virtual table exists but FTS5/BM25 is unavailable, and `like_scan` as the last-resort SQL fallback. Recommended probe sequence: first inspect compile capability if Public exposes it; otherwise table-probe `memory_fts`; then treat runtime query failures such as `no such module: fts5`, `unable to use function bm25`, or missing virtual-table errors as forced-degrade signals into the next tier. ### Brief B: Normalized analytics tables
- **Goal statement:** `024/003` already ships the producer side of session-end analytics collection: an async Stop hook, incremental JSONL transcript parsing, hook-state token snapshots, summary extraction, and token-pressure feedback. What is still missing is an analytics-grade cross-session read model, because those facts live in hook-state JSON and current cost estimates exclude cache read and cache write pricing. [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md:69-85] [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md:134-138] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/research.md:577-580]
- **Non-goals:** Do not rewrite `hooks/claude/session-stop.ts`, do not move the existing producer from hook-state JSON into a new live-write SQLite table, and do not freeze dashboard contracts before normalized ingestion lands. [SOURCE: .opencode/specs/system-spec-kit/024-compact-code-graph/003-stop-hook-tracking/implementation-summary.md:103-110] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/research/research.md:577-580]
- **Schema sketch for the follow-on packet:** Add four additive tables and keep the producer unchanged. `sessions` should hold one row per replayed session with stable session id, transcript path, project/spec affinity, first/last seen offsets, token totals, lower-bound cost totals, parser version, and replay source hash. `turns` should hold replayed per-message or per-turn facts keyed by session id plus ordinal, including role, timestamp, model, token counts, cache-read/cache-write counts when derivable, and transcript offsets for idempotent replays. `model_pricing_versioned` should hold model name, effective window, input/output/cache-read/cache-write unit prices, and pricing source version so historical sessions can be re-valued without mutating old facts. `cache_tier_normalized` should hold the normalized cache-cost breakout keyed to a session and optional turn, so rollups can separate uncached input, cache write, and cache read instead of collapsing them into one estimate. - **Producer/consumer separation:** Keep `024/003` as the unchanged writer of raw session facts into hook-state JSON plus transcript offsets; add a new analytics roll-up reader that replays hook-state plus the referenced JSONL transcripts into the normalized tables offline or on demand. This preserves the exact simplification decision `024/003` documented while still closing the cross-session query gap. Minimum checks: replay the same session twice and confirm one logical session row plus deterministic turn counts; replay two or more completed sessions and confirm cross-session cost totals no longer require direct hook-state reads; compare replayed totals against the existing hook-state lower-bound numbers so metric drift is explicit rather than silent. Pricing policy should be versioned, not in-place edited, so historical records stay reproducible even if pricing changes later.
