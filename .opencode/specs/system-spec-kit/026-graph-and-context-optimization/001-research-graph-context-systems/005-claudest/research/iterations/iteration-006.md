# Iteration 6: dashboard contract + borrowable sections

## Focus
This iteration treated `dashboard.html` as a payload contract, not as a UI artifact. The goal was to enumerate every surfaced section, map each section to the exact JSON fields it consumes, and identify the smallest set of high-signal dashboard slices that `Code_Environment/Public` should borrow without inheriting the full HTML, CSS, Chart.js, and neon presentation bundle.

## Findings
- `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/skills/get-token-insights/templates/dashboard.html` and `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/skills/get-token-insights/scripts/ingest_token_data.py` show that the dashboard is a single-file embedded artifact, not server-rendered and not JSON-fetch-rendered: `deploy_dashboard()` replaces `/* __INLINE_DATA_PLACEHOLDER__ */` with `const _INLINE_DATA = {json}` and boot reads `_INLINE_DATA` directly (`1936-1944`, `1290-1303`). The header, ticker, and KPI bands consume `generated_at`, `total_sessions`, `date_range.earliest`, `date_range.latest`, and `kpis.{total_sessions,total_turns,total_output_tokens,global_cache_ratio,cache_cliffs,max_token_stops,bash_antipatterns,tool_error_rate,total_cost_usd}`; the ticker only mirrors a subset of those KPI values. For `Code_Environment/Public`, the borrowable contract is the top summary object plus KPI fields, while the scanline overlay, marquee ticker, neon theme, and font/CDN setup are presentation-only chrome. Recommendation: borrow the summary contract now, reject the page chrome. Recommendation: (adopt now)
- `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/skills/get-token-insights/templates/dashboard.html` surfaces an "Overview" plus "Project Intelligence" catalog that already maps well to Public-native reporting: "Sessions by Day" uses `sessions_by_day[].{date,session_count}`; "Token Composition by Day" extends the same rows with `{output_tokens,input_tokens,cache_read,cache_creation}`; "Top Tools" uses `top_tools[].{tool,count}`; "Model Cost Split" uses `model_split[].{model,input_tokens,output_tokens,thinking_tokens,cost_usd}`; "Cost by Day" uses `cost_by_day[].{date,cost_usd}`; "Cost by Project" uses `cost_by_project[].{project,cost_usd}`; "Per-Project Token Spend" uses `project_spend[].{project,input_tokens,cache_creation}`; and "Project Tool Profiles" uses `project_tool_profile[].{project,tools}` where `tools` is a per-project map keyed by tool name (`593-710`, `932-977`, `1379-1418`). The highest-value borrow here for `Code_Environment/Public` is not the full chart deck, but three compact contracts: daily totals, per-project spend, and per-project tool profile. Recommendation: borrow `cost_by_project`, `project_spend`, and `project_tool_profile` first; keep model/day charting as a later prototype. Recommendation: (adopt now)
- `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/skills/get-token-insights/templates/dashboard.html` organizes the diagnostic middle of the page into "Context Management", "Tool Efficiency", and "Behavioral Patterns". The section contracts are explicit: `cache_trajectory[].{project,session_id,turns[]}` where each turn is `{turn,ratio}`; `ephem_split[].{project,ephem_5m,ephem_1h}`; `bash_antipatterns[].{project,antipatterns,total_bash}`; `tool_errors_by_tool[].{tool,errors,total}`; `redundant_reads[].{file,count}`; `edit_retries[].{project,retries}`; `agent_cost[].{project,parent_cost,agent_cost}`; `turn_complexity.{minimal,light,medium,heavy,runaway}`; `thinking_in_complexity.{minimal,light,medium,heavy,runaway}`; `response_time_dist.{under_30s,30s_2m,2m_5m,5m_15m,over_15m}`; and `hook_overhead[].{project,hook_ms}` (`735-954`). For `Code_Environment/Public`, the most reusable slices are `cache_trajectory` plus `tool_errors_by_tool` and `hook_overhead`, because they align with existing session, hook, and observability work; `redundant_reads`, `edit_retries`, and turn-bucket histograms are useful, but they are second-order diagnostics that do not need to ship in the first borrowing pass. Recommendation: take cache, error-rate, and hook-overhead contracts now; postpone the more behavioral widgets. Recommendation: (prototype later)
- `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/005-claudest/external/plugins/claude-memory/skills/get-token-insights/templates/dashboard.html` ends with the most selectively borrowable section set for Public: "Claude Code Ecosystem", "Findings", "Recommendations", and the optional "Week-on-Week Trends" block. Their contracts are concise and portable: `skill_usage[].{skill,count,errors}` feeds both the skill-usage SVG and the disable-candidate table; `agent_delegation[].{subagent_type,count,errors}` powers delegation counts; `hook_performance[].{hook_command,total_ms,runs,avg_ms,errors}` powers hook ranking; `findings[].{severity,title,text}` and `recommendations[].{priority,text}` are already presentation-agnostic compatibility layers over the deeper `insights` model; and `trends` contains `current_window.{sessions,cost_usd,turns}`, optional `prior_window.{sessions,cost_usd,turns}`, `metrics.{cost_per_session,cache_ratio,cliffs_per_session,antipatterns_per_session,tool_error_rate,hook_avg_ms}` with each metric row exposing `{label,prior,current,change_pct}`, plus `improved[]`, `regressed[]`, `new_skills[]`, `retired_skills[]`, and `hook_trends[].{hook,prior_ms,current_ms,change_pct}` (`979-1288`, `1372-1421`, `1917-1930`). These are the strongest borrow candidates for `Code_Environment/Public` because they can be rendered in markdown tables, changelog snippets, or lightweight internal dashboards without any dependency on the Claudest visual shell. Recommendation: borrow `trends`, `skill_usage`, `agent_delegation`, `hook_performance`, `findings`, and `recommendations` as the core selective-import package. Recommendation: (adopt now)

## Ruled Out
- Borrowing the full `dashboard.html` presentation layer: inline CSS theme tokens, scanline overlay, fixed ticker, collapsible section behavior, Chart.js wiring, Google Fonts dependency, and the SVG-specific skill chart are not the durable value. They are transport and styling decisions, not the underlying contract.
- Reusing the "Skills to Consider Disabling" widget as-is. The useful part is still `skill_usage[].{skill,count,errors}`; the disable-table copy and threshold (`count <= 2`) is opinionated product policy, not a reusable observability primitive.
