### 5.3 Q-C — Composition risk
V2 keeps the high-level composition verdict from v1: bounded adapters survive, monolithic facades do not. What changes is the explanatory model. V1 mostly explained composition risk in terms of architecture fit. Iter-13 and iter-15 show that many medium- and high-risk candidates are risky not only because they centralize control, but because they presume unification contracts Public does not yet own. A static artifact layer is risky not merely because it adds a packaging tier; it is risky because it needs freshness authority and a stable interchange contract. Evidence tagging is risky not merely because it touches multiple surfaces; it is risky because Public lacks one cross-surface trust contract. Two-layer invalidation is risky not merely because it spans Code Graph and CocoIndex; it is risky because there is no shared invalidation ledger yet. [SOURCE: research/iterations/public-infrastructure-iter-13.md:158-169] [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:77-96]

This matters because it changes what “low-risk” should mean in the next implementation cycle. In v2, low-risk means not only “stays inside one owner surface” but also “does not require a missing global contract to become trustworthy.” That keeps graph-first nudges, the runtime FTS cascade, detector regression, search/browse separation, and dashboard contracts in the safer bucket. It pushes artifact-default packaging, merged trust vocabularies, and scan-first orchestration further out unless Public funds the missing contract work explicitly. [SOURCE: research/iterations/public-infrastructure-iter-13.md:171-181] [SOURCE: research/iterations/cost-reality-iter-14.md:436-448]

Meanwhile, the runtime FTS cascade became less strategic and more opportunistic because its substrate is already present. 1. Does the candidate stay inside one owner or clearly route to an existing owner?

## 12. New cross-phase patterns surfaced after iter-8
Iter-15’s biggest contribution was not another recommendation or another system score. It was a better abstraction over what the five phase packets repeatedly got wrong in similar ways. ### Pattern 1 — Precision laundering at the presentation layer
The strongest new pattern is that weak evidence repeatedly becomes crisp-looking output. Phase 001 turns one field-observed schema-payload delta into a large session-scale arithmetic story despite denominator drift. CodeSight’s Go extractor emits `confidence: "ast"` even where the implementation is explicitly regex-plus-brace-tracking. None of these moves is identical, but together they show the same failure mode: the producer knows some of the signal is approximate, yet the presentation layer makes the output look stronger than the underlying evidence really is. [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:13-30]

For Public, that pattern has direct design consequences. V2 should not allow a single “confidence” field to stand in for provenance, certainty, and freshness at once. It should not let dashboards silently turn defaulted or heuristic fields into rows that appear equal to exact provider-counted ones. And it should not allow formatters to hide where a number came from merely because a consumer surface wants a single neat scalar. The remedy is not merely “be conservative.” It is to carry uncertainty as first-class data: `exact`, `estimated`, `defaulted`, `unknown`, plus separate trust axes where necessary. [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:27-33] [SOURCE: research/iterations/counter-evidence-iter-16.md:49-57] [SOURCE: research/iterations/counter-evidence-iter-16.md:85-93]

### Pattern 2 — Outcome claims are validated one seam too early
The second pattern explains why so many apparently “implemented” ideas still felt softer under skeptical review. Multiple phases validate a local mechanism while leaving the user-visible claim untested. Phase 001 can justify smaller upfront payload, but not first-tool ergonomics. CodeSight evaluates routes/models/components/env/middleware, but not blast radius or user-facing token-savings claims. Contextador measures internal counter arithmetic, not paired benchmark performance against Public’s typed stack. Graphify tests hook install and uninstall hygiene, not whether the hook improves behavior. Claudest has a plausible startup fast path, but no proof that it preserves pass rate when selected sessions are lossy or heuristic. The recurring error is not falsehood at the unit-test level. It is seam mismatch between what was validated and what later prose asked the reader to believe. [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:35-55]

That pattern implies a stronger adoption standard for Public. V2 should treat unit tests, parser fixtures, and contract validation as necessary but insufficient. Whenever the adoption language claims a user-visible outcome, Public should create a with/without lane at the same seam as the claim: routing correctness for a routing claim, pass-rate preservation for a cached-startup claim, stale-artifact rate for a static-brief claim, or follow-up count for a structural-guidance claim. This is why the warm-start combo fell and why the graph-first hook wording narrowed. The local mechanism is real; the broader outcome still needs proof. [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:49-55] [SOURCE: research/iterations/counter-evidence-iter-16.md:31-39] [SOURCE: research/iterations/counter-evidence-iter-16.md:40-48]

### Pattern 3 — Durable context arrives before freshness authority
The third pattern is a design-direction warning more than a registry headline, but it is one of the most important v2 additions. CodeSight’s generated profiles tell the agent to read the static artifact first and query live surfaces second, without a strong freshness contract. Graphify’s hook only helps if the graph already exists and is current. Claudest’s summary fast path is explicitly lossy and can select the wrong prior session. Across three systems, durable context is introduced early because it feels efficient and ergonomic, yet the freshness and authority boundary is left loose enough that the durable surface can become a wrong first source of truth. [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:57-75]

This pattern directly reshapes Public’s roadmap. V2 should not package static-first structural artifacts or treat persisted summaries as default truth until each durable surface can answer at least four questions: who produced this, when, for what scope, and under what invalidation rule? Public already has stronger startup and recovery surfaces than any external single system. The goal is to preserve that advantage, not trade it for a polished artifact-first UX that hardens stale or partial state. That is why R2, R3, and R8 are all downgraded rather than removed: the ideas are still useful, but only once freshness authority is explicit. [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:69-75] [SOURCE: research/iterations/counter-evidence-iter-16.md:22-39] [SOURCE: research/iterations/counter-evidence-iter-16.md:76-84]

### Pattern 4 — Cost spikes at contract seams, not inside owner surfaces
The fourth pattern explains why iter-14’s effort corrections felt disproportionate relative to the visible feature names. Several candidates looked small or medium when read as isolated behaviors, but became medium or large once Public’s actual stack was inventoried. Honest reporting is not “add a nicer metrics formatter”; it crosses token metrics, logs, envelopes, and dashboard rollups. Deterministic Stop summaries are not “store one extra field”; they cross Stop hooks, session state, SessionStart, and bootstrap consumers. Static artifacts are not “render one brief”; they need freshness/versioning, transport integration, prompt-surface consumers, and stale fallback behavior. By contrast, the runtime FTS cascade shrank because the substrate was already there. The expensive work is therefore concentrated at seams where multiple producers and consumers must agree, not inside the already-strong owner surfaces. [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:77-96] [SOURCE: research/iterations/cost-reality-iter-14.md:5-12] [SOURCE: research/iterations/cost-reality-iter-14.md:436-448]

That pattern should change how Public scopes follow-on packets. Any proposal that crosses memory, structural, and semantic surfaces should default upward in effort unless it can prove it is mostly packaging over an existing contract. Conversely, a proposal inside one owner surface should not be inflated just because its name sounds foundational. This is why v2 promotes the FTS cascade as substrate-ready hardening while demoting static artifacts and shared invalidation to later architectural work. It is also why the replacement for R10 is a trust-contract recommendation rather than another packaging recommendation. [SOURCE: research/iterations/cost-reality-iter-14.md:436-448] [SOURCE: research/iterations/counter-evidence-iter-16.md:94-100]

Taken together, these four patterns explain the rigor lane better than any single question lane does. V1 mostly asked which ideas Public should borrow. Iter-15 shows the deeper answer: borrow ideas that sharpen seams without laundering uncertainty, validate them at the claim seam rather than the mechanism seam, refuse durable-first UX without freshness authority, and budget cross-surface contract work honestly. [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:105-109]

## 13. Public's moats and hidden prerequisites
Iter-13 forced the biggest baseline correction in the whole packet: Public is not a thin substrate waiting for a superior external design to replace it. It is a large, already-shipped platform whose moat comes from combining several capabilities that the external systems only show in fragments. V2 needs this section because recommendation language that only compares external systems to one another will otherwise under-protect what Public already does better. [SOURCE: research/iterations/public-infrastructure-iter-13.md:5-11] [SOURCE: research/iterations/public-infrastructure-iter-13.md:194-198]

### Public’s 8 moats
1. **Canonical non-hook recovery surface.** `session_bootstrap` already unifies resume, health, structural readiness, transport planning, and next actions in one recovery contract. No external system in this packet ships an equivalent one-call recovery surface. [SOURCE: research/iterations/public-infrastructure-iter-13.md:185-185]
2. **Cross-runtime compact-cache startup pipeline.** Public has startup/compaction flows for more than one runtime plus a startup-brief builder, which makes its warm-start story broader than any single external system’s. [SOURCE: research/iterations/public-infrastructure-iter-13.md:186-186]
3. **Five-channel retrieval fused with graph and degree signals.** Public does not merely have semantic plus lexical search; it fuses vector, FTS5, BM25, graph, and degree signals behind routing logic. [SOURCE: research/iterations/public-infrastructure-iter-13.md:187-187]
4. **Research-grade eval stack inside the shipped product.** Ablation, ground-truth alignment, shadow scoring, and persisted eval storage are part of the live product surface rather than separate research-only machinery. [SOURCE: research/iterations/public-infrastructure-iter-13.md:188-188]
5. **Constitutional memory plus tier/state cognition.** Public’s memory system already combines always-surface rules, explicit importance tiers, state categories, and save gates in one runtime. [SOURCE: research/iterations/public-infrastructure-iter-13.md:189-189]
6. **Shared-memory governance with deny-by-default spaces.** Governance depth around spaces, membership, kill switches, and audits exceeds what any external system here exposed. [SOURCE: research/iterations/public-infrastructure-iter-13.md:190-190]
7. **Explicit split-topology contract with bounded merges.** Public explicitly names semantic, structural, and continuity owners, then merges them only in constrained recovery/compaction surfaces. That architectural self-awareness is itself a moat. [SOURCE: research/iterations/public-infrastructure-iter-13.md:191-191]
8. **Runtime capability staging.** Public already ships staged capability flags and in-runtime phase thinking rather than one static on/off feature surface. [SOURCE: research/iterations/public-infrastructure-iter-13.md:192-192]

These moats matter because they explain why some external ideas should be treated as supplements rather than replacements. Claudest’s continuity patterns are compelling because Public already has a strong continuity surface to receive them. Graphify’s hook ideas are compelling because Public already has hook infrastructure and clear retrieval owners. CodeSight’s packaging patterns are compelling because Public already has a large command/agent/document surface to project into. Contextador’s source code is not compelling because its main differentiators lie in ergonomics and surface posture, not in substrate Public lacks, and the 20-iteration closeout further narrowed those differentiators to bootstrap/setup, bounded local budgeting, and scoped automation rather than retrieval depth. [SOURCE: research/iterations/public-infrastructure-iter-13.md:171-181] [SOURCE: 003-contextador/research/research.md:47-51] [SOURCE: 003-contextador/research/research.md:214-242] [SOURCE: research/iterations/q-e-license-runtime.md:50-69]

### The 11 hidden prerequisites v2 must treat as explicit
1. **No canonical scan artifact** for monolithic scan-first orchestration. [SOURCE: research/iterations/public-infrastructure-iter-13.md:159-159]
2. **No cross-surface evidence/confidence contract** spanning CocoIndex and Code Graph. [SOURCE: research/iterations/public-infrastructure-iter-13.md:160-160]
3. **No shared invalidation ledger** spanning structural and semantic freshness. [SOURCE: research/iterations/public-infrastructure-iter-13.md:161-161]
4. **No ORM-specific extraction logic** for SQLAlchemy schema detection despite existing AST plumbing. [SOURCE: research/iterations/public-infrastructure-iter-13.md:162-162]
5. **No generated `.mcp.json` command surface** that emits the activation scaffold directly. [SOURCE: research/iterations/public-infrastructure-iter-13.md:163-163]
6. **No authoritative boundary validator contract** because the stable interchange artifact does not yet exist. [SOURCE: research/iterations/public-infrastructure-iter-13.md:164-164]
7. **No multimodal rebuild classifier / freshness coordinator** for the modality-aware rebuild lane. [SOURCE: research/iterations/public-infrastructure-iter-13.md:165-165]
8. **No durable, versioned interchange artifact** that multiple consumers can rely on. [SOURCE: research/iterations/public-infrastructure-iter-13.md:166-166]
9. **No canonical persisted Stop-summary artifact** equivalent to Claudest’s assumed fast path. [SOURCE: research/iterations/public-infrastructure-iter-13.md:167-167]
10. **No first-class auditor/discoverer workflow contract** even though governance and eval pieces exist. [SOURCE: research/iterations/public-infrastructure-iter-13.md:168-168]
11. **No dedicated tier-priced ledger or cache-cliff schema** despite strong telemetry substrate. [SOURCE: research/iterations/public-infrastructure-iter-13.md:169-169]

What v2 adds, beyond the bullet list itself, is the relationship between moats and missing prerequisites. The moats tell us why Public does not need wholesale replacement. The prerequisites tell us why several attractive external ideas are still expensive even though Public is strong. Public is “strong but seam-indebted”: it has the owners and much of the substrate, but it lacks the handful of contracts that would let cross-surface trust, freshness, and analytics claims stay honest under load. That combination is exactly why v1’s strongest overstatements happened where they did. [SOURCE: research/iterations/public-infrastructure-iter-13.md:8-11] [SOURCE: research/iterations/cross-phase-patterns-iter-15.md:77-96]

This section also changes how to read some downgrades. Downgrading the cached startup fast path or the FTS cascade does not mean the ideas got weaker in absolute terms. It means iter-13 showed that Public already ships adjacent substrate for them, so the strategic question is no longer “should Public adopt this category at all?” It is “what exactly is still missing, and does that missing piece belong at the top of the roadmap?” That reframing demotes some items while simultaneously making their implementation easier once chosen. [SOURCE: research/iterations/public-infrastructure-iter-13.md:177-181] [SOURCE: research/iterations/counter-evidence-iter-16.md:67-73]

The moat/prerequisite pairing also explains why the replacement for v1’s R10 is a contract recommendation. Public already has enough substrate to produce artifacts, overlays, startup briefs, shared payloads, and structural exports. What it does not have yet is the trust-axis and freshness-authority contract that would make those artifacts worth trusting as first-class reusable context. Without that contract, more packaging would mostly amplify ambiguity. With that contract, later packaging work could become valuable rather than misleading. [SOURCE: research/iterations/counter-evidence-iter-16.md:94-100] [SOURCE: research/iterations/public-infrastructure-iter-13.md:160-167]
