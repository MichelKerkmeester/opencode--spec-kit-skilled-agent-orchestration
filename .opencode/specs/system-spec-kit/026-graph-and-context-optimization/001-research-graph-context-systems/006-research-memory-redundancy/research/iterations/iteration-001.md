## Generator code observations

- `generate-context.ts` is mostly a JSON-primary CLI wrapper; it advertises structured session fields (`toolCalls`, `exchanges`, `preflight`, `postflight`) and says the memory file is for indexing by the Spec Kit Memory system, which is a recall-time concern rather than a spec-doc authoring concern. [SOURCE: .opencode/skill/system-spec-kit/scripts/memory/generate-context.ts:51-124]
- The renderer contract is packet-shaped, not recall-shaped. `memory-template-contract.ts` treats project snapshot, implementation guide, detailed changes, decisions, and recovery hints as first-class memory sections. [SOURCE: .opencode/skill/system-spec-kit/shared/parsing/memory-template-contract.ts:47-54]
- `context_template.md` reserves large body areas for packet-style prose and decision expansion, then adds recovery commands and a heavy metadata tail. That is where the "mini implementation-summary + mini decision-record" feel comes from. [SOURCE: .opencode/skill/system-spec-kit/templates/context_template.md:175-184] [SOURCE: .opencode/skill/system-spec-kit/templates/context_template.md:482-657]
- The clearly unique memory-only material is near the tail: postflight learning delta and machine-readable memory metadata (session id, git provenance, decay, dedup fingerprint). Those sections are defensible even if the packet-summary prose is removed or shrunk. [SOURCE: .opencode/skill/system-spec-kit/templates/context_template.md:676-760]
- Provisional change direction from this pass: keep the unique recovery/provenance/search metadata; collapse the packet-summary sections into short pointers to canonical docs (`decision-record.md`, `implementation-summary.md`, `research/research.md`) instead of re-rendering their content.
