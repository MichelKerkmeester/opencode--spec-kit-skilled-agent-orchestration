## 2. TOPIC

Eliminate redundancy between memory saves (generated by `generate-context.js` into each packet's `memory/` folder) and the static spec docs `decision-record.md` and `implementation-summary.md`. Identify which artifact should own which content, where the redundancy is produced, and what changes preserve recall semantics while removing duplication.

## 3. KEY QUESTIONS (remaining)

- [x] Q1. **Content overlap audit.** Across 026 child phases (002, 003 children, 005–013), what specific content patterns does the generated memory file duplicate from the same packet's `decision-record.md` and `implementation-summary.md`? Quantify by section (executive summary, decisions, outcomes, file changes, etc.).
- [x] Q2. **Unique value of memory saves.** What does a memory save provide that a static spec doc cannot? (Recall-time semantic search? Auto-surface triggers? Cross-packet retrieval? Constitutional-tier lookups? Semantic indexing budget?)
- [x] Q3. **Source of redundancy.** Is the duplication produced by (a) `generate-context.js` over-collecting content via its extractors, (b) the memory template encouraging full-content restatement, (c) the spec doc templates being too narrow so memory saves compensate, or (d) the AI composing the JSON payload restating things?
- [x] Q4. **Recall semantics preservation.** What does the memory system actually need to find a packet later? Title + trigger phrases + a short distinguishing summary, or full content? What is the smallest payload that still passes auto-surface and search?
- [x] Q5. **Change proposal.** Concrete recommendations: (a) what to change in `generate-context.js` extractors, (b) what to change in the memory frontmatter/body template, (c) whether `decision-record.md` or `implementation-summary.md` should be the canonical source the memory file points to, (d) any spec doc template additions needed.
