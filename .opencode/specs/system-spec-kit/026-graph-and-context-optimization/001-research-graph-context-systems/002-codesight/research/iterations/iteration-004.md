## Summary
Per-tool profile generation in `external/src/generators/ai-config.ts` is meaningfully different from "rename a file": there is one shared `generateContext()` summary plus a `generateProfileConfig()` switch with explicit per-tool instruction overlays for Claude Code, Cursor, Codex, Copilot, Windsurf, and a generic fallback. ## Findings

### Finding 1 - `generateAIConfigs` ships one shared summary into 5 distinct assistant files with safe append semantics
- Source: `external/src/generators/ai-config.ts:14-69`, `external/src/generators/ai-config.ts:71-158`
- What it does: `generateContext(result)` builds one narrative paragraph from `project`, `routes`, `schemas`, `components`, `middleware`, `config.envVars`, and `graph.hotFiles`, ending with a "Read .codesight/CODESIGHT.md" pointer. `generateAIConfigs(result, root)` then writes that same summary into `CLAUDE.md`, `.cursorrules`, `.github/copilot-instructions.md` (only if `.github` already exists), `codex.md`, and `AGENTS.md`. The CLAUDE.md path is the only one with append-aware behavior: if `CLAUDE.md` exists and does not already mention "codesight"/"CODESIGHT", it appends an `# AI Context (auto-generated by codesight)` block instead of overwriting. The other four paths only write when the file does not exist, so they will silently no-op on repos that already have those files.
- Why it matters: For `Code_Environment/Public`, this proves a useful mid-cost adoption path: re-use one canonical project summary, but treat each assistant target as a write boundary that can either no-op, append a marker section, or refuse to clobber. The Codesight code is also a good cautionary example: only CLAUDE.md gets a real upsert path; the others are first-write-wins, which would be unsafe in repos that already maintain assistant-specific guidance.
- Evidence type: source-confirmed
- Recommendation: prototype later
- Affected area: AI-assistant context generation
- Risk/cost: medium - direct emission to top-level instruction files conflicts with repos that already own `CLAUDE.md`/`AGENTS.md`/etc. unless namespacing or marker comments are added.

### Finding 2 - `generateProfileConfig` is the real per-tool overlay layer and contains tool-specific instructions, not just file renames
- Source: `external/src/generators/ai-config.ts:164-264`
- What it does: `generateProfileConfig(result, root, profile)` first builds a compact ~1-2k token summary (stack, monorepo workspaces, route/model/env counts, top API areas, top 5 hot files, required env vars), then `switch (profile)` writes meaningfully different content per target:
  - `claude-code` → `CLAUDE.md`: instructs the assistant to read `.codesight/CODESIGHT.md` first and lists 4 specific MCP tool calls (`codesight_get_summary`, `codesight_get_routes --prefix /api/users`, `codesight_get_blast_radius --file src/lib/db.ts`, `codesight_get_schema --model users`), and reports `tokenStats.saved` to motivate the workflow. It also has a regex-based replace path that overwrites a prior `# AI Context...` block in-place.
  - `cursor` → `.cursorrules`: gives a 4-step file-reading order (`CODESIGHT.md`, `routes.md`, `schema.md`, `graph.md`) and an explicit "Do not crawl the file tree" directive.
  - `codex` → `codex.md`: minimal single-line pointer to `.codesight/CODESIGHT.md` plus a "Routes/Schema/Dependencies" file map, written as a self-contained file (no append).
  - `copilot` → `.github/copilot-instructions.md`: similar minimal pointer; creates `.github/` if missing.
  - `windsurf` → `.windsurfrules`: same minimal pointer using Windsurf's expected filename.
  - `default` (no profile) falls back to `generateAIConfigs()` and writes everything.
- Why it matters: This is the most directly transferable pattern in Codesight. `Code_Environment/Public` could borrow the "shared summary + per-tool instruction overlay" split for its `CLAUDE.md` / `.codex` / `.cursor` / `.gemini` directories, but the cargo-cult risk is real: Codesight's current implementation hardcodes example MCP tool names like `codesight_get_blast_radius` into the Claude profile, which would only make sense after a similar MCP surface is built locally.
- Evidence type: source-confirmed
- Recommendation: adopt now
- Affected area: AI-assistant context generation, per-tool config profiles
- Risk/cost: low - the layout is small and easy to lift; the work is in choosing which instructions are accurate for each runtime.

### Finding 3 - `eval.ts` is a real precision/recall/F1 harness, but its scope is detector accuracy, NOT token savings
- Source: `external/src/eval.ts:43-68`, `external/src/eval.ts:85-149`, `external/src/eval.ts:161-244`, `external/eval/README.md:1-37`
- What it does: `runEval()` reads each fixture's `repo.json` (file content map), materializes a temp directory, runs `detectProject` + `collectFiles` + the 5 detectors (`detectRoutes`, `detectSchemas`, `detectComponents`, `detectConfig`, `detectMiddleware`) in parallel, then compares detected sets to the fixture's `ground-truth.json` for routes (`method:path`), models (lowercased name), env vars, and optionally components and middleware. `calcMetrics()` computes precision = TP/(TP+FP), recall = TP/(TP+FN), F1 = harmonic mean with the same `(empty,empty) → 1.0` convention as scikit-learn. Output is per-fixture metric tables plus an unweighted average across all fixture/category pairs. There is no token-savings measurement, no statistical test, no confidence interval, no run-to-run variance, and `runtime` is wall-clock for one detector pass per fixture.
- Why it matters: For `Code_Environment/Public`, the right takeaway is: an F1 harness against ground-truth fixtures is the strongest piece of "evidence" Codesight ships, and it could be lifted as a pattern for detector or context-generator regression. But it does NOT validate the headline "11.2x token reduction" claim; the reproducible eval is about detector correctness only.
- Evidence type: source-confirmed
- Recommendation: adopt now
- Affected area: benchmark methodology, regression testing
- Risk/cost: low - the harness is ~245 lines, has no external dependencies, and its limitations (no variance, no token-savings) are explicit in source.

### Finding 4 - The "11.2x average token reduction" claim is README-only and is not covered by any fixture in this checkout
- Source: `external/README.md:52-64`, `external/eval/README.md:1-37`, `external/eval/fixtures/` (5 directories: `express-prisma`, `fastapi-sqlalchemy`, `gin-gorm`, `hono-monorepo`, `nextjs-drizzle`), `external/src/eval.ts:43-244`
- What it does: The README presents one Benchmarks table with three projects ("SaaS A: Hono + Drizzle 4 workspaces, 12.9x", "SaaS B: raw HTTP + Drizzle, 11.7x", "SaaS C: Hono + Drizzle 3 workspaces, 9.1x") averaged to 11.2x. None of these three SaaS codebases are present under `external/eval/fixtures/`; the fixture set covers different stacks (Express/Prisma, FastAPI/SQLAlchemy, Gin/GORM, Hono monorepo, Next.js/Drizzle). The reproducible `eval.ts` harness measures precision/recall/F1 only, never token counts, so there is no path inside this checkout to reproduce or falsify the 11.2x headline.
- Why it matters: For `Code_Environment/Public`, treat token-savings claims as promotional unless we own the measurement. Adopt the F1 fixture pattern, but if we want to make our own savings claim we must (a) instrument a token counter against the same fixture set and (b) define an explicit "manual exploration" baseline rather than borrow Codesight's number.
- Evidence type: README-level (claim) + source-confirmed (gap)
- Recommendation: reject
- Affected area: benchmark methodology, marketing-vs-engineering boundary
- Risk/cost: low - the gap is purely a documentation/credibility risk, easy to side-step if we resist quoting the number.

### Finding 5 - `tests/detectors.test.ts` is broader than the eval fixtures and is the strongest validation of "25+ frameworks"
- Source: `external/tests/detectors.test.ts:34-269` (Route Detection block: 14 tests covering Hono, Express, Fastify, NestJS, tRPC, SvelteKit, Remix, Nuxt, Next.js App Router, FastAPI, Django, Elysia, plus a raw HTTP catch-all), `external/tests/detectors.test.ts:271-321` (Schema Detection: Drizzle and Prisma), `external/tests/detectors.test.ts:325-498` (Components, Dependency Graph, Config, Middleware, Framework Detection blocks)
- What it does: The test file uses Node's built-in `node:test` runner, builds inline fixtures with `writeFixture(...)`, and dynamically imports compiled detector modules from `dist/`. Route Detection runs 14 named cases for HTTP frameworks and asserts both count and at least one specific path/method. Schema Detection covers Drizzle (with chained `.notNull().unique()` field flag assertions) and Prisma (file-existence + count check). The Framework Detection block adds 7 more frameworks asserted from package.json alone (NestJS, tRPC, SvelteKit, Remix, Nuxt, Elysia, monorepo).
- Why it matters: This is the strongest test-confirmed evidence that "25+ frameworks" is more than marketing: at least ~14 unique frameworks have live integration assertions, plus another 7 are package.json-only. For `Code_Environment/Public`, the lift is the test pattern itself: in-process fixtures, minimal setup, count + spot-check assertions, parallel detectors. The skepticism for our own port is also useful: only Drizzle and Prisma have schema tests; the README's "8 ORM parsers" claim is NOT covered by tests in this checkout.
- Evidence type: test-confirmed
- Recommendation: adopt now
- Affected area: detector regression testing
- Risk/cost: low - the pattern is about 500 LOC for full detector coverage and would be a small lift.

### Finding 6 - The `hono-monorepo` ground-truth file shows what a "fixture-as-evidence" record actually looks like
- Source: `external/eval/fixtures/hono-monorepo/ground-truth.json:1-33`
- What it does: Hand-curated JSON with 11 expected routes (some duplicated paths across HTTP methods), 2 expected models with ordered field arrays and named relations, 2 expected components with named props, 3 expected env vars, and 1 expected middleware ("auth"). Notably, `routes` has multiple identical `/` and `/:id` entries because the harness compares `method:path` strings; the duplication is real, not a typo.
- Why it matters: For `Code_Environment/Public`, this is a tiny but instructive ground-truth schema: it costs almost nothing to write, produces verifiable F1 numbers, and is the kind of artifact a context-generation regression test could share with `system-spec-kit`'s validation infrastructure. The risk to flag: ground-truth quality directly determines the meaning of every metric, and Codesight has only 5 such fixtures.
- Evidence type: source-confirmed
- Recommendation: adopt now
- Affected area: benchmark methodology, regression testing
- Risk/cost: low - the format is trivially easy to author and to consume.
