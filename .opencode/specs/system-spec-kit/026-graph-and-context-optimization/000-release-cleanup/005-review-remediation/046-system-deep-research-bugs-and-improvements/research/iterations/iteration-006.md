# Iteration 006 — B1: Hook contract drift across runtimes

## Focus
Audited the prompt-time skill-advisor hook contracts across Claude, Copilot, Gemini, Codex, and the OpenCode plugin bridge. The focus was whether they render the same advisor-brief text and whether daemon-down or degraded paths fail open consistently.

## Actions Taken
- Enumerated hook and advisor files with `rg --files` / `rg -n` across `.opencode/skill/system-spec-kit/mcp_server/hooks`, `.opencode/plugins`, and the skill-advisor bridge.
- Read the prompt adapters: `hooks/claude/user-prompt-submit.ts`, `hooks/copilot/user-prompt-submit.ts`, `hooks/gemini/user-prompt-submit.ts`, `hooks/codex/user-prompt-submit.ts`, and `hooks/codex/session-start.ts`.
- Read the shared renderer and builder: `skill_advisor/lib/render.ts`, `skill_advisor/lib/skill-advisor-brief.ts`, `skill_advisor/lib/freshness.ts`, and `skill_advisor/lib/subprocess.ts`.
- Read the OpenCode bridge and plugin entrypoint: `mcp_server/plugin_bridges/spec-kit-skill-advisor-bridge.mjs` and `.opencode/plugins/spec-kit-skill-advisor.js`.
- Checked runtime docs/config references for event-name differences, especially Gemini `BeforeAgent` and Copilot's file-based custom-instructions workaround.

## Questions Answered
- Same advisor-brief format? Mostly yes on normal paths. Claude calls `buildSkillAdvisorBrief()` and `renderAdvisorBrief()` at `.opencode/skill/system-spec-kit/mcp_server/hooks/claude/user-prompt-submit.ts:164-170`; Gemini does the same at `.opencode/skill/system-spec-kit/mcp_server/hooks/gemini/user-prompt-submit.ts:178-184`; Copilot renders the same brief then writes it into the managed custom-instructions block at `.opencode/skill/system-spec-kit/mcp_server/hooks/copilot/user-prompt-submit.ts:197-217`; Codex normal path renders through the shared renderer at `.opencode/skill/system-spec-kit/mcp_server/hooks/codex/user-prompt-submit.ts:320-332`; OpenCode native/legacy bridge paths render through compat `renderAdvisorBrief()` at `.opencode/skill/system-spec-kit/mcp_server/plugin_bridges/spec-kit-skill-advisor-bridge.mjs:232-255` and `.opencode/skill/system-spec-kit/mcp_server/plugin_bridges/spec-kit-skill-advisor-bridge.mjs:286-295`.
- Rendering function and format strings: the canonical renderer is `renderAdvisorBrief()` at `.opencode/skill/system-spec-kit/mcp_server/skill_advisor/lib/render.ts:111-159`. Its two format strings are the ambiguous form at `.opencode/skill/system-spec-kit/mcp_server/skill_advisor/lib/render.ts:149-151` and the single recommendation form at `.opencode/skill/system-spec-kit/mcp_server/skill_advisor/lib/render.ts:155-157`.
- Same fallback when daemon/advisor state is degraded? Mostly. `buildSkillAdvisorBrief()` checks freshness at `.opencode/skill/system-spec-kit/mcp_server/skill_advisor/lib/skill-advisor-brief.ts:425-445`, runs the Python shim on live/stale artifacts at `.opencode/skill/system-spec-kit/mcp_server/skill_advisor/lib/skill-advisor-brief.ts:477-500`, and `runAdvisorSubprocess()` invokes `skill_advisor.py --stdin` at `.opencode/skill/system-spec-kit/mcp_server/skill_advisor/lib/subprocess.ts:258-263`. OpenCode first probes native availability at `.opencode/skill/system-spec-kit/mcp_server/plugin_bridges/spec-kit-skill-advisor-bridge.mjs:329-343`, then falls back to the same legacy builder unless `forceNative` is set.
- Gemini naming: despite the requested "Gemini UserPromptSubmit" label, the checked-in Gemini hook is registered as `BeforeAgent` in `.gemini/settings.json:95-107` and documents that registration at `.opencode/skill/system-spec-kit/mcp_server/hooks/gemini/README.md:24-41`.
- Codex SessionStart is not an advisor renderer. It formats startup/resume/clear context via `formatHookOutput()` and returns `hookEventName: 'SessionStart'` at `.opencode/skill/system-spec-kit/mcp_server/hooks/codex/session-start.ts:160-175`.

## Questions Remaining
- Should Codex's timeout fallback be treated as an intentional compatibility extension or collapsed into the shared `renderAdvisorBrief()` contract?
- Should OpenCode disabled mode be operator-visible only through plugin status, rather than model-visible system context?
- Should Copilot's next-prompt freshness behavior be represented in the shared contract metadata so synthesis can distinguish "same text format" from "same turn visibility"?

## Next Focus
Follow-on work should audit the parity tests around exceptional paths: timeout, disabled, force-native, missing prompt, and stale artifacts. The normal rendering path is covered reasonably well; the drift lives in fail-open and runtime workaround behavior.
