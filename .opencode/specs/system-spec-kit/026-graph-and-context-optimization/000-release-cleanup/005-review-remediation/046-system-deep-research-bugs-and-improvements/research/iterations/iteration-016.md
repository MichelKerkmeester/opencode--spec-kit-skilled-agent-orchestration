## Questions Answered

- Responsibilities are not fully clean: several modules are doing both domain logic and orchestration (`daemon/watcher.ts`, `skill-graph-db.ts`, `shared-payload.ts`).
- The strongest cross-boundary offenders are lib-to-handler imports, core lib-to-advisor-internal imports, and reuse of helpers from modules whose names imply a narrower responsibility.
- No P0/P1 production bug was found in this pass; the issues are maintainability/coupling risks that can harden into test brittleness and circular ownership as the advisor surface grows.

## Questions Remaining

- Whether the repo already has an intended neutral home for generic SQLite utilities, source-ref sanitization, and spec-document discovery.
- Whether `daemon/lifecycle.ts` is intended to become the composition root for watcher/indexer/generation responsibilities, or whether a new `daemon/service.ts` should own that wiring.
- Whether advisor projection should own lifecycle status normalization, or whether lifecycle status should be materialized earlier into graph metadata/index rows.

## Next Focus

Follow-on D1 work should map tests to these boundary edges and identify the smallest reorganization that keeps public behavior stable: likely extracting shared utilities first (`sqlite-integrity`, `busy-retry`, `payload-sanitizer`, `spec-document-discovery`), then slimming `daemon/watcher.ts`.
