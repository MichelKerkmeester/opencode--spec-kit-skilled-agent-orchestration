# Iteration 3: Q4 parser-fallback feasibility + Q5 glossary precedent

## Focus

Two question advances driven by iter-1 finding 4's parser-fallback recommendation and the spec.md REQ-009 vocabulary contract. Track: **migration-and-governance** — distinct from iter-1's concept-layer work and iter-2's tier-stratification work, this iteration converts strategy into mechanics.

**Q4 (`_memory.continuity` parser-fallback feasibility)**: Iter-1 finding 4 recommended a parser-fallback that reads both legacy `_memory:` and a new key (e.g., `_continuity:`), prefers the new key, and warns on the legacy path. This iteration validates mechanical feasibility by (a) locating every parser surface in the TS/JS source, (b) sizing the existing migration corpus across all spec-folder doc types, and (c) verdict on parser-fallback complexity vs hard-rename.

**Q5 (glossary scaffolding precedent)**: REQ-009 requires a vocabulary contract. This iteration surveys whether a glossary doc, lint rule, or contract precedent already exists in the repo, and recommends a scaffolding plan sized to actual enforcement infrastructure.

## Findings

### Q4: Parser-fallback feasibility — VERDICT: HIGH FEASIBILITY, ~7 distinct source-side touchpoints

1. **Parser surfaces are concentrated in a single TypeScript codebase with one canonical anchor constant repeated across 5+ sites.** The string `'_memory.continuity'` is hard-coded as `CONTINUITY_ANCHOR_ID` in **three** independent locations: `mcp_server/handlers/save/create-record.ts:23` (compiled `.js:23`), `mcp_server/handlers/memory-search.js:49`, and `mcp_server/lib/parsing/trigger-matcher.js:64`. A fourth equivalent constant `DEFAULT_METADATA_ANCHOR = '_memory.continuity'` exists at `mcp_server/lib/routing/content-router.js:44`. A fifth literal anchor `_memory.continuity` appears in `mcp_server/handlers/memory-save.js:67` (`ROUTED_CONTINUITY_ANCHOR_ID`). This duplicated-constant pattern means a parser-fallback must either (a) introduce a single shared module exporting both `LEGACY_CONTINUITY_ANCHOR_ID = '_memory.continuity'` and `CONTINUITY_ANCHOR_ID = '_continuity'` (the new name), or (b) accept the constant repetition and update each site individually. Option (a) is structurally cleaner and matches the existing pattern in `mcp_server/lib/search/vector-index-schema.ts` where `MEMORY_LINEAGE_TABLE` and `LEGACY_MEMORY_LINEAGE_TABLE` already coexist. [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/handlers/save/create-record.ts:23`] [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-search.js:49`] [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/dist/lib/parsing/trigger-matcher.js:64`] [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/dist/lib/routing/content-router.js:44`] [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/dist/handlers/memory-save.js:67`]

2. **The actual YAML reader uses `js-yaml`'s `yaml.load()` with type-guarded property checks — pure-JS object access, NOT a custom alias-aware extractor.** `scripts/validation/continuity-freshness.ts:62-77` shows the canonical read pattern: `yaml.load(frontmatterBlock)` produces a plain object, and the code probes `'_memory' in parsed`, `'continuity' in parsed._memory`, `'last_updated_at' in parsed._memory.continuity`. Because the YAML payload is a regular JS object, supporting both `_memory` and `_continuity` is mechanically a 6-line change at this site: read `parsed._continuity ?? parsed._memory?.continuity` (or equivalent) and emit a deprecation warning if the legacy path is taken. The trigger-matcher and resume-ladder follow the same pattern. This is the HIGH FEASIBILITY signal: there is no custom YAML AST parser to retrofit, no schema lockdown to migrate, no validation grammar to extend. Standard property-access fallback works. [SOURCE: `.opencode/skill/system-spec-kit/scripts/validation/continuity-freshness.ts:49-83`] [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/dist/lib/resume/resume-ladder.js:70` — regex `^_memory:\s*$/u` is the only regex-based fallback site, easily extended to `^_(memory|continuity):\s*$`]

3. **One parser surface — `resume-ladder.js` — uses a string-regex extractor (NOT yaml.load), which makes it the highest-cost rename site.** Lines 65-90 of `mcp_server/dist/lib/resume/resume-ladder.js` (compiled from `.ts`) implement `extractContinuityBlock` via line-by-line regex matching: `lines.findIndex((line) => /^_memory:\s*$/u.test(line))` finds the parent key, then `^  continuity:\s*$/u` finds the nested subkey, then indentation-anchored regexes extract leaf values. Adding a parser-fallback here requires extending two regexes (parent key + indentation depth) AND deciding the new key's nesting structure: does `_continuity` keep `continuity:` as a subkey (preserving `_continuity.continuity.last_updated_at`), or collapse to `_continuity.last_updated_at`? Recommendation: collapse the redundant nesting — a top-level `_continuity:` block with direct subkeys (`last_updated_at`, `current_focus`, `next_steps`, etc.) eliminates the awkward `_memory.continuity` double-nesting. The collapse is cleaner but increases the rename's structural blast radius (every fixture, schema check, and template emits a flatter shape). [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/dist/lib/resume/resume-ladder.js:65-113`]

4. **The migration corpus is BIGGER than spec.md §3 implies: 1,916 .md files in `.opencode/specs/**` use `_memory:` frontmatter (not the ~501 implementation-summary count).** Breakdown by filename:
   - `spec.md`: 344 files
   - `implementation-summary.md`: 334 files
   - `plan.md`: 334 files
   - `tasks.md`: 334 files
   - `checklist.md`: 282 files
   - `decision-record.md`: 92 files
   - `handover.md`: 11 files
   - `README.md`: 0 files

   Total: 1,731 files in the named-doc breakdown above + 185 other .md files (research iteration files, debug-delegation notes, etc.) for a grand total of 1,916. spec.md §3's "every existing implementation-summary doc" framing undercounts by **5.7×**. This is critical for Q4: a parser-fallback supports ALL 1,916 files for free; a hard-rename in-frontmatter requires a migration script with 1,916 file rewrites, plus the validation churn from `_memory.continuity` → `_continuity` schema changes propagating into every `validate.sh --strict` run during the migration window. [SOURCE: `find .opencode/specs -type f -name '*.md' -exec grep -lE '^_memory:' {} \;` returns 1,916; per-filename counts from same find with `-name '$f'` filter, iteration 3]

5. **VERDICT: parser-fallback is the unambiguously cheaper path.** Cost comparison:
   - **Hard rename in-frontmatter**: ~7-8 source-side parser changes + 1,916 file rewrites + a one-time migration tooling step + validation churn during the cutover. Estimated work: 4-6 hours of mechanical edits, plus one merge-storm risk window where in-flight work-in-progress branches break.
   - **Parser-fallback**: ~7 source-side parser changes (same touch surface) + zero file rewrites + a ParseWarn telemetry channel that surfaces "legacy `_memory:` key detected at $path; consider migrating to `_continuity:`" when the fallback fires. Files migrate organically as docs are touched in normal work. Estimated work: 2-3 hours of source edits + 1 hour of warning instrumentation.
   - **Recommended**: parser-fallback + soft migration. The "soft" component is a one-line `npm run migrate:continuity-key` script that batch-rewrites the 1,916 files when the team chooses to retire the alias (e.g., one or two releases after the rename ships). The parser keeps the alias forever in code; the corpus becomes alias-free at a chosen later checkpoint. This matches the iter-2 tier-stratification logic: T1 surfaces (the tools) get 2-release alias windows; the YAML key gets a permanent code-side alias and an OPTIONAL corpus-side migration. ## Ruled Out

- **Hard-rename of `_memory.continuity` in-frontmatter as the primary migration path** — finding 4 + finding 5 show the migration corpus is 1,916 files (5.7× the spec.md §3 estimate) and parser-fallback's mechanical work is strictly less than the hard-rename's. Hard-rename remains a SECONDARY option for the optional `npm run migrate:continuity-key` cleanup, but the primary path is parser-fallback. (iteration 3)
- **Adopting an existing glossary or terminology doc as the rename's vocabulary contract** — finding 6 shows zero glossary/terminology/vocabulary docs exist in the repo. The rename must scaffold this artifact greenfield. Any plan that assumes a "drop a section into existing glossary.md" approach is mechanically impossible. (iteration 3)
- **Building a TypeScript AST-aware vocabulary linter as the FIRST enforcement mechanism** — finding 7 shows the project has zero current vocabulary enforcement, and the existing `validate.sh` ecosystem is built around regex-based markdown linters (continuity-freshness, evidence-marker, spec-doc-structure). A regex-based `glossary-drift-lint.ts` aligns with existing patterns. AST-aware linting is a future upgrade path, not an MVP requirement. (iteration 3)

## Dead Ends

- **Treating the parser-fallback question as "is js-yaml capable of multi-key reads?"** — js-yaml is just a YAML→object parser; "alias support" is purely an application-layer concern. The right framing is "how many literal `'_memory.continuity'` strings are in the TS/JS source?" — answered in finding 1 (5 distinct sites) + finding 3 (one regex-based site). The js-yaml-capability question was a false abstraction.
- **Searching for glossary precedent in `templates/` or `addendum/` template-composition docs** — these folders own template *structure*, not vocabulary contracts. Confirmed by reading SKILL.md §1 Distributed Governance Rule which lists `glossary` zero times among the authored doc types.
