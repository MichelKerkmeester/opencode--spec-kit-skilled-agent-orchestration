## Analysis
The corpus is immediately useful as a frozen advisor-routing benchmark because it already has the fields the Phase 024 harness consumes: `id`, prompt text, and `skill_top_1` labels, and the harness has already processed all 200 prompts with an auditable 112/200 baseline (`smart-router-measurement.ts:19`-`smart-router-measurement.ts:23`; `implementation-summary.md:48`-`implementation-summary.md:52`). It also covers the categories Phase 027 cares about: direct writes, read-only suppression, memory-save/resume writes, mixed ambiguous requests, deep-loop workflows, and connector/skill-routing prompts (`labeled-prompts.jsonl:1`-`labeled-prompts.jsonl:60`; `labeled-prompts.jsonl:61`-`labeled-prompts.jsonl:200`). The evidence-map path is stale, but the harness default path and Phase 024 report establish the active corpus location (`deep-research-strategy.md:107`; `smart-router-measurement.ts:104`).

It should not be treated as enough data to promote learned production weights directly. The current labels are prompt-level top-1 skill labels, while the existing learned-combiner training API expects candidate-level feature vectors plus `[0,1]` relevance labels (`learned-combiner.ts:73`-`learned-combiner.ts:80`). We can derive candidate examples by running the advisor over each prompt, extracting each candidate's lexical, semantic, graph, explicit-mention, conflict, margin, and uncertainty features, and labeling the corpus top-1 skill as positive; but that derivation is a modeling step, not evidence that a live learned ranker is ready. This aligns with C4, which keeps learned scoring shadow/prototype until C7/C8 calibration exists (`iteration-019.md:45`-`iteration-019.md:63`).

The adopt-now split strategy should be deterministic and stratified: reserve a frozen 20% holdout, keep 80% for threshold tuning and optional shadow-model fitting, and stratify by a combined class such as `bucket + skill_top_1` so rare but critical classes do not disappear from the holdout. This fits the shipped shadow-scoring design, which already defaults to a seeded 20% holdout, supports intent classes, and uses stratified sampling when class labels are provided (`shadow-scoring.ts:28`-`shadow-scoring.ts:35`; `shadow-scoring.ts:95`-`shadow-scoring.ts:112`; `shadow-scoring.ts:235`-`shadow-scoring.ts:314`). The holdout must stay untouched for C8 promotion decisions; any prompt engineering, threshold tuning, feature selection, or learned weights should use the training split plus LOOCV where sample sizes are thin (`learned-stage2-weight-combiner.md:19`-`learned-stage2-weight-combiner.md:27`).

This yields a narrow but valuable adoption: use the corpus now for benchmark gating, error slicing, threshold calibration, and shadow learned-scoring experiments; reject any immediate production learned-weight replacement from this corpus alone. The Phase 024 caveats matter here: static measurement does not prove live AI tool-read behavior or causal AI-reasoning outcomes, so any learned scorer should remain shadow-only and paired with C6's performance lane and C8's accuracy target before promotion (`smart-router-measurement-report.md:50`-`smart-router-measurement-report.md:55`; `iteration-021.md:44`-`iteration-021.md:57`).

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Adopt the 019/004 corpus now as a frozen, deterministic, stratified benchmark and tuning/holdout dataset. Do not promote learned advisor weights from it directly; use learned scoring only in shadow until C8 defines target accuracy and live/holdout evidence clears the gate.
