## Analysis
Adopt precision safeguards now, but keep them deterministic and corpus-gated rather than turning Phase 027 into a learned-routing project. The repo already has the right primitives: confidence and uncertainty are separate, recommendations require a dual threshold by default, confidence is capped at `0.95`, score-margin and ambiguity calibration already exist, and previous graph work established the principle that indirect evidence needs separate accounting. External calibration guidance reinforces the same shape: confidence should track empirical correctness, and calibration data should be independent of the extraction/source data.

The main gap is that B3's new derived-trigger lane will currently look like direct intent evidence. `_apply_signal_boosts()` records `(signal)` reasons and adds capped boosts, but `has_boost = total_intent_boost > 0` means derived signals enter the high-confidence curve. The existing graph haircut then only counts `!graph:` reasons, so derived trigger phrases loaded from `graph-metadata.json.derived` can raise confidence without the same indirect-evidence penalty that prior ADRs applied to graph propagation. That is a precision risk because the 019 corpus shows the advisor already over-fires and wrong-skill-fires at meaningful rates before adding more derived phrases.

The safeguard framework should be: explicit author signals remain the high-precision lane; auto-derived triggers are treated as a separate derived lane with a lower per-phrase cap, a total derived contribution cap, and a post-calibration confidence haircut when derived evidence exceeds half the reasons. A derived-only match may be shown as a candidate, but should not pass the default dual threshold unless it is supported by at least one direct lexical/name/explicit keyword/phrase signal or by corpus validation proving that exact derived phrase meets a target precision floor. This mirrors the accepted ghost-candidate and graph-evidence separation ADRs without reopening the shipped 020-026 architecture.

Validation should use the 019/004-style corpus as the hard gate: run full 200-row top-1 metrics, per-skill precision/recall/F1, source-type slices, and before/after deltas whenever derived phrases are added or regenerated. For the first implementation slice, use conservative acceptance gates: no regression in P0 fixture pass rate, no increase in false fires on gold `none`, no per-skill precision drop greater than a small agreed tolerance for already-overfiring skills, and at least neutral or improved exact-match accuracy. If B6 later produces a broader corpus-statistics baseline, these gates can be tuned, but B4's minimum safeguard should ship with the derived-trigger feature.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Derived triggers are useful enough to adopt, but the evidence shows they need first-class evidence separation and corpus-gated precision checks before they can influence default routing. The current advisor already has the calibration hooks needed; B4 mainly requires extending those safeguards to `(signal)` / derived-trigger evidence.
