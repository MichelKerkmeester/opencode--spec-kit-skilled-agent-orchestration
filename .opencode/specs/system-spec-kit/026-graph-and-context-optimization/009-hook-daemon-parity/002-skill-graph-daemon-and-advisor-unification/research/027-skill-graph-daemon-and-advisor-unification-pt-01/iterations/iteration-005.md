## Analysis
The core daemon lifecycle should be MCP-owned for the first Phase 027 implementation slice: start when the Spec Kit Memory MCP server starts, run a startup skill-graph scan after stdio connection, optionally attach the real-time watcher behind `SPECKIT_FILE_WATCHER`, and cleanly shut down with the existing MCP process. This matches the current architecture: memory MCP already boots as a local Node stdio process from `opencode.json`, initializes its databases and skill-graph database in `context-server.ts`, connects with `StdioServerTransport`, then schedules startup memory and skill-graph scans. It also avoids introducing a second always-on process before A6 measures resource budget and before D-track consolidation decides final install/bootstrap surfaces.

macOS launchd is a viable packaging substrate later, but it should not be the Phase 027 core lifecycle. The launchd man pages show how a user agent would be managed: no self-daemonizing, respond to `SIGTERM`, use `Program` or `ProgramArguments`, set working directory/environment, and allow operator control through `bootstrap`, `bootout`, `enable`, `disable`, and `kickstart`. They also warn against `RunAtLoad` for speculative launches and strongly discourage `WatchPaths` because file monitoring is race-prone and may observe inconsistent files. That aligns with A1's Chokidar decision: if a launch agent is eventually offered, it should keep a Node process alive or on-demand kickstart a repo-owned watcher, not outsource file-change correctness to launchd `WatchPaths`.

For crash and recovery behavior, the repo already has most of the lifecycle shape needed in-process. `fatalShutdown()` closes background jobs, both watcher types, SQLite handles, and stdio transport under a deadline, and signal/exception/rejection handlers route into it. Watcher-triggered skill-graph indexing catches indexer failures, serializes in-process scans, and queues one follow-up rescan, while the shared file watcher has tested close-waits-for-in-flight-work and `SQLITE_BUSY` retry behavior. The one lifecycle gap is cross-process locking: memory indexing has a database-backed scan lease, but skill-graph indexing currently uses only process-local `skillGraphScanInProgress`. A8 should decide whether to add a skill-graph lease for multi-client or future launch-agent deployments.

Runtime hooks should treat daemon health as freshness, not as a hard dependency. The existing advisor hook path already probes freshness before prompt-time matching, skips or degrades absent/unavailable cases, invalidates cache on source-signature changes, and fail-opens on parse errors, unknown schemas, subprocess failures, and unhandled hook exceptions. That is exactly the recovery contract Phase 027 should preserve: hooks should never block the prompt waiting for daemon repair; they should surface live or stale advisor briefs only when safe, otherwise fail open and let `skill_graph_scan` or a future daemon supervisor repair state asynchronously.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Adopt the current MCP-owned lifecycle now: startup scan plus opt-in in-process watcher, graceful MCP shutdown, freshness-gated hooks, and fail-open prompt behavior. Keep macOS launchd/systemd-user packaging as a later install/bootstrap option, and reject launchd `WatchPaths` as the source of file-watcher truth.
