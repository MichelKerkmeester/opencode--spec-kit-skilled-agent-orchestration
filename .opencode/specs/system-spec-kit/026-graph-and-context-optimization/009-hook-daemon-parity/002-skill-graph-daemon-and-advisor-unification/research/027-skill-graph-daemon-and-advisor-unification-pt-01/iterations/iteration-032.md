## Analysis
The A and B designs are coherent at the durable-index boundary. A2/A3 intentionally keep the graph index's direct input narrow: refreshed `graph-metadata.json` files flow into the hash-aware SQLite indexer, and the current code persists `intent_signals`, `derived`, `source_path`, and `content_hash` together in `skill_nodes` (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-002.md:36`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-002.md:43`, `.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:118`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:129`). B3 and B5 use the same boundary: auto-derived trigger companions live in `graph-metadata.json.derived`, not in `SKILL.md` rewrites or a separate advisor sidecar (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-011.md:29`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-011.md:36`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-013.md:34`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-013.md:41`).

The only required adjustment is wording, not architecture: "freshness trigger" should name two stages. Stage 1 is A-scope detection: the watcher or freshness probe notices `SKILL.md`, source docs, key files, or `graph-metadata.json` drift, then marks the skill/advisor state stale (`.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/freshness.ts:136`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/freshness.ts:188`, `.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/freshness.ts:223`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/freshness.ts:263`). Stage 2 is B-specific regeneration: a per-skill derived-provenance fingerprint decides whether the derived trigger lane needs recomputation before the refreshed `graph-metadata.json.derived` is indexed (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-013.md:37`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-013.md:41`). That split matches A's two-tier scope instead of conflicting with it.

There is no evidence that B needs a separate dependency graph for the first implementation slice. A3's transaction already skips unchanged rows by content hash and refreshes changed entries inside one SQLite transaction (`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:520`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:586`), while B5 only needs extra provenance metadata inside the existing derived block to answer which generated scoring lane is stale. This keeps prompt-time freshness cheap, preserves A6's rejection of broad support-file watching for now, and lets A8's stale/unavailable fail-open behavior cover B-derived scoring when regeneration has not caught up (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-008.md:46`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-008.md:53`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-013.md:43`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-013.md:52`).

So X1 is coherent with one implementation caveat: the daemon must not treat current `*/graph-metadata.json` watch coverage as the complete B freshness story. That watcher is sufficient for durable graph-index updates today (`.opencode/skill/system-spec-kit/mcp_server/context-server.ts:1495`-`.opencode/skill/system-spec-kit/mcp_server/context-server.ts:1544`), but B-derived trigger freshness also depends on the provenance files that produced `derived.trigger_phrases`; those should be fingerprinted and checked by the derived refresh path rather than watched broadly by default (`.opencode/skill/skill-advisor/scripts/skill_graph_compiler.py:189`-`.opencode/skill/skill-advisor/scripts/skill_graph_compiler.py:231`).
