## Analysis
The compatibility rule should be "stable user-facing contracts, changing implementation authority." The evidence shows three different categories of consumers: operator/runbook surfaces that explicitly call the Python CLI, test and measurement harnesses that import or spawn `skill_advisor.py`, and runtime hook/plugin surfaces that already call TypeScript `buildSkillAdvisorBrief()` but still reach Python through `subprocess.ts`. Because D3 adopted MCP-native `advisor_recommend`, `advisor_status`, and `advisor_validate`, and D4 adopted native TypeScript subprocess elimination, new runtime integrations should move to `advisor_*` or native TypeScript APIs while the Python file remains as a shim for documented scripts.

Compatibility matrix:

| Surface | Current dependency | Phase 027 compatibility call |
|---|---|---|
| CLAUDE/AGENTS Gate 2 fallback | Direct `python3 .../skill_advisor.py "[request]" --threshold 0.8` | Keep documented fallback, but describe it as compatibility/diagnostics; primary remains hook or `advisor_recommend`. |
| Hook reference | `buildSkillAdvisorBrief()` primary, direct CLI fallback | Keep contract and disable flag; swap implementation behind `buildSkillAdvisorBrief()` from subprocess to native recommender after parity. |
| OpenCode plugin | Plugin -> bridge -> compiled TypeScript -> current subprocess path | Keep plugin and bridge API for D8, but route bridge to native `advisor_recommend`/brief producer; retain legacy disable env alias for one release. |
| Python CLI one-shot/stdin | `prompt`, `--stdin`, `--threshold`, `--uncertainty`, `--confidence-only`, `--show-rejections`, semantic hit flags | Keep as shim around native recommender so existing docs/scripts continue to return the same JSON recommendation array. |
| Python CLI health/validation | `--health`, `--validate-only`, `--force-refresh --health` | Map to `advisor_status`, `advisor_validate`, and shared generation/freshness refresh; preserve JSON shape where practical and add deprecation note for Python-specific fields only after tests migrate. |
| Python CLI batch | `--batch-file`, `--batch-stdin` | Keep in CLI for regression/measurement; do not expose `advisor_batch` in first MCP surface, matching D3 deferral. Internally call native recommender in-process. |
| Manual graph bootstrap | `init-skill-graph.sh` runs compiler/export plus `skill_advisor.py --health` | Keep as fallback for no-MCP environments; prefer MCP startup scan and `skill_graph_*` plus `advisor_status` in docs. |
| Regression/bench harnesses | Import or spawn `skill_advisor.py` | Preserve until native parity harnesses replace them; add native-vs-CLI parity gates before removing subprocess tests. |
| Manual testing playbook | CLI, hook, plugin, SQLite fallback, privacy/status checks | Update expected commands to include `advisor_*` equivalents while retaining CLI scenarios as compatibility smoke tests. |
| Disable env vars | `SPECKIT_SKILL_ADVISOR_HOOK_DISABLED`, legacy plugin alias | Keep both for backward compatibility; prefer the hook-disabled name as canonical. |

This is an adopt-now compatibility plan rather than a prototype because the repo already names the intended migration path. `skill_advisor.py` is no longer the desired implementation center, but it is still a broad contract surface: Gate 2 fallback, setup validation, troubleshooting, manual testing, graph bootstrap, regression, and benchmarks all rely on it. Removing or breaking it in Phase 027 would force simultaneous changes across operator docs, playbooks, and tests; keeping it as a thin compatibility wrapper lets D3/D4/D5 proceed without reopening shipped Phase 020-026 hook architecture.

The main design boundary is privacy and observability. Existing hook docs forbid raw prompts, subprocess stdout/stderr, and prompt excerpts in persisted or observable surfaces, while plugin status output is already prompt-safe. Native `advisor_*` tools and the compatibility CLI must preserve that boundary: direct CLI may still print recommendation JSON to stdout for the active caller, but hook/plugin/MCP diagnostics should keep using structured status, freshness, cache, generation, and error-code fields. `--force-refresh` should not clear arbitrary prompt data; it should trigger shared freshness/generation invalidation and then report via `advisor_status`.
