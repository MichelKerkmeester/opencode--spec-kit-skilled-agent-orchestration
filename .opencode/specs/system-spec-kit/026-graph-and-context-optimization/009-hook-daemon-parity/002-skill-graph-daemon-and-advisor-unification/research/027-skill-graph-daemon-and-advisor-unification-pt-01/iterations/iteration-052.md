## Analysis
The repo already points to a narrower parity contract than "identical scorer internals." r01's own measurement plan says native parity means the Python and TypeScript scorers must produce the same **top-1** and **pass-threshold** outcome on the canonical regression suite and the 200-prompt corpus before Python can be removed. That is stronger than "mostly similar top-1" but weaker than exact score or distribution matching, and it aligns with the actual migration risk: broken routing, abstain behavior, or false fires would surface in hooks, CLI fallback, and plugin integrations even if the internal score vectors looked numerically close. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/research.md:137-139,154-155`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/next-research-paths.md:86-92`

The current tests reinforce that operational definition. `advisor-corpus-parity.vitest.ts` runs all 200 labeled prompts, tracks `confidenceDelta`, and still treats top-1 mismatch as the actual failure condition. In parallel, `buildSkillAdvisorBrief()` only renders a visible brief from passing recommendations, and the Python scorer explicitly computes `passes_threshold` before ranking and output. Together that means the behavior users see is controlled by two things: which skill ends up top-1 after thresholding, and whether any recommendation clears the routing gate at all. If those two decisions match, small confidence or score deltas are observational noise, not a release blocker. `.opencode/skill/system-spec-kit/mcp_server/tests/advisor-corpus-parity.vitest.ts:88-111`, `.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/skill-advisor-brief.ts:119-134,148-159,436-444`, `.opencode/skill/skill-advisor/scripts/skill_advisor.py:2175-2195,2544-2565`

That leads to an adopt-now definition: parity for Python vs TypeScript should be **exact per-prompt agreement on (1) top-1 skill identity and (2) pass-threshold / abstain outcome across the full 200-prompt corpus and the full canonical regression dataset**. Exact equality of confidence values, full ranking order, or score distributions should not be required at ship time; the existing parity harnesses already observe confidence deltas without gating on them, and runtime-visible parity is enforced at the normalized brief layer instead. This keeps the contract strict where UX and safety depend on it, while avoiding brittle coupling to incidental scoring math that may differ between implementations without changing behavior. `.opencode/skill/skill-advisor/feature_catalog/feature_catalog.md:439-447`, `.opencode/skill/system-spec-kit/mcp_server/tests/advisor-corpus-parity.vitest.ts:29-35,97-111`, `.opencode/skill/system-spec-kit/mcp_server/tests/advisor-runtime-parity.vitest.ts:203-219`, `.opencode/skill/skill-advisor/scripts/skill_advisor_regression.py:193-197,223-229`
