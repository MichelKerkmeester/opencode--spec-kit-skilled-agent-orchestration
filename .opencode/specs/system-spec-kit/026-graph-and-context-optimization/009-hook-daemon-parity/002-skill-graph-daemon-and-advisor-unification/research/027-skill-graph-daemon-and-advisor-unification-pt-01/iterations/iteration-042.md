## Analysis
The current architecture makes **shared socket transport** the wrong adopt-now choice. The shipped server is a local stdio MCP process, not a socket daemon, and the runtime-facing surfaces are already standardized around local prompt hooks that call the same brief builder and renderer. Cross-runtime parity is therefore being enforced at the **shared payload + renderer layer**, not at a transport layer. Moving to a shared socket would mean inventing a new connection/auth/bootstrap surface for every runtime at the same time `027/001` is still supposed to stabilize watcher freshness and writer coordination. `opencode.json:19-24`, `.opencode/skill/system-spec-kit/mcp_server/context-server.ts:2114-2123`, `.opencode/skill/system-spec-kit/references/hooks/skill-advisor-hook.md:19-35,50-55,64-71`, `.opencode/skill/system-spec-kit/mcp_server/tests/advisor-runtime-parity.vitest.ts:203-219`

At the same time, **uncoordinated per-runtime spawn** is also not acceptable for the writer/watch path. The live skill-graph watcher currently protects only a single process through module-local booleans and debounce state, while the roadmap and risk register already identify scan lease/writer coordination and `SQLITE_BUSY` as load-bearing concerns. If Claude, Gemini, Copilot, and Codex each started their own active watcher+writer loop, they would multiply exactly the contention and freshness-publication hazards Phase 027 is trying to eliminate. `.opencode/skill/system-spec-kit/mcp_server/context-server.ts:1510-1527,1558-1561`, `.opencode/skill/system-spec-kit/mcp_server/context-server.ts:1485-1503`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/research.md:122-123,133-134`

The strongest adopt-now answer is a **workspace-scoped single-writer lease protocol** with runtime-local readers. The repo already has the right primitives: memory indexing uses a SQLite-backed lease with expiry and cooldown, advisor freshness is keyed to shared workspace artifacts, and the generation counter is a durable atomic sidecar write. That means any runtime can continue to do local prompt interception, but only the lease holder should run watchers, rebuild the skill graph, and publish new freshness generations; non-holders should stay read-only and rely on shared SQLite/JSON freshness state. This is the smallest design delta that actually solves cross-runtime sharing without inventing new RPC infrastructure. `.opencode/skill/system-spec-kit/mcp_server/core/db-state.ts:385-520`, `.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/generation.ts:44-49,103-121,161-192`, `.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/freshness.ts:80-97,136-188,223-263`

That lease-first design also fits the current prompt-time recovery contract. `buildSkillAdvisorBrief()` already checks freshness before cache or subprocess work and gracefully returns skipped, degraded, or fail-open results when shared artifacts are absent or unavailable. So non-leader runtimes do not need a shared daemon socket to remain safe: they can read live state when present, surface stale state when allowed, and otherwise bypass advisor output until the lease holder repairs freshness. A future standalone socket service can remain a later packaging experiment under D6, but it should not be the required protocol for `027/001`. `.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/skill-advisor-brief.ts:331-380,383-416,436-468`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-005.md:54-61,68-72`

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Adopt a workspace-scoped single-writer lease with runtime-local readers. It matches the repo’s existing freshness and lease primitives, avoids multiplying SQLite writer contention, and does not require inventing a new shared-socket transport before `027/001` ships.

- Keep “shared socket daemon” as an optional later packaging lane only if D6 needs a standalone service for install/bootstrap; do not make it a prerequisite for the first daemon foundation packet.
