## Analysis
The repo already contains the skeleton of the exact methodology G5 needs, so this should be an **adopt-now protocol**, not a fresh prototype. r01 fixed the policy boundary: semantic and learned channels stay out of live routing until `027/006` proves stronger corpus accuracy, holdout stability, safety-slice neutrality, and shadow-cycle readiness. C6 and C8 explain why: semantic or learned ranking is not allowed to "borrow" the advisor hook's prompt-cache latency assumptions, so promotion has to be evidence-driven and explicitly slower-moving than the deterministic lexical/graph lanes. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/research.md:127,137,147-159`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-021.md:39-46`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-023.md:40-47`

The concrete cycle should therefore mirror `runShadowEvaluation()` rather than inventing a bespoke advisor-only loop. Each cycle should: keep live fusion unchanged (`semantic-shadow = 0.00`; learned lane observational), select a deterministic seeded 20% holdout with intent stratification when available, replay the same prompt/query set through live and shadow scorers, log rank deltas plus cycle aggregates, and require two consecutive improving cycles before a "promote" recommendation is even eligible. The existing runtime/test contract already proves the desired properties: no live side effects, scheduled replay over recent queries, rollback on regression, and auditable cycle history. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-050.md:23-36`, `.opencode/skill/system-spec-kit/mcp_server/lib/feedback/shadow-scoring.ts:235-258,573-698`, `.opencode/skill/system-spec-kit/mcp_server/lib/feedback/shadow-evaluation-runtime.ts:399-455`, `.opencode/skill/system-spec-kit/mcp_server/tests/shadow-scoring-holdout.vitest.ts:1270-1291,1407-1444`

Learned signals need one extra guardrail layer, and the repo already has it. `learned-feedback.ts` starts with a one-week log-only shadow period, stores learned terms outside FTS, excludes top-3 clicks and too-new memories, applies a lower `0.7x` query weight only after the shadow period, and preserves a full audit trail. Adaptive ranking adds the second-stage promotion discipline: proposals are bounded (`maxAdaptiveDelta`), require minimum signal counts, and only move from `shadow` toward `promoted` after evaluation feedback. That means G5 does not need a single blended "semantic/learned" promotion jump; it should adopt a **two-lane shadow program** that shares the same cycle machinery but preserves lane-specific safeguards. `.opencode/skill/system-spec-kit/mcp_server/lib/search/learned-feedback.ts:7-24,74-90,257-327,415-465`, `.opencode/skill/system-spec-kit/mcp_server/lib/cognitive/adaptive-ranking.ts:39-55,84-95`, `.opencode/skill/system-spec-kit/mcp_server/tests/adaptive-ranking-e2e.vitest.ts:291-341`

So the recommended methodology is: (1) keep both lanes shadow-only and out of live threshold math; (2) run the G2 ablation matrix plus seeded holdout replay on the 200-prompt corpus and recent-query replays; (3) require the r01/C8 gate (`>=75%` full corpus, `>=72.5%` holdout, no safety regressions, no latency breach) and two consecutive positive shadow cycles; (4) only then let G6 assign the first non-zero promoted live weight, starting with bounded learned deltas before any prompt-time semantic contribution. This reuses proven repo machinery, keeps promotion reversible, and avoids mixing "can rank better offline" with "safe to affect a prompt-blocking hook now." `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/research.md:159-159`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-023.md:40-47`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-051.md:19-36`
