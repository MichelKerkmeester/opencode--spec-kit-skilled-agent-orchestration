# Iteration 7: hooks, agents, and startup-instruction boundaries

## Focus
This pass tested whether Copilot CLI's documented hook and agent surfaces can actually inject text into the live main-session prompt stream. I also verified whether the CLI exposes a startup-loaded instruction surface that could carry a static context payload instead.

[SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-copilot-cli] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features]

## Findings
- Copilot CLI now has a documented hook system with `sessionStart`, `userPromptSubmitted`, `preToolUse`, `postToolUse`, `agentStop`, and `subagentStop`; however, the hook reference says `sessionStart` output is ignored and `userPromptSubmitted` output is ignored because prompt modification is not currently supported, so hooks are not a direct parity path for injecting a startup payload or advisor brief into the model context. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/use-hooks] [SOURCE: https://docs.github.com/en/copilot/reference/hooks-configuration]
- The `--agent` / `/agent` surface selects `.agent.md` profiles from `.github/agents/` or `~/.copilot/agents/`, but GitHub describes custom-agent work as being carried out by a subagent with its own context window, i.e. delegated from the main agent rather than a documented main-session prompt replacement layer. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/about-custom-agents] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/create-custom-agents-for-cli]
- Copilot CLI does have a native startup-loaded instruction path: `AGENTS.md`, `.github/copilot-instructions.md`, path-scoped `.github/instructions/*.instructions.md`, and `$HOME/.copilot/copilot-instructions.md` are automatically loaded and added to requests. That makes custom-instruction files the closest built-in channel for a startup context payload, but only as static file content, not as dynamic hook-produced text. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features]
- Skills are documented as task-specific instruction bundles that Copilot loads when relevant or via `/skills`; they are not described as an always-on per-prompt injection surface, so they are closer to Claude-style skills than to Claude's `UserPromptSubmit` hook brief. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-agent-skills] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features]
- Plugins can package hooks, skills, custom agents, MCP configs, and LSP configs, but the plugin model is distribution/installation of those same surfaces, not a separate documented context-push API. Local inspection of `~/.copilot/config.json`, `~/.copilot/mcp-config.json`, and current `COPILOT_*` env vars also did not reveal a documented arbitrary prompt-payload environment channel. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/about-cli-plugins] [SOURCE: https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference]

## Questions Answered
- **Q1:** Copilot CLI has documented lifecycle hooks, but not a customer-facing prompt/session text injection API because `sessionStart` and `userPromptSubmitted` outputs are ignored. Evidence: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/use-hooks and https://docs.github.com/en/copilot/reference/hooks-configuration
- **Q2:** `--agent` / `/agent` route work into custom-agent profiles and subagents with separate context windows; the docs do not describe this as startup system-prompt injection into the main agent. Evidence: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/about-custom-agents and https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/create-custom-agents-for-cli
- **Q3:** Skills are task-scoped, just-in-time instruction/resource bundles, not a guaranteed per-prompt advisor brief lane. Evidence: https://docs.github.com/en/copilot/concepts/agents/about-agent-skills and https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features
- **Q5:** Copilot CLI has instruction files that are auto-loaded into requests, including repo, path, agent, and home-directory surfaces. Evidence: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions and https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features
- **Q7:** Documented env vars center on auth/control (`COPILOT_GITHUB_TOKEN`, `GH_TOKEN`, `GITHUB_TOKEN`, etc.), and local inspection surfaced only session/version flags rather than a payload-carrying prompt variable. Evidence: https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference plus local `~/.copilot`/env inspection

## Questions Remaining
- **Q4:** Whether MCP can ever push unsolicited context into the main session, versus only exposing pull-based tools.
- **Q6:** Whether ACP server mode gives any usable bidirectional prompt/context injection channel without wrapping Copilot itself as the client.
- **Q8:** What the least-friction shell-wrapper workaround looks like in practice, and what it breaks in interactive/TUI and stdin workflows.
- **Q2 residual nuance:** Whether `--agent` can ever replace the main agent in practice, or only explicitly select which custom subagent handles the task.
