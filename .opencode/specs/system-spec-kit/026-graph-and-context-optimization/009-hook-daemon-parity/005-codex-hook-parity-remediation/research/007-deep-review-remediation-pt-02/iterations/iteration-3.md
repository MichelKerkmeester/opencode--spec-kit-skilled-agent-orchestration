# Iteration 3: source-level hook parser contract
## Focus
This pass followed the iteration-2 recommendation by checking source-backed parser and schema behavior around the remaining contract gaps: non-`UserPromptSubmit` exit handling, unsupported prompt/tool transform fields, timeout failure shape, and multi-hook concurrency/order semantics.
## Actions Taken
1. Re-read the official Codex hooks documentation for the current event list, stdin/stdout, timeout, ordering, and event-specific blocker semantics: https://developers.openai.com/codex/hooks
2. Opened upstream hook discovery code to confirm the code-level event taxonomy and default timeout handling: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/discovery.rs
3. Opened upstream command execution code to confirm stdin delivery, timeout implementation, working directory, and inherited-process execution shape: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/command_runner.rs
4. Opened upstream dispatcher code to confirm same-event matching hooks are launched with `join_all` and then zipped back to handler order for result folding: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/dispatcher.rs
5. Opened upstream parser and schema code for `UserPromptSubmit`, `PreToolUse`, `PermissionRequest`, `PostToolUse`, and `Stop` output behavior: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/output_parser.rs and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/schema.rs
6. Rechecked local installed CLI/config evidence without modifying hook config: `codex-cli 0.122.0`, `~/.codex/hooks.json` still registers only Superset notify hooks, and the user config does not add extra hook definitions.
## Findings
- Q1: Source code confirms command hooks receive serialized JSON on stdin. `command_runner.rs` opens piped stdin and writes `input_json.as_bytes()` to the child process, while event modules build typed input structs before dispatch. Primary source: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/command_runner.rs
- Q2: `UserPromptSubmit` parser accepts either plain stdout as developer context or JSON `hookSpecificOutput.additionalContext`; invalid JSON-looking stdout becomes a failed hook run, not context. Relevance: confirms context injection surface and failure edge for Q2. Primary source: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/user_prompt_submit.rs
- Q3: For `UserPromptSubmit`, exit code `2` plus non-empty stderr is the blocker path; other non-zero exits are marked failed with `hook exited with code N` but do not set `should_stop`. Relevance: answers the ordinary non-zero fail-open vs blocker distinction for Q3. Primary source: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/user_prompt_submit.rs
- Q3/Q7: `PermissionRequest` has a separate decision contract: JSON deny or exit code `2` plus stderr yields a denial decision, any deny wins over allow, and if no hook decides Codex uses normal approval flow. Relevance: this is a hook-specific blocker/decline path, not a general turn-abort rule. Primary sources: https://developers.openai.com/codex/hooks and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/events/permission_request.rs
- Q4: Source confirms timeout uses `tokio::time::timeout(Duration::from_secs(handler.timeout_sec), child.wait_with_output())`; on timeout the result has no exit code, empty stdout/stderr, and error text `hook timed out after Ns`. Event parsers treat that as a failed hook run unless the event has already produced a supported decision. Primary source: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/command_runner.rs
- Q5: Multiple matching handlers are executed concurrently via `join_all`, not serially. Results are then zipped with the original handler list, so result aggregation observes handler selection order even though process execution is concurrent. Primary source: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/dispatcher.rs
- Q6: Code-level discovery currently enumerates exactly six configurable events: `PreToolUse`, `PermissionRequest`, `PostToolUse`, `SessionStart`, `UserPromptSubmit`, and `Stop`. This matches official docs. Primary sources: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/discovery.rs and https://developers.openai.com/codex/hooks
- Q7: `UserPromptSubmit` has no prompt-transform field in the output schema. Its schema allows only common output fields, legacy `decision`/`reason`, and `hookSpecificOutput.additionalContext`; `additionalProperties:false` rejects `updatedInput` or equivalent fields. Primary source: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/schema/generated/user-prompt-submit.command.output.schema.json
- Q7: Transform-like fields are explicitly not active elsewhere: docs say `PreToolUse.updatedInput` is parsed but unsupported and fails open, while `PermissionRequest.updatedInput`, `updatedPermissions`, and `interrupt` are reserved and fail closed. Relevance: reinforces that Codex 0.122.0 has blocker/decision surfaces, not prompt-rewrite semantics. Primary sources: https://developers.openai.com/codex/hooks, https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/output_parser.rs, and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/schema/generated/permission-request.command.output.schema.json
- Q8: Source-level command construction sets the hook child process current directory and stdio but does not call `env_clear` or construct a sanitized environment. Combined with iteration-2 empirical output showing custom `PROBE_CUSTOM_ENV`, fake `HOME`, and inherited `PATH`, the best-supported conclusion is parent/session environment inheritance with Codex path augmentation. Primary sources: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/command_runner.rs and `/tmp/codex-hook-probe/observed.log`
## Questions Answered
- Q3 advanced: source confirms `UserPromptSubmit` exit code `2` is the special block path; other non-zero exits fail the hook but do not stop the turn.
- Q4 advanced: source confirms timeout implementation and failure shape; iteration 2 already showed timeout does not abort a `UserPromptSubmit` turn.
- Q5 advanced: concurrent launch is source-confirmed, with result folding in selected handler order.
- Q6 advanced: source discovery and docs agree on six exposed events.
- Q7 advanced strongly: no `UserPromptSubmit` prompt transformation contract exists; block/deny/continue/additionalContext are the supported surfaces.
- Q8 advanced: source plus empirical evidence supports inherited environment, not a sanitized subset.
