# Iteration 3: D2 Root Cause — Generic Decision Placeholders

## Focus
Trace defect D2 to the exact branch that emits `observation decision N` / `user decision N`, then explain why JSON-mode runs can still hit that branch even when the originating payload was intended to carry real decisions.

## Hypothesis (from iter 1 architecture map)
Iteration 1 already isolated the lexical fallback in `decision-extractor.ts` as the only code path that manufactures the generic placeholder titles, so the likely failure was not template rendering but an upstream handoff miss that left `extractDecisions()` with no structured decision input. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-001.md:118-118`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:152-173`]

## Evidence collected
- The placeholder titles are created only by `buildLexicalDecisionObservations()`, which scans `observations[].narrative/title` and `userPrompts[].prompt`, extracts a sentence around a decision cue, and emits titles in the form `${source} decision ${index + 1}`. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:137-175`]
- `extractDecisions()` only treats two sources as authoritative before falling back: `_manualDecisions` and `observations` already typed as `decision`. If both are empty, it invokes the lexical fallback. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:367-388`]
- Both normalization paths are designed to surface JSON `keyDecisions` before the lexical fallback is even eligible. Fast-path clones them into `_manualDecisions` and, if needed, also synthesizes decision observations; slow-path does the same. [CITATION: `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:470-589`, `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:676-752`]
- The workflow explicitly normalizes preloaded JSON before extraction and overlays `_manualDecisions`, `observations`, and related normalized fields onto `collectedData` before calling `extractDecisions()`. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:617-637`, `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1014-1031`]
- The decision extractor does not read raw `keyDecisions` or a raw `decisions` field at all. By contrast, `conversation-extractor.ts` does read raw `keyDecisions` directly when constructing messages. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`, `.opencode/skill/system-spec-kit/scripts/extractors/conversation-extractor.ts:58-60`, `.opencode/skill/system-spec-kit/scripts/extractors/conversation-extractor.ts:87-107`, `.opencode/skill/system-spec-kit/scripts/extractors/conversation-extractor.ts:336-342`]
- The broken F1 output matches the lexical branch exactly: the saved decision titles are `observation decision 1` and `user decision 1`, with generic `Chosen Approach` content and 70% confidence. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/001-claude-optimization-settings/memory/06-04-26_13-47__extended-deep-research-run-13-iterations-total-8.md:226-300`]

## Root cause identified (file:line + trigger condition)
Primary root cause: `extractDecisions()` is contractually dependent on normalized decision carriers (`_manualDecisions` or decision-typed observations) and has no direct reader for raw JSON decision fields. The lexical fallback therefore triggers whenever `processedManualDecisions.length === 0` and `decisionObservations.length === 0`, provided some observation or user prompt still contains a decision cue sentence. That branch then manufactures placeholder titles such as `observation decision 1` / `user decision 1`. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:152-173`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:367-388`]

In other words, D2 is not caused by the lexical fallback outranking a direct JSON-field parser inside `extractDecisions()`. There is no such parser in that extractor. The intended higher-priority JSON handling lives upstream in `normalizeInputData()`, and only its normalized outputs are consumed by `extractDecisions()`. [CITATION: `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:566-584`, `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:676-752`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`]

## Verification
- Trigger condition verified: lexical fallback is gated behind `decisionObservations.length === 0 && processedManualDecisions.length === 0`. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:381-388`]
- Higher-priority path verified: both fast-path and slow-path normalization map `keyDecisions` into `_manualDecisions`. [CITATION: `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:566-584`, `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:751-752`]
- Workflow handoff verified: normalized `_manualDecisions` and `observations` are merged into `collectedData` before `extractDecisions()` runs. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:619-637`, `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1027-1031`]
- Raw-field asymmetry verified: `conversation-extractor.ts` still consumes raw `keyDecisions`, while `decision-extractor.ts` does not. That means a JSON payload can still visibly contain decisions elsewhere in the pipeline even if the decision extractor falls through to lexical mining. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/conversation-extractor.ts:87-107`, `.opencode/skill/system-spec-kit/scripts/extractors/conversation-extractor.ts:336-342`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`]
- Output shape verified: the broken F1 file shows the exact placeholder titles and generic option/rationale structure expected from the lexical fallback branch. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/001-claude-optimization-settings/memory/06-04-26_13-47__extended-deep-research-run-13-iterations-total-8.md:231-300`]

## Defect chain
JSON-mode payload enters workflow -> normalization is supposed to promote `keyDecisions` into `_manualDecisions` and/or decision observations -> `extractDecisions()` receives only `_manualDecisions`, `observations`, and `userPrompts` as decision inputs -> if normalized decision carriers are absent at that boundary, the extractor has no direct raw `keyDecisions`/`decisions` fallback -> lexical cue mining runs against summary-like observations and user prompts -> placeholder decisions are rendered into the final memory file. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:617-637`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:367-388`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/001-claude-optimization-settings/memory/06-04-26_13-47__extended-deep-research-run-13-iterations-total-8.md:231-300`]

## Findings (numbered list with citations)
1. The generic placeholder titles are emitted only by `buildLexicalDecisionObservations()`, not by the renderer or template layer. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:137-175`]
2. The lexical fallback is eligible only when both authoritative decision carriers are empty: `_manualDecisions` and decision-typed observations. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:367-388`]
3. The intended higher-priority JSON extraction path exists, but it is upstream in `normalizeInputData()`, not inside `extractDecisions()`. [CITATION: `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:566-584`, `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:676-752`]
4. `extractDecisions()` never reads raw `keyDecisions` or raw `decisions`; it depends entirely on normalized handoff fields. This is the design gap that lets lexical placeholders appear when normalization misses. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:381-388`]
5. The broken F1 file is consistent with the lexical fallback path and inconsistent with a successful `_manualDecisions` path, because the rendered titles are `observation decision 1` / `user decision 1` and the content collapses to generic `Chosen Approach` scaffolding. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/001-claude-optimization-settings/memory/06-04-26_13-47__extended-deep-research-run-13-iterations-total-8.md:231-300`]

## Question status (Q2 RESOLVED/PARTIAL/UNRESOLVED)
Q2 RESOLVED: the fallback triggers when normalized decision carriers are missing, and it can still affect JSON-mode runs because `extractDecisions()` does not directly consume raw JSON decision fields. The payload-specific reason F1 lost that normalized handoff remains an inference unless the original JSON artifact is inspected directly.
