- **Lines:** `skill_advisor_runtime.py:111-141,165-203`; `skill_advisor.py:105-116,140-152,180-187,1669-1694`; `graph-metadata.json:31-50`; `008-semantic-search-routing.md:59-73`; `skill-graph-schema.vitest.ts:1-12,40-156`
- **Severity:** P1
- **Description:** The live skill router does not turn per-skill `graph-metadata.json` `intent_signals` / `derived.trigger_phrases` into routing boosts. Instead, it builds its searchable corpus from `SKILL.md` frontmatter `name` + `description`, then scores requests through the hard-coded `INTENT_BOOSTERS`, `MULTI_SKILL_BOOSTERS`, and `PHRASE_INTENT_BOOSTERS` tables plus graph topology boosts. That leaves routing metadata looking authoritative in docs and playbooks even when changing it cannot affect direct routing.
- **Evidence:** `get_cached_skill_records()` only builds records from parsed frontmatter `name` and `description` (`skill_advisor_runtime.py:111-141,165-203`). The graph loader does parse and retain `intent_signals` into a `signals` map (`skill_advisor.py:105-116,140-152,180-187`), but the scoring path applies only hard-coded token/phrase tables and graph adjacency/family boosts (`skill_advisor.py:1669-1694`). Meanwhile the semantic-search playbook explicitly tells operators to inspect `mcp-coco-index` `graph-metadata.json` `trigger_phrases` / `intent_signals` as routing triage input (`008-semantic-search-routing.md:59-73`), and that metadata advertises phrases such as `"natural language code retrieval"` (`graph-metadata.json:31-50`). Local repro on 2026-04-16 from repo root with `python3 .opencode/skill/skill-advisor/scripts/skill_advisor.py "natural language code retrieval" --threshold 0.8` returned top-1 `sk-code-opencode` with `_graph_boost_count: 0`, proving the metadata phrase is not a live routing signal. The nearest automated coverage only checks MCP `skill_graph_*` dispatcher/schema routing, not advisor scoring parity with graph metadata (`skill-graph-schema.vitest.ts:1-12,40-156`).
- **Downstream Impact:** Skill authors can update `graph-metadata.json` trigger phrases and manual playbook expectations, rerun validation, and still ship unchanged or wrong routing behavior. That makes routing fixes look complete in metadata review while the actual advisor remains pinned to hand-maintained dictionaries and frontmatter prose.

### Finding R43-002
- **File:** `.opencode/command/spec_kit/assets/spec_kit_plan_auto.yaml`; `.opencode/command/spec_kit/plan.md`; `.opencode/skill/system-spec-kit/mcp_server/tests/transcript-planner-export.vitest.ts`; `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-planner-first.vitest.ts`
- **Lines:** `spec_kit_plan_auto.yaml:343-372,375-392`; `plan.md:93-99`; `transcript-planner-export.vitest.ts:146-217`; `memory-save-planner-first.vitest.ts:12-214`
- **Severity:** P2
- **Description:** `/spec_kit:plan` autonomous Step 0 still depends on an untyped expression DSL embedded in YAML strings. Branches such as `folder_state != populated` and `intake_only == TRUE` rely on a runtime interpreter understanding bare identifiers, mixed enum vocabularies, and uppercase boolean literals, but the asset does not define that grammar mechanically and nearby automated coverage never executes or validates those predicates directly.
- **Evidence:** The asset's intake gate and intake-only stop gate are raw string predicates: `when: "folder_state != populated"`, `when: "folder_state == populated"`, `when: "intake_only == TRUE"`, and `when: "intake_only == FALSE"` (`spec_kit_plan_auto.yaml:353-372,375-392`). The operator-facing `/spec_kit:plan` contract mirrors the same branch logic in prose (`plan.md:93-99`), so the workflow behavior still depends on all runtimes interpreting those string expressions the same way. The adjacent planner export suite exercises memory-save transcript routing and asserts only summary/status outputs (`transcript-planner-export.vitest.ts:146-217`), while the planner-first response suite is a response-builder contract test for follow-up actions and hints (`memory-save-planner-first.vitest.ts:12-214`). Neither suite evaluates the YAML branch predicates, the `TRUE`/`FALSE` literal handling, or the Step 0 `folder_state` comparisons.
- **Downstream Impact:** A parser/runtime change in expression evaluation can silently flip the control plane: healthy folders can re-enter intake, or `--intake-only` runs can fall through into Steps 1-7. Because the branch language is only expressed as strings, these failures would surface as workflow drift rather than a compile-time/schema-time error.

## Novel Insights
- The most important remaining Domain 4 seam is **authority drift between metadata and runtime**: graph metadata, playbook triage, and regression fixtures all imply a skill-local routing contract, but the live advisor still routes primarily from hard-coded token tables and frontmatter description text.
- The plan asset shows that string governance is not limited to documentation. Raw `when:` expressions now form a hidden control-language for workflow branching, but that language has no pinned interpreter contract or direct automated coverage.
- Re-reading `AGENTS.md` suggests the next unclaimed risk is semantic overbreadth rather than another prose-only finding: Gate 3 still uses trigger words like `analyze`, `decompose`, and `phase`, which likely collide with research/planning prompts, but that needs a prompt-by-prompt false-positive pass to avoid repeating Iteration 41.
