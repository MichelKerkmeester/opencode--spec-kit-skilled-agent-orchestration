# Iteration 5 (r2): Correctness — Content-router 8-category coverage audit

> r2 session. Overwrites r1's iteration-005 per explicit dispatch direction.

## Focus

Dimension: correctness. Audit `mcp_server/handlers/memory-save.ts` + `mcp_server/lib/routing/content-router.ts` for correct handling of all 8 routing categories (`narrative_progress`, `narrative_delivery`, `decision`, `handover_state`, `research_finding`, `task_update`, `metadata_only`, `drop`) now that the `memory/*.md` fallback is gone. Verify no code path that previously fell through to `memory/*.md` now errors or silently drops. Confirm `handover_state` preserves its contract (critical for `/memory:save` as canonical `handover.md::session-log` maintainer per v3.4.1.0:110-112).

## Findings

### P1

- **F034**: Content-router aborts saves on L1/L2 packets without `implementation-summary.md` or `spec.md` for categories `narrative_progress`, `narrative_delivery`, and `decision` (non-L3) — `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts:1263-1276` + `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts:1054-1061` + `.opencode/skill/system-spec-kit/mcp_server/lib/routing/content-router.ts:1020-1041` — `buildTarget` at `content-router.ts:1020-1041` resolves: `narrative_progress` → `implementation-summary.md`, `narrative_delivery` → `implementation-summary.md`, `decision` (non-L3) → `implementation-summary.md`, `metadata_only` → `spec-frontmatter` (which `resolveMetadataHostDocPath` at `memory-save.ts:1054-1061` resolves to `implementation-summary.md` OR falls back to `spec.md`). BUT the save path at `memory-save.ts:1263-1276` performs `fs.existsSync(targetDocPath)`; if false, it returns `status: 'abort', summary: 'Canonical routed save target document is missing'`. Problem: per CLAUDE.md/SKILL.md Rule 13, `implementation-summary.md` is "REQUIRED for all levels but created AFTER implementation completes, not at spec folder creation time". A brand-new L1 spec folder has only `spec.md`/`plan.md`/`tasks.md`/`implementation-summary.md` (placeholder from template) or sometimes lacks `implementation-summary.md` entirely. Before Path A, the fallback was a standalone `memory/*.md` write. Now, any mid-implementation save that routes to `narrative_progress` / `narrative_delivery` / `decision` on such a packet will abort with "Target document does not exist", leaving the operator no fallback. This is a regression-by-cutover: the runtime previously had a last-resort write path; now it hard-aborts. Mitigation present: `resolveMetadataHostDocPath` does fall back to `spec.md` (:1060) for `metadata_only`, but only for that ONE category — the other 3-4 categories that route to `implementation-summary.md` have no fallback. P1-CONTRACT-REGRESSION.

### P2

- **F035**: `handover.md` missing-file handling is identical to F034 pattern — `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts:1263-1276` + `.opencode/skill/system-spec-kit/mcp_server/lib/routing/content-router.ts:1030-1031` — `handover_state` category routes to `handover.md`, mergeMode `append-section`. If `handover.md` does not exist, the save aborts. Unlike `implementation-summary.md`, `handover.md` is NOT auto-scaffolded at spec-folder-creation time — operators are instructed to copy from `templates/handover.md` on first `/memory:save` call. v3.4.1.0:220 explicitly says "If your spec folder lacks a `handover.md`, copy from `.opencode/skill/system-spec-kit/templates/handover.md`." So the missing-file abort IS the documented UX — operators MUST initialize `handover.md` before the first save routes to it. This is a doc-contract UX friction: new-folder saves always abort on the first `handover_state` route until the operator manually seeds the file. Gate-relevant because v3.4.1.0:110-112 positions `/memory:save` as the canonical `handover.md` maintainer; the UX of "manually copy a template before first use" is awkward. P2-UX-FRICTION. Not a correctness bug in the runtime; a documented contract gap between `/memory:save` and template scaffolding.

- **F036**: `drop` category writes to `scratch/pending-route-{hash}.json` under the spec folder, but the `fs.existsSync(targetDocPath)` guard at `memory-save.ts:1264` is bypassed by the earlier `if (decision.refusal || decision.category === 'drop')` short-circuit at `memory-save.ts:1232-1245` — `.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts:1232-1245` — before targetDocPath resolution, `drop` returns `status: 'abort', success: false, status: 'rejected'`. So `drop`-routed content is NEVER written to disk; it is surfaced to the caller as a refused route. This is consistent behavior (drop = do not write). BUT: the `DROP_TARGET_PREFIX = 'scratch/pending-route-'` at `content-router.ts:44` and `buildTarget(..., 'drop')` at `content-router.ts:1038-1039` synthesize a docPath like `scratch/pending-route-{hash}.json` that the caller receives in `targetDocPath` of the abort response. Operators reading the response may believe the content was queued for review at that path — it was not. The phrase "Canonical routed save refused to merge content" (`:1239`) is ambiguous: does it mean "we refused and did nothing" or "we refused and queued for manual review at targetDocPath"? P2-DOC-AMBIGUITY. Code behavior is correct (content is dropped); the operator-facing response semantics are misleading.

## Ruled Out

- **All 8 categories resolve without `memory/*.md` fallback**: `buildTarget` at `content-router.ts:1020-1041` has complete `switch(category)` coverage — TypeScript `RoutingCategory` union is exhaustively discriminated (confirmed by the lack of `default:` and the clean compile against `ROUTING_CATEGORIES` at `:22-31`). No code path falls through to `memory/*.md`; no path produces a `memory/*.md` docPath.
- **Router refuse-to-route path goes to scratch/, not memory/**: `DROP_TARGET_PREFIX = 'scratch/pending-route-'` at `:44` is the refusal target; `refusalTarget(hash)` at `:1043-1049` returns `{ docPath: 'scratch/pending-route-{hash}.json', anchorId: 'manual-review', mergeMode: 'refuse-to-route' }`. Even if the caller ignored the abort short-circuit at `:1232-1245`, the path would not be a `memory/*.md` path.
- **`routing-prototypes.json` `legacy-memory` source-shape is a classifier input, not an output target**: Grep shows 10 prototype entries with `"sourceShape": "legacy-memory"` at `routing-prototypes.json:261-335`. These are training examples for the Tier-2 embedding classifier to RECOGNIZE legacy-memory-shaped content and route it to the correct canonical target. The sourceShape field influences classification, not the output target. Verified by `SOURCE_SHAPES` at `content-router.ts:34-41` being a frozen const tuple with zero consumers that decide write target. Retention is legitimate (training signal for backward-compat classification of historical memory-shaped input).
- **MCP handler `memory-save.ts` path guards**: `validateFilePath` from `@spec-kit/shared/utils/path-security` is imported at `memory-save.ts:12` and guards every caller-supplied path before it reaches the router. Path-based errors return before router dispatch. Not a routing-correctness concern.
- **`continuitySourceKey` override forcing `metadata_only` anchor**: `memory-save.ts:760` and `create-record.ts:111` both set `targetAnchorId` to `_memory.continuity` when `routeAs === 'metadata_only'` OR `continuitySourceKey !== null`. This is a deliberate override to force continuity writes to the frontmatter anchor regardless of router decision. Not a category-coverage bug.

## Dead Ends

- Searching for any residual branch in `memory-save.ts` that hand-crafts a `memory/*.md` path — none exists. All writes go through `resolveCanonicalTargetDocPath(specFolderAbsolute, decision.target.docPath)` which is either (a) `implementation-summary.md` / `spec.md` via the metadata-host resolver, or (b) `path.join(specFolderAbsolute, routedDocPath)` where `routedDocPath` is a canonical basename from `buildTarget`.
- Searching for a `memory/*.md` write in `save/create-record.ts`, `save/atomic-index-memory.ts`, `save/dedup.ts`, or `save/post-insert.ts` — the r1 iter-2 audit already confirmed MCP handlers have no independent `memory/*.md` write paths; re-grep of the same files shows the same result. None of these files write to `memory/*.md`.
- Searching for `handover_state` routing edge cases when `handover.md` is empty vs missing — empty file passes `existsSync`, `fs.readFileSync` returns empty string, then `upsertThinContinuityInMarkdown` or `anchorMergeOperation` treats it as a blank canvas and appends the requested section. No special handling needed; empty handover.md is a valid base case.
