# Iteration 50 — Domain 4: Stringly Typed Governance (10/10)

## Investigation Thread
I closed the remaining Domain 4 seams that earlier iterations left open: first, whether `AGENTS.md` Gate 3 has concrete false negatives for write-producing deep-research flows; second, whether the manual playbook runner's executable object-literal grammar already diverges from live scenario syntax in the checked-in playbook corpus.

## Findings

### Finding R50-001
- **File:** `AGENTS.md`; `.opencode/command/spec_kit/assets/spec_kit_deep-research_auto.yaml`; `.opencode/skill/system-spec-kit/scripts/tests/deep-research-reducer.vitest.ts`
- **Lines:** `AGENTS.md:182-186`; `spec_kit_deep-research_auto.yaml:159-167,521-526`; `deep-research-reducer.vitest.ts:53-58,145-146,264-270,286-295`
- **Severity:** P1
- **Description:** Gate 3's hard-block trigger list has a false-negative hole for write-producing deep-research resume flows. The only listed lexical triggers are verbs such as `create`, `edit`, `write`, `generate`, `analyze`, and `phase`, but the executable deep-research workflow treats `resume` as a first-class lineage mode that appends a `resumed` event to `deep-research-state.jsonl` and then continues into the iteration loop, where existence of a new `iteration-NNN.md` file and a newly appended JSONL iteration record are required outputs. A user request like "resume deep research" or "continue research" can therefore perform packet writes without matching the trigger words that supposedly guard *all* file-modifying work.
- **Evidence:** `AGENTS.md` defines Gate 3 as a pre-analysis hard block keyed to the literal trigger list `rename, move, delete, create, add, remove, update, change, modify, edit, fix, refactor, implement, build, write, generate, configure, analyze, decompose, phase` (`AGENTS.md:182-186`). But the actual deep-research workflow classifies `resume` as a valid session state, appends a `resumed` JSONL event, and skips directly into the loop (`spec_kit_deep-research_auto.yaml:159-167`); later in that same workflow, success requires both `file_exists: "{state_paths.iteration_pattern}"` and `jsonl_appended: "New iteration record in state.jsonl"` (`spec_kit_deep-research_auto.yaml:521-526`). The reducer test suite treats those writes as canonical runtime behavior: it seeds `lineageMode: 'resume'`, writes `research/iterations/iteration-001.md`, feeds a persisted `resumed` event plus later iteration records through the reducer, and asserts the resumed lineage is reflected in the registry/dashboard (`deep-research-reducer.vitest.ts:53-58,145-146,264-270,286-295`).
- **Downstream Impact:** The repository presents Gate 3 as a mandatory documentation/mutation guard, but the current lexical trigger contract does not mechanically cover at least one real write path that the runtime and tests already support. That means spec-folder setup can be skipped for research/resume prompts depending on wording, creating cross-runtime drift in a place the framework describes as a hard blocker.

### Finding R50-002
- **File:** `.opencode/skill/system-spec-kit/manual_testing_playbook/05--lifecycle/097-async-ingestion-job-lifecycle-p0-3.md`; `.opencode/skill/system-spec-kit/manual_testing_playbook/05--lifecycle/144-advisory-ingest-lifecycle-forecast.md`; `.opencode/skill/system-spec-kit/scripts/tests/manual-playbook-runner.ts`; `.opencode/skill/system-spec-kit/manual_testing_playbook/manual_testing_playbook.md`; `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-playbook-and-remediation/002-full-playbook-execution/spec.md`
- **Lines:** `097-async-ingestion-job-lifecycle-p0-3.md:35-37`; `144-advisory-ingest-lifecycle-forecast.md:35-36`; `manual-playbook-runner.ts:438-445,544-548,612-616`; `manual_testing_playbook.md:194-199`; `002-full-playbook-execution/spec.md:131-134`
- **Severity:** P2
- **Description:** The manual playbook's executable argument grammar is already inconsistent inside the live corpus. One checked-in lifecycle scenario uses explicit string syntax (`memory_ingest_status({ jobId:"<job-id>" })`), but another uses JavaScript shorthand (`memory_ingest_status({jobId})` / `memory_ingest_cancel({jobId})`). The runner does not define or inject a `jobId` variable into the evaluation scope; for any brace-prefixed arg source it bypasses default-tool args and executes the raw object literal with `Function(...)`. That makes the shorthand form dependent on undocumented JS scoping rather than a mechanically validated playbook DSL.
- **Evidence:** Scenario `144` uses the quoted placeholder form `memory_ingest_status({ jobId:"<job-id>" })`, while scenario `097` uses shorthand `memory_ingest_status({jobId})` and `memory_ingest_cancel({jobId})` (`144-advisory-ingest-lifecycle-forecast.md:35-36`; `097-async-ingestion-job-lifecycle-p0-3.md:35-37`). In the runner, `memory_ingest_status` and `memory_ingest_cancel` do have a safe default arg shape `{ jobId: runtimeState.lastJobId ?? 'job_missing' }`, but that default is used only when there is no explicit arg source (`manual-playbook-runner.ts:544-548,612-616`). As soon as the markdown contains `{...}`, the runner routes to `evaluateObjectLiteral()`, which evaluates `Function(\`return (${replaced});\`)()` with no bound `jobId` variable (`manual-playbook-runner.ts:438-445`). A direct Node repro against the same expression returns `ReferenceError: jobId is not defined`. This conflicts with the playbook's own release/readiness contract that coverage must include all scenario files and that active scenarios must not be silently dropped or depend on parser quirks (`manual_testing_playbook.md:194-199`; `002-full-playbook-execution/spec.md:131-134`).
- **Downstream Impact:** The playbook runner is not just executing markdown; it is executing an unstable, undocumented subset of JavaScript object syntax that already differs across shipped scenario files. Today that inconsistency is partly masked by manual/unautomatable classification, but any future automation expansion or copied scenario pattern can turn a documentation edit into a runner crash or a silently skipped lifecycle check.

## Novel Insights
- The open Gate 3 problem was not only false positives. The repository now has a concrete false-negative path: deep-research `resume` is a real, tested write flow, but the hard-block trigger vocabulary does not name it.
- The manual-playbook risk is deeper than "markdown is executed." The checked-in corpus already contains **two incompatible argument dialects** for the same tool family, and only one of them matches the runner's executable assumptions.
- Together, these show the final Domain 4 theme clearly: governance strings are no longer just drift-prone metadata; they are acting as hidden control languages whose gaps are exposed only when workflow/runtime files are read side-by-side with the test corpus.
