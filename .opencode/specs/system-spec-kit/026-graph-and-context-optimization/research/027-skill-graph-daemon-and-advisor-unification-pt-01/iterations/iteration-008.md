## Analysis
The failure-mode contract should be adopted now because the repo already contains most of the pieces, and the remaining gaps are crisp hardening requirements rather than research unknowns. SQLite WAL is the correct default for the derived skill graph because the repo already enables it for `skill-graph.sqlite`, and WAL fits prompt-time readers plus occasional scan writers. But WAL is not a complete failure model: SQLite still allows only one writer, can return `SQLITE_BUSY`, depends on same-host shared memory rather than network filesystems, and can suffer checkpoint starvation when readers overlap indefinitely. That means Phase 027 should keep the A4 single-transaction update boundary, but add A8 guardrails around writer coordination, `SQLITE_BUSY` handling, and network-filesystem degradation.

The watcher-side answer is also adoptable now. Chokidar's documented `atomic` and `awaitWriteFinish` options are built for editor write-then-replace and chunked-write patterns, and the current skill-graph watcher already uses `awaitWriteFinish`, add/change/unlink handlers, debounce, and non-fatal watcher-error logging. The shared memory watcher goes further: it treats ENOENT during realpath or hash reads as a benign race, verifies the resolved path stays inside the watched roots, retries `SQLITE_BUSY`, suppresses unchanged hashes, and keeps watcher callback failures from crashing the server. Phase 027 should lift those shared watcher failure semantics into the skill-graph daemon path instead of inventing a new policy.

The largest current gap is cross-process coordination and explicit corruption recovery for the skill graph. The skill-graph runner only has process-local `skillGraphScanInProgress`, while memory indexing already uses a database-backed lease with stale-lease expiry and user-facing E429 recovery. If Phase 027 keeps the daemon MCP-owned but later supports multi-client or launch-agent topologies, two processes can still contend on the same WAL writer. The adopt-now mitigation is to add a skill-graph scan lease modeled on `acquireIndexScanLease()`, then keep one queued in-process rescan as a convenience rather than as the only coordination mechanism.

Corrupted SQLite should degrade through derived-state recovery, not block prompts. The source of truth for this graph is the `graph-metadata.json` corpus; `skill-graph.sqlite` is a rebuildable index. The existing status and validate handlers can report staleness and graph invariants, but they currently return generic errors on database exceptions. A8 should set the implementation contract: advisor prompt-time paths treat graph/freshness failures as stale or unavailable and fail open; diagnostics should point to `skill_graph_status`, `skill_graph_validate`, and `skill_graph_scan`; and corrupted `skill-graph.sqlite` should have an explicit rebuild-from-source path analogous to memory MCP's checkpoint/force-scan recovery hints, with destructive DB replacement gated behind an operator action.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** The repo and upstream docs already identify the concrete failure modes and existing mitigation patterns: Chokidar stable-write handling, benign ENOENT races, WAL with single-writer/`SQLITE_BUSY` caveats, freshness fail-open behavior, and memory MCP scan leases/recovery hints. Phase 027 should adopt a failure-mode contract now, with a skill-graph scan lease and rebuild-from-source recovery as first-slice requirements.
