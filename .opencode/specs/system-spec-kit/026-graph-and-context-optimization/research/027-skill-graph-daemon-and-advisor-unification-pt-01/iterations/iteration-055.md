## Analysis
The repo evidence says promotion cannot be "75/72.5 and done." Those two accuracy numbers were already accepted in r01/C8, but they sit on top of a broader gate bundle: no safety regressions, no latency breach, and two passing shadow cycles before a shadow channel can leave observation-only mode. That is deliberately stricter than the normal routing threshold, where a recommendation can pass with `confidence >= 0.8` and `uncertainty <= 0.35`, or even with score `>=2.0` when intent boosts are present. Promotion therefore has to remain a **release gate for new signal authority**, not a restatement of ordinary recommendation readiness. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/027-skill-graph-daemon-and-advisor-unification-pt-01/research.md:154-160`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-023.md:40-47`, `.opencode/skill/skill-advisor/scripts/skill_advisor.py:2013-2037,2111-2136`

The missing numeric thresholds are already present in adjacent repo systems and should be adopted directly. First, shadow promotion must require the shipped weekly streak rule: deterministic 20% holdout, two consecutive improving cycles, and automatic rollback if the most recent cycle regresses. Second, learned signals must clear the existing warm-up and hygiene gates before they can count toward promotion evidence at all: one full week of log-only shadow mode, memory age at least 72 hours, no learning from top-3 results, and the existing rate caps and `0.7x` learned-weight haircut. Third, the promoted system must still satisfy the current compatibility gates: exact corpus top-1 parity, runtime-visible brief parity, regression-suite top-1 `>=0.92`, command-bridge false positives `<=0.05`, P0 pass rate `=1.0`, and zero failed regression cases. These numbers are already repo policy; G6's job is to require **all of them together** for promotion, not to invent a new scalar score. `.opencode/skill/system-spec-kit/mcp_server/lib/feedback/shadow-scoring.ts:28-35,239-257,539-565,672-689`, `.opencode/skill/system-spec-kit/mcp_server/tests/shadow-scoring-holdout.vitest.ts:1270-1290,1407-1444`, `.opencode/skill/system-spec-kit/mcp_server/lib/search/learned-feedback.ts:10-24,74-96,257-327,411-465`, `.opencode/skill/skill-advisor/scripts/skill_advisor_regression.py:193-197,223-229`, `.opencode/skill/system-spec-kit/mcp_server/tests/advisor-corpus-parity.vitest.ts:83-111`, `.opencode/skill/system-spec-kit/mcp_server/tests/advisor-runtime-parity.vitest.ts:203-219`

For the first actual live influence, the strongest in-repo numeric envelope is the adaptive-ranking path, not prompt-time semantic search. The repo already treats promoted adaptive effects as bounded deltas: start with `maxAdaptiveDelta = 0.08` and `minSignalsForPromotion = 3`, then relax only after later successful evaluation to `0.11` and `2`. By contrast, C6 still shows no approved prompt-time semantic latency ceiling, and G1/G5 kept `semantic-shadow` at `0.00` live weight pending this decision. So G6 should recommend a **staged promotion gate**: after the 75/72.5 + safety + parity + streak bundle passes, allow only the bounded learned/adaptive lane to gain live influence first (`delta <= 0.08`, `signals >= 3`), while semantic retrieval remains live weight `0.00` until a later packet proves a dedicated prompt-time latency budget. That makes promotion concrete without smuggling the slower semantic path into the hook hot path too early. `.opencode/skill/system-spec-kit/mcp_server/lib/cognitive/adaptive-ranking.ts:84-99`, `.opencode/skill/system-spec-kit/mcp_server/tests/adaptive-ranking-e2e.vitest.ts:317-341`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-021.md:39-47`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-050.md:23-36`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/027-skill-graph-daemon-and-advisor-unification-pt-01/iterations/iteration-054.md:28-40`
