## Analysis
The freshness trigger should reuse A3's skill-graph boundary for durable advisor scoring rows. B3 already chose `graph-metadata.json.derived` plus `skill-graph.sqlite` as the generated-trigger storage model, and the SQLite schema persists `derived`, `source_path`, `content_hash`, and `indexed_at` in the same row that the advisor reads for routing signals. That means a changed `graph-metadata.json` should invalidate exactly one skill node's derived and explicit graph-signal rows through the existing content-hash comparison, then refresh outgoing edges in the same transaction.

However, B5 should not rely on the current prompt-time mtime signal as the final invalidation truth. The runtime cache and advisor freshness probe already use mtimes/sizes for cheap cache invalidation and stale warnings, but the skill-graph indexer and status handler use SHA-256 content hashes for durable correctness. The right split is therefore: timestamps and sizes can wake the daemon, invalidate prompt caches, and mark the advisor `stale`; content hashes decide whether a source actually changed and whether scoring rows need to be regenerated or reindexed.

The only extra B-specific state needed is not a separate advisor dependency graph; it is a per-skill derived-provenance fingerprint inside the existing derived metadata. Because B1 inputs can include `SKILL.md`, `derived.source_docs`, and `derived.key_files`, the derived block should record a normalized input manifest/fingerprint for the files that fed `trigger_phrases`. If any file in that manifest changes content hash, that skill's derived trigger lane becomes stale and should be regenerated before it is allowed to contribute as fresh derived evidence. If only `graph-metadata.json` changes, the A3 indexer is sufficient; if an input file changes but `graph-metadata.json.derived` has not yet been regenerated, prompt-time output may still render as `stale` but B4's lower-trust derived lane should not be treated as fresh.

This aligns with A2 and A6: adopt narrow metadata watching now, but avoid broad live watches over every skill support file until a later benchmark/prototype proves the cost is acceptable. The first implementation can scan the small provenance manifest on startup, on prompt-time freshness probes, or on a debounced derived-refresh job, then update `graph-metadata.json.derived.last_updated_at` and the derived fingerprint before the existing A3 indexer writes SQLite. That keeps the system one graph deep while preserving enough provenance to answer "which scoring rows are stale?" per skill and per evidence lane.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Use the existing A3 skill graph and content-hash SQLite index as the durable invalidation authority, with a small derived-provenance fingerprint for B1 inputs inside `graph-metadata.json.derived`. Keep mtimes as fast stale/cache triggers only; do not create a separate advisor index or broad file watcher for the first slice.
