## Analysis
The best B2 answer is hybrid, but with a conservative split between extraction and enrichment. Regex-only is too brittle because B1 inputs include headings, body text, filenames, examples, and reference docs; however, regex/pattern extraction is still valuable for high-precision categories like failure terms, state terms, technical identifiers, and decision/action phrases. The existing shared trigger extractor already implements that shape: it strips Markdown, filters stop words, extracts n-grams, adds priority pattern extractors, deduplicates, caps phrase count, and exposes timing plus breakdown stats.

True TF-IDF should be adopted as the interpretable corpus filter, but not by adding Python scikit-learn into the advisor runtime. The scikit-learn docs confirm the right controls for corpus rarity and common-term filtering, but the local implementation is TypeScript-first and currently only approximates TF scoring inside one document. That means Phase 027 can adopt the existing pure-TypeScript extractor now, then let B6 define repo-level DF/IDF baselines so "skill", "context", "memory", and other corpus-common terms stop crowding out skill-specific terms.

Embeddings should not be the primary extraction pipeline for hook-time trigger generation. The repo has strong embedding infrastructure and current prices are low enough for offline enrichment, but provider selection can hit cloud APIs, HF local can incur a 15-30s first-load path, and vector search already treats >600ms as a warning. That is the wrong latency and explainability profile for the prompt-blocking advisor hook, especially when current trigger matching is documented as a no-embedding fast path.

The adopted design should therefore be: explicit metadata and names first; deterministic n-gram/pattern/technical-term extraction second; corpus-aware DF/IDF or BM25-style downweighting third; embeddings only as an offline/prototype-later enrichment channel for cluster labels, synonym suggestions, or evaluation against the labeled corpus. This keeps the B2 pipeline explainable enough for advisor briefs and leaves B4/B7 room to cap noisy or adversarial derived terms.

- **Call:** adopt now

- **Rationale:** Adopt a deterministic hybrid pipeline now: explicit signals plus the existing TypeScript n-gram/pattern extractor, upgraded with corpus-aware TF-IDF/DF statistics in B6. Keep embeddings as offline enrichment/prototype support rather than a hook-time trigger source.
