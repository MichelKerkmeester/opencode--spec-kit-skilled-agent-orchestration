## Analysis
The merge is feasible, but not as a literal replacement of advisor prompt-cache with the generic MCP `tool-cache.ts`. The generic cache is tool-output oriented: SHA-256 over tool name plus canonical args, 60-second default TTL, per-tool invalidation, in-flight dedupe, and write invalidation for memory retrieval tools. The advisor cache has stricter and more domain-specific requirements: HMAC keys to keep prompt text out of cache keys, source-signature participation in every key, token-cap and threshold dimensions, deleted-skill suppression, and separate freshness states that feed shared payload trust.

The shared opportunity is therefore at the primitive layer. Advisor freshness already implements the better domain model for Phase 027: source signatures over skill files, graph metadata, advisor scripts, SQLite/JSON graph artifacts, and a persistent generation counter. Those concepts should become a shared `lib/skill-advisor` or `lib/freshness` utility used by the daemon, native recommender, `advisor_status`, and any compatibility bridge. Similarly, the generic cache's bounded TTL, LRU, in-flight dedupe, stats, and lifecycle hooks are reusable patterns, but the advisor prompt cache must preserve HMAC keying and source-signature invalidation as first-class policy.

For the native TypeScript advisor from D4, adopt a two-layer arrangement now: keep exact prompt-result caching as an advisor-specific cache, factor common bounded-cache/source-signature/generation helpers so daemon rebuilds and MCP tools share freshness truth, and add advisor invalidation to the same write/rebuild pathway that currently invalidates memory tools. This avoids duplicating mtime/signature logic across the daemon and hook while keeping memory retrieval cache semantics separate from prompt privacy and skill freshness semantics.

The evidence also argues against persisting advisor prompt results in SQLite during Phase 027. Current tests assert process-local HMAC keys, short TTL behavior, cache-hit p95, replay hit-rate, and freshness invalidation. Changing storage would expand the migration beyond cache sharing and risk reopening shipped memory/cache architecture. The adopt-now scope should be primitive extraction plus shared invalidation signals, not a wholesale cache backend migration.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Adopt shared cache/freshness primitives now, but keep advisor prompt-result caching as a domain-specific wrapper over those primitives. The existing advisor cache already has source-signature, privacy, freshness, and latency contracts that the generic memory/tool cache does not satisfy by itself.
