## Analysis
The plugin should not be removed in Phase 027. D8 is explicitly about the OpenCode plugin/bridge relationship, and the current plugin owns an OpenCode-specific prompt hook: it extracts a prompt, calls `getAdvisorContext()`, and returns `additionalContext` from the bridge response rather than registering an MCP server or exposing the core advisor implementation itself (`.opencode/plugins/spec-kit-skill-advisor.js:343`-`.opencode/plugins/spec-kit-skill-advisor.js:380`). The install guide also notes that OpenCode MCP wiring is checked in while plugin registration is runtime-dependent and not represented in `opencode.json`, which makes the plugin a runtime adapter layer rather than duplicated MCP server ownership (`.opencode/skill/system-spec-kit/mcp_server/INSTALL_GUIDE.md:131`-`.opencode/skill/system-spec-kit/mcp_server/INSTALL_GUIDE.md:141`).

The bridge should remain, but its target should change. Today it is already a proxy into compiled system-spec-kit MCP-server code: it imports `buildSkillAdvisorBrief()` and `renderAdvisorBrief()`, renders the brief, and returns a compact JSON envelope (`.opencode/plugins/spec-kit-skill-advisor-bridge.mjs:93`-`.opencode/plugins/spec-kit-skill-advisor-bridge.mjs:128`). The next implementation should route that bridge to the native advisor runtime behind `buildSkillAdvisorBrief()` or to an internal equivalent of `advisor_recommend`, while preserving the plugin's existing OpenCode hook contract. A direct MCP-tool pass-through is conceptually right at the API boundary because D3 adopted `advisor_recommend`, `advisor_status`, and `advisor_validate` (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-026.md:47`), but the plugin process should keep the Node bridge pattern unless OpenCode exposes a reliable in-process MCP client. The sibling compact-code-graph plugin documents why: plugins avoid importing MCP-server native modules in the host runtime and instead call a plain Node bridge (`.opencode/plugins/spec-kit-compact-code-graph.js:9`-`.opencode/plugins/spec-kit-compact-code-graph.js:14`).

Deprecation should apply to implementation internals, not to the plugin surface. D4 adopted subprocess elimination and says `buildSkillAdvisorBrief()` should flip from Python subprocess to native TypeScript after parity (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-027.md:55`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-027.md:60`), while D7 says to keep the OpenCode plugin and bridge API during compatibility migration (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-030.md:55`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-daemon-parity/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-030.md:63`). Therefore the right post-consolidation posture is "adapter stays, backend consolidates": keep `spec-kit-skill-advisor.js` as the OpenCode hook adapter, keep `spec-kit-skill-advisor-bridge.mjs` as the process-isolation shim, and remove `subprocess.ts`/Python dependency from the primary runtime only after native parity.

The adapter must preserve privacy and operational controls. Current plugin status output is deliberately prompt-safe (`.opencode/plugins/spec-kit-skill-advisor.js:394`-`.opencode/plugins/spec-kit-skill-advisor.js:420`), manual tests assert bridge output and both disable paths (`.opencode/skill/skill-advisor/manual_testing_playbook/manual_testing_playbook.md:278`-`.opencode/skill/skill-advisor/manual_testing_playbook/manual_testing_playbook.md:288`), and hook privacy rules forbid raw prompt text, fingerprints, stdout, and stderr outside private cache (`.opencode/skill/system-spec-kit/references/hooks/skill-advisor-hook.md:585`-`.opencode/skill/system-spec-kit/references/hooks/skill-advisor-hook.md:607`). That argues against a broad deprecation/removal notice now: the user-visible plugin is a useful runtime-specific delivery mechanism, while the risky parts are the hidden subprocess and duplicated cache/freshness paths that D4/D5 already targeted for consolidation.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** Keep the OpenCode plugin and bridge as a thin runtime adapter, but route the bridge to the native advisor/`advisor_recommend` implementation after consolidation. Do not remove or deprecate the plugin surface now; deprecate only the Python subprocess backend and legacy alias once parity and documentation migration are complete.
