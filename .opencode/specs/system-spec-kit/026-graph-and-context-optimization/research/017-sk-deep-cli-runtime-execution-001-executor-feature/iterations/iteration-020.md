# Iteration 20: Dispatcher ordering and missing executor-audit runtime hook

## Focus
Inspect the production deep-loop dispatcher or workflow call site that runs non-native CLIs, confirm the exact ordering of iteration append, validation, executor-audit merge, and reducer sync, and check whether failure handlers preserve executor attribution when no clean iteration JSONL line exists.

## Actions Taken
1. Read the live autonomous command asset at `.opencode/command/spec_kit/assets/spec_kit_deep-research_auto.yaml` instead of the stale skill-local path reference, then traced the `step_dispatch_iteration` sequence for `cli-copilot`, `cli-gemini`, and `cli-claude-code`.
2. Read `.opencode/skill/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts` to verify what the validator actually checks before reducer sync.
3. Read `.opencode/skill/system-spec-kit/mcp_server/lib/deep-loop/executor-audit.ts` to confirm how executor attribution is supposed to be merged into the JSONL tail.
4. Ran an exact-match sweep over `.opencode/command/spec_kit` and `.opencode/skill/system-spec-kit/mcp_server` to see whether `appendExecutorAuditToLastRecord()` has any non-test production caller.

## Findings
- P1. The live workflow declares the ordering as `dispatch -> post_dispatch_validate -> record_executor_audit -> reduce-state`, which means executor attribution happens too late to influence validation and cannot rescue a malformed or missing tail record. Evidence: `.opencode/command/spec_kit/assets/spec_kit_deep-research_auto.yaml:494-581`, especially the validator block at lines 561-568 and reducer handoff at lines 579-581. Reproduction path: open the auto YAML and inspect `step_dispatch_iteration`; note that `record_executor_audit` is placed after `post_dispatch_validate`. Risk-ranked remediation candidate: move executor attribution into the first append of the iteration JSONL record, or run validation only after the audit merge has completed.
- P1. I did not find any non-test production caller for `appendExecutorAuditToLastRecord()`. The restricted exact-match sweep over the command asset and MCP deep-loop runtime returned only the helper definition itself, which means the YAML `record_executor_audit` stanza currently has no visible TypeScript/JavaScript runtime hook in this pass. Evidence: `.opencode/skill/system-spec-kit/mcp_server/lib/deep-loop/executor-audit.ts:26-53` plus exact-match search over `.opencode/command/spec_kit` and `.opencode/skill/system-spec-kit/mcp_server` returning only that definition. Reproduction path: `rg -n "appendExecutorAuditToLastRecord\\(" .opencode/command/spec_kit .opencode/skill/system-spec-kit/mcp_server --glob '!**/tests/**'`. Risk-ranked remediation candidate: either wire the helper into the command runner explicitly, or delete the dead declarative stanza and require `executor.kind` directly in the iteration JSONL contract so auditability is atomic rather than post-hoc.
- P1. Failure handling remains executor-blind when the iteration line is bad or absent. The validator only emits generic reasons (`iteration_file_missing`, `iteration_file_empty`, `jsonl_not_appended`, `jsonl_missing_fields`, `jsonl_parse_error`), and the YAML only promises a generic `schema_mismatch` escalation plus `stuck_recovery`; there is no executor-tagged fallback event for non-native runs when validation fails before a clean tail record exists. Evidence: `.opencode/skill/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:12-23`, `.opencode/skill/system-spec-kit/mcp_server/lib/deep-loop/post-dispatch-validate.ts:38-75`, `.opencode/command/spec_kit/assets/spec_kit_deep-research_auto.yaml:561-568`. Reproduction path: corrupt or suppress the expected iteration JSONL append, then inspect the failure reasons and emitted recovery path. Risk-ranked remediation candidate: add an `executor_failure` or `dispatch_failure` event emitted before validation escalation, carrying `executor.kind`, model, and raw validator reason so failure provenance survives even when the iteration record does not.
- P2. The skill documentation still points at a non-runtime YAML path (`assets/spec_kit_deep-research_auto.yaml` under the skill), while the live command asset is under `.opencode/command/spec_kit/assets/`. This is not the root bug, but it increases the chance of auditing the wrong artifact when chasing executor issues. Evidence: `.opencode/skill/sk-deep-research/SKILL.md:56` versus `.opencode/command/spec_kit/assets/spec_kit_deep-research_auto.yaml`. Reproduction path: follow the skill doc path literally and compare it to `rg --files . | rg 'spec_kit_deep-research_auto\\.yaml'`. Risk-ranked remediation candidate: fix the skill path reference so future incident/debug passes start from the real workflow asset.

## Questions Answered
- Q7 tightened further: the ordering question is now answered from the live workflow asset. Non-native iterations are validated before any declared audit merge, then handed to the reducer.
- Q7 tightened further: current failure handlers do not preserve executor attribution when the iteration JSONL tail is malformed or missing; they preserve only generic schema/recovery outcomes.

## Questions Remaining

- Phase 019+ still needs a product decision on whether executor provenance should be a required first-write field on every non-native iteration record or a separate failure event family.
