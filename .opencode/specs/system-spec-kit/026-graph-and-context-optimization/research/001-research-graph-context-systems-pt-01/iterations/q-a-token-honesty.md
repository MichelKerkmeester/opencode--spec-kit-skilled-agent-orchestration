## Per-System Deep Dive

### 001 Claude Optimization Settings — `ENABLE_TOOL_SEARCH` token claim
**Claim:** Deferred tool loading drops starting context from `45k` to `20k`, cuts tool-schema overhead from `20k` to `6k`, and therefore saves about `14,000` tokens "on every single turn." [SOURCE: 001-claude-optimization-settings/external/reddit_post.md:15-24]

**Method:** This is real field arithmetic, not synthetic constants. The source operator observed a before/after token delta, then multiplied it by average turns per session and total sessions. Phase 1 already narrowed the safe claim to upfront payload reduction rather than latency or ergonomics. [SOURCE: 001-claude-optimization-settings/research/research.md:36-60] [SOURCE: research/iterations/gap-closure-phases-1-2.json:10-55]

**Evidence:** The token delta itself is source-grounded, but the large aggregate numbers are not ledger-grade because the same post also contains unresolved denominator mismatches (`926` vs `858` sessions; `18,903` vs `11,357` turns). [SOURCE: 001-claude-optimization-settings/external/reddit_post.md:22-24] [SOURCE: 001-claude-optimization-settings/external/reddit_post.md:36-46] [SOURCE: research/iterations/gap-closure-phases-1-2.json:152-198]

**Honesty score (med):** More honest than the formula-driven claims because the savings vector starts from real observed token counts on real sessions, but still not publication-grade because the arithmetic cannot be fully reconciled and no first-tool ergonomics benchmark exists. [SOURCE: research/iterations/gap-closure-phases-1-2.json:10-55] [SOURCE: research/iterations/gap-closure-phases-1-2.json:152-198]

**What would falsify:** A matched A/B run showing that `ENABLE_TOOL_SEARCH` reduces upfront schema payload but causes enough first-tool lookup churn, latency, or extra follow-up turns that net task token use does not fall.

**Public-relevant takeaway:** Keep the config; do not reuse the Reddit extrapolation as a Public benchmark. Public can truthfully say "field evidence supports lower upfront schema payload," not "Public saves 14k tokens per turn." [SOURCE: 001-claude-optimization-settings/research/research.md:49-60]

## The Comparability Problem

These five numbers are not comparable as-is.

- `001` measures a configuration-induced per-turn schema reduction, then extrapolates to sessions. It is closest to "prompt overhead avoided." [SOURCE: 001-claude-optimization-settings/external/reddit_post.md:22-24]
- `002` measures one static summary artifact against an imagined manual exploration process. It is closer to "scan output vs manual repo spelunking." [SOURCE: 002-codesight/external/src/detectors/tokens.ts:23-55]
- `003` mixes a per-query marketing claim with aggregate session counters, but both denominators are fixed constants, not measured exploration traces. [SOURCE: 003-contextador/external/README.md:13-19] [SOURCE: 003-contextador/external/src/lib/core/stats.ts:75-101]
- `004` measures sample questions against an estimated corpus on a benchmark script. It is a per-corpus synthetic benchmark, not a live retrieval cost metric. [SOURCE: 004-graphify/external/graphify/benchmark.py:64-107]
- `005` is not a ratio at all. It optimizes session startup and analytics readiness through cached summaries and better producers. [SOURCE: 005-claudest/research/research.md:171-190] [SOURCE: 005-claudest/research/research.md:285-320]

Iter-4's matrix explains why this matters for Public: Public already dominates structural and semantic query while tying Claudest on observability. That means a fair comparison must hold capability constant and ask a narrower question: on the same tasks, how many tokens does each approach require to achieve the same answer quality? [SOURCE: research/cross-phase-matrix.md:12-17] [SOURCE: research/cross-phase-matrix.md:40-74]

Honest cross-system comparison therefore requires four invariants:

1. Same repo snapshot and same task set.
2. Same unit of comparison, reported separately for per-turn, per-query, per-session-start, and per-corpus batch cases.
3. Same answer-quality rubric.
4. Same counting authority for tokens.

Without those four controls, the five numbers stay marketing-adjacent rather than audit-grade.
