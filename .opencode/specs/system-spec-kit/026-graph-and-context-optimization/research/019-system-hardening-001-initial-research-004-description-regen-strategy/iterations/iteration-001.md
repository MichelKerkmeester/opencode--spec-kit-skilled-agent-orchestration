## Field Catalogue Draft

### Current regen path

- `generate-description.ts` is a thin CLI wrapper.
- Explicit `--description` mode builds a fresh object from the passed text and only directly carries forward `memorySequence` and `memoryNameHistory`.
- Default mode calls `generatePerFolderDescription(folderPath, basePath)`.
- `generatePerFolderDescription()` reads `spec.md`, extracts `description`, derives `keywords`, `specId`, `folderSlug`, `parentChain`, and always refreshes `lastUpdated`.
- `generatePerFolderDescription()` also loads the existing `description.json`, but only directly reuses `memorySequence` and `memoryNameHistory`.
- `savePerFolderDescription()` delegates the actual preservation decision to `getDescriptionWritePayload(desc, loadExistingDescription(folderPath))`.
- For schema-valid files, `getDescriptionWritePayload()` merges normalized existing data with canonical overrides, which means schema-recognized optional fields can survive a rewrite.
- For schema-invalid-but-parseable files, the write path uses `mergePreserveRepair(...)` behind `getRepairMergeSafe()`.

### Sample evidence

- 25/25 sampled files contained: `specFolder`, `description`, `keywords`, `lastUpdated`, `specId`, `folderSlug`, `parentChain`, `memorySequence`, `memoryNameHistory`.
- Minority fields in the same 25-file sample:
  - `title`: 8 files
  - `type`: 3 files
  - `trigger_phrases`: 2 files
  - `path`: 1 file
- Rich live examples:
  - `022-hybrid-rag-fusion/009-perfect-session-capturing/description.json` is large mostly because of long `memoryNameHistory`; optional rich fields are present but null.
  - `026-graph-and-context-optimization/007-release-alignment-revisits/description.json` contains populated `title`, `type`, and `trigger_phrases`.
  - `026-graph-and-context-optimization/010-search-and-routing-tuning/description.json` shows the same pattern as another populated coordination-parent file.

### Draft classification

#### Always derived/system-regenerated

- `specFolder`
- `specId`
- `folderSlug`
- `parentChain`
- `lastUpdated`

#### Derived from author input, but still regenerated by tooling

- `description`
- `keywords`

These are regenerated every run, but their source can be authored indirectly via `spec.md` content or directly via the explicit `--description` flag.

#### System tracking fields preserved across regen

- `memorySequence`
- `memoryNameHistory`

These are the only fields the generator itself explicitly carries forward before the save phase.

#### Can be authored / optional rich fields observed in valid live files

- `title`
- `type`
- `trigger_phrases`
- `path`

These appear to survive only because the save path merges normalized existing data for schema-valid files. Their preservation boundary therefore depends on schema support and normalization behavior, not on the generator object itself.

### Early invariants

- The regen object produced by `generatePerFolderDescription()` is intentionally narrow.
- Rich-field preservation, when it happens, is a save-path concern, not a generation-path concern.
- Any rich field not recognized by the schema-normalization layer remains at risk even if valid JSON is already present.
- The schema-error repair lane already has a merge-preserving concept, but this iteration did not yet evaluate whether it fully covers the same rich-field set as the schema-valid lane.
