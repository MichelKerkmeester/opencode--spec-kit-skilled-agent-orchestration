## 6. PER-QUESTION DEEP DIVE

### Q1

Q1 closed as a provenance and coupling problem, not a clock-skew problem. The strongest evidence shows that the historical refresh surface was already split across H-56-1 canonical-save side effects and a later description-only `jq` sweep, with no shared proof that `description.json` and `graph-metadata.json` were ever refreshed together for the whole sibling set.[.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-012.md:17-37][.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-012.md:63-82]

Later work tightened the story further. The current negative-delta cluster comes from newer graph-only regeneration on top of that historical baseline, while later description-only refreshes created the opposite polarity on other siblings. The result is a long-lived mixed-drift system with no same-pass lineage marker.[.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-027.md:31-73]

### Q2

Q2 produced the largest Phase-019+ scoping outcome. Valid `description.json` loads are mostly safe because canonical save mutates the existing snapshot in place. The destructive path appears only when the loader returns `null`, which currently happens for both parse failure and schema failure.[.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-028.md:15-35][.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-030.md:15-40]

Once that branch is taken, regeneration recomputes a short title-like summary from the packet's source specification file under the relevant spec folder, keeps only memory tracking fields and discards or flattens richer authored payload. The live corpus scan showed this is not a corner case. About one third of packet-level `description.json` files in the 026 tree are one invalid edit away from lossy repair.[.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-028.md:36-74][.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-029.md:18-42]

The final pass answered the implementation-design question. A deterministic repair helper is feasible for parseable schema-invalid JSON if canonical derived fields always win and only authored narrative or extension keys are preserved.[.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-030.md:42-81]

### Q3

Q3 resolved the vocabulary question cleanly. The readiness union remains eight values at the shared type level, but the code-graph sibling family only uses four of them in practice and no single handler reaches all four.[.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-014.md:17-44]

This matters because docs that say "shared canonical vocabulary" are true in a type-acceptance sense, but misleading in an operator-observable sense. `cached` is alive on hook and compact-cache surfaces, while `imported`, `rebuilt` and `rehomed` are effectively compatibility vocabulary on the audited handler surface.[.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/018-deep-loop-cli-executor/iterations/iteration-014.md:45-88]
