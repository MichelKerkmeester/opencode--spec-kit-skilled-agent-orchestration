## Summary
This iteration answered Q7 and Q8 by measuring the real size impact of compiling `domains` and `intent_signals` into `.opencode/skill/skill-advisor/scripts/skill-graph.json`, then tracing how CocoIndex semantic hits enter `skill_advisor.py` and interact with graph propagation. The main result is that the current `~2KB` target is only realistic for topology-only output. If the compiled artifact is expected to carry routing metadata as well, the practical target should move to roughly `3-4KB` with a compact indexed layout, because raw string-heavy inclusion lands closer to `5.2KB`. On the scoring side, semantic hits and graph boosts are complementary, but they are not independent: semantic evidence is blended first, then graph edges propagate that same evidence to related skills, which can legitimately help companion routing but can also double-count confidence.

## Findings
1. The current compiled graph is effectively at the edge of the stated 2KB budget, and the compiler is not actually emitting a fully minified payload today. The on-disk file is `1951` bytes including the newline, while the JSON payload itself is `1950` bytes (`wc -c .opencode/skill/skill-advisor/scripts/skill-graph.json`). Re-serializing the same object with true minified separators drops it to `1808` bytes, so about `142` bytes of the current budget are pure formatting overhead rather than topology (`.opencode/skill/skill-advisor/scripts/skill-graph.json:1`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/007-skill-advisor-graph/spec.md:31`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/007-skill-advisor-graph/spec.md:139`).

2. Adding raw `domains` and `intent_signals` verbatim to the compiled artifact blows past 2KB immediately. Across the 20 skill metadata files there are `104` domain entries and `73` intent-signal entries, with `88` unique domains and `73` unique signals. Using the compiler’s current JSON style, the file would grow from `1950` bytes to `3639` with `domains` only, `3536` with `intent_signals` only, and `5225` with both. That means the current 2KB target is only compatible with topology-only data, not with richer routing metadata (`.opencode/skill/skill-advisor/scripts/skill_graph_compiler.py:148-152`, `.opencode/skill/skill-advisor/scripts/skill_graph_compiler.py:234-292`).

3. The compiler currently validates `domains` and `intent_signals` but discards both during compilation, so the runtime graph cannot use the strongest packet-local routing metadata at all. This especially hurts zero-edge but high-value routing skills such as `mcp-coco-index`, `sk-doc`, `sk-git`, and `system-spec-kit`, because they contribute no adjacency yet still carry useful canonical signals in their source metadata. `system-spec-kit` is the clearest example: it has no compiled outbound or inbound edge presence, but its `graph-metadata.json` still contains routing-critical signals like `spec folder`, `save context`, `save memory`, `spec kit`, and `memory search` that never make it into `skill-graph.json` (`.opencode/skill/system-spec-kit/graph-metadata.json`, `.opencode/skill/skill-advisor/scripts/skill-graph.json:1`, `.opencode/skill/skill-advisor/scripts/skill_graph_compiler.py:249-291`).

4. A compact indexed single-file format can keep rich metadata in a reasonable range, but it still does not fit under 2KB. I simulated an indexed representation with short keys, shared vocab tables, aligned skill arrays, and integer-coded edge weights. That full single-file design came out to `3563` bytes for the current 20-skill corpus. For comparison, the same indexed approach for edge topology alone is only `776` bytes, which shows the cost is overwhelmingly in routing vocabulary rather than graph structure. This makes `3-4KB` the realistic sweet spot if a single compiled artifact is supposed to carry both topology and canonical routing metadata.

5. The semantic-search pipeline and graph pipeline are intentionally stacked, not isolated. In `analyze_request()`, the advisor first adds the auto-semantic intent hint for `mcp-coco-index`, then blends CocoIndex hits via `_apply_semantic_boosts()`, then applies token and phrase boosters, and only after that applies `_apply_graph_boosts()` and `_apply_family_affinity()`. Explicit skill-name boosts happen later and therefore do not cascade through the graph. The practical effect is that semantic hits can seed related skills through graph propagation, but explicit naming cannot (`.opencode/skill/skill-advisor/scripts/skill_advisor.py:1128-1154`, `.opencode/skill/skill-advisor/scripts/skill_advisor.py:1465-1499`, `.opencode/skill/skill-advisor/scripts/skill_advisor.py:1500-1510`).

6. A single semantic hit can materially change companion-skill routing because graph propagation is additive and happens before confidence/uncertainty are calculated. In a controlled simulation, `figma css` without semantic hits produced `mcp-figma` as the only passing recommendation, while `sk-code-web` (`0.79`) and `mcp-code-mode` (`0.78`) both stayed below threshold. Injecting one semantic hit for `mcp-figma` at score `0.90` flipped both companions to passing (`sk-code-web` `0.81`, `mcp-code-mode` `0.85`) because the semantic boost of `2.7` propagated through `enhances(0.4)` and `depends_on(0.9)`. So graph boosts and semantic boosts are complementary, but they absolutely compound (`.opencode/skill/skill-advisor/scripts/skill_advisor.py:1147-1149`, `.opencode/skill/skill-advisor/scripts/skill_advisor.py:1496-1498`).

7. The main double-counting risk is epistemic, not just numeric: graph-propagated evidence is treated like fresh evidence when confidence and uncertainty are computed. `calculate_confidence()` only checks whether any intent-style boost exists, and `calculate_uncertainty()` lowers uncertainty as the raw match count increases. Because graph reasons such as `!graph:depends(...)` and semantic reasons such as `!semantic(...)` both go into the same `matches` list, a skill can look more certain even when the graph signal is only a transformation of the original semantic hit rather than an independent confirmation. This is most visible on medium-strength companion edges, not on already-dominant phrase matches like `spec folder review`, where the added semantic hit barely changes the output (`.opencode/skill/skill-advisor/scripts/skill_advisor.py:1244-1289`, `.opencode/skill/skill-advisor/scripts/skill_advisor.py:1292-1315`).

8. There is also an operational failure mode in the CocoIndex interaction: built-in semantic search fails closed and effectively becomes invisible to the advisor. `_cocoindex_search_builtin()` returns an empty hit list on any non-zero `ccc` exit or subprocess error. In this sandbox, direct `ccc search` failed because the daemon wanted to write `~/.cocoindex_code/daemon.log`, which is outside the writable sandbox; in normal advisor execution that would collapse to `[]` with no routing-visible error. So part of the Q8 answer is that graph/semantic interaction can disappear entirely in restricted environments, not because the logic is wrong but because semantic retrieval is silently suppressed upstream (`.opencode/skill/skill-advisor/scripts/skill_advisor.py:1038-1056`).

9. The best compiled format depends on whether we want one artifact or two. If the goal is one authoritative runtime artifact, the best balance I found is a compact indexed structure with: sorted skill IDs array, family vocab plus per-skill family index, edge triples `[edgeType,targetIndex,weightTenths]`, domain vocab, intent vocab, and a metadata matrix aligned to the skill array. That gets rich metadata into roughly `3.6KB` today. If the 2KB ceiling is non-negotiable, then the system should keep `skill-graph.json` topology-only and emit a second compact routing-metadata sidecar instead of trying to force both concerns into a single tiny file.

10. The 2KB target should be relaxed if the project wants compiled routing metadata to become canonical. As long as `intent_signals` and `domains` stay outside the compiled artifact, hardcoded intent tables in `skill_advisor.py` will remain the primary source of truth. Relaxing the target to `3-4KB` would allow the graph system to carry the data needed to replace or auto-generate much of that manual routing layer, especially the zero-edge central skills and the already-identified `intent_signals` duplicates. Keeping the 2KB target is still reasonable only if “compiled graph” explicitly means “topology-only accelerator,” not “full runtime routing substrate.”

## New Questions
1. Should graph-propagated boosts be marked as derived evidence so they do not reduce uncertainty the same way direct semantic hits and phrase hits do?
2. Do we want one compiled runtime artifact at roughly `3.5KB`, or do we want to preserve the topology-only `~2KB` contract and add a second compact routing-metadata sidecar?
3. Should `_cocoindex_search_builtin()` surface a diagnostic reason when semantic search fails, so graph/semantic interaction is debuggable instead of silently degrading to keyword-only routing?
