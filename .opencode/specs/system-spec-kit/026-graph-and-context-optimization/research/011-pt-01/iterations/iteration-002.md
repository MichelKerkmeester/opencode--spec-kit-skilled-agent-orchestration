## Summary
Reviewed `_apply_graph_boosts()` and `_apply_family_affinity()` in `.opencode/skill/skill-advisor/scripts/skill_advisor.py` plus the compiled graph in `.opencode/skill/skill-advisor/scripts/skill-graph.json`. The snapshot pattern is correctly preventing same-pass cascades, but the current damping values make graph boosts mostly secondary nudges, with `siblings` and `family` especially close to dormant unless a skill already has a strong direct signal. Circular enhancement can add a bounded one-hop bonus when both sides are already seeded, but it does not create a runaway feedback loop.

## Findings
1. The current comparison baseline is slightly narrower than the prompt suggests. In the live code, `INTENT_BOOSTERS` spans `0.3` to `2.5`, not `3.2` (`skill_advisor.py` lines 321-542). The larger values come from later routing stages such as explicit skill-name matching, which adds `2.5 + 0.3 * (len(matched_variants) - 1)` after graph boosts (`skill_advisor.py` lines 1500-1509). That means the graph dampers should be compared mainly against a `0.3-2.5` intent-boost scale, with occasional larger post-graph explicit boosts.
2. `enhances = 0.30` is conservative but still meaningfully active. With current compiled edge weights (`0.3`, `0.4`, `0.5`, `0.7` in `skill-graph.json`), an `enhances` edge needs a source score of roughly `1.111`, `0.833`, `0.667`, or `0.476` respectively to survive the `>= 0.1` floor in `_apply_graph_boosts()` (`skill_advisor.py` lines 98-104). Relative to the direct boosters, this makes `enhances` act like a secondary hint rather than a primary router, which feels reasonable for transitive help.
3. `depends_on = 0.20` is slightly underpowered for a relation that is semantically stronger than `enhances`. With current `depends_on` weights (`0.7`, `0.8`, `0.9`), the source score must reach about `0.714`, `0.625`, or `0.556` to register (`skill_advisor.py` lines 114-120). That keeps prerequisites from overpowering direct intent, but it also means a fair number of medium-strength prerequisite relations will vanish unless the source skill already has a solid direct boost.
4. `siblings = 0.15` is extremely gated. With sibling weights of only `0.4` and `0.5`, the source score must be about `1.667` or `1.333` before any sibling boost survives the floor (`skill_advisor.py` lines 106-112). On the current scale, that means sibling propagation usually happens only when a skill is already strongly routed, so it behaves more like a tie-breaker than a meaningful discovery aid.
5. `family = 0.08` is the most restrictive signal and is effectively a last-mile fill-in, not a general affinity mechanism. `_apply_family_affinity()` requires at least one member in the family to already exceed `1.0`, then separately requires `max_boost > 1.5`, and only boosts members that are still at `<= 0` (`skill_advisor.py` lines 133-143). Since `0.1 / 0.08 = 1.25`, the explicit `> 1.5` guard is the real limiter. In practice, family affinity only fires after a very strong direct route, so it looks intentionally safe but close to dormant.
6. The snapshot pattern in `_apply_graph_boosts()` is correct for preventing same-pass cascades. The function copies `skill_boosts` into `snapshot = dict(skill_boosts)` before iterating (`skill_advisor.py` lines 90-93). Any boosts written into `skill_boosts` during the loop are not revisited until a future call, and the advisor only calls `_apply_graph_boosts()` once per request (`skill_advisor.py` lines 1496-1498). That blocks the classic feedback-loop bug where a newly boosted node immediately boosts its own neighbors in the same pass.
7. `_apply_family_affinity()` is also bounded, although for a different reason. It computes the family’s `boosted` members and `max_boost` once at the start of each family iteration, then only grants affinity to members whose score is still `<= 0` (`skill_advisor.py` lines 133-143). Newly boosted family members do not recursively create more family boosts in the same family pass, and the compiled families in `skill-graph.json` are disjoint, so there is no cross-family bouncing path in the current data.
8. Circular enhancement exists in the current graph, but it is bounded one-hop amplification rather than a runaway loop. Real mutual `enhances` pairs in `skill-graph.json` include `sk-code-review <-> sk-code-opencode`, `sk-code-review <-> sk-code-full-stack`, and `sk-code-review <-> sk-code-web`. If only one side is initially seeded, only the opposite side gets boosted in that pass. If both sides are already seeded, each side can add one increment from the other side’s snapshot score. Example with a mutual `0.7` pair: starting from `2.0` and `2.0`, one pass ends at `2.42` and `2.42`; it does not keep compounding because the loop never revisits the new scores.
9. No skill in the compiled graph has both `enhances` and `depends_on` edges to the same target. I checked `.opencode/skill/skill-advisor/scripts/skill-graph.json` directly and found zero overlaps of that form. I also spot-checked the source `graph-metadata.json` files and found no raw overlap there either. So the current graph is not double-counting the same relationship through both relation types from a single source skill.

## New Questions
1. Should `depends_on` be at least as strong as `enhances`, given that prerequisites are often more actionable than “helps with” relations?
2. Is the near-dormant behavior of `siblings` and `family` intentional, or was the goal for them to contribute more to ambiguous prompts?
3. Should mutual `enhances` pairs remain allowed, or would it be safer to reserve bidirectional relationships for `siblings` and keep `enhances` directional?
4. If explicit skill-name boosts are the strongest routing signal and they happen after graph propagation, should graph boosts run again afterward, or is the current one-pass ordering a deliberate guardrail?
