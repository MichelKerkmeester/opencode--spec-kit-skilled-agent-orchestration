# Iteration 17 — Domain 1: Silent Fail-Open Patterns (7/10)

## Investigation Thread
I re-read the requested runtime seams against Iterations 003 and 011-015 plus the Phase 015 review, then only kept degradations that were still additive. This pass focused on two residual shape-truth gaps: `code_graph_query` responses that stay `ok` even when the graph row set is internally inconsistent, and post-insert save responses that still report enrichment as `ran` after one or more enrichment stages threw and degraded to warnings.

## Findings

### Finding R17-001
- **File:** `.opencode/skill/system-spec-kit/mcp_server/handlers/code-graph/query.ts`
- **Lines:** `442-559`
- **Severity:** P2
- **Description:** One-hop `calls_*` and `imports_*` queries silently degrade dangling graph edges into successful-looking relationships. Trigger: `queryEdgesFrom()` or `queryEdgesTo()` returns an edge whose referenced source/target node no longer exists. Caller perception: the graph query found a real structural relationship that simply lacks optional file metadata. Reality: the graph is internally inconsistent, but the handler falls back to raw `edge.targetId` / `edge.sourceId`, keeps top-level `status: "ok"`, and emits no corruption marker.
- **Evidence:** The DB layer explicitly permits null node joins: `queryEdgesFrom()` / `queryEdgesTo()` return `{ edge, targetNode: null }` or `{ edge, sourceNode: null }` when the lookup by `symbol_id` misses [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/lib/code-graph/code-graph-db.ts:503-507`, `.opencode/skill/system-spec-kit/mcp_server/lib/code-graph/code-graph-db.ts:521-525`]. The handler then serializes `target: entry.targetNode?.fqName ?? entry.edge.targetId` and `source: entry.sourceNode?.fqName ?? entry.edge.sourceId`, still wrapping the payload in `status: 'ok'` [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/handlers/code-graph/query.ts:447-459`, `.opencode/skill/system-spec-kit/mcp_server/handlers/code-graph/query.ts:474-486`, `.opencode/skill/system-spec-kit/mcp_server/handlers/code-graph/query.ts:501-512`, `.opencode/skill/system-spec-kit/mcp_server/handlers/code-graph/query.ts:527-538`, `.opencode/skill/system-spec-kit/mcp_server/handlers/code-graph/query.ts:551-565`]. Direct tests only exercise edges with populated node objects and never force the null-node branch [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/tests/code-graph-query-handler.vitest.ts:109-145`].
- **Downstream Impact:** Graph-backed debugging, dependency tracing, and impact analysis can operate on phantom symbol IDs without surfacing that the code graph is partially corrupt or needs reindexing. Consumers that trust `status: "ok"` plus a returned edge list will treat this as authoritative graph truth.

### Finding R17-002
- **File:** `.opencode/skill/system-spec-kit/mcp_server/handlers/save/post-insert.ts`
- **Lines:** `106-109, 126-129, 148-151, 174-177, 201-214`
- **Severity:** P2
- **Description:** Exception-driven enrichment failures still report `executionStatus.status = 'ran'`. Trigger: any enrichment step throws (causal links, entity extraction, summary generation, entity linking, or graph lifecycle). Caller perception: post-insert enrichment completed, with at most advisory warnings. Reality: one or more stages failed and their per-step booleans remain `false`, but the only machine-readable execution status still says `ran` rather than partial or failed.
- **Evidence:** Each enrichment lane catches its own exception and only logs a warning [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/handlers/save/post-insert.ts:106-109`, `.opencode/skill/system-spec-kit/mcp_server/handlers/save/post-insert.ts:126-129`, `.opencode/skill/system-spec-kit/mcp_server/handlers/save/post-insert.ts:148-151`, `.opencode/skill/system-spec-kit/mcp_server/handlers/save/post-insert.ts:174-177`, `.opencode/skill/system-spec-kit/mcp_server/handlers/save/post-insert.ts:201-203`]. Even after those failures, the function always returns `executionStatus: { status: 'ran' }` [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/handlers/save/post-insert.ts:210-214`]. The response builder forwards that status unchanged and only adds a generic warning when some enrichment boolean is `false` [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/handlers/save/response-builder.ts:311-321`]. Existing tests only lock in the all-success `status: 'ran'` stub and the separate deferred contract; there is no regression for a throwing-step branch [SOURCE: `.opencode/skill/system-spec-kit/mcp_server/tests/handler-memory-save.vitest.ts:546-557`, `.opencode/skill/system-spec-kit/mcp_server/tests/post-insert-deferred.vitest.ts:11-47`].
- **Downstream Impact:** Automation that keys off `postInsertEnrichment.status` can treat partial enrichment failure as completed work, delaying retries or backfill for missing summaries, entities, or graph edges. The warning array becomes the only truthful signal, but it is optional and much easier for callers to ignore than the primary status field.
