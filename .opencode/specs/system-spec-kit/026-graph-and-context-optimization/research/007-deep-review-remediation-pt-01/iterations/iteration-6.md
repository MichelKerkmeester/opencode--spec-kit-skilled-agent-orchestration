## Findings
- **Custom instructions are the documented main-session startup text surface.** GitHub says Copilot CLI automatically adds repository instructions (`.github/copilot-instructions.md`), path-specific `.github/instructions/**/*.instructions.md`, agent instruction files such as `AGENTS.md`, and local instructions at `$HOME/.copilot/copilot-instructions.md`; `COPILOT_CUSTOM_INSTRUCTIONS_DIRS` adds more directories that are scanned for `AGENTS.md` and instruction files. This is real startup/prompt context loading, but it is file-based and applied on the next prompt, not a hook-return API for dynamic in-band payload injection. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions] [SOURCE: https://docs.github.com/en/copilot/how-tos/use-copilot-agents/use-copilot-cli]
- **Skills are documented as capability bundles, not an always-on briefing channel.** GitHub describes skills as folders of instructions, scripts, and resources loaded when relevant, with project locations such as `.github/skills`, `.claude/skills`, and `.agents/skills`, plus personal locations such as `~/.copilot/skills`. That makes the Copilot `/skills` model comparable to Claude-style skills at the "load specialized capability" layer, but not as a guaranteed per-session startup payload or per-prompt advisor-brief injector. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-agent-skills] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-copilot-cli]
- **MCP remains documented as a pull-based tool/data extension, not a push-based context channel.** GitHub says MCP servers are defined in `mcp-config.json` and made available to Copilot sessions so the agent can access extra tools and data sources. The docs describe registration and precedence, but do not describe any MCP server ability to inject text into the main session prompt stream without Copilot first deciding to call that server. [SOURCE: https://docs.github.com/en/copilot/how-tos/use-copilot-agents/use-copilot-cli] [SOURCE: https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-config-dir-reference] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-copilot-cli]
- **The official `~/.copilot` layout documents hooks, agents, skills, MCP, and plugins, but no prompt-template or system-payload config key.** GitHub's config-dir reference lists `config.json`, `mcp-config.json`, `agents/`, `skills/`, `hooks/`, and `installed-plugins/`, and says `config.json` supports inline `hooks`. The same docs document `COPILOT_HOME` and `COPILOT_CACHE_HOME`, while the custom-instructions doc adds `COPILOT_CUSTOM_INSTRUCTIONS_DIRS`; none are described as carrying arbitrary startup or per-prompt text payloads directly. [SOURCE: https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-config-dir-reference] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions]
- **ACP is the first documented surface that could emulate dynamic injection, but only from an external client/wrapper.** The ACP reference shows a client creating `newSession()` and then sending `prompt()` calls while receiving `sessionUpdate` events over NDJSON. That means Spec Kit could theoretically run *outside* the native TUI as an ACP client that prepends startup context and advisor briefs into ACP requests, but the docs do not expose any native in-process ACP or hook bridge that mutates Copilot CLI's own interactive prompts from inside the CLI. [SOURCE: https://docs.github.com/en/copilot/reference/copilot-cli-reference/acp-server] [SOURCE: https://docs.github.com/en/copilot/how-tos/use-copilot-agents/use-copilot-cli]

## Questions Answered
- **Q3 moved forward substantially:** `/skills` is documented as a specialized capability system using skill folders and resources, not as an always-on session bootstrap or prompt-brief surface. Evidence: https://docs.github.com/en/copilot/concepts/agents/about-agent-skills ; https://docs.github.com/en/copilot/concepts/agents/about-copilot-cli
- **Q4 moved forward substantially:** `/mcp` and `mcp-config.json` add tool/data servers Copilot can call, but the documented model is still pull-based and tool-oriented rather than server-pushed prompt injection. Evidence: https://docs.github.com/en/copilot/how-tos/use-copilot-agents/use-copilot-cli ; https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-config-dir-reference ; https://docs.github.com/en/copilot/concepts/agents/about-copilot-cli
- **Q5 moved forward substantially:** Copilot CLI does have startup-readable main-session instruction files (`.github/copilot-instructions.md`, `.github/instructions/**/*.instructions.md`, `AGENTS.md`, and `$HOME/.copilot/copilot-instructions.md`), but the official config-dir reference does not surface a separate prompt-template or system-payload key in `config.json`. Evidence: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions ; https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-config-dir-reference
- **Q6 moved forward substantially:** `--acp` exposes a viable external-client orchestration surface where a wrapper can send precomposed startup/prompt content via ACP requests, but not a native hook-like injection channel inside Copilot CLI's own TUI. Evidence: https://docs.github.com/en/copilot/reference/copilot-cli-reference/acp-server ; https://docs.github.com/en/copilot/how-tos/use-copilot-agents/use-copilot-cli
- **Q7 moved forward partially:** The documented environment variables relevant to these surfaces are `COPILOT_CUSTOM_INSTRUCTIONS_DIRS`, `COPILOT_HOME`, and `COPILOT_CACHE_HOME`; none are documented as direct carriers for arbitrary session/prompt payload text. Evidence: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions ; https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-config-dir-reference
