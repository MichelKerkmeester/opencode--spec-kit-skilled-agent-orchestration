## 4. Risk mitigation beyond spec.md §6

The risks below were surfaced in Iterations 1-5 and are **not** fully captured by spec.md §6:

| Risk | Why it matters | Mitigation |
| --- | --- | --- |
| **Graph-boost symmetry** | `_apply_graph_boosts()` reads the numeric `skill_boosts` snapshot after token, phrase, and normalization boosts have already been merged. It does not distinguish PHRASE-origin from INTENT-origin. Same-weight migrations therefore preserve graph propagation, but any newly higher phrase weight also raises transitive graph boosts. [SOURCE: `.opencode/skill/skill-advisor/scripts/skill_advisor.py:250-296,1639-1684`] | Keep the 23 non-violation migrations weight-stable. Treat every new phrase weight above the old INTENT value as behavior-changing and verify adjacent siblings in Phase 3. |
| **Explicit-name boost stacking** | Explicit skill variants are added after graph and family boosts at `2.5 + 0.3 * (n - 1)`. This is stronger than most new PHRASE weights, so prompts that name a skill directly will still route by explicit-name dominance. [SOURCE: `.opencode/skill/skill-advisor/scripts/skill_advisor.py:1686-1700`] | Keep new regression rows natural-language only. Do not mention exact skill IDs in the new P1 fixture prompts. |
| **Family-affinity bleed** | `_apply_family_affinity()` adds `max_boost * 0.08` to siblings that already have `0 < boost < 1.0` once one family member exceeds `1.5`. New sk-code PHRASE entries can therefore lightly lift siblings like `sk-code-web`, `sk-code-opencode`, and `sk-code-full-stack` together. [SOURCE: `.opencode/skill/skill-advisor/scripts/skill_advisor.py:298-318`] | For sk-code-family additions, inspect the top three recommendations, not just top one, and allow a sibling in `expected_top_any` only when the prompt truly deserves it. |
| **Hyphenated tokenizer blind spot** | The tokenizer uses `re.findall(r'\\b\\w+\\b', prompt_lower)`, so hyphenated INTENT keys are broken for the same reason as whitespace keys. This affects `proposal-only`, `5-dimension`, `openai-cli`, `claude-code`, `copilot-cli`, and related aliases. Iteration 4 treated those keys as safe. They are not. [SOURCE: `.opencode/skill/skill-advisor/scripts/skill_advisor.py:1634-1668`] [SOURCE: `.opencode/skill/skill-advisor/scripts/skill_advisor.py:545,547,550,681,686-688,694-695,701,704,728`] [SOURCE: bash tokenizer audit on 2026-04-15 in repo root] | Expand the migration lens from **contains spaces** to **tokenizer-broken**. At minimum, Phase 2 must add `5-dimension` and `5-dimension agent scoring`. Optional follow-up: audit all hyphenated INTENT keys and migrate the ones that are not already covered by PHRASE or explicit variants. |
| **Low-weight standalone phrase risk** | Preserving an unreachable INTENT weight does not guarantee a new PHRASE becomes threshold-worthy. `proposal only` at `1.4` is the clearest case. It still does not produce a passing bare prompt. [SOURCE: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/006-search-routing-advisor/003-advisor-phrase-booster-tailoring/research/iterations/iteration-004.md:38-47`] [SOURCE: hypothetical Phase 2 probe on 2026-04-15 in repo root for bare `proposal only`] | Do not add a standalone `proposal only` P1 fixture unless the phrase is intentionally raised above the old INTENT weight or paired with a new richer exact phrase that is meant to be routable. |
