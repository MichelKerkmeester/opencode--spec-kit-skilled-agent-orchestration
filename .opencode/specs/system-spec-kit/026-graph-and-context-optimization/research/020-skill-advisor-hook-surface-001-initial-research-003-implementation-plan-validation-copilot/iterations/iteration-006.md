## Findings

### P1

1. **Alert thresholds are not test-gated in 005.** The scaffold's documentation child already expects observability docs to publish fail-open and latency thresholds, but 005 never encodes threshold values in scope, requirements, or acceptance criteria. That leaves the hard-gate harness able to pass without asserting the actual operational SLO boundary. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/005-advisor-renderer-and-regression-harness/spec.md §3.6, §5 Acceptance Scenario 11` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/009-documentation-and-release-contract/spec.md §3.1.6, §4.1 REQ-006`

2. **005 under-specifies the exact JSONL contract.** Its schema is limited to timestamp/runtime/status/freshness/duration/cacheHit/error fields, while wave-2 X6's concrete contract expects a richer fixed-key record capable of supporting alerting and health correlation. Missing fields include hook event identity and structured cache/health state, so different implementations could all "pass" 005 while emitting incompatible telemetry. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/005-advisor-renderer-and-regression-harness/spec.md §3.6` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/020-skill-advisor-hook-surface-001-initial-research-extended-copilot/iterations/iteration-006.md §2 Structured Log Contract`

3. **The label/dimension contract is still too loose in 005.** "runtime, status, freshness, error_code" proves only a generic label set; it does not bind label placement per metric or protect against dimension drift across runtimes. Wave-2 X6 already tightened observability around fixed low-cardinality dimensions, so 005 should lock that precision before implementation starts. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/005-advisor-renderer-and-regression-harness/spec.md §3.6` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/020-skill-advisor-hook-surface-001-initial-research-extended-copilot/iterations/iteration-006.md §1 Metrics Catalog`

### P2

1. **`advisor-hook-health` is present but thinner than the runbook contract.** 005 requires the section to exist and expose last-N/fail-open/p95, which is directionally correct, but it does not pin richer operator fields or section-state semantics. This is not a child-boundary problem because 009's doc scope already covers the runbook side; it is a completeness gap in 005's validation surface, not a missing documentation child. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/005-advisor-renderer-and-regression-harness/spec.md §3.6, §4.1 REQ-009` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/009-documentation-and-release-contract/spec.md §3.1.5-§3.1.10`
