## Findings

### P1

1. **`004` drops wave-1's stale-reuse path for SQLite busy.** Wave-1 allows one bounded retry and then either `freshness: "stale"` from a matching prior live snapshot or `fail_open/unavailable`; `004` only specifies retry-then-`ok` or retry-then-`fail_open`. That weakens the current scaffold because a valid stale brief can disappear from the contract before implementation even starts. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/020-pt-01/research.md:226-226` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/004-advisor-brief-producer-cache-policy/spec.md:121-121` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/004-advisor-brief-producer-cache-policy/spec.md:222-223`

2. **`SIGNAL_KILLED` and deleted-skill fail-open branches are freshness-ambiguous in `004`.** Wave-1 pins signal-killed to `freshness: "unavailable"`, and wave-2 sharpened deleted-skill suppression into a producer-owned rule. `004` names both branches, but neither line states the resulting freshness, which leaves health/metrics parity open to drift across implementations. `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/020-pt-01/research.md:227-227` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/020-pt-02/research-extended.md:24-25` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/research/020-pt-02/research-extended.md:66-68` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/004-advisor-brief-producer-cache-policy/spec.md:121-122` `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/020-skill-advisor-hook-surface/004-advisor-brief-producer-cache-policy/spec.md:280-282`
