# Iteration 008 - V8 Migration compatibility

## Assessment

Wave-2 X7 requires migration-safe reuse to bind global source-signature invalidation to per-skill fingerprints, rename-as-delete-plus-add handling, and suppression of deleted skills from stale cache reuse (`research/020-pt-02/research-extended.md` §X7). Against that contract, `020/003` carries most of the freshness-side semantics and `020/004` carries most of the producer-side cache semantics, but two migration paths are still under-specified before implementation.

## Findings

1. **P1 - Non-live freshness does not yet force a producer-side migration posture.** `020/003` correctly detects stale graph state, but `020/004` never says whether `buildSkillAdvisorBrief()` must fail open, degrade, or continue invoking the subprocess when `freshness !== "live"`. That leaves an implementation path where an existing session can emit fresh-looking output from a stale `skill-graph.sqlite`, which is weaker than the X7 migration contract (`research/020-pt-02/research-extended.md` §X7; `020-skill-advisor-hook-surface/003-advisor-freshness-and-source-cache/spec.md` §5 Acceptance Scenario 2; `020-skill-advisor-hook-surface/004-advisor-brief-producer-cache-policy/spec.md` §4.1 REQ-001, REQ-007).
2. **P1 - Generation-counter corruption recovery is underspecified.** `020/003` covers monotonic generation, missing-file recreation, and read-only fallback, but it never states what happens when `.advisor-state/generation.json` exists and is malformed or unreadable. Migration rollout into existing workspaces is exactly where that bootstrap ambiguity appears, and no acceptance scenario closes it (`020-skill-advisor-hook-surface/003-advisor-freshness-and-source-cache/spec.md` §4.1 REQ-003, REQ-011, §6 Risks, L2 EDGE CASES).
3. **P2 - The edit -> stale -> rebuild -> generation+1 sequence is only implicit.** The current scaffold says edits make freshness stale and later rebuilds advance generation, while `020/004` invalidates exact-cache entries on source-signature change. That is enough to infer the intended chain, but the migration semantics are not written as one end-to-end acceptance path (`020-skill-advisor-hook-surface/003-advisor-freshness-and-source-cache/spec.md` §5 Acceptance Scenarios 2, 6, 8; `020-skill-advisor-hook-surface/004-advisor-brief-producer-cache-policy/spec.md` §5 Acceptance Scenario 4).
4. **P2 - Graceful first-run bootstrap is present by composition, not by an explicit migration scenario.** The scaffold implies a cold-start miss path rather than stale reuse, but neither child spec states the full "first session after hook lands" narrative in one acceptance scenario. This is probably implementable as-is, but the migration contract is easier to misread than the rename/delete case (`020-skill-advisor-hook-surface/003-advisor-freshness-and-source-cache/spec.md` §5 Acceptance Scenarios 1 and 3; `020-skill-advisor-hook-surface/004-advisor-brief-producer-cache-policy/spec.md` §5 Acceptance Scenario 3, L2 EDGE CASES).

## Novelty assessment

This pass produced limited but real new information: rename/delete handling is already well covered, while stale-graph producer behavior and corrupt generation-counter recovery remain the main pre-implementation migration gaps.
