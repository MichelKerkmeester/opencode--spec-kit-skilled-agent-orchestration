# Research: Graph Metadata Quality & Relationship Validation

Scan all 515+ graph-metadata.json files on disk to validate relationship integrity, entity quality, and derived field accuracy. No historical save data needed - this is a pure filesystem and code analysis.

## Background

Every spec folder now carries a `graph-metadata.json` with:

**Manual section** (human-authored):
- `depends_on` - declared dependencies on other spec folders
- `supersedes` - what this folder replaces
- `related_to` - cross-references
- `parent_id`, `children_ids` - tree structure

**Derived section** (auto-generated by parser):
- `trigger_phrases` - max 12
- `key_topics` - max 12
- `key_files` - max 20 referenced file paths
- `entities` - max 16, max 80 chars each, source-labeled
- `status` - derived from ranked doc sources, fallback "planned"
- `importance_tier` - derived from doc presence
- `causal_summary` - extracted from Overview/first paragraph
- `source_docs` - list of canonical packet doc paths
- Timestamps: `created_at`, `last_save_at`, `last_accessed_at`

**Known issues from deep review:**
- Shell commands leaking into key_files/entities (partially fixed but may recur on re-backfill)
- Non-canonical refs (basename-only duplicates alongside canonical paths)
- Ghost children_ids pointing at archived/moved folders
- Stale status values not matching implementation reality

## Research Questions

1. **RQ-1:** How many declared `depends_on` edges resolve to actual spec folders on disk? What is the broken-edge rate?
2. **RQ-2:** Are there dependency cycles in the graph? If so, which folders participate?
3. **RQ-3:** Do `children_ids` match actual child directories? What is the ghost-child rate?
4. **RQ-4:** What percentage of `key_files` entries point to files that actually exist? What types of non-existent refs appear?
5. **RQ-5:** How many entities are basename-only duplicates of canonical-path entries? What is the noise rate?
6. **RQ-6:** Do `status` values match reality? (Compare derived status against implementation-summary.md presence and checklist completion)
7. **RQ-7:** What is the distribution of entity counts, key_file counts, and trigger_phrase counts across all folders? Are the limits (16/20/12) appropriate?
8. **RQ-8:** How many graph-metadata.json files have stale `last_save_at` timestamps (older than their spec-doc mtimes)?

## Key Files to Investigate

- `mcp_server/lib/graph/graph-metadata-schema.ts` - Zod schema defining valid structure
- `mcp_server/lib/graph/graph-metadata-parser.ts` - derivation logic, normalization, limits
- `scripts/graph/backfill-graph-metadata.ts` - batch generation logic
- Every `graph-metadata.json` across `.opencode/specs/` (515+ files)

## Research Approach

1. Write a scanner that reads every graph-metadata.json under `.opencode/specs/`
2. For each file, validate:
   - All `depends_on` targets exist on disk
   - All `children_ids` match actual subdirectories
   - No cycles in the dependency graph (topological sort)
   - All `key_files` entries resolve to real files
   - Entity quality (no shell commands, no bare basenames when canonical path exists)
   - Status accuracy (cross-reference with implementation-summary.md)
   - Timestamp freshness (compare `last_save_at` with file mtimes)
3. Aggregate statistics across the full corpus
4. Identify systematic patterns in broken references (e.g., archived folders, renamed phases)

No historical saves needed. This is a pure filesystem scan of current state.

## Exit Criteria

- Full corpus scan results with statistics per research question
- Broken-edge inventory (every invalid depends_on/children_id)
- Cycle detection results
- Entity quality metrics (noise rate, duplicate rate)
- Status accuracy rate
- Recommendations for parser improvements and re-backfill parameters
