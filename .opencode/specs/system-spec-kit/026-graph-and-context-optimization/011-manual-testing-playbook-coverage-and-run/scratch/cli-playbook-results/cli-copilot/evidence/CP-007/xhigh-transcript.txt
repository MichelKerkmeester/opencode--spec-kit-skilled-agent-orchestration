**Best default:** design it as a **CRDT counter** so each partition can keep accepting updates locally and all replicas **converge automatically** after healing. If you need **exactly consistent reads during a partition**, you must trade away write availability; you cannot have both.

1. **Choose the semantics first**
   - **Increment-only:** use a **G-Counter**.
   - **Increment + decrement:** use a **PN-Counter**.
   - Decide whether you want:
     - **AP / highly available:** accept writes during partitions, converge later.
     - **CP / strongly consistent:** reject some writes during partitions to preserve a single exact value.

2. **Pick the replica model**
   - Run the counter on **N replicas** across failure domains.
   - Give every replica a stable **replica_id**.
   - Use **leaderless replication** or multi-writer replication if partition survival with local writes matters.

3. **Define the data model**
   For a PN-Counter:
   - `increments: Map<replica_id, uint64>`
   - `decrements: Map<replica_id, uint64>`
   - Value = `sum(increments) - sum(decrements)`
   - Optional metadata:
     - `version_vector` or per-replica logical clock
     - `seen_request_ids` for idempotency
     - membership/config version

   Why this works:
   - Each replica only increases its own slot.
   - Merge is monotonic and safe.

4. **Define the write path**
   - On `increment(k)` at replica `r`: do `increments[r] += k`
   - On `decrement(k)` at replica `r`: do `decrements[r] += k`
   - Attach a **client operation id** so retries do not double-apply.
   - Persist locally before acking if you want crash tolerance.

5. **Handle network partitions explicitly**
   During a partition:
   - Each side continues accepting writes on reachable replicas.
   - Replicas exchange state only with peers they can reach.
   - Reads can be:
     - **Local/fast but possibly stale**
     - **Quorum-based** when quorum is available
   Important tradeoff:
   - Under partition, different sides may return different temporary values.
   - After healing, all replicas converge to the same final value.

6. **Define conflict resolution**
   CRDT merge rule:
   - For every replica slot, merge with **element-wise max**
   - Example:
     - `merged.increments[i] = max(a.increments[i], b.increments[i])`
     - `merged.decrements[i] = max(a.decrements[i], b.decrements[i])`
   This resolves concurrent updates without losing either side’s work.

7. **Deal with practical edge cases**
   - **Duplicate requests:** dedupe by `request_id`
   - **Crash recovery:** use WAL or durable local storage
   - **Replica replacement:** do not reuse `replica_id`
   - **Membership changes:** treat config changes carefully; old slots may need retention or compaction
   - **Counter bounds/invariants:** if the counter must never go below 0 or above a limit, plain AP CRDT is not enough; use escrow/reservations or a CP design

8. **Design the read model**
   - Expose at least:
     - **Eventual value**: local merged state
     - **Freshness metadata**: config/version/vector timestamp
   - If users need stronger guarantees, offer:
     - `read_local()`
     - `read_quorum()`
     - maybe `read_linearizable()` only in CP mode

9. **Verification approach**
   Verify at three levels:
   - **Unit tests**
     - merge is commutative, associative, idempotent
     - retries do not double-count
   - **Property tests / model tests**
     - any interleaving of updates and merges converges to the same value
     - partition + heal preserves all acknowledged operations
   - **Fault-injection / distributed simulation**
     - drop/delay/reorder messages
     - split cluster into partitions
     - crash and restart replicas
     - confirm eventual convergence and no lost increments

10. **Know when this design is wrong**
   Use something else if you need:
   - a **single exact answer at all times**
   - strict bounds like “never exceed inventory”
   - transactional coupling with other objects

**Rule of thumb:** if the requirement is “keep counting through partitions and reconcile later,” use a **CRDT PN-Counter**. If the requirement is “always show one exact globally correct number,” use a **CP quorum/consensus-based counter** and accept reduced availability during partitions.
