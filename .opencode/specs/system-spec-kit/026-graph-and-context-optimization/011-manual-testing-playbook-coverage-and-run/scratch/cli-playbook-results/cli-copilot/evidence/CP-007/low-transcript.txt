A fault-tolerant distributed counter that survives partitions is usually designed as an **AP system**: it keeps accepting increments during network splits and converges later. The simplest reliable design is a **CRDT counter**, typically a **G-Counter** if you only need increments, or a **PN-Counter** if you need both increments and decrements.

1. **Define the requirements first**
   - Decide whether you need only `increment` or also `decrement`.
   - Decide whether availability during partitions is more important than linearizable reads.
   - Decide whether “exactly up-to-date right now” is required, or whether eventual convergence is acceptable.

2. **Pick the right model**
   - For **increment-only**: use a **G-Counter**.
   - For **increment/decrement**: use a **PN-Counter** built from two G-Counters.
   - Avoid a single shared integer with leader-based writes if partition survival is the priority; that design tends to reject writes or risk split-brain.

3. **Design the data model**
   - Each replica/node keeps its **own component** of the counter.
   - Example for a G-Counter:
     ```text
     counter = {
       replicaA: 5,
       replicaB: 3,
       replicaC: 9
     }
     value = sum(all replica components)
     ```
   - For a PN-Counter:
     ```text
     p = { replicaA: 7, replicaB: 2 }
     n = { replicaA: 1, replicaB: 0 }
     value = sum(p) - sum(n)
     ```
   - Each replica only updates its own slot.

4. **Write path**
   - On `increment()` at replica X, do: `counter[X] += 1`.
   - Persist locally before acknowledging if durability matters.
   - This makes writes local, fast, and still available during partitions.

5. **Read path**
   - Local read: sum the local CRDT state.
   - Merged/global read: sum after merging state from peers.
   - Be explicit that reads may be **stale during a partition**, but will converge.

6. **Partition handling**
   - During a network split, each side continues accepting updates independently.
   - No cross-partition coordination is required for increments.
   - This preserves availability: both sides keep working.

7. **Conflict resolution**
   - CRDT merge rule for each replica component is **element-wise max**.
   - Example:
     ```text
     node1 sees {A: 4, B: 2}
     node2 sees {A: 3, B: 5}
     merged = {A: max(4,3), B: max(2,5)} = {A: 4, B: 5}
     ```
   - This works because each replica’s own component is monotonic.
   - For PN-Counters, merge `p` and `n` independently with the same rule.

8. **Anti-entropy / synchronization**
   - Replicas periodically gossip their state, or exchange deltas.
   - After healing, all replicas merge until they converge to the same value.
   - Prefer **delta-state CRDTs** if full-state exchange is too expensive.

9. **Durability and recovery**
   - Store counter state in a write-ahead log or durable snapshot.
   - On restart, reload the last durable state before resuming gossip.
   - Replica identity must be stable; if node IDs change on restart, you can corrupt semantics.

10. **Membership and replica IDs**
   - Every writer needs a unique, durable replica ID.
   - If nodes are ephemeral, use logical actor IDs, not process IDs.
   - Handle retired replicas carefully so old slots are not lost or reused incorrectly.

11. **Limits and tradeoffs**
   - This design gives **eventual consistency**, not strict serializability.
   - If you need an exact globally current value at all times, partitions force a tradeoff: you must sacrifice availability or use stronger coordination.
   - Decrements are safe with PN-Counters, but “prevent counter going below zero globally” is harder and usually needs coordination.

12. **Verification approach**
   - Verify these core properties:
     1. **Convergence**: after communication resumes, all replicas reach the same value.
     2. **No lost updates**: every acknowledged increment is reflected in the final total.
     3. **Monotonic merge safety**: merge never decreases any component incorrectly.
     4. **Partition availability**: writes succeed on isolated replicas.
     5. **Crash recovery**: persisted acknowledged writes survive restart.
   - Test with:
     - unit tests for merge algebra: commutative, associative, idempotent
     - simulation tests with random partitions, duplicate messages, reordering, and retries
     - Jepsen-style fault injection if this is production-critical
     - model/property testing for “final value equals total successful operations”

A good default design is: **PN-Counter CRDT + durable local storage + periodic gossip + stable replica IDs + property-based convergence tests**. That is the standard path when partition survival matters more than immediately consistent global reads.
