## Analysis
Adopt adversarial resilience now, because B7 is a scoring-integrity problem at the same trust boundary as B3/B4/B6, not a speculative future security layer. The threat model is: a repo user or dependency author can edit a skill's `SKILL.md` or `graph-metadata.json`, add repeated generic terms, broad keywords, hidden/instruction-shaped text, or many derived trigger phrases, and cause that skill to outrank better matches. Existing renderer protections reduce the chance that malicious skill text becomes model-visible instructions, but they do not prevent ranking poisoning before the renderer picks the top skill.

The current implementation already has useful building blocks but also a key weakness. It caps total signal boost per skill, deduplicates normalized signal variants, requires phrase-boundary matches, uses confidence/uncertainty dual thresholds, and has precedent for graph-derived confidence haircuts. However, runtime graph loading merges `intent_signals` and `derived.trigger_phrases` into one signal map, and any positive signal boost enters the high-confidence intent curve. That means a stuffed derived trigger set can look like precise author intent unless B7 extends evidence separation from graph evidence to all untrusted or generated evidence lanes.

The mitigation should follow search-system spam practice: cap influence, demote suspicious signals, and keep ranking manipulation observable. Add per-source trust lanes (`explicit_author`, `derived_generated`, `body_corpus`, `graph`, `semantic`), source-specific max contribution, signal cardinality limits, repetition-density penalties, corpus DF/IDF downweighting, and a negative/suspicious-pattern demotion lane for instruction-shaped or stuffing-shaped phrases. Derived or body-only evidence may surface a candidate, but should not pass default routing unless supported by direct non-suspicious evidence or by corpus validation. Keep generated terms in `graph-metadata.json.derived` and SQLite with provenance/fingerprint fields; do not write them back into human-authored `SKILL.md`.

Testing should become part of the first implementation slice. The existing regression harness can absorb an adversarial suite that covers repeated broad keywords, hidden markdown/HTML text, "ignore previous instructions" phrases, encoded/typoglycemia variants, metalinguistic skill mentions, and a malicious skill that copies every high-value trigger from other skills. Required gates should be no P0 regression, no increase in gold-`none` false fires, no command-bridge false-positive regression, and no stuffed skill passing default routing without direct legitimate task evidence.

## Verdict
- **Call:** adopt now
- **Confidence:** high
- **Rationale:** The repo already has scoring caps, confidence calibration, renderer sanitization, corpus-stat direction, and a regression harness; B7 mainly needs to make adversarial scoring lanes explicit and enforce anti-stuffing caps/demotions before derived triggers become stronger. Deferring this would let B2/B3/B6 increase routing recall while widening a known ranking-poisoning surface.
