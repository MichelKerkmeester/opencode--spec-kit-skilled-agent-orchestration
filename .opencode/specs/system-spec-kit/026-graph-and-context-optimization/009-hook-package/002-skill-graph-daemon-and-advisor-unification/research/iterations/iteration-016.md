## Analysis
The memory MCP server already has the primitives Track C needs for an advisor matching prototype and likely for the first production slice. The public retrieval ladder is present: `memory_context` for intent-aware orchestration, `memory_search` for configurable hybrid retrieval, `memory_quick_search` as a safe preset over the full search path, and `memory_match_triggers` for cheap trigger-phrase matching with cognitive tiering. The current code layout does not actually use `mcp_server/lib/memory/**`; memory behavior is split across MCP handlers, `tools/*` dispatchers, schemas, and `lib/search/**`. That is a naming/layout mismatch in the research prompt, not a missing capability.

The hybrid search internals are richer than C1 strictly requires. Stage 1 can generate embeddings, run multi-concept search, expand or decompose queries, collect raw hybrid candidates, inject constitutional memories, and filter by tier/context/quality. The hybrid module collects vector, FTS5, BM25, graph, and degree channels, then fuses them through adaptive intent-weighted RRF. Stage 2 is already the single scoring point for boosts and demotions, including session boost, recency, causal boost, co-activation, graph signals, artifact routing, learned feedback, negative demotion, validation scoring, and a shadow learned combiner. That gives C4 a real local fusion substrate instead of forcing a brand-new advisor scorer.

The causal primitives are also available but should be reused carefully. `memory_causal_link`, `memory_causal_stats`, `memory_causal_unlink`, and `memory_drift_why` are memory-ID oriented; their relation vocabulary and traversal weights are useful for skill dependencies, but C2/C3 still need an adapter for skill IDs, skill graph rows, or synthetic memory-like skill records. The existing causal-boost caps are a good safety model for advisor scoring: causal evidence should nudge rankings, not dominate explicit task/keyword evidence.

Therefore the immediate design direction should be to adopt the memory MCP primitives as an internal library/API pattern for advisor matching, not to fork the memory system wholesale. The first Track C implementation should wrap the existing search/fusion concepts behind an advisor-specific adapter: map skills into searchable documents/signals, preserve explicit-vs-derived trust lanes from Track B, call or port only the scoring primitives needed, and keep diagnostics through stats/health-style surfaces. Track D then decides whether these tools live in the consolidated MCP server as `advisor_*` tools or remain internal modules behind the advisor hook.
