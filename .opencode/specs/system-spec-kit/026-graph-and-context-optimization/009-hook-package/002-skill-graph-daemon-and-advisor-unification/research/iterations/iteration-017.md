## Analysis
The memory entry schema and the skill schema overlap in the parts needed for retrieval, but they are not the same object. A memory is a mutable indexed document with scope, lifecycle, embedding state, content text, quality/validation, retention, access, decay, and cognitive memory metadata. A skill is a stable capability node with an identity, family/category, domains, intent signals, derived trigger phrases/key topics/key files/entities/source docs, and typed graph edges. That makes the useful mapping a projection, not a table reuse: `title/content_text/trigger_phrases` map to skill `name/description + derived text/intent_signals`, and memory causal/graph notions map to skill edges, but memory `spec_folder`, session scope, retention, FSRS decay, governance fields, and validation counters should not be copied into the canonical skill node.

The advisor already partially implements this projection, but across three thin paths. `SKILL.md` frontmatter supplies name/description/keywords, per-skill `graph-metadata.json` supplies intent signals and derived triggers, and `skill-graph.sqlite` supplies families and typed edges. The gap versus memory search is not source availability; it is that skills lack a first-class searchable projection with vector/FTS fields, channel provenance, trace metadata, and the same kind of query-time fusion that memory search uses. C2 should therefore map memory retrieval concepts to an advisor-specific `AdvisorSkillDocument` or `SkillSearchRecord`, not push skills into `memory_index`.

The clean mapping is: memory `id` -> skill `skill_id`; `title` -> display/name; `content_text` -> concatenated description, domains, intent signals, derived trigger phrases, key topics, causal summary, and selected source docs; `trigger_phrases` -> `intent_signals + derived.trigger_phrases + keyword variants`; `document_type` -> constant `skill`; `importance_tier/base_importance` -> bounded skill priority lane only if explicitly declared later; `quality_score/confidence/validation_count` -> advisor calibration evidence only when produced by routing tests, not by memory save validation; `causal_links` -> skill graph edges; `governance/session/retention/FSRS` -> no mapping for static skills. This preserves existing skill graph authority while letting C4 reuse memory-style semantic/keyword/graph fusion.

Adopt the mapping now, but keep the storage separate. The existing `skill_graph_*` MCP surface and `skill-graph.sqlite` already own skill identity and topology, while memory search owns a richer retrieval architecture. The next implementation phase should add a projection layer that can be regenerated from `SKILL.md` plus `graph-metadata.json`, optionally backed by advisor-specific FTS/vector tables, and feed those results into the advisor scorer with the same explicit-evidence and graph-only safety rules that exist today.
