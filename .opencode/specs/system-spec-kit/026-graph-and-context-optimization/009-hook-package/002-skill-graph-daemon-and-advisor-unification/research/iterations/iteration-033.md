## Analysis
A and C are coherent because C3's adopted causal design uses the same `skill_edges` that A's transaction already rebuilds. The live schema keeps skill nodes and typed skill edges in one SQLite database with foreign keys, cascade semantics, uniqueness, and edge indexes (`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:118`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:158`). A3 and A4 also established that the indexer parses and hashes inputs first, then performs missing-node deletion, changed-node upsert, and outgoing-edge replacement inside one SQLite transaction (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-003.md:34`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-003.md:41`, `.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:520`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:586`). That means prompt-time C traversal reads either the old committed graph or the new committed graph, not a half-rebuilt causal topology.

The main coherence guard is to keep C causal behavior as a skill-graph traversal lane, not as a mirror into memory `causal_edges`. C2 and C3 explicitly keep canonical skills separate from memory lifecycle storage and route causal-style skill behavior over `skill_edges` (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-017.md:37`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-017.md:44`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-018.md:32`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-018.md:39`). If implementation honors that boundary, A does not need a second transaction for causal rows; edge replacement and C traversal stay in the same committed SQLite graph.

C's prompt-time operations also fit A's performance and failure constraints if they stay bounded and SQL-backed. The query layer uses prepared relation lookups, bounded BFS, depth-limited subgraphs, deduplication, and deterministic sorted returns (`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-queries.ts:76`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-queries.ts:77`, `.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-queries.ts:322`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-queries.ts:432`). C6 already rejects embedding lookup as an adopt-now prompt-time dependency while accepting lexical and skill-graph SQL work as the hot-path candidate (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-021.md:39`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-021.md:46`).

The remaining adjustment is sequencing around post-commit publication. A4 says generation/freshness/cache publication belongs after the graph commit, and the current code writes scan summary metadata after transaction completion (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-004.md:43`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-004.md:50`, `.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:588`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-graph/skill-graph-db.ts:603`). C's fusion and ambiguity layers should therefore consume freshness/generation as a post-commit invalidation signal; when freshness is stale or unavailable, advisor behavior should fail open or degrade rather than reading a speculative rebuild (`.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/freshness.ts:223`-`.opencode/skill/system-spec-kit/mcp_server/lib/skill-advisor/freshness.ts:263`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-008.md:46`-`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/009-hook-package/002-skill-graph-daemon-and-advisor-unification/research/iterations/iteration-008.md:53`).
