## Analysis
The evidence points to an MCP-native target layout, but not by literally copying a `lib/memory/` directory. In this checkout, memory MCP behavior is organized as tool dispatcher modules (`tools/memory-tools.ts`, `tools/context-tools.ts`), lazy handler exports (`handlers/index.ts`), handler files (`handlers/memory-search.ts`, `handlers/memory-save.ts`, `handlers/memory-context.ts`), and domain libraries under `lib/search`, `lib/storage`, `lib/parsing`, `lib/context`, `lib/cache`, and related folders. The reusable pattern is therefore the layer boundary: tools expose names and argument validation, handlers own MCP response orchestration, and `lib/*` owns durable domain behavior.

Skill graph already matches that target shape. It has `tools/skill-graph-tools.ts`, `handlers/skill-graph/{scan,query,status,validate}.ts`, strict schema entries, tests, and `lib/skill-graph/{skill-graph-db,skill-graph-queries}.ts` over `skill-graph.sqlite`. The migration target should preserve that as the graph plane rather than creating a second graph home under advisor. In practice, `lib/skill-graph/` remains the source of truth for scan/query/status/validate and database lifecycle.

The advisor plane should converge inside system-spec-kit MCP beside the existing `lib/skill-advisor/` hook plane. Target modules should be: `tools/advisor-tools.ts` for public advisor tool dispatch; `handlers/skill-advisor/{recommend,status,validate,bench?}.ts` for MCP handlers; `lib/skill-advisor/{runtime,scoring,skill-records,signals,conflicts,health,compat-cli}.ts` for the ported Python brain; existing `prompt-policy.ts`, `prompt-cache.ts`, `freshness.ts`, `render.ts`, `skill-advisor-brief.ts`, metrics, and diagnostics stay in `lib/skill-advisor/`; and `schemas/tool-input-schemas.ts` receives advisor schemas/allowed parameters. Tests should mirror the current MCP pattern: schema tests, dispatcher tests, runtime parity tests, and compatibility tests around the legacy Python CLI outputs.

That target is adoptable now as a layout decision even though D4 still must decide whether subprocess elimination is a full TypeScript rewrite or a staged bridge. The layout can support both: `skill-advisor-brief.ts` can call a native `recommendSkills()` once available, while a `compat-cli` adapter or existing subprocess runner preserves the Gate 2 CLI contract until D7 confirms all call sites can move. This avoids reopening shipped Phase 020-026 graph architecture while giving later D3-D8 iterations stable paths for tools, cache/freshness, install, compatibility, and plugin decisions.
