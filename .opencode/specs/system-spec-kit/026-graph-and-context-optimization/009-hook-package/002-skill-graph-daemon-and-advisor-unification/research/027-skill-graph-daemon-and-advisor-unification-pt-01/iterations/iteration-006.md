## Analysis
The re-index compute budget is not the bottleneck for the current corpus. The legacy Python compiler validates 21 top-level metadata files in about 30ms wall time and about 15-16MB max RSS, and cProfile shows the work is dominated by interpreter startup/main validation rather than expensive graph operations. The TypeScript path selected by A3 is slightly broader because it recursively discovers metadata files and parses all discovered metadata before content-hash skip, but the current metadata payload is only about 81KB top-level and the code uses hash-aware row skipping inside one SQLite transaction.

The watcher budget is more nuanced. If Phase 027 stays with A2/A3's narrow `graph-metadata.json` trigger, an explicitly expanded Chokidar v4 file list watches 21 files with 21 additional descriptors and about 48MB RSS in a minimal Node process, with a 2s debounce already below A6's 10s ceiling. That is acceptable only if the daemon remains MCP-owned as A5 recommended, because a separate always-on watcher process would spend nearly the whole <50MB RAM target before SQLite, MCP transport, and advisor code are loaded.

The full A6 target of watching `SKILL.md`, supporting files, references, scripts, and asset directories should not be adopted now. The current code passes a glob path to Chokidar, but the installed v4 package explicitly removed glob support, and a live probe with the same glob shape watched zero entries. When the glob is replaced by explicit paths for the broader source set, this current repo already presents 91 top-level watch roots and about 1,528 supporting files, and the probe hit `EMFILE` before reaching the planned 50-100 skill scale. That makes broad live watching a benchmark/prototype item, not a safe Phase 027 default.

The practical resource budget for Phase 027 should therefore be: adopt narrow metadata-only watch after fixing the v4 glob mismatch, keep the all-root scan because the measured parse/validation work is cheap at current scale, and defer broad `SKILL.md`/asset/reference watching until B-track freshness design can combine directory-level filters, batching, and hard limits. A6 should also feed D6/D7: no launch-agent packaging should be claimed under the <50MB target until the watcher runs inside the existing MCP process or a separate daemon has measured end-to-end RSS.

## Verdict
- **Call:** prototype later
- **Confidence:** medium
- **Rationale:** The narrow metadata-only daemon budget is acceptable after replacing the Chokidar v4 glob with explicit paths or directory filters, but the broad A6 watch target hit descriptor pressure and has not demonstrated the <50MB RAM target. Prototype broad watching with a benchmark harness before adopting it as default behavior.
