## Analysis
The port target is feasible as a TypeScript rewrite rather than a PyOdide bridge. The runtime helper named in the D4 prompt is only 320 lines and consists of frontmatter parsing, keyword variant generation, mtime/size cache signatures, cloned record returns, and inventory set comparison. Those are ordinary Node/TypeScript operations, and the current MCP server already has `better-sqlite3`, `zod`, cache modules, freshness modules, and skill-graph libraries available in the same package.

The heavier Python file is larger at 2,922 lines, but its core recommendation logic is still deterministic table-driven scoring: constants, regex phrase matching, graph boosts, semantic-hit boosts, confidence/uncertainty formulas, tiebreakers, sorting, and health metadata. The one part that should not be copied literally is the built-in `ccc search` subprocess lane; D3 already proposed `semanticHits` as an MCP input, so the native runtime can accept precomputed semantic hits and keep semantic lookup orchestration outside the scorer. That makes the native `recommendSkills()` core smaller than the full CLI because it does not need argparse, stdout JSON, batch stdin, direct `ccc` spawning, or Python-specific fail-open handling.

PyOdide is technically possible, but it is the wrong bridge for Phase 027. It would add a new WebAssembly Python runtime dependency where the active package manifests currently have none, still keep Python semantics embedded in the MCP process, and introduce deployment/runtime artifact questions that are disproportionate for a deterministic scorer already expressible in TypeScript. PyOdide also would not remove the need for a compatibility story around `skill_advisor.py`; D7 still needs to enumerate old call sites and decide which CLI modes remain as shims.

The safest adoption path is therefore: port the advisor brain into `lib/skill-advisor/{runtime,scoring,skill-records,signals,conflicts,health}.ts`, keep the existing `subprocess.ts` bridge only as `compat-cli` during migration, and use the 200-prompt corpus parity test as the promotion gate. Once native parity matches direct Python top-1 and confidence behavior, `buildSkillAdvisorBrief()` and `advisor_recommend` should call native TypeScript directly. The current subprocess fail-open tests can remain as compatibility tests, not as the primary runtime contract.

## Verdict
- **Call:** adopt now

- **Rationale:** Adopt subprocess elimination now via a native TypeScript rewrite, not PyOdide. The code is deterministic and already has a parity harness, but the 2,922-line Python file and CLI compatibility surface make this a staged migration rather than a one-step mechanical port.
