## Shim Exit-Code Fix

Finding closed: 4 of 6 `skill-advisor/tests/compat/shim.vitest.ts` cases regressed to exit 2 ("Native advisor unavailable"), failing the `--force-native` contract that the compat shim must work without a daemon.

### Source

- Failing tests: `routes to native advisor_recommend when forced native...`, `preserves --stdin mode and does not leak prompt text...`, `lets --force-native override the shared disabled flag`, `tries native mode when --force-native is combined with semantic flags`. All four asserted `result.status === 0` but received `2`.
- Direct subprocess output: `{ "error": "Native advisor unavailable", "reason": "advisor_status found freshness artifacts, but no live daemon PID/process evidence", "freshness": "unavailable" }`.

### Root Cause

Commit `cd766a05f` (P1 batch CF-016 / Theme 4-5 daemon hardening) added `hasRunningDaemon(pid)` to `skill-advisor/handlers/advisor-status.ts:37` and changed the live/stale gate from `generation.state !== 'unavailable'` to `daemonAvailable = hasRunningDaemon(parseDaemonPid())` (`advisor-status.ts:106`). With no `SPECKIT_SKILL_GRAPH_DAEMON_PID` env var (the test environment), `daemonAvailable=false`, `createTrustState` collapses to `state: 'unavailable'`, the bridge `probe()` (`scripts/skill_advisor.py:108-109`) sees `freshness === 'unavailable'`, declares native unavailable, and `--force-native` returns exit 2 (`scripts/skill_advisor.py:3247-3253`). The native MCP daemon-liveness invariant is correct for in-process callers, but the Python compat shim is by design a one-shot CLI with no daemon to probe.

### Files Modified

- `.opencode/skill/system-spec-kit/mcp_server/skill-advisor/scripts/skill_advisor.py`
  - Lines 67–90: added `NATIVE_GENERATION_MODULE` path constant pointing at `dist/skill-advisor/lib/freshness/generation.js`.
  - Lines 92–93, 105–122: bridge now imports `readSkillGraphGeneration` and, when `readAdvisorStatus` reports `unavailable`, falls back to the underlying `generation.state` — if it is `live` or `stale`, the shim probe is treated as eligible. Native MCP callers (`handleAdvisorRecommend`) still flow through the strict daemon-checked `readAdvisorStatus` path unchanged.
  - Lines 207–214: `_native_bridge_source()` substitutes the new `GENERATION_MODULE` placeholder; `_native_bridge_available()` also checks the generation module exists before declaring the bridge usable.

## advisor-status freshness-vs-trustState decoupling (debug agent, 2026-04-25)

Finding closed: `cd766a05f` daemon-hardening regressed 5 tests in
`skill-advisor/tests/handlers/advisor-status.vitest.ts` because the handler
fed `trustState.state` (daemon-gated) into the `freshness` output field
(data/artifact-only).

### Source

Same regression cluster as the shim exit-code fix above. Commit `cd766a05f`
replaced `daemonAvailable = generation.state !== 'unavailable'` with a real
PID/process check. The handler then assigned `freshness: trustState.state`,
which collapses to `'unavailable'` whenever the daemon isn't live. Tests
don't simulate a daemon, so all `freshness` values flattened to
`'unavailable'`. Per F10 (data/artifact axis) and F18 (caller-trust axis)
these axes are conceptually different.

### Files Modified

- `.opencode/skill/system-spec-kit/mcp_server/skill-advisor/handlers/advisor-status.ts`
  - Line 110: gated daemon-evidence error on `daemonPid !== undefined`
    (no PID file → daemon never started → not a divergence to report).
  - Lines 113–119: added `sourcesNewerThanArtifact` derived from physical
    mtime evidence (`hasArtifact && hasSources && sourceScan.maxMtimeMs >
    fileMtimeMs(dbPath)`).
  - Lines 135–139: replaced `freshness: trustState.state` with
    `freshness: freshnessOutput`, where `freshnessOutput = state === 'live'
    && sourcesNewerThanArtifact ? 'stale' : state`. `trustState` field
    itself is unchanged (still daemon-gated).

### Decision: daemon-evidence error gating

Suppressed when `daemonPid === undefined` rather than tolerated via
`arrayContaining` in the cap-test. The error message asserts a
*divergence* — "freshness artifacts exist, but no live daemon evidence" —
which has no meaning when a PID file was never written. Firing it in that
state turns a genuine warning into noise and would force fixture-only test
changes for a non-issue. The error remains correct and useful when a PID
file IS present but the process is dead, which is the actual divergence the
message describes.

### Verification

- `cd .opencode/skill/system-spec-kit/mcp_server && npx vitest run skill-advisor/tests/handlers/advisor-status.vitest.ts` → 7/7 pass (was 5 failing).
- `cd .opencode/skill/system-spec-kit/mcp_server && npx tsc --noEmit` → exit 0.
