# Iteration 4: Q1/Q2 hook input and context injection contract
## Focus
This pass targeted the hook command stdin format and stdout-to-model-context semantics for Codex CLI 0.122.0, using current official Codex hooks documentation, generated schemas, source-level implementation evidence, and local read-only config evidence.
## Actions Taken
1. Read the official OpenAI Codex hooks documentation for lifecycle behavior, event taxonomy, stdin/stdout contract, timeout defaults, concurrency semantics, and event-specific blocker behavior: https://developers.openai.com/codex/hooks
2. Opened generated upstream input schemas for `UserPromptSubmit` and `SessionStart` to verify exact command stdin JSON fields: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/schema/generated/user-prompt-submit.command.input.schema.json and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/schema/generated/session-start.command.input.schema.json
3. Opened generated upstream output schema for `UserPromptSubmit` to verify the supported context/blocker output shape and absence of prompt-transform fields: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/schema/generated/user-prompt-submit.command.output.schema.json
4. Rechecked source implementation for command execution, session-start parsing, prompt-submit parsing, and concurrent dispatcher behavior: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/command_runner.rs, https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/session_start.rs, https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/user_prompt_submit.rs, and https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/dispatcher.rs
5. Re-read local installed evidence without modifying hook configuration: `codex --version` reports `codex-cli 0.122.0`; `~/.codex/hooks.json` registers only Superset `notify.sh` for `SessionStart`, `UserPromptSubmit`, and `Stop`; `notify.sh` accepts stdin when Codex does not pass an argument.
## Findings
- Q1: Every command hook receives one JSON object on stdin, not plain text or empty input. The official docs state this directly, and `command_runner.rs` writes the serialized `input_json` bytes to the child process stdin. Primary sources: https://developers.openai.com/codex/hooks and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/command_runner.rs
- Q1: `UserPromptSubmit` stdin is a JSON object with required fields `cwd`, `hook_event_name:"UserPromptSubmit"`, `model`, `permission_mode`, `prompt`, `session_id`, `transcript_path`, and `turn_id`; additional properties are rejected by the generated schema. Primary source: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/schema/generated/user-prompt-submit.command.input.schema.json
- Q1: `SessionStart` stdin is a JSON object with required fields `cwd`, `hook_event_name:"SessionStart"`, `model`, `permission_mode`, `session_id`, `source`, and `transcript_path`; `source` is one of `startup`, `resume`, or `clear`. Primary source: https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/schema/generated/session-start.command.input.schema.json
- Q2: For `UserPromptSubmit`, plain text written to stdout is added as extra developer context. JSON stdout can instead use `hookSpecificOutput.additionalContext`, which is also added as extra developer context. Primary source: https://developers.openai.com/codex/hooks
- Q2: Source-level parser evidence matches the docs: `UserPromptSubmit` stores accepted stdout context in `additional_contexts_for_model`, while tests show `decision:"block"` with `additionalContext` both blocks and records the context entry. Primary source: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/user_prompt_submit.rs
- Q2: For `SessionStart`, plain text stdout is accepted as model context unless it looks like malformed JSON; JSON stdout may use `hookSpecificOutput.additionalContext`. Source code appends either accepted shape to `additional_contexts_for_model`. Primary sources: https://developers.openai.com/codex/hooks and https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/session_start.rs
- Q2: The injection role is developer context, not replacement of the user prompt. Official docs repeatedly say `additionalContext` is added as extra developer context; an upstream issue identifies the core path as `DeveloperInstructions::new(additional_context).into()`, confirming model-visible developer-message injection semantics. Primary sources: https://developers.openai.com/codex/hooks and https://github.com/openai/codex/issues/16486
- Q5/Q6 collateral update: Current official docs and discovery source expose six configurable events, not only the three in the pre-research state: `PreToolUse`, `PermissionRequest`, `PostToolUse`, `SessionStart`, `UserPromptSubmit`, and `Stop`. The docs also state multiple matching command hooks are launched concurrently. Primary sources: https://developers.openai.com/codex/hooks and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/discovery.rs
- Q4/Q5 collateral update: `timeout` is in seconds, `timeoutSec` is an alias, omitted timeout defaults to 600 seconds, and the dispatcher launches matching handlers with `join_all`, then folds results in handler order. Primary sources: https://developers.openai.com/codex/hooks and https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/dispatcher.rs
- Q8 collateral update: Hook commands run with the session `cwd` as working directory. Source command construction sets current directory and stdio but does not call `env_clear` or define a sanitized environment in the runner, supporting the current inference that hook processes inherit the parent/session environment unless a higher-level shell environment policy intervenes. Primary sources: https://developers.openai.com/codex/hooks and https://raw.githubusercontent.com/openai/codex/main/codex-rs/hooks/src/engine/command_runner.rs
## Questions Answered
- Q1 advanced strongly: `SessionStart` and `UserPromptSubmit` pass structured JSON on stdin; generated schemas give the exact required fields for each event.
- Q2 advanced strongly: `UserPromptSubmit` stdout can inject model-visible developer context via either plain text stdout or JSON `hookSpecificOutput.additionalContext`; `SessionStart` also supports plain text or JSON additional context. Evidence supports additive developer-context injection, not prompt replacement.
- Q4 advanced slightly: official docs confirm default and per-hook timeout budget, but this pass did not run a fresh timeout probe.
- Q5 advanced slightly: official docs and dispatcher source confirm concurrent launch of matching hooks.
- Q6 advanced: current docs/source expose six configurable events; this supersedes the earlier three-event pre-research assumption for the current upstream contract.
- Q8 advanced slightly: docs/source reinforce working-directory behavior and inherited-environment inference, but a higher-level env-policy audit remains open.
