# Iteration 7: Q1/Q2 stdin and stdout contract consolidation
## Focus
This pass rechecked Q1 and Q2 against current official Codex hook documentation, upstream source, generated schemas, and local 0.122.0 installation evidence. The goal was to decide whether the stdin and stdout-to-context contract is now synthesis-ready, without modifying `~/.codex/hooks.json`.
## Actions Taken
1. Read the official Codex hooks documentation for runtime behavior, common stdin fields, timeout defaults, `SessionStart`, and `UserPromptSubmit`: https://developers.openai.com/codex/hooks
2. Opened upstream `command_runner.rs` to verify command hooks are spawned with piped stdin/stdout/stderr, receive serialized `input_json` via stdin, run with the session cwd, and use per-handler timeout seconds: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/command_runner.rs
3. Opened generated `SessionStart` and `UserPromptSubmit` command input schemas to confirm exact JSON stdin fields: https://github.com/openai/codex/blob/main/codex-rs/hooks/schema/generated/session-start.command.input.schema.json and https://github.com/openai/codex/blob/main/codex-rs/hooks/schema/generated/user-prompt-submit.command.input.schema.json
4. Re-read `SessionStart` and `UserPromptSubmit` event parsers to verify plain stdout and JSON `hookSpecificOutput.additionalContext` become `additional_contexts_for_model`, while malformed JSON-like stdout fails: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/session_start.rs and https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/user_prompt_submit.rs
5. Opened upstream runtime context injection path to verify accepted hook additional contexts are carried through `SessionStartOutcome` and `UserPromptSubmitOutcome` into runtime additional contexts: https://github.com/openai/codex/blob/main/codex-rs/core/src/hook_runtime.rs
6. Rechecked local read-only evidence: `codex --version` reports `codex-cli 0.122.0`; `~/.codex/hooks.json` still registers only Superset `notify.sh` for `SessionStart`, `UserPromptSubmit`, and `Stop`; `notify.sh` can read stdin JSON. No modification was made to `~/.codex/hooks.json`.
## Findings
- Q1: Codex's documented command-hook transport is a single JSON object on stdin, not argv or plain text; the common fields are `session_id`, `transcript_path`, `cwd`, `hook_event_name`, and `model`, with `turn_id` added on turn-scoped events. Primary source: https://developers.openai.com/codex/hooks
- Q1: Source confirms the docs at the transport layer: `command_runner.rs` builds a process with piped stdin/stdout/stderr and writes the serialized `input_json` bytes to the child stdin. Primary source: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/command_runner.rs
- Q1: `SessionStart` stdin is a JSON object requiring `cwd`, `hook_event_name:"SessionStart"`, `model`, `permission_mode`, `session_id`, `source`, and `transcript_path`; the schema's `source` enum includes `startup`, `resume`, and `clear`, which is slightly broader than the docs table that mentions startup/resume. Primary source: https://github.com/openai/codex/blob/main/codex-rs/hooks/schema/generated/session-start.command.input.schema.json
- Q1: `UserPromptSubmit` stdin is a JSON object requiring `cwd`, `hook_event_name:"UserPromptSubmit"`, `model`, `permission_mode`, `prompt`, `session_id`, `transcript_path`, and `turn_id`, with `additionalProperties:false`. Primary source: https://github.com/openai/codex/blob/main/codex-rs/hooks/schema/generated/user-prompt-submit.command.input.schema.json
- Q2: For both `SessionStart` and `UserPromptSubmit`, official docs say plain stdout is added as extra developer context, and JSON stdout can provide `hookSpecificOutput.additionalContext`, also added as extra developer context. Primary source: https://developers.openai.com/codex/hooks
- Q2: Source confirms the stdout parser semantics: valid plain text and valid `additionalContext` are appended into `additional_contexts_for_model`; stdout that begins like JSON but fails the expected schema is marked as failed rather than silently injected. Primary sources: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/session_start.rs and https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/user_prompt_submit.rs
- Q2: The context path is additive model context, not prompt replacement. The docs call it extra developer context, and the runtime maps `SessionStartOutcome` and `UserPromptSubmitOutcome` additional contexts into the hook runtime's `additional_contexts` path for later recording. Primary sources: https://developers.openai.com/codex/hooks and https://github.com/openai/codex/blob/main/codex-rs/core/src/hook_runtime.rs
## Questions Answered
- Q1: Advanced to synthesis-ready. Official docs, generated schemas, source transport code, prior 0.122.0 probe evidence from iteration 2, and this run's local version/config check all converge on JSON-object stdin for `SessionStart` and `UserPromptSubmit`.
- Q2: Advanced to synthesis-ready. Official docs and source parsers agree that `SessionStart` and `UserPromptSubmit` stdout can inject extra developer context through either plain text or JSON `hookSpecificOutput.additionalContext`; this is additive context, not prompt transformation or replacement.
## Questions Remaining

- Q4: Timeout budget and timeout outcome are substantially evidenced, but final synthesis should reconcile default `600`, configured `timeout`, source timeout errors, and empirical skip/continue behavior.
- Q5: Multi-hook behavior is documented as concurrent; final synthesis should explain that registration order is not an execution-order guarantee.
- Q6: Full event taxonomy is well evidenced as six configurable events, but final synthesis should reconcile that with the local home config only registering three.
- Q7: `UserPromptSubmit` blocking is evidenced and prompt transformation appears unsupported; final synthesis should cite output schema/parser absence explicitly.
- Q8: Environment propagation remains the weakest open item; source implies inherited environment because the command runner does not clear or override env, but this still needs focused source audit or an isolated probe.
