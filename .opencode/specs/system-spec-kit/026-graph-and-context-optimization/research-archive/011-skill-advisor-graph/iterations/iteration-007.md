## Summary
This iteration audited the `sk-util` and `sk-code` skill families requested in the prompt. The `sk-code-*` sibling mesh is already clean and fully symmetric at the expected `0.4` weight, so no sibling normalization is needed there. The meaningful graph-quality gaps in this slice are narrower: `sk-doc` still looks under-modeled as a zero-edge orphan despite explicit downstream integration with `system-spec-kit`, and `sk-improve-prompt` still looks under-connected to the four CLI orchestrator skills even though those skills now ALWAYS load the mirrored prompt-quality card before dispatch. `sk-git` looks adjacent to code-quality skills, but not like a true overlay that should get direct `enhances` edges to `sk-code-*`.

## Findings
1. The `sk-code` sibling weights are already perfectly consistent and do not need adjustment. `sk-code-full-stack`, `sk-code-opencode`, and `sk-code-web` each declare the other two as `siblings` at `0.4`, with matching `code quality peer` context strings, which is exactly within the packet's `siblings` weight band of `0.4-0.6` (`.opencode/skill/sk-code-full-stack/graph-metadata.json:6-19`, `.opencode/skill/sk-code-opencode/graph-metadata.json:6-19`, `.opencode/skill/sk-code-web/graph-metadata.json:6-19`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/007-skill-advisor-graph/spec.md:113-121`). This part of the graph looks internally complete.

2. `sk-doc` is probably under-modeled as a zero-edge utility skill. Its metadata currently declares no edges at all (`.opencode/skill/sk-doc/graph-metadata.json:1-15`), but both sides of the workflow say it participates directly in spec-folder documentation flows. `system-spec-kit` defines `Documentation Quality` as `system-spec-kit -> sk-doc` and lists `sk-doc` as a downstream related skill for documentation alignment (`.opencode/skill/system-spec-kit/SKILL.md:921-922`, `.opencode/skill/system-spec-kit/SKILL.md:943-950`), while `sk-doc` itself lists `system-spec-kit` as an integration point because spec-folder documentation structure is validated there (`.opencode/skill/sk-doc/SKILL.md:802-808`). The cleanest graph fix looks like an `enhances` relationship, most plausibly `sk-doc -> system-spec-kit` around `0.5-0.6`, because `sk-doc` adds documentation-quality value alongside the foundational spec workflow without being a hard prerequisite.

3. This pass strengthens the earlier "missing `system-spec-kit` edges" finding rather than weakening it. `system-spec-kit` still has a completely empty edge set in `graph-metadata.json` (`.opencode/skill/system-spec-kit/graph-metadata.json:1-15`) even though its own related-skills table names downstream `sk-code-opencode`, `sk-git`, and `sk-doc` relationships (`.opencode/skill/system-spec-kit/SKILL.md:943-950`). The `sk-doc` evidence above shows that the hub's sparsity is not limited to code paths; documentation workflow links are also missing from the graph surface.

4. `sk-git` should probably remain unlinked from the `sk-code-*` family directly, even though those skills refer users to it. `sk-git` currently has no edges (`.opencode/skill/sk-git/graph-metadata.json:1-15`), and the code-quality skills consistently frame git as a separate hand-off rather than an overlay: `sk-code-full-stack` says "Git/version-control workflows (use sk-git)" and lists `sk-git` in related skills (`.opencode/skill/sk-code-full-stack/SKILL.md:34-40`, `.opencode/skill/sk-code-full-stack/SKILL.md:648-653`); `sk-code-web` does the same for git/version control operations (`.opencode/skill/sk-code-web/SKILL.md:40-45`, `.opencode/skill/sk-code-web/SKILL.md:641-644`); `sk-code-opencode` also treats `sk-git` as a separate neighbor, not a code-standard overlay (`.opencode/skill/sk-code-opencode/SKILL.md:40-44`, `.opencode/skill/sk-code-opencode/SKILL.md:538-542`). Because the packet's `enhances` edges receive meaningful transitive damping (`0.30`) (`.opencode/specs/system-spec-kit/026-graph-and-context-optimization/007-skill-advisor-graph/plan.md:90-95`), direct `sk-git -> sk-code-*` links would likely over-promote git workflow advice on ordinary implementation intents. If this relationship is modeled at all, `system-spec-kit -> sk-git` is higher-signal than code-quality-to-git edges.

5. `sk-improve-prompt` looks under-connected to the CLI skill family and now has concrete evidence for explicit graph links. Its own skill body says it supports indirect invocation from `cli_prompt_quality_card.md` fast-path flows, treats `assets/cli_prompt_quality_card.md` as the external-consumption surface, and explicitly says that file is the entry point whose mirrors must stay synced across all four CLI skills (`.opencode/skill/sk-improve-prompt/SKILL.md:23-28`, `.opencode/skill/sk-improve-prompt/SKILL.md:339-343`, `.opencode/skill/sk-improve-prompt/SKILL.md:424-432`). The four CLI skills now all declare `assets/prompt_quality_card.md` in their `ALWAYS` loading set and all require loading that asset before building dispatch prompts (`.opencode/skill/cli-claude-code/SKILL.md:108-113`, `.opencode/skill/cli-claude-code/SKILL.md:429-431`, `.opencode/skill/cli-codex/SKILL.md:104-109`, `.opencode/skill/cli-codex/SKILL.md:446-447`, `.opencode/skill/cli-copilot/SKILL.md:102-107`, `.opencode/skill/cli-copilot/SKILL.md:384-385`, `.opencode/skill/cli-gemini/SKILL.md:101-106`, `.opencode/skill/cli-gemini/SKILL.md:373-374`). The recent packet work also documented that the router contract is now to ALWAYS load `references/cli_reference.md` plus `assets/prompt_quality_card.md` across all four CLI skills, with the canonical source living under `sk-improve-prompt` (`/Users/michelkerkmeester/.codex/memories/rollout_summaries/2026-04-11T18-24-33-993y-cli_prompt_quality_mirror_card_and_router_alignment.md:62-66`, `/Users/michelkerkmeester/.codex/memories/rollout_summaries/2026-04-11T18-24-33-993y-cli_prompt_quality_mirror_card_and_router_alignment.md:68-74`). That is strong enough to justify `enhances` edges from `sk-improve-prompt` to `cli-claude-code`, `cli-codex`, `cli-copilot`, and `cli-gemini`, likely in the `0.4-0.5` band. I would not model these as `depends_on`, because the CLI skills can still function with mirrored assets even when `sk-improve-prompt` is not directly invoked.
