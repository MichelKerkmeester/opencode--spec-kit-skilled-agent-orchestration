# Iteration 5: D4 Root Cause — Importance Tier Mismatch

## Focus
Trace defect D4 to the exact code paths that populate `importance_tier`, explain why the top frontmatter can diverge from the `## MEMORY METADATA` YAML block, determine whether either layer is authoritative for indexing, and verify whether post-save review reconciles the mismatch. The key clarification from this pass is that the initial template render uses one shared placeholder for both locations, so the split must be introduced later by a frontmatter-only rewrite path. [CITATION: `.opencode/skill/system-spec-kit/templates/context_template.md:1-6`, `.opencode/skill/system-spec-kit/templates/context_template.md:755-757`]

## Hypothesis
Because the template repeats `{{IMPORTANCE_TIER}}` in both frontmatter and the bottom metadata block, the working hypothesis was that D4 could not originate in `populateTemplate()` itself. The likely defect shape was a second writer that mutates only managed frontmatter after the file is rendered, plus a reviewer/indexer path that treats frontmatter as authoritative and ignores the bottom block. [CITATION: `.opencode/skill/system-spec-kit/templates/context_template.md:1-6`, `.opencode/skill/system-spec-kit/templates/context_template.md:755-757`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:8-9`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:77-107`]

## Evidence
- The save pipeline extracts an explicit tier from JSON payload fields `importanceTier` or `importance_tier`, passes that into `detectSessionCharacteristics(...)`, and returns the resolved value as `IMPORTANCE_TIER` for template population. This is the writer for the original rendered value in both locations. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:883-900`, `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:1024-1065`]
- `context_template.md` uses the same `{{IMPORTANCE_TIER}}` placeholder in top frontmatter and again under `# Classification` in the `## MEMORY METADATA` YAML block. On first render, those two values must match unless a later stage rewrites only one of them. [CITATION: `.opencode/skill/system-spec-kit/templates/context_template.md:1-6`, `.opencode/skill/system-spec-kit/templates/context_template.md:755-757`]
- `frontmatter-editor.ts` is not the second tier writer. The exported functions in this module inject quality metadata, spec-folder health metadata, trigger phrases, and semantic topics, but they do not add or recalculate `importance_tier`. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/frontmatter-editor.ts:27-92`, `.opencode/skill/system-spec-kit/scripts/core/frontmatter-editor.ts:95-152`]
- The separate frontmatter-only writer lives in `frontmatter-migration.ts`. `buildManagedFrontmatter()` computes `importance_tier` through `inferImportanceTier(...)`, and `serializeFrontmatter(...)` writes that managed value back into the frontmatter block only. `inferImportanceTier(...)` prefers an existing frontmatter tier, otherwise parses the body table row `| Importance Tier | ... |`, otherwise falls back to a document-type default. It does not read or update the `## MEMORY METADATA` YAML block. [CITATION: `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1006-1024`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1112-1125`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1150-1183`]
- The template includes a body table row `| Importance Tier | {{IMPORTANCE_TIER}} |`, so the migration path has its own body-derived source for rehydrating frontmatter later. That means frontmatter can be rewritten from the session summary table even if the bottom metadata block still carries an older value. [CITATION: `.opencode/skill/system-spec-kit/templates/context_template.md:129-130`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1015-1023`]
- Post-save review can parse both frontmatter and the bottom `## MEMORY METADATA` block, but the D4 check compares only payload tier vs frontmatter tier and proposes only a frontmatter fix. There is no reconciliation check for frontmatter vs metadata-block mismatch, and no automatic patch step in this module. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:8-9`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:56-75`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:77-107`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:279-289`]
- The immediate script-side workflow indexer does not pass an `importance_tier` string into `vectorIndex.indexMemory(...)`; it only computes a numeric `importanceWeight`. By contrast, the MCP save path parses `importanceTier` from the memory content and persists `parsed.importanceTier` into `memory_index.importance_tier`. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/memory-indexer.ts:139-172`, `.opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:251-279`, `.opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:629-666`, `.opencode/skill/system-spec-kit/mcp_server/handlers/save/create-record.ts:215-219`]
- The MCP parser’s `extractImportanceTier(...)` reads only the extracted frontmatter block (`extractFrontmatterBlock(...)`) when resolving `importance_tier`. Despite the comment mentioning “content or metadata,” this implementation does not inspect the bottom `## MEMORY METADATA` YAML block. [CITATION: `.opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:629-666`]

## Root Cause
D4 is a multi-path consistency defect, not a template bug.

1. **Original save path:** `collect-session-data.ts` resolves one `IMPORTANCE_TIER` and the template renders that same value into both frontmatter and the bottom metadata block. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:883-900`, `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:1024-1065`, `.opencode/skill/system-spec-kit/templates/context_template.md:1-6`, `.opencode/skill/system-spec-kit/templates/context_template.md:755-757`]
2. **Second writer:** a later frontmatter-maintenance path in `frontmatter-migration.ts` can independently recompute `importance_tier` for the top frontmatter from either existing frontmatter, the body table row, or document defaults, then serialize only the managed frontmatter block. The bottom `## MEMORY METADATA` YAML block is untouched. [CITATION: `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1006-1024`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1112-1125`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1150-1183`]
3. **Why values diverge:** the two paths use different sources and defaults. The original renderer takes payload-or-session-characteristic tier from `collect-session-data.ts`, while the migration path can rehydrate frontmatter from the body summary table or document defaults. Once the frontmatter-only path runs, the top value can drift away from the unchanged bottom block. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:883-900`, `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:1024-1065`, `.opencode/skill/system-spec-kit/templates/context_template.md:129-130`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1015-1024`]
4. **Which one wins:** for the authoritative MCP parsing/persistence path, frontmatter wins. `memory-parser.ts` extracts `importanceTier` from the frontmatter block only, and `create-record.ts` writes that parsed value into `memory_index.importance_tier`. The bottom metadata block is ignored for tier parsing. The script-side workflow indexer does not consume either rendered tier directly; it only computes `importanceWeight` in-process. [CITATION: `.opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:251-279`, `.opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:629-666`, `.opencode/skill/system-spec-kit/mcp_server/handlers/save/create-record.ts:215-219`, `.opencode/skill/system-spec-kit/scripts/core/memory-indexer.ts:139-172`]
5. **Why review does not patch it:** post-save review detects only payload-vs-frontmatter mismatches and emits advisory issues. It does not compare frontmatter against the metadata block and does not mutate the saved file. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:8-9`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:77-107`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:279-289`]

## Verification
- There is no D4-producing second tier assignment in the `workflow.ts` write slice itself; the save path computes `IMPORTANCE_TIER` once and passes that through template population. The inspected workflow section contains trigger/topic extraction and then `populateTemplate('context', ...)`, but no later `importance_tier` override. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1198-1325`]
- `frontmatter-editor.ts` cannot explain `high` in frontmatter vs `normal` in metadata because its mutation utilities never touch tier fields. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/frontmatter-editor.ts:27-92`, `.opencode/skill/system-spec-kit/scripts/core/frontmatter-editor.ts:95-152`]
- `frontmatter-migration.ts` can explain a frontmatter-only override because its serializer rewrites just the top `--- ... ---` block. [CITATION: `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1150-1183`]
- The authoritative parser ignores the bottom metadata block for tier extraction, so a mismatched frontmatter value is the one that gets stored and later searched. [CITATION: `.opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:629-666`, `.opencode/skill/system-spec-kit/mcp_server/handlers/save/create-record.ts:215-219`]

## Defect Chain
JSON payload or session heuristics -> `collect-session-data.ts` resolves `importanceTier` -> `context_template.md` renders one shared `{{IMPORTANCE_TIER}}` into both frontmatter and `## MEMORY METADATA` -> later frontmatter maintenance/migration can recompute and rewrite only the top frontmatter tier from existing frontmatter, the body `Importance Tier` table row, or document defaults -> bottom metadata block remains stale -> post-save review checks payload vs frontmatter only and emits advice without patching -> MCP memory parser reads frontmatter tier only -> `create-record.ts` persists that frontmatter-derived tier into `memory_index.importance_tier`. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:883-900`, `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:1024-1065`, `.opencode/skill/system-spec-kit/templates/context_template.md:1-6`, `.opencode/skill/system-spec-kit/templates/context_template.md:129-130`, `.opencode/skill/system-spec-kit/templates/context_template.md:755-757`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1006-1024`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1112-1125`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1150-1183`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:279-289`, `.opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:629-666`, `.opencode/skill/system-spec-kit/mcp_server/handlers/save/create-record.ts:215-219`]

## Findings
1. The original save writer for `importance_tier` is `collect-session-data.ts` feeding `IMPORTANCE_TIER` into `context_template.md`, and that single value is rendered into both locations on first write. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:883-900`, `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:1024-1065`, `.opencode/skill/system-spec-kit/templates/context_template.md:1-6`, `.opencode/skill/system-spec-kit/templates/context_template.md:755-757`]
2. The second independent tier writer is not `frontmatter-editor.ts`; it is the managed-frontmatter rewrite path in `frontmatter-migration.ts`. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/frontmatter-editor.ts:27-92`, `.opencode/skill/system-spec-kit/scripts/core/frontmatter-editor.ts:95-152`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1112-1125`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1150-1183`]
3. The two paths diverge because they use different tier sources: payload/session-characteristic resolution during initial render versus existing frontmatter, body table extraction, or document-type defaults during frontmatter migration. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:883-900`, `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1006-1024`, `.opencode/skill/system-spec-kit/templates/context_template.md:129-130`]
4. For authoritative MCP parsing and DB persistence, frontmatter wins. The bottom metadata block is not consulted by `extractImportanceTier(...)`. [CITATION: `.opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:629-666`, `.opencode/skill/system-spec-kit/mcp_server/handlers/save/create-record.ts:215-219`]
5. For the script-side workflow indexer, neither rendered tier directly drives `vectorIndex.indexMemory(...)`; only `importanceWeight` is computed there. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/memory-indexer.ts:139-172`]
6. The post-save reviewer cannot repair D4 today because it only checks payload-vs-frontmatter mismatch and returns advisory issues instead of mutating the file, while ignoring frontmatter-vs-metadata-block drift entirely. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:8-9`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:77-107`, `.opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:279-289`]

## Question Status
Q4 RESOLVED: D4 is caused by two separate tier paths. Initial save renders one shared `IMPORTANCE_TIER` into both places, but a later managed-frontmatter rewrite can change only the top frontmatter tier. Post-save review does not reconcile the split, and the authoritative MCP parser/indexing path reads the frontmatter value rather than the bottom metadata block.
