# Iteration 5: get-token-insights ingestion + dashboard contract

## Focus
This iteration examined how the `get-token-insights` skill turns raw Claude JSONL session logs into queryable analytics tables, then converts those tables into pricing, cache, workflow, and dashboard outputs. The goal was to isolate reusable observability patterns for Code_Environment/Public without copying plugin-specific presentation code.

## Findings
- `external/plugins/claude-memory/skills/get-token-insights/scripts/ingest_token_data.py` in `discover_jsonl_files()` and `parse_session()` (`282-297`, `394-580`) ingests both top-level session JSONL files and nested `*/subagents/*.jsonl`, then groups assistant events by `message.id` into one logical turn while updating usage to the latest event, extracting `input_tokens`, `output_tokens`, `cache_read_input_tokens`, `cache_creation_input_tokens`, ephemeral 5m/1h cache tiers, tool metadata, tool-result errors, `turn_duration`, and hook summaries. This matters for Code_Environment/Public because it goes beyond stop-hook token snapshots and yields a reusable turn/session/hook relational model for later analytics. Recommendation: (adopt now)
- `external/plugins/claude-memory/skills/get-token-insights/scripts/ingest_token_data.py` in `MODEL_PRICING`, `_get_pricing()`, `_turn_cost()`, and `build_output()` (`62-96`, `910-967`) normalizes model economics by substring-matching model IDs, defaulting unknown models to Sonnet pricing, and splitting cost into uncached input, output, cache read, 5-minute cache writes, and 1-hour cache writes; any cache creation not explicitly classified is conservatively billed as 5-minute cache write. This matters for Code_Environment/Public because the existing `system-spec-kit/024-compact-code-graph/003-stop-hook-tracking` work appears to be token-tracking-first by reference, while this skill already provides the missing per-model and per-cache-tier cost normalization needed for trustworthy spend reporting. Recommendation: (adopt now)
- `external/plugins/claude-memory/skills/get-token-insights/scripts/ingest_token_data.py` in `compute_session_analytics()`, `build_output()`, `build_trends()`, and `_build_insights()` (`604-655`, `974-989`, `1314-1323`, `1427-1518`, `1669-1874`) defines a cache cliff as a `cache_read_ratio` drop greater than `0.5` after a `user_gap_ms` longer than `300_000`, stores per-session cliff counts, and then derives week-on-week metrics such as `cost_per_session`, `cache_ratio`, `cliffs_per_session`, `tool_error_rate`, and `hook_avg_ms`. For Code_Environment/Public, this is the strongest contrast with the stop-hook tracking packet by reference: it does not just count hook time or tokens, it turns idle gaps plus cache behavior into workflow diagnostics and dollar-impact insights. Recommendation: (adopt now)
- `external/plugins/claude-memory/skills/get-token-insights/scripts/ingest_token_data.py` and `external/plugins/claude-memory/skills/get-token-insights/templates/dashboard.html` in the skill/agent/hook sections (`1238-1310`, `2011-2018`; `473-496`, `981-1067`, `1118-1286`) derive `skill_usage` from normalized Skill tool calls, `agent_delegation` from `subagent_type` counts plus error counts, `agent_model_dist` from explicit model overrides, and `hook_performance` from per-command run count, total runtime, average runtime, and errors; the dashboard then consumes those arrays directly for a Section 6 "Claude Code Ecosystem" view plus trend badges and disable-candidate tables. This matters for Code_Environment/Public because the existing stop-hook token work by reference should borrow the JSON contract and metric shapes, but probably not the full HTML bundle; the reusable piece is the analytics payload, not the plugin-specific dashboard chrome. Recommendation: (prototype later)
