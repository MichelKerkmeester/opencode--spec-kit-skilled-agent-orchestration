## Findings
- [F-MR-022] The smallest recall-safe payload is not just `title + trigger phrases`. Save-time sufficiency requires a specific title, at least one primary evidence item, at least two total evidence items, enough semantic substance (`>=250` chars or `>=40` unique words), at least one anchor, and meaningful trigger phrases. The evidence snapshot can satisfy those requirements from compact files, observations, decisions, outcomes, next action, blockers, and anchor-bearing body text; it does not require full repeated ADR or implementation-summary prose. - [F-MR-023] Auto-surface trigger matching gets most of its value from trigger phrases, not from long body sections. The trigger handler returns full content only for HOT memories, truncates WARM memories to the first 150 characters, and excludes COLD or DORMANT bodies entirely. That means large DECISIONS and CONVERSATION sections add little trigger-time value for most surfaced memories. [SOURCE: .opencode/skill/system-spec-kit/mcp_server/handlers/memory-triggers.ts:156-170] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/handlers/memory-triggers.ts:444-520]
- [F-MR-024] Search and parsing still want compact semantic body text, but not a second packet narrative. `parseMemoryContent()` indexes title, trigger phrases, context type, importance tier, and full content hash, while `memory_search` treats `includeContent` as optional in the hybrid pipeline. The right minimum is therefore a short distinguishing overview plus a few evidence-rich bullets and anchors, not a replay of every decision, file list, and conversation turn. - [F-MR-025] The true memory-only payload lives in continuation and machine metadata, not in packet-summary restatement. The existing code already carries unique recall value through continuation state, tool-call and exchange summaries, memory classification/decay settings, dedup fingerprinting, causal links, and postflight learning deltas. These fields should survive any reduction because no static spec doc currently owns them. - [F-MR-026] The collector is currently assembling the exact redundancy we want to remove. `collect-session-data.ts` builds `OUTCOMES`, `SUMMARY`, `OBSERVATIONS_DETAILED`, `SPEC_FILES`, `implementationGuide`, and project-state snapshots, while `buildImplementationGuideData()` expands implementation observations into “what was built / key files / how to extend / common patterns.” Those fields are useful as raw evidence inputs, but they should no longer be rendered as first-class packet-summary sections when canonical static docs already exist. - [F-MR-027] The concrete fix belongs in the workflow binding plus template contract. `workflow.ts` currently injects full `DECISIONS`, embedded implementation-summary fragments, `IMPL_*` arrays, key files, and other packet-summary material into `context_template.md`, while the template contract still marks `CONTINUE SESSION`, `PROJECT STATE SNAPSHOT`, `DECISIONS`, `CONVERSATION`, `RECOVERY HINTS`, and `MEMORY METADATA` as mandatory. To eliminate redundancy, the default memory should instead render: (1) a compact continuation block, (2) canonical-doc pointers, (3) a short distinguishing overview plus evidence bullets, (4) compact recovery hints, (5) optional postflight delta, and (6) metadata. Full decisions, implementation guide, detailed changes, and conversation transcript should become conditional or removed from default saves. - [F-MR-028] Canonical ownership can be simplified without changing static templates. `decision-record.md` should remain the canonical owner of ADR prose when that file exists; `implementation-summary.md` should remain the canonical owner of build story, file changes, verification, and limitations; and research-heavy packets may add a third canonical pointer to `research/research.md` or `review/review-report.md`. No required addition to the static templates is necessary because those templates already expose the sections the memory file is duplicating; at most, a lightweight “Canonical companion docs” note could be added later for author convenience, but it is not required for the contract change.
