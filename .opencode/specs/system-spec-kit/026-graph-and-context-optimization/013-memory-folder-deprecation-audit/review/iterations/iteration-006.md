# Iteration 6: Security audit — memory/*.md write path (path traversal, content injection, TOCTOU, containment)

## Focus

Single-dimension security audit of the `[spec]/memory/*.md` write-path surface (filename construction at `workflow.ts:1252`, slug generation at `slug-utils.ts:104-276`, atomic commit + containment at `file-writer.ts:132-255`, timestamp formatting at `message-utils.ts:106-109`). Four attack vectors evaluated: (1) path traversal via user-influenceable slug/topic/trigger-phrases/session_summary, (2) content-injection into the `${DATE}_${TIME}__${slug}.md` template, (3) TOCTOU race between `existsSync` dedup preflight (`workflow.ts:1630`) and atomic commit (`file-writer.ts:187`), (4) containment-assertion gaps in `writeFilesAtomically`.

### P2
- **F022**: Two-phase containment relies on symbolic-guarantee from `path.sep` suffix — `file-writer.ts:56,156` — The `startsWith(resolvedContextDir + path.sep)` check at both `writeFilesAtomically` (line 156, pre-rename) and `verifyResolvedWriteTarget` (line 56, via `!==` + `startsWith` pair) is semantically correct but relies on `path.sep` being appended. On platforms where the separator changes (e.g., Windows with forward slashes in a mixed-normalization path from realpath), a sibling directory whose name shares a prefix with `contextDir` (e.g., `memory/` vs `memory-archive/`) could theoretically evade the check IF `resolvedContextDir` were computed without the trailing separator AND the sibling existed AND its realpath collided on prefix. Current code appends `path.sep` explicitly at both sites, closing this gap. This is a defense-in-depth P2 observation, not an exploit — the current implementation is correct. Recommendation (informational): add a vitest fixture asserting that a path like `/tmp/memory` vs `/tmp/memory-archive/file.md` correctly fails containment, to lock the invariant.

## Ruled Out

- **Path traversal via `slug`**: `slug-utils.ts:53-61` `toUnicodeSafeSlug` whitelists only `[a-z0-9-]` after Unicode normalization — `/`, `\`, `..`, `\0`, control chars, and `.md` are all collapsed to `-`. User-supplied topic/sessionSummary/trigger-phrases/memoryNameHistory cannot inject path separators. Truncation to 200 chars at entry (`slug-utils.ts:106`) plus `DEFAULT_SLUG_MAX_LENGTH=50` at `truncateSlugAtWordBoundary` (`slug-utils.ts:180-188`) bound the component size.
- **Path traversal via `DATE`/`TIME`**: `collect-session-data.ts:1554-1555` assigns `DATE = formatTimestamp(now, 'date-dutch')` and `TIME = formatTimestamp(now, 'time-short')`. `message-utils.ts:97-109` derives both from `Date.toISOString()` split on `T` / `:` — the output is strictly `DD-MM-YY` / `HH-MM`. `now` is a machine-generated `Date` object, NOT user input. Zero injection surface.
- **Content injection into filename template**: The template `${sessionData.DATE}_${sessionData.TIME}__${contentSlug}.md` (`workflow.ts:1252`) concatenates three machine-sanitized strings — no shell/YAML/markdown parsing occurs on the filename; it's passed to `path.join` which treats the basename as an opaque string. The only downstream parsers (frontmatter YAML, markdown title) operate on file CONTENT, not the filename.
- **TOCTOU between `existsSync` (workflow.ts:1630) and atomic commit (file-writer.ts:187)**: Race window analysis — if a concurrent process writes `existingRawPath` between `existsSync` (line 1630) and `ensureUniqueMemoryFilename` (line 1636), the race affects only the DEDUP OUTCOME (whether the file is treated as duplicate vs suffixed), not filesystem safety. `ensureUniqueMemoryFilename` (`slug-utils.ts:198-251`) uses atomic `O_CREAT|O_EXCL` probe-and-unlink, which correctly handles concurrent creation: on `EEXIST` it advances to `-1`, `-2`, ... `-100`, then 100 random hex suffixes. The final `writeFilesAtomically` leg at `file-writer.ts:162-187` uses `.tmp.<randomSuffix>` write + rename — also atomic. Worst-case outcome of the race is a `-N` suffixed file; no overwrite of an unrelated file, no escape from `contextDir`, no filesystem corruption. This is an acknowledged dedup weakness (see F017/F019), but NOT a security issue.
- **Symlink traversal inside `memory/`**: If an attacker pre-creates `memory/evil -> ../../etc/passwd` and the system is tricked into writing `evil`, `verifyResolvedWriteTarget` (`file-writer.ts:35-59`) calls `realpathSync(parentDir)` at line 48 before the rename at line 187, which resolves the symlink and then checks `!realFilePath.startsWith(realContextDir + path.sep)` — the check fails and the error is thrown. Defense-in-depth is intact.
- **`writeFilesAtomically` explicit `..` rejection** (`file-writer.ts:151-153`): Rejects `path.isAbsolute(filename) || filename.includes('..')` before any I/O — belt-and-suspenders rejection that complements the realpath-based check.
- **Race on `O_CREAT|O_EXCL` fd leakage in `ensureUniqueMemoryFilename`** (`slug-utils.ts:201-205`): The fd is always closed in the `try` block; on `EEXIST` the catch re-throws or advances. Error-path fd leak analysis: `fs.closeSync` at line 203 is unconditional on successful open; `fs.unlinkSync` at line 205 removes the placeholder. No leak.
- **`contextDir` user-influenceability**: `contextDir` is computed at `directory-setup.ts:85` as `path.join(sanitizedPath, 'memory')` where `sanitizedPath` has already passed `resolveSpecFolderAbsolute` (caller of `setupContextDirectory`). The `'memory'` literal is hard-coded. User cannot steer `contextDir` outside the resolved spec folder.
- **Trigger-phrase injection into filename**: `extractTriggerPhrases` output is consumed into FRONTMATTER (`workflow.ts:1296-1330+`), NOT into the filename. The filename only uses `contentSlug` derived from `preferredMemoryTask` which flows through `pickBestContentName → normalizeMemoryNameCandidate → slugify`.
