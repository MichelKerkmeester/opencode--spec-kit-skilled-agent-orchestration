### P1
- **F011**: v3.4.0.0 changelog mirrored under both `.claude/changelog/01--system-spec-kit/v3.4.0.0.md` and `.opencode/changelog/01--system-spec-kit/v3.4.0.0.md` asserts CURRENT-state retirement in four distinct passages — `.claude/changelog/01--system-spec-kit/v3.4.0.0.md:1` ("scattering knowledge across standalone memory files, the system NOW stores continuity directly inside the spec documents"), `:14` ("The old `memory/*.md` corpus IS retired"), `:18` ("The entire `memory/*.md` surface IS retired in this release"), `:30` ("There IS NO archived-memory fallback because the old memory corpus IS GONE"). These are present-tense commitments about the shipped v3.4.0.0 system, not historical description of a migration in progress. Runtime (F001-F007) writes `[spec]/memory/*.md` via `workflow.ts:1861 writeFilesAtomically(contextDir, files)` and `directory-setup.ts:85 mkdir [spec]/memory/`, and reads them via `workflow.ts:1629-1646` and `file-writer.ts:99-129`. The doc-vs-runtime contradiction is cited from two changelog locations that are referenced as authoritative by `/spec_kit:resume`, `/memory:save`, and the v3.4.0.0 release surface. Changelogs are normally frozen (NON-FINDING per strategy.md classification rule), BUT the explicit P1-DOC escalation rule in the audit question triggers here: the changelog is making claims about CURRENT state, not historical state. Tier: P1-DOC.
- **F012**: `.opencode/command/memory/manage.md:50` asserts "the retired `memory/*.md` surface IS NO LONGER ACCEPTED BY THE RUNTIME" as a top-of-command-card instruction. This is a present-tense runtime-behavior claim in a live operator command card, contradicted by F001-F007 (runtime actively writes AND accepts these files — `workflow.ts:1861` is the write path, `workflow.ts:1629-1646 + file-writer.ts:99-129` are the read paths that specifically expect memory/*.md to exist for dedup/glob/causal-link resolution). This doc surface directly guides operator expectations when running `/memory:manage` and is a sibling card to the already-flagged `save.md:551` (F008). Tier: P1-DOC. Severity justification: unlike a README, `manage.md` is loaded by the command-card runtime and quoted back to operators, so the false claim has operational impact (operators may assume they can "clean up" memory/*.md files and break the workflow.ts dedup read path).

### P2
- **F013**: `.opencode/skill/system-spec-kit/README.md:149` describes the current-state behavior of `create.sh`: *"The script sets up the folder, copies the right templates for the chosen level, initializes `description.json` and creates `memory/` and `scratch/` subdirectories."* This statement is ACCURATE per runtime (F001 at `directory-setup.ts:85` confirms unconditional `[spec]/memory/` mkdir still happens). But it directly contradicts the changelog's "memory/*.md corpus IS retired" claim when read side-by-side — the README casually documents memory/ directory creation while the changelog claims the surface is gone. This is a doc-surface-vs-doc-surface inconsistency (README tells truth about runtime; changelog lies about runtime; their juxtaposition leaves operators unable to reconcile which surface is authoritative). Tier: P2-DRIFT (not an independent falsehood, but amplifies the F011/F012 contradiction by showing the docs tell both stories simultaneously). The README line itself is NOT a finding — it's accurate; F013 captures the DRIFT that arises when an accurate current-state doc coexists with inaccurate "retired" claims elsewhere.
