# Iteration 5: Maintainability — Duplicate-new spawn root cause (save-side Session Deduplication contract vs runtime)

## Focus
Dimension: **maintainability**. Root-cause analysis of the 14 near-identical `[spec]/memory/YYYY-MM-DD__verification-save-*.md` duplicates spawned in `012-spec-kit-commands/memory/` during the Step 11.5 verification loop on 2026-04-14 at 18:32–18:49. Trace the exact save-time code path from `generate-context.js` → `runWorkflow` dedup/filename decision → atomic commit, and compare against the documented contract in `.opencode/command/memory/save.md:685-702` ("Session Deduplication"). Scope: `workflow.ts` (duplicate preflight 1628-1647 + pre-save overlap 1818-1853), `file-writer.ts::checkForDuplicateContent` (99-129), `slug-utils.ts::ensureUniqueMemoryFilename` (198-251), and a broad grep for the contract's named artefacts (`session_hash`, `dedup_status`, `previous_session_id`, `related_sessions`, `1h`/`3600000` threshold).

## Findings

### P0
- **F017**: Documented Session Deduplication contract is entirely unimplemented in the save runtime — `.opencode/command/memory/save.md:685-702` vs `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1628-1647, 1818-1853` + `.opencode/skill/system-spec-kit/scripts/core/file-writer.ts:99-129` + `.opencode/skill/system-spec-kit/scripts/utils/slug-utils.ts:198-251` — The command card specifies a four-step interactive dedup contract: (1) "Generate SHA-256 fingerprint of (topic + files + timeframe), compare against most recent memory in target folder" (save.md:689), (2) "If match AND time delta < 1 hour → DUPLICATE DETECTED" (save.md:689), (3) interactive operator prompt with four routes *Overwrite / Append / New / Cancel* (save.md:695-700), (4) persist `session_hash`, `dedup_status`, `previous_session_id`, `related_sessions[]` metadata to frontmatter (save.md:702). **None of this is implemented.** The actual save path executes three independent fallback heuristics that do not match the contract in any dimension:
  - **Heuristic A — filename-exact existsSync (workflow.ts:1630)**: `fsSync.existsSync(path.join(contextDir, rawCtxFilename))` where `rawCtxFilename = ${sessionData.DATE}_${sessionData.TIME}__${contentSlug}.md` (workflow.ts:1252). `sessionData.TIME` is recomputed at every `collect-session-data.ts` invocation via `formatTimestamp(now, 'time-short')` at **minute granularity (`HH-MM`)** (collect-session-data.ts:1554-1555). Two saves one minute apart produce different `rawCtxFilename`s, so `existsSync` returns false and dedup short-circuits to "no collision". This is why the 14 test-loop files each got a **fresh filename with different `HH-MM`** — the existsSync check was not the "Overwrite/Append/New/Cancel" gate; it silently produced the "New" outcome every time without the operator being asked.
  - **Heuristic B — content-hash sibling scan (file-writer.ts:99-129)**: `checkForDuplicateContent` hashes the full rendered file via SHA-256 and compares against every `.md` sibling. It is NOT a fingerprint of (topic + files + timeframe) as the contract specifies; it is a full-content hash that is trivially invalidated by any variation (a new timestamp embedded in the rendered template, a new message in the captured session). In the 14-file test loop every content-hash differed → no match → silent skip of the dedup route entirely. Also: this is the ONLY component whose "skip on hash match" semantics even roughly resemble the documented contract, and it applies no time window at all.
  - **Heuristic C — pre-save overlap check (workflow.ts:1818-1853)**: SHA-1 hash of `files[ctxFilename]`, compared against last 12 size-matching siblings, logs `[PRE-SAVE OVERLAP] ...possible duplicate save` and proceeds anyway (line 1842). This is the closest thing to an "advisory" dedup signal. It uses **SHA-1 not SHA-256**, fingerprints the rendered content (not topic+files+timeframe), is **advisory/fail-open** (set `SPECKIT_PRE_SAVE_DEDUP=false` to disable, line 1819), applies no 1h window, and never produces the operator-facing prompt the contract specifies.
  - **Heuristic D — ensureUniqueMemoryFilename (slug-utils.ts:198-251)**: When Heuristic A *does* fire (identical filename), this produces `{slug}-1.md`, `{slug}-2.md`, ..., `{slug}-N.md` via atomic `O_CREAT|O_EXCL`. This is NOT the "+1 minute" behavior the dispatch described; the "+1 minute" observed in the test loop is an **emergent artefact** of `sessionData.TIME` being freshly computed each save, not a conscious dedup decision.

  **None of the four heuristics:** (a) computes a SHA-256 fingerprint over (topic + files + timeframe), (b) applies a 1-hour threshold, (c) surfaces the Overwrite/Append/New/Cancel prompt, or (d) writes `session_hash`/`dedup_status`/`previous_session_id`/`related_sessions[]` to frontmatter. The claimed contract is **entirely documentary** — the 14-file spawn was not "dedup routed to new-file" as the operator hypothesised; it was **dedup-never-fired** because the runtime has no session-dedup layer whatsoever. Classifying as P0-maintainability: this is a correctness gap (contract claim ≠ behavior) that is silently accumulating database bloat and cross-session noise, and the documentation cannot be relied on by any operator or agent (including the future AI agent that will inherit this session).

- **F020**: Frontmatter metadata keys enumerated by the documented contract have zero runtime producers — `.opencode/command/memory/save.md:702` vs grep across `.opencode/skill/system-spec-kit/scripts/` — Search for `session_hash`, `dedup_status`, `previous_session_id`, `related_sessions`, `duplicate_new`, `duplicate_overwrite` produced exactly **three hits total in the entire scripts tree**, all in test fixtures (`test-folder-detector-functional.js:210,213` for `-1 hour` SQL literal; `test-extractors-loaders.js:227` for `3600000` millisecond literal). Zero runtime code sets any of these frontmatter keys. Zero runtime code reads them. The YAML contract is not just unimplemented — it is a phantom. Any downstream consumer (search result assembler, session-resume, `/memory:search`) that attempts to branch on `dedup_status` or traverse `related_sessions` will silently receive `undefined` for every memory in the corpus. Classifying as P0 because this is **not documentation drift** (which would be P1) but a **contract-shaped hole in the runtime** — the 14-file test-loop duplicates are the *observable* symptom of F017; F020 is the *structural* symptom showing the same gap at the metadata layer. Remediation paths diverge: F017 needs either a fingerprint+prompt implementation OR a contract retraction; F020 additionally requires either emitting the metadata OR removing it from the operator-facing schema.

### P1

This is a second operator-facing surface (distinct from save.md:685-702) that will mislead any agent following the documented recovery path. Classifying as P1-DOC-CONTRACT-VIOLATION at gate severity because the Recovery matrix is one of the two primary surfaces an agent reads after a save failure. Not P0 because the Recovery matrix itself is documentary; the write-path behavior is already fully captured by F017. Gate-relevant because any post-save triage by an agent will hit this table first and be misled.

### P2
- **F021**: The pre-save overlap check (`workflow.ts:1818-1853`) is structurally misaligned with the documented contract on four axes even if one chose to treat it as the implementation — `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1818-1853` vs `.opencode/command/memory/save.md:689` — (a) hashes via **SHA-1** (line 1823) vs contract's **SHA-256**, (b) fingerprints the rendered file vs contract's `(topic + files + timeframe)` composite, (c) applies **no time window** (contract: 1h strict, 1-4h review, >4h proceed), (d) is **advisory/fail-open** with `SPECKIT_PRE_SAVE_DEDUP` kill switch (1819) vs contract's interactive operator prompt. Classifying P2 not P1 because: this is structural misalignment (not a new family of bad operator-facing claims), it is already subsumed by F017's maintainability verdict, and its runtime effect is limited to a log line. Value of calling it out separately: it gives a remediation starting point — a minimal fix for F017 could extend THIS check (tighten to SHA-256, add timeframe window, convert warn to error-or-prompt) rather than writing a new dedup layer from scratch.

## Ruled Out
- **"+1 minute" as a deliberate dedup route**: The dispatch hypothesised that the runtime routed to "new file +1 minute" without operator intervention. Evidence shows the runtime has no dedup routing at all — the +1 minute pattern is an emergent artefact of `sessionData.TIME` being freshly computed (`formatTimestamp(now, 'time-short')`) on each save invocation. There is no code that adds a minute to an existing timestamp; the test loop's 18:32–18:49 progression is elapsed wall-clock time, not a collision-resolution strategy. — `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:1554-1555`, `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1252`
- **`ensureUniqueMemoryFilename` as the spawn mechanism**: `-1/-2/-N` suffix fallback only fires when the filename-exact existsSync check hits (workflow.ts:1630). In the 14-file test loop, filenames already differed by minute — existsSync returned false — this code path never triggered. The observed duplicates have distinct `HH-MM` timestamps, not `-N` suffixes; visual inspection of the observed filename list would confirm this. — `.opencode/skill/system-spec-kit/scripts/utils/slug-utils.ts:198-251`
- **MCP search-side session dedup as the referenced contract**: `mcp_server/lib/session/session-manager.ts:184-185`, `mcp_server/handlers/memory-search.ts:573-1247`, `references/memory/memory_system.md:508-580` all describe the SEARCH-side session deduplication (preventing duplicate memories from being returned in the same session via `enableDedup` + `sessionId` → content-hash match → skip sending). This is a completely distinct feature from the SAVE-side dedup contract in save.md:685-702. The naming collision explains why a cursory grep finds "session deduplication" hits; adjudication separates them clearly. Rolling the search-side dedup into this finding would be a category error.
- **Historical changelog retirement of save-side dedup**: Grep across `.claude/changelog/` and `.opencode/changelog/` for "session dedup" + "save" + "retired" returned only v1.2.0 entries referring to the search-side feature. No changelog entry retires or marks deferred the save-side contract. The contract remains present-tense in save.md.
