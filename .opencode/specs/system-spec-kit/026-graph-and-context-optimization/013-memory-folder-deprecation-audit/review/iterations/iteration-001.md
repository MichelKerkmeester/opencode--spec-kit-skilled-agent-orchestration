# Iteration 1: Correctness — Memory write/read paths in scripts/core runtime

## Focus
Dimension: **correctness**. Audit the six scope files (`workflow.ts`, `memory-indexer.ts`, `file-writer.ts`, `subfolder-utils.ts`, `topic-extractor.ts`, `memory-metadata.ts`) plus the immediate helper `spec-folder/directory-setup.ts` (entered transitively from `workflow.ts:790`) to verify whether the runtime still writes and reads `[spec]/memory/*.md` files. The v3.4.0.0 changelog and `command/memory/save.md:551` both claim the `memory/*.md` corpus is retired and rejected.

## Findings

### P0
- **F001**: Runtime creates `[spec]/memory/` directory unconditionally — `.opencode/skill/system-spec-kit/scripts/spec-folder/directory-setup.ts:85-104` — `setupContextDirectory` constructs `contextDir = path.join(sanitizedPath, 'memory')` (line 85) and calls `fs.mkdir(contextDir, { recursive: true })` (line 88) during every `runWorkflow` invocation. `workflow.ts:790` imports and calls this helper, logging `"Created: ${contextDir}"`. This is the root write path that makes every downstream memory/*.md finding real, and it fires on every save invoked via `generate-context.js` even though documentation states the surface is retired.
- **F002**: `workflow.ts` Step 9 writes memory/*.md via `writeFilesAtomically(contextDir, files)` — `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1861` — `writtenFiles: string[] = await writeFilesAtomically(contextDir, files)`. `files[ctxFilename]` contains the rendered memory markdown, and `contextDir` is the `[spec]/memory/` path returned by `setupContextDirectory`. The batch also carries `files['metadata.json']` (line 1673) into the same directory. This is the primary P0-WRITE — the code path that produced the 14 duplicate-new memory files documented in the session context.
- **F003**: `file-writer.ts::writeFilesAtomically` persists each entry into `contextDir` — `.opencode/skill/system-spec-kit/scripts/core/file-writer.ts:132-254` — Lines 137, 143, 154, 187 resolve `contextDir`, check duplicates inside it, `path.join(contextDir, filename)`, and `fs.rename(tempPath, filePath)` to commit. The function is generic but its only runtime caller (`workflow.ts:1861`) passes the memory directory, so this is the atomic-commit leg of the P0-WRITE path. Equally critical: lines 229 and 231 re-enter the memory directory during rollback (`fs.rename(prev.backupPath, path.join(contextDir, prev.filename))` and `fs.unlink(path.join(contextDir, prev.filename))`), so even the failure path writes into `memory/`.
- **F004**: `memory-indexer.ts::indexMemory` indexes the written memory/*.md file into the vector DB — `.opencode/skill/system-spec-kit/scripts/core/memory-indexer.ts:179-189` — `vectorIndex.indexMemory({ specFolder, filePath: path.join(contextDir, contextFilename), ... })`. `workflow.ts:2100-2108` invokes this helper with the freshly-written file. This is the downstream P0-WRITE effect: the retired surface is both persisted to disk AND registered as memory #N in the semantic index, meaning every subsequent search / continuity path can surface the file. This alone would violate the "retired and rejected" claim even if the filesystem artifact were deleted afterwards.
- **F005**: `workflow.ts::ensureUniqueMemoryFilename` and duplicate-preflight read `[spec]/memory/*.md` — `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1629-1646, 1825-1848` — Line 1629 opens `path.join(contextDir, rawCtxFilename)`; line 1636 calls `ensureUniqueMemoryFilename(contextDir, ...)`; lines 1825-1839 `readdirSync(contextDir).filter(f.endsWith('.md'))` and `readFileSync(path.join(contextDir, existingFile))` to hash-compare prior memory files. This is P0-READ: the runtime expects `[spec]/memory/*.md` siblings to exist and actively inspects them.
- **F006**: `file-writer.ts::checkForDuplicateContent` globs and reads existing `memory/*.md` siblings — `.opencode/skill/system-spec-kit/scripts/core/file-writer.ts:99-129` — Lines 107-108 `fs.readdir(contextDir)` filtered to `*.md`, line 116 `fs.readFile(path.join(contextDir, existing))` — the helper is invoked for every write (line 143) and hashes sibling memory files to decide whether to skip. This formalises the expectation that the retired surface is live and queryable.
- **F007**: `memory-metadata.ts::autoPopulateCausalLinks` reads `[spec]/memory/*.md` to auto-derive causal-link context — `.opencode/skill/system-spec-kit/scripts/core/memory-metadata.ts:322-331` — Constructs `memoryDir = path.join(specFolderPath, 'memory')`, `readdirSync(memoryDir).filter(entry.endsWith('.md'))`, then `parseExistingMemoryHeader(path.join(memoryDir, entry))` to hydrate `CausalLinksContext` from prior memory files. This is P0-READ and contradicts the retirement claim at a different layer: not only does the runtime write the files, it relies on them for causal-link enrichment.
