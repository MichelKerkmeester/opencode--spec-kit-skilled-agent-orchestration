# Iteration 4 (r2): Security — 4-vector re-audit of post-Path A surface

> r2 session. Overwrites r1's iteration-004 per explicit dispatch direction.

## Focus

Dimension: security. The retired `[spec]/memory/*.md` write path was r1's primary security surface (r1 iter-6 audited path-traversal / content-injection / TOCTOU / containment). Verify the post-Path A surface (canonical-doc save via content-router + graph-metadata refresh at `workflow.ts:1632-1647` + `ensureSpecFolderExists` at `directory-setup.ts:22-85`) inherits at least the same guards. Four vectors: (1) path-traversal of `specFolder` input, (2) content-injection in derived metadata, (3) TOCTOU between validation and write, (4) containment (symlink-ancestor escape).

## Findings

### P1

- **F031**: `writeGraphMetadataFile` has no realpath-based containment guard — regression vs retired `file-writer.ts` — `.opencode/skill/system-spec-kit/mcp_server/lib/graph/graph-metadata-parser.ts:949-956` — `writeGraphMetadataFile(filePath, metadata)` does `fs.mkdirSync(path.dirname(filePath), { recursive: true })` → `fs.writeFileSync(tempPath, ...)` → `fs.renameSync(tempPath, filePath)`. The only upstream guard is `canClassifyAsGraphMetadataPath(filePath)` at `graph-metadata-parser.ts:969`, which delegates to `isSpecsScopedPath(filePath)` at `spec-doc-paths.ts:90`. `isSpecsScopedPath` is a STRING-PATTERN check (normalized path must contain `/specs/`), NOT a realpath-based containment check. This is a defense-in-depth regression vs the retired `file-writer.ts` which (per r1 F022) used `verifyResolvedWriteTarget` with `fs.realpathSync(parentDir)` containment. Exploit sketch: if an attacker can (a) supply a spec-folder-shaped path that normalizes to contain `/specs/` but (b) has a symlinked ancestor pointing outside the repo, `mkdirSync({recursive:true})` will follow the symlink and `renameSync` will write `graph-metadata.json` into the symlink target. Mitigations present: (a) `ensureSpecFolderExists` at `directory-setup.ts:22-85` validates the spec folder exists and passes through `sanitizePath(specFolder, getSpecsDirectories())` — but does NOT do `realpathSync`, so a valid spec folder dir containing a symlinked ancestor escapes detection; (b) `SPEC_FOLDER_PATTERN.test(leafFolder)` at `directory-setup.ts:36` whitelists `NNN-name` shape on the LEAF ONLY (not ancestors). Practical exploitability: LOW — requires write access to create a symlink inside specs/, which already requires commit access. Gate-relevant because the retired surface WAS hardened against this exact class of attack.

### P2

- **F032**: `refreshGraphMetadataForSpecFolder` lacks a reentrancy / mutex guard — `.opencode/skill/system-spec-kit/mcp_server/lib/graph/graph-metadata-parser.ts:965-984` + `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1643` — every save triggers a read-modify-write of `graph-metadata.json` via `loadGraphMetadata` → `deriveGraphMetadata` (which walks the full spec folder via `collectPacketDocs`) → `mergeGraphMetadata` → `writeGraphMetadataFile`. No mutex, no backoff. If two `/memory:save` calls fire concurrently against the same spec folder (e.g., stop-hook autosave + user-initiated save), both read the same `existing` snapshot, both compute `refreshed`, and the later `renameSync` wins while the earlier `renameSync` output is clobbered. No data loss because both outputs are deterministic derivations of disk state, but if `options.statusOverride` or `options.now` differ between the two callers, the later call's value wins unpredictably. The MCP handler path DOES use `withSpecFolderLock` at `memory-save.ts:115`, but the script-side `workflow.ts` invokes `refreshGraphMetadataForSpecFolder` at :1643 directly without going through the lock. P2-RACE-DRIFT — no exploit, only surprising-under-concurrency behavior.

- **F033**: `log` path emits retirement-transition internals to stderr on every save — `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1554,1651-1652,1752` — emits `"   Skipping legacy [spec]/memory/*.md writes"` (:1554), `"Legacy memory artifact write skipped."` (:1651), and `"   Skipping retired legacy memory indexing"` (:1752) on every save. These are not security issues per se, but they (a) leak retirement-transition internals to operators who do not need to know (information hygiene), (b) encode brittle marker strings that automated post-save parsers will latch onto, and (c) teach operators a concept (`memory/*.md`) that the changelog claims is retired. P2-NOISE. Retention may be intentional for cutover-period debugging, but the strings should be gated behind a debug-log level rather than printed to stderr unconditionally.

## Ruled Out

- Path-traversal via `specFolder` argument: `workflow.ts:1643` calls `path.resolve(specFolder)` (normalizes), `refreshGraphMetadataForSpecFolder` calls `canClassifyAsGraphMetadataPath` (requires `/specs/` substring AND no excluded segments), upstream `ensureSpecFolderExists` runs `sanitizePath(specFolder, getSpecsDirectories())` (rejects paths escaping allowed bases). Three layers of defense; direct traversal not exploitable.
- Content-injection via graph-metadata derivation: `deriveGraphMetadata` at `graph-metadata-parser.ts:870-909` runs all derived fields through `graphMetadataSchema.parse(...)` (Zod schema validation on every write). Trigger phrases / key files / entities / causal summary are string-truncated via `.slice(0, 12)` or similar. `serializeGraphMetadata` uses `JSON.stringify(metadata, null, 2)`; schema validation prevents `toJSON`-hook bypass. Injection into JSON output not exploitable.
- TOCTOU between `ensureSpecFolderExists` and write: `directory-setup.ts:46-83` does `fs.stat` then throws on ENOENT, returns `sanitizedPath`. `writeGraphMetadataFile` subsequently calls `fs.mkdirSync(path.dirname(filePath), { recursive: true })` — if the spec folder was deleted between stat and mkdir, mkdirSync recreates it. Benign recreate, not a TOCTOU exploit. r1-F022's original TOCTOU concern (existsSync-based dedup vs atomic-commit) is CLOSED because the dedup path itself is deleted.
- Symlink-traversal INSIDE `[spec]/`: F031 addresses the symlinked-ANCESTOR case (most severe). For symlinks INSIDE the spec folder (e.g., `[spec]/graph-metadata.json` being a symlink to `/etc/passwd`), `renameSync` replaces the symlink with the new file atomically — symlink-as-file-target is replaced, not followed. Not exploitable on Linux/macOS.
- fd / file-handle leak in write path: `writeFileSync` + `renameSync` both use short-lived fds; Node's `fs.writeFileSync` closes on completion or exception (verified in r1 iter-6 audit of same-pattern code). Not exploitable.
- Content-injection via `specFolder` name into logs: `workflow.ts:1652` interpolates `validatedSpecFolderPath` into a log message. Since `sanitizePath` + `SPEC_FOLDER_PATTERN` guarantee the leaf matches `/^\d{1,3}-[a-z0-9-]+$/`, there is no log-injection vector (no newline, no control chars). NON-FINDING.
- Sensitive-content leakage into `graph-metadata.json`: derived fields come only from canonical spec docs (`spec.md`, `plan.md`, `tasks.md`, `checklist.md`, `decision-record.md`, `implementation-summary.md`, `research/research.md`, `handover.md` per `CANONICAL_PACKET_DOCS` at `graph-metadata-parser.ts:29-39`). These are operator-authored docs already in the repo; no new sensitivity exposure.

## Dead Ends

- Searching for a user-controllable path that bypasses `canClassifyAsGraphMetadataPath` — the string-pattern check is loose (a path like `/tmp/evil/specs/001-fake` would pass the classification gate). BUT upstream `ensureSpecFolderExists` via `sanitizePath(specFolder, getSpecsDirectories())` rejects paths outside known specs directories. Bypass requires compromising BOTH guards, which is not a realistic single-vector attack.
- Searching for a missing auth layer on `refreshGraphMetadataForSpecFolder` — it is not exposed via MCP directly; called only from `workflow.ts:1643` (script path, operator-initiated) and from MCP handlers that have already gone through `validateFilePath` / `withSpecFolderLock`. No missing auth vector.
