## Proposed PR-7 operation (from APPENDIX B + iter 14 + iter 20)
At save time, PR-7 would add a helper immediately before `buildCausalLinksContext()` that inspects the current spec folder's `memory/` directory, enumerates sibling markdown files, reads enough of each file to recover its frontmatter title, applies the continuation gate frozen by iteration 14 (`extended` / `continuation`-style signals, explicitly excluding noisy `phase N` / `vN` families), selects only the single immediate predecessor when exactly one qualifying older sibling is unambiguous, and injects `causal_links.supersedes` into the collected payload before template render. If the folder is ambiguous, or no qualified predecessor exists, the helper must skip without writing lineage. - Folder size N = 100: same shape, doubled siblings. Measured warm-cache cost is ~1.39 ms median / ~1.77 ms p95 full-read, or ~1.00 ms with a 2 KB head-read helper. [CITATION: NONE — local microbenchmark]
- Folder size N = 500: 1 directory read + 500 sibling inspections. Measured warm-cache cost is ~7.86 ms median / ~8.38 ms p95 full-read, or ~6.01 ms with a 2 KB head-read helper. Even with a conservative 5x cold-cache multiplier, this stays under ~40 ms. [CITATION: NONE — local microbenchmark]

## Big-O analysis
- Discovery: `O(N)` to enumerate sibling files and inspect each title once; `O(N log N)` only if the implementation does a one-time sort to establish recency.
- Disambiguation: `O(K)` in the number of qualifying continuation candidates (`K <= N`) if the helper gathers candidates and then checks whether exactly one survives; effectively `O(1)` extra state if it tracks "best candidate so far" plus an ambiguity counter during the same pass.
- Total: `O(N)` expected, `O(N log N)` if a single sort is used, still not `O(N²)`.
- Is the implementation `O(N)`? `O(N log N)`? `O(N²)`? Justify. The proposed design from iterations 6/14/20 is linear if it performs one folder walk and one candidate-selection pass. It becomes `O(N log N)` only if it sorts siblings once, which is still fine at these folder sizes. I found no inherent reason for `O(N²)` unless the implementation introduces pairwise candidate comparisons, rescans the directory inside the per-file loop, or invokes a heavier lookup (`memory_search`, `parseMemoryFile`) once per sibling and then compares those results against each other. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-006.md:19-30] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-014.md:35-39] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-020.md:120-134]

## Cliffs / anti-patterns
- Cliff 1: reusing a full-file parser for every sibling (`readFileSync` / `parseMemoryFile`) instead of a header-only frontmatter reader. Trigger condition: large folders plus cold-cache files. Mitigation: add a tiny `readFrontmatterHead(file, 2048)` helper; the measured 500-file walk drops from ~7.86 ms to ~6.01 ms (~24% cheaper). [SOURCE: .opencode/skill/system-spec-kit/scripts/extractors/spec-folder-extractor.ts:50-82] [SOURCE: .opencode/skill/system-spec-kit/scripts/core/post-save-review.ts:224-233] [SOURCE: .opencode/skill/system-spec-kit/mcp_server/lib/parsing/memory-parser.ts:187-233] [CITATION: NONE — local microbenchmark]
- Cliff 2: repeated rescans or pairwise ambiguity resolution. Trigger condition: implementing "find predecessor" by rescanning the same folder for each candidate or comparing every candidate against every other candidate. Mitigation: one pass, one candidate accumulator, one ambiguity counter. [SOURCE: .opencode/skill/system-spec-kit/scripts/core/workflow.ts:1305-1372] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-020.md:128-134]
- Cliff 3: paying the scan cost on saves that can never qualify. Trigger condition: running predecessor discovery even when the current save title does not carry any continuation signal. Mitigation: check the current memory title first and skip the sibling walk entirely for non-continuation saves. Iteration 14 found only 13/140 titles (9.29%) with any continuation hit, so this early exit would eliminate the new cost on ~90% of historical saves. [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-014.md:16-20] [SOURCE: .opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-014.md:35-39]

## Optimization proposals (if latency is concerning)
- O1: Skip predecessor discovery when caller already provides `causal_links.supersedes`. Cost reduction: removes 100% of the new scan on explicitly linked saves. [SOURCE: .opencode/skill/system-spec-kit/scripts/core/memory-metadata.ts:227-236]
- O2: Skip the folder walk unless the current memory title itself matches the continuation gate. Cost reduction: based on iteration 14's corpus, likely avoids the new work on ~90% of historical saves.
