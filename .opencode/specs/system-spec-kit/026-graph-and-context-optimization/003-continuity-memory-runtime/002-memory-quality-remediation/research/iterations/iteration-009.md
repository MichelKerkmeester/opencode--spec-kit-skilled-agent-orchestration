## Skeptical pass on D2 fix
MODIFY.

Adding a raw `keyDecisions` reader in `extractDecisions()` is still the right direction, but the iteration-8 recommendation to broadly disable lexical fallback for JSON-mode inputs is too aggressive. The normalizer already promotes `keyDecisions` / `key_decisions` into `_manualDecisions` on both fast and slow paths, so the safer hardening is precedence-based: if raw JSON decisions are present and normalized carriers are missing, read them directly before lexical mining. [CITATION: `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:566-584`, `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:676-752`, `.opencode/skill/system-spec-kit/scripts/types/session-types.ts:137-148`]

Blanket `_source === 'file'` lexical suppression could regress legitimate JSON saves that intentionally omit `keyDecisions` and currently rely on lexical extraction as a degraded fallback. Capture-mode behavior should stay untouched, and JSON-mode should only bypass lexical fallback when authoritative raw decision arrays are actually present. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:367-388`, `.opencode/skill/system-spec-kit/scripts/extractors/conversation-extractor.ts:58-107`]

Migration story: no feature flag is needed if the fix is scoped to precedence hardening. Well-formed payloads improve immediately; malformed or decision-less JSON payloads keep existing degraded behavior instead of silently losing all decisions. A feature flag would only be justified if the team insists on fully disabling lexical fallback for all JSON-mode saves. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:367-388`]

## Skeptical pass on D5 fix
MODIFY.

The current D5 proposal is directionally right but too broad. `buildCausalLinksContext()` is only a pass-through reader, so any new auto-discovery logic has to define the semantics itself. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/memory-metadata.ts:227-236`]

For 3+ memory files in one folder, `supersedes` should point only to the immediate predecessor, not every earlier file. Linking all prior runs would duplicate transitive history and make the causal graph noisier than necessary. Also, folder recency alone is too weak: the current corpus mostly shows paired continuation runs, but the heuristic was not proven against mixed-topic histories or more crowded folders. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/002-codesight/memory/06-04-26_16-34__5-iteration-deep-research-session-investigating.md:2-3`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/002-codesight/memory/06-04-26_17-58__continuation-deep-research-run-for-002-codesight.md:2-3`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/003-contextador/memory/06-04-26_12-25__completed-deep-research-phase-003-contextador-via.md:2-3`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/003-contextador/memory/06-04-26_13-49__extended-deep-research-phase-003-contextador-from.md:2-3`]

Safer recommendation: only auto-link when explicit lineage is absent and there is one unambiguous predecessor with the same folder, an earlier timestamp, and a strong continuation signal in title/description/payload such as `extended`, `continuation`, or an iteration carry-forward. If multiple candidates tie or the continuity signal is weak, skip auto-linking rather than guessing. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/memory-metadata.ts:227-236`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/001-research-graph-context-systems/002-codesight/memory/06-04-26_17-58__continuation-deep-research-run-for-002-codesight.md:2-3`]
