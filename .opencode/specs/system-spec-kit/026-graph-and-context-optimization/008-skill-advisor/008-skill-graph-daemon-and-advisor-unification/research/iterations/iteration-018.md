## Analysis
`memory_causal_link` can technically accept skill-shaped string identifiers because `causal_edges.source_id` and `target_id` are text and `insertEdge` does not enforce memory foreign keys. That is not enough to make direct skill insertion safe. The public tool contract says it links memories, orphan cleanup validates endpoints against `memory_index`, and graph search/boost paths seed from `memory_fts` and memory row IDs. Directly placing skill IDs in `causal_edges` would therefore create a mixed graph whose maintenance, cleanup, FTS search, and neighbor injection paths do not consistently understand skill nodes.

The right adoption path is to keep skills in `skill_edges` and add a causal-style adapter over that graph. The skill graph already has first-class skill IDs, foreign keys, self-loop prevention, relation indexes, bounded path/subgraph traversal, hub/orphan queries, and relationship context. Its edge vocabulary maps naturally to advisor behavior: `depends_on` and `prerequisite_for` express dependency/enablement, `enhances` expresses support/extension, `siblings` expresses family affinity, and `conflicts_with` expresses negative routing evidence. That is a cleaner model than forcing skill relations into the six memory-causal relation names.

Traversal cost looks acceptable for hook-time advisor use if it follows the existing guardrails. Memory causal search already uses capped seed fractions, max-hop traversal, sparse-first 1-hop mode, fan-effect normalization, typed-degree caps, and per-DB degree caching. Skill graph traversal is simpler: indexed source/target/type lookups plus BFS with a visited set and max depth. The current advisor is even more conservative because graph boosts only propagate to skills that already have prompt evidence, which prevents topology alone from causing surprising recommendations.

So C3 should adopt causal graph behavior now, but as an advisor-owned skill graph traversal lane rather than as `memory_causal_link` reuse. Implementation should expose or reuse `skill_graph_query`/`skill_edges` for causal-style features, map skill edge types to scorer priors in C4, and reserve direct `memory_causal_link` for memory lineage. This preserves the C2 storage boundary while still bringing memory-search lessons into skill routing.
