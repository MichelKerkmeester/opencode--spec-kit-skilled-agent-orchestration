## Analysis
| Area | Standalone `.opencode/skill/skill-advisor/` | MCP-side `.opencode/skill/system-spec-kit/mcp_server/` |
| --- | --- | --- |
| Runtime brain | Python `skill_advisor.py` owns deterministic scoring, Gate 2 JSON output, CLI modes, graph loading, validation, semantic flags, and health reporting. | TypeScript `buildSkillAdvisorBrief()` owns hook-time policy, freshness, cache, subprocess invocation, shared payload, diagnostics, and fail-open behavior. |
| Graph artifacts | Python compiler still validates per-skill metadata and exports legacy `scripts/skill-graph.json`; Python runtime reads SQLite first and JSON second. | TypeScript `lib/skill-graph` owns live SQLite schema, indexing, queries, status, validation, and MCP tool dispatch. |
| Database | Python code references the MCP-owned SQLite path and treats it as preferred runtime input. | Database file is `skill-graph.sqlite` under the shared MCP database directory, with WAL mode initialized by the DB layer. |
| Tests | Python has one adjacent direct test file plus regression/bench harnesses and fixtures. | Vitest has broad hook, runtime parity, subprocess, freshness, prompt cache/policy, renderer, privacy, observability, corpus parity, plugin, shared-payload, tool schema, and skill graph dispatcher coverage. |
| Docs/playbooks | Standalone package contains README, setup guide, feature catalog, manual testing playbook, fixtures, and reports. | MCP package contains TypeScript modules, handlers, tools, tests, generated `dist/`, and database artifacts. |

The current split is not merely "Python implementation, TypeScript wrapper." Python still owns the advisor scoring contract, cache-backed skill discovery, graph-derived scoring overlays, CLI health/validation, legacy JSON compilation, regression harness, and benchmark harness. TypeScript owns the modern runtime integration layer around that Python brain: whether the advisor should fire, whether source state is live/stale/absent, how prompt-time cache keys are derived, how recommendations are rendered safely, how runtime adapters normalize output, and how hook metrics/fail-open diagnostics are represented.

The graph split is especially important for D2-D8. The live graph database has already moved into the system-spec-kit MCP server, with `skill-graph.sqlite` stored beside other MCP databases and exposed through four `skill_graph_*` tools. However, Python still has to understand that SQLite schema to load families, edges, signals, conflicts, metadata, and topology warnings, and it still preserves a JSON fallback plus auto-compile path. That means consolidation cannot delete the standalone package outright without first replacing Python scoring, Python health semantics, JSON fallback behavior, and the direct CLI contract that AGENTS.md Gate 2 still names as fallback.

The inventory points to an adopt-now consolidation baseline: keep the observed boundaries as the migration source of truth, then move in layers. First, D2 should define a target layout that keeps the live SQLite graph and MCP tools in system-spec-kit. Second, D4 should decide whether to port Python scoring into TS or keep a compatibility subprocess during transition. Third, D7 must preserve Python CLI compatibility until hook users, scripts, plugin bridge, docs, and tests have a stable replacement path.
