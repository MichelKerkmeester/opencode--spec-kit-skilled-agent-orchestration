## Analysis
The advisor hook's shipped performance story is built around exact prompt caching, not live semantic retrieval. The cache key includes the prompt, source signature, runtime, token cap, and thresholds, and the cache-hit branch avoids subprocess work entirely while restamping metrics and shared-payload provenance (`prompt-cache.ts:61`-`prompt-cache.ts:73`; `skill-advisor-brief.ts:383`-`skill-advisor-brief.ts:407`). Phase 020's recorded cache-hit p95 is `0.016 ms`, which is orders of magnitude below the `<=50 ms` release gate (`implementation-summary.md:99`-`implementation-summary.md:116`; `skill-advisor-hook.md:396`-`skill-advisor-hook.md:419`).

Memory MCP search is a different latency class. On cache miss, `memory_search` may wait for embedding readiness, then enters a four-stage pipeline with hybrid/vector search, optional query expansion, fusion, reranking, MMR, chunk reassembly, and filtering (`memory-search.ts:846`-`memory-search.ts:938`; `pipeline/README.md:34`-`pipeline/README.md:51`). The code path can generate one embedding for normal hybrid search, more embeddings for facets or variants in deep mode, and another embedding channel for R12 expansion; vector and MMR paths also touch `vec_memories` through SQLite (`stage1-candidate-gen.ts:676`-`stage1-candidate-gen.ts:953`; `vector-index-queries.ts:168`-`vector-index-queries.ts:280`; `hybrid-search.ts:1384`-`hybrid-search.ts:1472`). The only local memory-search benchmark found is a mocked handler benchmark with a `300 ms` p95 gate, and enhanced vector search warns only above `600 ms`, so current repo evidence does not support putting memory embedding search on every prompt-blocking advisor invocation (`gate-d-benchmark-memory-search.vitest.ts:170`-`gate-d-benchmark-memory-search.vitest.ts:196`; `vector-index-aliases.ts:405`-`vector-index-aliases.ts:430`).

The affordable adopt-now path is narrower: keep exact HMAC caching first, keep deterministic lexical/trigger and skill-graph SQL work in the hot path, and treat embedding lookup as a shadow or prototype channel. That still advances C4/C5 because skill edges needed for ambiguity are cheap indexed SQL lookups, not embedding calls (`skill-graph-db.ts:132`-`skill-graph-db.ts:158`; `skill-graph-queries.ts:144`-`skill-graph-queries.ts:157`). It also lines up with existing trigger guidance: `memory_match_triggers` is explicitly the sub-50 ms no-embedding proactive path, while semantic search is the heavier fallback (`trigger_config.md:82`-`trigger_config.md:100`).

Benchmark plan: add an advisor-specific shadow lane before promotion. Measure p50/p95/p99 for four paths over the 019/004 corpus and C7 holdout: exact cache hit, uncached lexical+skill-graph SQL, uncached embedding/hybrid shadow, and cache-miss-with-source-signature-churn. Promotion criteria should be cache-hit p95 `<=50 ms`, uncached lexical+graph p95 `<=60 ms`, embedding shadow p95 not above a separately approved prompt-time ceiling, zero top-1 regression, and evidence that the exact HMAC cache hit rate remains at least the Phase 020 `60%` replay target.

## Verdict
- **Call:** prototype_later
- **Confidence:** high
- **Rationale:** Do not adopt prompt-time embedding lookup on every advisor prompt now; repo evidence shows the advisor cache lane is sub-millisecond, while memory embedding/hybrid search is a multi-stage pipeline with much looser 300-600 ms benchmarks and warning thresholds.
