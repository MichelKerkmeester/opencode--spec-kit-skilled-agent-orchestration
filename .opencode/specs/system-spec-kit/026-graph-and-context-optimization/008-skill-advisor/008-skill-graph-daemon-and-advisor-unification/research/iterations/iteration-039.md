## Analysis
The measurement plan should be adopted as a packet-wide release gate, but not as a new runtime feature. The existing static harness already measures the core accuracy baseline against the 200-row corpus, emits per-skill/resource summaries, and preserves the caveat that it cannot prove live AI reads (`smart-router-measurement.ts:5`-`smart-router-measurement.ts:7`; `smart-router-measurement-report.md:50`-`smart-router-measurement-report.md:55`). C7 and C8 provide the target layer on top: freeze a deterministic 20% holdout, require 70% full-corpus and holdout top-1 for deterministic improvements, reduce `UNKNOWN` from 37 to <=10, avoid gold-`none` false-fire increases, and track top-k/ranking metrics before semantic or learned channels leave shadow mode (`iteration-022.md:41`-`iteration-022.md:59`; `iteration-023.md:35`-`iteration-023.md:53`).

The plan needs four harness lanes. First, extend `smart-router-measurement.ts` for Phase 027 slices: full corpus, frozen holdout, per-skill precision/recall/F1, gold `none` false fires, `UNKNOWN`, explicit-skill prompts, ambiguity/close-top-two prompts, derived-lane ablations, and top-k/MRR output. Second, keep the Python `skill_advisor_regression.py` suite as the compatibility and safety lane until the TypeScript scorer reaches parity, because it already gates top-1, P0, all-cases pass, and command-bridge false positives (`skill_advisor_regression.py:223`-`skill_advisor_regression.py:230`; `regression-report.json:25`-`regression-report.json:31`). Third, keep `skill_advisor_bench.py` and the C6 advisor-specific lanes for latency: cache-hit p95 <=50ms, uncached lexical+graph p95 <=60ms, cold subprocess p95 <=60ms while the shim remains, warm p95 <=20ms, and any embedding/semantic lane shadow-only until an explicit prompt-time ceiling is accepted (`skill_advisor_bench.py:227`-`skill_advisor_bench.py:260`; `iteration-021.md:46`-`iteration-021.md:57`).

The fourth lane is operational and manual verification for the daemon, MCP, hook, and plugin surfaces. A6/A8 require watcher and recovery measurements, so `027/001` should record watcher root count, descriptor delta, idle CPU, RSS, debounce-to-index time, stale/unavailable fail-open behavior, `SQLITE_BUSY` retry/lease behavior, and rebuild-from-source recovery (`iteration-006.md:35`-`iteration-006.md:47`; `iteration-008.md:46`-`iteration-008.md:58`). D3 defers public batch/benchmark MCP tools, so these can remain scripts/tests/manual scenarios during Phase 027 while `advisor_status` and `advisor_validate` expose readiness, graph source, validation, and recovery state (`iteration-026.md:44`-`iteration-026.md:90`). The manual playbook already has routing accuracy, regression safety, SQLite graph, hook, plugin, and measurement-run sections that can become acceptance rows for each sub-packet (`manual_testing_playbook.md:190`-`manual_testing_playbook.md:300`).
