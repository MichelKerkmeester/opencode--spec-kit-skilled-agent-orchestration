## Detailed proposals

### D1 — Truncated overview
**Root cause:** `collectSessionData()` owns the `SUMMARY` value used by the `OVERVIEW` section and currently hard-clamps `sessionSummary` with `substring(0, 500)`, while the template just renders `{{SUMMARY}}` directly. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:875-881`, `.opencode/skill/system-spec-kit/templates/context_template.md:330-352`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-002.md:18-35`]

**Proposed fix:** Replace the raw character clamp with a shared truncation helper that trims on a word or sentence boundary and optionally adds `"..."` or `" [truncated]"` when the input exceeds the soft target. The existing observation-narrative path in `buildSessionSummaryObservation()` is the best in-repo reference because it already trims at a word boundary and adds ellipsis instead of cutting a token mid-word. [CITATION: `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:274-283`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-002.md:9-16`]

**Code change:** Introduce a small helper used by both `collect-session-data.ts` and `input-normalizer.ts`, or at minimum replace `substring(0, 500)` at `collect-session-data.ts:876-877` with the same trim pattern already used at `input-normalizer.ts:281-282`. Keep the cap soft rather than removing it entirely unless downstream size assumptions are audited. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/collect-session-data.ts:875-881`, `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:279-283`]

**Risk:** A looser summary cap could increase duplication between `OVERVIEW` and `DETAILED CHANGES`, or grow markdown size enough to affect downstream indexing and template readability. [CITATION: `.opencode/skill/system-spec-kit/scripts/utils/input-normalizer.ts:279-280`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-002.md:16-16`]

**Verification:** Add unit tests for `collectSessionData()` with 450-character, 520-character, and 900-character `sessionSummary` inputs. Then do a template-only replay asserting that the rendered `OVERVIEW` ends cleanly and that the truncation marker appears only when expected. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-002.md:28-43`]

### D2 — Generic decision placeholders
**Root cause:** `extractDecisions()` has no direct reader for raw `keyDecisions` or `decisions`; it trusts `_manualDecisions` and decision-typed observations, then falls through to lexical cue mining when both are empty. That lexical path is the only place that manufactures `observation decision N` and `user decision N`. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:137-175`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`, `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:367-388`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-003.md:17-40`]

**Proposed fix:** Add a new high-priority `readRawJsonDecisions()` step inside `extractDecisions()` that consumes raw `keyDecisions` or `decisions` before lexical fallback. For `_source === 'file'`, treat lexical mining as recovery-only or disable it entirely unless the payload looks like a capture transcript rather than a structured save. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`, `.opencode/skill/system-spec-kit/scripts/extractors/conversation-extractor.ts:87-107`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-003.md:12-20`]

**Code change:** Widen the extractor's accepted subset so it can inspect raw `keyDecisions`, raw `decisions`, and `_source`. Then gate the lexical branch behind both "no normalized decisions" and "not JSON-mode authoritative input." That keeps transcript-mode mining available without letting structured JSON silently degrade into placeholders. [CITATION: `.opencode/skill/system-spec-kit/scripts/extractors/decision-extractor.ts:182-185`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-003.md:17-30`]

**Risk:** D2 has the highest behavior-change risk. Some existing JSON payloads may currently depend on lexical fallback rescuing incomplete `keyDecisions`, so a strict disable in JSON mode could reduce decision counts unless the raw-field reader is implemented first. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-003.md:22-27`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-003.md:39-40`]

**Verification:** Build three fixtures: valid `keyDecisions`, missing `_manualDecisions` but valid raw `keyDecisions`, and transcript-style observations with decision cue sentences. Assert only the transcript fixture can emit lexical decisions; the JSON fixtures must preserve authored titles. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-003.md:22-27`] 

### D3 — Garbage trigger phrases
**Root cause:** D3 is actually two sibling defects. Frontmatter `trigger_phrases` are polluted because `workflow.ts` appends `folderTokens` after `filterTriggerPhrases()` runs, while `Key Topics` produces fragment bigrams because topic extraction counts n-grams after aggressive stopword filtering collapses the token stream. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1270-1295`, `.opencode/skill/system-spec-kit/scripts/lib/semantic-signal-extractor.ts:260-284`, `.opencode/skill/system-spec-kit/shared/trigger-extractor.ts:116-149`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-004.md:20-33`]

**Proposed fix:** For D3a, stop appending folder tokens directly or rerun them through the same quality filter plus a slash-stopword guard. For D3b, change bigram generation so adjacency is preserved from the original source text, either by counting bigrams before stopword collapse or by validating each candidate against the source corpus before keeping it. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1270-1295`, `.opencode/skill/system-spec-kit/scripts/lib/semantic-signal-extractor.ts:260-284`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-004.md:23-25`]

**Code change:** In `workflow.ts`, either delete the `folderTokens` append block or pass `folderTokens` through a new `filterFolderTriggerTokens()` helper that rejects slashes, path fragments, and generic English singletons such as `and`. In `semantic-signal-extractor.ts`, add a source-adjacency check for ranked n-grams before they are returned to `extractKeyTopics()`. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1274-1295`, `.opencode/skill/system-spec-kit/scripts/lib/semantic-signal-extractor.ts:291-351`, `.opencode/skill/system-spec-kit/shared/trigger-extractor.ts:176-183`]

**Risk:** Over-tightening the filters could remove useful short product names or repo names, and changing topic extraction could reduce recall for legitimate bigrams that currently rely on normalized tokens. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1277-1286`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-004.md:15-18`]

Preserve a control fixture where a real adjacent bigram such as `spec kit` still survives if it appears verbatim in the source. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-004.md:13-18`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-004.md:27-31`]

This is the lowest-risk path because it preserves the current authoritativeness of frontmatter while keeping the bottom block honest. [CITATION: `.opencode/skill/system-spec-kit/scripts/lib/frontmatter-migration.ts:1112-1183`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-005.md:22-27`]

Then extend `post-save-review.ts` to parse and compare the metadata block tier, not just payload-vs-frontmatter. ### D5 — Missing causal supersedes
**Root cause:** The script pipeline has no predecessor discovery. `buildCausalLinksContext()` only forwards `collectedData.causal_links`, and nothing in JSON mode scans prior memory files or asks the memory index for earlier runs in the same folder before rendering. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/memory-metadata.ts:227-236`, `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1305-1372`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-006.md:19-39`]

**Proposed fix:** Add a new pre-render `discoverPredecessors(specFolder, currentPayload)` helper in `workflow.ts` that inspects the local `memory/` directory, resolves the latest earlier memory for the same spec folder, extracts its `session_id`, and injects `causal_links.supersedes` when the current run looks like an extension or continuation. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:612-740`, `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1305-1372`, `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-006.md:22-25`] 

**Code change:** Call the helper before `buildCausalLinksContext(collectedData)`, merge the discovered predecessor only when the payload does not already provide explicit lineage, and keep the current MCP-side causal parser/edge builder unchanged as the downstream consumer. [CITATION: `.opencode/skill/system-spec-kit/scripts/core/workflow.ts:1305-1372`, `.opencode/skill/system-spec-kit/scripts/core/memory-metadata.ts:227-236`, `.opencode/skill/system-spec-kit/mcp_server/handlers/causal-links-processor.ts:341-405`]

**Risk:** Auto-linking the wrong predecessor is the main hazard, especially when a spec folder contains multiple unrelated research threads or reruns. False positives would create misleading causal history and harden incorrect lineage into the graph. [CITATION: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/003-memory-quality-issues/research/iterations/iteration-006.md:23-30`, `.opencode/skill/system-spec-kit/mcp_server/handlers/causal-links-processor.ts:341-405`]
