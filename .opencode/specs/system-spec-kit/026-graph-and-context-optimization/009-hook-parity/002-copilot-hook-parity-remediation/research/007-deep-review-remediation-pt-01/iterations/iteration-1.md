# Iteration 1: Documented Copilot CLI injection surfaces
## Focus
This iteration mapped the publicly documented Copilot CLI extension surfaces around hooks, custom agents, skills, MCP, and custom-instruction files. The goal was to separate "can run logic or expose tools" from "can inject startup or per-prompt payload text into model context."

## Actions Taken
1. Reviewed the main Copilot CLI docs plus the customization overview to identify all documented extension surfaces and how GitHub classifies them: custom instructions, hooks, skills, MCP servers, custom agents, and plugins. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-copilot-cli] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features]
2. Read the CLI-specific hooks docs and the hooks reference to determine which lifecycle events exist and whether hook output can modify prompt or session context. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/use-hooks] [SOURCE: https://docs.github.com/en/copilot/reference/hooks-configuration]
3. Read the custom-agent concept, creation guide, and programmatic reference to map `/agent` and `--agent`, including where agent profiles live and what they can configure. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/about-custom-agents] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/create-custom-agents-for-cli] [SOURCE: https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-programmatic-reference]
4. Read the skills, MCP, and custom-instructions docs to identify which surfaces are startup-loaded, task-scoped, or tool-only, and to inventory the documented config/env entry points. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-agent-skills] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-mcp-servers] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions] [SOURCE: https://docs.github.com/en/copilot/reference/custom-instructions-support] [SOURCE: https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference]

## Findings
- **Copilot CLI has a documented hook system, but the docs do not expose prompt/session-content injection through hook return values.** `sessionStart`, `sessionEnd`, `userPromptSubmitted`, `preToolUse`, `postToolUse`, and `errorOccurred` are supported; however, the hook reference explicitly says output is ignored for `sessionStart`, `userPromptSubmitted`, `postToolUse`, and `errorOccurred`, and only `preToolUse` can affect execution by returning a deny/ask/allow decision. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/use-hooks] [SOURCE: https://docs.github.com/en/copilot/reference/hooks-configuration]
- **`/agent` and `--agent` select custom subagent profiles, not a documented main-session startup-context injector.** Custom agents are `.agent.md` profiles stored in `.github/agents/` or `~/.copilot/agents/`, and they define prompt, tools, and optional MCP servers; the docs describe them as delegated subagents with their own context window rather than a way to prepend dynamic system-prompt payloads to the main session. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/about-custom-agents] [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/create-custom-agents-for-cli] [SOURCE: https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-programmatic-reference]
- **Skills are task-scoped "just-in-time" instructions, not session-global instruction injection.** GitHub describes skills as folders of instructions/scripts/resources that Copilot loads when relevant or when manually invoked via `/skills`; the comparison page contrasts them with custom instructions, which are the session-start persistent guidance mechanism. [SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-agent-skills] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features]
- **MCP is documented as a tool/data-source integration surface, not a push channel for unsolicited context.** The MCP docs say MCP servers "provide additional tools" and that Copilot can use those tools when relevant; the CLI config surfaces are `~/.copilot/mcp-config.json`, `/mcp add`, and `--additional-mcp-config`, with no documented mechanism for an MCP server to inject startup or per-prompt brief text into the model without being called as a tool. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-mcp-servers] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-copilot-cli]
- **Copilot CLI does support automatic startup-loaded instruction files, and these are the closest documented parity surface for a "startup context" payload.** The docs say custom instructions are automatically added to requests, support `.github/copilot-instructions.md`, `.github/instructions/**/*.instructions.md`, `AGENTS.md`, and `$HOME/.copilot/copilot-instructions.md`, and also allow extra directories via `COPILOT_CUSTOM_INSTRUCTIONS_DIRS`; this is static/file-based rather than dynamic hook-generated payload injection. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-custom-instructions] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/copilot-cli/comparing-cli-features] [SOURCE: https://docs.github.com/en/copilot/reference/custom-instructions-support]
- **The documented env/config surface is rich for discovery and file locations, but not for arbitrary prompt-payload injection.** The CLI reference documents env vars such as `COPILOT_CUSTOM_INSTRUCTIONS_DIRS`, `COPILOT_SKILLS_DIRS`, `COPILOT_HOME`, `COPILOT_MODEL`, and auth tokens, plus config keys for hooks and related behavior; none of the docs reviewed describe an env var that carries a startup-context blob or a per-prompt advisor brief into model context. [SOURCE: https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-command-reference] [SOURCE: https://docs.github.com/en/copilot/reference/copilot-cli-reference/cli-programmatic-reference]
- **A shell-wrapper fallback is documented for one-shot prompt prepending, but only in programmatic mode.** GitHub documents `copilot -p "..."` and `echo "..." | copilot` as supported programmatic entry points, which means a wrapper can prepend an advisor/startup brief into each non-interactive prompt; this does not replicate interactive-session hook parity, and piped input is ignored if `-p/--prompt` is also supplied. [SOURCE: https://docs.github.com/en/copilot/how-tos/copilot-cli/automate-copilot-cli/run-cli-programmatically] [SOURCE: https://docs.github.com/en/copilot/concepts/agents/about-copilot-cli]

## Questions Answered
- **Q1:** Yes, Copilot CLI has a documented lifecycle hook API (`sessionStart`, `userPromptSubmitted`, `preToolUse`, etc.), but the documented customer-hook outputs are ignored for prompt/session mutation; only `preToolUse` can currently influence execution, and only through permission decisions. Evidence: hooks how-to + hooks reference.
- **Q2:** `/agent` and `--agent` are documented as selecting custom subagents defined by `.agent.md` profiles in repo/user agent folders. This moved Q2 forward by showing the model is subagent-specialization, not a documented main-session system-prompt injection layer. Evidence: about-custom-agents + create-custom-agents-for-cli + programmatic reference.
- **Q3:** `/skills` is documented as a task-scoped skill system that loads instructions/scripts/resources when relevant or manually invoked, distinct from always-on custom instructions. Evidence: about-agent-skills + comparing-cli-features.
- **Q4:** `/mcp` and `--additional-mcp-config` are documented as MCP server/tool registration surfaces. This moved Q4 forward by showing MCP adds callable tools and data sources, not a documented push-style context channel. Evidence: add-mcp-servers + comparing-cli-features + about-copilot-cli.
- **Q5:** Yes, Copilot CLI has documented startup-loaded instruction files and directories: `.github/copilot-instructions.md`, `.github/instructions/**/*.instructions.md`, `AGENTS.md`, `$HOME/.copilot/copilot-instructions.md`, and directories supplied through `COPILOT_CUSTOM_INSTRUCTIONS_DIRS`. Evidence: add-custom-instructions + custom-instructions-support + comparing-cli-features.
- **Q7:** The documented env vars/config keys include directory overrides, auth, model selection, skill directories, and hook/config toggles, but this iteration found no documented env var for arbitrary startup/prompt payload injection. Evidence: cli-command-reference + cli-programmatic-reference.
- **Q8:** Partially answered: the least-friction fallback is a shell wrapper around `copilot -p` or piped stdin that prepends the desired brief, but this only cleanly applies to non-interactive/programmatic usage and does not mirror interactive TUI lifecycle parity. Evidence: run-cli-programmatically + about-copilot-cli.

## Questions Remaining

- **Q8:** What breaks in practice for an interactive wrapper approach (TUI, stdin handling, token overhead, remote session behavior)?
