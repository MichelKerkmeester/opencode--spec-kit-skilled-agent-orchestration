# Iteration 6: Q1/Q2 stdin and stdout context confirmation
## Focus
This pass rechecked the hook stdin and stdout-to-model-context contract for Codex CLI hooks, prioritizing the current official Codex hooks documentation and upstream source paths. Because iteration 4 had already strongly answered Q1/Q2, the goal was to confirm whether any version-specific or local-installed evidence contradicted that contract.
## Actions Taken
1. Read the current official Codex hooks documentation for common input fields, per-event `SessionStart` and `UserPromptSubmit` stdout behavior, event taxonomy, and timeout/concurrency collateral: https://developers.openai.com/codex/hooks
2. Opened upstream `command_runner.rs` to confirm command hooks are spawned with piped stdin/stdout/stderr, receive serialized `input_json` via stdin, run in the session `cwd`, and use the configured timeout path: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/command_runner.rs
3. Re-read upstream `user_prompt_submit.rs` to confirm plain stdout and JSON `hookSpecificOutput.additionalContext` are accumulated in `additional_contexts_for_model`: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/user_prompt_submit.rs
4. Re-read upstream `session_start.rs` to confirm `SessionStart` plain stdout and JSON `additionalContext` are also accumulated as model context, and that invalid JSON-like stdout fails rather than becoming context: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/session_start.rs
5. Read the generated upstream `SessionStart` input schema to verify exact required stdin fields for that event: https://github.com/openai/codex/blob/main/codex-rs/hooks/schema/generated/session-start.command.input.schema.json
6. Read upstream issue evidence describing the model-visible injection path as `DeveloperInstructions::new(additional_context).into()`, with TUI rendering as a separate display concern: https://github.com/openai/codex/issues/16486
7. Rechecked local installed evidence without modifying `~/.codex/hooks.json`: `/opt/homebrew/bin/codex` reports `codex-cli 0.122.0`; the Homebrew cask path contains `/opt/homebrew/Caskroom/codex/0.122.0/codex-aarch64-apple-darwin`; `~/.codex/hooks.json` still registers only Superset `notify.sh`; `notify.sh` reads either argv or stdin and parses `hook_event_name`.
## Findings
- Q1: The current official contract is unambiguous: every command hook receives one JSON object on stdin, with common fields including `session_id`, `transcript_path`, `cwd`, `hook_event_name`, and `model`. Primary source: https://developers.openai.com/codex/hooks
- Q1: Source confirms the transport is stdin, not argv: `command_runner.rs` spawns the hook with piped stdin and writes the serialized `input_json` bytes to the child process. Primary source: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/command_runner.rs
- Q1: `SessionStart` stdin is a JSON object whose schema requires `cwd`, `hook_event_name:"SessionStart"`, `model`, `permission_mode`, `session_id`, `source`, and `transcript_path`; `source` includes `startup`, `resume`, and `clear` in the generated schema. Primary source: https://github.com/openai/codex/blob/main/codex-rs/hooks/schema/generated/session-start.command.input.schema.json
- Q1: Local Superset `notify.sh` is compatible with stdin JSON because it falls back to `cat` when no argument is passed and extracts `hook_event_name`; its comment claiming "Codex passes as argument" is contradicted by the upstream `command_runner.rs` stdin implementation and should not be treated as authoritative contract evidence. Primary sources: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/command_runner.rs and local `~/.superset/hooks/notify.sh` read-only inspection.
- Q2: For `SessionStart`, plain stdout is model-visible additional developer context, and JSON stdout can use `hookSpecificOutput.additionalContext`; source tests call this `plain_stdout_becomes_model_context`. Primary sources: https://developers.openai.com/codex/hooks and https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/session_start.rs
- Q2: For `UserPromptSubmit`, plain stdout is also added as extra developer context, and JSON `hookSpecificOutput.additionalContext` enters `additional_contexts_for_model`; malformed JSON-like stdout fails instead of being treated as plain context. Primary sources: https://developers.openai.com/codex/hooks and https://github.com/openai/codex/blob/main/codex-rs/hooks/src/events/user_prompt_submit.rs
- Q2: The injection is additive developer context, not prompt replacement. The public docs say "extra developer context," and issue #16486 identifies the core path as `DeveloperInstructions::new(additional_context).into()`, while noting the TUI rendering can flatten that same content for display. Primary sources: https://developers.openai.com/codex/hooks and https://github.com/openai/codex/issues/16486
- Q4/Q8 collateral: `command_runner.rs` shows no `env_clear` or environment override in the hook command construction path inspected here, while it does set `current_dir(cwd)` and uses the configured timeout with error text `hook timed out after Ns`; this supports inherited environment as the current source-level inference but still needs a focused Q8 audit or empirical probe to close. Primary source: https://github.com/openai/codex/blob/main/codex-rs/hooks/src/engine/command_runner.rs
## Questions Answered
- Q1 advanced but mostly confirmed prior work: both official docs and source say hooks receive a single JSON object on stdin; `SessionStart` schema gives exact fields. UserPromptSubmit exact fields were already captured in iteration 4 and remain supported by the same docs/source family.
- Q2 advanced but mostly confirmed prior work: `SessionStart` and `UserPromptSubmit` stdout can become extra developer context; source and issue evidence clarify this is additive developer-instruction context, not replacement of the prompt.
- Q4 advanced slightly: timeout error text and timeout implementation path were confirmed in source, though this pass did not run a timeout probe.
- Q8 advanced slightly: source shows cwd setup and no explicit env sanitization in the runner path, but inherited-env behavior remains an inference until a focused audit/probe verifies higher-level process policy.
## Questions Remaining
- Q3: Event-by-event exit code behavior is substantially mapped from iteration 5, but final synthesis should still consolidate it into one matrix.
- Q4: Timeout default is documented and source shows timeout error handling, but interactive/non-interactive surfacing and child-process kill behavior still need confirmation.
- Q5: Docs/source say matching hooks launch concurrently; side-effect ordering under multiple hooks remains empirically open.
- Q6: Six-event taxonomy is well evidenced, but final synthesis should reconcile the pre-research three-event local wiring with current upstream support.
- Q7: Block/decline is evidenced; prompt transformation appears unsupported, but final synthesis should cite schema absence explicitly.
- Q8: Environment propagation remains the main open item: command runner source implies inherited parent environment, but a higher-level env policy audit or isolated empirical probe is still needed.
## Next Focus
Iteration 7 should target Q4 and Q8 together: inspect timeout child-kill/reporting behavior and environment propagation using source-level audit plus, if acceptable, an isolated `--config-dir` probe that does not modify `~/.codex/hooks.json`.
