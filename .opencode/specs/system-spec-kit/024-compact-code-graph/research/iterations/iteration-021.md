# Iteration 021: Test Framework Patterns Analysis
## Focus
Analyze the existing Vitest suite under `.opencode/skill/system-spec-kit/mcp_server/tests/` to extract the dominant test-framework patterns, especially setup/teardown, in-memory SQLite usage, fixture structure, MCP handler testing, and the specific behaviors covered by `dual-scope-hooks.vitest.ts` and `crash-recovery.vitest.ts`.

## Findings
1. The suite is configured as a Node-only Vitest package with globals enabled, a 30-second default timeout, and a narrow include pattern of `tests/**/*.{vitest,test}.ts`; the test README groups the suite into cognitive, search, handler/protocol, save/index, eval, and governance categories, which matches the file layout and explains why most tests are self-contained instead of routed through a single global harness. (`.opencode/skill/system-spec-kit/mcp_server/vitest.config.ts:8-23`, `.opencode/skill/system-spec-kit/mcp_server/tests/README.md:25-43`, `.opencode/skill/system-spec-kit/mcp_server/tests/README.md:76-94`)

2. The dominant setup/teardown pattern is suite-local lifecycle control: `beforeEach` resets mocks, caches, and environment flags; `afterEach` restores spies and closes or clears temporary state; `beforeAll`/`afterAll` are used only when a suite needs a long-lived database or filesystem sandbox. `dual-scope-hooks.vitest.ts` clears the constitutional cache and re-primes mocked dependencies per block, `handler-memory-search.vitest.ts` initializes and closes an in-memory DB once per suite while restoring spies per test, and `shared-memory-handlers.vitest.ts` recreates a fresh `:memory:` database and shared-memory env state for every case. (`.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:78-84`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:115-121`, `.opencode/skill/system-spec-kit/mcp_server/tests/handler-memory-search.vitest.ts:35-50`, `.opencode/skill/system-spec-kit/mcp_server/tests/shared-memory-handlers.vitest.ts:64-80`)

3. In-memory SQLite is used in two recurring ways. The simplest pattern is `new Database(':memory:')` plus inline `CREATE TABLE` statements and optional seeding helpers, as seen in `handler-helpers.vitest.ts` and `memory-save-extended.vitest.ts`. The heavier pattern is a temp on-disk DB under `os.tmpdir()` or a suite-owned temp directory, with `MEMORY_DB_PATH` pointed at that file and the production `vectorIndex.initializeDb()` or schema helpers used to bootstrap the database; `checkpoint-completeness.vitest.ts` combines `createSchema()` with additional auxiliary tables, and `memory-save-ux-regressions.vitest.ts` uses a real temp DB path to exercise save-pipeline behavior closer to production. (`.opencode/skill/system-spec-kit/mcp_server/tests/handler-helpers.vitest.ts:101-190`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-extended.vitest.ts:32-51`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-extended.vitest.ts:64-154`, `.opencode/skill/system-spec-kit/mcp_server/tests/checkpoint-completeness.vitest.ts:66-193`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-ux-regressions.vitest.ts:32-41`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-ux-regressions.vitest.ts:184-233`)

4. Fixture structure is mostly local and composable rather than centralized. The common fixture styles are: small object factories such as `makeTriggerMatch()` and `makeRow()` for pure logic tests; DB builders such as `createTestDb()` plus `seedTestMemories()` for storage-heavy tests; and filesystem fixtures where a suite synthesizes a temporary spec folder or markdown memory file on demand. `memory-save-ux-regressions.vitest.ts` is the clearest file-backed fixture example: it builds a realistic markdown memory document with frontmatter, anchored sections, and recovery metadata, writes it into `tmp-test-fixtures`, then cleans matching DB rows after each test. The README’s mention of `fixtures/` is accurate, but the representative suites reviewed here rely more on inline factory functions and generated temp fixtures than on broad shared static fixture imports. (`.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:36-48`, `.opencode/skill/system-spec-kit/mcp_server/tests/handler-memory-search.vitest.ts:135-214`, `.opencode/skill/system-spec-kit/mcp_server/tests/handler-helpers.vitest.ts:101-190`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-ux-regressions.vitest.ts:43-152`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-ux-regressions.vitest.ts:158-233`, `.opencode/skill/system-spec-kit/mcp_server/tests/README.md:91-94`)

5. `dual-scope-hooks.vitest.ts` tests both hook entry points and their guardrails, not just happy paths. The covered scenarios are: exported token-budget constants; basic null-return behavior when there is no DB or no extractable context; recursion prevention for memory-aware tools; config-flag opt-out for both tool-dispatch and compaction hooks; context-hint extraction from `query`, `input`, `prompt`, and `concepts`; a lifecycle check proving one matcher call at tool dispatch and one at compaction; result-shape assertions for surfaced data; compaction-specific whitespace trimming; estimated-token enforcement against the 4000-token budgets; regression protection for `autoSurfaceMemories`; and export verification for the new hook APIs. (`.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:54-72`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:78-109`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:115-185`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:191-282`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:288-307`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:313-373`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:379-532`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:538-627`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:633-733`)

6. `crash-recovery.vitest.ts` deliberately avoids a real SQLite dependency and instead uses a hand-built SQL-aware mock harness plus temp files to cover recovery semantics. The recovery scenarios include: schema creation and index creation for `session_state`; saving session state with identity fields; resetting active sessions to `interrupted`; listing interrupted sessions; recovering an interrupted session with `_recovered=true`; generating and checkpoint-writing `CONTINUE_SESSION.md`; verifying schema columns and status constraints; immediate persistence of session-state fields and JSON payloads; status transitions to `completed` and `interrupted`; reactivation after recovery; and transaction-manager recovery for committed pending files, stale pending files, and mixed multi-file recovery results. (`.opencode/skill/system-spec-kit/mcp_server/tests/crash-recovery.vitest.ts:7-30`, `.opencode/skill/system-spec-kit/mcp_server/tests/crash-recovery.vitest.ts:87-322`, `.opencode/skill/system-spec-kit/mcp_server/tests/crash-recovery.vitest.ts:324-347`, `.opencode/skill/system-spec-kit/mcp_server/tests/crash-recovery.vitest.ts:350-420`, `.opencode/skill/system-spec-kit/mcp_server/tests/crash-recovery.vitest.ts:423-551`, `.opencode/skill/system-spec-kit/mcp_server/tests/crash-recovery.vitest.ts:553-760`, `.opencode/skill/system-spec-kit/mcp_server/tests/crash-recovery.vitest.ts:763-855`)

7. MCP tool handlers are usually tested by importing the handler module directly, invoking the exported `handle...` function with plain argument objects, parsing `response.content[0].text` as a JSON envelope, and then asserting either `isError` or specific payload fields. Dependency boundaries are isolated with `vi.mock`, `vi.hoisted`, and `vi.spyOn`: `handler-memory-search.vitest.ts` spies on `core.checkDatabaseUpdated()` and validates envelope errors plus `__testables`; `shared-memory-handlers.vitest.ts` hoists a mutable DB holder and mocks `requireDb`; `dual-scope-hooks.vitest.ts` mocks transitive DB and trigger-matcher dependencies before importing the hook module. This means the suite prefers direct handler invocation over server-level black-box tests except where protocol-format tests are explicitly the goal. (`.opencode/skill/system-spec-kit/mcp_server/tests/handler-memory-search.vitest.ts:17-33`, `.opencode/skill/system-spec-kit/mcp_server/tests/handler-memory-search.vitest.ts:35-115`, `.opencode/skill/system-spec-kit/mcp_server/tests/handler-memory-search.vitest.ts:135-214`, `.opencode/skill/system-spec-kit/mcp_server/tests/shared-memory-handlers.vitest.ts:13-31`, `.opencode/skill/system-spec-kit/mcp_server/tests/shared-memory-handlers.vitest.ts:47-80`, `.opencode/skill/system-spec-kit/mcp_server/tests/shared-memory-handlers.vitest.ts:82-219`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:11-34`)

8. The suite does have reusable patterns, but they are mostly “copyable local helpers” rather than one shared `test-utils` module. Repeated helper families include `createTestDb`, `seed...`, `parseEnvelope`/`parseResponse`, `createMock...`, and temp-dir builders; optional or environment-sensitive coverage is gated with `it.skipIf`, and more integration-like suites sometimes use module-scope DB initialization so the skip condition can be evaluated at test-registration time. For richer evaluation cases, fixtures are still encoded inline in the test file itself, such as `EVAL_CORPUS`, `QUERY_DOCUMENTS`, and provider response fixtures in `reranker-eval-comparison.vitest.ts`. (`.opencode/skill/system-spec-kit/mcp_server/tests/ablation-framework.vitest.ts:45-153`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-extended.vitest.ts:32-51`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-extended.vitest.ts:64-154`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-extended.vitest.ts:160-240`, `.opencode/skill/system-spec-kit/mcp_server/tests/reranker-eval-comparison.vitest.ts:21-109`, `.opencode/skill/system-spec-kit/mcp_server/tests/reranker-eval-comparison.vitest.ts:111-180`)

9. New tests for hook scripts should follow the `dual-scope-hooks` model first and only escalate to file-backed fixtures when hook behavior truly depends on recovery files or persisted DB state. Concretely, that means: mock transitive DB/search dependencies before importing the hook module; keep helper factories local to the spec file; reset caches and mocked return values in `beforeEach`; assert null/no-op paths before positive results; verify envelope/result shape and token-budget boundaries explicitly; and add one regression block proving the new hook does not change existing hook behavior. If the hook touches persisted session or pending-file behavior, then adopt the `crash-recovery` or `memory-save-ux-regressions` style of temp directories, production-like markdown fixture content, and explicit cleanup. (`.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:11-48`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:78-109`, `.opencode/skill/system-spec-kit/mcp_server/tests/dual-scope-hooks.vitest.ts:538-699`, `.opencode/skill/system-spec-kit/mcp_server/tests/crash-recovery.vitest.ts:324-347`, `.opencode/skill/system-spec-kit/mcp_server/tests/crash-recovery.vitest.ts:811-855`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-ux-regressions.vitest.ts:43-152`, `.opencode/skill/system-spec-kit/mcp_server/tests/memory-save-ux-regressions.vitest.ts:158-233`)

## Evidence
- Configuration baseline: Vitest runs in Node with globals, a 30-second timeout, and a tests-only include pattern. (`.opencode/skill/system-spec-kit/mcp_server/vitest.config.ts:13-23`)
- Coverage map: the test README explicitly frames the suite as a mixed unit/integration/handler/eval/regression package and recommends pattern-based focused runs like `tests/handler-*.vitest.ts`. (`.opencode/skill/system-spec-kit/mcp_server/tests/README.md:25-43`, `.opencode/skill/system-spec-kit/mcp_server/tests/README.md:46-72`)
- Representative setup/fixture files reviewed in depth: `dual-scope-hooks.vitest.ts`, `crash-recovery.vitest.ts`, `handler-memory-search.vitest.ts`, `handler-helpers.vitest.ts`, `shared-memory-handlers.vitest.ts`, `checkpoint-completeness.vitest.ts`, `memory-save-extended.vitest.ts`, `memory-save-ux-regressions.vitest.ts`, `ablation-framework.vitest.ts`, and `reranker-eval-comparison.vitest.ts`.

## Novelty Justification
The highest-value new information is that the suite does not revolve around a single shared harness. Its dominant style is file-local setup, file-local DB builders, file-local fixture factories, and direct handler invocation with parsed JSON envelopes. That matters for compact-code-graph planning because new hook tests should be added as another self-contained Vitest file in the same style, not as a new cross-suite abstraction. The second important insight is that there are already two proven escalation paths: lightweight hook-style mocks from `dual-scope-hooks.vitest.ts`, and recovery/file-system realism from `crash-recovery.vitest.ts` plus `memory-save-ux-regressions.vitest.ts`.

## Recommendations for Implementation
- Model new compact-code-graph hook tests on `dual-scope-hooks.vitest.ts`: mock transitive dependencies before import, use local factory helpers, and reset mock return values in `beforeEach`.
- Add tests in layers: null/no-op behavior first, guardrails second, positive result shape third, and regression protection last.
- Use `better-sqlite3` `:memory:` databases when the hook only needs storage semantics; use a temp on-disk DB plus `MEMORY_DB_PATH` only when production path handling is part of the behavior.
- Prefer local `parseEnvelope` or `parseResponse` helpers over introducing a new shared test utility unless at least several new files would reuse it.
- If the hook needs persisted crash/recovery semantics, reuse the `crash-recovery.vitest.ts` pattern of temp directories, explicit pending files, and a dedicated mock DB harness instead of bolting recovery assertions onto a lightweight unit test.
- If the hook needs realistic saved-memory content, generate markdown fixtures in the style of `memory-save-ux-regressions.vitest.ts` so the test exercises frontmatter, anchors, and durable-memory gating together.
