# Iteration 020: Hook-State Design Patterns
## Focus
Analyze existing Spec Kit session-state patterns so `hook-state.ts` can safely bridge Claude hook sessions to MCP-server sessions, preserve compacted context, and recover cleanly after restarts.

## Findings
1. The current design is split into two layers, not one: `session-state.ts` is an ephemeral retrieval overlay, while the real MCP session lifecycle lives in `session-manager.ts`. The ephemeral layer tracks goal, seen results, questions, and anchors in memory only; the durable layer tracks status, scope, recovery data, and timestamps in SQLite. ([session-state.ts:13](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/session-state.ts#L13), [session-state.ts:33](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/session-state.ts#L33), [session-manager.ts:84](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/session/session-manager.ts#L84))

3. Session IDs are server-minted and server-trusted. If a caller omits `sessionId`, `resolveTrustedSession()` generates a UUID and treats it as the `effectiveSessionId`. 4. The durable lifecycle is `active -> interrupted -> active` on recovery, or `active -> completed` on normal end. `saveSessionState()` always upserts to `active`; startup crash recovery marks leftover active sessions as `interrupted`; `recoverState()` flips interrupted sessions back to `active` and sets `_recovered`; `completeSession()` marks `completed` and clears working memory. Stale cleanup later removes old `working_memory`, `session_sent_memories`, and only `completed`/`interrupted` rows from `session_state`, never active ones. This only runs in hybrid search when `config.sessionId` is present. 7. The bridge rule for `hook-state.ts` should be: never pass Claude’s raw `session_id` into Spec Kit as `sessionId`. That value is not trusted unless the server already minted and tracks it. The hook should store a local mapping `claudeSessionId -> effectiveSessionId`, mint the Spec Kit ID via `memory_context` on first contact, and reuse only the returned `effectiveSessionId` afterward. `memory_match_triggers` already documents this contract in its recovery hint. This is an inference from the trust model, but it is strongly implied by the server’s validation path. ([session-manager.ts:401](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/session/session-manager.ts#L401), [memory-context.ts:1049](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-context.ts#L1049), [memory-context.ts:1056](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-context.ts#L1056), [memory-triggers.ts:207](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-triggers.ts#L207), [memory-triggers.ts:225](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-triggers.ts#L225))

8. The best existing temp/cache-file pattern is the folder-discovery cache I/O pattern: create parent dirs recursively, write to a random-suffixed temp file, `fsync`, `close`, `rename` atomically, and always attempt temp cleanup in `finally`. ## Novelty Justification
The highest-value new information is that the bridge cannot treat Claude’s `session_id` as the Spec Kit session key. The MCP server has a trust boundary around `effectiveSessionId`, and lifecycle recovery is DB-backed in `session-manager.ts`, not in `session-state.ts`. That means `hook-state.ts` needs a mapping/cache layer, not just a thin pass-through wrapper.

## Recommendations for Implementation
- Create `hook-state.ts` as a small bridge module exporting through [hooks/index.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/hooks/index.ts).
- Use a local type like `HookSessionState { claudeSessionId, effectiveSessionId, specFolder, status, createdAt, updatedAt, lastCompactedAt, cacheKey, transitionTrace }`.
- Store hook state under `os.tmpdir()/speckit-claude-hooks/sessions/<sha256(claudeSessionId)>/`.
- Keep two files per session: `state.json` for ID/status metadata and `compact-context.json` for the cached compacted payload plus `generatedAt`, `sourceHash`, and `ttlMs`.
- Persist both files with the folder-discovery atomic-write pattern: temp suffix, write, fsync, rename, cleanup.
- On first use, call `memory_context` without `sessionId`, read back `sessionLifecycle.effectiveSessionId`, and persist the mapping. Never send raw Claude `session_id` into MCP tools.
- On reuse, pass only the cached `effectiveSessionId`. If the server returns session-scope rejection, delete the mapping, mint a new session, and fall back to `memory_context({ mode: "resume", specFolder, ... })`.
- Reuse the `session-transition.ts` model for hook traces so state changes remain explainable: ordered signals, normalized trace payload, confidence, and best-effort attach/read behavior.
