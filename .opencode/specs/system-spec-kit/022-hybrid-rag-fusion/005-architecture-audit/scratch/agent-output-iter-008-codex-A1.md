## Findings

- High: Shared DB path resolution can silently diverge from the MCP server and scripts. [`shared/paths.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/paths.ts#L12) only derives `DB_PATH` from `SPEC_KIT_DB_DIR`/`SPECKIT_DB_DIR`, while the server/vector store still gives precedence to `MEMORY_DB_PATH` in [`vector-index-store.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/vector-index-store.ts#L213) and uses a separate resolver in [`mcp_server/core/config.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/core/config.ts#L42). In practice, the “shared bridge” can point scripts at one SQLite file while the MCP server reads another.

- High: `shared/paths.ts` and `shared/config.ts` allow database paths to escape the project root with no validation. [`shared/paths.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/paths.ts#L13) and [`shared/config.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/config.ts#L42) call `path.resolve(process.cwd(), configuredDir, ...)` directly, so `SPEC_KIT_DB_DIR=/etc` or similar resolves outside the workspace. The shared layer already has a hardened validator in [`shared/utils/path-security.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/utils/path-security.ts#L17), but these path builders do not use it.

- High: Custom embedding dimensions are accepted by the OpenAI and Voyage providers but never sent to the provider APIs, so non-default dims fail at runtime. Both constructors honor `options.dim` in [`shared/embeddings/providers/openai.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/openai.ts#L98) and [`shared/embeddings/providers/voyage.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/voyage.ts#L110), but the request bodies in [`openai.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/openai.ts#L126) and [`voyage.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/voyage.ts#L131) never include a dimensions/output-dimension field. The later length check then throws a mismatch error instead of producing the requested size.

- High: `getEmbeddingDimension()` hardcodes provider defaults and ignores model-specific or custom dimensions until a provider instance already exists. [`shared/embeddings.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings.ts#L625) returns `1024` for any Voyage config and `1536` for any OpenAI config before initialization, even though the providers explicitly define other valid dimensions like `text-embedding-3-large = 3072` and `voyage-code-2 = 1536` in [`openai.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/openai.ts#L19) and [`voyage.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/embeddings/providers/voyage.ts#L19). Any pre-init schema check or DB/profile selection using this helper can make the wrong compatibility decision.

- Medium: Cross-variant RRF is mathematically wrong by default because each variant is normalized before the variants are merged. [`fuseResultsCrossVariant()`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L378) calls [`fuseResultsMulti()`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L223) per variant, and that function normalizes scores at [`rrf-fusion.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L281) before the cross-variant merge at [`rrf-fusion.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/rrf-fusion.ts#L405). That destroys raw reciprocal-rank magnitudes, so a weak variant can contribute as much as a strong one.

- Medium: `semanticChunk()` duplicates overlapping text on medium-sized documents and can overrun its own budget rules. In [`shared/chunking.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/chunking.ts#L49), the first 500 chars and last 300 chars are always reserved; when `text.length` is between those totals and `maxLength`, the overview and outcome overlap, so the result duplicates content before being blindly truncated at [`chunking.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/chunking.ts#L103). Also, all “critical” sections are added without any budget check at [`chunking.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/chunking.ts#L81), which can push the function into arbitrary end truncation.

- Medium: MMR silently pads vectors with different lengths instead of treating them as incompatible. [`computeCosine()`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/mmr-reranker.ts#L48) zero-pads any mismatch up to 10% at [`mmr-reranker.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/algorithms/mmr-reranker.ts#L57). That is not cosine similarity in a single embedding space, and it can hide dimension drift instead of surfacing it immediately.

- Medium: Token estimation has no bounded accuracy guarantee, so `maxTokens` is not a real token ceiling. [`estimateTokenCount()`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/utils/token-estimate.ts#L9) and the structure-aware chunker’s `CHARS_PER_TOKEN = 4` in [`structure-aware-chunker.ts`](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/shared/lib/structure-aware-chunker.ts#L39) assume a fixed 4 chars/token for every model and content type. That can be off by multiples, not percentages: commonly 2x-4x low on code/CJK/punctuation-heavy text, and sometimes much too high on repetitive or whitespace-heavy text.
