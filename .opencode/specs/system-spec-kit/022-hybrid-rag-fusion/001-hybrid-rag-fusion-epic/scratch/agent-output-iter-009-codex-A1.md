**2. Dedup behavior**
- `checkExistingRow()` is same-path dedup. It matches the latest top-level row in the same scope and returns `unchanged` only when `content_hash` matches exactly, `embedding_status` is one of `success|pending|partial`, trigger phrases and quality flags are equivalent, quality score matches within epsilon, and `force` is false. Refs: [dedup.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/dedup.ts#L14), [dedup.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/dedup.ts#L67), [dedup.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/dedup.ts#L90), [dedup.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/dedup.ts#L146).
- `checkContentHashDedup()` is cross-path exact-hash dedup. It looks for a top-level row in the same spec folder/scope with identical `content_hash` and `embedding_status` in `success|partial`; it can exclude the current path via `file_path` and `canonical_file_path`. Refs: [dedup.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/dedup.ts#L15), [dedup.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/dedup.ts#L162), [dedup.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/dedup.ts#L248).
- `dedup.ts` has no similarity threshold. It is exact path/hash dedup only. Near-dupes can absolutely slip past it if the content hash changes. Later layers are what catch semantic duplicates: TM-04 save-quality gate rejects at `0.92` similarity, and PE candidate fetch starts at `50%` similarity before handing off to prediction-error logic. Refs: [save-quality-gate.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/validation/save-quality-gate.ts#L100), [save-quality-gate.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/validation/save-quality-gate.ts#L578), [pe-gating.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/pe-gating.ts#L114).

**3. Quality gates that can reject a save**
- Preflight can block the save unless `skipPreflight=true`. Refs: [memory-save.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts#L860), [memory-save.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts#L938).

- Quality loop rejects when weighted score stays below `0.6` after up to 2 auto-fix retries. Its dimensions are triggers `0.25`, anchors `0.30`, budget `0.20`, coherence `0.25`; it can reject for missing/low triggers, broken anchors, oversized content, empty/short/no-heading content, future-dated completion claims, and bad causal references. Auto-fix only re-extracts triggers, normalizes unclosed anchors, and trims to budget. **6. Concurrency / race conditions**
- Same-process concurrent saves for the same `specFolder` are serialized by an in-memory promise mutex around the critical section, which materially reduces TOCTOU races between dedup and insert. Refs: [memory-save.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts#L119), [memory-save.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts#L493).
- Record creation/update/governance steps are transactional once entered. Refs: [create-record.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/save/create-record.ts#L120), [pe-gating.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/pe-gating.ts#L256), [memory-save.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/memory-save.ts#L980).
- Remaining risk: the mutex is process-local only. Two separate MCP server processes can still race through dedup-before-insert. There is no DB uniqueness on `content_hash` or canonical path for ordinary top-level memories; `memory_index` only declares `UNIQUE(spec_folder, file_path, anchor_id)`, which does not protect rows where `anchor_id` is `NULL`. Refs: [vector-index-schema.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/vector-index-schema.ts#L1843), [vector-index-schema.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/vector-index-schema.ts#L1901).
- The embedding cache has an acknowledged benign race: concurrent writers can temporarily overshoot the cache-size limit because eviction is count-then-delete outside a transaction. Refs: [embedding-cache.ts](/Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/cache/embedding-cache.ts#L95).
