**Answers**
1. **Is the ablation framework complete? Can it actually disable individual scoring signals?**  
No. It is connected enough to run against `hybridSearchEnhanced()`, but it only toggles top-level channel flags plus a trigger-phrase routing hint. It cannot disable Stage 2 scoring signals like session boost, causal boost, co-activation, graph signals, feedback signals, validation scoring, or later rerank/truncation behavior. It also misses the live `degree` channel entirely. References: [eval-reporting.ts:84](//Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/handlers/eval-reporting.ts#L84), [ablation-framework.ts:82](//Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/eval/ablation-framework.ts#L82), [hybrid-search.ts:561](//Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/hybrid-search.ts#L561), [stage2-fusion.ts:21](//Users/michelkerkmeester/MEGA/Development/Opencode%20Env/Public/.opencode/skill/system-spec-kit/mcp_server/lib/search/pipeline/stage2-fusion.ts#L21)

2. **What does ground truth data look like? Is it sufficient for calibration?**  
It is a synthetic, structured benchmark: query text plus intent/category/source metadata, and graded `(queryId, memoryId, relevance)` judgments. That is enough for offline benchmarking, but weak for calibration because the loader schema has no annotator identity, disagreement, confidence, timestamp, tier, or candidate-negative structure. From local JSON inspection, the corpus is also very uniform: 110 queries, 297 labels, exactly 3 judged memories per query, and an even 1/2/3 relevance split.
