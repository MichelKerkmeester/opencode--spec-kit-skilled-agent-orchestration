## Findings

### Q1 - Skill Auto-Loading

- Slim does not implement file-type or prompt-keyword skill selection. OpenCode core injects `<available_skills>` globally, and Slim's hook only rewrites that block before API send based on the current agent's skill permissions: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/filter-available-skills/index.ts:2`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/filter-available-skills/index.ts:103`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/filter-available-skills/index.ts:127`.
- The current agent is inferred from the latest user message's `info.agent`, defaulting to `orchestrator`; this means filtering is per active agent/session, not per file or task content: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/filter-available-skills/index.ts:37`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/filter-available-skills/index.ts:137`.
- Permission matching is strict name/wildcard matching. A specific skill rule wins; otherwise `* === allow` grants all skills, and no rule means denied for non-orchestrator agents: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/filter-available-skills/index.ts:68`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/skills.ts:120`.
- Explicit `agents.<agent>.skills` config overrides recommendation defaults, including empty arrays. `*` allows all, `!name` denies a skill, and named entries allow specific skills: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/skills.ts:125`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/skills.ts:128`.
- Default skill recommendations come from three registries: installable recommended skills (`agent-browser` for `designer`), bundled custom skills (`simplify` for `oracle`, `codemap` for `orchestrator`), and permission-only skills (`requesting-code-review` for `oracle`): `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/skills.ts:39`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/custom-skills.ts:30`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/skills.ts:57`.
- Install-time and runtime are separated. Recommended skills are installed via `npx skills add`, bundled skills are copied into the OpenCode skills directory, and runtime access is still controlled through permissions/filtering: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/skills.ts:71`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/custom-skills.ts:84`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/docs/skills.md:3`.
- The generated provider preset surface pre-populates `skills`: orchestrator gets `["*"]`; other agents get only registry-matched skills; designer force-includes `agent-browser`: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/providers.ts:61`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/cli/providers.ts:78`.
- `<load_skills>` or `load_skills` is not a Slim skill-loader implementation. It appears in delegate retry guidance as a required OpenCode `task` argument when the task tool reports invalid arguments: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/delegate-task-retry/patterns.ts:15`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/delegate-task-retry/guidance.ts:35`.

Reusable pattern for `@code`: separate skill installation/registry from runtime visibility. Let the agent see only a filtered `<available_skills>` block derived from explicit agent policy, and treat task-call `load_skills` as a caller contract rather than an implicit auto-loader.

### Q5 - Council Child-Session Dispatch

- `CouncilManager.runCouncil()` validates depth, resolves a preset, notifies the parent session, runs councillors, and returns formatted results for the council agent to synthesize: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:77`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:142`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:177`.
- Councillors are created as child sessions with `parentID` set to the current council parent session and `query.directory` set to the plugin context directory: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:243`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:245`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:248`.
- The councillor prompt body is explicit and LEAF-safe: `agent: options.agent`, parsed `model`, `tools: { task: false }`, and a single text prompt part. Optional `variant` is copied into the body: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:271`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:278`.
- Councillor dispatch specifically passes `agent: 'councillor'`, `model: config.model`, a formatted prompt combining the base prompt and councillor-specific prompt, optional variant, timeout, and no reasoning extraction: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:427`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:430`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:432`.
- Child session depth is registered immediately after creation and cleaned in `finally`; the session itself is aborted after result extraction: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:257`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:304`.
- Parallel execution is default, serial is configurable, and parallel launches can be staggered only when multiplexer panes are enabled: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:132`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:349`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/council/council-manager.ts:353`.
- Multiplexer session lifecycle listens to OpenCode `session.created` events with a `parentID`, records the child, spawns a pane, closes on idle/deleted/missing/timeout, and clears everything on cleanup: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/multiplexer/session-manager.ts:87`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/multiplexer/session-manager.ts:171`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/multiplexer/session-manager.ts:291`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/multiplexer/session-manager.ts:439`.
- Multiplexer factory is environment-gated: `auto` only activates inside tmux or zellij via environment variables, with no binary-availability fallback: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/multiplexer/factory.ts:38`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/multiplexer/factory.ts:39`.
- Documented council mental model matches the code: `@council` is the synthesizer agent, while `council.presets` contains fan-out councillor models: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/docs/council.md:56`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/docs/council.md:64`.

Reusable pattern for `@code`: when spawning internal helper sessions, use an explicit child-session contract (`parentID`, title, directory, chosen agent/model/variant), hard-disable nested `task`, and guarantee cleanup via abort plus depth-tracker cleanup.

### Q5 - Task Payload Validation Surface

- Slim itself does not validate the core OpenCode `task` schema. Its task-session manager only inspects `description`, `prompt`, `subagent_type`, and `task_id` after the runtime has accepted a task call: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/task-session-manager/index.ts:12`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/task-session-manager/index.ts:220`.
- For session reuse, Slim recognizes only a fixed agent-name set including `orchestrator`, `oracle`, `designer`, `explorer`, `librarian`, `fixer`, `observer`, `council`, and `councillor`: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/task-session-manager/index.ts:27`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/task-session-manager/index.ts:65`.
- If `subagent_type` is not one of those names, the hook returns without rewriting or remembering anything; this is a management filter, not an error validator: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/task-session-manager/index.ts:231`.
- The retry hook reveals the OpenCode-level contract Slim expects errors for: missing `run_in_background`, missing `load_skills`, mutual exclusion between `category` and `subagent_type`, missing both category and subagent, unknown category, unknown agent, unknown skills, and caller not allowed for an agent: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/hooks/delegate-task-retry/patterns.ts:7`.

Reusable pattern for `@code`: keep runtime task schema validation in the tool/runtime layer; use plugin hooks for ergonomic retry hints and session reuse, not as the source of authorization truth.

### Q4 - Default File-Tool Permission Inheritance

- `applyDefaultPermissions()` only adds or preserves `question`, `council_session`, and nested `skill` permissions. It does not add default denials for `bash`, `edit`, `write`, or other file tools: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/agents/index.ts:149`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/agents/index.ts:178`.
- Built-in subagents, custom subagents, and orchestrator all receive `applyDefaultPermissions()`, but only after their own agent definitions or overrides are merged: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/agents/index.ts:290`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/agents/index.ts:315`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/agents/index.ts:339`.
- Config merging is intentionally shallow so user-provided `tools` and `permission` fields in `opencode.json` survive plugin agent defaults: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/index.ts:399`.
- `councillor` is the explicit read-only exception: it sets `'*': 'deny'`, `question: 'deny'`, and allows only read/search/list/LSP/codesearch-style tools: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/agents/councillor.ts:69`.
- `fixer` has a write-capable prompt but no explicit permission object in its agent definition, so write capability depends on OpenCode defaults and any user/runtime permission config, plus apply-patch hook safety rather than agent-level deny/allow entries: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/agents/fixer.ts:59`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/agents/index.ts:178`.
- MCP permissions are a separate allowlist surface: the config hook creates `<mcp>_*` allow/deny rules per agent from each agent's `mcps` list and respects user-defined permission keys: `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/index.ts:656`, `.opencode/specs/z_future/improved-agent-orchestration/external/oh-my-opencode-slim/src/index.ts:684`.

Reusable pattern for `@code`: do not infer write safety from Slim's default agent permissions. If `@code` must be write-capable but bounded, define explicit file-tool permissions or hard runtime guards; prompts and apply-patch validation alone are insufficient as an authorization boundary.

## Questions Answered

- Q1: Mostly resolved. Slim provides install-time skill recommendation plus runtime skill visibility filtering, not semantic auto-loading. `<load_skills>` is a task-call contract surfaced through retry guidance, not a plugin-side loader.
- Q4: Resolved. Non-councillor agents inherit OpenCode/user defaults for `bash`/`edit`/`write`; Slim's default permission helper does not broadly lock those down.
- Q5: Mostly resolved. Council child sessions use direct `client.session.create` + `client.session.prompt`, not the OpenCode `task` tool. Parent-child lifecycle is anchored by `parentID`, event-driven multiplexer panes, session abort, and depth cleanup.

- Q1: OpenCode core behavior for actual `<available_skills>` injection remains external to this target tree. Within Slim, filtering is confirmed.
