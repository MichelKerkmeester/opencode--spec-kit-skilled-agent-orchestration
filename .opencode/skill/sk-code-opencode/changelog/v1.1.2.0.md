This release makes `sk-code-opencode` safer to use as the standards guide for current OpenCode system code. The docs now distinguish current NodeNext ESM TypeScript packages, legacy CommonJS utilities and OpenCode plugin ESM files instead of treating one module style as universal. The verifier guidance is also clearer about what the script checks automatically and what still needs human review.

> Spec folder: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/000-release-cleanup/002-sk-code-opencode-alignment` (Level 2)

---

## What Changed

#### Module Guidance

- **TypeScript package boundaries** -- The previous guidance could make CommonJS look like the default for all TypeScript. The references now describe NodeNext ESM for current MCP server code, ES2022 ESM for scripts and CommonJS only where the package boundary actually uses it.

- **JavaScript plugin boundaries** -- The JavaScript standards now keep CommonJS rules for non-plugin `.js` and `.cjs` utility files while preserving OpenCode plugin ESM default exports. That prevents future edits from forcing plugin files into the wrong module shape.

#### Verification

- **Verifier contract** -- The README, shared references and checklists now say the alignment verifier checks supported code and config files for deterministic marker-level issues. Broader judgment calls such as naming, comments, header polish and KISS/DRY/SOLID review remain manual checklist gates.

- **Command examples** -- The verifier quick-start now states that `--root` defaults to the current directory and can be repeated for scoped checks. That makes local validation easier to run without guessing whether the flag is mandatory.

#### Documentation

- **Evidence paths** -- Stale examples in shared, Python, JavaScript and TypeScript references were refreshed so future agents land on real files when following the standards.

- **Skill template alignment** -- The primary skill file now exposes the required `REFERENCES` heading while keeping the related-resource navigation that existing readers expect.

- **Release-cleanup linkage** -- The packet and parent release-cleanup maps now point at `002-sk-code-opencode-alignment` instead of the old feature-catalog phase name, which keeps resume and phase navigation aligned with the folder on disk.
