This release changes where review artifacts land on disk. Review folders now always go to the spec tree root instead of inside nested child phases. A shared path resolver handles the lookup. The reducer, command YAMLs, agent definitions and documentation are all updated to match.

> Spec folder: `.opencode/specs/skilled-agent-orchestration/042-sk-deep-research-review-improvement-2/010-root-level-review-research-convention` (Level 2)

---

## Features

#### Root-level review folder convention

Running deep-review on a child phase used to create `review/` inside that child folder. In a spec tree with many nested phases, review artifacts ended up scattered across dozens of directories three or four levels deep. Finding a specific review meant knowing the exact nesting path.

Review folders now always go to the root of the spec tree. When the target is a child phase, a subfolder matching the phase path is created inside the root `review/` folder. A standalone spec with no parent behaves the same as before.

Examples:
- Standalone `040-feature/` writes to `040-feature/review/`
- Child `026/009-playbook/` writes to `026/review/009-playbook/`
- Grandchild `026/010-search/001-fusion/` writes to `026/review/010-search-001-fusion/`

&nbsp;

#### Shared path resolver

A new `resolveArtifactRoot(specFolder, mode)` function in `system-spec-kit/shared/review-research-paths.cjs` walks up the directory tree to find the root parent spec. It returns `{ rootDir, subfolder }` so every consumer resolves the same path. The reducer, command YAML assets, and agent dispatch context all use it.

&nbsp;

#### Iteration schema updated across all runtimes

The deep-review agent docs in all four runtimes (OpenCode, Claude, Codex, Gemini) now describe the live iteration format: Dispatcher, Files Reviewed, Findings - New (with P0/P1/P2 subsections), Traceability Checks, Confirmed-Clean Surfaces, Next Focus. The reducer accepts both the old and new schemas for backward compatibility.
