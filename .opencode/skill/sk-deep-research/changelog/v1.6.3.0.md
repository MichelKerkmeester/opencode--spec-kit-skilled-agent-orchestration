This release changes where research artifacts land on disk. Research folders now always go to the spec tree root instead of inside nested child phases. A shared path resolver handles the lookup. The reducer, command YAMLs, agent definitions and documentation are all updated to match.

> Spec folder: `.opencode/specs/skilled-agent-orchestration/042-sk-deep-research-review-improvement-2/010-root-level-review-research-convention` (Level 2)

---

## Features

#### Root-level research folder convention

Running deep-research on a child phase used to create `research/` inside that child folder. In a spec tree with many nested phases, research artifacts ended up scattered across many directories. Finding a specific research output meant knowing the exact nesting path.

Research folders now always go to the root of the spec tree. When the target is a child phase, a subfolder matching the phase path is created inside the root `research/` folder. A standalone spec with no parent behaves the same as before.

Examples:
- Standalone `040-feature/` writes to `040-feature/research/`
- Child `026/014-memory-save/` writes to `026/research/014-memory-save/`
- Grandchild `026/010-search/001-fusion/` writes to `026/research/010-search-001-fusion/`

&nbsp;

#### Shared path resolver

The same `resolveArtifactRoot(specFolder, mode)` function from `system-spec-kit/shared/review-research-paths.cjs` is used by both deep-research and deep-review. The reducer, command YAML assets, and agent dispatch context all resolve through it.

## Bug Fixes

Deep-research agent definitions in Claude, Gemini, and Codex runtimes now reference their own runtime directory instead of `.opencode/agent/`. The Gemini runtime path was also added to the capability matrix.
