A 291-scenario manual testing playbook run against the memory system found 12 failures. Four of those had fixable root causes -- three startup tests that broke when the server gained an import safety guard, ten code quality warnings that accumulated during the ESM migration, stale evaluation test data pointing at memory records that no longer exist, and a leftover import from the old module system. The other eight failures were knowledge gaps in the database (missing documentation about environment variables) and test harness limitations, not code bugs. The goal is to make the workflow easier to follow and easier to trust.

> Spec folder: `.opencode/specs/system-spec-kit/023-hybrid-rag-fusion-refinement/006-review-remediation/` (Level 2)

## What Changed

#### Testing

- **Three startup tests stopped working after the import safety guard landed** -- The previous release added a guard so that importing the server package no longer starts the full MCP server as a side effect. That guard also prevented the test harness from running the startup code it needed to verify dimension checks, dynamic instruction generation, and per-connection instruction refresh. The test harness now explicitly calls the startup function after importing the module, and all three tests pass again (359 out of 359, up from 356).

- **Evaluation test data pointed at memory records that no longer exist** -- The evaluation framework (ablation studies) uses a file of 290 "ground truth" entries that map test queries to known-good memory records by their database ID. After months of database changes, 294 of those IDs no longer matched any record. Re-running the existing mapping script updated all 290 entries to point at the correct current records, with zero left unmapped.

&nbsp;

#### Bug Fixes

- **Ten code quality warnings cleared across seven files** -- Five unused imports and variables were removed (they were declared but never referenced, which clutters the code and triggers lint failures). Two variables that were declared as changeable but never actually changed were tightened to constants. Two type annotations that the MCP SDK (the server framework) requires to be loosely typed were marked with explicit exceptions so the linter no longer flags them. The codebase now passes lint with zero warnings.

- **A leftover import from the old module system was still present** -- During the ESM migration, one file kept an import from the old CommonJS module loader (`createRequire`) even though nothing in that file used it anymore. Removed.
