This release fix MCP Server "no such table" Errors in /memory:manage -- Eliminated SQLite "no such table" errors (memory_index, checkpoints) that occurred when the LLM fell back to raw sqlite3 Bash commands against uninitialized databases. Four-layer fix: removed Bash from manage allowed-tools, added missing checkDatabaseUpdated() guard to health handler, extracted idempotent ensureCompanionTables() for schema resilience, and improved error messages with actionable recovery instructions. Code review pass: normalized all snake_case locals to camelCase, consolidated toErrorMessage usage in checkpoints, fixed stale section references in manage, and updated health table list. It focuses on fixing important problems and explaining the results in clearer terms. The goal is to make the workflow easier to follow and easier to trust.

---

## What Changed

#### Removed Bash fallback path

There was a reliability problem around Removed Bash fallback path. Removed Bash fallback path: Removed Bash from manage allowed-tools to prevent the LLM from running raw sqlite3 commands against uninitialized database files. Added explicit tool enforcement block prohibiting direct database access. This matters because it reduces confusing failures and makes the system more dependable.

&nbsp;

#### Health handler guard

There was a reliability problem around Health handler guard. Health handler guard: Added await checkDatabaseUpdated() to handleMemoryHealth() - the only handler missing this standard guard call. Added no such table detection that returns E_SCHEMA_MISSING with recovery instructions instead of silently reporting "degraded" status. This matters because it reduces confusing failures and makes the system more dependable.

&nbsp;

#### Companion table resilience

There was a reliability problem around Companion table resilience. Companion table resilience: Extracted ensureCompanionTables() function containing idempotent CREATE TABLE IF NOT EXISTS statements for memory_history, checkpoints, and memory_conflicts plus their indexes. Called from both the existing-database early-return path and the new-database path of create_schema(), ensuring companion tables are always created even if memory_index already exists. This matters because it reduces confusing failures and makes the system more dependable.

&nbsp;

#### Actionable error messages

There was a reliability problem around Actionable error messages. Actionable error messages: Improved and aligned error messages in requireDb(), checkpoints.getDatabase(), handleMemoryList(), and handleMemoryStats() null-db responses to suggest specific recovery actions. This matters because it reduces confusing failures and makes the system more dependable.

&nbsp;

#### CamelCase normalization

This release updates CamelCase normalization. CamelCase normalization: Renamed all snake_case local variables in handleMemoryDelete, handleMemoryUpdate, handleMemoryList, and handleMemoryStats to camelCase (spec_folder to specFolder, checkpoint_name to checkpointName, allow_partial_update to allowPartialUpdate, etc.). This matters because the release is clearer and easier to work with.

&nbsp;

#### ToErrorMessage consolidation

There was a reliability problem around ToErrorMessage consolidation. ToErrorMessage consolidation: Imported and applied toErrorMessage() from db helpers across all 8 inline error instanceof Error ? error.message : String(error) patterns in checkpoints. &nbsp;
