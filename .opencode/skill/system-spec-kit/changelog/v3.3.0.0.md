Two major programs land in this release. The compact code graph program (packet 024, 30 phases) gives AI sessions durable context recovery after compaction, restart, or runtime switches -- with Claude hook rescue, structural code-graph retrieval, CocoIndex bridging, hookless bootstrap, cross-runtime parity, and an OpenCode transport/startup plugin. The agent improver program (packet 041, 11 phases) adds a new skill that evaluates any agent across 5 integration-aware dimensions with deterministic scoring, dynamic profiling, and guarded promotion gates. The goal is to make the workflow easier to follow and easier to trust.

> Spec folders:
> - `.opencode/specs/system-spec-kit/024-compact-code-graph/` (Level 3, 30 phases)
> - `.opencode/specs/skilled-agent-orchestration/041-sk-recursive-agent-loop/` (Level 3, 11 phases)

## Context Preservation Stack (024)

Long AI coding sessions no longer depend on the user re-explaining context after compaction, restart, or runtime switches.

#### Compaction rescue for Claude sessions

Claude compaction wiped out the AI's short-term memory. After a compact, the assistant lost the active spec folder, recent code focus, and the rules guiding the work. A two-step rescue path: PreCompact caches a recovery payload before compression, and SessionStart injects it back for `source=compact`. Working-memory attention signals, spec-folder detection, and token-budget controls keep the recovered brief small enough to fit while still being useful.

&nbsp;

#### Source-aware session priming

Startup, resume, clear, and compact were treated too similarly, giving the wrong context for each situation. Session priming now branches by source. Compacts restore cached context, resumes point at the compact brief, startups surface tool and freshness guidance, and clears fall back to minimal recovery instructions.

&nbsp;

#### Structural code graph retrieval

Semantic search could find code by meaning but could not answer structural questions like "who calls this?" or "what does this file import?". Added a SQLite-backed code graph with scan, query, status, and context-expansion tools. Progressive migration from regex-only to tree-sitter-backed parsing, with regex retained as fallback.

&nbsp;

#### Freshness-aware startup and bounded auto-indexing

Structural tools need to tell the truth about graph freshness without quietly launching expensive full rebuilds in the background. Startup and recovery surfaces now report whether the graph is fresh, lightly stale, or needs a full rebuild. Structural read paths can auto-index small stale deltas inline with bounded `selective_reindex`, while empty or broadly stale states return explicit `full_scan` readiness and direct operators to `code_graph_scan`.

&nbsp;

#### CocoIndex bridge and intent-aware routing

Semantic search, structural search, and memory search lived beside each other without guidance about when to use which. Seed-resolution and context orchestration now let CocoIndex hits expand into structural neighborhoods. Intent metadata and routing enforcement push semantic questions toward CocoIndex, structural questions toward the graph, and session-history questions toward memory.

&nbsp;

#### Hookless auto-priming and bootstrap recovery

Claude hooks covered only one runtime family. Other runtimes depended on the AI remembering to prime itself. Added MCP first-call priming, `session_resume`, `session_health`, and `session_bootstrap` so non-hook runtimes recover with a single main path. The OpenCode transport layer and repo-local Copilot startup wrapper extend recovery beyond hook-based runtimes.

&nbsp;

#### Repo-relative MCP wrapper paths

Some checked-in MCP config surfaces drifted to absolute `/Users/...` wrapper paths, which made the JSON noisier and less portable than the rest of the repo-local setup. Restored repo-relative `spec_kit_memory` wrapper paths in the checked-in MCP JSON surfaces so the configuration stays consistent with the workspace-relative command style used elsewhere in the project.

&nbsp;

#### Stop-hook tracking and best-effort auto-save

The system had no reliable way to capture end-of-session state automatically. Stop-hook tracking parses transcript usage data, stores metrics, estimates token spend, and can trigger a best-effort `generate-context.js` save when configured thresholds are met.

&nbsp;

#### Cross-runtime contract parity

Runtime instructions, context-prime agents, and bootstrap surfaces drifted apart across runtimes. Standardized non-hook recovery around explicit expected behaviors: ready/stale/missing structural state, main `session_bootstrap` path, and mirrored routing/recovery tables across all runtime-facing docs.

## Hook Durability and Correctness (024)

#### Hook security and cache hardening

Cache races, weak temp-file handling, session-state collisions, and insufficient provenance markers made the hook path fragile. Tightened the hook-state lifecycle with TTL checks, SHA-256 state names, restrictive file permissions, provenance fencing, and stale-on-read checks.

&nbsp;

#### Correctness and boundary repair

Early graph work had correctness holes: collapsed line spans, unsafe DB initialization, transaction gaps, depth leaks. Hardened the graph pipeline with multi-line symbol spans, DB initialization guards, atomic replace operations, strict depth handling, and safer handler validation.

&nbsp;

#### Tree-sitter and classifier repair

First tree-sitter/classifier pass misclassified abstract methods, class expressions, multi-import capture, and nested Python class names. Repaired parser patterns and intent-classifier confidence so the parser and routing layers match the structural expected behavior.

&nbsp;

#### Startup highlight remediation

Startup highlights wasted space on duplicates, vendored code, and the wrong ranking signal. Rewrote the highlight query with deduplicated grouping, hard path exclusions, and incoming-call ranking.

## Agent Improvement Skill (041)

A new skill for evaluating and improving any agent definition file through a bounded, deterministic loop.

#### 5-dimension integration-aware scoring

No systematic way to evaluate agent prompt files. Manual review was inconsistent and could not track improvement over time. Created `sk-improve-agent` with 8 scripts: `scan-integration.cjs` discovers all surfaces an agent touches (main, mirrors, commands, YAML, skills); `generate-profile.cjs` derives scoring from any agent's own rules; `score-candidate.cjs` scores across 5 weighted dimensions (structural 0.20, ruleCoherence 0.25, integration 0.25, outputQuality 0.15, systemFitness 0.15). All scoring is deterministic -- regex, string matching, file-existence checks -- no LLM-as-judge.

&nbsp;

#### Proposal-only improvement loop

Editing main agent files directly during evaluation is dangerous -- a bad change could break the agent. The `@agent-improver` subagent writes candidates to a packet-local runtime area, never touching main targets. The `/improve:agent` command orchestrates the full loop: scan, profile, candidate, score, benchmark, reduce, stop-check. Promotion requires passing all evidence gates plus explicit operator approval.

&nbsp;

#### Self-referential test and fixes

The skill had never been tested on itself, leaving uncertainty about self-referential edge cases. Phase 009 ran the loop on `agent-improver.md` itself and found a real bug (stale command path). Phase 010 fixed that plus 4 reducer issues (family hardcoding, plateau window, accepted counting, mirror path) and promoted 6 candidate improvements from the test.

&nbsp;

#### Skill advisor routing and README documentation

The skill advisor had no routing for the new 5-dimension capabilities, and the root README was missing the agent-improver entries. Phase 011 added 28 routing entries to the skill advisor, bumped the skill version to 1.0.0.0, and added Agent-Improver to the root README's Agent Network and Skills Library sections. Skill count updated from 19 to 20.

## Upgrade

No migration required. Both programs are additive. The code graph, hooks, and bootstrap surfaces are backward compatible. The `sk-improve-agent` skill is entirely new and does not affect existing workflows. The `plateauWindow` config field defaults to 3 (same as previous hardcoded value). If you had custom references to `sk-recursive-agent`, update them to `sk-improve-agent`.
