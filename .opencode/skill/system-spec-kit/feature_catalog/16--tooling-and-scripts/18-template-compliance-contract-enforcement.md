## 1. OVERVIEW

Template Compliance Contract Enforcement is a 3-layer defense-in-depth system that ensures all spec folder documents generated by AI agents pass structural validation on first write.

The problem: AI agents writing spec folder markdown files frequently produce documents that fail structural validation — wrong heading levels, missing anchors, skipped required sections. Fixing these after the fact wastes cycles and clutters commit history.

The solution uses three complementary layers:

1. **Layer 1 — Agent Knowledge (Prevention):** A compact structural contract is embedded directly in all four CLI `@speckit` agent definitions. Before an agent writes any spec folder document, it already knows the exact heading structure, required anchors, and content minimums for each documentation level. The canonical source of truth is a shared reference file; each agent definition includes a 49-line compact version.

2. **Layer 2 — Post-Write Validation (Detection + Repair):** Immediately after writing any spec folder `.md` file, the agent runs `validate.sh --strict` against the spec folder. Exit codes are parsed (0 = pass, 1 = warnings, 2 = errors). On failure, validation flags the exact issues so the agent can apply targeted fixes and re-run validation before continuing.

3. **Layer 3 — Content Minimum Rules (Depth Safeguard):** Section-count and minimum-content rules enforce that documents are not only structurally valid, but also substantive enough to pass the level-specific quality bar. These checks backstop the agent contract and the validator so thin placeholder content cannot slip through.

Together, the three layers mean agents know structure before writing, validate after writing, and must satisfy content minimums before the workflow can be considered compliant.

## 2. CURRENT REALITY

The system comprises 4 components spanning references, agent definitions, and validation workflow:

**1. Shared reference file (`references/validation/template_compliance_contract.md`):** Canonical source of truth for structural contracts at all documentation levels (L1, L2, L3, L3+). Generated from `template-structure.js loadTemplateContract()` output. Defines the exact heading hierarchy, required anchors, and enforcement rules that all agents must follow. Versioned at 1.0.0.

**2. Compact structural contract in 4 CLI @speckit agent definitions:** A 49-line compact version of the structural contract is embedded in each of the four `@speckit` agent definitions — Claude Code (`.claude/agents/speckit.md`), Copilot (`.opencode/agent/speckit.md`), Codex (`.codex/agents/speckit.toml`), and Gemini (`.gemini/agents/speckit.md`). This gives every agent pre-write knowledge of anchor-to-H2 mappings for each document type at each level.

**3. Content minimums table (SECTION_COUNTS thresholds per level):** The contract defines minimum section counts per document per level. These thresholds are enforced both by the embedded agent contracts (Layer 1 prevention) and by the validation engine (Layer 2/3 detection). Agents reference these minimums when generating content to ensure documents are substantive, not just structurally correct.

**4. Post-Write Validation Protocol:** After writing any spec folder `.md` file, the agent must run `validate.sh --strict` against the spec folder. Exit code semantics: 0 = pass, 1 = warnings (acceptable), 2 = errors (must fix). On exit 2, the validator reports the failing sections and rules so the agent can apply targeted fixes and re-run validation. The contract is post-write validation plus explicit violation reporting, not a validator-managed automatic retry engine.
