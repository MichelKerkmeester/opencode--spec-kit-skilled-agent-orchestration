# Template Compliance Contract Enforcement

## 1. OVERVIEW

Template Compliance Contract Enforcement is a 2-layer defense-in-depth system that ensures all spec folder documents generated by AI agents pass structural validation on first write.

The problem: AI agents writing spec folder markdown files frequently produce documents that fail structural validation — wrong heading levels, missing anchors, skipped required sections. Fixing these after the fact wastes cycles and clutters commit history.

The solution uses two complementary layers:

1. **Layer 1 — Agent Knowledge (Prevention):** A compact structural contract is embedded directly in all four CLI `@speckit` agent definitions. Before an agent writes any spec folder document, it already knows the exact heading structure, required anchors, and content minimums for each documentation level. The canonical source of truth is a shared reference file; each agent definition includes a 49-line compact version.

2. **Layer 2 — Post-Write Validation (Detection + Repair):** Immediately after writing any spec folder `.md` file, the agent runs `validate.sh --strict` against the spec folder. Exit codes are parsed (0 = pass, 1 = warnings, 2 = errors). On failure, the agent enters a fix loop of up to 3 attempts before escalating.

Together, the two layers mean agents know structure before writing and validate after writing.

---

## 2. CURRENT REALITY

The system comprises 6 components spanning references, agent definitions, configuration, and scripts:

**1. Shared reference file (`references/template-compliance-contract.md`):** Canonical source of truth for structural contracts at all documentation levels (L1, L2, L3, L3+). Generated from `template-structure.js loadTemplateContract()` output. Defines the exact heading hierarchy, required anchors, and enforcement rules that all agents must follow. Versioned at 1.0.0.

**2. Compact structural contract in 4 CLI @speckit agent definitions:** A 49-line compact version of the structural contract is embedded in each of the four `@speckit` agent definitions — Claude Code (`.claude/agents/speckit.md`), Copilot (`.opencode/agent/speckit.md`), ChatGPT (`.opencode/agent/chatgpt/speckit.md`), and Codex (`.codex/agents/speckit.toml`). This gives every agent pre-write knowledge of anchor-to-H2 mappings for each document type at each level.

**3. Content minimums table (SECTION_COUNTS thresholds per level):** The contract defines minimum section counts per document per level. These thresholds are enforced both by the embedded agent contracts (Layer 1 prevention) and by the validation engine (Layer 2/3 detection). Agents reference these minimums when generating content to ensure documents are substantive, not just structurally correct.

**4. Post-Write Validation Protocol:** After writing any spec folder `.md` file, the agent must run `validate.sh --strict` against the spec folder. Exit code semantics: 0 = pass, 1 = warnings (acceptable), 2 = errors (must fix). On exit 2, the agent reads the error output, applies targeted fixes, and re-validates — up to 3 attempts maximum. If validation still fails after 3 attempts, the agent halts and reports the failure.
