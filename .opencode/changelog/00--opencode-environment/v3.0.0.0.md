## [**3.0.0.0**] - 2026-03-27

This release updates the OpenCode environment with work on platform architecture and surface area, spec kit memory engine: from storage to retrieval intelligence, save intelligence, query intelligence, and session behavior. The goal is to make the platform clearer, more reliable, and easier to use day to day.

### Platform Architecture And Surface Area
- Need: This part of the release improves how the environment saves, finds, and uses context.
- What changed: OpenCode becomes a full agentic environment instead of a collection of point tools. The release standardizes a platform footprint of 12 agents, 18 skills, 22 commands, 42 tool connection layer tools, 3 gates, 4 runtimes, 81 templates, and 6 analysis lenses so the same operational model spans planning, implementation, review, context recovery, and handoff. The Custom Agent Layer Is Now Broad Enough To Cover The Full Development Lifecycle. In Addition To The Built-in Roles, The Environment Now Formally Centers 10 Custom Agents: @orchestrate, @context, @speckit, @debug, @deep-research, @deep-review, @review, @write, @handover, and @ultra-think. Gate-driven workflow is now a first-class platform contract. The 3-gate model turns intent classification, skill routing, and spec-folder discipline into operational safety rails rather than optional conventions.
- Why it matters: That matters because saved context is easier to find, easier to trust, and less likely to get in the way.

### Spec Kit Memory Engine: From Storage To Retrieval Intelligence
- Need: This part of the release improves how the environment saves, finds, and uses context.
- What changed: The memory platform lands as a 33-tool, 7-layer system rather than a thin search feature. Retrieval, saving, analysis, management, constitutional memory, shared memory, and session continuity now form a cohesive memory operating system inside system-spec-kit. Five retrieval channels now participate in every serious memory lookup. Vector Voyage 1024d semantic search, SQLite full-text search, exact-word ranking lexical ranking, causal graph traversal, and degree-oriented graph signals all contribute evidence into the same retrieval pipeline. The query path is now a 4-stage pipeline. Requests move through Gather, Score and Fuse, local local ranking model reranking, and final filtering so the system can balance breadth, precision, and context pressure instead of relying on a single ranking step.
- Why it matters: That matters because saved context is easier to find, easier to trust, and less likely to get in the way.

### Save Intelligence, Query Intelligence, And Session Behavior
- Need: This part of the release improves how the environment saves, finds, and uses context.
- What changed: Saving memory is now an evaluated decision rather than a write-everything policy. PE-gated save intelligence classifies candidate writes into CREATE, REINFORCE, UPDATE, or SUPERSEDE operations and protects the store with 3 layers of gating. Query execution now adapts to user intent. Seven memory intents drive retrieval weighting, decomposition strategy, query expansion-style query expansion, and context-pressure management so different query types can surface different evidence blends. Session memory behavior was tuned around practical continuity. The session system now uses 0.85^distance decay, targets roughly 50 percent deduplication, and hits about 93 percent ANCHOR capture to preserve continuity without flooding prompts with repeated context.
- Why it matters: That matters because saved context is easier to find, easier to trust, and less likely to get in the way.

### Shared Memory, Governance, And Safety Controls
- Need: This part of the release improves how the environment saves, finds, and uses context.

Spaces, 3 explicit roles, deny-by-default access, and a kill switch allow multi-actor sharing without flattening tenant boundaries or weakening local control. Tenant-aware operations are part of the production path. Checkpoints and related persistence flows are now scoped so that shared-memory administration does not leak into normal tenant behavior. Governance text and operational rules were aligned around memory-first workflows. Commands, agents, skill documentation, and AGENTS templates now describe the same memory system rather than overlapping but inconsistent snapshots of earlier implementations.

### Autonomous Deep Research And Automated Code Review
- Need: This part of the release improves how the environment saves, finds, and uses context.
- What changed: Sk-deep-research Is Now A Dual-mode Autonomous Investigation System. It Operates In Two Modes: research mode for iterative technical investigation and review mode for automated code quality auditing - both built on the same LEAF (Loop, Externalize, Analyze, Finish) architecture with externalized line-based JSON log state, convergence detection, and fresh context per iteration. Research mode provides governed, observable deep investigation. Persistent dashboards track iteration progress, quality guards block premature convergence, negative knowledge is captured as first-class output alongside positive findings, novelty justification is required per iteration, per-iteration token budgets prevent runaway cost, and diagnosis-led recovery handles stuck iterations. Research charter validation and source-hygiene confirmation ensure quality from the start. Review Mode Turns The Same Iteration Loop Into A Systematic Code Quality Auditing Framework. Instead Of Ad Hoc Review Prompts, The: review mode dispatches @deep-review agents that produce severity-ranked highest-priority/high-priority/medium-priority findings with file:line evidence across 7 review dimensions, running iteratively until the review converges or the iteration limit is reached.
- Why it matters: That matters because saved context is easier to find, easier to trust, and less likely to get in the way.

### Documentation Tooling, Creation Standards, And Publish-Ready Knowledge
- Need: This part of the release improves how the environment saves, finds, and uses context.
- What changed: sk-doc becomes a full documentation production system, not just a style checker. The skill now governs README creation, feature catalog authoring, testing playbook generation, install guides, folder documentation, and general markdown quality through a unified DQI (Document Quality Index) scoring model with Structure 40%, Content 35%, and Style 25% weighting. README creation is now a repeatable, standards-driven workflow. The new readme_creation.md reference defines a 9-section matrix, section-specific writing guidance, type decision trees, and a pre-publish checklist so contributors follow a named authoring path instead of reverse-engineering expectations from existing files. HVR V0.210 Upgrades Documentation Quality From Cosmetic To Structural. The Ruleset Adds Heading Hierarchy Enforcement, Blockquote Tagline Conventions, TOC Format Rules, Two-tier Voice Guidance, Numbered Feature Subsection Patterns, Divider Usage Rules, Analogy Placement Controls, And 5 Recommended Table Patterns (How This Compares, Key Statistics, Key Features, Before/After, Parameter). Scoring Is Weighted (Punctuation 15%, Structure 25%, Content 25%, Words 20%, Voice 15%) With Explicit Thresholds: 85+ publish-ready, 70-84 needs revision, below 70 failing.

## Upgrade

- Pull the latest repository state so v3.0.0.0 lands as a consistent bundle instead of a partial cherry-pick of memory, docs, or command-line interface orchestration changes.
- Restart the Spec Kit Memory tool connection layer services so the hybrid retrieval stack, reranking path, shared-memory controls, tenant-aware checkpoint behavior, and new query/save intelligence are loaded together.
- Reindex or refresh memory state if your local workflow depends on immediate retrieval parity, especially if you are resuming saved context or testing retrieval behavior after upgrade.
- Review active feature flags before rollout. This release expands the flag surface to 26+ toggles and graduates more retrieval and observability behavior into normal operation.
- Validate core memory workflows after startup: retrieval, save, session continuity, checkpoint access, and any shared-memory paths you rely on for collaboration.
- Adopt the updated research and review workflows. Use the upgraded sk-deep-research flow for dashboarded research runs, and use /spec_kit:deep-research:review when you want severity-ranked highest-priority/high-priority/medium-priority review output across the 7 review dimensions. Review-mode durable state now lives under {spec_folder}/review/, and legacy scratch-based review packets migrate on resume.
- Switch concept-based code exploration to CocoIndex-first behavior when exact symbol names are unknown. Grep remains useful for exact matches, but semantic search is now the preferred path for intent-driven discovery.
- Update documentation habits to the new standards. New READMEs and major markdown rewrites should follow readme_creation.md and HVR v0.210, including the stronger structure and publish-readiness expectations.
- Align any local runtime assumptions with the current platform model. The active runtime set is now Copilot, Claude, Codex command-line interface, and Gemini command-line interface; the prior ChatGPT runtime should be considered removed.
- Run repository validation after upgrade. At minimum, verify tests, documentation-critical workflows, and any local memory or command automations you depend on before treating the environment as fully upgraded.
