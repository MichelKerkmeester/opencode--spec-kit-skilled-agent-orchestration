## [**3.0.0.0**] - 2026-03-27

**OpenCode v3.0.0.0 is the platform-scale release that turns the Spec Kit stack into a fully integrated operating environment for agentic development, hybrid memory retrieval, deep research, deep review, documentation production, and multi-runtime collaboration.** This release rolls up **57 included component releases** across the system-spec-kit, agents, commands, skills, research loops, semantic code search, documentation tooling, and CLI orchestration layers. It consolidates the complete `022-hybrid-rag-fusion` delivery track and lands the production-ready memory engine that the surrounding workflow, governance, and authoring systems now depend on.

At the platform level, this release establishes a coherent surface area of **12 agents** (2 built-in and 10 custom), **18 skills**, **22 commands**, **42 MCP tools** (33 memory, 7 code mode, 1 CocoIndex, 1 sequential thinking), **3 execution gates**, **4 runtimes**, **81 templates**, and **6 analysis lenses**. Instead of shipping isolated feature work, `v3.0.0.0` aligns these pieces into a single environment where planning, retrieval, implementation, review, handoff, and long-horizon memory all operate against the same governance model.

The centerpiece is the **33-tool memory engine**, implemented as a **7-layer** platform with a **7600-token budget**, **5 retrieval channels**, a **4-stage retrieval pipeline**, **3-tier fallback behavior**, **complexity-based routing**, and explicit save/query intelligence. Memory retrieval now fuses Vector Voyage embeddings, FTS5, BM25, causal graph traversal, and degree-aware graph signals through **Reciprocal Rank Fusion (RRF)** plus **8 post-fusion signals**, then reranks with a local GGUF model before applying context-pressure filtering. The engine also introduces lifecycle state tracking, importance-tiered decay rules, tenant-aware shared memory controls, session anchoring, and a broader evaluation harness spanning **25 database tables**, **26+ feature flags**, **12 evaluation metrics**, **110 ground-truth questions**, and **255 documented features across 21 categories**.

Around that core, the release expands OpenCode's operating model in three major directions. First, **`sk-deep-research`** advances from iterative research into a fuller quality system with dashboards, negative knowledge capture, novelty justification, per-iteration budgets, and then extends into **`:review` mode** with the new `@deep-review` agent and severity-ranked P0/P1/P2 findings across 7 review dimensions. Second, **documentation quality** receives a major uplift through **HVR v0.210**, `readme_creation.md`, 23 README rewrites, and two-tier technical voice guidance that makes the platform more legible for both newcomers and returning maintainers. Third, OpenCode broadens its external execution surface through **four CLI skills** and a normalized **four-runtime** model covering Copilot, Claude, Codex CLI, and Gemini CLI.

This is also a cleanup-and-alignment release. Dead code was removed, dormant paths were collapsed, governance text was reconciled across commands, skills, templates, agents, and `AGENTS.md`, and multiple ecosystem components were updated to reflect the new memory-first workflow. The result is a release that is larger than a feature drop: it is a full platform stabilization pass that makes hybrid retrieval, deep research, deep review, semantic code search, and documentation generation feel like one connected system rather than a loose set of tools.

## Highlights

### Platform Architecture and Surface Area

- **OpenCode becomes a full agentic environment instead of a collection of point tools.** The release standardizes a platform footprint of 12 agents, 18 skills, 22 commands, 42 MCP tools, 3 gates, 4 runtimes, 81 templates, and 6 analysis lenses so the same operational model spans planning, implementation, review, context recovery, and handoff.
- **The custom agent layer is now broad enough to cover the full development lifecycle.** In addition to the built-in roles, the environment now formally centers 10 custom agents: `@orchestrate`, `@context`, `@speckit`, `@debug`, `@deep-research`, `@deep-review`, `@review`, `@write`, `@handover`, and `@ultra-think`.
- **Gate-driven workflow is now a first-class platform contract.** The 3-gate model turns intent classification, skill routing, and spec-folder discipline into operational safety rails rather than optional conventions.
- **Runtime fragmentation was reduced to a consistent four-runtime model.** Copilot, Claude, Codex CLI, and Gemini CLI now share aligned routing concepts, agent directories, and governance language, while the removed ChatGPT runtime is no longer treated as an active execution target.
- **Template scale now matches platform scale.** With 81 templates available across spec workflows, documentation, commands, skills, and related assets, the platform has a reusable authoring layer rather than relying on one-off markdown or hand-built scaffolding.

### Spec Kit Memory Engine: From Storage to Retrieval Intelligence

- **The memory platform lands as a 33-tool, 7-layer system rather than a thin search feature.** Retrieval, saving, analysis, management, constitutional memory, shared memory, and session continuity now form a cohesive memory operating system inside `system-spec-kit`.
- **Five retrieval channels now participate in every serious memory lookup.** Vector Voyage `1024d` semantic search, SQLite FTS5, BM25 lexical ranking, causal graph traversal, and degree-oriented graph signals all contribute evidence into the same retrieval pipeline.
- **The query path is now a 4-stage pipeline.** Requests move through Gather, Score and Fuse, local GGUF reranking, and final filtering so the system can balance breadth, precision, and context pressure instead of relying on a single ranking step.
- **Fusion is no longer naive ranking.** The engine now uses RRF plus 8 post-fusion signals, allowing the system to reward cross-channel agreement, penalize interference, and adapt to session freshness and causal structure.
- **The post-fusion signal set is materially richer.** Co-activation contributes `+0.25`, interference can subtract `-0.08`, cold-start handling uses a 12-hour half-life, session recency is capped at `0.20`, causal reasoning reaches two hops, intent weighting spans 7 query intents, and channel minimum representation is guarded at `0.005`.
- **Temporal behavior is now explicit instead of hand-wavy.** FSRS-style decay is incorporated with the same research lineage validated on 100M+ Anki reviews, giving recency and reinforcement a principled scoring role.
- **Fallback behavior has depth.** A 3-tier fallback strategy ensures the system can degrade gracefully when semantic, graph, or fused retrieval is under-supported.
- **Routing now matches problem complexity.** Simple, Moderate, and Complex routing paths prevent small prompts from paying the full cost of deep retrieval while still allowing heavier prompts to unlock richer decomposition and evidence fusion.
- **Memory importance is governed, not flat.** The engine now distinguishes 6 importance tiers ranging from constitutional memories that carry a `3.0x` multiplier and never decay through to deprecated memories that remain available but intentionally de-emphasized.
- **Lifecycle state is now visible and actionable.** Memories move across 5 states from HOT through ARCHIVED, allowing the system to distinguish active working context from stable long-term reference or cold historical material.
- **Causality is modeled structurally.** Six causal relation types enable retrieval and analysis flows to connect events, decisions, outcomes, and supersessions instead of treating all memory edges as equivalent.

### Save Intelligence, Query Intelligence, and Session Behavior

- **Saving memory is now an evaluated decision rather than a write-everything policy.** PE-gated save intelligence classifies candidate writes into CREATE, REINFORCE, UPDATE, or SUPERSEDE operations and protects the store with 3 layers of gating.
- **Query execution now adapts to user intent.** Seven memory intents drive retrieval weighting, decomposition strategy, HyDE-style query expansion, and context-pressure management so different query types can surface different evidence blends.
- **Session memory behavior was tuned around practical continuity.** The session system now uses `0.85^distance` decay, targets roughly 50 percent deduplication, and hits about 93 percent ANCHOR capture to preserve continuity without flooding prompts with repeated context.
- **Context pressure is treated as a real budget, not an afterthought.** The engine can decide how much evidence to inject based on question complexity, token constraints, and channel agreement rather than blindly appending top-k results.
- **The result is a memory system that supports both recall and reasoning.** The platform can recover facts, preserve recent work, elevate governing rules, and pull causally related context into the same response path.

### Shared Memory, Governance, and Safety Controls

- **Shared memory is now a governed collaboration layer.** Spaces, 3 explicit roles, deny-by-default access, and a kill switch allow multi-actor sharing without flattening tenant boundaries or weakening local control.
- **Tenant-aware operations are part of the production path.** Checkpoints and related persistence flows are now scoped so that shared-memory administration does not leak into normal tenant behavior.
- **Governance text and operational rules were aligned around memory-first workflows.** Commands, agents, skill documentation, and AGENTS templates now describe the same memory system rather than overlapping but inconsistent snapshots of earlier implementations.
- **The platform now has enough instrumentation to support confidence in retrieval changes.** Twenty-five tables, 26+ feature flags, 12 evaluation metrics, and 110 ground-truth questions provide the scaffolding needed for iterative measurement rather than intuition-driven tuning.
- **The feature inventory was hardened at scale.** The memory and platform catalog reached 255 documented features across 21 categories, with alignment work pushing that inventory to a fully audited state instead of a drifting reference artifact.

### 022 Hybrid RAG Fusion Program

- **This release completes the 19-phase `022-hybrid-rag-fusion` program as the backbone of the v3 platform.** The work moved from prototype retrieval ideas to a productionized, evaluated, and governed memory engine with documentation, tests, and release packaging around it.
- **Hybrid retrieval is now the default mental model of the memory stack.** Vector, lexical, full-text, graph, and trigger-based evidence are treated as complementary signals and fused into one search path instead of existing as isolated experiments.
- **Ranking architecture consolidated around RRF.** Alternative fusion paths were evaluated, dead branches were removed, and the codebase kept the winning approach instead of preserving every experimental path.
- **Previously dormant or unwired work was either promoted or deleted.** Graph calibration profiles, temporal contiguity logic, and graduated consumption logging moved into meaningful release scope, while unused modules like `rsf-fusion.ts` and `retention.ts` were removed.
- **Quality debt was paid down during rollout.** Hydra review passes, ultra-think audits, recursive validation cleanup, and deep review cycles pushed the track from broad experimentation to a more disciplined production baseline.
- **Manual verification scaled with implementation scope.** The program reports 19 categories, 231 scenarios, and 272 scenario IDs passing, which matters because this release changes workflow behavior across retrieval, authoring, governance, and review.

### Autonomous Deep Research and Automated Code Review

- **`sk-deep-research` is now a dual-mode autonomous investigation system.** It operates in two modes: **research mode** for iterative technical investigation and **review mode** for automated code quality auditing — both built on the same LEAF (Loop, Externalize, Analyze, Finish) architecture with externalized JSONL state, convergence detection, and fresh context per iteration.
- **Research mode provides governed, observable deep investigation.** Persistent dashboards track iteration progress, quality guards block premature convergence, negative knowledge is captured as first-class output alongside positive findings, novelty justification is required per iteration, per-iteration token budgets prevent runaway cost, and diagnosis-led recovery handles stuck iterations. Research charter validation and source-hygiene confirmation ensure quality from the start.
- **Review mode turns the same iteration loop into a systematic code quality auditing framework.** Instead of ad hoc review prompts, the `:review` mode dispatches `@deep-review` agents that produce severity-ranked **P0/P1/P2 findings** with `file:line` evidence across **7 review dimensions**, running iteratively until the review converges or the iteration limit is reached.
- **The review taxonomy is intentionally simplified and canonical.** A contract manifest defines the authoritative 7-dimension set and scoring rules, smart router intent signals detect review-mode invocations automatically, and the testing playbook is aligned to the simplified taxonomy so review behavior is repeatable and testable.
- **The skill was designed and validated through its own methodology.** The v1.1.0 upgrade was driven by deep research across 7 auto-research repos, and the v1.2.0 review mode was designed via a 5-agent deep research session — the skill eating its own cooking.

### Semantic Code Search and Context Retrieval

- **CocoIndex becomes the platform's semantic code search layer.** The new `mcp-coco-index` skill gives OpenCode concept-based code search instead of forcing every exploration task through exact-text grep patterns.
- **The feature matured across multiple releases rather than a one-shot drop.** Versions `v1.0.0.0`, `v1.1.0.0`, and `v1.2.0.0` reflect iterative refinement of the skill and its place in the larger workflow.
- **Semantic retrieval is now wired into planning and creation paths.** Spec-kit planning flows and create workflows can reach for CocoIndex when developers know the intent of code they want but not the exact symbol names.
- **`@context` now reflects that retrieval strategy.** The agent is explicitly aligned around concept and intent discovery, making semantic search the first move for unfamiliar code exploration.

### Documentation Tooling, Creation Standards, and Publish-Ready Knowledge

- **`sk-doc` becomes a full documentation production system, not just a style checker.** The skill now governs README creation, feature catalog authoring, testing playbook generation, install guides, folder documentation, and general markdown quality through a unified DQI (Document Quality Index) scoring model with Structure 40%, Content 35%, and Style 25% weighting.
- **README creation is now a repeatable, standards-driven workflow.** The new `readme_creation.md` reference defines a 9-section matrix, section-specific writing guidance, type decision trees, and a pre-publish checklist so contributors follow a named authoring path instead of reverse-engineering expectations from existing files.
- **HVR `v0.210` upgrades documentation quality from cosmetic to structural.** The ruleset adds heading hierarchy enforcement, blockquote tagline conventions, TOC format rules, two-tier voice guidance, numbered Feature subsection patterns, divider usage rules, analogy placement controls, and 5 recommended table patterns (How This Compares, Key Statistics, Key Features, Before/After, Parameter). Scoring is weighted (Punctuation 15%, Structure 25%, Content 25%, Words 20%, Voice 15%) with explicit thresholds: 85+ publish-ready, 70-84 needs revision, below 70 failing.
- **23 READMEs were rewritten as a coordinated documentation program.** Nineteen skill READMEs and 4 command documents (converted from `.txt` to `.md`) now follow the `readme_creation.md` + `readme_template.md` + HVR v0.210 standard across 5 batches with **0 HVR hard blockers** across all files.
- **Feature catalog creation and maintenance is now a governed workflow.** The `/create:feature-catalog` command provides structured creation and update operations with auto-detection of component scope, category routing, and format validation against the established 255-entry catalog structure across 21 categories. Feature catalogs are no longer ad hoc inventories but template-driven packages.
- **Manual testing playbooks follow the same creation-standard pattern.** The `/create:testing-playbook` command scaffolds playbook packages with scenario structure, category assignment, and format compliance so testing documentation scales with the same rigor as feature documentation.
- **`sk-prompt-improver` adds structured prompt engineering to the documentation and authoring stack.** Seven proven frameworks (RCAF, COSTAR, RACE, CIDI, TIDD-EC, CRISPE, CRAFT), DEPTH thinking methodology, and CLEAR scoring across text modes give the platform a formal prompt quality layer rather than relying on unstructured prompt writing. Versions `v1.0.0` through `v1.2.0` landed across this release window.
- **Two-tier voice guidance improves accessibility across all documentation.** Docs now speak clearly to first-time readers in narrative sections while supporting deeper technical reference in tables and parameter listings, which is critical for the much larger memory, orchestration, and retrieval surface.
- **The Memory MCP README was reworked to match the system's true complexity.** It now explains all 33 tools across 7 layers, the 5-channel pipeline, scoring formulas, lifecycle states, and shared-memory governance instead of acting like a thin command index.
- **The `system-spec-kit` README was expanded to reflect the full platform.** Documentation now covers documentation levels, CORE+ADDENDUM v2.2 composition, validator flows, the 255-feature catalog, 22 scripts, 20 validation rules, and the full command and tooling surface.
- **The documentation program reports 0 hard blockers across all 23 files.** That matters because this release treats documentation as operational infrastructure — feature catalogs, testing playbooks, READMEs, and install guides are all governed artifacts, not post-hoc polish.

### CLI Skills and Multi-Runtime Execution

- **Four CLI skills meaningfully expand the environment's execution options.** `cli-copilot`, `cli-codex`, `cli-claude-code`, and `cli-gemini` turn external assistant CLIs into routable OpenCode capabilities.
- **`cli-copilot` adds breadth at the provider layer.** It ships support for 5 models across 3 providers and introduces an Autopilot execution mode.
- **`cli-codex` gives the environment a deeper OpenAI-native CLI path.** Dual-model operation, 7 profiles, sandbox awareness, and image-capable workflows broaden how OpenCode can delegate or inspect work.
- **`cli-claude-code` focuses on deep structured reasoning.** Its 3-model surface, extended thinking support, and 9-agent ecosystem make it a strong complement for planning, review, and collaborative execution.
- **`cli-gemini` adds search-grounded and long-context capabilities.** Google Search integration and 1M+ context support make it especially useful for large-context synthesis and externally grounded tasks.
- **The runtime model is now simpler to reason about.** Instead of proliferating ad hoc runtime language, the platform now clearly targets four active runtimes with explicit directory resolution rules.

### Alignment, Cleanup, and Ecosystem Consolidation

- **Multiple alignment tracks were completed so the platform description matches the shipped system.** Skills, commands, agents, AGENTS templates, and runtime docs were reconciled against the live memory architecture.
- **Template compliance strengthened cross-runtime consistency.** The two-layer template compliance system reduces drift between runtime-specific guidance and shared governance.
- **Infrastructure debt was reduced while the platform expanded.** npm audit issues were resolved, ChatGPT runtime references were removed, and dead code paths were deleted rather than preserved as legacy clutter.
- **`sk-prompt-improver` broadens the authoring and reasoning side of the stack.** Seven prompt frameworks, DEPTH processing, and CLEAR scoring extend the platform's ability to shape upstream prompt quality.
- **`mcp-clickup` starts the next wave of external integration.** Its inclusion marks early growth beyond internal tooling into a broader MCP-connected environment.

## Upgrade

1. Pull the latest repository state so `v3.0.0.0` lands as a consistent bundle instead of a partial cherry-pick of memory, docs, or CLI orchestration changes.
2. Restart the Spec Kit Memory MCP services so the hybrid retrieval stack, reranking path, shared-memory controls, tenant-aware checkpoint behavior, and new query/save intelligence are loaded together.
3. Reindex or refresh memory state if your local workflow depends on immediate retrieval parity, especially if you are resuming saved context or testing retrieval behavior after upgrade.
4. Review active feature flags before rollout. This release expands the flag surface to 26+ toggles and graduates more retrieval and observability behavior into normal operation.
5. Validate core memory workflows after startup: retrieval, save, session continuity, checkpoint access, and any shared-memory paths you rely on for collaboration.
6. Adopt the updated research and review workflows. Use the upgraded `sk-deep-research` flow for dashboarded research runs, and use `/spec_kit:deep-research:review` when you want severity-ranked P0/P1/P2 review output across the 7 review dimensions.
7. Switch concept-based code exploration to CocoIndex-first behavior when exact symbol names are unknown. Grep remains useful for exact matches, but semantic search is now the preferred path for intent-driven discovery.
8. Update documentation habits to the new standards. New READMEs and major markdown rewrites should follow `readme_creation.md` and HVR `v0.210`, including the stronger structure and publish-readiness expectations.
9. Align any local runtime assumptions with the current platform model. The active runtime set is now Copilot, Claude, Codex CLI, and Gemini CLI; the prior ChatGPT runtime should be considered removed.
10. Run repository validation after upgrade. At minimum, verify tests, documentation-critical workflows, and any local memory or command automations you depend on before treating the environment as fully upgraded.
