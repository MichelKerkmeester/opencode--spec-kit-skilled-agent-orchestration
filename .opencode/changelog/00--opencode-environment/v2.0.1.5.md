## [**2.0.1.5**] - 2026-02-14

This release updates the OpenCode environment with work on structured config files-first architecture refactoring, new architecture sections, bug fixes. The goal is to make the platform clearer, more reliable, and easier to use day to day.

### 1. Structured Config Files-First Architecture Refactoring
- Need: This part of the release improves how the environment saves, finds, and uses context.
- What changed: .md Files Refactored To Routing Documents: EXECUTION PROTOCOL banner → Phase 0 (@write verification) → Unified Setup Phase → structured config files loading → Reference context. Dual Structured Config Files Modes: Each command now has _confirm.yaml (interactive with checkpoints) and _auto.yaml (autonomous, no prompts). Agent.md Fully Rebuilt: Was a 670-line inline workflow with an orphaned structured config files file - now proper structured config files-first architecture with both mode files built from scratch.
- Why it matters: That matters because saved context is easier to find, easier to trust, and less likely to get in the way.

### 2. New Architecture Sections
- Need: This part of the release fixes rough edges and makes the environment more dependable.
- What changed: Workflow_enforcement: strict sequential mode with phase gates. Circuit_breaker: 3-failure threshold with closed/open/half-open states. Error_recovery: handlers for template_not_found, validation_failure, file_write_failure, user_rejects_approach.
- Why it matters: That matters because the environment is easier to maintain and less likely to break in everyday use.

### 3. Bug Fixes
- Need: This part of the release makes the written guidance easier to follow and easier to trust.
- What changed: Step Count Metadata Fixed In 3 YAMLs (create_install_guide, Create_skill_asset, Create_skill_reference): sequential_5_step → sequential_6_step.
- Why it matters: That matters because people can understand the system faster and use it more confidently.
