## Highlights

### YAML-First Architecture Refactoring

The create commands used an inverted workflow model ‚Äî the `.md` file ran the workflow inline and loaded YAML as a supplement. The spec_kit pattern is the opposite: `.md` is a router that loads YAML as the workflow engine. This release aligns all 6 create commands to the spec_kit pattern.

- **`.md` files refactored to routing documents**: EXECUTION PROTOCOL banner ‚Üí Phase 0 (@write verification) ‚Üí Unified Setup Phase ‚Üí YAML loading ‚Üí Reference context
- **Dual YAML modes**: Each command now has `_confirm.yaml` (interactive with checkpoints) and `_auto.yaml` (autonomous, no prompts)
- **`agent.md` fully rebuilt**: Was a 670-line inline workflow with an orphaned YAML file ‚Äî now proper YAML-first architecture with both mode files built from scratch

### New Architecture Sections

All 12 YAML files now include sections matching the spec_kit standard:

- `workflow_enforcement` ‚Äî strict sequential mode with phase gates
- `circuit_breaker` ‚Äî 3-failure threshold with closed/open/half-open states
- `error_recovery` ‚Äî handlers for template_not_found, validation_failure, file_write_failure, user_rejects_approach
- `quality_standards` ‚Äî DQI minimum scores and validation pipeline
- `termination` ‚Äî post-workflow next steps and conditions
- `rules` ‚Äî 6 ALWAYS / 6 NEVER rules adapted per command

### Bug Fixes

- Step count metadata fixed in 3 YAMLs (`create_install_guide`, `create_skill_asset`, `create_skill_reference`): `sequential_5_step` ‚Üí `sequential_6_step`
