This release fixes a silent routing bug in the skill advisor. 36 multi-character keys were sitting unreachable in `INTENT_BOOSTERS` because the tokenizer splits on whitespace and hyphens before the dict lookup runs. Those keys are now in `PHRASE_INTENT_BOOSTERS` where they actually fire, plus 15 new phrase routes cover previously-unrouted multi-word identifiers. Regression held at 100% top-1 accuracy through the whole migration.

> Spec folder:
> - `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-advisor-phrase-booster-tailoring/`

---

## What Changed

#### Multi-word intents now route correctly

The `INTENT_BOOSTERS` dict is keyed by single tokens — the tokenizer uses `re.findall(r'\b\w+\b', prompt_lower)` so whitespace and hyphens both split a key into parts before lookup. Over time 24 whitespace-separated keys (`"deep research"`, `"code audit"`, `"agent improvement"`, etc.) and 12 hyphenated keys (`"5-dimension"`, `"claude-code"`, `"copilot-cli"`, etc.) had been added. All 36 were dead code — they never matched anything.

Those 36 keys were deleted from `INTENT_BOOSTERS`. 33 landed in `PHRASE_INTENT_BOOSTERS` (which uses `phrase in prompt_lower` substring matching and correctly handles multi-character strings); 3 were already duplicated there at higher weights, so the `INTENT` copies were pure cleanup.

&nbsp;

#### `5-dimension agent scoring` now routes to the right skill

The query `"5-dimension agent scoring"` used to match `sk-improve-prompt` at 0.77 confidence via a weak single-token hit on `scoring`. That was a misroute — the intended target is `sk-improve-agent`. The hyphenated `INTENT` key was unreachable and nothing in `PHRASE` covered the phrase.

Two new `PHRASE` entries (`"5-dimension"` and `"5-dimension agent scoring"`) route both phrases to `sk-improve-agent`. Post-fix confidence for the full query is 0.95 and the top skill is correct.

&nbsp;

#### `code audit` schema conflict resolved

`"code audit"` existed in both dicts pointing to different skills — `sk-deep-review` in `INTENT_BOOSTERS`, `sk-code-review` in `PHRASE_INTENT_BOOSTERS`. Since the `PHRASE` path fires first and `sk-code-review` is the right baseline target for a one-pass audit (not an iterative loop), the `INTENT` pointer was pure dead conflict. It was deleted. Runtime behavior does not change — `sk-code-review` was already the live answer.

&nbsp;

#### 15 new phrase routes for under-covered identifiers

Multi-word phrases that users naturally type but had no phrase route were added across `system-spec-kit`, `sk-code-opencode`, `sk-code-full-stack`, `sk-code-web`, `mcp-code-mode`, and `sk-code-review`. Total `PHRASE_INTENT_BOOSTERS` entries went from 167 to 222.

&nbsp;

#### 8 new regression fixtures cover the new routes

The fixture suite grew from 44 to 52 cases. The new `P1-PHRASE-NNN` rows exercise the `code audit` phrase route, the hyphenated `5-dimension` fix, and representative new phrase additions. All 52 cases pass.

&nbsp;

#### Anti-pattern comment added

A comment block near `PHRASE_INTENT_BOOSTERS` documents the split-on-tokenizer-boundary rule: `INTENT_BOOSTERS` is for single tokens only; anything with whitespace or a hyphen goes in `PHRASE_INTENT_BOOSTERS`. This is meant to keep future contributors from re-introducing the same bug.

&nbsp;

#### Final invariant

`INTENT_BOOSTERS` now contains only single-token keys. The dict schema and the tokenizer behavior are aligned for the first time since the dict was introduced.

## Upgrade

No migration needed. This is a pure data-layer change — dict entries moved between two dicts the advisor was already consuming. Confidence formula, scoring logic, tokenizer, and graph-boost pathways are untouched. Any consumer that imports `INTENT_BOOSTERS` directly will find 36 fewer keys; if you relied on one of those keys to influence routing, it was dead code and had no effect — use `PHRASE_INTENT_BOOSTERS` instead.
