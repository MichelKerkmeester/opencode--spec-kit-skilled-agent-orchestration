This patch release closes the remaining 042 closing-audit issues that still affected `sk-deep-review`. It focuses on places where the workflow promised one thing but the runtime did another, so the review loop, reducer, and docs now tell the same story.

> Spec folder: `.opencode/specs/skilled-agent-orchestration/042-sk-deep-research-review-improvement-2/` (Level 3)

## Bug Fixes

Five gaps where the workflow advertised one thing and the runtime did another. Each fix is a behavior change: the documented promise and the shipped code now agree.

#### Claim adjudication was documented as a hard stop gate but never checked

Every new required-severity finding (a P0 blocker or a P1 must-fix) is supposed to carry a structured "claim-adjudication packet" - a short JSON block that names the claim, cites the file-and-line evidence, lists what counter-evidence the reviewer looked for, and records the final severity plus a confidence number. The documentation said the review loop would refuse to stop until every new P0 or P1 had a valid packet. In practice the review loop never read the flag that tracked this. Reviewers could produce a packet that was missing fields or missing entirely, the workflow would flag it internally, and then stop-checking would run as if nothing was wrong. A loop that was supposed to be release-gated could finish and render a PASS verdict on top of incomplete adjudication evidence. Both review workflows (interactive and autonomous) now run a universal claim-adjudication pre-check as the first step of every stop-check cycle. If the most recent adjudication result was a failure, stop is blocked immediately with a clear recovery message asking the reviewer to rewrite the missing packets. The legal-stop decision tree also gained a dedicated gate that surfaces the adjudication status alongside the other five gates. The result is persisted to the state log as a typed event so the next iteration can read it back. Every new P0 or P1 finding must now ship with a complete packet; anything less keeps the loop running.

&nbsp;

#### Two review sessions could overwrite each other's graph findings

The skill keeps a small SQLite cache of "coverage graph" nodes and edges - dimensions, files, findings, evidence, and their relationships - so future iterations can reason about what has already been covered. The cache was shared across all review sessions in a spec folder. The primary key on each row was the bare logical id (`d-correctness`, `f-001`, etc.). If two reviewers ran concurrent sessions against the same spec folder and both happened to emit a node with the same logical id, the second write overwrote the first. Filtered reads still worked, but the underlying rows were corrupted. The earlier phase-008 session-isolation test only checked filtered reads over disjoint fixtures, so it never caught the collision path. The cache schema is now on version 2. Node and edge tables use a composite primary key built from spec folder, loop type, session id, and logical id. The database migrates by dropping the old tables and recreating the new ones on first open - the cache is a derived artifact, not durable state, so a drop-and-recreate is safe. The session-isolation test now has a dedicated "shared-ID collisions" block that upserts identical node and edge ids across two sessions and asserts both rows survive with their own content. A future phase that introduces real cross-session analytics on top of this cache will find it correct from day one instead of walking into a silent-overwrite trap.

&nbsp;

#### Two lifecycle modes were advertised but never actually wired up

When an operator re-entered a review session, the interactive workflow asked which lifecycle mode to use: `resume` (continue where you left off), `restart` (archive the old run and start a new generation), `fork` (start a sibling run history), or `completed-continue` (reopen a finished run after snapshotting its synthesis). The first two were real. The last two were documented in several places but had no actual workflow wiring - selecting them did nothing different, and the state log silently kept the old session id. Anyone who believed they had branched a review was quietly overwriting their previous work. `fork` and `completed-continue` are now explicitly marked as deferred (reserved for a future release) everywhere the skill mentions them: `SKILL.md`, `README.md`, both loop-protocol references, both command workflows, the main agent definition, and the runtime mirrors across all three CLI runtimes. The interactive gate now offers three choices - `resume`, `restart`, or `cancel`. Both paths append a typed event to the state log that carries the full run history field set (session id, parent session id, run history mode, continued-from-run, generation, archive path, timestamp). A parity test locks the new option list so the retraction cannot silently drift back.

&nbsp;

#### The canonical review agent wrote iterations the reducer could not parse

The reducer that assembles the review registry, dashboard, and strategy parses each iteration markdown file with a strict set of section names (`## Focus`, `## Findings`, `## Ruled Out`, and so on) and a strict finding bullet shape. The main authoring template in the agent definition was completely different - it used a different top-level heading, different section names, and different finding formatting. A reviewer following the agent definition to the letter would produce an iteration file the reducer quietly dropped every finding from. The loop would keep running but the registry would go empty. The main authoring skeleton in the agent definition was rewritten so every heading, subsection, and bullet shape matches the reducer parser exactly. A note at the top of the section warns future editors that the skeleton is parser-sensitive and points at the exact reducer file and line the parser lives in. The same rewrite adds a ready-to-copy claim-adjudication packet example so reviewers do not need to go look up the typed-packet schema in a second file.

&nbsp;

#### The stop-gate contract reference described a data shape that never existed

The convergence reference described the shared stop expected behavior as a nested `legalStop` record - a single blob wrapping the blocker list, the per-gate results, and a replay snapshot. The shipped workflow never wrote that shape. What it actually writes on a blocked stop is a top-level `blocked_stop` event with a flat gate-results map and a recovery-strategy hint. Anyone building tooling on top of the documented shape would get zero matches. The reference also mis-labelled the third signal in the three-signal convergence vote as a "Novelty Ratio" - the actual signal is dimension coverage (every configured review dimension reached by at least one iteration, held stable for one extra pass). The shared-stop-expected behavior section now shows the real `blocked_stop` event shape verbatim, lists the six gate names the workflow actually emits, and includes a worked example with non-trivial values. The composite-convergence section corrects the third signal back to dimension coverage, renames the subsection accordingly, and clarifies that the severity-weighted new-findings ratio feeds the rolling-average and noise-floor signals rather than casting its own stop vote.

&nbsp;

#### Dashboard risks hid every problem that was not a P0

The reducer-owned "Active Risks" section at the bottom of every review dashboard was built to surface release-blocking debt. The check it actually ran was "is there a P0?" - nothing else. If a session had active P1 findings, or a failed claim-adjudication gate, or a blocked-stop event the legal-stop tree was rejecting, the dashboard still reported "None active beyond normal review uncertainty." Operators scanning the dashboard before clicking "synthesize" had no visual signal that the session was still carrying required work. The Active Risks section now walks the state log and surfaces every kind of non-zero debt: latest iteration error, active P0 count, active P1 count, the most recent claim-adjudication failure with the list of findings that are missing packets, the most recent blocked-stop event with its blocker list and recovery hint, and (only when nothing higher-severity is present) the P2 tech-debt count. A dashboard that says "None active" now means the session is genuinely clean instead of just "has no P0."

## Schema Changes

| Change              | Details                                                                      |
| ------------------- | ---------------------------------------------------------------------------- |
| Coverage graph DB   | `deep-loop-graph.sqlite` schema version 1 → 2                                |
| New primary key     | `(spec_folder, loop_type, session_id, id)` on `coverage_nodes` + `coverage_edges` |
| Migration strategy  | Drop and recreate on first open (dev-only cache, no durable state to preserve) |
| Bare-id API removed | `getNode(id)` / `getEdge(id)` / `getEdgesFrom(sourceId)` / `getEdgesTo(targetId)` / `updateEdge(id, updates)` / `deleteNode(id)` / `deleteEdge(id)` now take a namespace argument as their first parameter |

Existing review packets and state logs remain readable. The state-log format did not change. Consumers who called the bare-id helpers directly from outside this module break at the type level, which is intentional - the new signatures make it impossible to accidentally reference a node or edge without scoping it to a session.

## Upgrade

No state-log migration is required. The graph database is a derived cache and rebuilds itself from iteration events, so its version bump is transparent.

Reviewers must now emit a typed claim-adjudication packet for every new P0 or P1 finding. A missing or malformed packet vetoes stop on the next convergence check, and the dashboard Active Risks section will name the findings that are missing packets. Operators who were planning to use the `fork` or `completed-continue` lifecycle modes should archive the prior `review/` folder manually or pick `restart`, which does both in one step. Full closing-audit provenance lives at `.opencode/specs/skilled-agent-orchestration/042-sk-deep-research-review-improvement-2/review/review-report.md`.
