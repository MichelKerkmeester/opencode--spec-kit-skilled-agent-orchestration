## [**3.1.4.0**] - 2026-03-30

Ten search engine optimizations based on a 10-iteration deep research investigation of the hybrid RAG fusion pipeline. Each finding was implemented by a dedicated GPT-5.4 agent and targets a specific bottleneck or quality gap in the retrieval system. Together they improve ranking precision, recall coverage, result diversity, and operational observability.

## What Changed

### Ranking & Fusion

- **RRF fusion constant tuned for corpus size** -- The reciprocal rank fusion smoothing constant was set to k=60, which is designed for corpora with 10,000+ documents. At that value, the score difference between rank 1 and rank 10 is only 0.002 -- essentially flat. With a 999-memory database, k=40 doubles the rank separation to 0.004, making the top results meaningfully distinguishable from lower-ranked ones. (`shared/algorithms/rrf-fusion.ts`: `DEFAULT_K` 60 → 40)

- **Token budget raised to prevent result truncation** -- The search response token budget was 1500 tokens, which often truncated results to just 1 out of 7+ matches. Raised to 2500 to accommodate the p95 memory content length, so users see 3-5 results instead of 1. (`architecture/layer-definitions.ts`: L2 budget 1500 → 2500)

### Recall & Coverage

- **Deprecated memories now consistently excluded across all channels** -- Vector search excluded the 538 deprecated-tier memories, but FTS5 text search did not -- creating a channel asymmetry where deprecated content leaked back into results through the lexical path. FTS5 now applies the same exclusion filter. (`sqlite-fts.ts`: added `importance_tier != 'deprecated'` to WHERE clause)

- **Embedding expansion activates for more queries** -- The R12 embedding expansion feature (which broadens search by finding related terms from the embedding space) was suppressed for queries with 3 or fewer tokens. This blocked useful expansion for technical queries like "mcp runtime hardening". The gate now allows 3-token queries through. (`embedding-expansion.ts`: `isExpansionActive` threshold ≤3 → ≤2)

- **Multi-word queries now match exact phrases** -- Searching for "spec kit" previously only matched documents containing "spec" OR "kit" individually. Now it also generates an FTS5 phrase query `"spec kit"` to boost exact-phrase matches. (`bm25-index.ts`: `normalizeLexicalQueryTokens` adds phrase variant for 2+ token queries)

### Reranking & Diversity

- **Reranker provenance now tracked in metadata** -- Stage 3 previously reported `rerankApplied: true/false` without distinguishing which reranker ran (cross-encoder, local GGUF, or fallback sort). A new `rerankProvider` field records the actual path taken, making search traces and ablation studies trustworthy. (`stage3-rerank.ts`: added `rerankProvider` field)

### Data Quality

- **Co-activation now handles legacy memory format** -- The co-activation system (which boosts memories related to top results) expected `related_memories` in object format `[{id: 39}]` but the database stores them as scalar arrays `[39]`. The parser silently failed, meaning co-activation never contributed to search results. It now accepts both formats. (`co-activation.ts`: `parseRelatedMemoryReferences` handles scalar and object arrays)

- **Quality score backfill function available** -- 520 of 1002 memories (52%) have a quality score of 0.0 because they were ingested via index scan without the quality gate. A new `computeBackfillQualityScore()` function scores based on content length, trigger phrases, title presence, and importance tier. Available for maintenance scripts. (`save-quality-gate.ts`: new exported function)

- **Chunk children now linked to parents** -- When memories are chunked into smaller pieces, the child chunks were not recording their parent memory ID, breaking lineage traversal and making chunks appear as orphans. The chunking orchestrator now propagates `parent_id`. (`chunking-orchestrator.ts`, `vector-index-mutations.ts`)

### Observability

- **Embedding cache now reports hit/miss statistics** -- The embedding cache had no counters, making it impossible to measure cache effectiveness. Added `cacheHits`, `cacheMisses`, and `getCacheStats()` returning `{hits, misses, hitRate}`. (`embedding-cache.ts`)

### Ultra-Think Review P1 Fixes (post-optimization audit)

- **Shadow evaluation replay still used minState:'WARM'** -- The interactive handlers were fixed, but the shadow evaluation replay path still hardcoded `minState: 'WARM'`, meaning shadow evaluation, adaptive feedback, and replay-based monitoring were evaluating against empty result sets. Removed.

- **Co-activation similarity scale mismatch** -- Scalar `related_memories` entries were assigned similarity 0.5, but the co-activation system uses a 0-100 scale (divides by 100 for boosts). Scalar entries produced `0.5/100 = 0.005` boost — two orders of magnitude weaker than intended. Changed to 50 (matching the 0-100 scale).

- **Quality score backfill now runs at search time** -- `computeBackfillQualityScore()` was added but never called. Now wired into Stage 1: when a candidate has `quality_score === 0`, the backfill function computes a score based on content length, trigger phrases, title, and importance tier.

- **Tool schema tests reconciled** -- The schema simplification (removing oneOf/not for Claude compatibility) broke 2 contract tests. Updated assertions to match the current schema.

## Upgrade

No migration required. Restart the MCP server to pick up the changes. Zero-score memories are now automatically scored at search time via `computeBackfillQualityScore()`.
