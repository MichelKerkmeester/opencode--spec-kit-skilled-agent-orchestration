The memory search system now quietly watches how searches are used -- which results get shown, which get cited, whether follow-up searches rephrase the same question, and whether other tools fire right after a search. These observations are collected but do not change how results are ranked. The goal is to build up a reliable dataset first, then use it to improve ranking in a future release. The goal is to make the workflow easier to follow and easier to trust.

## What Changed

---

### Search

- **Search usage tracking** -- The system had no way to learn from how searches were actually used. It now records five types of usage signals after every search: which results were shown, which were delivered with full content (cited), whether the next search rephrased or repeated the same question, and whether a non-search tool ran shortly after. These signals are stored for later analysis but do not affect ranking today.

- **Rephrased and repeated search detection** -- When someone searches, then searches again with slightly different wording, the system now recognises that as a rephrased query. When they repeat nearly the same search, it recognises that as a retry. The detection works by comparing how much word overlap exists between consecutive searches in the same session.

- **Follow-on tool tracking after search** -- When someone runs a search and then immediately uses a different tool (like checking memory stats), that sequence is now recorded. The challenge was that many tools do not carry a session identifier, so the system remembers the most recent session and uses it as a fallback. This means even tools that know nothing about sessions still get linked back to the search that preceded them.

- **Observation-only design** -- All five signal types are collected in the background without affecting what users see. Search results are ranked exactly the same way as before. This is deliberate: the signals need enough volume and validation before they can safely influence ranking without introducing bias.

---

### Testing

- **Signal path test coverage** -- A dedicated test suite verifies the detection thresholds, deduplication timing, expiry rules, and the full cycle of citation, follow-on tracking, and rephrased query detection within a single session.

- **Performance overhead confirmed** -- Benchmarks show the signal collection adds 0.03ms on average per search. Well within the 5ms budget, so search speed is unaffected.
