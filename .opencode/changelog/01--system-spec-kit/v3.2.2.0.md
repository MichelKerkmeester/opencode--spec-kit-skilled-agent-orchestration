## [**3.2.2.0**] - 2026-04-03

Adds implicit feedback signal collection to the memory search pipeline. Five event types -- search result views, citation tracking, query reformulations, near-identical requeries, and follow-on tool use -- are now logged to the `feedback_events` ledger after every search interaction. The signals are shadow-only: they record how searches are used but do not influence ranking yet. This lets the system build a quality baseline before graduating signals into the scoring pipeline.

> Spec folder: `.opencode/specs/02--system-spec-kit/023-hybrid-rag-fusion-refinement/014-feedback-signal-pipeline` (Level 2)

## What Changed

### New Features

- **Query flow tracker** -- New module that maintains a bounded per-session query history and detects patterns across consecutive searches. When a follow-up query overlaps 30-80% with a prior query (measured by Jaccard token similarity), it emits `query_reformulated`. When overlap exceeds 80%, it emits `same_topic_requery`. Session history enforces a 50-entry cap and 10-minute TTL.

- **Result citation tracking** -- When a search runs with `includeContent: true` and returns results, the search handler now emits `result_cited` events for each shown memory ID. This tracks which memories are actually delivered with full content to the caller.

- **Sticky session fallback for follow-on tools** -- Tools like `memory_stats` have no `sessionId` parameter, so they could not correlate with a prior search. The server now stores the session ID from any tool call that provides one into a module-level `lastKnownSessionId` variable. When a non-search tool fires, the dispatcher falls back to that stored ID to emit `follow_on_tool_use` events.

- **Shadow-only signal design** -- All five event types write to the `feedback_events` table but do not affect search ranking. `feedbackSignalsApplied` remains `off` in search traces. This is intentional -- the system collects signal quality data first, validates it offline, then graduates to ranking influence in a future release.

### Testing

- **Dedicated tracker test suite** -- New `query-flow-tracker.vitest.ts` covers Jaccard thresholds, dedup window, TTL expiry, result_cited emission, follow_on_tool_use correlation window, and a DB-level packet flow test spanning citation, follow-on, and reformulation in a single session.

- **Sub-5ms signal overhead** -- Benchmark confirms the feedback signal path adds 0.03ms average and 0.07ms p95 overhead to searches. Well within the documented <5ms async-overhead budget.
