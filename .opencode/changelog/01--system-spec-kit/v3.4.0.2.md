# v3.4.0.2 — Canonical Save, Actually

This patch closes the 28 findings surfaced by the deep-review of v3.4.0.1 plus 14 more exposed by a follow-up 7-iteration meta-research. The previous version made failure modes loud; this one fixes the largest one of all — it turned out the canonical save path wasn't writing any metadata at all. Not "eventually stale." Zero. Every default `/memory:save` since the guard shipped had been a no-op on `description.json.lastUpdated` and `graph-metadata.json.derived.*`.

The fix is two lines in `scripts/core/workflow.ts`. A dead-code guard at `:1259` hardcoded `ctxFileWritten = false`, making the per-folder description block unreachable. A plan-only gate at `:1333` kept `refreshGraphMetadata` disabled on the default code path. Once both came out, every save since has written fresh timestamps across 38 folders in the 026 tree. The "0 of 16 sibling folders have fresh `lastUpdated`" symptom that the deep-review flagged as drift was actually the deterministic output of code that never wrote.

Three follow-on surfaces: code-graph sibling handlers now emit the same readiness vocabulary as `query.ts` instead of just 1 of 7 being hardened; Copilot finally has compact-cache parity with Claude and Gemini; and the MCP transport layer now carries caller identity through `AsyncLocalStorage`, closing a session-resume cross-UID leakage path without a 30-file signature refactor.

> Scope: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/017-review-findings-remediation/` — review-driven remediation. 25 commits to main across ~45 files + test files.

## Canonical Save, Actually

The symptom that the deep-review flagged was "0 of 16 sibling folders have fresh `description.json.lastUpdated`" and labeled it drift. It wasn't drift. Three conditions composed into a deterministic no-op:

```ts
// scripts/core/workflow.ts:1259
const ctxFileWritten = false;  // stub

// scripts/core/workflow.ts:1261-1331
if (ctxFileWritten) {
  // ... 70 LOC of description.json + memorySequence tracking ...
  // never reached.
}

// scripts/core/workflow.ts:1333
const shouldRunExplicitSaveFollowUps = options.plannerMode === 'full-auto';
if (shouldRunExplicitSaveFollowUps) {
  await refreshGraphMetadata(/* ... */);
  // ...
}

// scripts/memory/generate-context.ts:415
plannerMode: 'plan-only',  // the default
```

Three code paths that individually looked defensive. Together, they meant the default `/memory:save` path wrote zero metadata. Since the guards shipped.

The fix removes both. `ctxFileWritten` is now `true`; the block runs on every save. `shouldRunExplicitSaveFollowUps` is unconditional; `refreshGraphMetadata` runs on every save. A new `sequenceSnapshot.lastUpdated = new Date().toISOString()` line inside the retry loop actually writes the timestamp that the block was always supposed to.

The fix cascaded immediately. During implementation of the rest of Phase 017, `/memory:save` invocations refreshed 38 folder metadata files automatically — the visible proof that canonical save now writes what it says it writes.

&nbsp;

### The meta-irony

Fixing the canonical save path meant we couldn't verify the fix by running `/memory:save` on the 017 spec folder itself — until the fix landed, 017's own `description.json.lastUpdated` wouldn't update. The verification test had to prove the logic worked independent of any live folder's state.

`scripts/tests/workflow-canonical-save-metadata.vitest.ts` builds a synthetic spec-folder fixture inside a temp dir with `spec.md` + `description.json` stubs. It invokes the canonical save directly with `plannerMode: 'plan-only'` (the default), asserts `lastUpdated` gets written, asserts successive saves produce monotonically advancing timestamps, asserts `graph-metadata.json.derived.last_save_at` advances in plan-only mode. Only once the fixture test passed did we run the live `/memory:save`.

## Code-Graph Readiness Propagation

`handlers/code-graph/query.ts` had been hardened in v3.4.0.1 with `canonicalReadiness`, `trustState`, and `lastPersistedAt` emission. The deep-review flagged only `context.ts` as unhardened. A segment-2 token-grep sweep found the asymmetry was 6:1 — every other sibling in the `code-graph/` directory was also missing the readiness vocabulary.

`lib/code-graph/readiness-contract.ts` extracts four helpers:

```ts
export function canonicalReadinessFromFreshness(freshness: ReadyResult['freshness']): StructuralReadiness;
export function queryTrustStateFromFreshness(freshness: ReadyResult['freshness']): SharedPayloadTrustState;
export function buildQueryGraphMetadata(readiness: ReadyResult): Record<string, unknown> | undefined;
export function buildReadinessBlock(readiness: ReadyResult): ReadinessBlock;
```

`query.ts` refactored to consume the shared module. `scan.ts` and `status.ts` now emit the full 3-field readiness block. The CCC trio (`ccc-status`, `ccc-reindex`, `ccc-feedback`) emits a stub `trustState: 'unavailable'` + `reason: 'readiness_not_applicable'` where full readiness semantics don't apply — backward-compat with consumers keying on `trustState !== undefined` preserved.

No new enum. The existing canonical `SharedPayloadTrustState` at `lib/context/shared-payload.ts:36` already had 8 values (`live | cached | stale | absent | unavailable | imported | rebuilt | rehomed`); handler emissions use the 4-value subset that applies.

## Copilot Runtime Parity

Claude and Gemini compact cycles read `payloadContract?.provenance.trustState` at session-prime entry. Copilot didn't — because `hooks/copilot/` only had `session-prime.ts` and nothing else. A compound-hypothesis audit (H-56-4) confirmed the triple gap: no `compact-cache.ts`, no trustState read, and — by extension — no retry-exhaustion signal if Copilot-driven sessions accumulated silent enrichment failures across compaction.

`hooks/copilot/compact-cache.ts` mirrors the Gemini shape: writes `trustState: 'cached'` via the shared provenance module. `session-prime.ts` extended to read the contract. 5 new Vitest cases parallel to the Claude and Gemini compact-cycle suites.

Before Copilot-primary autonomous iteration launches (Phase 018+), this lane had to land — per the operator-feedback concurrency cap (3 parallel dispatches), accumulated per-session silent failures would have been consequential.

## Hooks Shared Provenance

`wrapRecoveredCompactPayload` and `sanitizeRecoveredPayload` had been living in `hooks/claude/shared.ts`. Gemini's `shared.ts:7` imported them via `import { hookLog } from '../claude/shared.js'` — which looked like a narrow import but actually pulled in the whole Claude module, including `RECOVERED_TRANSCRIPT_STRIP_PATTERNS`. If Copilot also needed these helpers, it'd be the third runtime doing that.

`hooks/shared-provenance.ts` now owns the three functions. Claude re-exports for backward-compat. Gemini switched to the direct import. Copilot imports it first-hand. Three runtimes, one canonical source, zero transitive coupling.

## MCP Caller Context

Session-resume auth needed caller identity. The MCP transport layer didn't expose any. Adding it via explicit argument threading would have touched ~30 handler signatures across the server surface.

`lib/context/caller-context.ts` adds `AsyncLocalStorage`-based propagation:

```ts
export interface MCPCallerContext {
  readonly sessionId: string | null;
  readonly transport: 'stdio' | 'sse' | 'ws' | 'unknown';
  readonly connectedAt: string;
  readonly callerPid?: number;
  readonly metadata: Record<string, unknown>;
}

export function runWithCallerContext<T>(ctx: MCPCallerContext, fn: () => T): T;
export function getCallerContext(): MCPCallerContext | null;
export function requireCallerContext(): MCPCallerContext;
```

`context-server.ts` wraps each `CallToolRequestSchema` dispatch with `runWithCallerContext`. Handlers that need caller identity call `getCallerContext()` at their boundary. Everything else stays untouched.

The test suite covers 12 cases — propagation across `await`, `setTimeout`, `Promise.all`, nested runs, concurrent requests. The nested case is important: if a handler internally runs another operation inside `runWithCallerContext`, the inner context correctly shadows the outer for the duration of the inner run.

## Session-Resume Auth

`handleSessionResume` used to accept `args.sessionId` as trusted input. A same-UID attacker with access to the process environment could pass a different session's ID and receive that session's cached summary — the classic same-machine cross-session leak.

The fix reads `getCallerContext()?.sessionId` at handler entry. If `args.sessionId` is provided and doesn't match, the handler throws. If it matches, or if either is absent, the original flow runs.

For production deployments that already had workflows with mismatched session IDs, the rollout is staged:

```bash
export MCP_SESSION_RESUME_AUTH_MODE=permissive
```

Permissive mode logs the mismatch and continues. Default is strict. The flag is intended for canary verification before flipping the default elsewhere.

## Unicode Sanitization

`shared/gate-3-classifier.ts:normalizePrompt` used to lowercase and collapse whitespace. That's enough for ASCII but not for homoglyph attacks. `"fileе write"` (with Cyrillic `е` U+0435 instead of Latin `e`) would pass the Gate 3 trigger check — the classifier was looking for `"file write"` literally. `"f\u00ADile write"` (with a soft hyphen between `f` and `i`) would also pass.

`normalizePrompt` now runs `.normalize('NFKC')` first, then strips `[\u00AD\u200B-\u200F\uFEFF]`. `sanitizeRecoveredPayload` applies the same normalization to each line before the system-role-prefix regex runs, so `"SYST\u0395M:"` (Greek epsilon U+0395) gets caught by the same pattern as `"SYSTEM:"`.

New test file at `scripts/tests/gate-3-classifier.vitest.ts` — 5 unicode cases: Cyrillic `е`, soft hyphen, zero-width space, Greek `Ε`, combined homoglyph + zero-width.

## Scope Normalizer Canonical

Four files had their own copy of `normalizeScopeValue(value: unknown): string | null`:

- `handlers/save/reconsolidation-bridge.ts:228-234`
- `lib/storage/lineage-state.ts:198-204`
- `handlers/save/types.ts:348-352`
- `lib/validation/preflight.ts:440-444`

Plus a fifth canonical version in `lib/governance/scope-governance.ts` that returned `string | undefined`. Consumers had null-specific narrowing (`if (x === null)`, `x === null ? ...`), so the canonical had to match — the new `normalizeScopeValue` export returns `string | null` to preserve caller semantics. `normalizeScopeContext` (the structural-object variant using `normalizeId`) stayed on `string | undefined`.

All 4 duplicates deleted. 15 fixture tests cover `undefined`, `null`, `""`, whitespace, non-string, simple string, trimmed string. A lint rule at `scripts/rules/check-normalizer-lint.sh` rejects new `normalizeScope*` or `getOptionalString` declarations outside the canonical file — `--exclude-dir` filters cover `dist/` and `node_modules/`.

## Retry Budget

`partial_causal_link_unresolved` outcomes used to schedule `runEnrichmentBackfill` with no attempt counter. If the underlying condition was structurally non-retryable (malformed causal-link reference, missing predecessor), the backfill would loop indefinitely, pinning CPU on the same failure mode.

`lib/enrichment/retry-budget.ts` tracks attempts keyed on `(memoryId, step, reason)`:

```ts
export function shouldRetry(memoryId: number, step: string, reason: string): boolean;
export function recordFailure(memoryId: number, step: string, reason: string): RetryBudgetEntry;
export function clearBudget(memoryId?: number): void;
```

`post-insert.ts` calls `shouldRetry` before scheduling the backfill and `recordFailure` on each failure. After 3 attempts on the same `(memoryId, step, reason)` tuple, the budget denies further retries and emits a structured warning. `clearBudget(memoryId)` fires on successful enrichment completion.

The budget is in-memory per process. If durability across restarts matters, that's a follow-up.

## Validators and Lints

Three new entries in `validate.sh --strict`:

- **EVIDENCE marker lint.** The `[EVIDENCE: ...]` format used to get regex-checked, but the regex false-positived on any marker whose description contained parens. `scripts/validation/evidence-marker-audit.ts` is a bracket-depth state machine that tracks `[` / `]` depth separately from `(` / `)` (content-legal). Skips fenced code blocks and inline backtick spans. The audit run found 1,962 markers across 16 folders, 210 of them actually malformed (closed with `)` instead of `]`) and rewrapped them in place. Future drift caught in strict mode.

- **Continuity freshness.** `scripts/validation/continuity-freshness.ts` compares `_memory.continuity.last_updated_at` in `implementation-summary.md` against `graph-metadata.json.derived.last_save_at`. Warns when continuity lags by >10 minutes. Non-strict warn; strict fail.

- **Normalizer lint** (mentioned above). Rejects new local `normalizeScope*` helpers outside the canonical file.

## P2 Maintainability

Three refactors landed under a lighter gate (`×3` deep-review instead of `×7`) because they're code-shape improvements, not behavior changes:

### Exhaustiveness checks

`lib/utils/exhaustiveness.ts` exports `assertNever(x: never, context?: string): never`. Applied to 8 typed unions (`OnIndexSkipReason`, `EnrichmentStepStatus`, `EnrichmentSkipReason`, `EnrichmentFailureReason`, `ConflictAbortStatus`, `HookStateLoadFailureReason`, `SharedPayloadTrustState`, `TriggerCategory`) at switch defaults. `post-insert.ts:302` gained a `satisfies Record<OnIndexSkipReason, EnrichmentSkipReason>` clause so future variant additions fail at compile time.

### Post-insert god function

`runPostInsertEnrichment` was 243 lines and contained five near-identical `try { ... } catch { ... }` enrichment blocks. Extracted to a single `runEnrichmentStep<T>(name, isEnabled, runner, successHandler, options)` helper. The orchestrator function reduced to 32 lines; each step is one-line invocation. Existing tests pass unchanged.

### Reconsolidation transaction helper

`executeConflict` had two near-identical transaction blocks (deprecate-path and content-update-path) sharing the same preconditions. Extracted to `runAtomicReconsolidationTxn(db, memory, snapshot, mode, operations)`. Both paths now share precondition checks and CAS guards; mode-specific `UPDATE` statements pass in as operations. No behavior change.

### Typed YAML predicates

`spec_kit_complete_confirm.yaml:1099` had a prose `when:` string that looked like a BooleanExpr but wasn't — it was timing narration ("Immediately after the canonical spec document is refreshed on disk"). The fix moved it to `after:` (the canonical timing field). Wave D agent caught the distinction during implementation; the catalog note clarifies that prose timing and typed predicates are different slots.
