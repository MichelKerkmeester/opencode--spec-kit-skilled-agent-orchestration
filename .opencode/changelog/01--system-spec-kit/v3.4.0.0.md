## [**3.4.0.0**] - 2026-04-12

Spec 026 turns the graph-and-context optimization program into a shipped continuity-first release for `system-spec-kit`. The release replaces legacy packet memory markdown with canonical spec-doc continuity, adds packet-level graph metadata as first-class indexed state, hardens save/read/runtime contracts, removes dead shared-memory infrastructure, and closes a long memory-quality/documentation cleanup train that spanned research, runtime, docs, and review.

> Active child train: `001-research-graph-context-systems` -> `002-implement-cache-warning-hooks`, `003-memory-quality-issues`, `004-agent-execution-guardrails`, `005-code-graph-upgrades`, `006-canonical-continuity-refactor`

## Canonical Continuity Model

### Three-rung resume ladder replaces packet-local memory markdown

**Problem:** Packet recovery still depended on a mix of legacy `memory/*.md` assumptions, runtime-specific bootstrap behavior, and loosely coupled save/read paths. After long sessions, compaction, or runtime switches, the system did not have one canonical packet-first recovery model.

**Fix:** Gate D retargeted the reader side around a shared `resumeLadder` contract and a doc-first recovery order: `handover.md` -> `_memory.continuity` in `implementation-summary.md` -> canonical spec docs. The active reader surfaces now align on that ladder across `memory-search`, `memory-context`, `session-resume`, `session-bootstrap`, `memory-index-discovery`, `memory-triggers`, operator docs, and `/spec_kit:resume`.

---

### Canonical saves now write into packet docs instead of standalone memory files

**Problem:** The canonical routing and validation layers existed, but the real atomic save path still treated them as metadata and could continue writing the source memory file instead of promoting the routed spec doc.

**Fix:** Gate C wired `atomicSaveMemory()` through `buildCanonicalAtomicPreparedSave()`, `atomic-index-memory.ts`, `create-record.ts`, and the target-doc identity path. Canonical saves now validate the routed destination, promote the real packet doc, persist the routed target identity in the index, and return the actual spec-doc path that changed.

---

### Phase-aware content routing and anchor-aware merge operations became real runtime behavior

**Problem:** Saved content still lacked a trustworthy packet-doc destination model. Decision content, narrative updates, task mutations, and handover state could not all be routed and merged with packet-structure awareness.

**Fix:** Gate C shipped the 3-tier `contentRouter`, the five canonical merge modes, and the ADR insertion path in `anchor-merge-operation.ts`. Narrative progress routes into `implementation-summary.md::what-built`, delivery notes route into `implementation-summary.md::how-delivered`, decisions can append table rows or synthesize `adr-NNN` inserts into `decision-record.md`, handover state targets `handover.md`, and metadata-only updates stay bounded to the thin continuity layer.

---

### Thin `_memory.continuity` records replaced heavyweight continuity artifacts

**Problem:** Small continuity updates still assumed a full script round-trip and the old standalone memory-artifact model, which made quick packet-local state updates heavier than they needed to be.

**Fix:** Phase 018 introduced a thin `_memory.continuity` record hosted directly in `implementation-summary.md` frontmatter. Gate C tightened the fallback continuity record so it satisfies the validator without reviving observation-era payload sprawl, and Gate D made that thin record a first-class resume rung instead of a secondary hint.

---

### ADR-004 allows direct continuity edits while keeping indexed saves on the script path

**Problem:** The old save rule treated every continuity change like a full generated-memory workflow, even after canonical continuity moved into spec docs.

**Fix:** ADR-004 formalized the split: AI may directly edit `_memory.continuity` in canonical spec docs for session continuity updates, while `generate-context.js` remains the indexed save path for DB writes, embeddings, `description.json` refresh, and full canonical-save workflows.

---

### Canonical writes now fail closed behind five new spec-doc guardrails

**Problem:** Routing content into packet docs without structure checks risked malformed frontmatter, illegal merges, missing packet surfaces, anchor contamination, or post-save drift.

**Fix:** Gate C added the five-rule `spec-doc-structure` bridge and exposed it through `validate.sh`: `FRONTMATTER_MEMORY_BLOCK`, `MERGE_LEGALITY`, `SPEC_DOC_SUFFICIENCY`, `CROSS_ANCHOR_CONTAMINATION`, and `POST_SAVE_FINGERPRINT`. Canonical saves now validate before indexing and restore pre-save content on failure instead of silently accepting corrupt packet state.

---

### Stop-hook continuity is now producer-first and autosave-backed

**Problem:** The earlier stop-hook lane stopped at metadata capture and replay fencing, leaving the continuity producer seam under-described and easy to misread as additive-only.

**Fix:** Packet `002-implement-cache-warning-hooks` hardened the producer boundary around `processStopHook()` and `runContextAutosave()`. Stop-path metadata now persists transcript identity, cache token carry-forward state, bounded transcript references, and replay-safe verification harnesses, while the live docs were later corrected to describe autosave as the intended default producer path.

## Graph Metadata System

### `graph-metadata.json` became the packet-level graph contract

**Problem:** Removing legacy packet `memory/` saves also removed the packet-scoped graph and ranking hints that the retrieval stack had been getting indirectly from those artifacts.

**Fix:** Packet `011-spec-folder-graph-metadata` introduced one root-level `graph-metadata.json` file per spec folder. The new file carries packet identity plus a merge-safe `manual` relationship section and a regenerated `derived` section for trigger phrases, status, topics, key files, entities, timestamps, and source documents, all indexed as `document_type='graph_metadata'`.

---

### Graph metadata refresh is now part of canonical save

**Problem:** Packet graph state would drift unless operators remembered to run a separate metadata or backfill step after canonical saves.

**Fix:** Canonical save now refreshes `graph-metadata.json` during the normal save workflow. The parser preserves `manual.*`, regenerates `derived.*`, writes atomically through temp-file + rename, and lets packet relationships project into `causal_edges` through the existing graph processor.

---

### Schema, parser, and backfill tooling now cover the whole packet tree

**Problem:** There was no typed contract or rollout tool for packet graph metadata, which made whole-tree adoption brittle and hard to verify.

**Fix:** The MCP server now ships `graph-metadata-schema.ts` and `graph-metadata-parser.ts`, while scripts add `backfill-graph-metadata.ts` for dry runs and full apply across `.opencode/specs/`. Packet creation scaffolds the file, validation checks presence with warning-first rollout semantics, and the implementation rollout backfilled `515/515` packet roots at the time of the apply run.

---

### Derived entity extraction is cleaner and more bounded

**Problem:** Packet-derived entities and file references needed normalization so graph metadata could stay compact and index-friendly.

**Fix:** `graph-metadata-parser.ts` now normalizes extracted entity names by collapsing embedded newlines to spaces, ignores empty values, rejects names longer than 80 characters, deduplicates against existing keys, and caps the emitted entity list to a bounded set. That keeps packet metadata additive without turning it into an unbounded dump.

## Memory Save Quality

### The D1-D8 memory-quality train was closed as a shipped remediation packet

**Problem:** The earlier memory-save pipeline still had boundary bugs across truncation, metadata ownership, sanitization precedence, continuation heuristics, and operational tail behavior, and the parent packet had drifted into a stale research-only rollup.

**Fix:** Packet `003-memory-quality-issues` was converted into a true implementation rollup with ten child phases. The train closed the D1-D8 defect set, synchronized the packet-local closeout evidence, and kept later follow-on work narrowly focused on compact-wrapper and canonical-doc-ownership seams instead of reopening the original five-phase repair lane.

---

### Fast-path structured saves no longer abort on real orchestrator payloads

**Problem:** `generate-context.js` could still reject the real orchestrator payload shape with `INSUFFICIENT_CONTEXT_ABORT` because the JSON fast path failed to coerce string-array entries and dropped mixed enrichments.

**Fix:** Phase `008-input-normalizer-fastpath-fix` taught `normalizeInputData()` to coerce plain-string `user_prompts`, `observations`, and `recent_context` entries and merge slow-path enrichments like `sessionSummary`, `keyDecisions`, `nextSteps`, `filesModified`, `toolCalls`, and `exchanges` into mixed fast-path payloads.

---

### Post-save render output now matches the compact-wrapper contract

**Problem:** Fresh saves still rendered with noisy titles, missing canonical packet-doc discovery, wrong file counts, prose-derived trigger junk, weak lineage, self-referential `parent_spec`, and one overloaded quality field.

**Fix:** Phase `009-post-save-render-fixes` corrected the render helpers around the existing wrapper contract. Fresh saves now emit clean dashboard titles, canonical packet-doc source lists, truthful counts, authored trigger phrases, deduplicated anchored evidence, honest phase/status, auto-linked lineage, non-self-referential `parent_spec`, and separate `render_quality_score` vs `input_completeness_score` surfaces.

---

### Structured saves now preserve authored metadata, manual triggers, and conservative continuation logic

**Problem:** The runtime still dropped explicit `title`, `description`, and `causalLinks`, over-matched V8/V12 validation heuristics, and let continuation/linker behavior drift between runtime and reviewer surfaces.

**Fix:** Phase `010-memory-save-heuristic-calibration` widened the structured save contract so authored fields survive normalization and workflow merge, preserved manual DR trigger phrases through the runtime pipeline, narrowed V8 false positives, normalized V12 slug/prose matching, aligned D5 continuation behavior across linker and reviewer, and verified the full path with a real compiled-dist save against the parent `026` packet.

## Shared Memory Removal

### Shared-memory runtime files and tool surfaces were hard-deleted

**Problem:** The repo still carried a dead shared-memory feature surface after the continuity model had moved away from shared spaces and shared-scope governance.

**Fix:** Packet `010-remove-shared-memory` deleted `mcp_server/handlers/shared-memory.ts`, `mcp_server/lib/collab/shared-spaces.ts`, and the surrounding lifecycle/tool registration surface. The shared-memory MCP tools and contracts disappeared from the active runtime, and the dedicated command/docs/catalog/playbook/test surfaces were trimmed to match the live post-delete system.

---

### Governance flags, HYDRA aliases, and archival ballast were removed with the feature

**Problem:** Even after the main delete, active code still carried fake control surfaces for shared-memory governance, HYDRA compatibility, and archive-manager behavior.

**Fix:** The follow-up cleanup removed `SPECKIT_MEMORY_SCOPE_ENFORCEMENT`, `SPECKIT_MEMORY_GOVERNANCE_GUARDRAILS`, the `SPECKIT_HYDRA_*` alias family, the `SPECKIT_ARCHIVAL` runtime path, and the old archival-manager bootstrap/module/tests. The live roadmap flags are now the only remaining checked runtime switches in that neighborhood.

---

### Schema compatibility remains only as an explicit migration exception

**Problem:** Dropping shared-space columns from SQLite in place would have turned a code cleanup into a risky live-schema rewrite.

**Fix:** `shared_space_id` columns remain in `vector-index-schema.ts` as a documented compatibility exception for backward-safe DB migration, but the active runtime no longer reads, writes, filters, or exposes shared-memory behavior through them.

## MCP Configuration Standardization

### Five checked-in MCP configs now share one minimal memory env contract

**Problem:** The Public repo still carried drift across `.mcp.json`, `.claude/mcp.json`, `.vscode/mcp.json`, `.gemini/settings.json`, and `opencode.json`, including redundant `MEMORY_DB_PATH` wiring and feature-flag noise that the runtime no longer needed.

**Fix:** Packet `012-mcp-config-and-feature-flag-cleanup` locked those five checked-in configs to the same minimal Spec Kit Memory env block. `MEMORY_DB_PATH` is gone, redundant checked-in `SPECKIT_*` flags stay removed, and the `_NOTE_*` guidance is now aligned across all five runtime surfaces.

---

### Default-on feature behavior moved fully into code

**Problem:** The repo still looked like it needed checked-in flags for features that the runtime already treated as enabled by default.

**Fix:** The cleanup formalized the actual default behavior: `rollout-policy.ts` treats undefined or empty flags as enabled, with only explicit `false` or `0` disabling a feature. The configs now document those switches as opt-out-only notes instead of pretending they must be pinned in versioned JSON.

---

### Gemini and other checked-in runtime surfaces now use the same profile-aware path story

**Problem:** The checked-in config surfaces had drifted toward inconsistent path guidance, including a stale absolute-path story in `.gemini/settings.json`.

**Fix:** The five-config cleanup normalized the checked-in Public MCP surfaces around repo-relative, profile-aware behavior and aligned their operator notes with the post-018 continuity and provider model.

## Search & Ranking Upgrades

### Reranking and embedding selection are now provider-aware instead of hardcoded

**Problem:** The search stack still looked and behaved like a fixed local-profile system even though the runtime already needed cloud-aware reranking and dimension-safe embedding behavior.

**Fix:** The checked-in config/docs/runtime now align on provider-aware search defaults. Voyage is documented as the preferred cloud path with `rerank-2.5` and `voyage-4` embeddings at 1024 dimensions, while the no-cloud-key default remains local HF at 768 dimensions. The runtime also derives `EMBEDDING_DIM` from the active provider profile instead of pinning a universal 768-dimension assumption.

---

### Causal edge identity is now anchor-aware

**Problem:** Packet continuity edges could collapse distinct relationships when they shared the same source and target IDs but pointed at different anchor regions inside those docs.

**Fix:** Gate B advanced the live schema to version 26 and added `source_anchor` / `target_anchor` columns plus anchor indexes on `causal_edges`. The runtime now preserves anchor-aware identity and query-plan usage instead of flattening those relationships into one doc-level edge.

---

### Code graph outputs became more precise about provenance, blast radius, and hot files

**Problem:** The adopt-now code-graph lane still had correctness holes around traversal depth, multi-file union, and the provenance vocabulary exposed to graph-owned outputs.

**Fix:** Packet `005-code-graph-upgrades` added `DetectorProvenance`, graph-local `graphEdgeEnrichment`, bounded blast-radius traversal, explicit multi-file union mode, and advisory `hotFileBreadcrumb` output. The release also kept those enrichments scoped to graph-owned outputs instead of reopening `session_resume` or `session_bootstrap` authority surfaces.
