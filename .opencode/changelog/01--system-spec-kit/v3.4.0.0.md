## [**3.4.0.0**] - 2026-04-12

This release rewires how the system remembers things between sessions. Instead of scattering knowledge across standalone memory files, the system now stores continuity directly inside the spec documents it already tracks — making recovery faster, more reliable, and simpler to understand. A new graph metadata layer gives every spec folder a machine-readable identity, and the old shared-memory feature is fully removed.

The work spans 21 phases under spec 026 (6 root phases + 15 child phases inside the canonical continuity refactor), backed by 40 research iterations, 80 deep-review iterations, and two remediation passes.

> Spec folder: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/` (Level 3, 21 phases)

## How AI Sessions Recover Context

The biggest change in this release is how the system picks up where it left off. Previously, session recovery relied on standalone memory files that could go stale, get lost, or conflict with the actual spec documents. Now the system follows a simple 3-step ladder.

---

### Session recovery follows a clear priority order

**Problem:** When resuming work, the system had no single source of truth. It checked memory files, spec docs, and handover notes inconsistently, often loading outdated context or missing recent progress.

**Fix:** A new resume ladder checks three sources in order: (1) `handover.md` if one exists, (2) the `_memory.continuity` block inside `implementation-summary.md`, (3) the canonical spec docs themselves. The first source that has useful content wins. There is no archived-memory fallback — the old memory files are gone.

---

### Quick continuity updates no longer require a full save workflow

**Problem:** Even small session-state updates (like "I just finished task 3") required running the full `generate-context.js` pipeline with database writes and embedding generation.

**Fix:** AI can now directly edit the `_memory.continuity` YAML block in `implementation-summary.md` for quick session notes. The full save pipeline is still used for indexed saves that need database visibility, but lightweight continuity updates are instant.

---

### Saves go into the right spec document automatically

**Problem:** When saving context, the system wrote standalone memory files. These files were disconnected from the spec documents they described, creating a parallel truth that drifted over time.

**Fix:** A content router now classifies incoming saves and directs them to the correct spec document: decisions go to `decision-record.md`, progress updates go to `implementation-summary.md`, handover notes go to `handover.md`. Five merge modes handle different content types (append, replace, table-row insert, ADR insert, metadata-only).

---

### Session stop hooks now auto-save context

**Problem:** When a session ended, accumulated context was lost unless the user manually saved. The stop hook existed but only captured metadata.

**Fix:** The stop hook now runs `runContextAutosave()` by default, persisting transcript identity, session metrics, and continuity state. This is the primary continuity producer — not a side effect.

---

### New validation rules catch bad saves before they land

**Problem:** With content now routing into spec documents, malformed saves could corrupt packet state — bad frontmatter, illegal merges, anchor contamination.

**Fix:** Five new validation rules run before every save: frontmatter structure check, merge legality check, spec-doc sufficiency check, cross-anchor contamination check, and post-save fingerprint verification. Failed saves roll back to pre-save content.

## Graph Metadata

Every spec folder now carries a `graph-metadata.json` file that describes what the folder contains, what it depends on, and what it relates to. This makes the spec tree navigable by tools, not just by humans reading markdown.

---

### Each spec folder now has a machine-readable identity

**Problem:** The only way to understand relationships between spec folders was to read markdown and follow prose references. Tools couldn't traverse the spec tree programmatically.

**Fix:** A new `graph-metadata.json` file in each spec folder carries two sections: a `manual` section for human-authored relationships (depends_on, supersedes, related_to) and a `derived` section that is auto-refreshed with trigger phrases, key files, entities, status, and timestamps.

---

### Canonical saves automatically refresh graph metadata

**Problem:** Even with graph-metadata files in place, they would go stale unless someone remembered to regenerate them after changes.

**Fix:** The `generate-context.js` save pipeline now refreshes `graph-metadata.json` as part of every canonical save. The parser preserves manual relationships while regenerating derived fields, using atomic writes (temp-file + rename) to prevent corruption.

---

### A schema, parser, and backfill tool cover the entire spec tree

**Problem:** Rolling out graph metadata across hundreds of spec folders needed typed tooling, not manual file creation.

**Fix:** Three new modules: `graph-metadata-schema.ts` (Zod schema + TypeScript types), `graph-metadata-parser.ts` (read/validate/merge), and `backfill-graph-metadata.ts` (retroactive population with dry-run support). The initial backfill populated 515 spec-folder roots.

---

### Entity extraction is cleaner and bounded

**Problem:** Early entity derivation produced garbage entries — multi-line names, shell commands parsed as file paths, unbounded lists.

**Fix:** The parser now strips embedded newlines from entity names, rejects entries longer than 80 characters, deduplicates against existing keys, and caps the output. Shell commands and non-file-path strings are filtered out.

## Shared Memory Removal

The shared-memory feature — which allowed cross-workspace memory sharing — has been fully removed. It was never used in production and added complexity to every save/search/config path.

---

### All shared-memory code and tools deleted

**Problem:** The codebase carried a complete shared-memory subsystem (handlers, storage layer, governance flags, MCP tools) that was never activated but touched every save and search path.

**Fix:** Hard-deleted: `handlers/shared-memory.ts`, `lib/collab/shared-spaces.ts`, `lib/cognitive/archival-manager.ts`, `SHARED_MEMORY_DATABASE.md`. Removed 4 MCP tools (`shared_memory_enable`, `shared_memory_status`, `shared_space_upsert`, `shared_space_membership_set`). Removed all HYDRA aliases and governance flags (`SPECKIT_MEMORY_SCOPE_ENFORCEMENT`, `SPECKIT_MEMORY_GOVERNANCE_GUARDRAILS`, `SPECKIT_HYDRA_*`, `SPECKIT_ARCHIVAL`).

---

### Database columns kept for migration safety

**Problem:** Dropping `shared_space_id` columns from the SQLite schema would turn a code cleanup into a risky live-database migration.

**Fix:** The `shared_space_id` columns remain in the schema as a documented migration-safe exception. The runtime no longer reads, writes, or filters on them. They exist only so existing databases open without errors.

---

## MCP Configuration Cleanup

All five checked-in MCP configuration files are now consistent and portable.

---

### Configs standardized across all runtimes

**Problem:** The five MCP config files (`.mcp.json`, `.claude/mcp.json`, `.vscode/mcp.json`, `.gemini/settings.json`, `opencode.json`) had drifted apart — different env blocks, leftover `MEMORY_DB_PATH` entries, redundant feature flags, and one hardcoded absolute path.

**Fix:** All five configs now share the same minimal Spec Kit Memory env block. `MEMORY_DB_PATH` is removed (auto-derived from the provider profile). Redundant feature flags are removed (the runtime treats undefined flags as enabled by default). The `.gemini/settings.json` absolute path was replaced with a repo-relative convention. Consistent `_NOTE_*` guidance blocks appear in all five files.

## Search and Ranking Upgrades

---

### Reranking model and embedding provider are now configurable

**Problem:** The search stack was hardcoded for local-only operation with a fixed embedding dimension.

**Fix:** The system now auto-detects the embedding provider: if `VOYAGE_API_KEY` is present, it uses Voyage-4 at 1024 dimensions; without a key, it falls back to local HuggingFace at 768 dimensions. The rerank model was upgraded from `rerank-2` (deprecated) to `rerank-2.5`, with `ms-marco-MiniLM-L-6-v2` as the local fallback. Embedding dimension is derived from the provider profile, not hardcoded.

---

### Causal edges can now distinguish different anchor regions

**Problem:** When two spec documents referenced each other at different anchor points, the system collapsed those into a single edge, losing the distinction.

**Fix:** Added `source_anchor` and `target_anchor` columns to `causal_edges` with dedicated indexes. The upsert logic now treats edges with different anchors as distinct relationships.

## Memory Save Quality

A train of 12 child phases under `003-memory-quality-issues` fixed bugs across the entire save pipeline.

---

### Structured saves no longer reject real orchestrator payloads

**Problem:** The JSON fast path in `generate-context.js` rejected payloads with mixed string arrays and enrichment objects, causing `INSUFFICIENT_CONTEXT_ABORT` on legitimate saves.

**Fix:** The input normalizer now coerces string-array entries and merges slow-path enrichments (sessionSummary, keyDecisions, nextSteps, filesModified) into the fast-path payload.

---

### Post-save output matches the dashboard contract

**Problem:** After saving, the render output showed noisy titles, wrong file counts, self-referential parent links, and junk trigger phrases derived from prose content.

**Fix:** Fresh saves now emit clean dashboard titles, truthful file counts, authored trigger phrases, deduplicated evidence, honest phase/status, and separate quality scores for render quality vs input completeness.

---

### Authored metadata survives the save pipeline

**Problem:** Explicit `title`, `description`, and `causalLinks` provided by the caller were being overwritten or dropped during normalization.

**Fix:** The structured save contract now preserves authored fields through normalization and workflow merge. Manual trigger phrases survive the pipeline. Continuation and linker behavior is consistent between runtime and reviewer surfaces.

## Documentation and Quality

---

### Architecture docs rewritten to match current reality

**Problem:** The `ARCHITECTURE.md` module tree and lib READMEs still described the pre-refactor layout, including modules that had been renamed, moved, or deleted.

**Fix:** `ARCHITECTURE.md` was fully rewritten. `lib/graph/README.md` and `lib/config/README.md` were updated to include graph-metadata files and reflect the post-memory continuity model. Dead imports, dead locals, and raw stdout logging were removed from production modules.

---

### All operator docs aligned on the same continuity contract

**Problem:** `CLAUDE.md`, `AGENTS.md`, `SKILL.md`, and the `/memory:save` command described continuity editing differently — some said "any spec doc", others said "implementation-summary.md", and one still forbade direct edits entirely.

**Fix:** All operator-facing docs now agree: `_memory.continuity` edits target `implementation-summary.md` specifically. The resume ladder reads from that file. Full saves go through `generate-context.js`. The contradiction between SKILL.md (which forbade edits) and AGENTS.md/CLAUDE.md (which allowed them) is resolved.

---

### Manual testing playbooks converted to prose format

**Problem:** Playbook scenarios used 9-column execution tables that were unreadable and impossible to follow as test instructions.

**Fix:** All scenarios across sk-deep-review, sk-deep-research, and system-spec-kit playbooks were converted to headed prose sections: Prompt, Commands, Expected, Evidence, Pass/Fail, Failure Triage. The root playbook index retains its table format (as a reference inventory), but individual scenarios use the prose format.

---

### Playbook execution results report honest coverage

**Problem:** The "full playbook execution" phase claimed complete execution but actually ran 24 of 297 scenarios through the direct-handler automation boundary, with the rest classified as UNAUTOMATABLE.

**Fix:** The phase was reframed as "coverage accounting and partial execution." Results honestly report 22 PASS, 1 PARTIAL, 1 FAIL, and 273 UNAUTOMATABLE for the current automation boundary. Previously permissive PASS entries with unmatched required signals were reclassified.

---

### CLI orchestration gotcha documented

**Problem:** When launching `codex exec` from another AI's Bash tool using `&`, the orchestrator reports instant completion while the codex process is still running — leading to empty output files and hallucinated results.

**Fix:** Added a "False-Completion Trap" section to `cli-codex/references/integration_patterns.md` explaining the root cause and correct pattern: use the orchestrator's own `run_in_background` mechanism instead of shell-level `&`.

The remaining items were fixed in a second remediation pass.
