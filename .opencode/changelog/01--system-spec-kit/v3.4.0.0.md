This release rewires how the system remembers things between sessions. Instead of scattering knowledge across standalone memory files, the system now stores continuity directly inside the spec documents it already tracks  -  making recovery faster, more reliable, and simpler to understand. 

A new graph metadata layer gives every spec folder a machine-readable identity, and the old shared-memory feature is fully removed. Alongside that core work, the deep-loop skills became auditable and scale-ready, and every CLI skill now runs a pre-dispatch prompt-quality check before calling external models.

## New Memory System

Standalone memory files are gone. Continuity now lives inside the spec documents the system already tracks, so there is no separate memory surface to keep in sync. The old `memory/*.md` corpus is retired. Saves route directly into `decision-record.md`, `implementation-summary.md`, and `handover.md`. Session resume reads from those same files, validation protects them on every write, and stop hooks persist continuity automatically.

#### Memory files deprecated, continuity moves into spec documents

The old system wrote standalone memory files that lived beside the spec documents they described. Those files could go stale, get lost, or contradict the spec docs they shadowed. The entire `memory/*.md` surface is retired in this release. Continuity state now lives inside spec documents: the `_memory.continuity` YAML block inside `implementation-summary.md` holds session progress, `decision-record.md` holds decisions, and `handover.md` holds resume hints. No parallel memory tree to drift against.

&nbsp;

#### Saves route into the right spec document automatically

Before, every save produced a new memory file disconnected from the spec docs. A content router now classifies incoming saves and writes them into the correct canonical document: decisions go to `decision-record.md`, progress updates go to `implementation-summary.md`, and handover notes go to `handover.md`. Five merge modes handle different content types (append, replace, table-row insert, ADR insert, metadata-only), so the same router covers both small updates and full sections.

&nbsp;

#### Session resume follows a clear priority order

When resuming work, the system had no single source of truth. It checked memory files, spec docs, and handover notes inconsistently, often loading outdated context or missing recent progress. A new resume ladder checks three sources in order: (1) `handover.md` if one exists, (2) the `_memory.continuity` block inside `implementation-summary.md`, (3) the canonical spec docs themselves. The first source with useful content wins. There is no archived-memory fallback because the old memory corpus is gone.

&nbsp;

#### Quick continuity updates skip the full save pipeline

Even small session-state notes like "I just finished task 3" used to require running `generate-context.js` end-to-end with database writes and embedding generation. AI can now edit the `_memory.continuity` YAML block directly in `implementation-summary.md` for quick session notes. The full pipeline is reserved for indexed saves that need database visibility. Lightweight continuity updates are instant.

&nbsp;

#### Session stop hooks auto-save on exit

When a session ended, accumulated context was lost unless the user saved manually. The stop hook existed but only captured metadata. The stop hook now runs `runContextAutosave()` by default, persisting bounded transcript identity, additive `producerMetadata`, and cache-token carry-forward. Replay is sandboxed with out-of-bound write detection and idempotency checks that prove double-replay does not duplicate rows. This is the primary continuity producer  -  not a side effect. Broader SessionStart fast paths, `UserPromptSubmit` hooks, and settings-file mutation were explicitly deferred.

&nbsp;

#### Validation rules protect spec documents on every save

With content now routing into spec documents, malformed saves could corrupt packet state  -  bad frontmatter, illegal merges, anchor contamination. Five new validation rules run before every save: frontmatter structure check, merge legality check, spec-doc sufficiency check, cross-anchor contamination check, and post-save fingerprint verification. Failed saves roll back to pre-save content.

## Graph Metadata

Every spec folder now carries a `graph-metadata.json` file that describes what the folder contains, what it depends on, and what it relates to. This makes the spec tree navigable by tools, not just by humans reading markdown.

#### Each spec folder now has a machine-readable identity

The only way to understand relationships between spec folders was to read markdown and follow prose references. Tools couldn't traverse the spec tree programmatically. A new `graph-metadata.json` file in each spec folder carries two sections: a `manual` section for human-authored relationships (depends_on, supersedes, related_to) and a `derived` section that is auto-refreshed with trigger phrases, key files, entities, status, and timestamps.

&nbsp;

#### Canonical saves automatically refresh graph metadata

Even with graph-metadata files in place, they would go stale unless someone remembered to regenerate them after changes. The `generate-context.js` save pipeline now refreshes `graph-metadata.json` as part of every canonical save. The parser preserves manual relationships while regenerating derived fields, using atomic writes (temp-file + rename) to prevent corruption.

&nbsp;

#### A schema, parser, and backfill tool cover the entire spec tree

Rolling out graph metadata across hundreds of spec folders needed typed tooling, not manual file creation. Three new modules: `graph-metadata-schema.ts` (Zod schema + TypeScript types), `graph-metadata-parser.ts` (read/validate/merge), and `backfill-graph-metadata.ts` (retroactive population with dry-run support). The initial backfill populated 515 spec-folder roots.

&nbsp;

#### Entity extraction is cleaner and bounded

Early entity derivation produced garbage entries  -  multi-line names, shell commands parsed as file paths, unbounded lists. The parser now strips embedded newlines from entity names, rejects entries longer than 80 characters, deduplicates against existing keys, and caps the output. Shell commands and non-file-path strings are filtered out.

&nbsp;

#### Graph metadata is searchable, ranked, and projected into causal edges

Graph metadata existed on disk but the runtime could not find packets through it  -  search never matched `graph_metadata`, ranking ignored packet signals, and manually authored `depends_on` / `supersedes` / `related_to` relationships never reached the edge store. `graph-metadata.json` is now indexed alongside spec documents with a `graph_metadata` content type, participates in packet-oriented ranking, and projects the `manual` relationship section into `causal_edges` on save. Searches return packet roots directly, ranking respects graph-derived signals, and graph queries can traverse authored relationships without a second lookup.

&nbsp;

#### Graph metadata status is derived, not guessed

The `derived.status` field was inferred from loose heuristics (file presence, timestamps) that disagreed with the packet's actual state. Status derivation now keys off `implementation-summary.md` existence plus checklist completion signals. `key_files` are sanitized before storage (no shell fragments, no multi-line paths). Basename-only entity duplicates are rejected, and the backfill/default traversal contract falls back cleanly when metadata is absent.

&nbsp;

#### All shared-memory code and tools deleted

The codebase carried a complete shared-memory subsystem (handlers, storage layer, governance flags, MCP tools) that was never activated but touched every save and search path. Hard-deleted: `handlers/shared-memory.ts`, `lib/collab/shared-spaces.ts`, `lib/cognitive/archival-manager.ts`, `SHARED_MEMORY_DATABASE.md`. Removed 4 MCP tools (`shared_memory_enable`, `shared_memory_status`, `shared_space_upsert`, `shared_space_membership_set`). Removed all HYDRA aliases and governance flags (`SPECKIT_MEMORY_SCOPE_ENFORCEMENT`, `SPECKIT_MEMORY_GOVERNANCE_GUARDRAILS`, `SPECKIT_HYDRA_*`, `SPECKIT_ARCHIVAL`). Live request contracts dropped the `sharedSpaceId` parameter, and shared-space filtering was removed from the retrieval, trigger, save, checkpoint, and preflight code paths so no hot path still checks scope. The `shared_space_id` columns were also dropped from the SQLite schema. Every startup the runtime calls `dropDeprecatedSharedSpaceColumn()`, an idempotent helper that issues `ALTER TABLE memory_index DROP COLUMN shared_space_id` when the column is still present and returns without doing anything when it is already gone or when the SQLite build does not support DROP COLUMN. The runtime never reads or writes the column, so installs that cannot execute DROP COLUMN keep an unused column on disk without side effects.

## Search and Ranking Upgrades

#### Reranking model and embedding provider are now configurable

The search stack was hardcoded for local-only operation with a fixed embedding dimension. The system now auto-detects the embedding provider: if `VOYAGE_API_KEY` is present, it uses Voyage-4 at 1024 dimensions; without a key, it falls back to local HuggingFace at 768 dimensions. The rerank model was upgraded from `rerank-2` (deprecated) to `rerank-2.5`, with `ms-marco-MiniLM-L-6-v2` as the local fallback. Embedding dimension is derived from the provider profile, not hardcoded.

&nbsp;

#### Causal edges can now distinguish different anchor regions

When two spec documents referenced each other at different anchor points, the system collapsed those into a single edge, losing the distinction. Added `source_anchor` and `target_anchor` columns to `causal_edges` with dedicated indexes. The upsert logic now treats edges with different anchors as distinct relationships.

&nbsp;

#### Search fusion profile for continuity and neutralized length penalty

Cross-encoder rerankers applied length penalties that de-ranked long but highly relevant documents. The Stage 3 rerank gate fired at a 2-candidate floor, producing noisy rerank passes on small result sets. There was no fusion profile tuned for continuity recovery. Neutralized live cross-encoder length penalties, raised the Stage 3 rerank gate from 2 to 4 candidates, and introduced an internal `continuity` adaptive-fusion profile (`0.52 / 0.18 / 0.07 / 0.23` for semantic / keyword / recency / graph weights). Reranker cache telemetry is now emitted so hit-rate and latency can be observed without code changes.

&nbsp;

#### Content routing accuracy hardened with Tier 3 LLM classifier

The save router conflated delivery events with progress updates and could not cleanly distinguish handover content from noise that should be dropped. A more accurate Tier 3 LLM classifier existed but was never wired into the live save path. Delivery-vs-progress classification and handover-vs-drop logic were both tightened. The Tier 3 LLM classifier is now wired into `memory-save` behind rollout gating, with cache reuse and fail-open fallback to Tier 2 when the LLM call errors. Routing accuracy improves without blocking saves when the upstream LLM is unavailable.

## Code Graph Upgrades

Phase 005 shipped a set of runtime improvements to the code-graph query surface that make traversal safer, more accurate, and better contextualized for multi-file work.

#### Query results carry provenance and richer edge evidence

Code-graph queries had no provenance signal, so callers could not tell which detector produced an edge. Edge evidence carried no additional context beyond a line number, which left callers guessing why a match was returned. Shipped a detector provenance taxonomy so every node and edge carries a labeled source, and enriched edge evidence metadata additively so existing consumers keep working while new consumers see the extra context.

&nbsp;

#### Traversal is bounded and supports multi-file queries

Blast-radius traversal could explode on large graphs with no guardrail, and multi-file queries had no explicit union mode. Added depth-safe blast-radius traversal with an explicit bound, an explicit multi-file union query mode, and advisory hot-file breadcrumbs in query results so callers can see which files dominate a traversal. Lexical fallback cascade, routing facade/clustering, GraphML/Cypher export, and rationale-node support were explicitly deferred.

## Skill Advisor Graph

Phase 011 (renumbered from the original top-level `007-skill-advisor-graph`) shipped the production-ready skill graph that powers skill routing, graph-boosted recommendations, and cross-skill traversal.

#### Skill graph is hardened against ghost candidates and missing edges

The initial skill graph had ghost candidates (skills referenced in routing but missing from the compiled graph), several central skills were missing edges that should have existed, and the review/research sibling pairing leaked unrelated routing signals. Shipped a ghost-candidate guard that fails compile when a skill is referenced but missing, filled missing edges on the central skills, and removed the bad sk-deep-review ↔ sk-deep-research sibling pairing. Compiler validation was hardened so future regressions fail fast instead of silently producing bad output.

&nbsp;

#### Skill graph compiles intent signals and prerequisites

Skills declared `intent_signals` and `prerequisite_for` metadata, but the compiler dropped those fields before runtime saw them. Graph-boost evidence was conflated with direct evidence, masking when a recommendation came from intent vs lexical match. The compiler now emits `intent_signals` and `prerequisite_for` into the runtime graph output. Confidence handling separates graph-derived evidence from direct lexical evidence so callers can tell which signal drove the match.

&nbsp;

#### Skill advisor package standardized around a single runtime layout

The skill-advisor package had runtime files, feature catalogs, and playbooks scattered across several paths, and different surfaces referred to them with inconsistent legacy literals. The package is now rooted at `.opencode/skill/skill-advisor/` with runtime under `scripts/`, a validated layout for `feature_catalog/` and `manual_testing_playbook/`, and aligned metadata. Repo-wide references were migrated to the new runtime layout and legacy path literals were removed from the 011 packet tree.

&nbsp;

#### Skill graph metadata corpus migrated to schema v2

Skill-graph metadata used schema v1, which lacked the `derived` block required for automated ranking and traversal signals. Migrating all 21 skill metadata files to v2 risked breaking the compiler for any legacy file that had not yet been upgraded. All 21 per-skill metadata files were migrated to schema v2 with additive `derived` blocks. Compiler validation remains compatible with both v1 and v2 so any late legacy file does not fail compile during the transition.

## Deep Loop Skills

Phase `042-sk-deep-research-review-improvement-2` converted consolidated agentic-systems research into a runtime truth foundation, a semantic coverage graph, a wave executor, and an offline loop optimizer. The bundle touched ~100 files and added +19K lines across contracts, libraries, MCP tools, reducer logic, agent instructions, and test suites. Shipped in eight dependency-ordered phases plus a five-lane remediation pass from the closing audit.

#### Every loop stop reports a named reason through a legal-stop gate

Loop stop conditions were scattered across reducers and workflow code with ad-hoc reasons, making it impossible to audit why a deep-research or deep-review loop halted. Resume semantics were inconsistent between the journal, the reducer, and the visible state log. All stop reasons now come from a shared taxonomy. Every stop passes through a legal-stop gate that checks convergence, dimension coverage, and quality together before the loop is allowed to end. Separate append-only journals, semantic convergence traces, and explicit reducer ownership boundaries make loop behavior auditable. Resume semantics carry deterministic continuation state so restart/resume does not silently diverge from the last persisted iteration.

&nbsp;

#### Convergence is computed over a real coverage graph, not a counter

Deep-loop convergence was a single abstract score. There was no substrate for representing which dimensions had been covered, which claims depended on which evidence, or what had been adjudicated across iterations. Coverage now lives in a dedicated `deep-loop-graph.sqlite` with coverage nodes, edges, and snapshots. Four CJS shared libraries extract and adapt graph primitives from the memory MCP code. Four MCP tools expose the graph to reducers: `deep_loop_graph_upsert`, `deep_loop_graph_query`, `deep_loop_graph_status`, `deep_loop_graph_convergence`. The reducer/MCP contract defines explicit authority order and a fallback chain so graph writes always reach the same source of truth.

&nbsp;

#### Graph is session-isolated after coverage v2 migration

Coverage graph tables used a single primary key on `id`, which meant concurrent sessions on the same spec folder could collide on shared node or edge IDs  -  silently overwriting each other's coverage records. `deep-loop-graph.sqlite` was bumped to v2 with a composite primary key of `(spec_folder, loop_type, session_id, id)` on both `coverage_nodes` and `coverage_edges`. Migration uses drop-and-recreate so the corpus reboots cleanly. A shared-ID collision regression test lives in `session-isolation.vitest.ts`, and the typed `graphEvents` payload shape and namespace rules are documented in both state-format references.

&nbsp;

#### Large targets run in bounded waves without turning workers into managers

Loops on large review or research targets (1000+ files, 50+ domains) either ran serially forever or had to be decomposed by hand. LEAF agents could not spawn sub-agents, so there was no clean way to fan out work while keeping execution auditable. A helper-module wave executor orchestrates fan-out and join at the workflow layer. LEAF workers stay LEAF. Segmentation comes from two deterministic planners: a heuristic v1 and a graph-enhanced v2. Keyed merge preserves per-segment provenance and dedupes overlapping claims. Activation gates ensure wave mode only triggers on genuinely large targets, so small loops keep their existing single-pass profile.

&nbsp;

#### Offline optimizer tunes numeric configs against real packet traces, advisory-only

Numeric knobs in the deep-loop stack (convergence thresholds, segmentation bands, stop gate weights) had no systematic tuning path. Changes were intuition-driven, and there was no way to evaluate a proposed config against historical runs before shipping it. An offline compile/evaluate loop replays real `040` packet traces against candidate configs, scores them with a multi-dimensional rubric, and emits candidate patches with full audit trails. All outputs are advisory: the promotion gate refuses production mutation until behavioral test suites and replay fixtures exist. Phase 4b (prompt-pack generation, cross-packet meta-learning, automatic promotion) remains explicitly deferred.

&nbsp;

#### sk-improve-agent, playbooks, and docs now match the shipped runtime

After the runtime-truth and coverage-graph work landed, the visible surfaces (sk-improve-agent, manual testing playbooks, SKILL.md files, and changelogs for the three deep-loop skills) still described the pre-refactor behavior. sk-improve-agent was aligned with the frozen stop-reason taxonomy, journal wiring, and advisory optimizer contract. Vitest coverage was added across the coverage-graph layers and matching playbook scenarios were authored so manual QA mirrors the automated checks. The graph convergence MCP calls are now wired into both loop workflows. The final phase closed 12 Codex-research recommendations plus 4 phase-008 P1 findings, adding 12 new vitest tests, 7 playbook scenarios, and SKILL.md + changelog bumps for `sk-deep-research`, `sk-deep-review`, and `sk-improve-agent`.

## CLI Prompt Quality Architecture

Phase `043-cli-skill-improved-prompting` shipped a two-tier prompt-quality architecture so every CLI skill (`cli-claude-code`, `cli-codex`, `cli-copilot`, `cli-gemini`) runs a pre-dispatch quality check before calling an external model, with escalation to a shared agent when the prompt is complex enough to need the full methodology.

#### Every CLI skill checks prompt quality before dispatch

CLI skills could dispatch poorly-structured prompts to external models. The full prompt-engineering methodology lived in `sk-improve-prompt`, but loading that skill inline from another skill violates the `_guard_in_skill()` contract and breaks same-skill markdown discovery. Four guard-safe mirror cards now live under each CLI skill's own `assets/prompt_quality_card.md`, and each CLI SKILL.md documents the local card in Resource Domains, loads it through `LOADING_LEVELS["ALWAYS"]`, and requires the pre-dispatch prompt-quality check before building any prompt. All four CLI prompt-template assets now carry `Framework:` annotations so callers can see which framework each template uses.

&nbsp;

#### Complex prompts escalate to the shared @improve-prompt agent

The mirror-card fast path handles routine prompts, but complex or compliance-sensitive prompts still need the full `sk-improve-prompt` methodology without polluting the caller's context. An `@improve-prompt` agent landed across all four runtime directories (`.opencode/agent/`, `.claude/agents/`, `.codex/agents/`, `.gemini/agents/`). Each CLI skill routes prompts with `complexity_hint >= 7` or explicit isolation/fresh-context signals to the agent instead of loading `sk-improve-prompt` inline. The `/improve:prompt` command now supports both Inline mode (ordinary interactive use) and Agent mode (fresh-context improvement).

&nbsp;

#### Canonical card and mirrors stay in sync via shell script

Five copies of the framework-selection table (canonical card + four mirrors) risked drift every time the canonical card changed. `.opencode/skill/skill-advisor/scripts/check-prompt-quality-card-sync.sh` hashes the framework-selection table across the canonical card and the four mirrors. It exits cleanly only when every copy matches, so drift fails a lightweight local verification before it reaches review.

- Every startup the runtime calls the idempotent `dropDeprecatedSharedSpaceColumn()` helper. It issues `ALTER TABLE memory_index DROP COLUMN shared_space_id` when the column is still present and returns silently when the column is already gone or when the SQLite build lacks DROP COLUMN support. The runtime never reads or writes the column, so installs that cannot execute DROP COLUMN keep an unused column on disk without side effects.
- MCP configs auto-derive paths that were previously hardcoded
- Feature flags default to ON  -  no config changes needed
- The resume ladder gracefully handles missing continuity blocks by falling through to spec docs
