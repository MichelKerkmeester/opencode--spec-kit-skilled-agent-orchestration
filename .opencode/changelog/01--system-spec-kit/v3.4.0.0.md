# v3.4.0.0 — Memory That Actually Sticks

> Your AI assistant used to forget. Every session started from zero. Every decision, every trade-off, every carefully reasoned choice — gone the moment the conversation window closed.
>
> **v3.4.0 fixes that.**

This release changes how the system remembers things between sessions. Your memory now lives inside the spec documents you already track — no separate files to maintain, no parallel history to drift against. Picking up where you left off is faster, more reliable, and easier to reason about.

Alongside the memory rewrite, the search stack got smarter, the code graph got safer to traverse, deep-loop skills became fully auditable, and every CLI skill now runs a quality check on its prompts before talking to external models. A cleanup pass on the skill advisor removed dead routing code and added coverage for the kinds of phrases people actually type. The intake command surface also consolidated: `/spec_kit:start` is gone, replaced by `/spec_kit:plan --intake-only` backed by a shared intake contract.

## Memory System

Standalone memory files are gone. Your session notes, decisions, and handover hints now live inside the spec documents the system already tracks. One place for the record. No parallel tree to keep in sync.

#### Legacy memory folder retired

`[spec]/memory/` directories are no longer created. New spec folders never get one. Older ones stay on disk as read-only archive — nothing gets deleted out from under you.

The code that wrote those files is gone. The save workflow no longer renders them. Indexing now flows directly through your canonical spec docs.

The old 589-line memory template is deleted. Four checklist templates lose their "Findings saved to memory/" row. Legacy test fixtures migrated over.

&nbsp;

#### Continuity in spec docs

Every save routes into the right document automatically. Decisions land in `decision-record.md`. Progress updates land in `implementation-summary.md`. Handover notes land in `handover.md`. A content router picks the destination without you thinking about it.

Session resume checks one clear order: `handover.md` first, then the `_memory.continuity` block inside `implementation-summary.md`, then the spec docs themselves. First source with useful content wins.

Quick session notes skip the full save pipeline — the AI can edit the continuity block directly. The full indexed save is reserved for work that needs database visibility.

Session stop hooks autosave by default. Shutting a session no longer loses context.

Five validation checks run before every save. A broken save rolls back instead of corrupting your spec folder.

&nbsp;

#### Graph metadata per spec folder

Every spec folder now carries a `graph-metadata.json` file describing what it holds, what it depends on, and what it relates to. The spec tree is navigable by tools, not just by humans reading markdown.

Each file has two sections: a `manual` block where you can declare relationships (`depends_on`, `supersedes`, `related_to`) and a `derived` block that auto-refreshes with trigger phrases, key files, entities, status, and timestamps.

Saves refresh the metadata automatically. Your manual relationships are preserved; the derived half regenerates on its own.

Search, ranking, and the causal graph all use this file. Searches surface packet roots directly. Ranking respects the relationships you've declared. Graph queries traverse the tree in a single lookup.

Entity extraction is clean and bounded — shell commands filtered out, long or multi-line names rejected, no duplicates.

The initial backfill populated 515 spec-folder roots.

**The dead shared-memory subsystem is fully removed.** Handlers, storage, governance flags, four MCP tools, and all related aliases are deleted. A one-time cleanup runs on startup to drop the stale database column if it's still there.

&nbsp;

#### Simpler commands

Two wrapper commands and two exclusive-writer agents became redundant once continuity moved into canonical spec docs. They're retired.

`/spec_kit:handover` and `/spec_kit:debug` are gone. So are the `@handover` and `@speckit` agents across all four runtimes — 15 paths total.

The old `@speckit` exclusive-writer rule is replaced with a distributed governance rule: any agent writing spec folder docs must use the provided templates, run strict validation after every write, and route continuity through `/memory:save`. `@deep-research` and `@debug` keep write access only for their own specific files.

`/memory:save` becomes the canonical maintainer of the handover log.

`/spec_kit:complete` loses its `:auto-debug` flag. If implementation fails three times, the workflow surfaces the diagnostic and you decide whether to dispatch `@debug` yourself.

## Search and Ranking Upgrades

The search stack is no longer hardcoded to a single backend or a single scoring profile. You can pick your embedding provider, the reranker and its thresholds are tuned for real result sets, continuity search has its own weights, save routing is smarter on mixed content, and causal edges finally distinguish different anchor regions.

&nbsp;

#### Configurable backends

The search setup no longer assumes one fixed backend. The checked-in configs in `.mcp.json`, `.claude/mcp.json`, `.vscode/mcp.json`, `.gemini/settings.json`, and `opencode.json` now share `EMBEDDINGS_PROVIDER=auto`, so the same repo can run with the default cloud path or a local fallback without rewriting config.

&nbsp;

#### Reranker and scoring

The live cross-encoder path is on Voyage `rerank-2.5`, and `getRerankerStatus()` now reports `hits`, `misses`, `staleHits`, and `evictions`, which makes cache behavior visible when a result moves.

Long but useful docs no longer lose ground just for being long. `applyLengthPenalty` still exists for compatibility, but it now resolves to a neutral `1.0`, so long handover and implementation docs keep their place when they are the best match.

Stage 3 reranking also waits for enough evidence to matter. The cutoff moved from 2 results to 4, and the same boundary now holds for both the cloud reranker and the local reranker path.

&nbsp;

#### Continuity search profile

Continuity search now has its own tuned profile instead of reusing the generic weights. Resume-style lookups use `0.52 / 0.18 / 0.07 / 0.23` weights for semantic, keyword, recency, and graph signals, and that profile was checked against a judged 12-query continuity fixture that kept the baseline `K=60` recommendation.

&nbsp;

#### Anchor-aware causal edges

Causal lineage keeps anchor detail now. Schema version `26` added `source_anchor` and `target_anchor` to `causal_edges`, so two links between the same docs can stay separate when they point at different sections.

## Code Graph Upgrades

Traversal is tighter and easier to trust in multi-file work. Every match carries clearer provenance, results ship with real evidence labels, and multi-file union queries are finally first-class.

#### Bounded traversal

`code_graph_query` now stops blast-radius walks before nodes past `maxDepth` are included, which removes the old out-of-bounds spill instead of hiding it after the fact.

This stays scoped to graph-owned outputs, not every startup surface. You get richer code-graph answers, but the trust boundaries from the other graph packets still stay in place.

&nbsp;

#### Detector provenance and evidence

Every node and edge can now carry a real source label instead of a loose parser string. The runtime serializes a dedicated detector-provenance vocabulary, with a compatibility mapper back to the older `ParserProvenance` names so existing surfaces do not break.

Graph-local results also carry better evidence. Shared payloads can now include `edgeEvidenceClass` and `numericConfidence`, which makes it easier to tell whether a relationship came from strong structural evidence or a weaker graph-local guess.

&nbsp;

#### Multi-file union queries

Multi-file union is now a real mode instead of an implied merge. `unionMode: 'multi'` combines several source files without duplicate results, and high-degree files add a `hotFileBreadcrumb` so you can see when one file is dominating the walk.

## Skill Advisor

The production skill graph that powers routing is hardened and future-proofed. A cleanup pass on the routing dicts removed dead code and added coverage for multi-word phrases.

&nbsp;

#### Graph hardening

The compiler now catches missing skills before they ship. If a skill is referenced but not defined, the build fails fast. Previously missing edges on central skills are filled. Bad sibling pairings are removed.

The compiler emits intent signals and prerequisite relationships into the runtime graph. Callers can tell whether a match came from a graph signal or a direct keyword hit.

All 21 per-skill metadata files migrated to schema v2 with new derived blocks. The compiler stays compatible with the old format during the transition so nothing breaks mid-migration.

&nbsp;

#### Routing dicts cleaned up

The advisor had 36 multi-character keys sitting in the wrong routing dict, they never matched anything because of how the tokenizer works. All 36 are removed.

33 of them moved into the right dict (the one that matches multi-word phrases). Three were dropped because the same routing already existed at a higher weight.

One conflicting routing was resolved: `"code audit"` used to point to two different skills in two different dicts. The version that was actually working at runtime stays; the dead pointer is deleted. Nothing changes at runtime.

15 new phrase routes cover identifiers that were going unrouted. A new inline comment warns future contributors not to put multi-word or hyphenated keys into the wrong dict.

The regression test fixture grew from 44 to 52 cases. Top-1 accuracy held at 100% through the whole migration.

**Concrete example.** The query `"5-dimension agent scoring"` used to misroute to `sk-improve-prompt` at 0.77 confidence via a weak keyword hit on `scoring`. After the fix it routes to `sk-improve-agent` at 0.95.

## Auditable Deep-Loop Skills

The runtime behind deep-research and deep-review skills became fully auditable and ready to scale. Stops now have names, convergence runs on a real coverage graph, large targets split into bounded waves, and an offline optimizer can tune configs against real traces without ever touching production.

#### Auditable stops

The deep-loop runtime now records stop decisions in a form you can inspect. Every stop carries a named reason from the same small taxonomy — `converged`, `max_iterations`, `blocked`, `user_stop`, or `error` — and the legal-stop gate now checks convergence, coverage, and quality together before a run is allowed to stop.

That audit trail is no longer implied by scattered notes. Blocked and claim-adjudication decisions now persist as explicit events, and the dashboards stay generated markdown backed by the same structured state underneath.

&nbsp;

#### Coverage graph

A real coverage graph now sits under convergence. `deep-loop-graph.sqlite` stores nodes, edges, and snapshots, and four graph tools expose gaps, contradictions, provenance chains, hot nodes, and stop blockers directly instead of rolling everything up into one abstract score.

Concurrent runs on the same packet no longer share graph state by accident. The graph store now keys records by `(spec_folder, loop_type, session_id, id)`, so two sessions can write the same local node id without colliding.

&nbsp;

#### Bounded waves for large targets

Large targets can now split into bounded waves, but only when the scale justifies it. Wave mode turns on at `1000+` review files or `50+` research domains, keeps smaller runs on the old single-stream path, and uses deterministic segmentation plus keyed merge so provenance survives the join.

## CLI Prompt Quality Architecture

Every CLI skill (`cli-claude-code`, `cli-codex`, `cli-copilot`, `cli-gemini`) now runs a prompt quality check before calling an external model.

A lightweight quality card lives inside each CLI skill. Routine prompts pass through it.

Complex prompts — those flagged by a complexity score or explicit isolation signals — escalate to a shared `@improve-prompt` agent that runs the full prompt-engineering methodology in a fresh context.

A sync script hashes the canonical card and all four mirror copies. Drift fails a local check before it reaches review.
