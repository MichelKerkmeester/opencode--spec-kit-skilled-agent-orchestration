## Runtime Correctness (4)

### Importing the server package started the MCP server as a side effect

**Problem:** `import '@spec-kit/mcp-server'` would call `main()` unconditionally, which means any module that imported the package for its exports would also start a full MCP server process as a side effect. This made the package unsafe to import in test harnesses or tooling that only needed access to the exported functions.

**Fix:** Wrapped the `main()` call in `context-server.ts` behind an entrypoint check that compares `import.meta.url` against `process.argv[1]`. The server only starts when the file is run directly, not when imported.

### Two ESM wrapper scripts still used the CommonJS `__dirname` global

**Problem:** `map-ground-truth-ids.ts` and `reindex-embeddings.ts` both referenced `__dirname`, which does not exist in native ESM modules. Running either script after the ESM migration would throw a `ReferenceError` at startup.

**Fix:** Replaced `__dirname` with `import.meta.dirname` in both files. This is a Node 20.11+ API that provides the same directory path without depending on CommonJS globals.

### The four workspace packages disagreed on the minimum Node version

**Problem:** The root, shared, mcp_server, and scripts packages each declared different `engines.node` floors. The ESM migration requires Node 20.11.0 or later (for `import.meta.dirname`), but not all packages enforced this.

**Fix:** Aligned all four `package.json` files to `engines.node >= 20.11.0`.

### The shared package's root export pointed at the wrong file

**Problem:** `shared/package.json` had its root export (`"."`) pointing at `dist/embeddings.js` instead of `dist/index.js`. Anyone importing `@spec-kit/shared` without a subpath got the embeddings module instead of the full public surface.

**Fix:** Changed the root export to `dist/index.js` and added a separate `./embeddings` subpath export so direct embeddings consumers are not broken.

## Security (4)

### Shared-memory admin operations accepted caller-supplied identity without verification

**Problem:** Admin operations like creating shared spaces or setting memberships accepted `actorUserId` and `actorAgentId` directly from the request payload. A malicious caller could supply any identity to perform admin actions on spaces they should not control.

**Fix:** Added a trusted-transport warning that logs when caller-supplied identity is used without session binding. The current MCP transport does not provide a session principal, so full enforcement is documented as a future requirement.

### The validation bridge silently succeeded when the validation module was missing

**Problem:** When `scripts/dist/memory/validate-memory-quality.js` was not built, the V-rule bridge returned a successful "unavailable" result. This meant saves would succeed without any quality validation, silently bypassing the quality gate.

**Fix:** The bridge now fails closed -- it returns a failure result that blocks the save and surfaces a clear "build scripts first" error message. A `SPECKIT_VRULE_OPTIONAL` environment variable allows development workflows to opt out when scripts are intentionally unbuilt.

### Path resolution could escape the workspace boundary

**Problem:** The path resolver in `shared/paths.ts` walked up the directory tree looking for a `package.json` but had no upper bound check. Running from an unexpected working directory (like `/tmp`) could resolve the database path to a location outside the Spec Kit workspace.

**Fix:** Added `findNearestSpecKitWorkspaceRoot()` which finds the monorepo root by looking for a `package.json` with a `workspaces` array. Any resolved path that falls outside this boundary triggers a fallback to `import.meta.dirname`-relative resolution with a warning.

### Duplicate detection ignored governance scope in multi-tenant databases

**Problem:** The duplicate content check in the preflight validator queried by content hash without filtering by tenant, user, agent, or shared-space scope. In a multi-tenant database, this could flag a memory as "duplicate" based on an identically-hashed memory belonging to a completely different scope. The dry-run response also leaked metadata from the cross-scope match.

**Fix:** The duplicate query now threads the governed scope into the WHERE clause. Cross-scope metadata is redacted from duplicate detection responses so dry-run payloads only show information the caller is authorized to see.

## Reliability (2)

### File-write failures after a successful save were reported as generic "anchor issues"

**Problem:** When a memory was saved to the database but the corresponding file write failed, the warning was lumped into the "anchor issues" category. This made it impossible for callers to distinguish between cosmetic anchor warnings and genuine data divergence between the database and filesystem.

**Fix:** Warnings now carry typed categories. File-persistence failures use a `[file-persistence-failed]` prefix that the response builder detects and exposes as a separate `file-persistence-failed` category in the `typedWarnings` array.

### Three different error-handling patterns for the same dynamic import

**Problem:** `scripts/core/workflow.ts` had three separate call sites that dynamically imported `@spec-kit/mcp-server/api`, each with a different error-handling pattern. One silently returned null, another logged a warning, and the third threw. This inconsistency made it hard to predict how import failures would propagate.

**Fix:** Consolidated the three patterns into a single `tryImportMcpApi()` helper that logs the failure reason and returns null, matching the existing degradation contract.

## Performance (2)

### Hot retrieval paths paid per-call async import overhead

**Problem:** `vector-index-store.ts` used `await import(...)` inside frequently-called retrieval methods. Each search or read operation re-resolved the module specifier at runtime, adding unnecessary overhead to the hottest code paths.

**Fix:** Replaced per-call imports with a module-level cached lazy loader that resolves the import once and caches the result for subsequent calls.

### CLI loaded the full database stack even for `--help`

**Problem:** `cli.ts` imported all command dependencies at the top level, including the database layer, vector index, and checkpoint system. Running `node dist/cli.js --help` loaded the entire stack before printing usage text.

**Fix:** Deferred heavy imports behind per-command handler functions. `--help` and `--version` now exit immediately without loading the database stack. Each subcommand loads only its own dependencies.
