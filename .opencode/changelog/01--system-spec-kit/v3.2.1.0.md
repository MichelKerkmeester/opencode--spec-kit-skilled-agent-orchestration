## [**3.2.1.0**] - 2026-04-03

This release expands System Spec Kit in three directions at once: packet-local nested changelog generation is now a shipped workflow, the `session_bootstrap` recovery contract and structural safeguards were remediated after review, and the deep-research/deep-review command surfaces now sit on executable parity-tested contracts. It also records the approved Codex command-discoverability direction from Packet 039 without overstating that packet as a downstream quick-reference implementation.

> Spec folders: `.opencode/specs/system-spec-kit/024-compact-code-graph/029-review-remediation/`, `.opencode/specs/system-spec-kit/025-nested-changelog-per-spec/`, `.opencode/specs/skilled-agent-orchestration/039-cmd-spec-kit-codex-skill-routing/`, and `.opencode/specs/skilled-agent-orchestration/040-sk-auto-deep-research-review-improvement/`

## Packet Changelogs (3)

System Spec Kit can now generate packet-local changelogs as a first-class workflow instead of leaving packet history to hand-written one-offs. Root packets and direct child phases now have a canonical place to record chronological completion history beside their implementation summaries.

---

### Nested changelog generator

**Problem:** Packet-local changelogs already existed in some specs, but there was no canonical generator, no approved naming rule, and no single workflow explaining how root packets and phase folders should produce those files.

**Fix:** System Spec Kit now includes a dedicated nested changelog generator that detects whether the target is a packet root or a direct child phase. Root output writes to `changelog/changelog-<packet>-root.md`, while phase output writes to the parent packet as `changelog/changelog-<packet>-<phase-folder>.md`.

---

### Templates and workflow

**Problem:** Even where packet changelog files were being created, the repo still lacked canonical templates and documentation for what those files should contain and how they relate to `implementation-summary.md`.

**Fix:** Dedicated root and phase changelog templates were added, plus a workflow reference that explains the evidence stack, output paths, and packet-local intent. Packet changelogs are now explicitly documented as additive to `implementation-summary.md`, not a replacement for it.

---

### Closeout commands

**Problem:** `/create:changelog`, `/spec_kit:implement`, and `/spec_kit:complete` could not speak consistently about packet-local history if the generator, templates, and workflow docs were updated in isolation.

**Fix:** All three command surfaces were updated in the same pass so packet-aware closeout now teaches one consistent behavior: implementation summary plus nested packet changelog, with root and phase naming handled by the shipped generator instead of manual improvisation.

---

## Recovery Fixes (4)

Packet 029 closed the open review findings on the Compact Code Graph / recovery surfaces. This release captures those fixes at the System Spec Kit level because they change the public recovery contract and the reliability of structural startup context.

---

### `session_bootstrap` and `nextActions`

**Problem:** The public `session_bootstrap` contract already described recommended next actions, but the handler did not return a real `nextActions` payload. The docs and the implementation were telling slightly different truths.

**Fix:** `session_bootstrap` now generates real `nextActions`, and the tool schema documents the full output shape explicitly. The recovery surface no longer depends on readers inferring behavior from docs that the handler itself did not fulfill.

---

### Structural budget enforcement

**Problem:** The structural startup contract documented a 500-token ceiling, but that limit was still advisory rather than enforced in the shared builder.

**Fix:** The shared structural bootstrap builder now applies that 500-token limit as code, which makes the startup structural context truthful, bounded, and consistent with the packet docs that promised a hard ceiling.

---

### Provenance and autosave targeting

**Problem:** Compact recovery and autosave behavior still had ambiguity at two risky points: recovered text could appear without the shared provenance fence, and transcript-tail packet detection could promote uncertain scope matches into autosave targets.

**Fix:** Gemini compact recovery now wraps recovered text with the shared provenance fence, and Claude stop-hook autosave now preserves validated active-session state while retargeting only when transcript evidence clearly shows a new unambiguous packet. Ambiguous transcript-only matches are no longer treated as authoritative.

---

### Runtime and packet docs

**Problem:** The implementation fixes alone were not enough. The recovery story also needed the runtime-facing agent copies and packet documentation to reflect the bootstrap-first contract and the shipped closeout state.

**Fix:** Claude and Codex `context-prime` definitions now include the explicit Structural Context guidance already present in the OpenCode version, and the related Packet 024 root/phase docs were refreshed so the shipped evidence, phase map, and recovery contract tell the same story.

---

## Research and Review Contracts (4)

Packet 040 brought both autonomous workflows onto one stronger contract model. This matters to System Spec Kit because the command assets, runtime mirrors, and contract tests now live on explicit executable rules instead of documentation-only alignment.

---

### Deep-research contract

**Problem:** The deep-research workflow still relied on scattered wording for lineage, lifecycle branches, reducer ownership, packet artifacts, and runtime parity. That made the contract easier to drift and harder to audit.

**Fix:** Deep-research now has canonical lineage fields, lifecycle modes, reducer-owned outputs, runtime capability surfaces, and executable helpers. The workflow YAML, skill docs, runtime mirrors, and packet-closeout rules all point at the same packet model.

---

### Deep-review contract

**Problem:** Deep-review still mixed `deep-research-*` and `deep-review-*` naming across docs, assets, and workflow surfaces. Release-readiness and reducer schemas were also easier to describe than to verify.

**Fix:** Deep-review now uses canonical `deep-review-*` artifact names across the live contract, limits legacy `deep-research-*` names to the intentional scratch migration path, and exposes explicit lifecycle, reducer, severity, and release-readiness schemas across the contract surfaces.

---

### Contract test coverage

**Problem:** Without targeted tests, deep-research and deep-review could drift back out of sync across docs, command assets, and runtime mirrors while still looking plausible on manual inspection.

**Fix:** Focused Vitest suites in the System Spec Kit scripts workspace now guard both contracts directly: deep-research parity plus reducer idempotency, and deep-review parity plus reducer/readiness schemas. That makes future drift visible quickly and keeps the contracts executable.

---

### Runtime mirror parity

**Problem:** Autonomous workflow instructions can silently diverge across OpenCode, Claude, Gemini, Codex, and compatibility wrappers when each runtime surface is patched separately.

**Fix:** Packet 040 synchronized all runtime mirrors for both deep-research and deep-review in one pass, so lifecycle vocabulary, reducer-owned surfaces, and packet file expectations now move together instead of fragmenting by runtime.

---

## Codex Command Direction (1)

Packet 039 does not represent a downstream quick-reference implementation release. It is still important here because it records the approved direction for how Codex-specific command discoverability should be handled inside the System Spec Kit documentation surfaces.

---

### Research result and override

**Problem:** The original research recommendation was a narrow four-command shortlist, but the user later approved a broader all-commands short-list direction. Without a packet update, the record could imply only one of those truths.

**Fix:** Packet 039 now preserves the original four-command recommendation while also recording the approved all-commands short-list override. The downstream implementation target stays explicit: `references/workflows/quick_reference.md` is the primary command surface, while `SKILL.md` remains a pointer-only surface to avoid duplicating the command matrix and reintroducing drift.
