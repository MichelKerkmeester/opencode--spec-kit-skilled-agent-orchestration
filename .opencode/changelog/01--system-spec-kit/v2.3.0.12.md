## [**2.3.0.12**] - 2026-02-26

This release fix dual importance_tier fields in memory export - Memory files were generated with two conflicting importance tier fields: a hardcoded importance_tier: "normal" (snake_case) and a dynamically resolved importanceTier: "critical" (camelCase). The stale default was read first by frontmatter parsers, causing incorrect tier classification. It focuses on making the instructions and release notes easier for non-developers to follow.

---

## What Changed

### 1. Root cause

- Need: People needed clearer guidance about Root cause.
- Change: Root cause: Template context template had a hardcoded importance_tier: "normal" alongside a placeholder importanceTier: "{{IMPORTANCE_TIER}}". After rendering, both fields persisted with different values.
- Why it matters: This matters because the release notes and instructions are easier to read and trust.

### 2. Template fix

- Need: People needed clearer guidance about Template fix.
- Change: Template fix: Replaced both fields with a single importance_tier: "{{IMPORTANCE_TIER}}" that receives the dynamically detected value during rendering.
- Why it matters: This matters because the release notes and instructions are easier to read and trust.

### 3. Interface cleanup

- Need: There was a reliability problem around Interface cleanup.
- Change: Interface cleanup: Removed importanceTierAlias from ManagedFrontmatter interface and its serialization path in serializeFrontmatter().
- Why it matters: This matters because it reduces confusing failures and makes the system more dependable.

### 4. Backward compatibility

- Need: There was a reliability problem around Backward compatibility.
- Change: Backward compatibility: Added resolveImportanceTierValue() helper in buildManagedFrontmatter() to handle existing memory files with legacy dual fields - prefers the non-"normal" value when both are present.
- Why it matters: This matters because it reduces confusing failures and makes the system more dependable.

### 5. All 10 frontmatter backfill tests pass

- Why it matters: This matters because the team has clearer proof that the change works.

### 6. TypeScript build clean with zero errors
