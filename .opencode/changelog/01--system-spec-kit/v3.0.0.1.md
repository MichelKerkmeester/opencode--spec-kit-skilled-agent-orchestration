## [**3.0.0.1**] - 2026-03-27

Fixes a bug where the indexing system rejected almost all spec folder documents. Spec files like `spec.md`, `plan.md`, and `tasks.md` were being run through a quality gate designed for memory files -- which requires metadata (like trigger phrases and importance tiers) that spec documents intentionally do not have. About 97% of spec documents were silently rejected. Now spec documents bypass this gate during indexing while memory files are still fully validated.

> Spec folder: `specs/02--system-spec-kit/022-hybrid-rag-fusion/001-hybrid-rag-fusion-epic/012-pre-release-remediation/`

---

## What Changed

### The core bug

When the system scans and indexes files, it runs them through a quality gate that checks for required metadata. This gate makes sense for memory files (which should have trigger phrases, importance tiers, etc.), but spec documents like `spec.md`, `plan.md`, and `tasks.md` are a different kind of content -- they follow their own templates and intentionally lack memory-specific metadata.

The indexer was treating all files the same, so spec documents failed the quality gate and were never indexed. This meant the system could not search or build relationships between spec documents.

**The fix:** Added a "warn-only" mode to the quality gate. During indexing scans, spec documents now pass through with warnings logged (but not rejected). The normal save path for memory files still enforces all checks.

### Why this matters beyond searchability

The system builds a "causal chain" across spec folders -- linking spec documents to their plans, tasks, checklists, and implementation summaries with "caused" and "supports" relationships. With spec documents rejected from the database, these chains were nearly empty (only about 7 links existed). After re-indexing with the fix, hundreds of causal chains can form, making it possible to trace how a spec led to a plan, which led to tasks, which led to implementation.

### Other fixes

- **Build error in save orchestration** -- A TypeScript error where the code assumed a value was always a list, but it could sometimes be an empty object. Added a check to handle both cases.

### Database cleanup (one-time maintenance)

- Purged 128 stale memories: 112 marked as deprecated, 14 leftover test entries, and 2 test stubs.
- Repaired 3,035 orphaned causal edges and 2 orphaned chunks (records pointing to deleted parents).
- Ran a full re-index, which brought the memory count from 358 to 407 after recovering previously-missed files.

## Upgrade

1. Rebuild: `npm run build` in `mcp_server/`.
2. Restart the MCP server.
3. Run `memory_index_scan({ force: true })` to re-index all spec documents and rebuild causal chains.
4. No database migration required.
