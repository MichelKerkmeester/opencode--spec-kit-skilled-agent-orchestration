## [**3.0.0.1**] - 2026-03-27

**Spec doc indexing: warn-only quality gate for spec folder documents** -- Fixes a bug where `memory_index_scan` rejected ~97% of spec folder documents (spec.md, plan.md, tasks.md, etc.) because they were routed through the memory-save quality gate, which requires memory-specific frontmatter (trigger phrases, YAML metadata) that spec documents intentionally do not have. Spec documents are now indexed with `qualityGateMode: 'warn-only'`, allowing the gate to log but never reject template-validated content.

> Spec folder: `specs/02--system-spec-kit/022-hybrid-rag-fusion/001-hybrid-rag-fusion-epic/012-pre-release-remediation/`

---

## Highlights

### Category: Bug Fix

- **Spec document quality gate bypass** -- Added a `qualityGateMode` option (`'enforce' | 'warn-only'`) that flows from `memory_index_scan` through `indexSingleFile` to `indexMemoryFile` and into `processPreparedMemory`. During scan indexing, spec folder documents (identified via `specDocKeySet`) now pass `qualityGateMode: 'warn-only'`, which logs quality gate failures but does not reject the file. The `memory_save` path remains unchanged at `'enforce'`.
- **Causal chain coverage restored** -- With spec documents no longer rejected, `createSpecDocumentChain` can now find all document types (spec, plan, tasks, checklist, implementation-summary, research, decision-record) in the database and build complete `caused` and `supports` edges across spec folders. Previously only ~7 `caused` edges existed; after re-indexing, hundreds of causal chains should form.
- **Pre-existing pe-orchestration build fix** -- Fixed a TypeScript `TS2488` error in `pe-orchestration.ts` where the spread operator was applied to `reinforced.warnings` / `updated.warnings` which could be `{}` instead of an array. Changed to `Array.isArray()` guard.

### Category: Database Maintenance

- **Memory cleanup** -- Purged 128 stale memories: 112 deprecated tier, 14 test entries (`999-atomic-save-fi`), and 2 test stubs (`alpha`, `beta`). Database reduced from 527 to 399 memories.
- **Orphaned edge cleanup** -- Auto-repaired 3,035 orphaned causal edges and 2 orphaned chunks via `memory_health({ autoRepair: true })`. Graph health status moved from `has_orphans` to `healthy`.
- **Force re-index** -- Ran full force scan to rebuild embeddings and causal chains. Memory count rose from 358 to 407 after re-indexing previously missing files.

## Upgrade

1. Rebuild: `npm run build` in `mcp_server/`.
2. Restart the MCP server so the new dist is loaded.
3. Run `memory_index_scan({ force: true })` to re-index all spec documents and rebuild causal chains.
4. No database migration required.
