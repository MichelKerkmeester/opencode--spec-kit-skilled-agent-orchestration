## [**2.0.0.0**] - 2026-02-07

This release moves the system spec kit from JavaScript to TypeScript. The goal was to make the codebase easier to maintain without changing how the runtime behaves for existing users.

---

## What Changed

---

### The whole codebase moved to TypeScript

**Problem:** The project had grown large enough that stronger type checks were worth the migration effort.

**Fix:** This release converted the shared code, the MCP server, the scripts, the tests, and the supporting documentation from JavaScript-based references to TypeScript-based ones. This gives the project a more consistent foundation and helps catch mistakes earlier during development.

---

### The migration was rolled out in clear phases

**Problem:** A change this large needed to be broken into manageable parts instead of done as one risky rewrite.

**Fix:** The work was split across infrastructure, shared utilities, core server code, handlers, scripts, tests, and documentation so each area could be updated in sequence. This lowered migration risk and made it easier to validate each part before moving on.

---

### Build and type infrastructure were added

**Problem:** A TypeScript codebase needs its own build rules, shared type definitions, and project-level settings.

**Fix:** This release added TypeScript configuration files, shared types, generated type declarations, source maps, and project references for multi-workspace builds. This keeps the build process predictable and makes cross-project imports easier to understand.

---

### Existing runtime behavior was preserved

**Problem:** The migration could not break the way people already start or use the system.

**Fix:** The compiled output stayed compatible with the existing CommonJS runtime, existing startup commands stayed in place, and the generated JavaScript behavior was kept aligned with the earlier version. This lets teams adopt the new source format without having to relearn or rewire the runtime setup.

---

### Tests and docs were brought along with the code

**Problem:** A language migration is hard to trust if the tests and documentation still describe the old setup.

**Fix:** This release converted the test suite, updated example code, and changed documentation references from `.js` files to `.ts` files where needed. This keeps the written guidance in sync with the actual code and makes the migration easier to follow.

---

### A few new shared files were added to support the shift

**Problem:** Some pieces of the TypeScript setup needed dedicated support files.

**Fix:** This release introduced shared type definitions and vector-extension type declarations so the new code could compile cleanly across workspaces. This removes guesswork around shared interfaces and gives the migration a cleaner long-term base.

## Upgrade

- Pull the latest changes and rebuild the project as usual.
- Existing startup commands stay the same, so no runtime migration is required.
- If you maintain local docs or examples, update any remaining `.js` source references to `.ts`.
