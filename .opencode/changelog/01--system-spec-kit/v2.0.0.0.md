## [**2.0.0.0**] - 2026-02-07

This release moves the system spec kit from JavaScript to TypeScript. The goal was to make the codebase easier to maintain without changing how the runtime behaves for existing users.

---

## What Changed

### The whole codebase moved to TypeScript

- Need: The project had grown large enough that stronger type checks were worth the migration effort.
- Change: This release converted the shared code, the MCP server, the scripts, the tests, and the supporting documentation from JavaScript-based references to TypeScript-based ones.
- Why it matters: This gives the project a more consistent foundation and helps catch mistakes earlier during development.

### The migration was rolled out in clear phases

- Need: A change this large needed to be broken into manageable parts instead of done as one risky rewrite.
- Change: The work was split across infrastructure, shared utilities, core server code, handlers, scripts, tests, and documentation so each area could be updated in sequence.
- Why it matters: This lowered migration risk and made it easier to validate each part before moving on.

### Build and type infrastructure were added

- Need: A TypeScript codebase needs its own build rules, shared type definitions, and project-level settings.
- Change: This release added TypeScript configuration files, shared types, generated type declarations, source maps, and project references for multi-workspace builds.
- Why it matters: This keeps the build process predictable and makes cross-project imports easier to understand.

### Existing runtime behavior was preserved

- Need: The migration could not break the way people already start or use the system.
- Change: The compiled output stayed compatible with the existing CommonJS runtime, existing startup commands stayed in place, and the generated JavaScript behavior was kept aligned with the earlier version.
- Why it matters: This lets teams adopt the new source format without having to relearn or rewire the runtime setup.

### Tests and docs were brought along with the code

- Need: A language migration is hard to trust if the tests and documentation still describe the old setup.
- Change: This release converted the test suite, updated example code, and changed documentation references from `.js` files to `.ts` files where needed.
- Why it matters: This keeps the written guidance in sync with the actual code and makes the migration easier to follow.

### A few new shared files were added to support the shift

- Need: Some pieces of the TypeScript setup needed dedicated support files.
- Change: This release introduced shared type definitions and vector-extension type declarations so the new code could compile cleanly across workspaces.
- Why it matters: This removes guesswork around shared interfaces and gives the migration a cleaner long-term base.

## Upgrade

- Pull the latest changes and rebuild the project as usual.
- Existing startup commands stay the same, so no runtime migration is required.
- If you maintain local docs or examples, update any remaining `.js` source references to `.ts`.
