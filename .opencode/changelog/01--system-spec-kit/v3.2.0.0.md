## ESM Module Migration (3)

The runtime told the truth about its module system for the first time. `@spec-kit/shared` and `@spec-kit/mcp-server` are now native ESM; `@spec-kit/scripts` stays on CommonJS by design.

### Shared package moved to native ESM

**Problem:** `@spec-kit/shared` used modern import syntax in source but the package contract did not match what Node actually runs. Consumers inherited that ambiguity — the package looked modern in one place and old in another.

**Fix:** `@spec-kit/shared` now ships as real native ESM. Package metadata, compiler settings, and 48 relative import paths were rewritten so runtime behavior matches the source. That gave every downstream package a clean base to build on.

### MCP server moved to native ESM

**Problem:** The MCP server had years of CommonJS assumptions buried inside it — `require()`, `__dirname`, `__filename`, and loose JSON loading all made the package look modern in source while depending on old runtime rules.

**Fix:** `@spec-kit/mcp-server` now runs as native ESM end to end. The migration rewrote 839 imports across 181 files, replaced CommonJS-only globals, converted the remaining `require()` holdouts, and repaired ESM startup details. The result is a server that starts and runs under the module system it claims to use.

### Scripts kept CommonJS without pretending to be ESM

**Problem:** `@spec-kit/scripts` still needed CommonJS for callers that depend on that contract. Flipping it to ESM for consistency would have broken the wrong thing.

**Fix:** Scripts stayed on CommonJS with Node 25's native `require(esm)` support used where needed, loading blockers removed at the package boundary, and the memory-save path hardened at the same time. Compatibility was preserved without dragging old assumptions back into the migrated packages.

## Session Recovery & Context Preservation (7)

Long AI coding sessions no longer depend on the user re-explaining context after compaction, restart, or runtime switches.

### Compaction rescue for Claude sessions

**Problem:** Claude compaction wiped the AI's short-term memory. After a compact, the assistant lost the active spec folder, recent code focus, and the rules guiding the work.

**Fix:** A two-step rescue path was introduced: PreCompact caches a compact recovery payload, and SessionStart injects that payload back only for `source=compact`. Working-memory attention signals, spec-folder detection, and token-budget controls make the recovered brief small enough to fit while still being useful.

### Source-aware session priming

**Problem:** Startup, resume, clear, and compact were treated too similarly, which meant the assistant either got too much context or the wrong context for the situation.

**Fix:** Session priming now branches by source. Compacts restore cached context, resumes point at the compact brief flow, startups surface tool and freshness guidance, and clears fall back to minimal recovery instructions instead of pretending a full recovery already happened.

### Stop-hook tracking and best-effort auto-save

**Problem:** The system had little durable insight into session usage and no reliable way to capture end-of-session state automatically.

**Fix:** Stop-hook tracking parses transcript usage data, stores hook-state metrics, estimates token spend, and can trigger a best-effort `generate-context.js` save path when configured thresholds are met.

### Hookless auto-priming and bootstrap recovery

**Problem:** Claude hooks covered only one runtime family. Other runtimes still depended on the AI remembering to prime itself manually.

**Fix:** MCP first-call priming, `session_resume`, `session_health`, and `session_bootstrap` were added so hookless runtimes can recover with a single canonical path. Structural bootstrap contracts and startup briefs were added for OpenCode-style flows.

### Hook durability and security hardening

**Problem:** Cache races, weak temp-file handling, session-state collisions, and insufficient provenance markers made the hook path more fragile than it looked on paper.

**Fix:** TTL checks, SHA-256-based state names, restrictive file permissions, provenance fencing, stale-on-read checks, and response-envelope cleanup tightened the hook-state lifecycle.

### Floors-plus-overflow budgeting

**Problem:** A fixed budget split wastes capacity when one source is empty, but a pure pool can starve the most important context.

**Fix:** Budgeting moved to a floors-plus-overflow model so constitutional context, graph context, semantic hints, and triggered memories each have a baseline allocation while sharing spare capacity when one channel is quiet.

### Startup orientation quality fixes

**Problem:** The first rollout of startup structural highlights was real but noisy — duplicates, vendored code, and the wrong centrality heuristic made the output harder to trust.

**Fix:** Follow-on phases repaired the experience with shared startup brief builders, an explicit structural bootstrap contract, and a final SQL remediation pass that deduplicates symbols, excludes non-project paths, and ranks by incoming callers instead of outgoing callers.

## Structural Code Graph (4)

A new SQLite-backed code graph answers structural questions like "who calls this?" and "what does this file import?" — something semantic search alone could never do.

### Structural indexer and graph storage

**Problem:** Semantic search could find code by meaning, but it could not answer structural questions about callers, imports, or dependency chains.

**Fix:** A structural indexer extracts symbols, relationships, and call edges from the codebase. A SQLite-backed graph store persists the results with scan, query, status, and context-expansion tools exposed through MCP.

### Tree-sitter migration with regex fallback

**Problem:** Regex-only parsing was fast to ship but too brittle for long-term graph correctness — abstract methods, class expressions, multi-import captures, and nested class names were all misclassified.

**Fix:** Parser adapters were introduced, progressively moving toward tree-sitter-backed extraction while retaining regex as a fallback. Follow-on phases repaired classification accuracy so the graph matches the codebase more faithfully.

### Freshness and auto-reindex

**Problem:** A stale graph is worse than no graph because it looks authoritative while pointing at the wrong branch state.

**Fix:** `ensureCodeGraphReady()` logic, HEAD-change detection, tracked-file mtime checks, and true fresh/stale/empty reporting were added so the system can reindex before structural answers drift too far from the repo state.

### CocoIndex bridge and intent-aware routing

**Problem:** Semantic search, structural search, and memory search lived beside each other without enough guidance about when to use which system.

**Fix:** Seed-resolution and context orchestration now let CocoIndex hits expand into structural neighborhoods, while intent metadata and routing-enforcement phases push semantic questions toward CocoIndex, structural questions toward the graph, and session-history questions toward memory surfaces.

## Search & Retrieval (10)

Search went from a zero-result outage to a trustworthy hybrid pipeline. Two silent filters were removing every result; once fixed, ranking, recall, diversity, and observability all improved.

### The zero-result search outage was fixed at the root cause

**Problem:** Two separate filters were silently discarding every search result despite a healthy 999-memory database. Access-scope enforcement treated unscoped queries as "this caller owns nothing," and a lifecycle-state filter checked against a column that did not exist. Both channels found good matches, then threw them all away.

**Fix:** Access-scope enforcement now requires deliberate opt-in. The lifecycle-state filter was removed while state storage remains unimplemented. After these two fixes, queries returned 4-5 results with both vector and FTS5 channels contributing.

### Ranking tuned for a smaller corpus

**Problem:** The reciprocal rank fusion smoothing constant (k=60) was designed for 10,000+ document corpora. At that value, the score difference between rank 1 and rank 10 was only 0.002 — essentially flat for a 999-memory database.

**Fix:** k=40 doubles the rank separation to 0.004, making top results meaningfully distinguishable from lower-ranked ones.

### Multi-word phrase queries now find exact phrases

**Problem:** Searching for "spec kit" only matched documents containing "spec" OR "kit" individually. And a double-quoting bug could wrap already-quoted phrases in a second pair of quotes, creating invalid FTS5 syntax that silently dropped results.

**Fix:** Multi-word queries now generate FTS5 phrase variants alongside individual tokens, and the FTS5 path detects already-quoted tokens to prevent double-quoting. Exact-phrase matches are boosted properly.

### Focused and deep search no longer lose candidates

**Problem:** Focused mode could promote a discovered folder into a hard search filter, and deep mode could let an ephemeral `sessionId` act like governance scope. Both cases made legitimate candidates disappear.

**Fix:** Folder discovery no longer turns into a hard scope filter. When a discovered-folder pass comes back empty, the handler retries without that narrow scope. Stage-1 candidate generation no longer treats a plain `sessionId` as governed scope input.

### Token budget raised and truncation improved

**Problem:** The search token budget (1,500 tokens) often truncated results to just 1 out of 7+ matches. And token-budget enforcement could remove whole results too early when one or two matches were unusually long.

**Fix:** Budget raised to 2,500 tokens. The response builder now truncates oversized content before dropping results, and keeps compact metadata-only entries for additional matches after full-content slots are exhausted.

### Deprecated memories excluded consistently across all channels

**Problem:** Vector search excluded deprecated-tier memories, but FTS5 text search did not — creating a channel asymmetry where deprecated content leaked back through the lexical path.

**Fix:** Both channels now apply the same exclusion filter. Results are consistent regardless of which internal path surfaced them.

### Related memories now enter search results correctly

**Problem:** The co-activation system expected `related_memories` in object format but the database stored them as scalar arrays. The parser silently failed, meaning co-activation never contributed. Additionally, scalar entries were assigned similarity 0.5 on a 0-100 scale, producing boosts two orders of magnitude weaker than intended.

**Fix:** Both format and scale issues were corrected. Co-activated memories are returned in the right format at the right scale and injected into the fusion candidate pool.

### Reranking scores content and title separately

**Problem:** The cross-encoder reranker concatenated title and body into a single block before scoring. A strong title match and a weak body could cancel each other. Additionally, the MMR diversity pass was skipped when the full reranker was unavailable.

**Fix:** Title and body are scored as separate inputs. The MMR diversity pass runs regardless of reranker availability. A new `rerankProvider` metadata field records which reranker path was actually taken.

### Search traces now show active channels

**Problem:** It was hard to tell which channels actually contributed to results and whether graph signals were active or merely configured.

**Fix:** Stage 1 reports `activeChannels`, Stage 2 emits zero-contribution graph diagnostics, and per-stage timing is now persisted. The embedding cache tracks hit/miss statistics. Search pipeline behavior is observable through monitoring instead of requiring live log inspection.

### Vector search no longer drifts onto an empty database

**Problem:** After lazy Voyage-4 initialization, the vector-index store could re-resolve its SQLite path and drift from the populated `context-index.sqlite` to an empty provider-specific database.

**Fix:** Database path resolution is stabilized around the active populated database, with conflict logging and post-validation connection caching. Startup now logs the active DB path and refuses to rebind consumers onto an empty database.

## Saving Memories (5)

Memory saves went from unusable 0/100 output to a repeatable 55-75/100 range for JSON-mode structured saves.

### Structured saves now keep the details they were given

**Problem:** Structured save requests arrived with the right information but not in the shape the rest of the pipeline expected. JSON and STDIN payloads bypassed normalization, so the extractor chain saw empty input.

**Fix:** The workflow routes structured payloads through the existing normalization path before extraction begins.

### JSON-only saves can now tell a readable story

**Problem:** Structured saves without a transcript had no narrative material for the extractor chain, collapsing into thin placeholders with generic titles and summaries.

**Fix:** The extractor stack synthesizes a minimal structured conversation when prompts are absent, then derives title and summary fields from real session data instead of boilerplate.

### Decision lists are less repetitive

**Problem:** Decision strings could repeat themselves and file lists could become noisy enough to bury important references.

**Fix:** The phase reduced repeated decisions and tightened key-file shaping so saved memories stay easier to scan.

### Related sibling phases no longer look like contamination

**Problem:** The quality validator treated valid references to nearby child phases from the same workstream as foreign contamination, dragging scores down unfairly.

**Fix:** The validator relaxes this check for structured same-parent sibling references without creating a broad cross-spec bypass.

### Memory-save and shared-memory boundaries were hardened

**Problem:** Shared-memory admin operations lacked transport warnings, path handling needed stronger workspace boundary checks, and duplicate-preflight queries needed governed scope threading.

**Fix:** Trusted-transport warnings for admin paths, fail-closed behavior in the V-rule bridge, workspace boundary validation in shared path handling, and governed-scope handling with metadata redaction in duplicate-preflight logic were all added.

## Security (2)

### Governance became explicit instead of silently breaking normal search

**Problem:** A security control that defaults to "deny everything" can be correct in theory and wrong in practice when the system is not running in a governed environment. Normal unscoped queries were being rejected as unauthorized, making valid searches look empty.

**Fix:** Scope enforcement changed from silent default-on to explicit opt-in. Normal single-user search no longer gets treated like an access violation.

### Duplicate checks stay inside governed boundaries

**Problem:** Duplicate detection did not fully respect governed scope. One scope could learn that matching content existed somewhere it should not be able to inspect.

**Fix:** Duplicate checks now stay inside the caller's governed boundary with cross-boundary details redacted.

## Cross-Runtime Parity (4)

### Hybrid hook-plus-tool transport

**Problem:** A hook-only design would make the system feel great in Claude and second-class everywhere else.

**Fix:** Hooks were kept where they are strong and tool-driven recovery was added everywhere else. The implementation shares logic between Claude, Gemini, and hookless runtimes instead of forking the recovery model.

### Gemini hook porting

**Problem:** Gemini had no hook-driven session recovery, leaving it entirely dependent on manual priming or tool-based fallback.

**Fix:** Gemini startup brief wiring and hook parity work were added so Gemini sessions get startup structural context comparable to Claude sessions.

### Cross-runtime contract parity

**Problem:** Runtime instructions, context-prime agents, and bootstrap surfaces drifted apart. The same recovery concept had different wording and different guarantees depending on where the AI looked.

**Fix:** Non-hook recovery was standardized around explicit contracts: ready/stale/missing structural state, a canonical `session_bootstrap` path, and mirrored routing/recovery tables across all runtime-facing docs and agent definitions.

### Tool-routing enforcement

**Problem:** "Must use CocoIndex" style instructions were too passive. The AI often defaulted back to Grep/Glob even when the better tool was available.

**Fix:** Five layers of routing enforcement were added across server instructions, PrimePackage state, tool hints, runtime docs, and context-prime agents. The system now nudges tool choice where those decisions are actually made.

## Data Quality & Indexing (4)

### Compliance audit across 186 spec folders

**Problem:** Spec documents and memory files carried enough structural drift that indexing them cleanly was no longer safe. Frontmatter was broken, anchors were missing, and 46 memory files had hard-block failures.

**Fix:** The audit applied 2,081 frontmatter fixes, added 4,291 anchors, removed 46 hard-block memory files, cleaned orphaned vectors, and rebuilt the database from zero. Final status: 175/175 in-scope folders at 0 errors.

### Bulk reindex stopped rejecting good files

**Problem:** The reindex validator blocked 1,106 legitimate files because two validation rules used the wrong context. Batch inputs lacked file-path awareness, and the topic check rejected memory notes for describing outcomes in plain language instead of repeating spec names.

**Fix:** File-path context was threaded through the validator, topic checks were skipped for memory notes and spec-defining documents, and forced rebuilds now treat quality issues as warnings. Blocked files dropped from 1,106 to 0.

### Context types and schema rules were normalized

**Problem:** The memory system had alias drift — values like `decision` and `discovery` still existed where the current model expected `planning` and `general`. That drift produced validation noise and brittle migrations.

**Fix:** Canonical context types were established across parser logic, migrations, runtime consumers, and the database constraint. Schema version 25 migrates old values before enforcing the cleaned contract. 2,009 legacy values were migrated and 13,211 duplicate rows removed.

### Indexing and adaptive fusion stabilized after repository move

**Problem:** A repository move broke three search indexes — CocoIndex was pointed at stale state, BM25 needed a rebuild, and adaptive-fusion defaults were not equally visible across clients.

**Fix:** CocoIndex was rebuilt against the active repository (51,820 files / 663,336 chunks), BM25 was re-indexed against 1,228 memories, the code-graph DB gained first-use lazy initialization, and lexical provenance was preserved through fused output.

## Upgrade

No migration required. Schema changes are backward-compatible and run automatically on startup.

**Behavioral changes to be aware of:**

- `@spec-kit/shared` and `@spec-kit/mcp-server` are now native ESM. `@spec-kit/scripts` remains CommonJS by design.
- New `contextType` values should use the canonical set (`research`, `implementation`, `planning`, `general`), not legacy aliases.
- Access-scope enforcement is now opt-in. Multi-tenant deployments should enable it explicitly.
- Session recovery now branches by source (`compact`, `startup`, `resume`, `clear`) — each source type gets tailored context instead of a generic recovery brief.
- Code graph structural queries are available through `code_graph_query` and `code_graph_scan` MCP tools.
- Tool routing enforcement nudges semantic questions toward CocoIndex, structural questions toward the code graph, and session-history questions toward memory surfaces.
