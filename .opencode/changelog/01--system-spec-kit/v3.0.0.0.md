## [**3.0.0.0**] - 2026-03-27

This is the release that turns the spec kit from a collection of scripts and tools into a full documentation and memory platform. It completes a 19-phase program that rebuilt how the system stores, searches, and retrieves memories -- and then aligned every piece of documentation, every agent definition, and every command description to match what was actually built.

## What Changed: The Big Picture

The memory system now searches for information using 5 different methods in parallel, processes results through a 4-stage pipeline, and falls back gracefully when individual methods fail. The documentation surface grew to cover 255 features across 21 categories, 81 document templates, 33 tools, 14 commands, 22 scripts, and 20 validation rules.

### How search works now

The system finds relevant memories by running 5 search channels at the same time:

1. **Vector search** -- Converts your query into a mathematical representation and finds memories with similar meaning, even if they use different words. Supports multiple embedding providers (Voyage, OpenAI, HuggingFace).
2. **Full-text search (FTS5)** -- SQLite's built-in search that matches exact phrases and word combinations.
3. **Keyword search (BM25)** -- A ranking algorithm that scores results based on how important your search terms are within each document.
4. **Causal graph traversal** -- Follows cause-and-effect links between memories (e.g., "this decision led to that implementation").
5. **Graph importance scoring** -- Ranks memories by how connected they are to other memories, with limits to prevent overly-connected "hub" memories from dominating results.

Results from all 5 channels flow through a 4-stage pipeline:

1. **Gather** -- Runs all channels in parallel and injects "constitutional" memories (always-relevant rules that should appear in every search).
2. **Score and Fuse** -- Combines results using ranked fusion (a technique that merges ranked lists fairly) plus 8 additional scoring signals including how recently a memory was accessed, how related memories reinforce each other, and how well a result matches what the system thinks you are trying to do.
3. **Rerank** -- Optionally re-scores the top results using a local AI model for higher accuracy.
4. **Filter** -- Applies final checks: removes results below a confidence threshold, respects memory lifecycle states (active, archived, etc.), and ensures scores from earlier stages are not silently changed.

If the main pipeline fails, a 3-tier fallback kicks in, starting with simpler search methods and progressing to broader queries.

## What Changed: Phase by Phase

### 1. Phases 1-9: Building the search and storage engine

- **Phase 1 (program coordination)** -- Defined the overall architecture and established the 5-channel, 4-stage pipeline as the single production search path, replacing earlier isolated experiments.
- **Phase 2 (indexing normalization)** -- Fixed duplicate indexing problems. The same document could be indexed multiple times under different paths or broken into overlapping sections. Now each artifact is indexed exactly once.
- **Phase 3 (constitutional memory manager)** -- Rebuilt the `/memory:learn` command from a simple "create a rule" tool into a full manager with list, edit, remove, and budget controls. Constitutional memories get a 3x search boost and never decay over time.
- **Phase 4 (automated post-save hooks)** -- Added consistent automation after any memory change (save, update, delete, checkpoint). Cache invalidation, hooks, and safety checks now happen automatically instead of being scattered across different code paths.
- **Phase 5 (architecture audit)** -- Drew clear ownership boundaries between the different parts of the system (scripts, server, shared code, validation) to prevent them from tangling together as the platform grew.
- **Phase 6 (feature catalog)** -- Expanded and normalized the catalog of all 255 features across 21 categories. Added code-level trace markers so each feature can be tracked from documentation to implementation.
- **Phase 7 (code audit)** -- Audited 222 runtime features against the catalog until every single one matched its documentation. No partial or mismatched entries remain.
- **Phase 8 (database features)** -- Hardened multi-tenant database operations: scoped checkpoints per tenant, server-owned admin identity for shared memory spaces, and proper composite scoring in graph search results.
- **Phase 9 (session capturing)** -- Repaired the spec folder hierarchy and aligned session behavior with the cognitive model: attention scores decay over time, follow-up messages deduplicate roughly 50% of redundant content, and anchor-based retrieval can reduce payload sizes by about 93%.

### 2. Phases 10-19: Making the documentation match the code

- **Phase 10 (template compliance)** -- Built a two-layer template system (81 templates total) and aligned agent contracts with it.
- **Phase 11 (skill alignment)** -- Updated the skill documentation to reflect the real platform: 33 tools, 14 commands, shared-memory governance, query intelligence, and evaluation infrastructure.
- **Phase 12 (command alignment)** -- Rewrote command documentation to describe what was actually shipped instead of earlier simplified descriptions.
- **Phase 13 (agent alignment)** -- Updated agent definitions across all runtimes to work with the memory-first operating model.
- **Phase 14 (governance templates)** -- Synchronized repository-level governance text (gates, commands, rules) with what the system actually does.
- **Phase 15 (manual testing)** -- Completed manual verification: 231 scenario files, 272 individual test scenarios, all passing across 19 categories.
- **Phase 16 (MCP server documentation)** -- Rewrote the MCP server README as a proper platform reference covering all 33 tools, the search pipeline, evaluation harness (12 metrics, 110 test queries), and safety model.
- **Phase 17 (installation guide)** -- Updated setup documentation with current provider options, configuration paths, and rollback guidance.
- **Phase 18 (skill README)** -- Rebuilt the skill README with current platform counts instead of pre-program numbers.
- **Phase 19 (repository README)** -- Updated the top-level repository description to match the component READMEs.

### 3. Pre-release hardening

Before locking the release, the codebase went through multiple rounds of automated and manual review:

- A 20-iteration deep review cycle that caught correctness issues and documentation drift.
- A 5-agent parallel review that applied 59 fixes and resolved 135 pre-existing test failures.
- Three separate high-depth review passes that surfaced 32 additional findings.
- Recursive validation was driven from 13 errors down to 0 across the entire spec tree.
- All counts, descriptions, and platform claims were cross-checked until documentation and code told the same story.

### 4. Code changes

- **Dead code cleanup** -- Removed experimental modules that were never shipped, including an alternative fusion algorithm and a retention module. The codebase no longer contains code that looks like product behavior but is not used.
- **Bug fixes** -- Fixed quick-mode filter propagation, governance bypass via dry-run flags, file recovery for interrupted saves, write-before-commit ordering, and keyword-aware graph scoring.
- **Save quality improvements** -- Tightened the save pipeline with prediction-error gating (only saving information that adds value beyond what is already stored), structural validation, semantic sufficiency checks, and duplicate detection.
- **Shared memory security** -- Shared memory spaces are now deny-by-default (you must be explicitly granted access), with server-owned admin identity for administrative operations.

## Upgrade

1. Pull the latest changes and restart MCP services.
2. Review your embedding provider configuration (Voyage, OpenAI, or HuggingFace) and any shared-memory admin identity environment variables.
3. Run `npm test` in the memory server to verify the release baseline (target: 4,500+ tests passing).
4. No database migration is required, but familiarize yourself with the new search pipeline, fallback behavior, and deny-by-default shared memory model before treating this as a routine update.
