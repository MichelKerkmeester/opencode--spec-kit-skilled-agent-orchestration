## [**3.0.1.3**] - 2026-03-29

Post-audit deep research refinement: 28 fixes across concurrency safety, search performance, database query optimization, error recovery, and dead code removal. Test suite grew from 8,858 to 8,892 passing tests with no type errors.

A 5-iteration deep research re-examined the entire MCP server codebase after the 121-finding audit was closed. It found 28 additional refinement opportunities -- race conditions that only appear under concurrent load, database queries that scan more rows than necessary, error recovery paths that leave partial data behind, and dead code that adds maintenance burden. All 28 were fixed and verified.

## Concurrency Guards (4)

These four fixes address race conditions that only manifest when multiple operations happen simultaneously.

---

### Database restore could corrupt data if saves happened during rebuild

**Problem:** When you restore a checkpoint, the system performs a large restore transaction and then rebuilds several derived tables (search indexes, text indexes, graph caches). During that rebuild window -- which can take seconds -- nothing prevented normal operations like saving new memories, running index scans, or bulk deleting. If a save happened mid-rebuild, the newly saved data could reference structures that were about to be overwritten by the restore, leaving the database in an inconsistent state.

**Fix:** The restore process now sets a maintenance barrier before it begins and clears it only after all rebuilds complete. Any mutation operation (save, index scan, bulk delete) that arrives while the barrier is up gets an immediate, clear rejection (`E_RESTORE_IN_PROGRESS`) instead of silently proceeding into a half-rebuilt database. If the restore fails or crashes, the barrier is released in the cleanup handler so the system doesn't get permanently locked.

---

### Two people creating the same shared space could both become owners

**Problem:** When two requests to create the same shared space arrived at nearly the same time, both checked whether the space existed, both saw "no," and both proceeded to create it. Because of the timing, both would report `created: true` and both would bootstrap themselves as the space owner. The second caller would be incorrectly upgraded to owner of a space they didn't actually create.

**Fix:** The system no longer decides "was this a create or update?" based on what it read before writing. Instead, it uses a single atomic database operation (`INSERT ... ON CONFLICT DO NOTHING RETURNING`) that tells you whether a row was actually inserted. Owner bootstrapping only happens when the insert actually created the row. If it was a conflict (space already exists), the request is treated as an update with no owner promotion.

---

### Memory merge could use stale data if the original changed during processing

**Problem:** When merging two similar memories, the system reads the original memory, then sends it to an AI service for embedding generation (which involves a network call that can take seconds). After the embedding returns, the merge proceeds using the data it read earlier. But if another save modified or deleted that original memory during the embedding wait, the merge would overwrite the newer version with stale data -- or archive a record that no longer exists.

**Fix:** Before committing the merge, the system now reloads the original memory inside the transaction and compares its content hash and last-modified timestamp against what it read before the embedding call. If the original changed, the merge aborts cleanly with a `predecessor_changed` status. If the original was deleted or archived by another process, it aborts with `predecessor_gone`. In both cases, the save falls through to the normal (non-merge) path instead of forcing stale data.

---

### Two index scans could start at the same time

**Problem:** The system prevents index scans from running too frequently using a cooldown timer. But the check was a classic "time-of-check vs time-of-use" race: read the last scan time, check if enough time has passed, then update the timestamp. Two requests arriving in the same millisecond could both pass the check and both start scanning, wasting resources and potentially conflicting.

**Fix:** The cooldown is now an atomic lease. A single database transaction checks the timer and reserves the scan slot in one step -- if the lease is fresh, the request is rejected immediately. The lease includes a crash-expiry timer, so if a scan process dies mid-way, the slot is automatically freed after 2 minutes instead of being permanently locked.

## Search Performance (7)

These fixes reduce redundant computation in the search pipeline. No search *results* change -- the same queries return the same answers, just faster.

---

### Fallback search was doing expensive work multiple times

**Problem:** When a search doesn't find enough results, the system tries progressively looser criteria ("fallback tiers"). Each tier ran the *entire* search pipeline from scratch -- collecting candidates, fusing scores, reranking with the cross-encoder model, computing co-activation boosts, applying folder relevance, truncating by confidence, adding context headers, and trimming to token budget. The expensive post-processing stages (reranking, co-activation, token truncation) were running 2-3 times instead of once.

**Fix:** The pipeline is now split into two phases. Phase 1 (collect and fuse candidates) runs for each fallback tier, building up the candidate pool. Phase 2 (reranking, co-activation, folder scoring, confidence filtering, context headers, token truncation) runs exactly once on the final merged candidate set after the last tier is chosen.

---

### Response size was estimated twice per result using slow serialization

**Problem:** When deciding how many results fit in a response, the system estimates each result's "token cost." It was calling this estimator twice per result -- once to compute the total, then again when greedily admitting results. Each call used `JSON.stringify()` to serialize the result object, which is surprisingly expensive for complex objects with nested metadata.

**Fix:** Token estimates are now cached in a per-request map keyed by result ID. Each result is sized once and the cached value is reused. The estimator itself was rewritten to count known fields directly instead of serializing to JSON and measuring the string.

---

### In-memory text search engine ran by default even though a better alternative exists

**Problem:** The system maintains two text search engines: SQLite's built-in FTS5 (fast, always consistent with the database) and an in-memory BM25 index (slower, requires full rebuild at startup, can drift out of sync). Both ran by default. The BM25 engine performed a synchronous full table scan at startup to build its index, blocking the server from accepting requests.

**Fix:** BM25 is now opt-in behind an explicit `ENABLE_BM25` flag. FTS5 is the default and only text search engine unless you specifically enable BM25. When BM25 is enabled, its startup rebuild is now incremental (syncing only changed rows) and deferred to background warmup instead of blocking the main thread.

---

### Graph connectivity scoring made one database query per result

**Problem:** The "degree channel" ranks search results by how well-connected they are in the knowledge graph. For each candidate result, it ran a separate SQL query to count that node's connections. With 50 candidates, that's 50 queries. On a cold cache (first search after restart), this was the slowest part of the pipeline.

**Fix:** All candidate degree scores are now computed in a single SQL query that counts connections for all candidates at once. The global maximum degree (needed for normalization) is cached alongside the per-node cache, eliminating another query.

---

### Graph text search produced duplicates that required JavaScript deduplication

**Problem:** The graph's text search used an OR-based SQL join that found the same edge through multiple paths (matching on source side, target side, or both). The duplicates were removed in JavaScript after the query returned, wasting both database and application time.

**Fix:** The query was rewritten as a CTE (Common Table Expression) that finds matching memory IDs once, then uses UNION ALL to look up edges from both directions without overlap. No duplicates are produced, so no JavaScript dedup is needed. Whether the FTS table exists is also cached per database connection to skip the availability check on subsequent queries.

---

### Adaptive fusion computed standard results even when only weights were needed

**Problem:** The adaptive fusion system (which adjusts how much weight to give different search channels based on query type) always ran the standard fusion first, then the adaptive fusion, then compared them. But in production mode, only the adaptive weights are needed -- the standard fusion was wasted work.

**Fix:** A new lightweight helper (`getAdaptiveWeights()`) returns just the channel weights without running any fusion. The search pipeline now builds its fusion lists once using those weights, instead of running two full fusions and discarding one.

---

### Diversity reranking re-fetched embeddings it already had

**Problem:** The MMR (Maximal Marginal Relevance) diversity reranker needs the vector embeddings of each result to compute pairwise similarity. But the vector search channel had already retrieved these embeddings moments earlier. MMR was fetching them again from the database, one by one, using a linear scan (`reranked.find()`) to match results.

**Fix:** The vector search channel now passes its retrieved embeddings forward in a request-scoped cache. MMR checks the cache first and only queries the database for any embeddings that weren't already retrieved. The linear `find()` lookup was replaced with a Map for O(1) access.

## SQLite Query Optimization (6)

These fixes improve database query efficiency by adding targeted indexes and rewriting queries that couldn't use existing indexes.

---

### Duplicate detection queries couldn't use any index

**Problem:** Every time a memory is saved, the system checks three times whether it already exists (by path, by content hash, and by canonical path). These queries used nullable OR predicates like `(? IS NULL AND col IS NULL) OR col = ?` for optional scope columns (tenant, user, agent, session, shared space). SQLite's query planner can't use indexes with this pattern, so every check scanned the entire table. On a database with thousands of memories, this was the slowest part of every save.

**Fix:** The dedup queries are now dynamically assembled to include only the scope columns present in the request as exact-match conditions. The `(canonical_file_path = ? OR file_path = ?)` pattern was split into two separate probes. Two new partial indexes were added: one for canonical-path lookups and one for content-hash lookups with scope columns, both filtered to non-chunk rows.

---

### Trigger cache reload scanned the entire memory table

**Problem:** After every mutation (save, delete, update), the trigger-phrase cache was rebuilt by querying all successfully embedded memories that have trigger phrases. Without a targeted index, this was a full table scan every time.

**Fix:** A new partial index covers exactly the rows the trigger cache needs: successfully embedded memories with non-empty trigger phrases. The prepared SQL statement is also cached per database connection so it doesn't need to be recompiled on each reload.

---

### Co-activation boost made one query per related memory

**Problem:** The co-activation system (which boosts results that are related to other high-scoring results) fetched details for related memories one at a time. The stage-2 fusion pipeline called `getRelatedMemories(row.id).length` for each boosted row, generating another round of individual queries. With 20 boosted rows having 5 related memories each, that's 120 queries.

**Fix:** Related memory details are now fetched in a single batch query using `WHERE id IN (...)`. The causal-neighbor lookup was rewritten as a single SQL JOIN instead of separate queries per direction. Stage-2 fusion precomputes all neighbor counts in one batch before the boosting loop.

---

### Temporal proximity search wrapped dates in functions, defeating indexes

**Problem:** The temporal-contiguity system finds memories created near the same time. Its query used `ABS(strftime('%s', created_at) - strftime('%s', ?))` in the WHERE clause, which wraps the indexed column in a function. SQLite can't use an index when the column is inside a function call, so every temporal proximity search scanned the entire table.

**Fix:** The query now uses a bounded range on the raw `created_at` column (`created_at BETWEEN ? AND ?`) to narrow results first, then computes the time delta only on the narrowed result set. A new composite index on `(spec_folder, created_at DESC)` makes the range scan fast.

---

### Causal link resolution used leading-wildcard LIKE patterns

**Problem:** When resolving references between memories (e.g., "this memory mentions that file path"), the system used `LIKE '%path%'` patterns that start with a wildcard. SQLite can't use any index for leading-wildcard matches, so every reference resolution scanned the entire table. With documents that reference many other files, this was slow.

**Fix:** Resolution now tries exact matches on `canonical_file_path` and `file_path` first (which can use indexes). Only if the exact match fails does it fall back to fuzzy LIKE matching. All references in a document are resolved in a single batch instead of one at a time.

---

### Working memory queries were missing order-aligned indexes

**Problem:** The working memory system (which tracks recently active memories per session) queries by session ID with various sort orders (by last-focused time, by attention score). Without matching indexes, these queries required SQLite to sort results in memory. The `upsertExtractedEntry()` function also ran a `SELECT COUNT(*)` before every upsert just to check if the row existed, even though the subsequent `ON CONFLICT` clause handles both cases.

**Fix:** Two new indexes align with the working memory query patterns: one for LRU eviction order and one for attention-score ranking. The unnecessary existence check was removed -- the `ON CONFLICT` clause already handles both insert and update.

## Error Recovery Hardening (4)

These fixes close gaps where a failure partway through an operation could leave the database in an inconsistent state.

---

### Failed supersede after chunking left orphaned data

**Problem:** When saving a large file, the system splits it into chunks and indexes each one. After all chunks are committed to the database, it marks the old version as superseded. But if the "mark as superseded" step failed, the handler returned an error -- even though the new chunks were already committed. The caller would see "save failed" and retry, creating a second copy of the chunks. Meanwhile, the old version was never marked as superseded.

**Fix:** The supersede logic now runs inside the same finalize transaction as the chunk commit. If the supersede fails, compensating cleanup deletes the newly created parent and child rows before returning the error. The caller sees a clean failure with no orphaned data, and retrying works correctly.

---

### Old chunk children were deleted outside the safety of a transaction

**Problem:** When replacing an existing chunked memory with a new version (a "safe swap"), the old children were deleted as a separate operation after the transaction committed. If the delete failed (e.g., disk full, process killed), the old children would remain as orphaned records alongside the new ones.

**Fix:** Old-child deletion is now part of the finalization transaction. The children's parent references are nulled first (making them orphans within the transaction), then they're bulk-deleted. If anything fails, the entire transaction rolls back, leaving the old version intact.

---

### Failed chunking left the text search index in a wrong state

**Problem:** When chunking a large file, the system updated the text search index (BM25) with the new parent summary immediately. If all chunks subsequently failed to index, the system correctly rolled back to the old parent -- but the text search index still had the new summary. The old parent's content was invisible to word-based searches until the next full rebuild.

**Fix:** The text search index update is now delayed until at least one chunk succeeds. If all chunks fail and the system falls back to the old parent, the text search index was never modified, so it stays correct automatically.

---

### Failed text search repair after merge was lost permanently

**Problem:** When merging two memories, the system commits the merged record to the database and then updates the text search index. If the text search update fails (e.g., corrupted index state), the merge is committed but the merged memory can't be found by word-based searches. The failure was logged but not recorded anywhere durable, so no future process could retry the repair.

**Fix:** A new `bm25_repair_needed` flag is persisted on the merged row when the text search repair fails. This flag survives server restarts and can be picked up by a future reconciliation process or manual repair tool.

## Dead Code Cleanup (7)

These changes remove unused code that increases maintenance burden and confuses code navigation.

---

### Startup warmup code that never ran

**Problem:** The server startup had a code branch for "eager warmup" of embedding models, guarded by a `shouldEagerWarmup()` function. But that function was hardcoded to return `false`. The warmup code was unreachable dead code.

**Fix:** Both the dead code branch and the always-false flag function were removed.

---

### Exported types that nothing imported

**Problem:** The `tools/types.ts` file exported `MCPResponseWithContext` (an interface) and `parseValidatedArgs` (a function). A search across the entire codebase found zero imports of either.

**Fix:** Both exports were removed.

---

### Handler barrel exported modules that nothing used

**Problem:** The handler barrel file (`handlers/index.ts`) exported 9 lazy-loaded module proxies (`memorySearch`, `memoryTriggers`, `memoryCrud`, `memoryIngest`, `sessionLearning`, `causalGraph`, `evalReporting`, `memoryContext`, `sharedMemory`). All actual handler usage goes through the named function exports, not through these module proxies.

**Fix:** All 9 unused proxy exports were removed. The named function exports that are actually used remain unchanged.

---

### Debug exports left over from development

**Problem:** `getLastDegradedState()` in the trigger matcher and `_resetInitTracking()` in shadow scoring were exported for debugging during development but never called from any production or test code.

**Fix:** Both exports were removed.

---

### Type exports with no consumers

**Problem:** Three type exports had zero imports across the codebase: `PipelineOrchestrator` from pipeline types, `InterferenceResult` from interference scoring, and `SurrogateMatchResult` from query surrogates.

**Fix:** All three type exports were removed.

---

### Test file with non-standard naming required a redirect shim

**Problem:** The shared memory handler tests were in a file named `shared-memory-handlers.test-suite.ts` instead of the standard `.vitest.ts` extension. A one-line shim file (`shared-memory.vitest.ts`) existed solely to re-export the test suite so Vitest could find it.

**Fix:** The test file was renamed to `shared-memory-handlers.vitest.ts` (standard naming). The shim file was deleted.

---

### Three copies of the same score-resolution function

**Problem:** The function `resolveEffectiveScore()` (which determines which score field to use for a search result) was defined in three different files: `pipeline/types.ts` (canonical), `confidence-scoring.ts` (copy), and `profile-formatters.ts` (copy). All three did the same thing.

**Fix:** The copies in `confidence-scoring.ts` and `profile-formatters.ts` were replaced with imports from the canonical version in `pipeline/types.ts`.

## Upgrade

No migration required. Schema changes are backward-compatible and run automatically on startup.

**One behavioral change to be aware of**:
- **BM25 is now opt-in.** If you were relying on the in-memory BM25 text search engine (in addition to FTS5), you need to set `ENABLE_BM25=true` to keep it active. Most users won't notice this change because FTS5 (the default) handles the same queries. BM25 is only needed for specific benchmarking or experimental scenarios.
