## Critical Fix

---

### Database restore could erase other tenants' data when filtering by folder

**Problem:** When restoring a checkpoint that was created for a specific spec folder, the system would delete all data matching that folder -- regardless of which tenant, user, or agent owned it. If Tenant A and Tenant B both had memories in the same spec folder, restoring Tenant A's checkpoint would wipe Tenant B's data too. The same issue affected the "clear before restore" path and the "merge restore" path.

**Fix:** Checkpoint restore now intersects the spec_folder filter with the caller's tenant/user/agent/sharedSpace scope. The clear path adds scope predicates alongside the folder predicate. Three new tests verify: scoped restore preserves other tenants' rows, unscoped restore still works folder-wide, and merge-mode respects scope boundaries.

## Required Fixes

---

### Search could return zero results when all matches exceeded the token budget

**Problem:** The token-budget truncation function uses a greedy loop to fit results within the allowed size. When every candidate exceeded the budget, the loop would accept zero results and return an empty array -- making it look like nothing matched, even though relevant results existed.

**Fix:** When the greedy loop accepts zero results but candidates exist, the system now keeps the highest-ranked candidate with overflow metadata, matching the existing single-result fallback pattern.

---

### Two concurrent saves could overwrite each other

**Problem:** The atomic save function renamed a pending file to its final path before acquiring the spec-folder mutex. A second concurrent save could rename its own pending file to the same path in the gap between rename and lock acquisition, causing one save to silently overwrite the other.

**Fix:** The spec-folder lock is now acquired before the file rename. Both the rename and the database mutation happen inside the same critical section. Rollback stays within the locked section too.

---

### Duplicate detection could miss in-scope matches in multi-tenant databases

**Problem:** The pre-existing duplicate detector (PE gating) fetched the top N*3 vector matches and then filtered by governance scope. In a noisy multi-tenant database, all top matches could belong to other tenants, causing the system to miss the real in-scope duplicate.

**Fix:** The vector search now uses a paging loop that expands the window (N*3, N*6, N*12, ...) until enough scoped matches are found or the candidate pool is exhausted.

---

### Deferred chunk children could overwrite each other's index entries

**Problem:** When a large document is split into chunks and some chunks are deferred for later processing, the deferred path omitted the anchor identity. All deferred children shared a null-anchor key, causing later chunks to overwrite earlier ones in the active projection table.

**Fix:** The deferred path now passes `anchorId: chunk.label` to `indexMemoryDeferred`, matching the embedded (immediate) path.

---

### Anchor extraction could return corrupted content from malformed documents

**Problem:** The anchor extraction function (regex-based) and the anchor validation function (stack-based) were independent. A document with crossed or unclosed anchors would be flagged as invalid by validation but still processed by extraction, returning wrong content boundaries.

**Fix:** `extractAnchors()` now calls `validateAnchors()` first. If validation detects structural errors, extraction returns an empty map instead of corrupted content.

---

### Graph-signal cache already invalidated from prior sprint

**Problem:** The meta-review flagged three files that mutate causal edges directly without clearing the graph-signal cache. Investigation showed this fix was already implemented in Sprint 4 across all three mutation paths (vector-index-mutations, corrections, graph-lifecycle).

**Fix:** No additional code changes needed. Confirmed 51/51 graph-signal tests pass.

---

### Shared-memory actor IDs accepted blank values

**Problem:** The shared-memory handler trusts actor identity from tool arguments. While full server-bound authentication is out of scope for this MCP server architecture, blank or whitespace-only actor IDs were silently accepted, making impersonation trivially easy.

**Fix:** Input validation now rejects empty/whitespace actor IDs with a structured error. A code comment documents the trust boundary and recommends authenticated transport middleware for untrusted environments.

---

### Constitutional memory cache leaked across database switches

**Problem:** The constitutional memory cache keyed entries only on (spec_folder, includeArchived). After switching to a different database via `set_active_database_connection`, cached results from the old database could appear in queries against the new one.

**Fix:** Cache entries now include the active database path in their key. The entire cache is cleared when the active connection changes.

---

### Schema DDL was duplicated in three places

**Problem:** The `memory_conflicts` table DDL and companion indexes were defined identically in the helper function, migration v4, and bootstrap. Changes to the table shape required updating all three locations in lockstep.

**Fix:** Bootstrap and migration v4 now call the shared `createMemoryConflictsTable()` and `createMemoryConflictIndexes()` helpers instead of inlining DDL.

---

### Fallback search policy was duplicated in two functions

**Problem:** The fallback ladder (threshold constants, channel overrides, retry markers) was implemented identically in both `searchWithFallback()` and `collectRawCandidates()`. Changes to fallback behavior required updating both in lockstep.

**Fix:** Extracted a shared `executeFallbackPlan()` that encapsulates the threshold constants, channel overrides, and retry tagging. Both callers consume staged results and decide their own post-processing.

---

### Documentation drift corrected across 5 spec files

**Problem:** Seven documentation P1s found claims in spec.md, plan.md, checklist.md, implementation-summary.md, and tasks.md that no longer matched the actual codebase state after 4 fix sprints and 2 triage passes.

**Fix:** Updated iteration counts (20 to 30), scope descriptions, threshold unit claims, file attributions, test count progressions, finding table gap notation, and Known Limitations status.

## Improvements

---

### Embedding cache primary key now includes dimensions

Two dimension variants for the same content/model can now coexist without one overwriting the other. Already present from a prior sprint; confirmed with 14 tests.

---

### Learning history normalizes session ID on read

`handleGetLearningHistory` now normalizes the sessionId before querying, matching the write-side behavior. Prevents misses when the caller passes a padded session ID.

---

### Checkpoint scope rejects whitespace-only tenant ID

`validateCheckpointScope` now trims and rejects empty-string tenantId when any actor or shared-space field is present.

---

### Lineage row loading deduplicated into shared helper

Extracted `loadLineageRowsForMemory()` and reused it across `inspectLineageChain`, `summarizeLineageInspection`, and the active/as-of resolution functions. Reduces code duplication in lineage-state.ts.

---

### Rollback errors preserved as structured metadata

Rollback/cleanup catch blocks now preserve the original error as structured metadata (`rollbackError`, `pendingCleanupError`) instead of reducing to a boolean. Surfaced in warning/error responses for debugging.

---

### Database rebind failure now logged

The listener path in db-state.ts now checks the boolean return of `rebindDatabaseConsumers` and logs an error if it returns false, instead of silently discarding the result.
