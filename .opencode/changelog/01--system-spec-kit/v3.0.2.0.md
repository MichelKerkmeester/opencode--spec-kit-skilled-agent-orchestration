## [**3.0.2.0**] - 2026-03-28

Following the v3.0.1.0 release that fixed all 80 critical and required findings, this release tackles the remaining 41 lower-priority improvement findings from the same code audit. Of those 41: 22 were fixed with code and tests, 16 were deferred because they require coordinated changes across multiple system components, and 3 turned out to not be real issues upon closer inspection. Test suite grew from 8,748 to 8,771 passing tests.

## Saving and Data Integrity (7 findings: 6 fixed, 1 rejected)

### 1. Dry-run mode was writing to the evaluation database

**Problem:** "Dry run" is supposed to preview what a save would do without side effects. But it was still logging metrics to the evaluation database, polluting benchmark data with synthetic entries.

**Fix:** Dry-run paths now explicitly suppress all evaluation logging.

### 2. Anchor auto-fix miscounted mismatches

**Problem:** The quality checker correctly identified when anchor tags (markers that define sections in documents) were opened multiple times but closed only once. But the auto-fixer only checked whether *any* closing tag existed, not whether there were *enough* of them. Multiple opens with one close stayed broken after "repair."

**Fix:** The fixer now counts missing closing tags per anchor name and adds exactly the right number.

### 3. Split-brain save window -- REJECTED

Not a real issue. Code inspection confirmed this was already fixed in v3.0.1.0.

### 4. Deleting a memory could leave orphaned vector data

**Problem:** When deleting a memory, if cleaning up the vector embeddings (the numerical data used for similarity search) failed, the system logged a warning and continued deleting the main record. This left orphaned vector data that would never be cleaned up and could cause stale search matches for deleted memories.

**Fix:** Vector cleanup failures now abort the entire delete transaction. The main record is only deleted if vector cleanup succeeds.

### 5. File changes between validation and save could cause stale indexing

**Problem:** The save process parsed a memory file during initial validation, then reused that parsed data later when actually saving. If the file changed between those two moments (e.g., from a concurrent editor), the indexed content wouldn't match the file on disk.

**Fix:** The save process now re-checks the file's modification time and size inside the write lock, and re-parses if anything changed.

### 6. Content hash got out of sync during backfill

**Problem:** When the system backfilled a missing content column for an existing memory, it updated the content but left the content hash (a fingerprint used for duplicate detection) pointing at the old content. This could cause false duplicate matches or missed updates.

**Fix:** The content hash is now recomputed and stored in the same operation as the content backfill.

### 7. Hash-based duplicate detection was too trusting

**Problem:** Duplicate detection relied entirely on matching content hashes (fingerprints). While hash collisions are astronomically unlikely, stale or corrupted hash data from prior bugs could cause the system to incorrectly flag a new memory as a duplicate of an unrelated one.

**Fix:** After a hash match, the system now also compares the actual content. If the content doesn't match, the hash hit is downgraded to a warning and processing continues normally.

## Search and Query Pipeline (7 findings: 3 fixed, 4 deferred)

### 8. Word-matching and full-text search handled punctuation differently

**Problem:** The memory search system has two text-based search channels: BM25 (word matching) and FTS5 (full-text search via SQLite). They each had their own way of handling punctuation, path separators, and code-style text. A query like `lib/search/hybrid-search.ts` would be split into different tokens by each channel, producing inconsistent rankings.

**Fix:** Both channels now use a shared text normalizer before their channel-specific processing, ensuring consistent tokenization.

### 9. Empty anchor chunks wasted search result slots

**Problem:** The chunk-thinning system (which decides which pieces of a split document to keep) gave high scores to chunks containing anchor markers, even if those chunks contained nothing but the markers themselves. These empty-but-high-scoring chunks took up search result slots that should have gone to chunks with actual content.

**Fix:** Anchor-containing chunks now must have a minimum amount of real content (at least 40 non-whitespace, non-marker characters) to receive the anchor score boost. Pure-marker chunks are thinned away.

### 10. Short queries lost their best search channel

**Problem:** The search system has 5 different ways to find memories. One of them (BM25, which matches exact words) was being turned off for short queries like "find spec 026" as a cost optimization. But short, specific lookups are exactly the kind of query where exact word matching works best. Disabling it forced these through similarity-only search, which is weaker for exact ID and title matching.

**Fix:** The query router now detects when a short query is looking for a specific document or spec and keeps BM25 enabled for those cases.

### Deferred (4)

Multi-facet query decomposition, traversal snapshot consistency, high-degree truncation signaling, and stale degree cache lifecycle were all valid but require coordinated changes across multiple modules.

## Configuration, Parsing, and Feature Flags (9 findings: 3 fixed, 5 deferred, 1 rejected)

### 11. Feature flags were frozen at startup

**Problem:** Several feature flags (on/off switches controlled by environment variables) were read once when the server started and stored as constants. Changing the environment variable while the server was running had no effect -- the old value was frozen. This defeated the purpose of using environment variables for runtime configuration.

**Fix:** All three affected flags now read the environment variable on each use instead of once at startup, so changes take effect without a server restart.

### 12. Files saved as UTF-16 without a byte-order mark were corrupted

**Problem:** The memory parser detected text encoding by looking for a byte-order mark (a special marker at the start of the file that says "this file is UTF-16"). Files saved as UTF-16 *without* this marker (common from certain Windows editors and PowerShell) fell through to the default UTF-8 decoder, corrupting every other character into a replacement symbol. The corrupted text was then indexed, making the memory unsearchable.

**Fix:** Added a detection heuristic: if the first 512 bytes show a pattern of alternating content and null bytes (characteristic of UTF-16), the parser decodes accordingly even without a byte-order mark.

### 13. Special characters in names could make identity keys ambiguous

**Problem:** The logical key (which uniquely identifies a memory's version history) was built by joining fields with `::` as a separator. If any field contained `::` in its value (e.g., a namespace like `module::init`), the resulting key was ambiguous -- the system couldn't tell where one field ended and the next began.

**Fix:** Normal keys keep the human-readable format. When a collision is detected (any field contains `::`), the key switches to an unambiguous hashed format.

### Rejected: Session learning scheduling

The finding suggested the session learning module should implement spaced-repetition scheduling. But that scheduling already exists in a separate dedicated module. The session learning module handles measurement, not scheduling -- its name is broader than its scope, but the behavior is correct.

### Deferred (5)

Concurrent supersede serialization, graph walk documentation, shared-memory defaults documentation, shared-memory enablement auth, and late spec-folder discovery were all valid but require broader changes.

## Evaluation, Merging, and Interference (9 findings: 6 fixed, 3 deferred)

### 14. Interference scoring counted archived memories

**Problem:** The interference scorer penalizes memories that have too many similar siblings in the same folder (to prevent one topic from dominating results). But it was counting archived and deprecated memories that were already excluded from search results. After a merge operation, the archived originals still counted as siblings, unfairly penalizing the surviving merged memory.

**Fix:** Interference scoring now only counts active, retrievable memories.

### 15. Batch processing had no size limit

**Problem:** The batch processor accepted arbitrarily large batch sizes from an environment variable. An oversized value (e.g., 10,000) could cause the system to process thousands of files simultaneously, exhausting memory or overwhelming the database.

**Fix:** Batch size is now capped at 200 items regardless of the configured value.

### 16. Duplicate file paths could be queued for processing

**Problem:** The file ingest system normalized paths for OS compatibility but didn't check for duplicates after normalization. Two paths that point to the same file (like `./spec.md` and `spec.md`) would both be queued, causing duplicate processing and potential conflicts.

**Fix:** Paths are deduplicated after normalization. The response includes a count of removed duplicates.

### 17. Zero-token benchmarks were recorded as real measurements

**Problem:** When benchmark runs had no actual token usage data (the common case for in-process evaluation), the system recorded `0/0/0` as real measurements. Downstream reporting treated these zeros as actual data, skewing aggregate calculations.

**Fix:** The system now skips recording token usage entirely when no real measurement exists, instead of writing misleading zeros.

### 18. "Most recent" sprints weren't actually the most recent

**Problem:** When limiting the dashboard to "the 5 most recent sprints," the system ranked by when each sprint started rather than when it was last active. An old sprint with recent activity could be dropped in favor of a newer but inactive one.

**Fix:** Sprint selection now ranks by most recent activity. The display still shows sprints in chronological order.

### 19. Missing benchmark test IDs were silently skipped

**Problem:** When running benchmarks against specific test queries, any query IDs not found in the reference dataset were silently filtered out. You could run a benchmark against a stale ID list without realizing half your test cases were being skipped.

**Fix:** Missing IDs now generate a structured warning in the report. If *all* requested IDs are missing, the benchmark fails with a clear error.

### Deferred (3)

BM25 reconsolidation atomicity, Unicode tokenization, and trigger scan performance were all valid but require broader changes.

## Schema, Response Format, and API Surface (9 findings: 4 fixed, 4 deferred, 1 rejected)

### 20. Database caches survived database swaps

**Problem:** After a database swap (e.g., restoring from a checkpoint), internal caches still held data from the previous database. Queries would return stale results until the caches naturally expired or the server restarted.

**Fix:** Cache invalidation now triggers automatically whenever a database change is detected.

### 21. Quick-search responses were mislabeled

**Problem:** The quick-search tool reused the regular search tool's response handler, which stamped responses with the wrong tool name. Any monitoring or debugging that tracked tools by name would misattribute quick-search traffic.

**Fix:** The response envelope now correctly identifies the originating tool.

### 22. API schema said "provide one," runtime said "provide exactly one"

**Problem:** The published API schema for shared-memory tools said both user ID and agent ID were optional. But the actual runtime validation required exactly one (not both, not neither). Clients performing pre-validation against the schema wouldn't catch the error until their request was rejected at runtime.

**Fix:** The schema now encodes the "exactly one" constraint, matching what the runtime actually enforces.

### 23. API schema was looser than runtime validation

**Problem:** Several runtime validation rules (minimum string lengths, numeric ranges, non-empty requirements) existed only in the code but weren't reflected in the published API schemas. Clients relying on the schema for validation or documentation saw a looser contract than what the system actually enforced.

**Fix:** All missing constraints were added to the published schemas to match runtime behavior.

### 24. A declared API parameter was silently ignored

**Problem:** The context tool's API schema declared a `profile` parameter, but the internal type definition didn't include it. The parameter was accepted by the schema but silently dropped during processing, so it never had any effect.

**Fix:** The parameter is now properly threaded through the internal type system and forwarded to all downstream calls.

### 25. Response size estimation undercounted

**Problem:** The response envelope calculated its size estimate from only the data payload, ignoring the metadata, pagination info, and other envelope overhead. Downstream budget enforcement would undercount the actual cost of responses.

**Fix:** Size estimation now includes the full serialized response, not just the data portion.

### Rejected: Split-brain save window -- already fixed in v3.0.1.0.

### Deferred (4)

Embedding timeout handling, a response-shape breaking change for auto-surfaced context, and handler import optimization were all valid but require broader changes.
