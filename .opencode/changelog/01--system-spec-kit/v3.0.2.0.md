## P2 Fixes — Save Pipeline & Transaction Safety (7 findings → 6 fixed, 1 rejected)

### 1. Dry-run eval mutation suppressed (`memory-save.ts`, `quality-loop.ts`) — P2-001

**Problem**: Dry-run mode was supposed to be side-effect free, but both dry-run code paths still invoked the quality loop's eval logger, causing `eval_metric_snapshots` table inserts during what should have been a read-only operation. This meant dry-run calls polluted eval metrics with synthetic data, skewing ablation results.

**Fix**: Patched dry-run paths to pass an explicit `suppressEvalLogging` flag through the quality-loop API. The quality loop now respects this flag and skips all eval-metric emission when set. Three new tests verify dry-run suppression across both save paths.

### 2. Anchor auto-fix count mismatch repaired (`quality-loop.ts`) — P2-002

**Problem**: The quality scorer correctly counted repeated anchor names (e.g., three `<!-- ANCHOR:metadata -->` opens with only one close), but the auto-fixer only checked for the *existence* of a closing tag, not the *count*. This meant repeated opens with a single close stayed broken after auto-fix, leaving orphaned anchor regions in spec documents.

**Fix**: The fixer now counts missing closers per anchor name using the same approach as the scorer, and appends the exact number of missing `<!-- /ANCHOR:name -->` tags. Updated tests verify multi-open scenarios are fully repaired.

### 3. Split-brain save window — P2-003 — **REJECTED**

**Reason**: Code inspection confirmed the current implementation already returns `success: false` plus `dbCommitted: true` when the file rename fails after DB commit. The finding described a stale code path that was already fixed in the P1 sprint (T027). No action needed.

### 4. Vector delete atomicity enforced (`vector-index-mutations.ts`) — P2-004

**Problem**: Delete flows treated `vec_memories` cleanup failures as warnings and continued committing the primary `memory_index` row deletion. This left orphaned vector embeddings in the secondary table that would never be garbage-collected, causing search drift as stale vectors continued to match queries for deleted memories.

**Fix**: Non-legacy vector deletion failures are now fatal inside the delete transaction — if `vec_memories` cleanup fails, the entire transaction rolls back. The legacy missing-table exemption is preserved for backward compatibility with pre-vector-index databases. New test verifies rollback behavior.

### 5. Stale parsed state re-evaluated under lock (`memory-save.ts`) — P2-028

**Problem**: The handler parsed the memory file once during preflight validation, then reused that stale parsed object later during the locked save phase. If the file changed between preflight and lock acquisition (e.g., from a concurrent editor or file-watcher), the indexed content wouldn't match the actual file on disk.

**Fix**: The save path now re-stats and reparses the memory file inside the folder lock whenever a preflight parse is being reused. If the file's mtime or size changed, the re-parsed content replaces the stale state. Two new tests verify re-parse triggers on file mutation.

### 6. PE reinforcement hash synchronization (`pe-gating.ts`) — P2-036

**Problem**: When PE reinforcement backfilled a missing `content_text` column, it left `content_hash` unchanged. Downstream change detection and hash-based dedup assumed the hash always matched the stored content, so the stale hash could cause false-positive duplicate rejection or missed content updates.

**Fix**: Reinforcement now updates `content_hash` in the same SQL UPDATE statement when it backfills content. The hash is computed from the same normalized text that gets stored. Test verifies hash/content synchronization.

### 7. Hash match secondary verification (`dedup.ts`, `preflight.ts`) — P2-037

**Problem**: Both the dedup checker and preflight validator trusted raw SHA-256 hash matches blindly. While SHA-256 collisions are astronomically unlikely, stale or corrupted hash metadata (e.g., from a prior bug or manual DB edit) could cause the system to incorrectly report a new memory as a duplicate of an unrelated one.

**Fix**: Dedup and preflight now verify the matched row's `content_text` against the incoming content after a hash hit. If the stored content doesn't match, the hash hit is downgraded to a "stale hash" warning and processing continues as if no match was found. New `content-hash-dedup.vitest.ts` test file covers both matching and mismatching scenarios.

## P2 Fixes — Search & Query Pipeline (7 findings → 3 fixed, 4 deferred)

### 8. Shared FTS/BM25 lexical normalization (`bm25-index.ts`, `sqlite-fts.ts`) — P2-007

**Problem**: The BM25 channel tokenized raw query text using its own word-boundary splitter, while SQLite FTS5 used a separate sanitizer with different punctuation handling. Path-style queries (e.g., `lib/search/hybrid-search.ts`) and code-style queries (e.g., `buildLogicalKey()`) would split differently across the two lexical channels, producing inconsistent ranking — one channel would match tokens the other missed entirely.

**Fix**: Added a shared `normalizeLexicalQuery()` function that both channels call before their channel-specific processing. The normalizer handles path separators, code punctuation, and camelCase boundaries uniformly. Existing BM25 and FTS5 tests updated to verify normalization consistency.

### 9. Anchor-only chunk density gate (`chunk-thinning.ts`) — P2-008

**Problem**: The thinning algorithm assigned `ANCHOR_WEIGHT = 0.6` to chunks containing anchors, which meant an anchor-only chunk (containing just `<!-- ANCHOR:name -->` and `<!-- /ANCHOR:name -->` with no meaningful content between them) started at a 0.6 composite score — easily clearing the 0.3 retention threshold. These noise chunks consumed retrieval slots without providing useful context.

**Fix**: Added an anchored-signal floor: anchor-containing chunks now require minimum character density (at least 40 non-whitespace, non-anchor characters) AND minimum meaningful content length before anchor weight applies. Chunks failing this floor are scored without anchor bonus. Tests verify that pure-anchor chunks are now thinned away while content-bearing anchored chunks retain their boost.

### 10. Router preserves BM25 for artifact lookups (`query-router.ts`) — P2-018

**Problem**: The query router classified short queries as `simple` complexity tier and disabled BM25 as a cost optimization. But short spec/decision lookups like "find spec 026" or "get decision record" are precisely the queries that benefit most from BM25's exact-match strength — stripping BM25 forced these through vector-only retrieval which is weaker for exact title/ID matching.

**Fix**: The router now checks for `find_spec` / `find_decision` intent classification and doc-like artifact class indicators before pruning channels. Short queries with these signals retain BM25. Tests verify channel preservation for artifact-intent short queries.

### Deferred: P2-019 Multi-facet query decomposition, P2-005 Traversal snapshot consistency, P2-006 High-degree truncation signaling, P2-027 Stale degree cache lifecycle

These four findings are valid but require changes to `hybrid-search.ts`, `handlers/causal-graph.ts`, `lib/storage/causal-edges.ts`, and `core/db-state.ts` respectively — files that span multiple module boundaries and need coordinated contract changes.

## P2 Fixes — Config, Parsing & Flags (9 findings → 3 fixed, 5 deferred, 1 rejected)

### 11. Runtime flag resolvers replace import-time snapshots (`co-activation.ts`, `corrections.ts`, `stage2-fusion.ts`) — P2-013

**Problem**: Several SPECKIT feature flags were captured as module-level constants at import time (e.g., `const CO_ACTIVATION_ENABLED = process.env.SPECKIT_CO_ACTIVATION === 'true'`). In a long-running MCP server process, changing an environment variable after startup had no effect — the flag was frozen to its initial value. This defeated the purpose of environment-based feature gating for hot-reload scenarios.

**Fix**: Replaced all three import-time snapshots with getter functions that read `process.env` on each call:
- `co-activation.ts`: `getCoActivationEnabled()` and `getCoActivationStrength()` replace frozen `CO_ACTIVATION_ENABLED` / `CO_ACTIVATION_STRENGTH`
- `corrections.ts`: `getRelationsGatingEnabled()` replaces frozen `RELATIONS_GATING`
- `stage2-fusion.ts`: `getFusionCoActivationWeight()` replaces frozen `CO_ACTIVATION_FUSION_WEIGHT`

### 12. BOM-less UTF-16 detection heuristic (`memory-parser.ts`) — P2-014

**Problem**: The memory parser's encoding detection only checked for byte-order marks (BOM). Files saved as UTF-16LE or UTF-16BE without a BOM (common from certain Windows editors and PowerShell output) fell through to the UTF-8 fallback, corrupting every other byte into replacement characters (U+FFFD). The corrupted content was then indexed, making the memory unsearchable and unreadable.

**Fix**: Added a parity/null-byte heuristic that runs before the UTF-8 fallback: if the first 512 bytes show a pattern of alternating null/non-null bytes, the parser detects BOM-less UTF-16LE (null in odd positions) or UTF-16BE (null in even positions) and decodes accordingly. If neither pattern matches, UTF-8 remains the fallback. The heuristic is conservative — it requires at least 60% of the check window to match the alternating pattern to avoid false positives on binary files.

### 13. Lineage delimiter collision hashing (`lineage-state.ts`) — P2-009

**Problem**: `buildLogicalKey()` constructed keys by concatenating `spec_folder::canonical_path::anchor` with `::` as delimiter. If a spec folder name or anchor contained `::` (e.g., a namespace-scoped anchor like `module::init`), the resulting key was ambiguous — the parser couldn't distinguish between `folder::path::anchor::name` meaning anchor="anchor::name" vs folder="folder::path" with anchor="name". The code had a warning comment acknowledging this but persisted the ambiguous key anyway.

**Fix**: The legacy readable format is preserved for normal (non-collision) rows. When a collision is detected (any field contains `::`), the key switches to a structured hash: the fields are JSON-serialized and SHA-256 hashed to produce an unambiguous 24-character key prefix. Existing test suite updated to cover both paths.

### Deferred: P2-010 Concurrent supersede serialization, P2-011 Graph walk rollout docs, P2-012 Shared-memory defaults docs, P2-015 Shared-memory enablement auth, P2-033 Late spec folder discovery

### Rejected: P2-016 Session learning FSRS state — **REJECTED**

**Reason**: FSRS scheduling already exists in `lib/cognitive/fsrs-scheduler.ts`. The session learning handler is scoped to epistemic preflight/postflight measurement, not FSRS scheduling. The module name is broader than its contract, but the implemented behavior is internally consistent.

## P2 Fixes — Eval, Reconsolidation & Interference (9 findings → 6 fixed, 3 deferred)

### 14. Active-only interference scoring (`interference-scoring.ts`, `vector-index-store.ts`) — P2-017

**Problem**: The interference scorer counted same-folder sibling memories when calculating interference penalties, but included archived and deprecated rows that were already excluded from the search pipeline. After reconsolidation merged memories, the archived originals still counted as siblings, over-penalizing the surviving merged memory and pushing its retrieval score down.

**Fix**: Restricted interference scoring to active/retrievable rows only by joining against the `active_memory_projection` table. Archived and deprecated memories are excluded from both the sibling count and the interference weight calculation. New `interference.vitest.ts` tests verify exclusion of archived rows.

### 15. Batch size clamp (`batch-processor.ts`) — P2-021

**Problem**: `processBatches()` accepted arbitrarily large `batchSize` values from the `SPEC_KIT_BATCH_SIZE` environment variable. An oversized value (e.g., 10,000) would fan out a massive `Promise.all()` batch during index scans, potentially exhausting memory or overwhelming the SQLite connection with concurrent writes.

**Fix**: Added a hard clamp of 200 items per batch in `processBatches()`. Values above the clamp are silently reduced. New `batch-processor.vitest.ts` tests verify the clamp applies and that oversized inputs produce correctly-sized batches.

### 16. Ingest path deduplication (`memory-ingest.ts`) — P2-022

**Problem**: The ingest handler normalized file paths for OS compatibility but didn't deduplicate after normalization. Two paths that normalize to the same location (e.g., `./spec.md` and `spec.md`, or paths differing only in trailing slashes) were queued as separate work items, causing duplicate processing and potential race conditions on the same file.

**Fix**: Normalized paths are now deduplicated before job creation using a `Set`. The response payload includes `deduplicatedCount` when paths were removed, giving callers visibility into what was dropped.

### 17. Token-usage null markers (`ablation-framework.ts`) — P2-023

**Problem**: When ablation runs had no token-usage measurement data (the common case for in-process evaluation without API metering), the framework persisted `token_usage = 0/0/0` snapshot rows. Downstream reporting treated these as real measurements of zero tokens, skewing aggregate token-budget calculations.

**Fix**: The ablation framework now skips persisting token-usage snapshot rows entirely when no real measurement exists. The in-memory report shape is unchanged for compatibility, but `storeAblationResults()` only writes rows where measurements are non-synthetic. Updated ablation tests verify omission.

### 18. Sprint ordering by lastSeen (`reporting-dashboard.ts`) — P2-024

**Problem**: The eval dashboard's sprint limiting used `MIN(created_at)` (first-seen) for ordering, which meant an older-starting sprint that was recently updated could be dropped in favor of a newer-starting but inactive sprint. For "show me the 5 most recent sprints" queries, this produced counterintuitive results.

**Fix**: Sprint selection now ranks by `MAX(created_at)` (lastSeen), while the returned sprint list maintains chronological ordering for trend visualization. Tests verify that recently-updated older sprints are preserved in the limit window.

### 19. Ground-truth missing ID warnings (`ablation-framework.ts`) — P2-025

**Problem**: When an operator specified `groundTruthQueryIds` to restrict ablation evaluation to specific test queries, any IDs not found in the static ground-truth dataset were silently filtered out. This made it easy to run an evaluation against a stale ID list without realizing half the queries were being skipped.

**Fix**: The ablation report now includes `requestedQueryIds`, `resolvedQueryIds`, and `missingQueryIds` fields. Missing IDs generate a structured warning in the report metadata. If *all* requested IDs are missing, the ablation run fails with a descriptive error rather than silently producing empty results.

### Deferred: P2-020 BM25 reconsolidation atomicity, P2-034 Unicode tokenization, P2-035 Trigger scan performance

## P2 Fixes — Schema, Envelope & API Surface (9 findings → 4 fixed, 4 deferred, 1 rejected)

### 20. Stale DB cache invalidation (`context-server.ts`) — P2-026

**Problem**: After an out-of-process database reinitialization (e.g., checkpoint restore, test fixture swap), process-global caches in the tool dispatcher and trigger matcher still held stale data from the previous DB generation. Queries would return results from the old DB state until the caches naturally expired or the process restarted.

**Fix**: Added a server-side invalidation step in the dispatch path: whenever `checkDatabaseUpdated()` reports a refresh, the tool cache and trigger matcher cache are explicitly cleared before the next tool dispatch. Tests verify cache invalidation triggers on DB update signal.

### 21. Quick-search envelope label fix (`memory-tools.ts`) — P2-029

**Problem**: `memory_quick_search` reused the `memory_search` handler's response envelope, which meant the `meta.tool` field in the MCP response identified the tool as `memory_search` instead of `memory_quick_search`. Any client behavior or observability dashboard keyed off the tool name would misattribute quick-search traffic.

**Fix**: The dispatcher now rewrites the envelope's `meta.tool` field to `memory_quick_search` before returning. Tests verify the correct tool name appears in the response metadata.

### 22. Shared-memory schema actor exclusivity (`tool-schemas.ts`) — P2-030

**Problem**: The published JSON schema for shared-memory admin tools accepted both `actorUserId` and `actorAgentId` together, but runtime Zod validation rejected requests with both fields set. Clients performing schema-based validation (e.g., MCP tool discovery UIs) wouldn't catch the error until runtime, producing confusing 400 responses.

**Fix**: The JSON schema now encodes "exactly one actor identity" using `oneOf` plus `not` constraints, matching the runtime validation behavior. New `tool-input-schema.vitest.ts` tests verify that dual-actor payloads fail schema validation.

### 23. JSON schema numeric bounds mirrored (`tool-schemas.ts`) — P2-031

**Problem**: Several runtime validation bounds (minimum string lengths, numeric ranges, non-empty constraints) only existed in the Zod layer but were absent from the published JSON schemas. Clients relying on the schema for pre-validation or documentation saw a looser contract than what runtime actually enforced.

**Fix**: Mirrored all missing constraints into `tool-schemas.ts`: `minimum`/`maximum` for numeric fields, `minLength` for required strings, and `minItems` for arrays. The affected tools include `memory_save`, `memory_search`, `memory_context`, `memory_causal_link`, `checkpoint_create`, and `eval_run_ablation`. Tests verify schema-level rejection matches Zod-level rejection.

### 24. ContextArgs profile field added (`types.ts`, `memory-context.ts`) — P2-032

**Problem**: The `memory_context` tool's public JSON schema exposed a `profile` parameter, but the dispatcher-local `ContextArgs` type in `tools/types.ts` didn't include it. The profile value was accepted by the schema but silently dropped during type narrowing, meaning profile-aware routing in `memory_context` never received the field.

**Fix**: Added `profile` to the `ContextArgs` type and forwarded it through all routed `memory_search` delegate calls within the context handler. Tests verify profile propagation through the handler chain.

### 25. Envelope tokenCount from serialized payload (`envelope.ts`) — P2-039

**Problem**: The MCP response envelope computed `meta.tokenCount` from only the `data` field, but the actual transport payload includes the full envelope structure (metadata, pagination cursors, warnings array). Token-budget enforcement downstream would undercount the actual cost of responses, potentially exceeding budget limits.

**Fix**: `meta.tokenCount` is now computed from `JSON.stringify()` of the final complete envelope, not just the data portion. This gives consumers an accurate byte/token budget. Tests verify the token count includes envelope overhead.

### Deferred: P2-038 Embedding timeout handling (requires dist artifact updates), P2-040 autoSurfacedContext breaking change, P2-041 Handler barrel eager imports

### Rejected: P2-003 Split-brain save window — already fixed in v3.0.1.0 (T027)
