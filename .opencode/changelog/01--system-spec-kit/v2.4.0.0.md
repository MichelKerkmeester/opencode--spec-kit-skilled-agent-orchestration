This release removes a large chunk of old, unused search code and fixes 35 bugs found during a thorough code review. The most important fixes address data integrity issues -- things like leftover database records after deletions, scores going out of range, and search filters that silently did nothing. The goal is to make the workflow easier to follow and easier to trust.

> Spec folder: `.opencode/specs/system-spec-kit/022-hybrid-rag-fusion/017-refinement-phase-6` (Level 3)

---

## What Changed

---

### Old search code removed

The memory system had two search pipelines: an original (V1) and a newer replacement (V2). V1 was no longer used but still sitting in the codebase -- roughly 550 lines of dead code including 8 functions that were never called. This release deletes all of it. The V2 pipeline is now the only path and the feature flag that toggled between them is ignored.

---

### Search and scoring fixes

- **Word stemming was broken** -- The system reduces words to their root form for matching (e.g., "running" should become "run"). It was producing "runn" and "stopp" instead. Fixed.
- **Exact-word search filter was a no-op** -- When filtering search results by spec folder, the code compared the wrong kind of identifier, so the filter never actually filtered anything. Now does a proper database lookup.
- **Score double-counting in result merging** -- When combining search results from multiple query variations (a technique called "fusion"), a bonus was being applied twice. Fixed so each result gets the bonus once.
- **Scores could exceed valid range** -- Working memory scores (which track what the system is currently "paying attention to") could go above 1.0 due to an unchecked boost. Scores are now clamped to the 0-1 range.
- **Double decay on attention scores** -- A decay factor (which reduces old scores over time) was being applied twice, making scores drop too fast. Removed the duplicate.
- **Large arrays crashed the scoring math** -- A function that finds the highest score in a list used a technique that fails when the list has more than 100,000 items (a JavaScript engine limit). Replaced with a simple loop.
- **Recency was ignored in intent-based ranking** -- The system adjusts search results based on what it thinks you are trying to do (e.g., "find recent work" vs. "find a definition"). The time-based component of this adjustment was missing. Added.
- **Scoring weights did not add up to 1.0** -- When you override some scoring weights, the rest should adjust proportionally so they still sum to 1.0. They were not adjusting. Now they auto-normalize.

---

### Data integrity fixes

- **Deleting a memory left orphaned records** -- When a memory was deleted, 6 types of related records (graph snapshots, community assignments, summaries, linked entities, causal edges, and search index entries) were left behind. Deletion now cleans all of them up.
- **Orphaned chunks went undetected** -- If a parent memory was deleted, its chunks (smaller pieces stored for search) remained in the database with no way to find or clean them. The integrity checker now detects and optionally removes them.
- **Self-referencing graph edges were possible** -- The causal graph (which tracks how memories relate to each other) could create an edge from a memory to itself, which is meaningless. Now rejected.
- **Duplicate detection used weak hashing** -- Session deduplication used a 64-bit hash, which has a small but real chance of collisions (two different items producing the same hash). Extended to 128-bit.

---

### New capabilities

- **Search API exposes more options** -- Three parameters that existed internally but were hidden from the tool interface are now available: tracking access history, including archived items, and selecting search mode.
- **Smarter embedding on updates** -- When a memory is updated, the system now generates its search embedding from the full title and content, not just the title. This makes updated memories findable by their content.
- **Embedding is cached during search** -- The system computes a mathematical representation of your query for similarity search. It now reuses that computation instead of doing it twice when constitutional memories (always-present rules) are involved.
- **Configurable similarity threshold** -- A function that detects "interfering" memories (ones too similar to each other) now accepts a custom threshold instead of using a fixed value.
- **Evaluation run IDs survive restarts** -- The counter that assigns IDs to evaluation runs now reads the last-used ID from the database on startup, so IDs do not reset when the server restarts.

## Upgrade

No migration required. The `SPECKIT_PIPELINE_V2` environment variable is still accepted but now ignored -- V2 is the only path. All existing tests pass with zero regressions.
