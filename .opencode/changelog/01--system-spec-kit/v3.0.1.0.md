## P0 Blockers Fixed (5)

These five findings represented data loss, silent corruption, or security bypass risks.

### 1. Cross-scope lineage key collision (`lineage-state.ts`)

**Problem**: `buildLogicalKey()` constructed lineage identity from only `(spec_folder, canonical_path, anchor)`. Two governed memories from different tenants, users, or shared spaces but the same file path produced identical logical keys. This caused `UNIQUE(logical_key, version_number)` constraint violations on `memory_lineage` and overwrote each other's `active_memory_projection` rows -- silently merging unrelated version histories across tenant boundaries.

**Fix**: Added `buildScopePrefix()` which normalizes governance fields `(tenant_id, user_id, agent_id, session_id, shared_space_id)` into a labeled tuple, hashes it with SHA-256 (24-char prefix), and inserts the scope segment into the logical key. When all scope fields are null (unscoped memories), the key format is unchanged for backward compatibility. Three new tests verify tenant divergence, user divergence, and unscoped backward compat.

### 2. Wrong embedding dimensions at DB bootstrap (`vector-index-store.ts`)

**Problem**: `get_embedding_dim()` checked for raw API key presence (`VOYAGE_API_KEY`, `OPENAI_API_KEY`) without consulting `EMBEDDINGS_PROVIDER` or checking for placeholder values. The shared embeddings factory at `factory.ts` uses a different precedence: explicit provider first, then non-placeholder Voyage, then non-placeholder OpenAI, then hf-local (768d). A valid `EMBEDDINGS_PROVIDER=hf-local` configuration could bootstrap the database with 1024-dimension vectors (because a Voyage key happened to exist in the environment), then fail at embedding time when actual 768-d vectors were generated.

**Fix**: Replaced ad hoc env checks with `getStartupEmbeddingDimension()` which delegates to the shared factory's canonical resolution rules. Added a dimension mismatch fail-fast guard that reads the existing DB's vector width from `vec_metadata` (fallback: `vec_memories` schema) and throws `VectorIndexError` with `INTEGRITY_ERROR` code before schema bootstrap if the resolved dimension doesn't match. Three new tests cover hf-local override, placeholder key handling, and provider-resolved reopen mismatch.

### 3. Merge restore wiping auxiliary tables (`checkpoints.ts`)

**Problem**: Merge restore (`clearExisting=false`) called `clearTableForRestoreScope()` without passing the active scope. When `checkpointSpecFolder` was null and `hasScope` was false, the helper fell back to a full-table `DELETE`, emptying tables like `working_memory`, `session_state`, `session_sent_memories`, and `shared_spaces` before reinsertion. If any later insert failed, merge mode accepted the partial result without rollback -- the tables stayed empty.

**Fix**: Merge-mode restore now passes `scope: normalizedScope` into the pre-clear call and constrains delete behavior to the actual restore boundary. Each pre-clear + reinsert pair is wrapped in a `SAVEPOINT` so a failed auxiliary-table restore rolls that specific table back instead of leaving partial state. The restore result now includes `partialFailure` and `rolledBackTables` metadata, and the handler distinguishes clean partial success from actual failure with a `CHECKPOINT_RESTORE_PARTIAL_FAILURE` error code.

### 4. Unauthenticated shared-memory admin mutations (`shared-memory.ts`)

**Problem**: `handleSharedSpaceUpsert()` and `handleSharedSpaceMembershipSet()` resolved the acting identity from server-configured admin environment variables (`SPECKIT_SHARED_MEMORY_ADMIN_USER_ID`/`AGENT_ID`) regardless of who actually called the tool. Any MCP client could create spaces in any tenant, bootstrap admin ownership, and rewrite memberships -- the tool surface itself was an admin capability.

**Fix**: Added `validateCallerAuth()` that requires exactly one of `actorUserId`/`actorAgentId` in every request (returns `E_AUTHENTICATION` when omitted). New-space creation now requires the caller to match the configured admin identity. Existing space updates and membership changes allow either the configured admin or a verified space owner. The runtime schema was tightened to reject requests without actor identity. Existing tests were updated to provide identity, and new tests cover omitted actors, non-admin cross-tenant denial, and owner vs non-owner membership management. 66/66 shared-memory tests pass.

### 5. Merged memories disappearing from search (`reconsolidation.ts`)

**Problem**: `executeMerge()` archived the old row and inserted a raw replacement into `memory_index`, but never updated `active_memory_projection`, never ran post-insert metadata hooks (lineage, BM25, history), and the bridge at `reconsolidation-bridge.ts` returned the archived predecessor's ID instead of the new merged row's ID. Since vector retrieval joins through `active_memory_projection` and excludes archived rows, a "successful" merge made the surviving memory invisible to all search paths.

**Fix**: `executeMerge()` now follows the same append-first indexed write contract as normal saves: creates the survivor row, upserts `active_memory_projection` with the new logical key, records a lineage transition with the predecessor as root, syncs BM25 (remove old, add new with canonical document shape), and applies post-insert metadata. The bridge returns `reconResult.newMemoryId`. Tests verify: survivor appears in vector search, projection points to new row, archived predecessor excluded, BM25 sync called, lineage chain recorded.

## Sprint 1: Search + Data Integrity (15 P1 fixes)

### Hybrid search pipeline (`hybrid-search.ts`) -- 6 fixes

- **T020: Fallback threshold units** -- The tiered fallback pipeline passed `minSimilarity` values like `0.3`, `0.17`, `0.1` to the vector channel, but `vector_search()` divides by 100 (expecting percentages). Tier 1 was effectively using a 0.3% floor, making vector retrieval almost unfiltered. Changed constants to `30`, `17`, `10`.

- **T021: Disabled channels re-enabled in fallback** -- `collectRawCandidates()` and `searchWithFallbackTiered()` unconditionally called `ftsSearch()` and `bm25Search()` in fallback, and Tier 2 hard-coded `useBm25: true, useFts: true, useGraph: true, forceAllChannels: true`. Any caller running a channel ablation or targeted search lost their contract as soon as degradation triggered. Fixed by threading an "allowed channels" set through the entire fallback chain.

- **T022: `useGraph: false` not disabling degree** -- The `useGraph` option only suppressed direct graph retrieval. The `degree` channel still ran independently whenever routing included it, injecting graph-derived ranking changes. Fixed by removing both `graph` and `degree` from `activeChannels` when `useGraph` is false.

- **T023: Adaptive fusion lexical weight flipping** -- When only vector/fts/bm25 were active, `adaptiveResult.results` was returned directly (FTS+BM25 as one `keyword` channel). When graph/degree was present, the code switched to `fuseResultsMulti(lists)` and evenly split `keywordWeight` across FTS and BM25. This made the lexical share of RRF depend on an unrelated channel's presence. Fixed by always using one consistent lexical weighting model.

- **T024: Confidence truncation before rerankers** -- Confidence truncation ran immediately after fusion (line ~1124) but before MMR (~1154), co-activation spreading (~1234), and folder relevance scoring (~1265). Candidates cut at this stage were permanently lost, even though later stages are explicitly designed to rescue semantically useful items. The `evaluationMode` flag disabled truncation for benchmarks but not live traffic. Moved truncation to after the final ranking pipeline.

- **T025: Token budget miscounting** -- `estimateResultTokens()` counted only top-level primitive fields, ignoring nested `traceMetadata`, `sources`, and `sourceScores`. The budget cut happened before trace metadata was added. The greedy loop always accepted the first result even when it exceeded the budget. Fixed by estimating after enrichment, including nested fields, and handling single-item overflow.

### Save pipeline (`pe-gating.ts`, `memory-save.ts`) -- 4 fixes

- **T026: PE filtering matching unscoped rows** -- `findSimilarMemories()` used `if (tenantId && r.tenant_id && r.tenant_id !== tenantId) return false;` -- this accepted rows where `r.tenant_id` was NULL, so a scoped save could match and supersede an unscoped legacy memory. Fixed by requiring exact equality when a scope field is supplied; NULL treated as mismatch.

- **T027: Atomic save DB-first failure window** -- `atomicSaveMemory()` committed the DB write before promoting the pending file via `fs.renameSync()`. If rename failed, the DB had a committed record pointing at a file path that was never updated, returning `dbCommitted: true`. Fixed by promoting the file first and performing compensating cleanup when indexing fails.

- **T028: Write lock held across async work** -- `processPreparedMemory()` opened `BEGIN IMMEDIATE` before awaiting reconsolidation (which generates embeddings) and file finalization. This held the SQLite writer lock across network I/O. Fixed by splitting into a planning phase (similarity search, reconsolidation classification, file prep -- no lock) and a mutation phase (`database.transaction().immediate()` with SAVEPOINTs for nested calls).

- **T029: Large files skipping quality gate** -- When `needsChunking()` returned true, the handler exited through `indexChunkedMemoryFile()` before running the quality gate, content-hash dedup, PE gating, or reconsolidation. Save semantics changed based solely on content length. Fixed by moving quality gate and reconsolidation before the chunking fork.

### Transaction safety (`vector-index-mutations.ts`, `vector-index-store.ts`) -- 2 fixes

- **T030: Projection writes best-effort** -- `index_memory()` treated `active_memory_projection` updates as optional, swallowing errors. Since all read paths join through projection, a successful base-row write with a failed projection created a "committed but invisible" memory. Fixed by making projection upsert part of the required mutation contract within the same transaction.

- **T031: SQLiteVectorStore DB switching** -- The module kept the active DB handle in process-global variables. A `SQLiteVectorStore` instance pinned its path during `_ensureInitialized()`, but later mutations called `initialize_db()` without the custom path, using whichever global connection was current. Two store instances interleaved and read/wrote the wrong database. Fixed by having each store re-acquire and pass its own DB handle into all helper calls.

### BM25 index consistency (`bm25-index.ts`, callers) -- 3 fixes

- **T032: Reconsolidation merge BM25 sync** -- Already addressed by T014 (P0 fix); verified the merge path now uses `buildBm25DocumentText()` for the canonical document shape.

- **T033: Archived docs lingering in BM25** -- Assistive auto-merge marked the older memory as archived with a direct `UPDATE` but never called `bm25.removeDocument()`. Archived entries accumulated in the process-local singleton until restart. Fixed by adding immediate BM25 eviction after archive.

- **T034: Live update document shape mismatch** -- Rebuild indexed `title + normalizeContentForBM25(content_text) + trigger_phrases + file_path`, but live mutations indexed only raw content. Title, trigger phrases, and file-path terms were missing from real-time BM25 queries until a rebuild. Fixed by creating `buildBm25DocumentText()` in `bm25-index.ts` and wiring it into all 7 mutation paths: saves, merges, lineage writes, chunking, updates, and unarchive.

## Sprint 2: Correctness + Schema (13 P1 fixes)

### Causal graph (`causal-edges.ts`, `causal-graph.ts`) -- 3 fixes

- **T040: Diamond traversal** -- `getCausalChain()` used a single `visited` set keyed by node ID. In `A->B->D, A->C->D`, whichever branch reached D first won; the C->D edge was silently dropped. Fixed by tracking the active recursion path for cycle prevention instead of a global node set, preserving alternate inbound edges.

- **T041: Direction-aware drift response** -- `memory_drift_why` always returned `causedBy`, `enabledBy` buckets even for `direction: "outgoing"` (where those were effects, not causes). For `both`, forward and backward edges were merged. Fixed by returning separate `incoming` and `outgoing` buckets with per-edge `direction` field.

- **T042: Strength propagation** -- Traversal strength was `min(1, edge.strength * weight)` per-edge without multiplying by the parent's accumulated strength. Deeper paths never decayed. Fixed to propagate cumulatively: `childStrength = clamp(parentStrength * edge.strength * weight)`.

### Schema migrations (`vector-index-schema.ts`) -- 4 fixes

- **T043: Warnings advancing version** -- Many migrations caught DDL failures, logged a warning, and continued. `ensure_schema_version()` then wrote `schema_version = 23` even with incomplete DDL. Fixed by re-throwing DDL failures for required changes and only advancing version after a stronger compatibility check.

- **T044: History backfill ordering** -- `initHistory()` backfilled `memory_history.spec_folder` from `memory_index.spec_folder`, but v23 canonicalized those values later. History rows were permanently frozen with pre-canonicalized folder names. Fixed by running migrations before history backfill and adding a v23 repair step for `memory_history.spec_folder`.

- **T045: Constitutional-tier no-op** -- `migrate_constitutional_tier()` only logged warnings for legacy CHECK constraints. Modern saves expecting `'constitutional'` as a valid tier would fail at runtime. Fixed with a real table-rebuild migration; startup fails on legacy schemas that can't represent the current enum.

- **T046: v12 destroying audit history** -- v12 did `DROP TABLE IF EXISTS memory_conflicts; CREATE TABLE ...`, destroying all pre-v12 conflict audit rows. Fixed with a rename-copy-drop migration: rename old table, create new schema, copy rows mapping `similarity_score`->`similarity` and `notes`->`reason`, drop old table after successful transfer.

### Embedding lifecycle (`embedding-cache.ts`, `factory.ts`) -- 3 fixes

- **T047: Stale cached vectors** -- `lookupEmbedding()` keyed only by `(content_hash, model_id)`, ignoring the stored `dimensions` field. If the same model was reused with a different dimension, stale vectors from the old dimension were served. Fixed by including the resolved dimension in the cache key.

- **T048: Silent provider downgrade** -- Auto-mode fallback from Voyage/OpenAI to hf-local was only logged. `getProviderInfo()` still reported the env-derived provider. Fixed by tracking `requestedProvider`, `effectiveProvider`, `fallbackReason`, and `dimensionChanged` in structured metadata.

- **T049: API key validation timing** -- `validateApiKey()` was advisory; neither startup dimension resolution nor provider creation enforced it. Cloud providers could be selected and their dimensions committed to DB before discovering the key was invalid. Fixed with `resolveStartupEmbeddingConfig()` that couples validation to the first cloud-provider resolution.

### Chunking (`chunking-orchestrator.ts`, `anchor-chunker.ts`) -- 3 fixes

- **T050: Anchor-mode dropping text** -- Once anchor mode was selected (2+ matched sections), all text outside `<!-- ANCHOR:... -->` pairs was discarded. Intro, interstitial, and trailing content was never chunked or embedded. Fixed by preserving non-anchor regions as first-class chunks.

- **T051: Code blocks split by heading detection** -- The structure fallback used `content.split(/(?=^#{1,2}\s)/m)` which treated `# comment` inside fenced code as a heading. Fixed by switching to the AST-aware markdown chunker that keeps code blocks atomic.

- **T052: Orphaned chunk rows** -- Child rows were inserted before metadata assignment. If `applyMetadata()` threw, the catch block only logged. The row ID wasn't in `childIds` yet, so rollback paths never cleaned it up. Fixed by wrapping child insert + metadata in one transaction with immediate cleanup on partial failure.

## Sprint 3: Security + Governance (9 P1 fixes)

- **T060: Spoofable shared-memory status** -- `handleSharedMemoryStatus()` accepted arbitrary `tenantId`/`userId`/`agentId` from the request body and queried `getAllowedSharedSpaceIds()` without authentication. Fixed to derive scope from the authenticated caller using the same `validateCallerAuth` pattern.

- **T061: Wrong error codes** -- All handler errors were wrapped with `SEARCH_FAILED` even for non-search handlers (saves, causal graph, checkpoints). Added domain-specific codes: `CAUSAL_GRAPH_ERROR`, `TRAVERSAL_ERROR`, `CAUSAL_INVALID_RELATION`, `SAVE_FAILED`, etc.

- **T062: Exception text leaking** -- Internal stack traces and file paths were returned in MCP error responses. Added `createSanitizedError()` helpers that log the full error server-side and return only a generic public message to MCP clients.

- **T063: Causal graph error codes** -- Causal graph handlers used `SEARCH_FAILED` for graph operations. Replaced with graph-specific codes throughout all 4 handlers (drift-why, link, stats, unlink).

- **T064: Response profiles stripping fields** -- Response envelope profiles discarded non-`results` fields like `totalCount`, `truncated`, and `searchMetadata`. Fixed to preserve all metadata fields.

- **T065: Exceptions bypassing envelope** -- Server-level exceptions could return raw error objects. Added a top-level catch-all in `context-server.ts` that wraps all unhandled exceptions in the standard MCP error envelope with `wrapForMCP()`.

- **T066: Singleton DB rebind gaps** -- `initializeDb(newPath)` refreshed only some module-level DB consumers. Others kept stale references. Added a `db-state` listener that broadcasts rebind events to all registered consumers.

- **T067: Access-tracker bleed** -- Hit count accumulators persisted across DB swaps, leaking metrics from one DB context into another. Fixed by resetting all access-tracker state when the underlying connection changes.

- **T068: Unscoped continuation cursors** -- Progressive-disclosure cursors didn't validate that the resuming caller had the same scope as the original query. Fixed by including scope metadata in cursor tokens and validating on resume.

## Sprint 4: Remaining P1s (22 fixes)

- **T070: PE SUPERSEDE cross-file splicing** -- The save path treated any PE `SUPERSEDE` target as a lineage predecessor, even when it was a semantically-similar but different file. Fixed to only chain same-path files; cross-path contradictions recorded as causal `supersedes` links.

- **T071: YAML body override** -- YAML-looking body text after frontmatter could override parsed metadata fields. Fixed by stopping body re-parsing as YAML after frontmatter extraction.

- **T072: Malformed anchor nesting** -- Overlapping or improperly closed anchor tags were silently accepted. Added proper nesting validation.

- **T073: Checklist state erased** -- Content normalization rewrote `- [x]` checklist items, losing their checked state. Fixed to preserve checklist markers during normalization.

- **T074: Focused mode ranking** -- `memory_context` focused mode bypassed intent-aware ranking. Fixed to apply relevance scoring regardless of mode.

- **T075: Resume keyword dependency** -- Resume mode required specific keywords to activate. Fixed to detect resume intent from context (spec folder, prior session state) rather than exact keyword matches.

- **T076: Errors wrapped as success** -- Handler errors in `memory_context` were wrapped in success response envelopes. Fixed to return proper error responses with appropriate codes.

- **T077: Common-word triggers** -- Words like "the", "and", "is" were accepted as trigger phrases. Added a stopword filter and minimum specificity check.

- **T078: Frontmatter extraction errors dropped** -- Parse errors during trigger matching were caught and silently swallowed. Fixed to surface structured failure information so callers know matching was degraded.

- **T079: Force-save dedup rejection** -- Content-hash dedup rejected intentional force saves on the same path. Fixed to skip dedup when `force: true`.

- **T080: Chunked files skipping dedup** -- Chunked saves bypassed content-hash dedup entirely. Fixed by running dedup before the chunking decision (complementing T029).

- **T081: Cross-session postflight overwrite** -- `task_postflight` could overwrite another session's baseline when two sessions shared the same `task_id`. Fixed by including `session_id` in the postflight key.

- **T082: Write-once completion rows** -- Once a row was marked completed for a `task_id`, later learning cycles were lost. Fixed to allow multiple completion entries per task, preserving the full learning history.

- **T083: Trigger scope fields unreachable** -- `memory_match_triggers` tool schema exposed scope filter fields but the handler never read them. Fixed by wiring `specFolder`, `tenantId`, `userId`, `agentId`, `sharedSpaceId` through to the matching logic.

- **T084: Self-loop double-snapshot** -- Self-loop edges were counted twice in degree snapshots (once outgoing, once incoming). Fixed to detect and count once.

- **T085: Degree cache partial invalidation** -- Cache invalidation covered only link creation but missed unlink, archive, and merge. Fixed to invalidate on all mutation surfaces including `mutation-hooks` and `archival-manager`.

- **T086: Degree cap not enforced** -- The advertised `0.15` degree cap didn't actually cap fusion influence. Fixed by enforcing the cap in the score calculation at `graph-search-fn.ts`, not just in display. Set `DEGREE_CHANNEL_WEIGHT = 0.15` as the authoritative constant.

- **T087: Incremental scan mtime sensitivity** -- Subsecond file writes could be missed when mtimes didn't move enough. Fixed by using content-hash comparison as a secondary change detection signal alongside mtime.

- **T088: Stale row deletion before replacement** -- During re-scan, stale rows were deleted before replacement indexing completed. If re-indexing failed, the memory was lost. Fixed by indexing the replacement first, then deleting the stale row in the same transaction.

- **T089: Graph refresh flag ignored** -- `SPECKIT_GRAPH_REFRESH_MODE=off` didn't actually disable graph enrichment. Fixed by checking the flag in both `search-flags.ts` and `graph-lifecycle.ts`.

- **T090: Dashboard collapsing channels** -- The eval dashboard collapsed distinct channel metrics into aggregate values, losing per-channel visibility. Fixed to preserve and render per-channel breakdowns.

- **T091: Missing alignment guard** -- Ablation runs never enforced the ground-truth alignment check added in spec-023. Fixed by calling `assertGroundTruthAlignment()` before running iterations.

## Bonus Fixes (2)

- **T088b: Nested transaction failure** -- The save pipeline used manual `BEGIN IMMEDIATE`/`COMMIT`/`ROLLBACK` which conflicted with inner `database.transaction()` calls in `createMemoryRecord` and `index_memory`, causing "cannot start a transaction within a transaction" errors during bulk scan. Fixed by switching to `database.transaction().immediate()` which nests via SAVEPOINTs.

## Upgrade

No migration required for most deployments. Schema migrations are backward-compatible and run automatically on startup.

**Breaking change for legacy databases**: The constitutional-tier migration (T045) will now fail startup on databases that still have the pre-constitutional `importance_tier` CHECK constraint. If affected, run a manual table rebuild or drop and recreate the constraint before upgrading.

**Behavioral changes**:
- Shared-memory tools now require caller identity (`actorUserId` or `actorAgentId`) -- requests without identity will receive `E_AUTHENTICATION` errors
- Error responses now use domain-specific codes instead of generic `SEARCH_FAILED`
- `memory_drift_why` response shape changed from flat `causedBy`/`enabledBy` to `incoming`/`outgoing` direction-aware buckets
- Hybrid search fallback no longer re-enables caller-disabled channels
