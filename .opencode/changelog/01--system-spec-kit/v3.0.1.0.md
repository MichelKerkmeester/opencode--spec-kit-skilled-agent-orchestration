## [**3.0.1.0**] - 2026-03-28

Complete closure of the 121-finding memory database code audit across three implementation passes. 117 fixes implemented, 3 findings rejected as non-issues, zero remaining. Test suite grew from roughly 8,660 to 8,789 passing tests with no type errors.

This release consolidates three passes: Pass 1 fixed 80 critical/required findings (5 data-loss-risk blockers + 75 required fixes). Pass 2 fixed 22 improvement findings and rejected 3. Pass 3 fixed the remaining 15 deferred findings and aligned all documentation.

Previously tracked as v3.0.1.0 (Pass 1), v3.0.2.0 (Pass 2), and v3.1.0.0 (Pass 3) — now consolidated into a single release.

## Critical Fixes (5)

These five issues could cause data loss, silent corruption, or security bypass.

### Memories from different users could overwrite each other

**Problem:** The system identifies each memory using a "logical key" built from its spec folder, file path, and anchor. But it didn't account for *who* created the memory. Two users (or two AI agents, or two tenants) working on the same file path would generate identical keys, causing their version histories to collide -- one user's memory would silently overwrite the other's.

**Fix:** The logical key now includes a hashed identifier derived from the creator's identity (tenant, user, agent, session, or shared space). Existing memories without identity data keep their old keys for backward compatibility.

### Database could be created with wrong-sized vector storage

**Problem:** The system uses vector embeddings (numerical representations of text, used for similarity search). Different embedding providers produce vectors of different sizes -- for example, one provider outputs 768 numbers per vector, another outputs 1024. At database startup, the code checked which API keys existed to guess the right size, but used different logic than the code that actually generates embeddings. This could create a database sized for 1024-dimension vectors that then receives 768-dimension vectors, causing failures.

**Fix:** Startup now uses the exact same logic as the embedding generator to determine vector size. If the database already exists with a different size, startup fails immediately with a clear error instead of silently creating mismatched data.

### Database restore could accidentally wipe tables

**Problem:** The checkpoint restore system has a "merge" mode that's supposed to blend restored data with existing data (as opposed to full replace). But when certain conditions weren't met (no specific folder scope set), it would fall back to deleting *everything* from tables like working memory, session state, and shared spaces before reinserting. If any reinsert failed, those tables stayed empty.

**Fix:** Merge-mode restore now scopes its deletes to only the data being restored, and wraps each table's delete-and-reinsert in a savepoint (a mini-transaction) so a failure in one table doesn't leave others empty. The response now reports which tables had partial failures.

### Anyone could create shared memory spaces without authentication

**Problem:** The shared-memory management tools (creating spaces, setting memberships) pulled the "who's doing this" identity from server configuration variables instead of from the actual request. Any connected client could create spaces in any tenant, claim admin ownership, and rewrite memberships.

**Fix:** Every shared-memory request now requires the caller to identify themselves. Creating new spaces requires admin credentials. Modifying existing spaces requires the caller to be either an admin or the space owner.

### Merged memories became invisible to search

**Problem:** When the system merged two similar memories into one, it archived the old record and created a new one. But it didn't update the search index to point to the new record, didn't update the text search index, and didn't record the lineage chain. The merged memory existed in the database but couldn't be found by any search method.

**Fix:** Memory merges now follow the same full indexing process as normal saves -- updating the search projection, the text search index, and the lineage chain. The merged memory is immediately findable through all search paths.

## Sprint 1: Search and Data Integrity (15 fixes)

### Search pipeline (6 fixes)

### Similarity thresholds were 100x too low
The search system's fallback tiers (which progressively relax standards when initial search finds too few results) passed values like `0.3` meaning "30% similarity." But the underlying function expected percentages like `30`. So the actual threshold was 0.3%, making the filter accept almost everything. Fixed by using the correct scale.

### Fallback search re-enabled channels the caller had turned off
If you deliberately disabled a search channel (say, for testing or benchmarking), the fallback system would turn it back on when results were scarce. Fixed by tracking which channels the caller allowed and respecting that through all fallback tiers.

### Disabling graph search didn't fully disable it
The graph search has two parts: direct graph lookup and a "degree" channel (which ranks results by how connected they are). Turning off graph search only disabled the first part. Fixed by disabling both when graph search is turned off.

### Text search weighting changed depending on unrelated features
Whether the text search channels (full-text search and word matching) got equal or unequal weight in the final ranking depended on whether the graph channel happened to be active -- a completely unrelated feature. Fixed by using consistent weighting regardless of which other channels are active.

### Results were filtered by confidence too early
The search pipeline has several post-processing stages that can rescue useful results (diversity filtering, relationship boosting, folder relevance scoring). But confidence filtering happened *before* these stages, permanently removing results that the later stages might have promoted. Fixed by moving confidence filtering to after all ranking stages complete.

### Response size estimation was inaccurate
The system estimates how many "tokens" (roughly, words) each result costs to include in a response, to stay within budget. But it only counted top-level fields and ignored nested metadata, and it always accepted the first result even if it alone exceeded the budget. Fixed by counting all fields and respecting the budget from the first result.

### Save pipeline (4 fixes)

### Saving could match and replace memories from different users
When checking whether a new memory supersedes an existing one, the system accepted matches where the existing memory had no user/tenant scope. A scoped save could accidentally replace an unscoped legacy memory. Fixed by treating "no scope" as a mismatch, not a match.

### Database was committed before the file was ready
The save process wrote to the database first, then renamed the temp file to its final location. If the rename failed, the database had a record pointing to a file that didn't exist. Fixed by finalizing the file first.

### Database was locked during slow network operations
The save process held a write lock on the entire database while waiting for embedding generation (which requires a network call to an AI provider). This blocked all other writes. Fixed by splitting into a planning phase (no lock needed) and a fast mutation phase (lock held briefly).

### Large files skipped quality checks
Files above a certain size triggered a "chunking" path (splitting into smaller pieces for indexing) that bypassed the quality gate, duplicate detection, and other checks that normal saves go through. Fixed by running all checks before the chunking decision.

### Transaction safety (2 fixes)

### Failed search-index updates were silently ignored
The search projection table (which maps logical keys to active memory IDs) is required for all searches to work. But failures writing to this table were caught and swallowed. A memory could be "saved" but invisible to search. Fixed by making projection updates mandatory within the same transaction.

### Multiple database instances could interfere with each other
The code used global variables to track the active database connection. When two database instances existed simultaneously (e.g., during benchmarks with a test database), they could accidentally read from and write to each other's databases. Fixed by having each instance track its own connection.

### Text search index consistency (3 fixes)

### Merged memories had incomplete text search entries
The word-matching index (BM25, which finds exact word matches) wasn't updated when memories were merged, so the merged memory was missing from word-based searches. Fixed alongside the critical merge fix above.

### Archived memories lingered in the text search index
When a memory was archived through auto-merge, it was marked as archived in the database but not removed from the in-memory text search index. It would keep appearing in word-match results until the server restarted. Fixed by removing archived entries immediately.

### Text search indexed different fields at different times
The full rebuild of the text search index included the title, trigger phrases, and file path. But real-time updates only indexed the content body. This meant newly saved memories were harder to find by title until the next full rebuild. Fixed by creating a shared function that always indexes the same fields.

## Sprint 2: Correctness and Schema (13 fixes)

### Causal graph (3 fixes)

The causal graph tracks relationships between memories -- which memory caused, enabled, or led to another.

### Diamond-shaped relationships lost edges
When memory A connects to D through both B and C (a "diamond" pattern), whichever path was traversed first marked D as "visited," causing the other path's connection to D to be dropped. Fixed by tracking the current traversal path instead of globally visited nodes.

### Direction was ignored in drift analysis
When asking "why did this memory change," the system returned the same relationship types regardless of whether you asked about causes (incoming) or effects (outgoing). Fixed by separating the response into incoming and outgoing buckets.

### Relationship strength didn't decay over distance
When traversing a chain of relationships, each link's strength was calculated independently without considering how far from the starting point it was. A relationship 5 steps away appeared just as strong as one 1 step away. Fixed by accumulating strength decay through the chain.

### Database schema migrations (4 fixes)

Schema migrations update the database structure when the system is upgraded.

### Failed migrations were recorded as successful
If a migration step failed partway through, the system logged a warning but still advanced the version number. The next startup would skip the incomplete migration, leaving the database in a broken state. Fixed by failing hard on required migration steps.

### History records were populated with pre-migration data
The migration that standardizes folder names ran *after* the history table was already populated, locking in the old unstandardized names. Fixed by running migrations before history population.

### A required database feature was never actually added
The migration for "constitutional" importance tier (a special always-surface tier) only logged a warning if the old database couldn't support it, instead of actually updating the schema. Fixed with a real migration that rebuilds the table to support the new tier.

### One migration destroyed audit data
A migration dropped and recreated the conflicts table, losing all previously recorded conflict audit records. Fixed by renaming the old table, creating the new one, copying data with column mapping, then dropping the old table only after successful transfer.

### Embedding system (3 fixes)

Embeddings are the numerical representations of text that power similarity search.

### Cached embeddings could have wrong dimensions
The embedding cache looked up entries by content hash and model name, but ignored the vector size. If a model was reconfigured to use different dimensions, the cache would serve stale vectors of the wrong size. Fixed by including dimensions in the cache key.

### Provider fallback happened silently
If the preferred embedding provider (a cloud AI service) failed, the system silently fell back to a local alternative. Logs showed the switch, but the system still reported using the original provider. Fixed by tracking what was requested, what was actually used, and why the switch happened.

### Invalid API keys weren't caught early enough
The system could select a cloud embedding provider and commit its vector dimensions to the database before discovering the API key was invalid. Fixed by validating the API key before committing to the provider.

### Text chunking (3 fixes)

Chunking splits large documents into smaller pieces for indexing.

### Anchor-based chunking dropped text between anchors
When a document used anchor markers to define sections, any text outside those markers (introductions, transitions, conclusions) was silently discarded. Fixed by preserving non-anchor text as its own chunks.

### Code blocks were broken by heading detection
The fallback chunking method split text at markdown headings (`# Heading`). But it also split at `# comment` lines inside code blocks, breaking code examples apart. Fixed by using a markdown parser that understands code blocks are atomic.

### Failed metadata assignment left orphaned records
When creating chunks, the database record was inserted before metadata was applied. If metadata assignment failed, the record was left without metadata and never cleaned up. Fixed by wrapping both steps in a single transaction with cleanup on failure.

## Sprint 3: Security and Governance (9 fixes)

### Shared-memory status was viewable without authentication
The status-check tool accepted arbitrary identity parameters from the request body without verifying them. Anyone could check the status of any tenant's shared spaces. Fixed by deriving identity from the authenticated caller.

### All errors used the same error code
Whether a search failed, a save failed, or a graph operation failed, the system returned `SEARCH_FAILED`. Fixed by adding domain-specific error codes so clients can distinguish between different types of failures.

### Internal details leaked in error messages
Error responses included raw stack traces and internal file paths. Fixed by logging full details server-side and returning only generic messages to clients.

### Causal graph operations used search error codes
Graph-related handlers reported failures as `SEARCH_FAILED`. Fixed with graph-specific error codes across all 4 graph handlers.

### Response formatting stripped useful metadata
The response formatting system was discarding fields like total count, whether results were truncated, and search metadata. Fixed by preserving all metadata fields.

### Unhandled exceptions returned raw errors
Server-level exceptions that weren't caught by individual handlers returned unformatted error objects. Fixed by adding a catch-all that wraps all unhandled exceptions in the standard response format.

### Database reconnection left some components on the old connection
When the database was re-initialized with a new path, some internal components kept references to the old connection. Fixed by broadcasting reconnection events to all registered components.

### Usage metrics leaked across database swaps
Internal counters tracking how often memories were accessed persisted when switching databases, mixing metrics from different databases. Fixed by resetting all counters when the database connection changes.

### Pagination cursors weren't scoped
The progressive-disclosure system (which returns results in pages) didn't verify that the person requesting "page 2" had the same permissions as the person who requested "page 1." Fixed by encoding scope information in cursor tokens and validating on resume.

## Sprint 4: Remaining Fixes (22)

### Supersede chains crossed file boundaries incorrectly
When a new memory supersedes an old one, the system should only create version chains between memories for the same file. It was chaining memories from different files that happened to be semantically similar. Fixed by restricting version chains to same-path files.

### Document body text could override parsed metadata
If a memory file's body content looked like YAML, it could accidentally overwrite metadata that was already extracted from the frontmatter header. Fixed by stopping body re-parsing after frontmatter extraction.

### Overlapping anchors were silently accepted
Improperly nested or unclosed anchor tags were accepted without error, leading to unpredictable chunking behavior. Fixed by adding nesting validation.

### Checklist state was erased during normalization
Content normalization (which standardizes formatting) was rewriting `- [x]` (checked items) as `- [ ]` (unchecked items), losing completion tracking. Fixed by preserving checklist markers.

### Focused search mode skipped relevance ranking
The "focused" mode in the context tool was supposed to narrow results to a specific area, but it bypassed the relevance ranking system entirely. Fixed by applying ranking regardless of mode.

### Resume detection required exact keywords
To resume a previous session, you had to use specific trigger words. Fixed by detecting resume intent from context (like the presence of a spec folder or prior session state) instead of requiring magic words.

### Errors were wrapped as successes
Some handler errors in the context tool were returned inside a success response envelope, making failures look like successful results. Fixed by returning proper error responses.

### Common words were accepted as trigger phrases
Words like "the", "and", "is" could be stored as trigger phrases (keywords that surface a memory). Fixed by adding a stopword filter and minimum specificity check.

### Parse errors during trigger matching were silently dropped
If the system couldn't parse a memory file while checking triggers, it silently skipped it. Fixed by surfacing structured error information so callers know matching was degraded.

### Force-save was blocked by duplicate detection
When explicitly force-saving to the same path, the duplicate checker rejected it. Fixed by skipping duplicate detection when `force: true` is set.

### Chunked files skipped duplicate detection
Large files that were split into chunks bypassed the duplicate check entirely. Fixed by running dedup before the chunking decision.

### Concurrent sessions could overwrite each other's benchmarks
Post-task measurements could overwrite another session's baseline when two sessions shared the same task ID. Fixed by including the session ID in the measurement key.

### Completed tasks couldn't record new learning
Once a task was marked complete, later learning cycles for that task were discarded. Fixed by allowing multiple completion entries per task, preserving the full learning history.

### Trigger matching ignored scope filters
The trigger-matching tool's schema exposed scope filter fields (spec folder, tenant, user, agent, shared space), but the handler never actually read them. Fixed by wiring all scope fields through to the matching logic.

### Self-referencing relationships were double-counted
When a memory links to itself (a self-loop), degree calculations counted it twice (once as outgoing, once as incoming). Fixed by detecting self-loops and counting them once.

### Degree cache was only partially invalidated
The cache of "how connected is each memory" was cleared when new links were created, but not when links were removed, memories were archived, or memories were merged. Fixed by invalidating on all mutation types.

### The degree channel's influence cap wasn't enforced
The system advertised a 0.15 cap on how much the degree channel (which boosts well-connected memories) could influence final rankings, but the actual calculation didn't enforce it. Fixed by applying the cap in the score calculation.

### Subsecond file changes could be missed
The incremental scanner compared file modification timestamps, but very fast writes could produce the same timestamp. Fixed by using content-hash comparison as a secondary change detection signal.

### Re-indexing failures could lose the original memory
During re-scanning, the old record was deleted before the replacement was fully indexed. If re-indexing failed, the memory was lost. Fixed by indexing the replacement first, then deleting the old record.

### A feature flag for graph refresh was ignored
Setting `SPECKIT_GRAPH_REFRESH_MODE=off` didn't actually disable graph enrichment. Fixed by checking the flag in all relevant code paths.

### Benchmark dashboard collapsed per-channel metrics
The evaluation dashboard merged metrics from different search channels into aggregate values, losing the ability to see how each channel performed individually. Fixed by preserving per-channel breakdowns.

### Benchmark didn't enforce its own test-data check
The ground-truth alignment check added in the previous release (v3.0.0.4) wasn't actually called before benchmark runs. Fixed by calling it.

### Bonus fixes (2)

### Nested transactions caused crashes during bulk operations
The save pipeline used manual transaction commands that conflicted with internal transaction calls, causing "cannot start a transaction within a transaction" errors. Fixed by using the proper nesting mechanism (savepoints).

### Re-indexing could fail on unique constraint
During force re-indexing, the search projection table could hit a unique constraint violation if the old entry wasn't cleared first. Fixed by cleaning up the old entry before inserting the new one.

## Upgrade

No migration required for most deployments. Schema migrations are backward-compatible and run automatically on startup.

**One potential breaking change for old databases**: The constitutional-tier migration will now fail startup on databases that still have the legacy importance-tier constraint. If affected, run a manual table rebuild before upgrading.

**Behavioral changes to be aware of**:
- Shared-memory tools now require caller identity -- requests without it will get authentication errors
- Error responses use specific error codes instead of the generic `SEARCH_FAILED`
- The `memory_drift_why` response shape changed from flat lists to direction-aware incoming/outgoing buckets
- Search fallback no longer re-enables channels the caller explicitly disabled
