## [**2.2.11.0**] - 2026-02-15

This release improves the project’s README files so they are easier for the memory system to index and easier for people to read consistently. It also cleans up wording issues across the documentation set.

---

## What Changed

---

### Anchor tags were added across the README set

- Need: The memory system needed clearer section markers so it could find specific parts of the documentation more reliably.
- Change: This release added anchor pairs across the system spec kit README files so sections can be indexed at a finer level.
- Why it matters: This makes the docs more searchable and helps later tools retrieve the right section instead of the whole file.

---

### Human-writing cleanup was applied

- Need: Some README sections still used wording patterns that did not match the project’s human-writing rules.
- Change: This release fixed the identified wording violations and cleaned up list structures that were hard to read.
- Why it matters: This gives the documentation a more consistent voice and makes it easier for non-developers to follow.

---

### A missing frontmatter block was fixed

- Need: One README was missing the metadata block used by the documentation system.
- Change: This release added the missing YAML frontmatter to that file.
- Why it matters: This keeps the documentation package structurally complete and easier to process.

---

### Suspected anchor problems were reviewed

- Need: The team needed to make sure the new anchor work did not introduce false alarms.
- Change: This release reviewed the files flagged for stray anchors and confirmed those cases were harmless, such as anchor-like text inside code samples.
- Why it matters: This gives the team more confidence that the anchor rollout did not create hidden document issues.

---

### Coverage was spread across the full doc tree

- Need: The indexing improvements needed to cover more than just one or two top-level folders.
- Change: This release applied the work across configuration docs, shared code docs, templates, scripts, and the MCP server documentation tree.
- Why it matters: This keeps the documentation system consistent across the whole project instead of improving only one corner of it.
