## What Changed

### Decision records updated to match reality

Each spec folder can include "decision records" -- documents that capture important technical choices and their status (proposed, accepted, rejected, etc.). Three decision records in the database features spec (014) still said "Proposed" even though the work was finished, tested, and verified. Updated all three to "Accepted" with evidence from the passing test suite.

### Checklist honesty fixes

- **014 (database features)** -- The completion checklist lumped together two different things: actual verification items (all done) and organizational sign-offs like "Product Owner approval" and "Security review" (which are deferred until production handoff). Reworded to make this distinction clear, and added a note explaining why the sign-offs are not blocking.
- **015 (skill alignment)** -- The checklist claimed the validation script exited cleanly. It actually exits with errors -- but for a known, accepted reason: the validator flags certain file paths as broken because it does not understand that relative paths like `../../../../skill/...` resolve correctly from deep inside the spec folder tree. Rewritten to honestly document the exit code and explain why the errors are acceptable.

### Validator limitation documented

The 015 implementation summary now explains a subtle dependency: relative file paths in the spec folder work because the folder sits at a specific depth inside `.opencode/specs/...`, which means `../../../../skill/` correctly reaches `.opencode/skill/`. This was previously just assumed knowledge.
