This release completes the `[spec]/memory/*.md` retirement that v3.4.0.0 announced. The v3.4.0.0 release notes stated the surface was retired, but the runtime still created, wrote, indexed, and read from `[spec]/memory/` on every save — a deep-review audit (packet `026/013-memory-folder-deprecation-audit`, 8 iterations) surfaced 25 findings (9 P0 runtime contradictions) documenting the gap. v3.4.1.0 cuts the write path, deletes the manual-snippet template, migrates tests and docs, and leaves the runtime consistent with the v3.4.0.0 intent.

> Spec folder:
> - `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/013-memory-folder-deprecation-audit/` (Level 2, audit + Path A execution)

## Memory Folder Write Path Removed

The save workflow no longer produces `[spec]/memory/*.md` artifacts. Every category that used to land there now routes through the content-router into canonical spec docs, so the continuity surface is the same one the packet already tracks.

#### `[spec]/memory/` directories no longer created on save

`setupContextDirectory` used to unconditionally `mkdir [spec]/memory/` for every save, even when no memory file was produced. It has been replaced with `ensureSpecFolderExists`, which only validates that the target spec folder exists. New spec folders created via `create.sh` no longer get a `memory/` subdirectory, and the `--help` output no longer mentions one. Existing `memory/` directories in older packets are preserved as a read-only historical corpus.

&nbsp;

#### `writeFilesAtomically` and the memory/*.md write leg deleted

`scripts/core/file-writer.ts` was the primary producer of `[spec]/memory/YYYY-MM-DD__topic.md` files via atomic-commit semantics. The whole module is deleted. Step 9 of the save workflow no longer renders a memory file, no longer assigns a `ctxFilename`, and no longer calls the atomic-commit path. `writeFilesAtomically`, `checkForDuplicateContent`, and the sibling-glob dedup logic are gone.

&nbsp;

#### `indexMemory` wrapper removed; vector DB indexing goes through canonical docs

The script-side `scripts/core/memory-indexer.ts::indexMemory` wrapper previously called `vectorIndex.indexMemory({ filePath: path.join(contextDir, contextFilename) })` after every save, registering the new memory file as memory #N in the vector DB. That wrapper is deleted along with its `updateMetadataEmbeddingStatus` helper. Indexing now flows exclusively through Step 11.5 auto-index (introduced in this session's Step 11.5 patch), which re-indexes canonical spec docs + `graph-metadata.json` after save. The MCP handler call sites (`pe-gating.ts`, `chunking-orchestrator.ts`, `save/reconsolidation-bridge.ts`, `save/create-record.ts`) still call `vectorIndex.indexMemory()` directly with externally-supplied paths — those are unchanged.

&nbsp;

#### Dedup contract (phantom) and causal-link reads deleted

`save.md` §Session Deduplication previously specified a SHA-256 + 1h + interactive-prompt dedup contract with zero runtime implementation. The documentation section has been removed; the contract was a phantom spec. The pre-save SHA-1 overlap check that globbed `memory/*.md` siblings is also deleted. `autoPopulateCausalLinks` in `memory-metadata.ts` (the function that walked `[spec]/memory/*.md` to pre-populate causal graph edges) is deleted; packet-level relationships route through `graph-metadata.json.manual.depends_on/related_to/supersedes` and MCP causal-edge handlers.

## Template and Doc Cleanup

The template and documentation surface that taught operators the retired memory-file workflow has been purged or rewritten to match the new canonical-doc routing.

#### `context_template.md` deleted

The 589-line template used to render standalone memory snippets (`context_template.md`) has been deleted along with its couplings: the `LEGACY_TEMPLATE_BANNER_RE` regex in `shared/parsing/memory-template-contract.ts`, the banner-strip regex in `scripts/renderers/template-renderer.ts`, the filename-classification branch in `scripts/lib/frontmatter-migration.ts:1263`, and three test files that specifically exercised the retired template (`memory-quality-phase1.vitest.ts`, `memory-quality-phase6-template.test.ts`, `anchor-id-simplification.vitest.ts`). The `templates/memory/` directory (template + README) is gone.

&nbsp;

#### CHK-052 removed from four checklist templates

The Level 2, Level 3, Level 3+, and `addendum/level2-verify` checklist templates each carried a `CHK-052 "Findings saved to memory/"` verification row. That row has been removed. New spec folders no longer inherit a retirement-violating QA gate, and existing checklists that still carry CHK-052 can be safely unchecked (the underlying surface is gone).

&nbsp;

#### Reference and command docs aligned with retired state

`.opencode/command/memory/save.md` dropped the Session Deduplication section, the Recovery-table phantom-dedup row, and any step description that referenced `[spec]/memory/*.md` as an output artifact. `manage.md` already carried accurate retirement wording. `mcp_server/INSTALL_GUIDE.md` lost the stale `memory context in spec folders: specs/*/memory/` pointer. `references/memory/save_workflow.md`, `memory_system.md`, `trigger_config.md`, and `references/workflows/worked_examples.md` now describe the content-router + canonical-doc routing instead of the retired write path. `SKILL.md`, `README.md`, and the four `templates/level_{1,2,3,3+}/README.md` files have their retirement language harmonized so every audit surface agrees with every other audit surface.

&nbsp;

#### Test fixtures migrated or removed

Eight vitest files that asserted `[spec]/memory/*.md` behavior (`slug-uniqueness`, `tree-thinning`, `validate-memory-quality`, `artifact-routing`, `handler-memory-index`, `full-spec-doc-indexing`, `content-hash-dedup`, plus the three phase-1/6 template tests) have been deleted or migrated to canonical-doc fixtures. The self-contradictory `full-spec-doc-indexing.vitest.ts` ("returns memory" vs "rejects legacy memory") has been reduced to the "rejects legacy memory" half, which matches the new runtime.

## Backward Compatibility

- Historical `[spec]/memory/*.md` files in existing spec folders are preserved on disk as a read-only content archive.
- Existing `memory_index` DB rows pointing at those historical files remain indexed; no migration script deletes them. `/memory:search` still returns historical matches when relevant.
- `/spec_kit:resume` recovery is unchanged: the ladder reads `handover.md` first, then `_memory.continuity`, then canonical spec docs. It never read `[spec]/memory/*.md` in any earlier release, so no regression is possible.
- Stop-hook `runContextAutosave()` is unchanged: it routes through `generate-context.js`, which now produces canonical-doc updates instead of memory files. Continuity at session end is preserved.
- `_memory.continuity` YAML frontmatter key, `.opencode/command/memory/` command namespace, and MCP tool names (`memory_save`, `memory_index_scan`, `memory_context`, `memory_stats`, `memory_update`) are all unchanged.
- The `system-spec-kit` skill package, all Level 1/2/3/3+ templates except the retired `context_template.md`, all MCP server source code (handlers, routing, schemas, types, content-router) are unchanged.

## Known Limitations

1. **Residual test hits (205 matches).** Broad grep for `memory/.*\.md` across `scripts/tests/` and `mcp_server/tests/` still returns 205 hits outside the migrated-surface slice. These are memory-infrastructure tests (trigger phrases, frontmatter parsing, embedding quality) that reference `memory/*.md` as a file-shape token in fixtures, not as a retirement-violating surface. A full grep-zero state requires a broader follow-up pass through tests unrelated to the runtime surface this release closes.
2. **Vitest runner startup.** Both packages have pre-existing vitest startup issues (`scripts` can't resolve `vitest/config`; `mcp_server` hangs on summary emit in some environments). Those are unrelated to this release and can be diagnosed separately.
3. **Path C deferred.** The deep-review audit identified 4 findings (F017, F019, F020, F021) in the dedup-contract-phantom family. Path A supersedes them in context (`save.md` §Session Deduplication is deleted, so the contract is no longer a phantom — there's simply no dedup contract). If future work wants deduplication semantics, a new packet should design them against the canonical-doc content-router, not against the retired memory-file surface.
4. **DB row preservation.** Historical `memory_index` rows pointing at `memory/*.md` files remain indexed (option 8a from the plan). This was a deliberate choice: the historical corpus retains retrieval signal for older packet context. A future cleanup can delete stale rows if the corpus becomes a retrieval noise source.

## Upgrade

No migration required. The release is backward compatible:
- Existing spec folders keep their `memory/` directories and historical `.md` files as read-only archive.
- New spec folders never get a `memory/` directory.
- Runtime save path routes to canonical docs via the existing content-router; no new configuration needed.
- `SPECKIT_INDEX_SPEC_DOCS` env var behavior unchanged.
- `SPECKIT_AUTO_INDEX_TOUCHED` (added earlier this session for Step 11.5 auto-index) kill switch still works.
