This release bundles two deprecation cutovers plus a save-flow planner-first trim that together complete the v3.4.0.0 modernization arc.

**Phase 013 — `[spec]/memory/*.md` write path retirement.** The v3.4.0.0 release notes stated the surface was retired, but the runtime still created, wrote, indexed, and read from `[spec]/memory/` on every save — a deep-review audit (packet `026/013-memory-folder-deprecation-audit`, 8 iterations) surfaced 25 findings (9 P0 runtime contradictions) documenting the gap. Phase 013 cuts the write path, deletes the manual-snippet template, migrates tests and docs, and leaves the runtime consistent with the v3.4.0.0 intent.

**Phase 012 — Middleware cleanup (M9).** The v3.4.0.0 canonical-doc routing made the `/spec_kit:handover` and `/spec_kit:debug` wrapper commands plus the `@handover` and `@speckit` agents redundant: continuity now lives in canonical spec docs via `/memory:save`, and `@debug` dispatches via the Task tool. Phase 012 M9 deletes the 15 enumerated redundant paths, replaces the `@speckit` exclusivity narrative with a distributed-governance rule across the four canonical governance surfaces (`CLAUDE.md`, `AGENTS.md`, `AGENTS_example_fs_enterprises.md`, `system-spec-kit/SKILL.md`), and removes the `:auto-debug` flag from `/spec_kit:complete` in favor of explicit user escalation.

**Phase 015 — Save-flow planner-first trim.** After Phase 013 retired the `[spec]/memory/*.md` write path, a 20-iteration research pass (packet `026/014-save-flow-backend-relevance-review`) classified the 15 remaining save-flow subsystems and produced a `trim-targeted` verdict: keep the 4 load-bearing pieces (canonical atomic writer, routed record identity, content-router core, thin continuity sync), gate the 4 over-engineered pieces behind opt-in flags, and defer the 7 partial-value pieces out of the hot save path. Phase 015 implements that verdict: `/memory:save` now returns a non-mutating structured planner response by default (`SPECKIT_SAVE_PLANNER_MODE=plan-only`), the legacy full-auto pipeline is preserved behind explicit fallback, routing Tier 3 / quality-loop auto-fix / reconsolidation-on-save / post-insert enrichment are all off by default on the planner path, and `mcp_server/api/indexing.ts` exposes `refreshGraphMetadata`, `reindexSpecDocs`, and `runEnrichmentBackfill` as explicit follow-up entry points.

## Memory Folder Write Path Removed

The save workflow no longer produces `[spec]/memory/*.md` artifacts. Every category that used to land there now routes through the content-router into canonical spec docs, so the continuity surface is the same one the packet already tracks.

#### `[spec]/memory/` directories no longer created on save

`setupContextDirectory` used to unconditionally `mkdir [spec]/memory/` for every save, even when no memory file was produced. It has been replaced with `ensureSpecFolderExists`, which only validates that the target spec folder exists. New spec folders created via `create.sh` no longer get a `memory/` subdirectory, and the `--help` output no longer mentions one. Existing `memory/` directories in older packets are preserved as a read-only historical corpus.

&nbsp;

#### `writeFilesAtomically` and the memory/*.md write leg deleted

`scripts/core/file-writer.ts` was the primary producer of `[spec]/memory/YYYY-MM-DD__topic.md` files via atomic-commit semantics. The whole module is deleted. Step 9 of the save workflow no longer renders a memory file, no longer assigns a `ctxFilename`, and no longer calls the atomic-commit path. `writeFilesAtomically`, `checkForDuplicateContent`, and the sibling-glob dedup logic are gone.

&nbsp;

#### `indexMemory` wrapper removed; vector DB indexing goes through canonical docs

The script-side `scripts/core/memory-indexer.ts::indexMemory` wrapper previously called `vectorIndex.indexMemory({ filePath: path.join(contextDir, contextFilename) })` after every save, registering the new memory file as memory #N in the vector DB. That wrapper is deleted along with its `updateMetadataEmbeddingStatus` helper. Indexing now flows exclusively through Step 11.5 auto-index (introduced in this session's Step 11.5 patch), which re-indexes canonical spec docs + `graph-metadata.json` after save. The MCP handler call sites (`pe-gating.ts`, `chunking-orchestrator.ts`, `save/reconsolidation-bridge.ts`, `save/create-record.ts`) still call `vectorIndex.indexMemory()` directly with externally-supplied paths — those are unchanged.

&nbsp;

#### Dedup contract (phantom) and causal-link reads deleted

`save.md` §Session Deduplication previously specified a SHA-256 + 1h + interactive-prompt dedup contract with zero runtime implementation. The documentation section has been removed; the contract was a phantom spec. The pre-save SHA-1 overlap check that globbed `memory/*.md` siblings is also deleted. `autoPopulateCausalLinks` in `memory-metadata.ts` (the function that walked `[spec]/memory/*.md` to pre-populate causal graph edges) is deleted; packet-level relationships route through `graph-metadata.json.manual.depends_on/related_to/supersedes` and MCP causal-edge handlers.

## Save-Flow Planner-First Trim (Phase 015)

The v3.4.0.0 retirement plus Phase 013 cleanup left a save-flow backend that was larger than the simplified `[spec]/memory/*.md`-free contract needed. A 20-iteration research pass in packet `026/014-save-flow-backend-relevance-review` (cli-codex, gpt-5.4 high, fast tier) classified the 15 remaining subsystems into load-bearing / partial-value / over-engineered / dead. Phase 015 implements the resulting `trim-targeted` verdict: keep the 4 load-bearing pieces, gate the 4 over-engineered pieces behind opt-in flags, and expose the 7 partial-value pieces as explicit follow-up commands rather than always-on save-time automation.

&nbsp;

#### `/memory:save` returns a non-mutating planner by default

`memory-save.ts` now produces a structured planner response as its default behaviour. The response includes the resolved route target, proposed canonical-doc edits, blockers (hard structural/legality failures), advisories (score-heavy concerns), and a list of explicit follow-up actions (apply, refresh-graph, reindex, enrichment). The operator or the orchestrating AI applies the edits directly; the full-auto pipeline is reached only when the caller explicitly requests it. The feature flag is `SPECKIT_SAVE_PLANNER_MODE` with the three modes `plan-only` (default), `full-auto` (legacy pipeline), and `hybrid` (reserved for future mixed flows and currently behaviourally identical to `plan-only`).

&nbsp;

#### Routing, quality, reconsolidation, and enrichment gated off by default

- **Routing Tier 3 off:** Default planner-first path refuses low-confidence chunks with an explicit blocker instead of invoking the LLM reformulation layer. Tier 3 is available via `SPECKIT_ROUTER_TIER3_ENABLED=true`. The eight-category routing contract (`narrative_progress`, `narrative_delivery`, `decision`, `handover_state`, `research_finding`, `task_update`, `metadata_only`, `drop`) plus Tier 1 and Tier 2 dispatch are preserved, while the in-file Tier 3 default-disable/manual-review guard is a scoped exception documented in ADR-006.
- **Quality-loop auto-fix off:** Default planner-first path emits quality concerns as advisories in the planner response. Hard structural and canonical-doc legality checks still fail saves as blockers. The retry-driven auto-fix loop is available via `SPECKIT_QUALITY_AUTO_FIX=true`.
- **Reconsolidation-on-save off:** Default planner-first path skips supersede/reinforce/create-linked logic and surfaces "Reconsolidation available as follow-up action" advisories. The reconsolidation bridge is available via `SPECKIT_RECONSOLIDATION_ENABLED=true` or during `full-auto` fallback.
- **Post-insert enrichment off:** Default planner-first path skips entity extraction and cross-doc linking on save. The standalone follow-up path is `mcp_server/api/indexing.ts::runEnrichmentBackfill(specFolder)` plus `SPECKIT_POST_INSERT_ENRICHMENT_ENABLED=true` for in-save execution.

&nbsp;

#### Graph refresh and spec-doc reindex decoupled from the hot save path

`scripts/core/workflow.ts` no longer runs unconditional `graph-metadata.json` refresh or Step 11.5 spec-doc reindex on planner-default saves. Both operations remain callable as explicit follow-up actions via the new public indexing API:

- `mcp_server/api/indexing.ts::refreshGraphMetadata(specFolder)` — rebuilds `graph-metadata.json` from canonical docs.
- `mcp_server/api/indexing.ts::reindexSpecDocs(specFolder)` — triggers the Step 11.5 spec-doc reindex.
- `mcp_server/api/indexing.ts::runEnrichmentBackfill(specFolder)` — runs entity extraction + cross-doc linking against the canonical-doc state.

All three are re-exported from `mcp_server/api/index.ts` and are the operator-visible replacement for the implicit always-on behaviour the planner-default removes. The full-auto pipeline still runs them inline.

&nbsp;

#### Chunking is a size-driven fallback, not a default-path dependency

`handlers/chunking-orchestrator.ts` now gates chunking behind `shouldUseChunkedIndexing(content, plannerMode)`, which only activates in `full-auto` mode when content exceeds the chunk threshold. Planner-default saves never chunk, matching the non-mutating contract of the planner response.

&nbsp;

#### Load-bearing pieces preserved with one scoped router exception

Per the Phase 014 research verdict, four subsystems were classified `load-bearing` and MUST stay untouched:

- **Canonical atomic writer** (`handlers/save/atomic-index-memory.ts`): atomic promote/rollback semantics intact; unchanged.
- **Routed record identity** (`handlers/save/create-record.ts` + same-path lineage in `handlers/pe-gating.ts`): stable record identity across re-saves; unchanged.
- **Content-router core** (`lib/routing/content-router.ts` eight-category switch): core category mapping plus Tier 1 and Tier 2 dispatch preserved; ADR-006 records the scoped in-file Tier 3 default-disable/manual-review guard.
- **Thin continuity sync** (`lib/continuity/thin-continuity-record.ts`): `_memory.continuity` validation + upsert rules unchanged; tests verify parity.

### Phase 015 (Save-Flow Planner-First Trim)

- **4 load-bearing subsystems preserved with one documented router exception:** `handlers/save/atomic-index-memory.ts` (canonical atomic writer), `handlers/save/create-record.ts` (routed record identity), and `lib/continuity/thin-continuity-record.ts` (continuity validation + upsert) remain unchanged. `lib/routing/content-router.ts` keeps the eight-category routing core plus Tier 1 and Tier 2 dispatch intact, but Phase 015 added the scoped Tier 3 default-disable/manual-review guard documented in ADR-006. Same-path lineage in `handlers/pe-gating.ts` is also preserved.
- **Full-auto fallback preserves legacy behaviour:** Calling with `SPECKIT_SAVE_PLANNER_MODE=full-auto` (or passing `plannerMode: "full-auto"` via the CLI) reaches the exact same save pipeline that pre-015 callers saw, including reconsolidation, post-insert enrichment, Tier 3 routing, quality auto-fix, chunking, graph refresh, and Step 11.5 reindex. Seven env flags (`SPECKIT_ROUTER_TIER3_ENABLED`, `SPECKIT_QUALITY_AUTO_FIX`, `SPECKIT_RECONSOLIDATION_ENABLED`, `SPECKIT_POST_INSERT_ENRICHMENT_ENABLED`, plus the preserved `SPECKIT_INDEX_SPEC_DOCS` / `SPECKIT_AUTO_INDEX_TOUCHED`) let operators opt parts of the legacy behaviour back on without full-auto mode.
- **Follow-up API is additive:** `mcp_server/api/indexing.ts::refreshGraphMetadata / reindexSpecDocs / runEnrichmentBackfill` are new entry points and do not remove anything. Existing callers that reach the legacy pipeline are unaffected.
- **MCP tool names and content-router categories unchanged:** `memory_save`, `memory_index_scan`, `memory_context`, `memory_stats`, `memory_update`, and the eight routing categories keep their identifiers and input/output shape.
- **Trade-off accepted:** Default planner-first path does not produce immediate semantic reindex or graph refresh. Retrieval freshness on the default path depends on the operator/AI running the follow-up actions. `full-auto` remains available when immediate freshness matters; `hybrid` stays reserved until a distinct mixed-flow branch exists.
- **Trade-off accepted:** Default save path no longer performs reconsolidation, so duplicate canonical-doc entries are not collapsed automatically. Operators who depend on reconsolidation should either enable `SPECKIT_RECONSOLIDATION_ENABLED=true` or run `/memory:save` in `full-auto` mode.

### Phase 015 (Save-Flow Planner-First Trim)

The default behaviour of `/memory:save` changes from "apply mutations" to "return a non-mutating planner response". Any caller that depends on the legacy apply-on-save behaviour must opt into one of three preserved modes:

- **`full-auto` fallback (exact legacy pipeline):** Set `SPECKIT_SAVE_PLANNER_MODE=full-auto` in the environment, or pass `plannerMode: "full-auto"` in the CLI payload, or invoke `generate-context.js` with the `--full-auto` flag. This reaches the exact pre-015 pipeline, including reconsolidation, post-insert enrichment, Tier 3 routing, quality auto-fix, chunking, graph refresh, and Step 11.5 reindex.
- **`hybrid` mode (reserved):** `SPECKIT_SAVE_PLANNER_MODE=hybrid` is reserved for a future mixed-flow branch. Current runtime behaviour is identical to `plan-only`.
- **Per-subsystem env-flag opt-ins:** For operators that want planner-first default plus one or two legacy behaviours, enable `SPECKIT_ROUTER_TIER3_ENABLED=true`, `SPECKIT_QUALITY_AUTO_FIX=true`, `SPECKIT_RECONSOLIDATION_ENABLED=true`, or `SPECKIT_POST_INSERT_ENRICHMENT_ENABLED=true` individually. The flags stack: each enables only its own subsystem while leaving the others on the planner-default path.

**Follow-up API for callers that want planner-default plus explicit freshness:**

- `mcp_server/api/indexing.ts::refreshGraphMetadata(specFolder)` — rebuild `graph-metadata.json` from canonical docs.
- `mcp_server/api/indexing.ts::reindexSpecDocs(specFolder)` — run Step 11.5 spec-doc reindex.
- `mcp_server/api/indexing.ts::runEnrichmentBackfill(specFolder)` — entity extraction + cross-doc linking.

All three are exported from `mcp_server/api/index.ts`. They are additive and do not break existing callers.
