## [**3.0.0.4**] - 2026-03-28

The benchmarking system (used to measure how well the memory search works) was producing misleading results because it was comparing against the wrong database, and its test data was out of date. This release fixes those measurement problems and also points all 7 configuration files at the correct database -- the one in the repo with 2,417 memories instead of an external one with only 44.

> Spec folder: `.opencode/specs/02--system-spec-kit/022-hybrid-rag-fusion/023-ablation-benchmark-integrity/` (Level 2)

---

## What Changed

### Benchmarks now refuse to run against mismatched data

**Problem:** The ablation benchmark (a system that measures search quality by selectively disabling features) could run against a database that didn't match its reference test data. This produced meaningless scores that looked real.

**Fix:** Before running, the benchmark now checks that its test data and the active database share the same set of memory IDs. If they don't match, it stops immediately with a clear error instead of producing garbage results.

### Search has a special "evaluation mode" for benchmarks

**Problem:** During normal use, the search system applies confidence filters and token budgets (limits on how much text it returns). These are useful in production but distort benchmark measurements -- they hide results that the benchmark needs to see.

**Fix:** Benchmark callers can now pass `evaluationMode: true` to get raw, unfiltered results. Normal users are unaffected.

### Test data refreshed and previous conclusions corrected

The reference test data was rebuilt from scratch with 294 test cases across 126 memory IDs. With clean data, the full-text search feature (which finds exact word matches in stored memories) turned out to be helpful -- a previous benchmark had incorrectly concluded it was hurting search quality, but that was caused by the data mismatch described above.

### All configuration files now point to the right database

**Problem:** The system has 7 different configuration files (one per supported runtime: OpenCode, Claude, VS Code, Codex, Agents, Gemini, and a root config). They were pointing to an external database at `~/.codex/memories/` which only had 44 records. The actual database with 2,417 records lives inside the repo.

**Fix:** All 7 configs now point to the repo database. A backwards-compatibility shortcut (symlink) was added so the Codex CLI can still find the database at its old expected location.
