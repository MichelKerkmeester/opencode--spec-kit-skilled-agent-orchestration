## [**3.1.5.0**] - 2026-03-31

Sixteen fixes for the memory indexing pipeline, frontmatter compliance system, and spec templates. The `contextType` field -- which controls how memories are classified, decayed, and retrieved -- had the wrong default value ("decision") hardcoded in six separate locations across two codebases plus 41 template files. Every time the backfill script ran, it re-introduced the wrong value across hundreds of spec documents. The reindex pipeline also had three false-positive validation rules that silently blocked legitimate files from being indexed, leaving 36 memory files invisible to search. This release fixes all root causes, migrates all existing data, updates all templates, and removes 13,000 duplicate database rows.

> Spec folders: `.opencode/specs/02--system-spec-kit/023-esm-module-compliance/008-spec-memory-compliance-audit/` + `009-reindex-validator-false-positives/` (Level 2, Phases 8-9)

---

## 1. Cross-Spec Contamination Validator — False Positive on Bulk Reindex

**Problem:** During bulk reindex, the cross-spec contamination rule checked whether a file's body text referenced other spec IDs. It read the `spec_folder` from the file's YAML frontmatter to know which spec the file belonged to. But older memory files and all spec documents (spec.md, plan.md, checklist.md) don't have this field. Without it, the validator saw `current_spec: unknown` and treated every cross-reference as contamination -- blocking 1,106 files per reindex run.

**Fix:** When the frontmatter `spec_folder` field is empty, the validator now extracts the spec folder from the file's path (e.g., `.opencode/specs/02--system-spec-kit/024-compact-code-graph/memory/foo.md` resolves to `02--system-spec-kit/024-compact-code-graph`). The `filePath` parameter is threaded through the v-rule bridge and memory-save handler to make this possible.

## 2. Topical Coherence Validator — Blocked Memory and Spec Doc Files

**Problem:** The topical coherence rule blocked indexing when a file's content had zero keyword overlap with the parent spec's trigger phrases. This is too strict for memory files (which use different terminology than the spec) and spec documents (which define the triggers themselves). Six memory files were silently excluded from the search index.

**Fix:** The rule now skips the check entirely for files in `*/memory/` directories and for known spec document filenames (spec.md, plan.md, checklist.md, tasks.md, decision-record.md, implementation-summary.md, research.md, handover.md).

## 3. Template Contract Enforcement — Rejected Older Files During Force Reindex

**Problem:** Force reindex (`memory_index_scan` with `force: true`) rejected 90 memory files because they don't match the current template format. These files were created before the v2.2 template standard and are missing newer frontmatter fields or blank-line formatting.

**Fix:** When `force=true`, the quality gate uses `warn-only` mode for all files -- same as spec documents already did. Template issues are logged as warnings but don't block indexing. Interactive `memory_save` still enforces the full template contract.

## 4. Frontmatter Defaults — Source of Truth Used Wrong contextType

**Problem:** The `frontmatter-migration.ts` module (used by both `create.sh` and the backfill script) had three bugs: (1) `VALID_CONTEXT_TYPES` didn't include "planning", so any file with that value got rejected. (2) `DOC_DEFAULT_CONTEXT` mapped spec and plan documents to "decision" -- a value that downstream consumers don't recognize. (3) `normalizeContextType()` aliased "planning" back to "decision", actively reversing any manual fix. Every backfill run re-introduced the wrong value across all spec documents.

**Fix:** Added "planning" to `VALID_CONTEXT_TYPES`. Changed defaults: spec→implementation, plan→planning, decision_record→planning. Reversed the alias so "decision"→"planning" (migrating legacy values). `inferContextType()` now overrides existing "decision" values instead of preserving them.

## 5. MCP Parser — CONTEXT_TYPE_MAP Had Same Bug

**Problem:** The MCP server's memory parser (`memory-parser.ts`) had an independent copy of the context type mapping that also mapped "planning" to "decision". When files were reindexed, the parser converted the corrected frontmatter value back to the wrong one before storing it in the database.

**Fix:** Updated `CONTEXT_TYPE_MAP` to map "decision"→"planning" and "discovery"→"general". Added "planning" to the `ContextType` TypeScript union.

## 6. Session Extractor — detectContextType Returned "decision"

**Problem:** `detectContextType()` in `session-extractor.ts` returned "decision" when decisions were recorded and "discovery" for web-heavy sessions. These values propagated into saved memory files.

**Fix:** Returns "planning" for decision-heavy sessions and "research" for web-heavy sessions.

## 7. Runtime Consumers — Four More Locations Referenced "decision"

**Problem:** Four runtime systems used "decision" as a contextType value: (1) `intent-classifier.ts` weighted `find_spec` and `find_decision` intents toward "decision". (2) `save-quality-gate.ts` had a special exception for `context_type === 'decision'`. (3) `fsrs-scheduler.ts` treated "decision" as no-decay in both the stability multiplier and hybrid decay policy. (4) `memory-state-baseline.ts` validated against a set that excluded "planning".

**Fix:** All four updated to use "planning" as the primary value, with "decision" accepted as a legacy alias for backward compatibility.

## 8. Database Schema — CHECK Constraint Missing "planning"

**Problem:** The `context_type` column in `memory_index` had a CHECK constraint that only allowed 'research', 'implementation', 'decision', 'discovery', 'general'. Any attempt to store "planning" failed with a constraint violation.

**Fix:** Updated the CHECK constraint in `vector-index-schema.ts` and `schema-downgrade.ts` to include "planning". Applied the same constraint change to the live database.

## 9. Database Migration — 2,009 Rows With Legacy Values

**Problem:** The live database had 2,006 entries with `context_type = 'decision'` and 3 with `context_type = 'discovery'` from before the migration.

**Fix:** Direct SQL UPDATE migrated all legacy values: decision→planning, discovery→general.

## 10. Database Deduplication — 13,211 Duplicate Rows

**Problem:** Each force reindex created new rows instead of updating existing ones. After multiple reindex runs, the database had 14,415 rows but only 1,204 unique file paths -- 92% waste. This inflated search result counts and slowed queries.

**Fix:** Deleted all duplicate rows, keeping only the latest entry per file path. Database went from 14,415 to 1,200 unique entries.

## 11. Retroactive Backfill — 828 Spec Documents Fixed

**Problem:** After fixing the source defaults, all existing spec documents still had the old values in their frontmatter YAML.

**Fix:** Ran `backfill-frontmatter.js --apply --include-archive --allow-malformed` across all 2,383 spec documents including archived folders. 828 files were updated. Zero files with `contextType: "decision"` remain anywhere in the codebase.

## 12. Context Template — Detection Logic Updated

**Problem:** The `context_template.md` pseudo-code for contextType detection returned "decision" for decision-heavy sessions and "discovery" for web-heavy sessions. The comment on the `context_type` field listed "decision|discovery" as valid values.

**Fix:** Updated detection logic to return "planning" and "research" respectively. Updated the field comment to list "implementation|planning|research|general".

## 13. Templates — All 41 Template Files Had Wrong Default

**Problem:** Every template file in `templates/` (across level_1, level_2, level_3, level_3+, core, and addendum directories) used `contextType: "general"` regardless of document type. When a new spec folder was created from these templates, the resulting files started with the wrong contextType and required a backfill pass to correct.

**Fix:** Updated all 41 template files to use the correct doc-type-specific default: spec.md→"implementation", plan.md→"planning", decision-record.md→"planning", checklist.md→"implementation", tasks.md→"implementation", implementation-summary.md→"implementation", research.md→"research". README.md, handover.md, and debug-delegation.md remain "general". New spec folders now start with the correct value without needing a backfill.

## 14. Parser Tests — Expectations Still Encoded Old Contract (Review P1-1)

**Problem:** The `memory-parser-extended.vitest.ts` test suite still expected `planning` to alias to `decision` and `bug` to alias to `discovery`. Three test assertions failed after the CONTEXT_TYPE_MAP change.

**Fix:** Updated T06 canonical types to `['implementation', 'research', 'planning', 'general']`. Updated T07 aliases to `{ decision: 'planning', bug: 'implementation' }`. Added T08 for explicit legacy value assertions.

## 15. Regex Fallback — Single-Level Spec Paths Not Matched (Review P1-2)

**Problem:** The spec folder extraction regex required two-level nesting (`specs/track/NNN-name/`). Single-level paths like `specs/001-feature/memory/foo.md` didn't match, leaving `current_spec: unknown` for files in single-level spec folders.

**Fix:** Changed regex from `/[/\\]specs[/\\](.+?[/\\](?:\d{3}-[^/\\]+))/` to `/[/\\]specs[/\\]((?:[^/\\]+[/\\])*\d{3}-[^/\\]+)/`. Now captures everything from `specs/` up to the last `NNN-` prefixed segment, matching both nested and single-level paths.

## 16. Test Coverage — No Tests for filePath Fallback or V12 Skip (Review P1-6)

**Problem:** The exact reindex/save path that motivated the V8 and V12 fixes had no regression tests. All existing validator tests called `validateMemoryQualityContent(content)` without the new `{ filePath }` option.

**Fix:** Added 3 tests to `validate-memory-quality.vitest.ts`: (1) V8 filePath fallback extracts spec folder from path when frontmatter `spec_folder` is empty. (2) V12 skips topical coherence for files in `*/memory/` directories. (3) V12 skips topical coherence for spec doc files (spec.md).

## Upgrade

Run the backfill to fix frontmatter in your spec documents:
```bash
node .opencode/skill/system-spec-kit/scripts/dist/memory/backfill-frontmatter.js --apply --include-archive
```

Then force reindex to update the database:
```bash
node .opencode/skill/system-spec-kit/scripts/dist/memory/reindex-embeddings.js
```
