## [**1.1.0.0**] - 2026-04-11

This release lands the 042 Phase 5 "agent-improver deep-loop alignment" work plus the `sk-agent-improver to sk-improve-agent` rename. The improver now has its own frozen stop-reason taxonomy, append-only audit journal, mutation coverage graph with trade-off detection, directed candidate run history, and benchmark stability scoring - aligning its runtime truth model with `sk-deep-research` and `sk-deep-review`. The proposal-only agent constraint is preserved: all new journaling happens orchestrator-side, never from the improver itself. The goal is to make the workflow easier to follow and easier to trust.

---

### Skill Rename: sk-agent-improver → sk-improve-agent

- The skill and its sibling `sk-prompt-improver to sk-improve-prompt` were renamed to the verb-first `improve:*` convention used by the other `/improve:*` slash commands.
- Every playbook scenario, agent mirror, command doc, YAML workflow, asset, and reference path was updated. Legacy `sk-agent-improver` paths are fully removed from live surfaces.
- The `/improve:agent` command continues to drive the loop; no operator-facing command changed.

---

### Stop-Reason Taxonomy and Audit Journal

- Frozen `STOP_REASONS` enum published alongside a `SESSION_OUTCOMES` enum so improvement sessions report the exact gate that tripped (`convergedImprovement`, `benchmarkPlateau`, `rejectCandidate`, `blockedStop`, `maxIterationsReached`, `rollbackTriggered`, `manualStop`, `error`).
- Added `scripts/improvement-journal.cjs` - append-only JSONL journal with event-type validation. Event types: `session_start`, `candidate_generated`, `scored`, `gate_evaluation`, `session_end`. `session_ended` requires `stopReason` and `sessionOutcome`.
- Journal emission is **orchestrator-only** so the proposal-only agent constraint (ADR-001) is preserved - the improver never writes its own audit trail.
- A CLI entrypoint (`node .../improvement-journal.cjs --emit <event-json> | --read <session>`) allows replay tools and tests to consume the journal directly.

---

### Mutation Coverage and Trade-Off Detection

- Added `scripts/mutation-coverage.cjs` - coverage graph for candidate mutations, session scoped via `loop_type: "improvement"` so it shares the same SQLite-backed graph store used by the deep-research and deep-review loops without collisions.
- Added `scripts/trade-off-detector.cjs` - Pareto-aware dimension trade-off detection. When a candidate improves one dimension but regresses another, the detector surfaces the trade-off rather than silently accepting the regression.
- Added `scripts/candidate-run history.cjs` - directed run history graph for candidate genealogy, so promotion decisions can trace back through the full derivation history of a proposed candidate.
- Added `scripts/benchmark-stability.cjs` - benchmark stability coefficient computed as `1 - (stddev / mean)` across replicate runs, gating promotion on measurement reliability.

---

### Scoring Weight Optimizer and Benchmark Stability

- the score dynamic helper now accepts a `weights` override so evaluators can tune the 5-dimension scoring per target profile without forking the base scorer.
- The benchmark scanner output format was mapped into `run-benchmark.cjs` so the coverage graph can ingest actual run results rather than synthesized metrics.
- Reducer rejection enum gained `reject-candidate` so a failed candidate can be cleanly journaled without appearing as an error.

---

### Playbook and Operator Guidance

- Added a new `07--runtime-truth/` playbook category with seven scenarios (`025-stop-reason-taxonomy.md` through `031-parallel-candidates-opt-in.md`) covering the full runtime truth expected behavior: stop reasons, audit journal, resume semantics, legal-stop gates, benchmark stability, dimension trajectory, and opt-in parallel candidates.
- Added three graph-era scenarios under `06--end-to-end-loop/`: mutation coverage tracking (`022`), trade-off detection (`023`), and candidate run history (`024`).
- Every playbook scenario was executed against the updated runtime - **31/31 PASS**, covering both pre-existing loop scenarios and all new Phase 5 additions.

---

### Verification

- Five P1 review findings from the ten-iteration deep-review pass were closed: CLI entrypoint, reducer rejection enum, required stopReason/sessionOutcome, scanner output mapping, scoreDynamic weights override.
- Full vitest suite green (10,335 passing, 0 failing).
- Backward compatibility verified: existing sessions without new config fields still run to completion.

## Upgrade

Existing improvement sessions without the new runtime-truth fields continue to run unchanged (backward compatible). Operators invoking `/improve:agent` now get a reviewable audit journal, mutation coverage graph, trade-off detection, and stability-aware promotion gates - all enforced orchestrator-side so the proposal-only agent constraint is preserved. Legacy `sk-agent-improver` paths are fully removed; update any external references to `sk-improve-agent`.

---

> **Update (post-release) - 2026-04-11:**
>
> This release documented four lifecycle modes for re-entering an `/improve:agent` session: `resume`, `restart`, `fork`, and `completed-continue`. None of the four were ever wired up. The command has no input for lineage mode, the reducer does not carry ancestry across sessions, and the journal helper has no multi-session boundary event. Every invocation is a fresh `new` session.
>
> Release `v1.2.1.0` retracts the promise. Operators who want to continue evaluating an agent across sessions should archive the prior `improve/` folder under a timestamped path and re-invoke the command. See `v1.2.1.0.md` for the full story.
