## [**1.2.1.0**] - 2026-04-11

Patch release closing the 042 closing-audit remediation as it affects `sk-improve-agent`. This skill was called out in the review because `SKILL.md §Resume/Continuation Semantics` promised resumable lineage modes (`resume`, `restart`, `fork`, `completed-continue`) that the shipped workflow, reducer, and journal consumer never actually supported. This entry retracts the claim and locks the shipped contract.

---

### Resume/Continuation Retraction (Lane 3)

- `SKILL.md §Runtime Truth / Resume/Continuation Semantics (current release)` now explicitly states the runtime only supports a single lineage mode: `new`. Every invocation of `/improve:agent` starts a fresh session with a new session id and generation 1. Multi-generation lineage modes were described in earlier drafts but have no workflow inputs, reducer ancestry output, or journal boundary events to back them.
- Operators who want to continue evaluating an agent after a prior session should archive the prior session folder (e.g. move `improve/` to `improve_archive/{timestamp}/`) and re-invoke the command. The reducer treats each session independently and does not carry ancestry across sessions.
- `.opencode/command/improve/agent.md` line 339 `Supported lineage modes` callout was shrunk to `new` only, with an explicit note pointing at the SKILL.md retraction and flagging `resume`/`restart`/`fork`/`completed-continue` as deferred.
- Historical `v1.1.0.0` changelog now carries an append-only "Update (post-release)" block pointing at this entry so readers can trace the retraction.

If the long-form lineage feature is implemented later it will arrive with first-class event emission in `improve_agent-improver_{auto,confirm}.yaml`, reducer ancestry handling in `sk-improve-agent/scripts/reduce-state.cjs`, and replay fixtures. Until then, treat every `/improve:agent` session as a standalone evaluation.
