## What Changed

#### Provider Auth Pre-Flight (Smart Fallback) subsection added

New ┬¦3 subsection codifies the pre-dispatch auth check for OpenAI credentials. One-shot bash pre-flight detects `$OPENAI_API_KEY` env var and `codex auth status` ChatGPT OAuth output, capturing `OPENAI_KEY_OK` and `CODEX_OAUTH_OK` flags. A 3-state decision table (default-available / API-key-missing-OAuth-ready / both-missing) governs routing. Two user-prompt templates (key-missing-with-OAuth-fallback A/B/C menu; both-missing login-command menu including `codex login`) make the ASK-not-substitute contract explicit. The default-available action shows the canonical `codex exec --model gpt-5.5 -c model_reasoning_effort="medium" -c service_tier="fast"` shape. Cache invalidation on auth-error mid-dispatch reruns the pre-flight before retry.

#### Position in ┬¦3

Inserted between Prerequisites and the existing Default Invocation block, matching the cli-opencode donor ordering. No existing subsections renamed or reordered.

## Upgrade

No invocation breaking changes. Existing `codex exec` dispatches keep working. AI consumers of the skill should now run the Provider Auth Pre-Flight on session start (a one-time `codex auth status 2>&1; env | grep -E '^OPENAI_API_KEY'` call) before the first dispatch, cache the result for the session, and apply the 3-state decision tree if `OPENAI_KEY_OK=0`.

The pre-flight matches the cli-opencode v1.3 pattern exactly ŌĆö same ASK-not-substitute language, same cache-invalidation contract, same 3-state column structure. Substitutions are limited to the detection command, var names, model invocation (`gpt-5.5` with `model_reasoning_effort` config), and login commands.
