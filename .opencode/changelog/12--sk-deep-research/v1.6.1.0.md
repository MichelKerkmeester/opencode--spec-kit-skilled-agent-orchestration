## [**1.6.1.0**] - 2026-04-11

This patch release closes the remaining 042 closing-audit fallout for `sk-deep-research`. It retracts two lifecycle modes that never actually worked, documents the graph-event format the runtime already depends on, and fixes session-collision risk in the shared graph cache.

> Spec folder: `.opencode/specs/skilled-agent-orchestration/042-sk-deep-research-review-improvement-2/` (Level 3)

---

## Bug Fixes

These fixes bring the shipped documentation back into line with what the runtime actually does. The normal happy path stays the same, but unsupported or under-documented paths are now described honestly and stored more safely.

---

### Two lifecycle options no longer pretended to work

**Problem:** When a research packet already existed, the workflow asked operators to pick how to continue: resume the same run, start a fresh run, `fork` into a sibling run history, or `completed-continue` (reopen a finished run after snapshotting its synthesis). The first two were real. The last two were described in several references but had no workflow wiring - picking them did the same thing as `resume` and the packet quietly kept the old session id. Operators who thought they had branched a research run history ended up overwriting the original.

**Fix:** `fork` and `completed-continue` are now explicitly marked as deferred (reserved for a future release) everywhere the skill mentions them - `SKILL.md`, `README.md`, the two loop-protocol references, both command workflows, and the runtime agent definitions across all three CLI mirrors (Claude Code, Codex, Gemini). The visible lifecycle choices are now `new`, `resume`, and `restart`. `resume` keeps the current session id and appends a typed `resumed` event to the state log. `restart` archives the existing `research/` folder under a timestamped `research_archive/` directory, mints a fresh session id, and appends a typed `restarted` event with the archive path. Both events carry the full run history field set the reducer reads.

---

### Graph events had no documented shape or namespace rules

**Problem:** Each research iteration can emit a `graphEvents` array into its state-log record. The coverage-graph MCP tools consume those events and persist them to a shared SQLite cache. But nothing in `state_format.md` said what each event looked like (what keys are required, what values are allowed, how nodes and edges relate). Reviewers and future maintainers had to read the MCP handler source to figure out the expected behavior, and at least one external consumer had started salting ids with the session id to "avoid collisions" - which the storage layer handles automatically and which actively broke provenance tracing.

**Fix:** `references/state_format.md` now has a dedicated Graph Events section that documents every field (`type`, `id`, `label`, optional `relation`, `source`, `target`, `metadata`), lists the accepted type values (`question`, `finding`, `claim`, `source`, `edge`), gives a worked example, and spells out the four session scope rules the storage layer enforces. Producers are told in plain terms: do not salt ids with the session id - the database already handles isolation.

---

### Session isolation in the shared graph database was incomplete

**Problem:** Two research sessions that accidentally picked the same logical node or edge id (`q-reconnect`, `f-001`, etc.) could silently overwrite each other's rows in the shared coverage-graph SQLite cache. The filtered reads worked fine, but the writes keyed on the bare id. This was caught by the closing audit and is documented here because `sk-deep-research` consumes the same storage layer through the graph convergence MCP tool.

**Fix:** The shared coverage-graph database schema is now on version 2 with a composite primary key of (spec folder, loop type, session id, logical id) on both node and edge tables. Migration on first open drops the old tables (the cache is dev-only, not durable state) and creates the new ones. The change is transparent to `sk-deep-research` callers: every iteration's `graphEvents` still round-trip exactly the same way, but two sessions can now safely reuse the same logical id.

## Upgrade

No migration is required. Existing research packets continue to resume exactly as before. Operators who were relying on the `fork` or `completed-continue` lifecycle choices should archive the prior `research/` folder manually or pick `restart`, which does both in one step. The two deferred modes remain reserved for a future release and will return only when they ship with first-class event emission and replay coverage.
