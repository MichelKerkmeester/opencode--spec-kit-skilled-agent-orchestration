This release finishes what v1.8.0.0 started. Phase 018 wired `cli-codex` as the only new executor; three more (`cli-copilot`, `cli-gemini`, `cli-claude-code`) were reserved in the config enum but rejected at load time. This release wires all three. Each CLI has a different flag surface so the config schema now validates per-kind: setting a flag the chosen CLI does not support rejects with a clear error naming the incompatible combination. The YAML workflow keeps owning state, convergence, lifecycle, and the reducer. Cross-CLI delegation is documented as design intent, not coded as enforcement. Ships as a symmetric pair with `sk-deep-review` v1.6.0.0.

> Spec folder: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/019-cli-runtime-matrix` (Level 3)

## New Features

#### cli-copilot wired as executor

Before this release, picking `kind: cli-copilot` in the config rejected at parse time with `ExecutorNotWiredError`. Now the config accepts cli-copilot, the YAML dispatch step has an `if_cli_copilot` branch, and iterations run via `copilot -p "PROMPT" --model X --allow-all-tools --no-ask-user`. Copilot has no stdin support so the prompt is passed positionally via command substitution. Copilot also has no CLI flag for reasoning effort (that is set in `~/.copilot/config.json` ahead of time) and no service-tier flag, so the config validator rejects either combination with a message pointing to those limits.

&nbsp;

#### cli-gemini wired as executor

Setting `kind: cli-gemini, model: gemini-3.1-pro-preview` now dispatches through `gemini "PROMPT" -m gemini-3.1-pro-preview -s none -y -o text`. Gemini CLI currently supports a single model, so the config schema enforces a whitelist. Users who pass an unknown gemini model string get a clear error listing the supported models. Gemini has no reasoning-effort or service-tier flags; the config rejects either of those combinations.

&nbsp;

#### cli-claude-code wired as executor

Setting `kind: cli-claude-code, model: claude-opus-4-6` dispatches through `claude -p "PROMPT" --model claude-opus-4-6 --permission-mode acceptEdits --output-format text`. Optional `reasoningEffort` adds `--effort X`. Claude Code's default permission mode is `plan` which is read-only, so this branch overrides to `acceptEdits` so iterations can actually write their outputs. Claude Code has no service-tier flag; the config rejects that combination.

&nbsp;

#### Per-kind flag-compatibility validation

Phase 018's flat config block is preserved (same six fields in both deep_research_config.json and deep_review_config.json). What is new is a per-kind support matrix inside `executor-config.ts`. The validator iterates over the non-default flag values and checks each against the kind's supported set. Incompatible combinations produce typed errors with messages like: *"field 'serviceTier' is not supported by executor kind 'cli-gemini'. Supported fields for cli-gemini: model, sandboxMode, timeoutSeconds."* The alternative design (a Zod discriminated union with per-kind sub-schemas) was rejected because it would break the config symmetry between the two skills and churn the optimizer manifest every time a new kind lands.

&nbsp;

#### Executor audit on new kinds

The `record_executor_audit` step from Phase 018 now fires on any non-native run. Users running cli-copilot, cli-gemini, or cli-claude-code get the same `executor: {kind, model, reasoningEffort, serviceTier}` block appended to the last JSONL record that cli-codex runs already produce. The audit lets anyone reading the state log later see exactly which CLI produced each iteration, regardless of which of the five executor kinds was active.

&nbsp;

#### Five setup flag options

The consolidated setup prompt in `deep-research.md` previously offered two executor choices: native (A) and cli-codex (B). It now offers all five: A native, B cli-codex, C cli-copilot, D cli-gemini, E cli-claude-code. Each option briefly notes the CLI's canonical flags and its notable limitations (copilot has no reasoning-effort flag, gemini has a single supported model, claude-code has no service-tier). Users who pass `--executor=cli-copilot` on the command line never see the question.

## Documentation

#### Executor Selection Contract table updated

Both `SKILL.md` files had a contract table listing the four CLI kinds and marking three of them as "Not wired — awaits future spec." Those rows now say "Shipped (spec 019)." The contract no longer lies about availability.

&nbsp;

#### Cross-CLI Delegation subsection

Both SKILL.md files get a new subsection under the Executor Selection Contract titled Cross-CLI Delegation. The subsection documents what is possible (any CLI executor can shell out to any other CLI from within its iteration sandbox), what is forbidden (self-recursive invocation, which every individual CLI skill already warns against), and what is user-responsibility (auth and environment propagation between CLIs — each CLI uses its own credentials, the deep-loop workflow does not mediate). Runtime enforcement of cross-CLI rules is explicitly out of scope: it is design intent documented in prose, not a code path.

&nbsp;

#### Loop protocol references expanded

Section 3 of both `references/loop_protocol.md` files previously described Executor Resolution for spec 018 only (native + cli-codex). The section now describes all five kinds with their canonical dispatch shapes and cites the per-kind flag-compatibility enforcement in `executor-config.ts`.

## Upgrade

No migration required. Default `kind: native` is unchanged. Users already on `cli-codex` get no behavior change. Users previously blocked from `cli-copilot`, `cli-gemini`, or `cli-claude-code` by the `ExecutorNotWiredError` rejection can now run those CLIs.

To opt into a new CLI, invoke:

```
/spec_kit:deep-research :auto "topic" \
  --executor=cli-copilot \
  --model=gpt-5.4
```

Or for gemini:

```
/spec_kit:deep-research :auto "topic" \
  --executor=cli-gemini \
  --model=gemini-3.1-pro-preview
```

Or for claude-code:

```
/spec_kit:deep-research :auto "topic" \
  --executor=cli-claude-code \
  --model=claude-opus-4-6 \
  --reasoning-effort=high
```

Each CLI must be on PATH and authenticated. Setting a flag that the chosen CLI does not support (for example `--service-tier=fast` with `--executor=cli-gemini`) rejects at config parse time with a message naming the unsupported field.
