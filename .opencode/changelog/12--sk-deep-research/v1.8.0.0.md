This release lets you pick which AI actually runs each research iteration. Before, every iteration dispatched through the native research agent using Claude Opus. Now you can opt into running the iteration through a CLI executor such as `codex exec` with a chosen model, reasoning effort, and service tier. The skill's YAML workflow still owns all the loop machinery (state tracking, convergence detection, lifecycle events, the reducer). The CLI is a tool inside the workflow, not a replacement for it. Ships as a symmetric pair with `sk-deep-review` v1.5.0.0.

> Spec folder: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/018-deep-loop-cli-executor` (Level 3)

## New Features

#### Executor branch in the iteration step

Before this release, the YAML step that runs each iteration had a hardcoded line saying "dispatch the research agent using Opus". There was no way to pick a different executor without rewriting the YAML. The iteration step now branches on a config field called `executor.kind`. The `native` branch (the default) keeps the exact same agent dispatch as before. The new `cli-codex` branch runs `codex exec` with the model and flags pulled from the config, piping the iteration prompt through stdin. A user who does not set anything sees identical behavior to previous releases, so nothing breaks for existing sessions.

&nbsp;

#### Shared prompt pack template

Before, the iteration context (state summary, focus area, constraints, file paths) lived inline inside the YAML file. That worked for the native agent because the YAML runtime could pass the inline string directly into the agent's context. It did not work for a CLI executor which needs a file it can read. The YAML block has been extracted into a template file at `assets/prompt_pack_iteration.md.tmpl`. Both dispatch paths render the same template and see the same content. No implicit context gap between the native agent path and the CLI path.

&nbsp;

#### Config schema for executor selection

Adds a new `executor` block to `deep_research_config.json` with six fields: `kind`, `model`, `reasoningEffort`, `serviceTier`, `sandboxMode`, `timeoutSeconds`. Default is `kind: "native"` with every other field null. That means any existing config file keeps working without changes. Invalid combinations (picking `cli-codex` without setting a model, or picking an executor that is reserved but not yet wired) are caught at config load time with typed errors.

&nbsp;

#### Setup flags in the command doc

Five new flags were added to the consolidated setup prompt in `deep-research.md`: `--executor`, `--model`, `--reasoning-effort`, `--service-tier`, `--executor-timeout`. Precedence is CLI flag first, then config file, then the schema default. If none of these are specified and the research topic does not mention an executor, the setup prompt adds a single optional question so the user can pick native or cli-codex. Existing invocations that use none of the new flags keep the native default.

&nbsp;

#### Executor audit on the iteration record

When a non-native executor runs an iteration, the workflow appends an `executor: {kind, model, reasoningEffort, serviceTier}` block to that iteration's JSONL record. Anyone reading the state log later can see exactly which executor produced each iteration. Native runs skip the audit because they are recoverable from the YAML itself.

&nbsp;

#### Post-dispatch validation

Every executor (native or CLI) must produce two things per iteration: a markdown narrative file and a JSONL delta record with required fields. A new validator checks both after dispatch. If the iteration file is missing or empty, or if the JSONL delta was not appended, or if it is missing required fields, the validator emits a canonical `schema_mismatch` conflict event. Three consecutive validation failures trigger the existing stuck-recovery path.

## Documentation

#### Executor Selection Contract in SKILL.md

The forward-looking note at line 61 of `SKILL.md` that said "if the user specifies an executor CLI, that is the how" has been replaced with a full contract section. The section documents which kinds are shipped, which are reserved, what invariants every executor must respect, how failure is detected and recovered, and where the config and template live. Anyone reading SKILL.md now sees the real behavior, not a forward-looking promise.

&nbsp;

#### Executor Resolution in loop_protocol.md

Section 3 of the loop protocol reference gets a new subsection explaining how the branch selection interacts with the existing `schema_mismatch` and stuck-recovery paths. Branch selection does not change any recovery semantics. The existing recovery flow handles CLI failures the same way it handles agent failures.

## Upgrade

No migration required. `executor.kind` defaults to `native`, which preserves pre-release behavior exactly. Same agent dispatch, same iteration file, same JSONL record.

To opt into `cli-codex` dispatch, invoke:

```
/spec_kit:deep-research :auto "topic" \
  --executor=cli-codex \
  --model=gpt-5.4 \
  --reasoning-effort=high \
  --service-tier=fast
```

Requires `codex` on PATH. Reserved kinds (`cli-copilot`, `cli-gemini`, `cli-claude-code`) are declared in the config enum but throw `ExecutorNotWiredError` at config load with a pointer to future specs.
