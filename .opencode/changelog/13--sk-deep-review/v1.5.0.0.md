This release lets you pick which AI actually runs each review iteration. Before, every iteration dispatched through the native review agent using Claude Opus. Now you can opt into running the iteration through a CLI executor such as `codex exec` with a chosen model, reasoning effort, and service tier. The skill's YAML workflow still owns all the loop machinery (state tracking, convergence detection, findings registry, dashboard, the reducer). The CLI is a tool inside the workflow, not a replacement for it. Ships as a symmetric pair with `sk-deep-research` v1.8.0.0.

> Spec folder: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/018-deep-loop-cli-executor` (Level 3)

## New Features

#### Executor branch in the iteration step

Before this release, the YAML step that runs each review iteration had a hardcoded line saying "dispatch the review agent using Opus". There was no way to pick a different executor. The iteration step now branches on a config field called `executor.kind`. The `native` branch (the default) keeps the exact same agent dispatch as before. The new `cli-codex` branch runs `codex exec` with the model and flags pulled from the config, piping the review iteration prompt through stdin. A user who does not set anything sees identical behavior to previous releases. Review dimensions, findings severity, and verdict logic all stay exactly where they were.

&nbsp;

#### Shared prompt pack for review

The review iteration context (state summary, dimension, target, scope files, prior findings counts, quality gates, verdicts, claim adjudication requirements) lived inline inside the YAML. It has been extracted into a template file at `assets/prompt_pack_iteration.md.tmpl`. Both dispatch paths render the same template and see the same content. Every review-specific constraint is preserved in the template: correctness, security, traceability, maintainability dimensions, evidence, scope, coverage quality gates, FAIL, CONDITIONAL, PASS verdicts with `hasAdvisories=true` option, and the claim adjudication schema for new P0 and P1 findings.

&nbsp;

#### Config schema for executor selection

Adds a new `executor` block to `deep_review_config.json` that is bit-identical to the one in `deep_research_config.json`. Same six fields, same defaults, same validation. One contract, two consumers. The two configs stay aligned by design.

&nbsp;

#### Setup flags in the command doc

Five new flags were added to the consolidated setup prompt in `deep-review.md`: `--executor`, `--model`, `--reasoning-effort`, `--service-tier`, `--executor-timeout`. Precedence is CLI flag first, then config file, then the schema default. Users get the same UX across both iterative skills, so no need to learn two different configurations.

&nbsp;

#### Executor audit on review records

When a non-native executor runs a review iteration, the workflow appends an `executor: {kind, model, reasoningEffort, serviceTier}` block to that iteration's JSONL record. The review JSONL schema (dimensions, filesReviewed, findingsSummary, newFindingsRatio) is preserved unchanged. The audit is a new top-level field, not a replacement for any existing field.

&nbsp;

#### Post-dispatch validation for review

Every executor (native or CLI) must produce a markdown narrative file and a JSONL delta record with the required review fields. The shared validator checks both after dispatch. If either is missing or malformed, a canonical `schema_mismatch` conflict event fires. Three consecutive validation failures trigger the existing stuck-recovery path.

## Upgrade

No migration required. `executor.kind` defaults to `native`, which preserves pre-release behavior exactly. Same agent dispatch, same iteration file, same JSONL record with review dimensions, same findings registry semantics.

To opt into `cli-codex` dispatch for review:

```
/spec_kit:deep-review :auto "target" \
  --executor=cli-codex \
  --model=gpt-5.4 \
  --reasoning-effort=high \
  --service-tier=fast
```

Requires `codex` on PATH. Reserved kinds (`cli-copilot`, `cli-gemini`, `cli-claude-code`) are declared in the config enum but throw `ExecutorNotWiredError` at config load with a pointer to future specs.

## Symmetry with sk-deep-research v1.8.0.0

Both skills ship the same feature in the same release cycle. Design rationale in the spec packet's `decision-record.md` ADR-005: asymmetric shipping creates the bug class Phase 017 Cluster D was built to prevent, where six code-graph handlers lacked contract vocabulary for months because only one was updated. Co-shipping avoids that pattern here. All four shared TypeScript modules are consumed by both skills. All four YAMLs (research auto, research confirm, review auto, review confirm) use the same two-branch structure.
