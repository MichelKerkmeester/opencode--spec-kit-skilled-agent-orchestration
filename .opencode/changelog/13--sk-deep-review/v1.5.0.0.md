This release adds CLI executor selection to `sk-deep-review` in symmetry with `sk-deep-research` v1.8.0.0. The review loop's iteration dispatch now branches on `config.executor.kind` — `native` (default, preserves current `@deep-review` + Opus dispatch bit-identically) or `cli-codex` (invokes `codex exec` with configurable model, reasoning effort, and service tier). YAML keeps owning state, convergence, lifecycle, findings registry, and the reducer. The CLI is a tool inside the workflow, not a replacement for it.

> Spec folder: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/018-deep-loop-cli-executor/` (Level 3)

---

## What Changed

#### Two-branch dispatch in the YAML workflow

`step_dispatch_iteration` in both auto and confirm YAMLs was refactored to match the research skill's structure. The old hardcoded `agent: deep-review, model: opus` block is now the `if_native` branch. Review sessions that never touch the new config see identical behavior to previous releases — same agent, same model, same iteration file, same JSONL record with dimensions + findings.

The new `if_cli_codex` branch invokes:

```bash
codex exec \
  --model <config.executor.model> \
  -c model_reasoning_effort="<config.executor.reasoningEffort>" \
  -c service_tier="<config.executor.serviceTier>" \
  -c approval_policy=never \
  --sandbox workspace-write \
  - < {spec_folder}/review/prompts/iteration-{n}.md
```

Both branches share pre-dispatch prompt rendering (review-flavored template), post-dispatch invariant validation (asserts iteration file + JSONL delta + required review fields), and executor audit recording on non-native runs.

#### Executor-agnostic prompt pack

A new template at `.opencode/skill/sk-deep-review/assets/prompt_pack_iteration.md.tmpl` replaces the inline `context: |` block inside the YAML. The template preserves every review-specific constraint: `correctness / security / traceability / maintainability` dimensions, `evidence / scope / coverage` quality gates, `FAIL / CONDITIONAL / PASS` verdicts with `hasAdvisories=true` option, claim adjudication requirements for new P0/P1 findings, and the `graphEvents` optional field on the JSONL delta.

#### Config schema additions

`assets/deep_review_config.json` gains the same `executor` block as the research skill (bit-identical six fields). Validation uses the shared `executor-config.ts` module — one contract, two consumers.

#### Setup flag parsing

The command doc at `.opencode/command/spec_kit/deep-review.md` now accepts the same five flags as the research command: `--executor`, `--model`, `--reasoning-effort`, `--service-tier`, `--executor-timeout`. Precedence is CLI > config file > schema default. Users get the same UX across both iterative skills — no need to learn two different configurations.

#### Contract documentation

`SKILL.md` replaces the forward-looking executor-CLI note with a full **Executor Selection Contract** section parallel to the research skill's, with review-specific JSONL schema requirements (`dimensions`, `filesReviewed`, `findingsSummary`, `newFindingsRatio`). `references/loop_protocol.md` §3 adds the **Executor Resolution** subsection.

#### Shared modules reused

All four shared modules from the research release (executor-config, prompt-pack, post-dispatch-validate, executor-audit) are consumed unchanged. No review-specific TypeScript was needed — the symmetric ship guarantees the same failure modes, recovery paths, and audit trail across both skills.

## Upgrade

No migration needed for existing users. `executor.kind` defaults to `"native"`, which preserves pre-release behavior exactly — same agent dispatch, same JSONL records with review dimensions, same findings registry semantics.

To opt into `cli-codex` dispatch for review:

```
/spec_kit:deep-review :auto "target" \
  --executor=cli-codex \
  --model=gpt-5.4 \
  --reasoning-effort=high \
  --service-tier=fast
```

Requires `codex` on PATH. The review path shares shared modules with `sk-deep-research` (see that skill's v1.8.0.0 entry for the shared infrastructure).

## Symmetry with sk-deep-research v1.8.0.0

Both skills ship the same feature in the same release cycle. Design rationale in the spec packet's `decision-record.md` ADR-005: asymmetric shipping creates the bug class Phase 017 Cluster D was built to prevent (code-graph sibling asymmetry where 6 handlers lacked contract vocabulary for months). Co-shipping avoids that pattern here.
