This release finishes what v1.5.0.0 started. Phase 018 wired `cli-codex` as the only new executor for review iterations; three more (`cli-copilot`, `cli-gemini`, `cli-claude-code`) were reserved in the config enum but rejected at load time. This release wires all three for review symmetric to `sk-deep-research` v1.9.0.0. Each CLI has a different flag surface so the shared config schema validates per-kind. The YAML workflow keeps owning state, convergence, findings registry, and the reducer. Cross-CLI delegation is documented as design intent.

> Spec folder: `.opencode/specs/system-spec-kit/026-graph-and-context-optimization/019-cli-runtime-matrix` (Level 3)

---

## New Features

#### cli-copilot wired as review executor

Setting `kind: cli-copilot` now dispatches review iterations through `copilot -p "PROMPT" --model X --allow-all-tools --no-ask-user` with the prompt rendered from the review-flavored prompt-pack template (the one shipped in v1.5.0.0). Review JSONL schema (dimensions, filesReviewed, findingsSummary, newFindingsRatio) is preserved unchanged across all executors. The positional-prompt constraint (copilot has no stdin) is handled the same way as the research path.

&nbsp;

#### cli-gemini wired as review executor

Review iterations dispatch through `gemini "PROMPT" -m gemini-3.1-pro-preview -s none -y -o text`. The model whitelist enforced in `executor-config.ts` applies to both skills. Review-specific constraints from the prompt pack (`correctness / security / traceability / maintainability` dimensions, `FAIL / CONDITIONAL / PASS` verdicts, claim adjudication requirements) are preserved in the rendered template that gemini reads.

&nbsp;

#### cli-claude-code wired as review executor

Review iterations dispatch through `claude -p "PROMPT" --model claude-opus-4-6 --permission-mode acceptEdits --output-format text` with optional `--effort`. The permission-mode override from `plan` to `acceptEdits` is required so claude-code can write the review narrative file and append to the JSONL state log.

&nbsp;

#### Per-kind validation shared with research

Both skills consume the same `executor-config.ts` module. The per-kind `EXECUTOR_KIND_FLAG_SUPPORT` map applies identically to review configs. Users setting a flag the chosen CLI does not support get the same typed error (for example `serviceTier` with `cli-gemini` rejects for review too).

&nbsp;

#### Executor audit on review records

The `record_executor_audit` step from v1.5.0.0 fires on any non-native review run. cli-copilot, cli-gemini, and cli-claude-code review iterations now produce the same `executor` audit block as cli-codex review iterations. The block is added to the last JSONL record per iteration so the state log retains the full review dimension schema plus the executor audit.

&nbsp;

#### Five setup flag options

The Q-Exec question in `deep-review.md` now offers all five executor kinds (A native, B cli-codex, C cli-copilot, D cli-gemini, E cli-claude-code) with the same brief per-CLI notes as the research command. Users get the same UX across both iterative skills so switching executors is identical regardless of which loop is active.

## Documentation

#### Executor Selection Contract table updated

The `SKILL.md` contract table now reports all four CLI kinds as "Shipped (spec 019)." Review-specific JSONL schema requirements (dimensions, filesReviewed, findingsSummary, newFindingsRatio) are unchanged from v1.5.0.0.

&nbsp;

#### Cross-CLI Delegation subsection

A new subsection under the Executor Selection Contract documents that any review executor can, in theory, shell out to other CLIs during iteration execution. Self-recursion is forbidden. Auth and environment propagation between CLIs is user-responsibility. Runtime enforcement is out of scope.

&nbsp;

#### Loop protocol references expanded

Section 3 of `references/loop_protocol.md` now describes Executor Resolution for all five kinds with their canonical dispatch shapes tailored to review (`review/prompts/` path, review JSONL schema, `@deep-review` agent). Cites per-kind flag-compatibility enforcement in the shared validator.

## Upgrade

No migration required. Default `kind: native` is unchanged. Users already on `cli-codex` for review get no behavior change. Users previously blocked from `cli-copilot`, `cli-gemini`, or `cli-claude-code` by `ExecutorNotWiredError` can now run those CLIs for review.

To opt into a new CLI for review, invoke:

```
/spec_kit:deep-review :auto "target" \
  --executor=cli-claude-code \
  --model=claude-opus-4-6 \
  --reasoning-effort=high
```

Or for gemini:

```
/spec_kit:deep-review :auto "target" \
  --executor=cli-gemini \
  --model=gemini-3.1-pro-preview
```

Each CLI must be on PATH and authenticated. Per-kind flag-compatibility is enforced in the shared validator; setting a flag the chosen CLI does not support rejects at config parse.
