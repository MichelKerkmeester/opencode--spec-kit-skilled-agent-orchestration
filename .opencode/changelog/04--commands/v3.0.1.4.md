## [**3.0.1.4**] - 2026-03-29

The create:changelog command can now publish GitHub releases in addition to creating changelog files, and it now has a canonical template that defines two formats -- compact (bullet points) for small releases and expanded (Problem/Fix paragraphs) for major ones. Previously, creating a changelog and publishing a release were separate manual steps, and the changelog format was only defined inline in the YAML workflows with no standalone template. Now the command handles the entire release flow with a single `--release` flag and generates content using a shared template referenced by both YAML workflows and PUBLIC_RELEASE.md.

> Spec folder: `.opencode/specs/03--commands-and-skills/038-cmd-create-changelog-and-releases/` (Level 2)

## What Changed

### Commands

- **create:changelog now supports GitHub release creation** -- Added a `--release` flag and Q3 prompt that optionally creates an annotated git tag, pushes it, composes plain-English release notes (stripping local wrapper lines), and runs `gh release create`. The release notes follow the PUBLIC_RELEASE.md Section 7 format: explain what was broken, what was fixed, and why it matters, written for a smart person who is not a developer.

- **Release notes are auto-composed from the changelog file** -- The command reads the changelog it just created, strips the local-only headers (`# vX.X.X`, `> Part of...`, `## [**X.X.X**] - date`), and uses the remaining content as the GitHub release body. A link back to the full changelog file is appended.

- **Error handling for release failures** -- New error recovery for tag conflicts (ask user to force-update or increment BUILD), auth failures (suggest `gh auth login`), and release conflicts (offer to update existing or skip).

### Changelog Template

- **New canonical template file** -- Created `.opencode/command/create/assets/changelog_template.md` defining two formats: compact (bullet points with bold labels, for releases under 10 changes) and expanded (numbered titles with Problem/Fix paragraphs, for 10+ changes or major releases). Includes writing style rules, format selection guide, category vocabulary, and links to real examples.

- **YAML workflows now reference the template** -- Both `create_changelog_auto.yaml` and `create_changelog_confirm.yaml` replaced their inline `changelog_template` block with a reference to the new template file. Step 4 (content generation) now reads the template first and selects compact or expanded format based on change count. Category headings updated to use PUBLIC_RELEASE.md vocabulary (Search, Saving Memories, Security, etc.) instead of generic labels (New, Changed, Fixed).

- **PUBLIC_RELEASE.md cross-references the template** -- Section 7 now points to the template file and the `/create:changelog --release` command. The pre-publish checklist includes an expanded format requirement for 10+ change releases.

- **Format reference section replaced** -- The inline format example in changelog.md Section 3 was replaced with a format selection table, links to the template, and pointers to real compact and expanded examples.
