The create:changelog command can now publish GitHub releases and has a proper template for how release notes should be written. Previously, creating a changelog and publishing a release were separate manual steps, and the format for writing release notes was scattered across inline code in workflow files with no single source of truth. Now the command handles the entire flow -- changelog creation, git tagging, and GitHub release publishing -- with one flag, and a standalone template defines two writing formats depending on how many changes a release contains.

> Spec folder: `.opencode/specs/03--commands-and-skills/038-cmd-create-changelog-and-releases/` (Level 2)

---

## What Changed

---

### Commands

- **Publishing a release no longer requires manual steps** -- Previously, after creating a changelog file you had to separately create a git tag, push it, and run the GitHub CLI to create a release page with properly formatted notes. The create:changelog command now does all of this when you add the `--release` flag. It creates the tag, strips the local-only header lines from the changelog, and publishes the release with correctly formatted notes.

- **Release notes are composed automatically from the changelog** -- The command reads the changelog it just created, removes the lines that only belong in the local file (the title line, the project link, and the date header), and uses the remaining content as the GitHub release body. A link back to the full changelog file is appended at the end.

- **Release failures are handled gracefully** -- If the tag already exists, the command asks whether to update it or bump the version. If the GitHub CLI isn't authenticated, it tells you to run the login command. If a release page already exists for that tag, it offers to update the existing one or skip.

---

### Documentation

- **Changelog format now has a standalone template** -- The format for writing changelogs was previously defined inline inside the YAML workflow files, with no way to read or reference it independently. A new template file defines two formats: a compact format (bullet points) for releases with fewer than 10 changes, and an expanded format (numbered sections with "Problem" and "Fix" paragraphs) for larger or more important releases. Both YAML workflows and the release documentation now reference this single template.

- **Writing style rules are consistent across all surfaces** -- The YAML workflows, the command documentation, and the public release guide all point to the same template and the same writing style rules. Category headings now use plain names (Search, Security, Commands, Documentation) instead of generic labels (New, Changed, Fixed).

- **The git-finish workflow now mentions the guided alternative** -- When you use the git-finish workflow and choose to create a release, it now tells you about the create:changelog command as a structured alternative to the manual release steps.
